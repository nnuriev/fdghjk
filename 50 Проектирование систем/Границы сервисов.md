# Границы сервисов

## TL;DR

Хорошая граница собирает внутри одного владельца данные, правила и изменения, которые должны оставаться согласованными, и оставляет снаружи стабильный бизнес-контракт. Её ищут по bounded context, инвариантам, языку предметной области, причинам изменения и требованиям к отказам, а не по таблицам, CRUD-ресурсам или слоям `controller/service/repository`.

Bounded context ещё не обязан быть отдельным процессом. Сначала границу дешевле выразить модулем с закрытыми данными и направленными зависимостями; независимый deploy оправдан, когда даёт измеримую автономию выпуска, масштабирования, безопасности или команды. Если обычная операция требует распределённой транзакции, синхронно обходит много сервисов и выпускается только согласованным набором, границы, скорее всего, проведены не там.

## Область применимости

Заметка описывает выделение логических и процессных границ в backend-системах. Она использует понятия bounded context из Domain-Driven Design (DDD), но не требует полной DDD-методологии и не задаёт организационную структуру компании.

## Ментальная модель

Граница отвечает на три вопроса:

1. **Кто принимает решение?** Один owner поддерживает инварианты и меняет состояние.
2. **Как остальные взаимодействуют?** Через контракт в терминах бизнеса, а не через внутренние таблицы.
3. **Что происходит при разрыве связи?** Внешний caller видит конечный набор исходов, а владелец сохраняет корректное локальное состояние.

Внутри границы модель должна быть связной: термины имеют одно значение, правила часто меняются вместе, а критичные инварианты помещаются в локальную транзакцию. Между границами допустимы разные модели одного реального объекта. `Customer` для Billing — плательщик и реквизиты, для Support — участник обращений; принуждать оба контекста к общей «универсальной» сущности значит связать их изменениями, которые им не принадлежат.

Сервисная граница — это граница владения, а process boundary — один из способов её развернуть. [[50 Проектирование систем/Монолит и сервисы|Монолит и набор сервисов]] могут реализовать одинаковые логические contexts с разной ценой коммуникации и эксплуатации.

## Как это устроено

### 1. Начать с capability и языка

Разбиение по business capability отвечает на устойчивый вопрос «за какой результат отвечает часть системы?»: принять заказ, зарезервировать запас, провести платёж. Горизонтальный `OrderController` отдельно от `OrderRepository` не создаёт автономии: изменение правила заказа всё равно проходит через все слои.

Признак context boundary — изменение смысла термина или набора правил. Если два подразделения спорят, что такое `active account`, возможно, это две модели с явным переводом, а не одна сущность с десятками необязательных полей.

### 2. Найти владельца инварианта

Инвариант должен проверяться там, где атомарно меняются данные, от которых он зависит. Если «available stock не бывает отрицательным», Inventory владеет остатком и операцией reservation. Order может запросить reservation, но не должен напрямую обновлять таблицу склада.

Сильный сигнал неверной границы — штатный use case, который требует синхронного двухфазного commit между сервисами. Некоторые workflows по природе распределены, но локальный инвариант не следует разносить только ради меньших процессов. Межконтекстный процесс выражают state machine, saga или событиями, принимая промежуточные состояния явно.

### 3. Закрыть данные и открыть контракт

У владельца данных может быть отдельная БД, schema или хотя бы модульный API, но другие части не обходят его правила прямыми writes. Shared database особенно опасна не физически, а правом произвольно читать и изменять чужую схему: контракт становится неявным, а deploy — координированным.

Публичный контракт передаёт необходимый факт или команду. Он не экспортирует ORM-модель целиком и не заставляет consumer повторять внутренние joins. Для асинхронной интеграции [[40 Распределённые системы/Transactional outbox и Change Data Capture|outbox]] позволяет владельцу атомарно связать своё изменение с публикацией события.

### 4. Выбрать форму связи

Синхронный вызов нужен, когда caller не может продолжить без ответа и отказ downstream должен войти в немедленный исход операции. Он добавляет latency и availability dependency: deadline верхнего запроса должен вместить всю цепочку, а fan-out повышает вероятность частичного отказа.

Событие подходит для уже состоявшегося факта и независимых реакций. Оно разрывает время жизни сторон, но вводит eventual consistency, schema evolution, повторную доставку и наблюдаемость lag. Команда через очередь остаётся адресованной владельцу действия; название topic не меняет семантику.

### 5. Проверить архитектурную автономию

Process boundary оправдана, если часть системы действительно может:

- выпускаться и откатываться без синхронного deploy consumers;
- владеть схемой данных и совместимостью контракта;
- масштабироваться по собственной нагрузке;
- иметь отдельную fault containment или security boundary;
- поддерживаться командой, способной нести on-call и менять её end to end.

Эти свойства конфликтуют. Независимый deploy требует backward compatibility, а изоляция отказа исчезает, если каждый запрос синхронно зависит от сервиса. Граница определяется не диаграммой, а наблюдаемым поведением изменений и сбоев.

## Сквозной пример: оформление заказа

Исходная система разделена по сущностям: `OrderService` пишет `orders`, `InventoryService` пишет `order_items.stock`, `PaymentService` меняет `orders.payment_status`. Для оформления заказа нужен общий lock и согласованный deploy трёх сервисов.

Переразбиение идёт по инвариантам:

1. **Order** владеет жизненным циклом заказа и решает, можно ли перейти из `draft` в `confirming`.
2. **Inventory** владеет остатком и создаёт reservation по уникальному `order_id`; только он проверяет, что доступный остаток не отрицателен.
3. **Billing** владеет payment attempt и не разрешает второй capture для того же бизнес-ключа.
4. Order запускает workflow. Успешные reservation и payment переводят заказ в `confirmed`; окончательный отказ — в `rejected`, а уже выполненные шаги получают явные compensation-команды.

При повторной доставке каждый владелец узнаёт свой бизнес-ключ и возвращает тот же outcome. Временное `confirming` видно как нормальное состояние, а не как «наполовину закоммиченная» транзакция. При этом contexts можно сначала оставить модулями одного deployable: границы данных и контрактов уже проверяемы, а сетевую цену добавляют только при появлении причины.

## Trade-offs и альтернативы

### Крупная или мелкая граница

Крупный context проще сохраняет локальные транзакции и требует меньше сетевого взаимодействия, но объединяет больше причин изменения. Мелкий даёт точечное владение и независимость, однако умножает контракты, промежуточные состояния и operational overhead. Выбирать следует минимальную границу, внутри которой связные инварианты и изменения ещё естественны, а не минимальное число строк кода.

### Общая модель или перевод между contexts

Общая модель уменьшает mapping в начале, если язык и жизненный цикл действительно совпадают. При разных значениях она превращается в shared kernel: любое изменение требует согласования. Явный anti-corruption layer дублирует часть структуры, зато локализует перевод и не даёт внешней модели проникнуть в доменные правила.

### Прямой запрос или локальная проекция

Запрос владельцу отдаёт актуальный ответ, но создаёт runtime dependency и latency. Локальная read projection быстрее и доступна при отказе owner, зато отстаёт и требует event contract. Решение зависит от допустимой stale-ness, а не от желания «не делать лишнюю копию».

## Типичные ошибки

### Один сервис на таблицу

- **Неверное предположение:** техническая сущность задаёт business capability.
- **Симптом:** один use case делает chatty fan-out и распределённый commit.
- **Причина:** данные одного инварианта разделены по CRUD-владельцам.
- **Исправление:** группировать поведение и данные по решению и транзакционной согласованности.

### Shared database как скрытый API

- **Неверное предположение:** прямой SQL — только оптимизация чтения.
- **Симптом:** миграцию нельзя выпустить без проверки неизвестных consumers.
- **Причина:** внутренняя schema стала неверсируемым контрактом.
- **Исправление:** один write owner, явные API/events и миграция зависимостей.

### Граница по оргструктуре без capability

- **Неверное предположение:** отдельная команда автоматически создаёт связный сервис.
- **Симптом:** команды постоянно координируют мелкие изменения и спорят о владельце данных.
- **Причина:** организационный разрез пересёк инварианты и поток изменений.
- **Исправление:** совместить ownership с end-to-end capability; оргструктуру считать фактором, а не единственным критерием.

### Преждевременно фиксированная декомпозиция

- **Неверное предположение:** границы можно точно вывести до изучения домена.
- **Симптом:** появляется distributed monolith с циклическими вызовами.
- **Причина:** сетевые boundaries закрепили гипотезу слишком рано.
- **Исправление:** сначала модульная изоляция и наблюдение за изменениями; extraction по подтверждённому seam.

## Когда применять

Границы пересматривают при появлении нового capability, конфликте моделей, распределённой транзакции, постоянных coordinated deploys или отличающихся требований к scale, security и availability. Не каждое неудобство требует нового сервиса: иногда достаточно исправить направление зависимостей внутри [[60 Go/Пакеты, модули и направление зависимостей|модульной структуры]].

Проверка границы проста по формулировке, но строга по последствиям: назовите владельца каждого инварианта, запретите внешние writes в его данные и смоделируйте отказ каждой внешней связи. Если корректный исход нельзя описать без знания внутренних таблиц соседа, контракт ещё не отделён.
