# Code readability

## TL;DR

Code readability, или читаемость кода, — способность читателя восстановить поведение, причины решений и границы ответственности без догадок и постоянных прыжков по репозиторию. Форматирование и стиль снижают визуальный шум, но не исправляют скрытый control flow, неявный lifecycle или boolean-параметр с непонятным смыслом.

Практический критерий: инженер, который не писал этот код, способен по локальному контексту предсказать happy path, один failure path и последствия изменения. Если объяснение существует только в голове автора или в обсуждении code review, код ещё не передал модель читателю.

## Ментальная модель

Чтение — это восстановление программы из сигналов. Имена дают предметные сущности, структура показывает границы, типы ограничивают допустимые состояния, control flow показывает порядок, а комментарии сохраняют причины, которых в синтаксисе нет.

У читателя ограничен рабочий контекст. Каждый скрытый alias, глобальный side effect, неявный порядок вызовов и переключатель `true/false` заставляет удерживать ещё одну гипотезу. Поэтому читаемый код не обязательно короткий. Он минимизирует число одновременно открытых вопросов:

- что здесь меняется;
- кто владеет ресурсом;
- где операция считается завершённой;
- какие ошибки ожидаемы;
- почему выбран этот необычный путь.

Читаемость оценивают со стороны reader, а не автора. Это прямо совпадает с принципом clarity в Google Go Style Guide: нужно ясно видеть и что делает код, и почему он устроен именно так.

## Как устроено

### Сначала сделать поведение явным

Имя должно обозначать роль в текущем контексте. `elapsed`, `retryBudget` и `pendingPayments` несут больше информации, чем `data`, `tmp` и `manager`; при этом длинное имя не компенсирует расплывчатое понятие. В Go имя package входит в каждое обращение, поэтому `http.Client` читается лучше, чем `httputils.HTTPClient`: контекст уже дал package.

Control flow читается легче, когда основной путь виден сверху вниз, а проверки предусловий и ошибок завершают ветку рано. Но механическое дробление в функции тоже имеет цену: десять одноразовых helpers заставляют постоянно переходить между файлами. Извлечение полезно, если фрагмент получил собственное имя, invariant или уровень абстракции.

Типы должны убирать неоднозначность на границе. Пара одинаковых `string`, пять `bool` или map со скрытым набором keys перекладывают контракт на память читателя. Предметный value type, options struct или отдельные операции делают допустимые комбинации заметнее. Эта связь раскрыта в [[50 Проектирование систем/Public API компонента|проектировании public API компонента]].

### Разделить «что» и «почему»

Код и тесты лучше всего показывают observable behavior. Комментарий нужен для информации, которую нельзя надёжно вывести из реализации: внешнее ограничение, доказательство корректности, причина обходного пути, ownership или совместимость. Комментарий, пересказывающий следующую строку, добавляет второй текст, который придётся синхронно менять.

Для exported Go API doc comment фиксирует назначение, результат, ошибки и необычные гарантии. Например, concurrency safety или смысл zero value должны быть названы, если caller не способен вывести их из type signature. Официальные Go Doc Comments отдельно требуют документировать более сильные гарантии concurrent use.

### Автоматизировать механические споры

Formatter и linter полезны там, где правило однозначно. Они убирают шум из review и делают необычный фрагмент заметным. Но formatter не знает предметный invariant, а метрика complexity не доказывает понятность. Автоматический сигнал должен вести к вопросу, а не заменять review поведения.

Review идёт по риску: contract, invariants, ownership, concurrency и failure semantics раньше вкусовых замечаний. Полный порядок разобран в [[50 Проектирование систем/Code review и refactoring LLD-решения в Go|заметке о code review и refactoring]].

## Пример или трассировка

Caller встречает вызов:

```go
send(ctx, order, 3, 500*time.Millisecond, true, false)
```

Чтобы объяснить его, нужно открыть declaration и выяснить смысл двух чисел и двух flags. Непонятно, `3` — число попыток или concurrency, `true` включает retry или asynchronous delivery, а `false` относится к logging или idempotency. Ошибка перестановки компилируется.

Граница становится явнее после введения предметной policy:

```go
send(ctx, order, DeliveryPolicy{
	MaxAttempts:  3,
	RetryBackoff: 500 * time.Millisecond,
	RequireAck:   true,
})
```

Теперь читатель видит единицы, defaults можно описать в одном type contract, а новый независимый параметр не меняет позиционный вызов. Цена тоже видна: для одного локального helper struct может оказаться лишним. Выбор оправдан, когда параметры образуют устойчивую policy или несколько call sites уже ошибаются в их смысле.

Наблюдаемый результат review: инженер объясняет retry и acknowledgement по call site, не открывая implementation; невозможный или конфликтующий набор options всё равно должен отклонять constructor/validation. Хорошие имена не заменяют invariant.

## Trade-offs

- Короткая функция снижает локальную нагрузку, пока её имя обозначает отдельную мысль. Чрезмерное дробление разрывает control flow и увеличивает navigation cost.
- Комментарий сохраняет rationale и внешние ограничения. Если он объясняет запутанный алгоритм, сначала стоит проверить, нельзя ли упростить код; иначе комментарий быстро расходится с реализацией.
- Предметный type делает контракт явным и запрещает часть ошибок, но для одноразовой локальной операции добавляет ceremony. Primitive уместен, если смысл очевиден из функции и позиции.
- Единообразие ускоряет чтение знакомого кода. Новый нестандартный приём оправдан, когда его локальная выгода доказана; «у нас везде так» не защищает неясный pattern.
- Читаемость call site иногда требует более сложной реализации API. Такая сложность допустима, если она действительно локализована и предотвращает misuse, а не маскирует неявный protocol.

## Типичные ошибки

- **Неверное предположение:** `gofmt` сделал код читаемым. **Симптом:** diff аккуратен, но reviewer не может назвать lifecycle и failure path. **Причина:** formatter решает presentation, а не semantics. **Исправление:** восстановить contract, ownership и control flow; форматирование оставить автоматике.
- **Неверное предположение:** больше комментариев всегда помогает. **Симптом:** комментарий утверждает одно, код после refactoring делает другое. **Причина:** продублирован observable behavior. **Исправление:** выразить «что» именами, типами и структурой; комментировать rationale, ограничения и неочевидные гарантии.
- **Неверное предположение:** чем короче имя, тем idiomatic Go. **Симптом:** вне трёх строк `c`, `m` и `s` теряют смысл. **Причина:** правило локальных коротких имён применили к wider scope. **Исправление:** выбирать длину по области видимости и роли; учитывать имя package на call site.
- **Неверное предположение:** helper на каждые пять строк снижает complexity. **Симптом:** для понимания одного branch нужно открыть шесть функций. **Причина:** механическая длина принята за cognitive boundary. **Исправление:** извлекать связную операцию с именем и contract, оставлять линейные детали рядом.
- **Неверное предположение:** reviewer обязан понять сложный код после устного объяснения. **Симптом:** решение остаётся только в review thread. **Причина:** будущий читатель этого контекста не увидит. **Исправление:** упростить структуру или перенести необходимое rationale в код и документацию.

## Когда применять

Проверяйте читаемость при каждом изменении public API, concurrency/lifecycle, error handling и предметного правила. Полезный review-вопрос звучит конкретно: «какой результат будет при timeout после первого side effect?» Если ответ требует читать несвязанные packages или угадывать defaults, граница нуждается в улучшении.

Style guide задаёт общий язык, но не превращайте readability в gate личного вкуса. Google Engineering Practices советует отделять обязательные замечания от `Nit` и опираться на технические факты. Цель — уменьшить вероятность неверного прогноза и следующего дефекта, а не унифицировать почерк любой ценой.
