# Мультитенантность

## TL;DR

Мультитенантная система обслуживает несколько tenants на общей платформе, но обязана сохранять изоляцию данных, доступа, производительности, отказов и стоимости. `tenant_id` в таблице закрывает только часть задачи. Tenant context должен появиться на доверенной границе, пройти через API, jobs, cache keys, events, logs и billing, а каждый слой обязан либо проверить его, либо получить уже ограниченный capability.

Архитектура лежит на спектре: shared rows, отдельные schemas/databases, выделенные cells или полностью dedicated stack. Более сильная изоляция повышает стоимость и operational cardinality. Часто выигрывает tiered/cell-based модель: большинство tenants разделяют стандартизированную cell, крупные или регулируемые получают отдельное placement по тем же control-plane контрактам.

## Ментальная модель

Tenant — это boundary владения и blast radius, а не только поле фильтра.

```text
identity -> tenant context -> policy -> placement -> resource budget -> audit/cost
```

Если хотя бы один hop теряет context, система способна прочитать чужую строку, положить private response в общий cache или дать одному tenant исчерпать общий pool.

## Как устроено

### Tenant identity и authorization

Tenant context выводится из проверенной identity/membership, а не принимается из произвольного header. Пользователь может состоять в нескольких tenants, поэтому выбор active tenant связывается с session/token и повторно проверяется владельцем resource.

Service-to-service call передаёт subject/tenant либо получает scoped internal capability. Background job хранит tenant ID в durable payload; worker не полагается на process-local context. Audit event фиксирует actor, tenant, resource и policy version.

### Модели изоляции данных

**Shared tables** с `tenant_id` дают высокую плотность и простую эксплуатацию. Every primary/unique/foreign key и query plan должны учитывать tenant. Уникальность email «внутри tenant» означает key `(tenant_id, normalized_email)`, а не global unique. PostgreSQL Row-Level Security добавляет storage-level policy, но не отменяет application tests и privileged-role review.

**Schema/database per tenant** усиливает isolation, backup/restore и residency, но умножает migrations, connections, catalogs и monitoring series. Тысячи idle connection pools способны стать bottleneck раньше данных.

**Cell architecture** размещает ограниченный набор tenants в автономной копии data plane. Control plane ведёт directory `tenant -> cell`, provisioning и policy. Failure cell затрагивает bounded cohort, а новая cell добавляет capacity. Цена — routing, fleet/version skew, rebalancing tenants и отдельное восстановление control plane.

### Isolation за пределами базы

Cache key начинается с tenant/authorization domain. Event partition и consumer state сохраняют tenant context. Object path/bucket policy не позволяет выбрать чужой prefix. Search filter применяется server-side до pagination и ranking; post-filter результата способен утечь через facets/counts.

Compute isolation использует per-tenant rate, concurrency, queue и cost-unit budgets. Общий FIFO допускает noisy neighbor: один bulk import заполняет очередь раньше интерактивных задач остальных. Weighted fair queues, tenant caps и separate priority pools сохраняют fairness.

### Control plane и lifecycle

Control plane создаёт tenant, выбирает plan/region/cell, выдаёт keys/policy, запускает suspension/deletion и перенос. Data plane продолжает безопасно обслуживать последнюю валидную конфигурацию при кратком outage control plane; неизвестный tenant deny-ится.

Deletion охватывает primary rows, events, object versions, caches, indexes, backups и audit по различным retention rules. Tenant export/restore требует consistent boundary и проверки ownership.

### Observability и cost attribution

Нельзя помещать миллионы tenant IDs в metric labels. SLO измеряется глобально и по bounded cohorts/cells/tier, а top offenders анализируются через logs/analytics. Для enterprise tenant допустима отдельная SLI series, если cardinality budget контролируется.

Unit economics связывает tenant plan с storage, requests, egress и async work. Quota без cost attribution защищает capacity, но не показывает убыточный workload.

## Сквозной пример: B2B document service

Платформа обслуживает 50 тыс. малых tenants и 20 крупных регулируемых клиентов.

1. Control plane хранит `tenant_directory(tenant_id, cell_id, region, isolation_tier, policy_version)` и не лежит на каждом document read.
2. Edge проверяет user token и membership, затем маршрутизирует запрос в назначенную cell. Cell повторно проверяет audience/tenant.
3. Малые tenants разделяют SQL cluster; ключ документа `(tenant_id, document_id)`, unique name ограничен tenant. RLS служит дополнительным guard.
4. Body хранится в object storage по opaque object key; metadata связывает key с tenant. Pre-signed URL ограничен exact object/method/expiry.
5. Cache key `(cell, tenant, document, version, authorization_variant)` не смешивает representations.
6. Bulk OCR идёт через weighted queues с per-tenant concurrency. Исчерпавший quota tenant получает delayed/rejected jobs, остальные сохраняют latency.
7. Регулируемый tenant получает dedicated cell и key hierarchy в требуемом регионе, но использует тот же API и control-plane lifecycle.

При отказе одной shared cell затронут ограниченный cohort. Directory не перенаправляет tenant на пустую cell автоматически: сначала восстанавливаются data/RPO и write authority. После recovery выполняется reconciliation jobs и cache warm-up.

## Trade-offs

Shared database дешевле и проще для fleet-wide changes, но требует безошибочного tenant scoping и создаёт общий failure/performance domain. Dedicated database повышает isolation и точечный restore, зато migrations и connections растут с числом tenants.

Hard quotas защищают соседей, но могут отклонять burst платящего клиента при свободной global capacity. Token bucket с guaranteed floor и borrowable burst лучше использует запас, если возврат borrowed capacity быстрый.

Cell уменьшает blast radius и даёт линейный scaling unit, но создаёт skew: разные cells могут иметь разные versions, load и hot tenants. Нужны uniform automation, placement/rebalance и cell-level SLO.

## Типичные ошибки

- **Неверное предположение:** фильтр `WHERE tenant_id = ?` достаточен. **Симптом:** один забытый query/cache key возвращает чужие данные. **Причина:** isolation держится на ручной дисциплине одного слоя. **Исправление:** scoped repositories/keys, storage policy, tests и audit.
- **Неверное предположение:** gateway передал tenant, downstream доверяет header. **Симптом:** внутренний caller подменяет tenant. **Причина:** context не связан с authenticated identity/capability. **Исправление:** signed/scoped identity и повторная resource authorization.
- **Неверное предположение:** отдельная database автоматически даёт изоляцию. **Симптом:** общий worker pool или credentials создают общий blast radius. **Причина:** учтён только data plane storage. **Исправление:** проверить compute, queues, network, secrets, observability и control plane.
- **Неверное предположение:** один global queue справедлив. **Симптом:** bulk tenant увеличивает latency всех jobs. **Причина:** FIFO не учитывает ownership/cost. **Исправление:** per-tenant budgets, fair scheduling и bounded queues.
- **Неверное предположение:** удаление primary rows завершает deletion. **Симптом:** данные остаются в object versions, search или backups. **Причина:** derived copies не вошли в lifecycle. **Исправление:** deletion workflow, lineage, retention и verification.

## Когда применять

Tenant boundary проектируют до schema/cache/event contracts. Степень изоляции выбирают по regulatory, blast radius, workload skew, restore и unit economics. Tiered модель оправдана, если дорогая dedicated topology нужна небольшой части клиентов; она должна сохранять один автоматизированный platform contract.
