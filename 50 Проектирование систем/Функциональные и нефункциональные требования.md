# Функциональные и нефункциональные требования

## TL;DR

Functional requirements описывают наблюдаемый результат: кто выполняет операцию, над каким объектом и какой исход получает. Non-functional requirements ограничивают способ и качество выполнения: latency, availability, durability, consistency, privacy, scale, recovery и стоимость.

На System Design интервью требования нужны как фильтр решений. Формулировка «поддержать загрузку файла» ещё не говорит, нужен ли multipart upload, а «система должна быть быстрой» не задаёт никакого ограничения. Хорошее требование имеет scope, измеримый outcome, приоритет и условие, при котором его можно проверить.

## Ментальная модель

Функция задаёт разрешённый переход состояния. Нефункциональное требование задаёт коридор, в котором переход должен произойти.

```text
command + preconditions -> state transition -> observable outcome
                           |                |
                           +-- latency -----+
                           +-- consistency -+
                           +-- durability --+
```

Разделение не абсолютно. «Экспортировать отчёт» выглядит как функция, но после требования «результат должен пережить закрытие браузера и быть доступен 24 часа» появляется durable operation resource и object storage. Качество изменило форму API.

## Functional requirements

### Формула сценария

Минимальная запись содержит:

1. **actor:** пользователь, внешний сервис, оператор или scheduler;
2. **trigger:** команда или событие;
3. **preconditions:** право доступа, состояние объекта, idempotency key;
4. **outcome:** созданный ресурс, изменённое состояние либо терминальная ошибка;
5. **observable boundary:** когда caller вправе считать операцию завершённой.

Для notification system «отправить уведомление» слишком расплывчато. Полезнее: «producer создаёт уведомление с business key; API долговечно принимает его один раз; подписанные каналы получают попытки доставки; producer читает итог по каждому каналу». Здесь уже видны дедупликация, async state и status API.

Функции приоритизируют. В core flow оставляют два-четыре сценария, которые определяют архитектуру. Admin UI, аналитика, рекомендации и редкие bulk-операции можно вынести за scope, если интервьюер не обозначил их критичными.

### Команды, запросы и события

Команда просит владельца изменить состояние и допускает отказ. Query читает состояние без обещания изменить его. Event утверждает, что факт уже состоялся. Смешение приводит к ложным гарантиям: событие `CreateOrder` на самом деле является командой, а consumer не знает, кто отвечает за отказ.

Для каждой мутации зафиксируйте идемпотентность по правилам [[20 Бэкенд/Идемпотентные и неидемпотентные операции|семантики операций]] и стабильный business key по [[20 Бэкенд/Ключи идемпотентности и дедупликация запросов|контракту дедупликации]].

## Non-functional requirements

### Качества надо измерять

Заменяйте прилагательное на SLI, target, окно и population:

- вместо «высокая доступность» — `99,95%` успешных `GET /timeline` за 28 дней, исключая невалидные запросы;
- вместо «низкая задержка» — p99 server latency ниже 200 мс для payload до 64 KiB;
- вместо «данные не теряются» — acknowledged ledger entries имеют RPO = 0 для отказа одного узла и проверяемый restore из backup при потере региона;
- вместо «данные быстро сходятся» — собственное сообщение видно автору сразу, остальным участникам не позже 2 секунд в 99,9% случаев.

Детали таких контрактов разобраны в [[50 Проектирование систем/SLO в System Design|заметке о SLO]].

### Ограничения и policy

Не все non-functional requirements выражаются одним процентом. Data residency задаёт разрешённое место хранения. Retention определяет, когда данные обязаны быть удалены. RTO задаёт время восстановления, а security boundary — кто может принять решение об авторизации. Такие ограничения всё равно требуют проверяемого теста или процедуры.

Нагрузка тоже входит в контракт: DAU, peak requests/s, размер объекта, read/write ratio, burst и прогноз роста. Без workload нельзя отличить requirement от пожелания. Один SQL-узел и шардированный wide-column cluster способны реализовать одинаковую функцию, но рассчитаны на разную цену и пределы.

### Согласовать конфликты

Требования часто противоречат друг другу. Global write availability во время partition конфликтует с единым строгим порядком; RPO = 0 в другом регионе добавляет межрегиональную задержку; низкая стоимость конфликтует с горячим standby полной мощности.

Задача кандидата — показать конфликт и запросить продуктовый приоритет. CAP не выбирает за бизнес. Он объясняет, почему одновременно обещать все свойства в конкретном failure mode нельзя.

## Сквозной пример: требования к чату

Начальная формулировка: «спроектировать групповой чат».

Core functional requirements:

1. участник отправляет текстовое сообщение в комнату;
2. участники получают новые сообщения и могут дочитать историю после reconnect;
3. клиент видит `accepted`, `delivered` и окончательный отказ;
4. повтор submit с тем же client message ID не создаёт второе сообщение.

Non-functional requirements как interview assumptions:

- 10 млн DAU, пик 200 тыс. accepted messages/s;
- p99 durable acceptance внутри домашнего региона ≤ 150 мс;
- 99,99% accepted requests за 28 дней;
- acknowledged message переживает потерю одного узла без потери;
- порядок гарантируется внутри комнаты, глобального порядка нет;
- сообщения других участников появляются в 99,9% случаев не позже 2 секунд;
- история хранится год, удаление пользователя исполняется по отдельной policy.

Вне scope: звонки, end-to-end encryption, moderation ML и поиск по вложениям.

Из этих требований следуют решения: `client_message_id` становится idempotency key; ordering key совпадает с `room_id`; synchronous path заканчивается после durable append, а fan-out выполняется асинхронно; история partitioned по комнате и времени. Если потом понадобится E2EE, data model и server-side search придётся пересмотреть. Это нормальная эволюция, а не скрытый дефект первого scope.

## Trade-offs

Подробный список функций снижает риск забыть edge case, но съедает время и маскирует critical path. На интервью лучше глубоко зафиксировать core flow и явно назвать исключённые возможности.

Жёсткий SLO облегчает проверку, но стоит денег и ограничивает изменения. Начальный target должен вытекать из пользовательского ущерба, а не из текущей красивой метрики. Google SRE рекомендует избегать 100%: абсолютная цель лишает систему error budget и часто ведёт к неоправданной сложности.

Один общий SLO проще, но смешивает разные классы workload. Интерактивный read и batch export обычно получают отдельные latency/availability objectives.

## Типичные ошибки

- **Неверное предположение:** feature list равен требованиям. **Симптом:** неизвестно, когда операция считается успешной и что происходит при повторе. **Причина:** не описаны preconditions, outcome и observable boundary. **Исправление:** пройти один сценарий как state transition.
- **Неверное предположение:** «scalable» — проверяемое качество. **Симптом:** невозможно оценить storage и partitioning. **Причина:** отсутствуют workload и growth horizon. **Исправление:** назвать baseline, peak, distribution и горизонт прогноза.
- **Неверное предположение:** все желаемые качества совместимы. **Симптом:** обещаны global strong consistency, zero latency и write availability при partition. **Причина:** failure mode не был сформулирован. **Исправление:** разбирать свойства для конкретного отказа и просить приоритет.
- **Неверное предположение:** out-of-scope можно не проговаривать. **Симптом:** интервьюер и кандидат оценивают разные системы. **Причина:** граница задачи осталась неявной. **Исправление:** назвать исключения и условие их возврата в scope.

## Когда применять

Технику используют в начале design interview, RFC и architecture review. Требования пересматривают после появления измерений: допущение не надо защищать, его надо заменить наблюдаемым распределением и пересчитать зависимые решения.
