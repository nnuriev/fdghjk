# Проектирование платформы метрик и логов

## TL;DR

Платформа принимает telemetry локально в каждом регионе, быстро подтверждает батч после записи в реплицированный журнал и асинхронно строит два разных serving-слоя: time-series storage для метрик и полнотекстовый индекс плюс object storage для логов. Общий ingestion экономит сетевые и операционные расходы, но модель данных и запросы объединять нельзя: метрики агрегируются по series и time range, логи ищутся по полям и тексту.

Главный компромисс — freshness против стоимости и изоляции. Синхронная индексация уменьшает задержку появления данных, но связывает доступность приёма с поисковым кластером. Durable log разрывает эту связь: при отказе indexer приём продолжается, backlog растёт, а пользователи видят явно измеряемое отставание.

## Контекст и ментальная модель

Пользователи — команды разработки и эксплуатации. Они отправляют метрики и структурированные логи, строят dashboards, выполняют ad hoc запросы и настраивают alerts. Платформа не заменяет бизнес-аналитику: её задача — быстро объяснить состояние production-системы при заведомо неполных и иногда дублированных данных.

Ментальная модель состоит из трёх независимых плоскостей:

- ingestion plane принимает, проверяет tenant и ограничивает нагрузку;
- storage plane преобразует поток в долговечные time-partitioned данные;
- query plane читает готовые сегменты и не может остановить приём тяжёлым запросом.

Семантика доставки — at-least-once. Для логов source-generated `event_id` позволяет убрать точные повторы в ограниченном окне. Для метрик ключом служит `(tenant, series_id, timestamp)`; поздняя повторная точка либо перезаписывает то же значение по явной политике, либо учитывается один раз при compaction. Механика delivery и backlog уже разобрана в [[40 Распределённые системы/Очереди, streams, группы потребителей и DLQ|заметке о messaging]], а здесь важна её роль в архитектуре telemetry.

## Требования

### Функциональные

- принимать метрики и структурированные логи через OTLP 1.10.0 по отдельным signal-specific контрактам;
- проверять tenant, schema limits, timestamp и размер батча;
- выполнять range query и агрегации по метрикам;
- искать логи по tenant, времени, индексированным полям и тексту;
- задавать retention, sampling и cardinality policy по tenant;
- строить recording rules, downsampling и alerts асинхронно;
- экспортировать холодные данные и выполнять replay после исправления pipeline.

### Нефункциональные и SLO

Интервью-допущение задаёт такие цели:

| Характеристика | Цель |
| --- | --- |
| Доступность ingestion API | 99,95% за 30 дней в регионе |
| Доступность query API | 99,9% за 30 дней; отказ query не расходует ingestion budget |
| Приём батча | p99 ≤ 200 ms до durable acknowledgement |
| Freshness | p99 ≤ 30 s для метрик и ≤ 60 s для логов в штатном режиме |
| Запрос метрик за последние 24 h | p95 ≤ 2 s при ограниченной cardinality |
| Долговечность | после ack допускается RPO 0 внутри региона; для потери региона RPO ≤ 5 min |
| Consistency | eventual для query; read-your-writes не обещается, внутренний status API лишь показывает продвижение `ingest_offset` в serving-слое |

Availability не скрывает stale data. Если ingestion отвечает, но индекс отстаёт на 20 минут, SLO freshness нарушен. Это отдельный индикатор, а не «успешный» запрос к старому индексу.

### Вне scope

Распределённая трассировка, машинное обучение над логами, бессрочное хранение и запросы без tenant/time bounds остаются за границей дизайна.

## Оценка нагрузки и ёмкости

Все числа ниже — интервью-допущения, а не характеристики конкретного продукта.

**Метрики.** 20 000 instances отправляют в среднем 100 samples/s: `20 000 × 100 = 2 000 000 samples/s`, пик — 5 000 000/s. При допущении 16 B на sample после batching и compression:

```text
2 000 000 × 16 B × 86 400 = 2,7648 TB/day
2,7648 × 30 days × 2 replicas = 165,888 TB
```

**Логи.** Средний поток — 500 000 events/s, пик — 1 500 000/s. Средний сжатый event принят равным 300 B:

```text
500 000 × 300 B × 86 400 = 12,96 TB/day
12,96 × 14 days × 2 replicas = 362,88 TB
search index = 20% × 12,96 × 14 × 2 = 72,576 TB
```

Итого hot footprint около `601,344 TB`. С 30% headroom планируется примерно `782 TB`. Object storage для cold retention считают отдельно: цена там определяется сроком хранения, классом storage, retrieval и межрегиональной копией. Реальный bytes/sample и коэффициент индекса получают только из нагрузочного прогона на собственной cardinality.

Пропускную способность проверяют по трём осям: network ingress на пике, sequential write журналов и CPU на parsing/compression/indexing. Среднее значение не годится для выбора числа partitions; брать нужно peak с запасом и сценарием повторного проигрывания backlog.

## API и модель данных

```http
POST /v1/metrics
Authorization: Bearer <tenant credential>
Content-Type: application/x-protobuf
Content-Encoding: gzip

<ExportMetricsServiceRequest>

POST /v1/logs
Authorization: Bearer <tenant credential>
Content-Type: application/json

<JSON mapping of ExportLogsServiceRequest>
```

Это внешний OTLP/HTTP-контракт. Collector отвечает только после durable append внутреннего batch. Полный успех возвращает `200 OK` и соответствующий `ExportMetricsServiceResponse` либо `ExportLogsServiceResponse`. Частичный успех тоже возвращает `200 OK`, но заполняет `partial_success` с числом отклонённых points/records и сообщением; такой request клиент не повторяет. Retryable failures используют только предусмотренные OTLP коды `429`, `502`, `503` и `504`. Метрики и логи не объединяются в один OTLP request.

Между regional collector и ingestion gateway действует отдельный внутренний контракт:

```http
POST /internal/v1/durable-batches
Content-Type: application/json

{
  "source_id": "collector-eu-17",
  "batch_id": "01J...",
  "metrics": [...],
  "logs": [...]
}
```

Gateway возвращает `202 Accepted`, `ingest_offset` и детальный список отклонённых элементов только после quorum commit. Collector переводит этот результат в signal-specific OTLP response. Повтор внутреннего batch с тем же `(tenant, source_id, batch_id)` безопасен в deduplication window; молча отбросить series из-за cardinality limit нельзя.

```http
GET /v1/metrics/query_range?query=...&start=...&end=...&step=...
POST /v1/logs/search
GET /internal/v1/ingestion/offsets/{ingest_offset}
```

Модель метрики: `series_id = hash(tenant_id, metric_name, canonical_labels)`, затем пары `(timestamp, value)`. Metadata dictionary хранит labels отдельно от compressed chunks. Модель логов: `(tenant_id, stream_id, timestamp, event_id, indexed_fields, body, ingest_offset)`. Большой body остаётся в object storage, а индекс хранит searchable terms и pointer.

Высококардинальные labels создают новую series на каждое значение. Поэтому `request_id`, email и случайный URL запрещены как metric labels, но допустимы как поля лога с отдельной политикой индексации. Причина выбора TSDB объяснена в [[30 Данные/Time-series database|заметке о time-series storage]], а устройство полнотекстового serving-слоя — в [[30 Данные/Search index|заметке о search index]].

## Архитектура и критические потоки

```text
SDK/agent -> regional collector -> ingestion gateway -> durable event log
                                                    |-> metric processors -> TSDB chunks -> object storage
                                                    |-> log processors -> search index + object storage
query API -> planner -> metadata/cache -> hot shards -> cold object storage
alert/rule workers <- committed metric stream
```

Collector выполняет batching, compression и bounded buffering на узле приложения. Gateway аутентифицирует tenant, применяет quota, валидирует envelope и маршрутизирует batch в partition журнала. Parsers и enrichers версионируют schema; ошибки уходят в quarantine, а не блокируют partition навсегда.

### Write path и end-to-end trace

1. Collector `collector-eu-17` объединяет 10 000 samples и 2 000 log events, присваивает `batch_id`.
2. Gateway проверяет пределы tenant и пишет batch с replication quorum в региональный event log. После этого отвечает `202` и `ingest_offset=eu/42/991`.
3. Metric consumer канонизирует labels, вычисляет `series_id`, сортирует точки внутри time bucket и пишет immutable chunk. Log consumer сохраняет compressed body в object storage и обновляет inverted index.
4. Оба consumer публикуют committed offset. Запрос статуса показывает, дошёл ли `eu/42/991` до нужного serving-слоя.
5. Повтор batch после потерянного HTTP-ответа не создаёт второй лог благодаря `event_id`; повторная metric point схлопывается по ключу sample.

Наблюдаемый результат: потеря ответа не теряет telemetry и не удваивает видимый результат. Если indexer сломан, шаг 2 остаётся доступным, а freshness SLO и backlog alarm показывают деградацию.

### Read path

Query frontend сначала авторизует tenant и ограничивает time range, cardinality и cost. Planner выбирает time partitions, дробит запрос на shard-local subqueries, применяет partial aggregation рядом с данными и сливает результат. Metadata и результаты коротких dashboard-запросов кэшируются; отрицательный результат не кэшируется надолго, потому что index ещё догоняет ingestion.

## Масштабирование и надёжность

**Storage.** Event log нужен для replay и backpressure, TSDB — для columnar/time-range чтения, search index — для поиска, object storage — для дешёвых immutable chunks. SQL остаётся для tenant policies, schemas и audit, но не для основного telemetry потока.

**Partitioning и replication.** Журнал делится по `(tenant, hash(series_or_stream))`, а storage ещё и по time bucket. Крупному tenant выделяют отдельные virtual shards. Репликация журнала синхронна внутри региона; chunks копируются в другую failure domain и асинхронно в DR-регион. Partition key не должен содержать только timestamp: иначе текущая минута станет одним hot shard.

**Caching.** Кэшируются compiled query plans, metadata dictionaries, recent immutable chunks и dashboard results. Cache key включает tenant, query, time range, step и data watermark. Это предотвращает выдачу результата от старой watermark как будто он свежий.

**Async processing.** Indexing, compaction, downsampling, retention deletion и alert evaluation работают как независимые consumer groups. Каждый имеет собственный lag budget. Backpressure ограничивает producers по tenant; бесконечный локальный buffer лишь переносит OOM ближе к приложению.

**Multi-region и DR.** Writes идут в ближайший регион. Tenant queries по умолчанию читают регион происхождения; глобальный запрос fan-out делает явно и дороже. Cross-region replication копирует immutable chunks и offsets. При потере региона DNS переключает ingestion, но новый регион начинает с нового epoch; overlapping data дедуплицируется. План DR регулярно проверяет восстановление catalog, object manifests и encryption keys, а не только наличие файлов. Связь RPO/RTO и цены резервной мощности разобрана в [[40 Распределённые системы/Disaster recovery|заметке о disaster recovery]].

**Capacity и стоимость.** Capacity owner еженедельно следит за samples/s, events/s, bytes/s, active series, index ratio, compression, compaction debt и query scanned bytes. Главные статьи расходов — replicated hot storage, CPU индексации, cross-region egress и ad hoc scans. Снижение retention, selective indexing и downsampling обычно дешевле общего hardware upgrade, но меняет продуктовый контракт.

## Failure modes

| Отказ | Наблюдаемый симптом | Обнаружение | Реакция |
| --- | --- | --- | --- |
| Search index недоступен | приём логов работает, поиск отстаёт | consumer lag и freshness watermark | остановить тяжёлые queries, сохранить backlog, восстановить index replay-ем |
| Broker partition потерял quorum | gateway не может durable-ack | publish latency/errors | вернуть retryable error, не подтверждать batch, collector хранит bounded spool |
| Hot tenant/series | lag и p99 растут на одном shard | top partition throughput, active series | split virtual shard, quota, cardinality guard |
| Object storage отвечает медленно | cold queries и compaction зависают | dependency latency, queue age | deadline, circuit breaker, оставить hot path изолированным |
| Duplicate или out-of-order batch | скачки counter и повторные логи | dedup hits, late sample ratio | dedup window, event-time policy, idempotent compaction |
| Регион потерян | локальные ingestion/query недоступны | synthetic probes из других регионов | failover epoch, асинхронный replay, объявить фактический RPO |
| Ошибка schema rollout | quarantine резко растёт | reject ratio по schema version | rollback parser, replay raw batches |

## Безопасность

Tenant identity выводится из mTLS или service credential, а не из пользовательского поля. Авторизация применяется на ingestion и на каждом shard-read. Данные шифруются в transit и at rest; ключи разделены по средам, а доступ к raw logs жёстче доступа к агрегированным метрикам.

Нужны limits на batch size, decompression ratio, label count, field length, regex complexity и query scanned bytes. Logs часто содержат PII и credentials, поэтому collector выполняет redaction до durable log, а сервер применяет вторую policy. Audit фиксирует чтение чувствительных streams и изменение retention.

## Observability и SLO

Платформа наблюдает за собой через отдельный минимальный канал, иначе её отказ ослепит команду. Основные SLI:

- доля durable acknowledgements и их latency;
- end-to-end freshness от event timestamp и от ingest timestamp;
- consumer lag в bytes и seconds;
- rejected samples/events по причине;
- active series, index growth и compaction debt;
- query success, p95/p99 и scanned bytes;
- доля partial results и stale watermark.

Burn-rate alerts привязаны к ingestion, query и freshness SLO отдельно. Operational ownership включает on-call, tenant abuse playbook, восстановление индекса из журнала, capacity review и регулярный DR exercise.

## Эволюция решения и миграции

1. **Первый год:** один регион, общий durable log, managed TSDB/search, строгие limits. Raw batch сохраняется коротко для replay.
2. **Рост:** отделить query fleet от ingestion, добавить per-tenant virtual shards, object storage и downsampling.
3. **Несколько регионов:** local writes, глобальный catalog, asynchronous chunk replication, проверяемый failover epoch.
4. **Смена storage:** processors делают dual write в старый и новый format. Shadow queries сравнивают counts, aggregates и freshness; backfill идёт по immutable time buckets. Затем read-by-time-range переключается canary-группами, старое хранилище остаётся read-only до истечения rollback window.

Миграция без остановки опирается на versioned envelope и raw replay. Нельзя сначала удалить старый parser, а потом надеяться перечитать исторические batches новым несовместимым кодом.

## Trade-offs и альтернативы

- **Единый кластер или отдельные metrics/logs stacks.** Общий ingestion и control plane дешевле. Отдельные serving-кластеры лучше изолируют модели нагрузки и failure domains.
- **Push или pull.** Push естественен для short-lived jobs и multi-region agents; pull проще контролирует discovery и scrape pressure. Gateway может принимать оба, но приводит их к одной envelope model.
- **Синхронный index или durable log.** Первый даёт меньшую freshness latency при малом масштабе. Второй переживает отказ downstream и поддерживает replay ценой eventual consistency и storage журнала.
- **Полный индекс или selective fields.** Полный индекс улучшает ad hoc поиск, но резко увеличивает CPU и storage. Практический default — индексировать известные поля, body держать в object storage.

## Типичные ошибки

### Query и ingestion делят один ресурсный пул

- **Неверное предположение:** средняя нагрузка запросов мала.
- **Симптом:** incident вызывает много dashboards и одновременно telemetry drops.
- **Причина:** diagnostic reads вытесняют writes в момент, когда данные нужнее всего.
- **Исправление:** разные pools, quotas и failure budgets; degraded query не останавливает durable ingestion.

### Cardinality считают позже

- **Неверное предположение:** число metric names определяет объём.
- **Симптом:** один новый label умножает series и память index.
- **Причина:** capacity зависит от комбинаций label values.
- **Исправление:** per-tenant budgets, pre-ingest estimator, top offenders и schema review.

### Ack возвращается до durable boundary

- **Неверное предположение:** in-memory queue переживёт deploy.
- **Симптом:** accepted batches исчезают после crash.
- **Причина:** HTTP-успех опередил replicated append.
- **Исправление:** ack после quorum commit либо честный контракт best-effort.

## Когда применять

Архитектура подходит для внутренней multi-tenant observability-платформы, где ingestion значительно тяжелее control plane, данные допускают eventual query consistency, а replay важнее мгновенного общего индекса. Для небольшой команды managed backend дешевле собственной платформы: здесь операционная сложность оправдывается только масштабом, требованиями к изоляции или контролем стоимости.
