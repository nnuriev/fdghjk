# Архитектурная схема и критические потоки

## TL;DR

High-level architecture — это карта ответственности и потоков, а не каталог технологий. Схема должна позволить пройти минимум один write path, один read path и один failure path: где принимается решение, когда изменение становится durable, как строятся derived copies, какую версию читает клиент и что происходит при недоступности каждого обязательного звена.

Начинайте с минимального skeleton. Добавляйте cache, broker, search и replicas только после того, как названо требование, которое компонент выполняет, и новая failure boundary, которую он создаёт.

## Ментальная модель

У схемы есть узлы и типизированные рёбра.

Узел получает одну роль: policy/edge, stateless compute, source of truth, derived store, durable log, worker или external dependency. Ребро получает семантику: synchronous request, replicated write, event, background scan либо control-plane update.

```text
client -> edge -> service -> source of truth
                    |              |
                    |              +-> outbox/log -> workers -> projections
                    +-> cache/read projection
```

Если нельзя сказать, что происходит при timeout на стрелке, связь описана недостаточно.

## Как строить high-level architecture

### 1. Выделить trust и ownership boundaries

Покажите client/Internet, edge, service boundaries, data owners и внешних providers. [[50 Проектирование систем/Границы сервисов|Владелец инварианта]] принимает mutation и один управляет canonical state. Отдельный process не доказывает отдельный bounded context, а общая таблица между сервисами скрывает API.

### 2. Назвать data plane и control plane

Data plane обслуживает пользовательские операции. Control plane меняет routing, schema, flags, membership и policy. Если control plane временно недоступен, data plane по возможности продолжает работать на последней валидной конфигурации. Их общий synchronous dependency превращает редкое административное действие в пользовательский outage.

### 3. Показать source и derived state

Для каждого хранилища подпишите:

- какие данные canonical;
- какая операция их меняет;
- consistency/ack boundary;
- retention;
- можно ли rebuild-ить содержимое;
- что читает система во время lag или rebuild.

Cache, search index и precomputed feed являются derived state, если их можно восстановить из durable source/log. Это определяет backup priority и degraded mode.

### 4. Наложить topology

Сначала один region/zone-neutral logical view, затем failure domains: instances, zones, regions и external providers. Replicas без placement не показывают отказоустойчивость. Multi-region diagram также обязана назвать write authority, traffic steering, replication direction и failback, а не только две одинаковые колонки.

## Write path

Write path проговаривают до точки, где клиент получает необратимое обещание:

1. routing, authentication, authorization, validation и admission;
2. idempotency lookup и concurrency precondition;
3. owner service и transaction/conditional write;
4. WAL/quorum/replication, необходимые для acknowledgment;
5. ответ клиенту;
6. outbox/event, projections, notifications и reconciliation.

Порядок commit и публикации критичен. Если business row committed, а event потерян, downstream расходится; если event опубликован до rollback, он сообщает несуществующий факт. [[40 Распределённые системы/Transactional outbox и Change Data Capture|Transactional outbox]] связывает локальное изменение и намерение публикации.

У write path должны быть outcome при client timeout и duplicate request. Идемпотентность определяется бизнес-ключом, а не надеждой, что сеть не повторит пакет.

## Read path

Read path начинается с требуемой версии данных:

1. cache key и authorization context;
2. cache hit/miss/stale policy;
3. replica/projection/source choice;
4. query/index/partition fan-out;
5. merge, ordering и pagination;
6. response и cache fill.

Read replica снижает нагрузку primary, но lag меняет semantics. Cache ускоряет ответ, но cache key должен включать все representation dimensions, влияющие на результат. Для read-your-writes используют primary, session token, version fence или overlay недавней mutation.

Fan-out увеличивает tail latency: запрос к множеству shards ждёт медленный shard либо возвращает partial result. На схеме надо показать fan-out width, deadline и допустимую деградацию.

## Сквозной пример: публикация статьи

Система хранит article metadata в SQL, body в object storage, публичную projection в cache/CDN и поисковый документ в search index.

Write path:

1. author загружает body по pre-signed URL в object storage;
2. `POST /articles` приходит с object checksum и idempotency key;
3. service проверяет ownership object, создаёт article `DRAFT` и outbox row в одной SQL transaction;
4. publish command условно меняет `DRAFT -> PUBLISHED`, фиксирует immutable version и пишет `ArticlePublished`;
5. после commit API возвращает version; publisher отправляет event;
6. workers строят public cache entry и search document. Их lag не откатывает canonical publish.

Read path:

1. public `GET` проходит CDN по key `(article_id, version, encoding)`;
2. miss попадает в origin cache, затем в article service;
3. service читает published metadata и immutable body version;
4. response получает validator и TTL; private draft никогда не входит в public cache namespace.

Failure path: search worker отстаёт. Прямой URL статьи остаётся доступным, search freshness SLI нарушается отдельно, lag alert поднимает capacity consumer. Rebuild читает event log или canonical rows. Если object upload есть, а metadata transaction не состоялась, lifecycle job удалит orphan после grace period.

Этот trace объясняет четыре хранилища без фразы «используем polyglot persistence»: у каждого есть конкретная роль и recovery.

## Trade-offs

Синхронное обновление всех projections даёт свежий read сразу, но связывает latency и availability с каждым index/cache. Async projection оставляет canonical write коротким, зато требует freshness SLO, repair и read-your-writes strategy.

Одна база упрощает transactions и backup. Разделение body, metadata и search оптимизирует разные access patterns, но добавляет orphan cleanup, version coordination и operational ownership.

Прямой service-to-service call даёт актуальный ответ, но создаёт runtime dependency. Локальная projection повышает автономность read path ценой staleness и schema evolution.

## Типичные ошибки

- **Неверное предположение:** чем больше компонентов, тем зрелее дизайн. **Симптом:** broker и cache не меняют ни одного path. **Причина:** схема рисовалась по checklist. **Исправление:** удалять компонент, пока не сформулированы requirement и failure mode.
- **Неверное предположение:** стрелка означает надёжную доставку. **Симптом:** commit и event расходятся после crash. **Причина:** не определены protocol, ack и retry. **Исправление:** типизировать ребро и пройти crash window.
- **Неверное предположение:** replicas взаимозаменяемы. **Симптом:** failover нарушает read-your-writes или теряет acked write. **Причина:** не указаны lag, quorum и authority. **Исправление:** подписать replication/consistency и failure domains.
- **Неверное предположение:** cache прозрачен. **Симптом:** один пользователь читает private representation другого. **Причина:** cache key не включает authorization/variant. **Исправление:** key contract, private/public boundary и bypass rules.
- **Неверное предположение:** background projection восстановится сама. **Симптом:** после пропущенного event index навсегда расходится. **Причина:** нет source position и reconciliation. **Исправление:** replay/rebuild, version checkpoints и consistency audit.

## Когда применять

Схему используют после требований, SLO и грубой оценки. На интервью достаточно логической topology и одного слоя failure domains. В production design к ней добавляются deployment, network/security, ownership и operational views; одна картинка редко отвечает на все вопросы.
