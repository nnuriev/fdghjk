# Проектирование новостной ленты

## TL;DR

Основной вариант — hybrid fanout. Обычные публикации асинхронно раскладываются в materialized inbox подписчиков (fanout-on-write), а posts авторов с огромной аудиторией подмешиваются при чтении (fanout-on-read). Канонические post и follow graph отделены от производной feed projection: lag или rebuild ленты не должны терять публикацию.

Hybrid здесь — проектный синтез двух подтверждённых механизмов, а не утверждение, что конкретный продукт использует такой же threshold. Materialized inbox ограничивает стоимость обычного read path; Multifeed-подобный aggregator показывает противоположную границу, где request fanout собирает recent actions из distributed leaves. Порог между режимами выводится из измеренной стоимости публикации и чтения для каждого cohort.

Сложность здесь создаёт не CRUD, а skew. Средний автор дешёв, celebrity способен породить десятки миллионов inserts одним событием. Hybrid threshold ограничивает write amplification, но заставляет read path сливать несколько упорядоченных потоков и явно определять freshness.

## Контекст и ментальная модель

Пользователь публикует post, подписывается на авторов и читает персональную reverse-chronological ленту с лёгким ranking. Feed — индекс доставки, а не source of truth. Его можно отстать, перестроить и даже временно упростить; post и follow relation должны остаться корректными.

## Требования

### Functional requirements

- Создать, получить и удалить post.
- Follow/unfollow автора.
- Прочитать персональную ленту cursor-страницами.
- Исключить заблокированных и недоступные posts.
- Подмешать лёгкий ranking без нарушения privacy и удаления.

### Non-functional requirements

- Лента остаётся доступной при отставании fanout workers.
- Один post не появляется дважды на странице или после retry cursor.
- Celebrity post не перегружает весь write path.
- Delete и privacy change доходят до производных inbox в ограниченное время.
- Feed допускает eventual consistency, но канонический post — нет.

### SLO: availability, latency, durability, consistency

Interview targets:

| Путь | SLO |
| --- | --- |
| Read feed | 99,95% за 30 дней; p95 ≤ 200 ms, p99 ≤ 500 ms |
| Create post | 99,9%; p99 ≤ 500 ms до durable commit |
| Freshness | 99% posts обычных авторов появляются у активных followers ≤ 30 s; celebrity posts доступны через read merge ≤ 5 s |
| Durability | подтверждённый post переживает отказ зоны; region-loss RPO ≤ 5 min, RTO ≤ 30 min |
| Consistency | read-your-writes автору; follow/unfollow и delete могут сходиться до 30 s, но authorization проверяется по текущему source of truth |

### Вне scope

- Сложная ML-рекомендация, ads auction и обучение моделей.
- Федерация между независимыми доменами.
- Комментарии и direct messages.

## Traffic и storage estimates

Все величины — interview assumptions.

- 20 млн DAU, 20 открытий ленты в день: `400M reads/day`, в среднем `≈4 630 RPS`, при peak factor 10 — `≈46k RPS`.
- 2 млн posts/day: `≈23 posts/s` в среднем, `≈230/s` на пике.
- Среднее число followers для fanout cohort — 300: `2M × 300 = 600M feed inserts/day`, `≈6 944 inserts/s` в среднем.
- Feed entry 32 bytes (`post_id`, author/time/rank metadata): `600M × 32 B = 19,2 GB/day`; 30 дней hot retention — `576 GB` logical до индексов и реплик.
- Post body и metadata 2 KiB: `≈4 GiB/day`, около `1,4 TiB/year`; media считается отдельно в object storage.
- Если 0,1% posts принадлежат авторам со средним 5 млн followers, fanout-on-write дал бы `2 000 × 5M = 10B` inserts/day только для этой группы. Это и есть причина hybrid fanout.

Среднее число followers недостаточно для capacity. Нужны histogram fanout, p99 feed length, active-followers ratio и отдельная модель celebrity traffic.

## API contract

```http
POST /v1/posts
Idempotency-Key: 72a...
{"text":"...","media_ids":[]}

PUT /v1/users/{author_id}/followers/me
DELETE /v1/users/{author_id}/followers/me

GET /v1/feed?cursor=...&limit=50
DELETE /v1/posts/{post_id}
```

Cursor кодирует snapshot/ranking version и последний `(sort_score, published_at, post_id)`. Он opaque и подписан сервером. Offset pagination не подходит: новые posts сдвигают начало выдачи и создают дубли, что подробно объясняет [[20 Бэкенд/Пагинация offset, cursor и continuation token|заметка о cursor pagination]].

Create возвращает durable `post_id`; он не обещает, что fanout уже завершён. Response содержит `feed_state=pending`, если product contract должен показывать этот нюанс.

## Data model

```text
post(post_id, author_id, body_ref, visibility, published_at,
     deleted_at, version)

follow(follower_id, followee_id, state, updated_at)

feed_entry(user_id, bucket, sort_key, post_id, author_id,
           source_version, inserted_at)

author_stream(author_id, published_at, post_id)

outbox(event_id, aggregate_id, type, payload, created_at, published_at)
```

`feed_entry` — projection. Unique key `(user_id, post_id)` или deterministic upsert устраняет duplicate fanout. Tombstone post остаётся доступным для filtering, пока cleanup не удалит все copies.

## High-level architecture

```text
write API -> post DB + outbox -> event stream -> fanout workers -> feed store
                      |                              |
                      +-> author stream              +-> cache invalidation

read API -> feed cache -> feed store ----+
       |                                  +-> merge/ranker -> hydrate posts
       +-> celebrity author streams ------+
```

Follow service владеет graph. Feed generator читает событие post, определяет fanout cohort и пакетно пишет inbox. Celebrity posts остаются в author stream. Read service получает candidates, фильтрует privacy/block state, hydrates bodies и ранжирует.

## Read path и write path

### Write path

1. Create post проверяет author, сохраняет post и outbox event одной транзакцией.
2. Event попадает в partition по `author_id`, сохраняя порядок posts одного автора.
3. Для обычного автора fanout worker читает followers pages и idempotent batch-upsert делает feed entries.
4. Для celebrity сохраняется только author-stream entry; active users увидят его через merge.
5. Delete пишет tombstone event; read path начинает фильтровать сразу по canonical state, cleanup удаляет copies позже.

### Read path

1. API валидирует cursor и получает precomputed candidates из cache/feed store.
2. Параллельно читает последние posts celebrity-authors пользователя.
3. Merge сортирует candidates по стабильному score tuple; ranking version фиксируется в cursor.
4. Batch hydrate получает canonical posts, отбрасывает deleted/private/blocked.
5. Если после фильтра осталось меньше limit, система дочитывает следующий candidate window с bounded work.

### Сквозная трассировка

Author `A17` с 800 followers публикует `P91`. Транзакция сохраняет post и event `E44`, клиент сразу получает `201`. Worker падает после 500 inserts и повторно читает `E44`; deterministic key `(user_id,P91)` превращает повтор в upsert. Через 8 секунд follower `U7` открывает feed, читает entry P91, hydrate подтверждает visibility и возвращает post. Duplicate отсутствует, хотя delivery события была at-least-once.

## Выбор storage

- Canonical posts и follow mutations удобно начать в SQL: constraints, idempotency и delete state укладываются в транзакции.
- Feed entries требуют append/upsert по user partition и range scan по sort key. Подходит wide-column или partitioned key-value; модель таких workload разобрана в [[30 Данные/Wide-column store|заметке о wide-column store]].
- Media лежит в [[30 Данные/Object и blob storage|object storage]].
- Event stream переносит fanout и delete events; он не заменяет post DB.

## Partitioning и replication

- Feed store partition key: `user_id`; sort key: bucket + descending score/time. Крупный inbox делится на time buckets.
- Follow edges хранятся в двух projections: `following_by_user` для read merge и `followers_by_author` для fanout. Их обновление асинхронно reconcile-ится.
- Event partition по `author_id` сохраняет author order, но hot celebrity partition требует достаточного числа producers/consumers вокруг неё; сам порядок одного автора нельзя распараллелить без sequence rules.
- Feed store и post DB реплицируются между зонами. Cross-region feed projection можно перестроить, а post/follow state входит в DR boundary.

## Caching

Кэшируем первые одну-две страницы active users и hydrated post objects. Feed cache имеет короткий TTL и version token; fanout invalidation — hint, а не единственный механизм корректности. Negative caching удалённых posts уменьшает повторные hydrate misses.

Celebrity streams хорошо кэшируются один раз на регион: один и тот же post нужен многим readers. Полный персональный feed на CDN не подходит из-за authorization, ranking version и огромной cardinality keys.

## Async processing

Fanout, ranking features, delete cleanup и notification triggers идут через [[40 Распределённые системы/Очереди, streams, группы потребителей и DLQ|event stream и consumer groups]]. Delivery at-least-once требует idempotent upsert. Lag измеряется временем самого старого необработанного post, а не только числом records.

Backpressure policy разделяет live fanout и backfill. Историческая перестройка не должна занимать все worker slots; bounded queues и отдельные quotas сохраняют freshness новых posts.

## Failure handling

| Failure | Сигнал | Реакция | Остаточный эффект |
| --- | --- | --- | --- |
| Fanout lag растёт | oldest-event age, freshness SLI | autoscale, остановить backfill, временно расширить read merge | лента дороже читается |
| Feed shard недоступен | shard errors, empty-rate anomaly | replica failover; fallback к author streams последних подписок | выдача неполная, но не пустая |
| Celebrity storm | per-author events, queue skew | переключить автора в fanout-on-read, cancel ещё не начатые batches | часть inbox уже содержит entry, merge дедуплицирует |
| Delete cleanup отстал | tombstone age | canonical filter на read, приоритет delete events, reconciliation | storage временно растёт |
| Ranking service медленный | deadline budget, fallback count | chronological fallback, circuit breaker | relevance хуже, доступность сохраняется |
| Follow projection расходится | dual-view mismatch | source-of-truth check для privacy, repair job | новый follow может появиться позже |
| Region потерян | regional journey failure | route reads к replica, promote post/follow authority по fencing runbook | feed projection прогревается заново |

## Security

- Каждый hydrate повторно применяет current visibility, block и audience policy; materialized feed entry сам по себе не даёт право чтения.
- Follow, post и delete требуют authentication и object-level authorization.
- Cursor подписан и не раскрывает internal shard/ranking state.
- Text/media проходят limits, malware/content policy; HTML не отдаётся без escaping/sanitization.
- Rate limits разделены по reader, author и fanout cost. Один post celebrity стоит дороже одного token.
- Audit log хранит privacy changes и administrative removals.

## Observability

Основные SLI: feed read success/latency, post commit latency, post-to-visible freshness по cohort, fanout lag, candidates-before/after-filter, duplicate rate, cache hit, celebrity merge cost и stale-delete exposure. Trace связывает `post_id`, `event_id`, fanout batch и read request; sampled trace не заменяет exhaustive counters.

Synthetic journey создаёт post тестового автора, ждёт появления у follower, удаляет его и проверяет исчезновение. Отдельный canary проверяет chronological fallback при выключенном ranker.

## Capacity planning

Средняя fanout write rate `≈6,9k/s`. С peak factor 10 и 30% headroom target равен `69k / 0,7 ≈ 99k upserts/s`. Конкретное число shards выводится из load test выбранного store с реальным batch size и replication factor.

Feed hot retention `576 GB logical` при replication factor 3 становится `≈1,7 TB` до overhead. Если хранить 90 дней вместо 30, стоимость растёт линейно, но пользователь редко дочитывает этот хвост; cold recomputation из author streams часто дешевле.

## Cost trade-offs

Fanout-on-write покупает дешёвый read ценой write amplification и feed storage. Fanout-on-read экономит writes, но каждый refresh читает и сливает много streams. Hybrid threshold должен зависеть от active followers и read frequency, а не от красивого круглого follower count.

Ranking service и feature store добавляют compute и operational surface. Для первой версии reverse chronology с небольшой эвристикой дешевле и лучше проверяет data paths; ML вводится после появления измеримой relevance цели.

## Migration и rollout plan

1. Начать с fanout-on-read для ограниченного follow graph и canonical posts.
2. Добавить outbox и precomputed inbox в shadow; сравнивать top-N с исходным read merge.
3. Включить fanout-on-write для малого normal cohort, сохранив fallback.
4. Ввести celebrity threshold как dynamic config и hysteresis, чтобы автор не прыгал между режимами на каждом изменении followers.
5. Переключить read traffic по процентам; cursor содержит algorithm version, старые cursors обслуживаются до TTL.
6. Для нового ranking generation писать обе projections, измерить divergence и переключить alias/version без остановки.

Rollback возвращает read к author streams; precomputed entries остаются безопасной производной и удаляются позже.

## Bottlenecks

- Fanout write amplification и skew по celebrity.
- Hydration N+1 вместо batch get.
- Filtering после получения слишком малого candidate window.
- Один hot `user_id` при ботах или shared account.
- Ranking dependency в synchronous critical path.
- Rebuild/backfill, который вытесняет live events.
- Cross-region follow/post lag при failover.

## Trade-offs и альтернативы

| Подход | Выигрыш | Цена | Подходящие условия |
| --- | --- | --- | --- |
| Fanout-on-write | быстрый предсказуемый read | много inserts и storage | много reads, умеренный fanout |
| Fanout-on-read | дешёвая публикация | merge cost и tail latency | celebrity и редко читаемые followers |
| Hybrid | ограничивает оба хвоста | сложная дедупликация и threshold | реальный social graph со skew |
| Chronological | объяснимость и простота | ниже relevance | старт и fallback |
| Ranked | engagement/relevance | features, versioning, debugging | есть метрика качества и данные |

## Типичные ошибки

- **Неверное предположение:** среднее число followers задаёт capacity. **Симптом:** один celebrity post заполняет очередь. **Причина:** power-law skew. **Исправление:** histogram, hybrid fanout и per-author admission.
- **Неверное предположение:** feed entry можно считать разрешением на read. **Симптом:** private или удалённый post остаётся видимым. **Причина:** projection устарела. **Исправление:** canonical authorization/filter и bounded cleanup SLO.
- **Неверное предположение:** retry fanout безопасен сам по себе. **Симптом:** duplicate posts. **Причина:** at-least-once delivery без deterministic key. **Исправление:** idempotent upsert `(user_id, post_id)`.

## Когда применять

Архитектура подходит для social timeline, activity stream и подписочной ленты с сильным skew между авторами. Для небольшого каталога без follow graph достаточно query по общему индексу; для строгого аудита событий feed projection не заменяет immutable event log.
