# Проектирование чата

## TL;DR

Chat разделяется на connection plane и durable message plane. WebSocket gateways держат миллионы соединений, но не владеют историей. Message service проверяет membership, дедуплицирует `client_message_id`, назначает sequence внутри conversation partition и подтверждает send только после durable commit. Fanout материализует отдельный durable device inbox; именно он даёт монотонный resume cursor между разными conversations. Delivery online-устройствам, push, read receipts и presence выполняются асинхронно.

Глобальный порядок сообщений не нужен и слишком дорог. Нужен стабильный порядок внутри conversation, а для multi-region — один write authority на conversation с fencing при failover. End-to-end encryption (E2EE) меняет границу доверия: сервер хранит ciphertext, но теряет server-side search, moderation body и простой multi-device key recovery.

## Контекст и ментальная модель

Поддерживаем direct и group conversations, online delivery, offline history, attachments, read receipts и push. Клиент может потерять ответ после send и повторить команду. Поэтому transport connection и identity операции нельзя смешивать: reconnect не создаёт новое сообщение, а retry с тем же `client_message_id` возвращает прежний result.

Message log — каноническая история. WebSocket delivery и durable inbox устройства — projections над ней. Inbox хранит ограниченный хвост lightweight events, поэтому старый cursor не обещает бесконечный replay: после retention клиент получает `resync_required` и дочитывает каноническую историю conversations.

## Требования

### Functional requirements

- Создать conversation, добавить и удалить участника.
- Отправить text/attachment message и получить server acknowledgment.
- Получать новые messages online, догонять историю после reconnect.
- Поддержать edit/redaction как новые events, не переписывая прошлое молча.
- Хранить per-device cursor, delivered/read receipts и push для offline device.

### Non-functional requirements

- Retry после неизвестного исхода не создаёт duplicate logical message.
- Порядок стабилен внутри conversation; между conversations порядок не задаётся.
- Gateway restart не теряет durable message.
- Hot group не перегружает все partitions.
- Backpressure ограничивает slow consumers и не удерживает неограниченную память gateway.

### SLO: availability, latency, durability, consistency

Interview targets:

| Путь | SLO |
| --- | --- |
| Connect/reconnect | 99,95%; p99 ≤ 2 s без учёта client network |
| Send acknowledgment | 99,95%; p95 ≤ 250 ms, p99 ≤ 600 ms |
| Online delivery | 99% messages доставлены подключённому получателю ≤ 1 s после commit |
| History read | 99,95%; p99 ≤ 500 ms для страницы 50 messages |
| Durability | после `accepted` message переживает отказ зоны; region-loss RPO ≤ 1 min, RTO ≤ 15 min |
| Consistency | per-conversation total order; read receipts и presence eventual, target freshness ≤ 5 s и ≤ 30 s соответственно |

`accepted` означает durable server commit, а не прочтение получателем. Push provider и выключенное устройство не входят в end-to-end delivery SLO.

### Вне scope

- Голосовые и видеозвонки.
- Федерация независимых chat servers.
- Полная криптографическая спецификация E2EE и key transparency.
- Arbitrary server-side full-text search при E2EE.

## Traffic и storage estimates

Все числа — interview assumptions.

- 10 млн DAU, 50 messages/user/day: `500M messages/day`.
- Средний send rate: `500M / 86 400 ≈ 5 787 messages/s`; peak factor 10 — `≈58k/s`.
- Message envelope вместе с индексируемой metadata в среднем 800 bytes: `500M × 800 B = 400 GB/day`, около `146 TB/year` logical до реплик.
- При replication factor 3 message storage — `≈438 TB/year` до compression и compaction.
- 2 млн concurrent connections. Если одна gateway replica проходит load test на 40k соединений при нужном p99, базово нужно `2M / 40k = 50`; с 30% headroom `50 / 0,7 ≈ 72 replicas`.
- Если в среднем 1,5 target recipient-device на message, fanout создаёт `750M device events/day`, а peak delivery — `58k × 1,5 ≈ 87k events/s`; крупные groups моделируются отдельно.
- Lightweight inbox entry 96 bytes даёт `750M × 96 B = 72 GB/day`; 30 дней resume retention — `≈2,16 TB logical`, до реплик и storage overhead. Это planning assumption, размер entry и retention надо измерить.

Attachments не входят в 800 bytes и считаются в object storage/egress. Presence traffic может превосходить message traffic, поэтому heartbeat interval и online audience нельзя оставить «мелочью».

## API contract

История и mutations доступны по HTTP; live delivery — WebSocket. Protocol message имеет version и type.

```http
PUT /v1/conversations/{conversation_id}/messages/{client_message_id}
{"type":"text","body":"hello","reply_to":null}

GET /v1/conversations/{conversation_id}/messages?before=...&limit=50
POST /v1/conversations/{conversation_id}/receipts
{"read_through_seq":1842}
```

`PUT` естественно идемпотентен по client-chosen ID. Response содержит `message_id`, `conversation_seq`, `committed_at`. Повтор с другим payload получает conflict.

WebSocket frames:

```text
client -> {type: "resume", device_id, cursor}
server -> {type: "event", delivery_cursor, source_event_id,
           conversation_id, seq, message_id, payload}
client -> {type: "ack", through_cursor}
```

Resume cursor opaque, monotonic только внутри одного `(user_id, device_id)` stream и не заменяет conversation sequence. ACK кумулятивен: client отправляет наибольший непрерывно применённый `through_cursor`, server монотонно продвигает checkpoint, а повтор ACK безопасен. Пропуск неприменённого cursor нельзя подтвердить более новым числом. Если cursor старше retention, server отвечает `resync_required` с conversation watermarks; молча начинать с текущего хвоста нельзя.

## Data model

```text
conversation(conversation_id, home_region, current_epoch, created_at)

membership(conversation_id, user_id, role, joined_seq, left_seq, version)

message(conversation_id, seq, message_id, sender_id, client_message_id,
        type, payload_ref_or_ciphertext, created_at, redacts_seq)

device_inbox(user_id, device_id, delivery_cursor, source_event_id,
             conversation_id, conversation_seq, event_type,
             payload_ref, created_at, expires_at)
device_checkpoint(user_id, device_id, acknowledged_cursor, updated_at)
receipt(conversation_id, user_id, read_through_seq, updated_at)

outbox(event_id, conversation_id, seq, type, payload, published_at)
```

Unique `(conversation_id, sender_id, client_message_id)` обеспечивает request deduplication. Device inbox имеет unique `(user_id, device_id, source_event_id)` и monotonic `delivery_cursor`, который назначает владелец device partition; composite key не позволяет retry fanout создать второй logical event. Membership хранит sequence boundaries: бывший участник не должен читать messages после `left_seq`, а новый — раньше `joined_seq`, если policy такова.

## High-level architecture

```text
clients <-> regional WebSocket gateways <-> session/delivery router
                         |
                         v
                 message service -> message store + outbox
                         |                 |
                         |                 +-> event stream -> fanout workers -> device inbox
                         |                                      |-> online gateways
                         +-> membership store                   |-> push service
                                                                +-> receipts/index
```

Gateway аутентифицирует connection и применяет transport limits. Message service остаётся write authority. Router хранит soft mapping `user/device -> gateway`; потеря mapping лечится reconnect, а не data recovery.

## Read path и write path

### Write path

1. Gateway проверяет token, frame size и rate limit, затем передаёт command с deadline.
2. Message service на conversation owner проверяет membership и idempotency key.
3. Owner в своей partition назначает следующий `seq`, сохраняет message и outbox event в одной commit boundary.
4. Только после commit sender получает acknowledgment.
5. Fanout consumer для каждого target device идемпотентно добавляет lightweight row в durable inbox по `(user_id, device_id, source_event_id)` и получает следующий `delivery_cursor` внутри device partition.
6. Только после inbox append event отправляется online gateway; для offline device инициируется push. Duplicate transport delivery допустим, client дедуплицирует composite `source_event_id`/`message_id`.

### Read path

1. После reconnect client отправляет последний device cursor.
2. Delivery service делает range scan device inbox после cursor. Если cursor ещё в retention, он возвращает пропущенные lightweight events в порядке device stream.
3. Если cursor неизвестен, старше retention или в stream обнаружен gap, server возвращает `resync_required` и последние conversation watermarks. Client синхронизирует каждую затронутую conversation по её `seq`, затем принимает новый device checkpoint.
4. History API делает range scan `(conversation_id, seq)` с cursor, применяет membership interval и redactions.
5. Attachments скачиваются по отдельному signed URL из object storage.

### Сквозная трассировка

Client отправляет `client_message_id=C55` в conversation `R9`. Service назначает `seq=1842`, commit проходит, но acknowledgment теряется. Client повторяет PUT. Unique key находит прежний row и возвращает `message_id=M77, seq=1842`; второго message нет. Recipient gateway дважды получает outbox event после consumer retry, но client хранит M77 и показывает его один раз. История остаётся упорядоченной по seq.

## Выбор storage

Message access — append и range scan по conversation. Partitioned wide-column/KV store хорошо соответствует этой форме; [[30 Данные/Wide-column store|wide-column model]] даёт нужный sort внутри partition. Device inbox использует тот же паттерн, но с partition `(user_id, device_id)` и sort key `delivery_cursor`; это отдельная projection с TTL, не замена message history. Conversation membership и idempotency можно держать в SQL, пока scale и transaction boundary это позволяют, либо collocate с conversation owner.

Attachments лежат в [[30 Данные/Object и blob storage|object storage]]. Event stream развязывает durable commit и online delivery. Presence/session routing подходит ephemeral distributed KV с TTL, но не message history.

## Partitioning и replication

- Основной shard key — hash `conversation_id`; все writes одной conversation идут к одному logical owner.
- `seq` назначается owner/leader внутри epoch. После failover новый owner получает больший epoch; storage отвергает запись со старым fencing token по модели [[40 Распределённые системы/Leases, distributed locks и fencing tokens|leases и fencing]].
- Device inbox partition-ится по `(user_id, device_id)`. Inbox owner атомарно назначает следующий `delivery_cursor` и вставляет unique `source_event_id`; порядок между conversations произволен, но стабилен для resume одного device.
- Message partitions синхронно реплицируются между зонами. Cross-region replica асинхронна, если latency важнее zero-RPO при потере региона.
- Огромная group conversation становится hot partition. Варианты: отдельный high-capacity shard, batching, substreams с causal merge или ослабление total order; механически hash-раскидать messages нельзя, не изменив guarantee.

## Caching

Кэшируем membership snapshots с version, recent messages и routing mappings. Negative membership cache короткий: invite/join должен быстро вступить в силу. Authorization на send сверяет version или source при stale risk.

История хорошо читается из recent-message cache, но cursor pages не кэшируются общим ключом без membership scope. Presence TTL не означает точный offline: network partition и suspended mobile app делают статус приблизительным.

## Async processing

Device-inbox append, online delivery, push, receipts aggregation, search projection и media processing выполняются после message commit. Гарантии очереди разбираются в [[40 Распределённые системы/At-most-once, at-least-once и effectively-once processing|заметке о processing semantics]]: at-least-once плюс dedup лучше ложного обещания exactly-once до UI. Inbox retention по базовому assumption равен 30 дням; TTL worker удаляет entry только после expiry, а checkpoint сам по себе не служит доказательством, что всем устройствам больше не нужен replay.

Slow connection получает bounded send buffer. При заполнении gateway прекращает live fanout, закрывает connection с resumable cursor; message остаётся в history. Бесконечный buffer превратил бы одного клиента в memory leak.

## Failure handling

| Failure | Сигнал | Реакция | Остаточный эффект |
| --- | --- | --- | --- |
| Gateway падает | connection drop, gateway heartbeat | client reconnect + range scan durable device inbox; router TTL очищает mapping | краткая задержка, history и resume stream целы |
| Message commit прошёл, ack потерян | client retry | lookup по `client_message_id`, вернуть тот же seq | duplicate transport, один logical message |
| Conversation owner split-brain | epoch/fencing rejects | старый owner прекращает writes, elect/promote новый | send latency растёт, order сохраняется |
| Fanout lag | oldest-event age, online-delivery SLI | scale consumers, отключить noncritical receipts/search, clients sync history | live delivery задержана |
| Slow consumer | send-buffer bytes, ack lag | stop live fanout, disconnect with durable resume cursor | client догоняет device inbox |
| Resume cursor вышел из retention | cursor lookup miss, inbox oldest cursor | вернуть `resync_required`, range-sync conversations по `seq`, затем выдать новый checkpoint | дороже reconnect, но события не считаются доставленными молча |
| Inbox append отстал | inbox oldest-event age, online-delivery SLI | scale inbox writers, ограничить noncritical projections; gateway не обходит durable inbox | live delivery задержана, canonical messages целы |
| Hot group | per-partition QPS/latency | dedicated shard, batching, admission; product limit при необходимости | одна room деградирует изолированно |
| Region потерян | regional connect/send probes | route reconnects, promote home authority с новым epoch | writes ждут safe promotion; RPO по replica lag |

## Security

- Gateway проверяет access token и device identity; message service повторно проверяет membership.
- TLS защищает transport. E2EE требует отдельного threat model, device keys, rotation, backup и abuse metadata policy; «зашифровать body» недостаточно.
- Attachments используют random immutable keys, checksum, malware scan и short-lived URLs.
- Per-user, per-device и per-conversation rate limits учитывают fanout cost.
- Logs не содержат message body, tokens и encryption keys; administrative access аудируется.
- Block/redaction не гарантируют стирание уже доставленной копии на чужом устройстве; product contract должен говорить это прямо.

## Observability

Измеряем connect success, concurrent connections, reconnect rate, send commit latency, ack-loss retries, duplicate ratio, per-partition sequence gap, inbox append lag, cursor gap/resync rate, oldest retained cursor, online delivery latency, slow-consumer disconnects, push handoff и history catch-up. Cardinality `conversation_id` не кладём в каждую metric label; для hot rooms используем sampled logs/top-K.

Trace проходит gateway → message commit → outbox → device inbox → delivery, а `message_id`, `source_event_id` и device cursor дают точную корреляцию. Synthetic users отправляют message, теряют connection, повторяют command и проверяют один seq после resume; отдельный canary проверяет `resync_required` на cursor старше retention.

## Capacity planning

Peak send `≈58k/s`; с 30% headroom target `58k / 0,7 ≈ 83k commits/s`. Shard count выводится из нагрузочного теста с реальным group-size distribution и replication. Нельзя делить только на среднюю throughput replica: hot partition может исчерпать один owner раньше cluster.

Для 2 млн connections и измеренного limit 40k/gateway расчёт дал 72 replicas с headroom. Дополнительно резервируем failure of one zone: оставшиеся zones должны принять reconnect storm, а handshake/auth dependencies — выдержать burst.

При assumption `750M` device events/day и 96 B на entry inbox растёт на `72 GB/day`; 30 дней дают `≈2,16 TB logical`, а replication factor 3 — `≈6,48 TB` до индексов и compaction. Inbox write capacity при 30% headroom планируется на `87k / 0,7 ≈124k events/s` и отдельно на hot group fanout.

## Cost trade-offs

Основные расходы: always-on gateways, replicated message storage, attachments/egress, push и cross-region copies. Persistent connection уменьшает polling overhead и latency, но держит memory/file descriptors даже без messages.

Хранить вечную историю дорого. Retention, export и legal hold задаются по conversation class. Compression снижает bytes, но не заменяет deletion index и cryptographic erasure policy.

## Migration и rollout plan

1. Начать с HTTP send/history и одной durable region; WebSocket только доставляет committed messages.
2. Ввести `client_message_id` и unique constraint до retries клиента.
3. Добавить outbox и durable device inbox в shadow, сравнить composite event IDs и resume gaps со старым path.
4. Переключать conversations на partition owners по consistent hash ring; directory version входит в routing response.
5. Для нового message schema readers сначала принимают обе версии, затем writers начинают новую; backfill не переписывает ordering key.
6. Cross-region replica проходит restore/failover drill. Traffic переводится только после fencing test, reconnect capacity test и проверенного failback.

## Bottlenecks

- Concurrent connections и reconnect storm.
- Hot conversation, требующая единого order.
- Membership lookup на каждый message.
- Fanout multiplication по devices и group members.
- Slow consumers и неограниченные gateway buffers.
- Cross-region sequence authority.
- Attachment egress, которое не видно в message QPS.

## Trade-offs и альтернативы

| Выбор | Выигрыш | Цена | Когда выбирать |
| --- | --- | --- | --- |
| WebSocket | low-latency bidirectional delivery | connection state и reconnect | активный chat |
| Long polling | простой infrastructure path | больше requests и latency | малый scale/fallback |
| Per-conversation order | понятная история | hot shard | почти всегда достаточно |
| Global order | единая шкала | coordination latency и availability | редко имеет бизнес-смысл |
| Server-readable body | search/moderation проще | server trust и breach impact | если E2EE не требование |
| E2EE | сервер не читает content | key lifecycle, search/moderation сложнее | высокий privacy requirement |

## Типичные ошибки

- **Неверное предположение:** WebSocket send равен durable delivery. **Симптом:** message исчез после gateway restart. **Причина:** ack выдан до storage commit. **Исправление:** подтверждать после durable commit, live delivery строить поверх log.
- **Неверное предположение:** timestamp задаёт порядок. **Симптом:** messages переставляются при clock skew. **Причина:** часы клиентов и серверов не дают total order. **Исправление:** per-conversation sequence у authority.
- **Неверное предположение:** lease останавливает старого owner. **Симптом:** два sequencer пишут одновременно. **Причина:** paused process ожил после expiry. **Исправление:** epoch/fencing validation в storage.

## Когда применять

Дизайн подходит для direct/group messaging с offline history и near-real-time delivery. Для ephemeral presence без истории durable log избыточен; для collaborative document нужен CRDT/OT и другая consistency model.
