# Проектирование поиска и автодополнения

## TL;DR

Канонические documents живут в transactional/object storage, а search index остаётся versioned projection. CDC/outbox доставляет изменения в indexing pipeline; query service выполняет lexical search, filters и ranking по index generation. Autocomplete получает отдельный prefix-oriented индекс и более строгий latency budget. Один тяжёлый full-text query не должен вытеснять короткие suggestions. Delete и permission revoke дополнительно проходят через обязательный deny overlay: ждать обычного refresh ради безопасности нельзя.

Главные компромиссы: freshness против indexing throughput, relevance против latency и index size против возможностей query. Blue-green index generations позволяют менять analyzer/mapping без остановки и откатываться переключением alias. Stable pagination опирается на point-in-time snapshot (PIT), а не только на sort tuple. Dual write из application прямо в два индекса не даёт атомарности и плохо восстанавливает пропущенные updates.

## Контекст и ментальная модель

Ищем по каталогу из 100 млн documents. Пользователь вводит prefix и получает suggestions, затем отправляет full query с filters и cursor pagination. Documents меняются независимо от index.

Ментальная модель: source of truth отвечает «что существует и кому доступно», index отвечает «какие candidates быстро подходят запросу». Index можно отставить или перестроить; authorization и окончательная версия документа не должны зависеть только от него.

## Требования

### Functional requirements

- Полнотекстовый поиск по нескольким полям с filters и stable pagination.
- Autocomplete по prefix с typo/locale policy и популярностью.
- Add/update/delete document через indexing stream.
- Reindex новой schema/analyzer generation без downtime.
- Учитывать tenant/visibility и не показывать forbidden suggestions.

### Non-functional requirements

- Query service деградирует предсказуемо при shard/indexer failures.
- Search result может быть слегка stale, но delete/permission revoke имеет отдельный короткий SLO.
- Один expensive query ограничен по CPU, clauses, timeout и result window.
- Index rebuild не вытесняет live updates.
- Ranking version фиксируется в cursor/experiments.

### SLO: availability, latency, durability, consistency

Interview targets:

| Путь | SLO |
| --- | --- |
| Autocomplete | 99,95%; p95 ≤ 50 ms, p99 ≤ 100 ms server-side |
| Full search | 99,95%; p95 ≤ 150 ms, p99 ≤ 350 ms для bounded query |
| Index freshness | 99% create/update видны ≤ 60 s; delete и permission revoke ≤ 10 s |
| Durability | canonical document следует SLO source system; index generation восстанавливается из snapshot + change log |
| Consistency | eventual между source и index; read-your-writes обеспечивается direct lookup/overlay, а не обещанием мгновенного refresh |

SLO revoke измеряет момент, когда document перестал появляться и в full search, и в autocomplete во всех serving regions. Команда revoke считается завершённой только после durable записи monotonic access version и deny overlay; обычное удаление из index затем догоняет асинхронно. Если overlay для private scope недоступен или старше своего 10-секундного freshness budget, query закрывается с ошибкой, а не возвращает потенциально запрещённый result.

### Вне scope

- Обучение large ranking/embedding model.
- Web crawling и борьба со spam на уровне интернета.
- Реализация search engine с нуля.
- Глобальный semantic/vector search; его можно добавить как candidate source позже.

Version-dependent примеры API ниже относятся к Elasticsearch 8.19. Архитектурный контракт не зависит от конкретного engine, но mapping и PIT API при смене версии нужно проверить заново.

## Traffic и storage estimates

Все числа — interview assumptions.

- 20 млн DAU, 10 full searches/day: `200M/day`, в среднем `≈2 315 QPS`, peak factor 10 — `≈23k QPS`.
- 50 autocomplete requests/day: `1B/day`, `≈11 574 QPS` average, `≈116k QPS` peak.
- 100 млн documents, средний canonical searchable text 2 KiB: `≈191 GiB` raw.
- Index expansion factor 2,5 (terms, postings, doc values) принят как planning assumption: `≈477 GiB logical`; одна replica — `≈954 GiB` до segment overhead.
- 1 млн updates/day: `≈11,6/s` average, `≈116/s` peak; bulk reindex создаёт совершенно другой workload.
- Autocomplete словарь 20 млн normalized entries по 100 bytes metadata: `≈1,9 GiB` raw; prefixes/ngrams увеличивают размер, это измеряется на реальном analyzer.

Factor 2,5 не продуктовая гарантия. Его заменяют измерением sample corpus до заказа capacity.

## API contract

```http
GET /v1/search?q=wireless+headphones&category=audio&limit=20&cursor=...
GET /v1/suggest?prefix=wirel&locale=en-US&limit=10
```

Search response возвращает `results`, `next_cursor`, `index_generation`, `ranking_version` и `took_ms`. Первый запрос открывает server-owned PIT с коротким TTL. Подписанный cursor ссылается на этот snapshot и включает hash исходного query/access scope, ranking version и `search_after` tuple, например `(score_bucket, updated_at, document_id)`. Native PIT ID наружу не выдаётся и не превращается в бессрочный public contract.

PIT стабилизирует index candidates между страницами. Security overlay проверяется заново на каждой странице, поэтому revoke может удалить уже недоступный candidate и сократить страницу. Возвращать запрещённый result ради формальной стабильности pagination нельзя.

Indexing interface — event, не публичный search API:

```text
DocumentChanged {
  event_id, document_id, source_version, operation,
  changed_at, tenant_id
}
```

Indexer читает актуальную canonical representation либо self-contained event. Версия не позволяет старому retry перезаписать новый document.

## Data model

```text
canonical_document(document_id, tenant_id, source_version, visibility,
                   title, body_ref, attributes, updated_at, deleted_at)

index_document(document_id, tenant_id, source_version,
               analyzed_fields, filter_fields, rank_features, tombstone)

access_overlay(tenant_id, document_id, access_version, denied,
               committed_at, cleared_at)

index_generation(generation_id, schema_version, analyzer_version,
                 state, created_at, high_watermark)

indexing_checkpoint(consumer, partition, offset, generation_id)
```

Autocomplete document отделяется от full index: `normalized_text`, locale, prefix/shingle fields, popularity bucket и visibility scope. Source version сравнивается перед upsert/delete.

## High-level architecture

```text
source DB -> outbox/CDC -> event stream -> normalizer -> bulk indexer
                                            |              |
                                            |              +-> search generations
                                            +-> autocomplete index

client -> API gateway -> query service -> query/result cache -> search cluster
                                      |                       |
                                      +-> policy/filter ------+
                                      +-> deny/access overlay
                                      +-> source authorization
```

Control plane создаёт mapping, generation, snapshot и alias switch. Data plane выполняет queries. Autocomplete получает отдельные pools/quotas, чтобы full search не съел latency budget keystrokes.

## Read path и write path

### Write path

1. Source service commit-ит document и outbox event. Delete или permission revoke вместе с canonical state получает новую monotonic `access_version`.
2. Revoke publisher до успешного ответа управляющей команде durable записывает deny overlay во всех serving regions. Повтор безопасен по `(tenant_id, document_id, access_version)`.
3. Indexer читает event, fetch-ит canonical version, нормализует locale/analyzers и формирует deterministic document.
4. Bulk request делает conditional upsert только если `source_version` не старее indexed.
5. Checkpoint продвигается после подтверждения index; retry повторяет тот же version безопасно.
6. Delete индексируется как tombstone/удаление с высоким priority и инвалидирует query cache. Deny overlay нельзя убрать только потому, что активные generations прошли нужную `access_version`: старый PIT всё ещё видит доревоковый snapshot. Его low-watermark GC разрешён лишь после трёх условий: все generations применили deny/delete, все PIT/cursors, открытые до этой `access_version`, закрылись или истекли, а старые candidate-cache entries purged либо вышли из TTL; сверху остаётся safety grace. Regrant публикуется как новая monotonic `access_version`, а не как преждевременное удаление deny record.

### Read path

1. API нормализует query, применяет limits и access scope. Первый full-search request открывает PIT; следующий использует тот же PIT и `search_after`.
2. Autocomplete идёт в public-only либо tenant/ACL-scoped prefix index. Каждый candidate проходит тот же deny overlay; private titles никогда не попадают в общий suggestion index.
3. Full search fanout-ится только по нужным shards/tenant routing и собирает candidates/ranking из PIT.
4. Все candidates проходят обязательный deny/access overlay. Для policy, которую overlay не может выразить, query service batch-hydrate-ит source authorization; при недоступности проверки private scope закрывается fail-closed.
5. Cursor продолжает тот же PIT/generation/ranking version. После expiry клиент начинает новый snapshot; сервер закрывает PIT по завершении или TTL.

### Сквозная трассировка

Document `D7` обновлён с version 41 до 42. Event 42 индексируется, затем после rebalance consumer повторно получает старый event 41. Conditional script сравнивает version и отклоняет downgrade. Первый search открывает PIT P3 и возвращает version 42; следующая страница использует P3 и `search_after`, поэтому refresh не переставляет results. Через минуту запускается generation G2: snapshot строит D7=42, change log догоняется до high-watermark, alias переключается с G1 на G2. PIT P3 обслуживается на G1 до TTL, новый query идёт в G2. Если D7 отзывают до следующей страницы, live deny overlay удалит его и из старого PIT.

## Выбор storage

[[30 Данные/Search index|Search index]] подходит для inverted postings, analyzers и ranking, но не заменяет canonical DB. SQL/document/object storage выбирается по source domain; index хранит projection.

Autocomplete варианты: engine `search_as_you_type`/edge n-grams, completion trie/FST или заранее вычисленный popular-prefix table. Trie/FST экономен для mostly-static dictionary; general search index проще при frequent updates и filters.

Event stream хранит ordered changes и checkpoints. Object storage хранит snapshots для bootstrap/recovery.

## Partitioning и replication

- Index shard routing по `tenant_id` или hash `document_id`. Tenant routing уменьшает scatter, но крупный tenant создаёт hot shard.
- Число primary shards выбирают по corpus, query fanout, recovery time и future growth; слишком много мелких shards расходует heap/metadata.
- Replicas обслуживают reads и отказ shard. Они не заменяют snapshot: operator delete/mapping corruption реплицируются тоже.
- Indexing stream partition key `document_id` сохраняет order changes одного document.
- Autocomplete index можно реплицировать в каждом read region; updates асинхронны, generation switch региональный и наблюдаемый.

## Caching

Кэшируем normalized popular queries и autocomplete prefixes коротким TTL. Cache key включает tenant/locale/filter/access cohort, generation и ranking version. Персональный ACL в shared key запрещён. Каждый cached candidate повторно проходит deny overlay. Permission decision кэшируется не дольше 10 s и несёт monotonic access version; push invalidation сокращает окно, но не заменяет TTL.

Negative result cache особенно осторожен: новый document должен появиться в freshness SLO. Query cache не лечит expensive unbounded queries; сначала нужны admission и limits.

## Async processing

Indexing, embedding/rank-feature computation, spell dictionary и popularity aggregation идут асинхронно. Live changes имеют отдельную очередь от backfill. Backpressure останавливает bulk reindex до того, как freshness live stream выйдет за budget.

At-least-once processing безопасен благодаря `(document_id, source_version)`. DLQ хранит schema/normalization failure с owner; после исправления events replay-ятся, а не пропускаются вручную.

## Failure handling

| Failure | Сигнал | Реакция | Остаточный эффект |
| --- | --- | --- | --- |
| Indexing lag | high-watermark age | pause backfill, scale bulk workers, prioritize delete | results stale в пределах/за SLO |
| Search shard unavailable | shard failure rate | replica promotion; partial results только с explicit flag или fail request | availability/relevance trade-off виден клиенту |
| Bad mapping/analyzer rollout | reject/error spike, relevance canary | stop G2, оставить alias G1, repair/rebuild | новые features задержаны |
| Query explosion | timeout/CPU/clauses | validate limits, timeout, circuit/load shedding | expensive query rejected |
| Cache hot prefix | per-key QPS | request coalescing, local prefix cache, precompute | небольшое stale window |
| Delete/revoke event до index потерян | overlay/index watermark mismatch, reconciliation | deny overlay продолжает блокировать выдачу; high-priority replay чинит index | storage/index stale, но exposure не выходит за revoke SLO |
| Deny overlay недоступен или stale | overlay age/error rate | fail-closed для private search/autocomplete; public-only path продолжает работу | private availability ниже, privacy сохраняется |
| Region/index lost | synthetic failures | route к replica region или restore snapshot + replay log | freshness падает до catch-up |

## Security

- Query parser принимает typed DSL/allowlist, а не произвольный native engine JSON.
- Tenant/ACL filter добавляет server, клиент не может его удалить. Затем каждый result проходит deny overlay; сложная policy дополнительно проверяется по source.
- Autocomplete использует public-only или ACL-scoped index, поверх которого обязателен deny overlay. Private titles, banned users и low-frequency PII не должны просачиваться через prefix probing.
- Rate limit учитывает query cost, result window и wildcard/fuzzy/vector expense.
- Search logs редактируют PII и задают retention; raw queries часто содержат секреты.
- Admin mapping/reindex/alias APIs отделены от query credentials.

## Observability

Нужны query/suggest success и latency, timeout/load-shed rate, shard fanout, cache hit, result count, zero-result rate, indexing lag по partition, reject/DLQ, generation divergence, snapshot age и delete freshness. Отдельно измеряем overlay propagation p99, stale-overlay rejects и время от revoke commit до исчезновения из full search и autocomplete во всех регионах. Relevance измеряется offline judgments и online product metric отдельно от latency SLO.

Trace связывает query normalization, cache, shard requests и source hydration. Synthetic corpus проверяет exact, prefix, locale, ACL и delete. Golden queries сравнивают G1/G2 до alias switch.

## Capacity planning

Autocomplete peak `≈116k QPS`; с 30% headroom target `116k / 0,7 ≈ 166k QPS`. Full search target: `23k / 0,7 ≈ 33k QPS`. Их нельзя складывать и делить на общую replica capacity: cost одного full query выше и зависит от shard fanout.

Sample corpus даёт measured bytes/document, indexing CPU и query p99. При `≈477 GiB` primary + одна replica получаем `≈954 GiB`; добавляем disk watermarks, merge и rebuild headroom, обычно отдельный коэффициент, подтверждённый load/recovery test.

PIT удерживает старые segments, file descriptors и heap metadata. Capacity model ограничивает TTL, число одновременно открытых PIT на tenant и принудительно закрывает завершённые cursors; стабильная pagination имеет измеримую storage/heap цену.

## Cost trade-offs

Index replicas, RAM/heap, SSD и merge I/O доминируют. Больше replicas снижают read latency и улучшают availability, но увеличивают indexing fanout и storage. Prefix/ngram fields ускоряют autocomplete ценой index size.

Hydrating каждый result из source даёт строгую authorization, но добавляет latency и load. Coarse ACL в index отсекает основную массу candidates, deny overlay закрывает revoke/delete, а source check остаётся для policy, которую эти две проекции не выражают.

## Migration и rollout plan

1. Ввести outbox/CDC и построить index G1 рядом с прежним database search.
2. Shadow queries сравнивают IDs, latency и authorization; расхождения классифицируются.
3. Canary read traffic идёт в G1 с fallback на старый path.
4. Для schema/analyzer change создаётся G2, выполняются snapshot, live catch-up и golden-query check.
5. Alias переключается атомарно; старые PIT/cursors обслуживаются G1 до TTL, затем G1 удаляется после закрытия PIT и rollback window.
6. Multi-region replicas вводятся по одному региону; traffic включается после freshness/recovery check.

## Bottlenecks

- Shard fanout и coordination tail latency.
- Merge/refresh I/O во время bulk indexing.
- Prefix/ngram index expansion.
- Hot tenant или hot query.
- Source hydration N+1.
- Backfill, вытесняющий live deletes.
- Too many shards и долгий recovery.

## Trade-offs и альтернативы

| Решение | Выигрыш | Цена | Когда выбирать |
| --- | --- | --- | --- |
| DB full-text | меньше infrastructure | ограниченное scale/relevance | небольшой corpus, простые queries |
| Separate index | rich search и независимый scale | eventual consistency, rebuild | search — важный product path |
| Edge n-grams | простой autocomplete | большой index | frequent updates, flexible filters |
| Trie/FST | компактный быстрый prefix | rebuild/update complexity | стабильный словарь |
| Refresh чаще | ниже freshness lag | меньше indexing throughput, больше segments | freshness важнее bulk efficiency |

## Типичные ошибки

- **Неверное предположение:** index — source of truth. **Симптом:** потеря shard удаляет бизнес-данные или stale ACL даёт доступ. **Причина:** projection получила authority. **Исправление:** canonical store, replay и source authorization boundary.
- **Неверное предположение:** dual write в DB и index атомарен. **Симптом:** один write прошёл, второй потерян. **Причина:** независимые systems. **Исправление:** outbox/CDC, versions и reconciliation.
- **Неверное предположение:** больше shards всегда быстрее. **Симптом:** coordination/heap overhead и recovery растут. **Причина:** query касается слишком многих мелких shards. **Исправление:** measured shard sizing и routing.

## Когда применять

Отдельный search service оправдан при большом corpus, relevance, text analysis, autocomplete и независимом read scale. Для exact lookup по нескольким полям сначала достаточно индекса transactional DB.
