# Проектирование файлового хранилища

## TL;DR

File service разделяет control plane и byte data plane. API хранит metadata, ownership, versions и upload state; clients передают bytes напрямую в object storage по scoped signed URLs. Publish/finalize делает object видимым только после проверки размера и checksum. Download идёт через CDN/object storage, а API выдаёт authorization decision и immutable version URL.

Multipart upload, metadata transaction и malware scan не образуют одну атомарную транзакцию. Поэтому нужен явный lifecycle `initiated → uploading → verifying → available | rejected | expired`, reconciliation и сборка abandoned parts. Immutable versions упрощают cache, rollback и concurrency; mutable filename остаётся pointer на version.

## Контекст и ментальная модель

Пользователи загружают files, скачивают их, создают новые versions, удаляют и делятся ссылкой. Размеры распределены неравномерно: p99 file способен быть на порядки больше median. API-сервер не должен проксировать bytes, иначе bandwidth и long connections станут его главным bottleneck.

Metadata отвечает «чей object, какая version доступна, кто вправе читать». Object store отвечает «где лежат bytes и совпадает ли их integrity metadata».

## Требования

### Functional requirements

- Initiate resumable/multipart upload и получить scoped part URLs.
- Завершить upload с checksum, создать immutable file version.
- Скачать whole file или byte range; поддержать conditional GET.
- List files cursor-страницами, rename logical filename, delete/restore.
- Share/revoke access и задать retention/lifecycle.
- Асинхронно сканировать malware и строить previews.

### Non-functional requirements

- Подтверждённая version не указывает на incomplete/corrupt object.
- Retry initiate/finalize не создаёт две logical versions.
- Большой upload не удерживает application worker.
- Authorization действует на metadata и signed URLs; угадывание key не даёт доступ.
- Orphaned uploads/objects обнаруживаются и очищаются безопасно.

### SLO: availability, latency, durability, consistency

Interview targets:

| Путь | SLO |
| --- | --- |
| Metadata/read authorization | 99,95%; p99 ≤ 300 ms |
| Initiate/finalize | 99,9%; p99 ≤ 500 ms без передачи bytes и scan |
| Byte download | 99,95% successful starts; throughput/complete latency задаётся по size/network class |
| Durability | available version не теряется при отказе зоны; region-loss RPO ≤ 5 min, RTO ≤ 30 min для critical class |
| Consistency | read-after-finalize для version metadata; revoke share запрещает новые download starts ≤ 10 s; delete/GC физически eventual |

Malware scan может задержать `available`. Если product разрешает «quarantine download owner only», это отдельный state и policy, а не скрытый exception. Revoke SLO относится к новой авторизации и началу нового HTTP/range request. Уже переданные bytes не отозвать, а stream, открытый до revoke, базовый object-storage path не прерывает; для mid-stream revoke нужен online edge proxy.

### Вне scope

- POSIX filesystem semantics и random in-place writes.
- Collaborative document editing.
- Видео transcoding/streaming protocol.
- Реализация erasure coding/object store internals.

## Traffic и storage estimates

Все числа — interview assumptions.

- 5 млн DAU, 0,2 uploads/user/day: `1M uploads/day`, `≈11,6/s` average, peak factor 10 — `≈116/s`.
- Средний file 8 MiB: `≈7,6 TiB/day`, около `2,7 PiB/year` logical bytes без version deletion.
- 5 downloads/user/day: `25M/day`, `≈289/s` average, `≈2,9k/s` peak.
- Если каждый download передаёт средние 8 MiB, egress — `≈191 TiB/day`. Реальный range/cache hit радикально меняет эту цифру.
- Metadata 1 KiB/version: `≈0,95 GiB/day`, `≈348 GiB/year` без индексов/реплик.
- При 20% abandoned multipart uploads и среднем 2 MiB до abandon временный leak: `200k × 2 MiB ≈391 GiB/day`, пока lifecycle его не удалит.

Capacity обязательно считает size histogram, concurrent uploads, part size, CDN hit, geographic egress и retention classes.

## API contract

```http
POST /v1/files:begin-upload
Idempotency-Key: 3c1...

{"name":"report.pdf","size":8388608,"checksum":{"sha256":"..."}}
```

Response содержит `file_id`, `version_id`, `upload_id`, recommended part size и scoped URLs. Server не принимает client path как object key.

```http
POST /v1/uploads/{upload_id}:complete
{"parts":[{"number":1,"etag":"..."}],"checksum":{"sha256":"..."}}

GET /v1/files/{file_id}
GET /v1/files/{file_id}/versions/{version_id}:download
DELETE /v1/files/{file_id}
POST /v1/files/{file_id}:share
```

Complete идемпотентен и возвращает прежнюю version. Download response может быть redirect/signed URL; metadata содержит `ETag`, size и content type.

## Data model

```text
file(file_id, owner_id, logical_name, current_version_id,
     state, created_at, deleted_at, row_version)

file_version(file_id, version_id, object_key, size, checksum,
             media_type, state, created_at, available_at)

upload_session(upload_id, file_id, version_id, expected_size,
               expected_checksum, state, expires_at, provider_upload_id)

share_grant(grant_id, file_id, principal_or_token_hash,
            permissions, expires_at, revoked_at, version)

outbox(event_id, aggregate_id, type, payload, published_at)
```

Object key случайный и versioned. Logical filename не участвует в physical identity. Metadata хранит checksum целого object независимо от provider ETag semantics.

## High-level architecture

```text
client -> file API -> metadata DB -> policy/audit
   |          |
   |          +-> signed upload/download URLs
   v
object storage <-> CDN
   |
   +-> events -> verifier / malware scanner / preview / lifecycle reconciler
```

File API — control plane. Object storage и CDN — byte plane. Event consumer не делает version available без conditional transition в metadata DB.

## Read path и write path

### Write path

1. Begin upload проверяет quota/policy, создаёт `upload_session` и immutable version в `uploading`.
2. Client загружает parts напрямую, может повторять part с тем же number.
3. Complete передаёт authoritative list parts; service завершает provider multipart и HEAD/checksum-проверку.
4. Verifier/malware scanner переводит version в `available`; metadata pointer обновляется conditional transaction.
5. Outbox запускает preview, audit и cache invalidation. Expired sessions abort-ятся lifecycle worker.

### Read path

1. Client запрашивает file/version; API проверяет owner/share grant и state.
2. API выдаёт signed URL либо CDN token, привязанный к immutable version и authorization decision. Decision живёт не больше 10 s, а token истекает не позже `decision_checked_at + 10 s`; выдача из старого cache не начинает новое 10-секундное окно.
3. CDN обслуживает hit. На miss object store отдаёт whole/range bytes и validators.
4. Revocation инвалидирует authorization decision. Уже выданный token может открыть новый request только до bounded expiry, поэтому не позже 10 s после revoke все новые whole/range starts проходят повторную authorization и получают отказ. Request, начатый раньше, может завершиться; это отдельная метрика и явно не входит в базовую гарантию остановки.

### Сквозная трассировка

Client начинает upload U8 из трёх parts. После parts 1–2 сеть рвётся; повторная загрузка part 2 с тем же number заменяет прежнюю часть. Complete request дошёл до object store, но response API потерян. Client повторяет complete. Session уже `verifying`, service HEAD-проверкой находит object с ожидаемыми size/checksum и возвращает ту же version V4. Scanner помечает V4 `available`, и только тогда current pointer меняется. Дубликата version и частичного файла нет.

## Выбор storage

Bytes хранятся в [[30 Данные/Object и blob storage|object/blob storage]]: whole-object PUT/GET, range read, lifecycle и большая ёмкость соответствуют workload. Metadata требует транзакций, unique idempotency и indexed list — SQL подходит до необходимости sharding.

CDN — cache, не durability layer. Search index для filenames/content — projection. Event stream переносит verification и lifecycle work, но upload session source остаётся DB.

## Partitioning и replication

- Metadata shard key — hash `file_id`; listing owner требует secondary projection/index по `(owner_id, created_at, file_id)`.
- Object key включает hash prefix + file/version ID, без user filename и PII.
- Metadata синхронно реплицируется между zones. Object store использует выбранный durability class; cross-region replication включается только для classes с region-loss SLO.
- Cross-region metadata и object replication должны иметь общий readiness rule: нельзя failover-ить pointer на version, object которой ещё нет в target region.
- Share grants partition-ятся с file либо проверяются отдельным strongly consistent policy service.

## Caching

Immutable version URL допускает долгий public cache. Private bytes можно кэшировать на edge лишь при online token/grant-version validation; bearer signed URL сам по себе не делает shared cache безопасным. Mutable `/files/{id}` имеет короткий TTL/version. Authorization decision хранит `checked_at` и `valid_until ≤ checked_at + 10 s`; token expiry не может продлить это окно. Version event инвалидирует cache раньше.

Range cache улучшает large-download resume, но fragments должны разделять один strong validator. Private bytes не попадают в public shared key.

## Async processing

Malware scan, preview, checksum audit, lifecycle transition, replication reconciliation и orphan GC идут в background. Durable job хранит obligation; worker lease/fencing не позволяет stale worker сделать rejected version available.

Queues разделены для fresh verification и bulk lifecycle. Backfill previews не задерживает доступность новых uploads. Retry idempotent по `(version_id, job_type, algorithm_version)`.

## Failure handling

| Failure | Сигнал | Реакция | Остаточный эффект |
| --- | --- | --- | --- |
| Part upload оборвался | client retry, incomplete parts | resumable part retry; session TTL | временные bytes |
| Complete outcome неизвестен | timeout + session state | HEAD/list known parts, idempotent finalize | extra provider calls |
| Metadata commit без object readiness | invariant monitor | state не выше `verifying`; reconciliation/rollback pointer | file ещё недоступен, не corrupt |
| Scanner outage | verification queue age | quarantine, scale/failover scanner; не публиковать unsafe class | availability upload completion ниже |
| CDN/object start после revoke | deny-after-revoke probe, token/edge logs | token expiry ограничен исходным 10-секундным decision window; online grant-version check для более строгого path | начатый до revoke stream может завершиться; новые starts блокируются ≤ 10 s |
| Object region lag | replica lag vs pointer | не объявлять target ready; fallback source region | failover/egress медленнее |
| Orphan GC ошибся | referenced-object check | mark-and-sweep grace period, version retention | storage выше, но data не теряется |

## Security

- Signed upload ограничивает exact key, method, part/size, checksum policy и expiry; credentials client не получает.
- Filename и declared content type не доверяются; response ставит безопасный `Content-Disposition`, sniffing policy и malware status.
- Object store закрыт от public listing; random key не заменяет authorization.
- Share token хранится hash-ом, имеет scope/expiry/revoke и не попадает в logs/referrer.
- Download token несёт immutable version и истекает не позже authorization `valid_until`; повторная подпись из stale cache не продлевает grant. Для требования прерывать активный stream используется online edge authorization, а не bearer URL.
- Encryption at rest и TLS обязательны; tenant-specific keys/crypto-erasure применяются по threat/compliance model.
- Quotas учитывают total bytes, number of parts, concurrent uploads и egress; одного request count недостаточно.

## Observability

SLI: begin/complete latency, upload success by size bucket, bytes/s, checksum mismatch, verification age, scanner failure, orphan bytes, CDN hit, range/whole traffic, signed-URL denial, object/metadata replication lag и lifecycle deletion. Для revoke отдельно измеряем authorization decision age, время от revoke commit до отказа нового whole/range start и число streams, начатых до revoke и завершившихся после него. Metrics по file ID недопустимы; trace/log correlation использует sampled opaque IDs.

Synthetic flow загружает multi-part object, повторяет part/complete, скачивает range, проверяет checksum, revoke и deletion. Restore drill проверяет не bucket existence, а доступный file через metadata/policy.

## Capacity planning

Storage growth `≈7,6 TiB/day`; с 30-day hot retention это `≈229 TiB logical`, с годовым retention — `≈2,7 PiB`. Versioning, replicas и incomplete uploads умножают число. Lifecycle экономически важнее API micro-optimization.

Peak download full-file bandwidth при `2,9k/s × 8 MiB ≈ 23 GiB/s`. Даже 90% CDN hit оставляет origin `≈2,3 GiB/s`. Эти оценки определяют CDN/egress contract и region placement.

## Cost trade-offs

Storage, replication и egress доминируют. Cross-region copy всех files может стоить больше control plane; применяем durability classes. CDN уменьшает origin egress/latency для repeated downloads, но cold unique files платят за fill без hit benefit.

Мелкие multipart parts улучшают retry granularity, но увеличивают request cost и metadata; крупные parts хуже повторять. Recommended size выбирают по file histogram/network, сохраняя provider limits.

## Migration и rollout plan

1. Вынести bytes из application DB в object storage: новые uploads пишутся туда, old files читаются fallback path.
2. Backfill copies immutable objects, сверяя size/checksum; metadata pointer пока остаётся old.
3. Dual read сравнивает headers/checksum и постепенно переключает cohorts.
4. Write authority переводится на object path; старое blob поле остаётся rollback source до retention window.
5. CDN включается сначала для immutable public versions, затем для tokenized private traffic после cache-key/security test.
6. Cross-region replication и failover вводятся per storage class; promotion разрешён после object+metadata watermark.

## Bottlenecks

- Byte egress и concurrent long transfers.
- Very large multipart session/part metadata.
- Scanner/preview CPU и queue lag.
- Hot shared file и CDN miss stampede.
- Metadata listing index по owner.
- Orphan lifecycle/GC safety.
- Cross-region object replication backlog.

## Trade-offs и альтернативы

| Выбор | Выигрыш | Цена | Когда выбирать |
| --- | --- | --- | --- |
| Proxy bytes через API | единая auth точка | bandwidth/connection bottleneck | малые files, строгая inline transform |
| Direct signed transfer | scale и дешевле app tier | token policy и finalize workflow | большие files |
| Mutable object key | простой URL | cache/concurrency/rollback сложнее | редко оправдан |
| Immutable versions | cache, audit, rollback | storage и GC | основной вариант |
| Cross-region all data | низкий region-loss RPO | replication/egress cost | critical class |

## Типичные ошибки

- **Неверное предположение:** multipart complete и metadata commit атомарны. **Симптом:** orphan object или pointer на missing data. **Причина:** разные systems. **Исправление:** lifecycle states, idempotent finalize и reconciliation.
- **Неверное предположение:** provider ETag всегда checksum целого файла. **Симптом:** integrity check ложно проходит/падает для multipart. **Причина:** provider-specific ETag semantics. **Исправление:** хранить явный checksum algorithm/value.
- **Неверное предположение:** delete row достаточно. **Симптом:** bytes, versions, CDN и shares живут дальше. **Причина:** distributed deletion workflow. **Исправление:** tombstone, revoke, purge/GC и audit до terminal state.

## Когда применять

Дизайн подходит для user uploads, attachments, backups и media с whole/range access и immutable versions. Для low-latency random block updates, shared POSIX locks или database pages нужен block/file storage с другим consistency contract.
