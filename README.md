# Репозиторий знаний для backend-собеседований

Локальный Git-репозиторий и Obsidian-vault для русскоязычных технических конспектов единой Senior+-глубины и отдельного направления Go. Базовые концепции начинаются с первых принципов, но доводятся до инвариантов, failure modes и практических trade-offs.

Первая версия содержит только навигацию, тематические планы и шаблоны. Предметные заметки из планов ещё не созданы.

## Начало работы

1. Откройте Obsidian.
2. Выберите **Open folder as vault**.
3. Укажите корень этого репозитория.
4. Начните с [Главной](Главная.md) или единого маршрута [Backend — от основ к архитектуре](<01 Маршруты/Backend — от основ к архитектуре.md>).

Vault использует только встроенные возможности Obsidian. Community plugins не требуются.

## Как создать предметную заметку

1. Откройте нужную тематическую карту.
2. Выберите конкретную ссылку в секции `План заметок`; не создавайте все запланированные файлы заранее.
3. Создайте файл по пути, указанному в wiki-link.
4. В пустой заметке выполните команду **Templates: Insert template** и выберите подходящий файл из `99 Шаблоны`.
5. Проверьте frontmatter, статус и теги, затем наполните разделы и добавьте источники.
6. После завершения перенесите ссылку из `План заметок` в `Готовые заметки` соответствующей карты.

Встроенный Templates вставляет шаблон по команде и не применяет его автоматически при создании файла.

## Frontmatter

У vault-страницы должны быть только три поля:

```yaml
---
aliases: []
tags: []
статус: черновик
---
```

Допустимые значения `статус`: `черновик`, `проверено`, `требует обновления`.

Теги пишутся в нижнем регистре и образуют простую русскую иерархию, например `тип/карта`, `тип/маршрут`, `область/go`. Статус тегами не дублируется. Используйте один–три тега, только если они улучшают поиск.

## Глубина и объём заметки

- Все предметные заметки пишутся с Senior+-глубиной, но не предполагают, что читатель уже знает базовый механизм.
- Одна заметка отвечает на один связный вопрос и строится по цепочке `проблема → ментальная модель → механизм → наблюдаемый результат → правило выбора`.
- Для нетривиального поведения нужен один минимальный сквозной пример: код, запрос, trace, последовательность событий или failure-сценарий.
- Подробности добавляются только тогда, когда они меняют понимание, прогноз поведения или практическое решение.
- Жёсткого лимита слов нет: после написания удаляются повторы, общие вступления и декоративные списки.

## Ссылки

- Внутренние ссылки используют wiki-формат и полный путь от корня vault без `.md`.
- Постоянная ссылка должна вести на существующий файл и, если указан заголовок, на существующую секцию.
- Ссылки на ещё не созданные страницы допустимы только внутри явно обозначенной секции `План заметок`.
- В `Готовые заметки`, маршрутах и связанных темах размещаются только существующие цели.
- Для внешних источников используются обычные Markdown-ссылки.

Obsidian настроен на [wiki-links и автоматическое обновление ссылок](https://obsidian.md/help/links), отдельную [папку вложений](https://obsidian.md/help/attachments) и встроенный [Templates](https://obsidian.md/help/plugins/templates).

## Источники

Ключевые утверждения подтверждаются подходящими первичными источниками:

1. спецификациями, стандартами и RFC;
2. официальной документацией, release notes, proposals и migration guides;
3. исходным кодом и тестами на фиксированном tag или commit.

Вторичные материалы допустимы для дополнительного объяснения, но не заменяют первичный источник. Для версионно-зависимого утверждения указывайте точную версию и дату проверки. Ссылка на исходный код должна вести на permalink конкретного tag/commit, файла и по возможности символа, а не на плавающую ветку.

Рекомендуемый формат:

```text
[Название](URL) — организация или репозиторий, версия или commit/tag, проверено YYYY-MM-DD.
```

Если поведение менялось, сравните подтверждённые состояния «раньше» и «сейчас», укажите практический эффект и приложите минимальные примеры для обеих версий, когда это помогает пониманию.

## Входящие и вложения

1. Сырые ссылки, фрагменты и идеи временно помещайте в `00 Входящие`.
2. Определите тематическую карту и существующую плановую ссылку.
3. Создайте одну предметную заметку по правилам выше и перенесите полезный материал с указанием источника.
4. Файлы, изображения и PDF сохраняйте в `90 Вложения`.
5. Удалите или переработайте исходный материал из входящих только после переноса полезного контекста.

`00 Входящие` — временная очередь, а не постоянное место хранения знаний.

## Соглашения репозитория

- Предметные папки и файлы именуются по-русски; стандартные служебные файлы сохраняют общепринятые имена.
- Текстовые файлы используют UTF-8 и LF.
- Markdown должен оставаться понятным без Obsidian.
- Состояние окон, устройства, локальные настройки редактора и community plugins не хранятся в Git.
- Labs, локальный каталог Playground, CI, remote, публикация и лицензия не входят в первую версию.

Правила для нейросетевых агентов находятся в [AGENTS.md](AGENTS.md).
