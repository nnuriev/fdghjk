# Multi-leader replication

## TL;DR

Multi-leader replication разрешает принимать записи в нескольких узлах или регионах, а затем обмениваться ими, обычно асинхронно. Она сокращает локальную write latency и позволяет площадке продолжить автономную работу, но превращает concurrent writes в штатный случай. Поэтому система обязана определить, как обнаружить независимые версии, упорядочить повторную доставку и разрешить конфликт без нарушения бизнес-инвариантов.

Главный вопрос здесь не «сколько лидеров», а «какие операции можно безопасно выполнить без общего порядка». Независимые изменения разных объектов часто сливаются. Уникальное имя, остаток денег или свободный временной слот требуют единственного владельца, координации либо специального протокола резервирования. Last-write-wins даёт сходимость, но может молча потерять подтверждённое намерение.

## Область применимости

Концептуальная модель опирается на Bayou, опубликованный в 1995 году. Современные примеры поведения сверены с Azure Cosmos DB multi-region writes и режимом MREC в Amazon DynamoDB Global Tables на 2026-07-18. Конкретные продукты отличаются гранулярностью conflict detection, consistency levels и доступными merge policies; термин `multi-leader` сам по себе гарантий не задаёт.

Заметка рассматривает логические записи в нескольких write locations. Внутренняя репликация каждого лидера по большинству и маршрутизация всего приложения остаются вне scope.

## Ментальная модель

У каждого лидера есть локальный упорядоченный журнал. Пока связь работает, журналы обмениваются операциями. При задержке или partition каждый лидер продолжает свою ветку:

```text
Region A: v0 -> a1 -> a2
                    \
                     merge -> v3
                    /
Region B: v0 -> b1
```

`a2` и `b1` нельзя честно назвать «последними» только по положению в разных журналах. Если между ними нет causal dependency, они concurrent. После восстановления связи нужны три шага: обнаружить расхождение, выбрать или вычислить результат, распространить его всем репликам. Это та же цепочка `detection → reconciliation → propagation`, которая подробно разобрана в [[30 Данные/Data repair и reconciliation|заметке о repair и reconciliation]].

Полезный инвариант: все реплики должны прийти к одному результату при любом порядке повторной доставки. Но convergence слабее correctness. Два региона могут одинаково сохранить отрицательный остаток и тем самым сойтись на неверном бизнес-состоянии.

## Как устроено

### Локальный commit и межлидерная доставка

Клиент пишет в ближайший лидер. Тот фиксирует операцию согласно локальной durability policy и может ответить до получения удалённым регионом. В ответе поэтому заключена локальная гарантия, а не глобальная видимость.

Репликационная запись несёт стабильный operation ID, origin/epoch, позицию источника и metadata причинности либо версии. Повторная доставка обязательна после reconnect, значит применение записи должно быть идемпотентным. В кольцевой или all-to-all топологии без origin ID одна операция может вернуться к источнику и примениться повторно.

Топология меняет failure modes. All-to-all сокращает число hops, но число каналов растёт квадратично. Hub-and-spoke проще контролировать, зато hub становится пределом распространения и местом, где накапливается backlog. Каскад A → B → C увеличивает время сходимости и требует сохранять исходную идентичность операции, а не назначать новую на каждом hop.

### Обнаружение concurrent writes

Есть три распространённых подхода:

- causal metadata, например version/vector context, отличает потомка от независимой ветки;
- optimistic concurrency отклоняет write, устаревший относительно уже известной локальной версии; во время partition два лидера всё равно могут принять операции с одним parent, поэтому межлидерный конфликт обнаруживается позже;
- timestamp-based LWW выбирает максимальную метку времени и скрывает сам факт конкуренции от приложения.

Bayou использовал dependency checks и merge procedures: write могла проверить условие на текущем состоянии, а при нарушении вызвать предметную процедуру слияния. Dynamo-подобная causal versioning сохраняет siblings, если ни одна версия не наследует другую. LWW дешевле, но физические часы не доказывают причинный или пользовательский порядок. Ошибка часов способна заставить старую запись побеждать новые долгое время.

### Разрешение конфликта

Resolution policy выбирают по семантике данных:

- детерминированный winner, например `(logical_version, region_id)`, обеспечивает convergence, но отбрасывает проигравшее изменение;
- field-level merge сохраняет независимые изменения разных полей, если операции действительно коммутируют;
- application merge проверяет предметные правила и может создать новую объединённую версию;
- CRDT кодирует операции так, чтобы все реплики сошлись без единого порядка при выполнении его алгебраических предпосылок.

Нельзя автоматически переносить merge с одного объекта на транзакцию из нескольких объектов. Слияние `shipping_address` и `notification_language` может быть безопасным. Два локально успешных `reserve(slot=10:00)` нарушают уникальность, даже если позже один winner вытеснит другого: оба клиента уже получили подтверждение.

Для несливаемых инвариантов применяют single-writer-per-key, home region, синхронную координацию, escrow/partitioned rights или отказ одной из конкурентных операций. Такой «карман координации» не отменяет multi-leader для остальных данных.

### Reconfiguration и возвращение региона

Регион после долгой изоляции нельзя сразу считать равноправным. Сначала определяют его topology epoch, дочитывают журналы, устраняют конфликты и только затем возвращают пользовательский трафик. Иначе stale node реплицирует старые операции как новые или принимает записи по устаревшим правилам.

Автоматический failover тоже может неожиданно превратить active-passive схему в multi-leader: старый primary продолжает принимать writes, а новый уже повышен. Без fencing это split brain, а не запланированная multi-leader semantics.

## Пример или трассировка

Профиль `user=42` хранит `email`, `language` и `credit_limit`. Регионы A и B начинают с версии `v7`.

1. Во время partition пользователь в A меняет `email`; возникает операция `a8` с parent `v7`.
2. Почти одновременно оператор в B меняет `language`; операция `b8` тоже наследует `v7`.
3. После reconnect causal metadata показывает, что `a8` и `b8` concurrent: ни одна не видела другую.
4. Field-level merge создаёт `v9`, где сохранены оба изменения. Идентификатор merge стабилен, поэтому повторная доставка не создаёт новые версии.

Для `credit_limit` та же стратегия опасна. Если A и B независимо уменьшают доступный лимит после двух покупок, выбор одного целого документа через LWW потеряет вторую покупку. Покомпонентное сложение тоже корректно лишь при модели операций, которая выдерживает повторы и отмены. Практическое решение: закрепить счёт за одним write owner, использовать координируемую транзакцию либо заранее разделить разрешённый лимит между регионами.

## Trade-offs

Multi-leader уменьшает latency локальной записи и выдерживает потерю межрегиональной связи. Цена переносится в metadata, replication lag, конфликтные состояния и бизнес-логику merge.

[[30 Данные/Репликация данных#Leader/follower|Один лидер]] проще поддерживает уникальность и единый порядок, но удалённые writers платят WAN latency и зависят от failover. [[30 Данные/Репликация данных#Leaderless|Leaderless-модель]] тоже допускает расходящиеся версии, однако authority задаётся набором реплик на каждую операцию, а не несколькими долгоживущими write leaders. В обоих случаях название архитектуры слабее точного протокола commit, versioning и repair.

Синхронная межрегиональная репликация избегает части конфликтов, но возвращает latency самого медленного необходимого региона и останавливает writes при потере требуемого quorum. Асинхронная сохраняет local availability, принимая ненулевой lag и необходимость reconciliation.

## Типичные ошибки

- **Неверное предположение:** active-active compute автоматически означает безопасные записи в обоих регионах. **Симптом:** после failover обнаруживаются два подтверждённых значения одного ключа. **Причина:** write authority и conflict policy не были спроектированы. **Исправление:** отдельно описать routing вычислений, владельца записи и merge для каждого класса данных.
- **Неверное предположение:** LWW выбирает последнее действие пользователя. **Симптом:** более позднее изменение исчезает. **Причина:** clock timestamp принят за causal order. **Исправление:** causal/version metadata, clock discipline либо явный предметный merge.
- **Неверное предположение:** одинаковый итог означает корректность. **Симптом:** все регионы сходятся на нарушенном лимите или двойном бронировании. **Причина:** convergence не сохраняет cross-object invariant. **Исправление:** координация или разделение прав для некоммутативных операций.
- **Неверное предположение:** reconnect сам безопасно вернёт регион. **Симптом:** stale writes снова расходятся после восстановления. **Причина:** не проверены epoch, backlog и schema compatibility. **Исправление:** staged rejoin, reconciliation и только затем write traffic.

## Когда применять

Multi-leader оправдана для geographically distributed или intermittently connected workloads, где локальная запись важнее немедленного общего порядка, а большинство изменений коммутируют, могут быть объединены или привязаны к региональному владельцу. Типичные примеры: пользовательские предпочтения, каталоги с раздельными owners, offline-first данные.

Не начинайте с неё для денег, глобальной уникальности, inventory без escrow и workflow, где два локальных подтверждения нельзя позднее отменить. На design review для каждого write type фиксируют conflict unit, causal metadata, merge function, replication topology, maximum tolerable lag и поведение клиента при проигравшей версии.
