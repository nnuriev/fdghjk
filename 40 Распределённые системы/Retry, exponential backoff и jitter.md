# Retry, exponential backoff и jitter

## TL;DR

Retry заменяет один неуспешный удалённый вызов новой попыткой. Он полезен только для временной ошибки, replayable запроса и достаточного остатка общего deadline. Каждый повтор создаёт дополнительную нагрузку именно тогда, когда система уже может быть неисправна.

Exponential backoff увеличивает паузу между попытками, cap ограничивает её сверху, а jitter разводит клиентов по времени. Без jitter клиенты, которые одновременно увидели сбой, просыпаются теми же волнами. Политика обязана также ограничивать число попыток, уважать server pushback и иметь retry budget; иначе recovery превращается в retry storm.

## Область применимости

- HTTP-семантика соответствует RFC 9110 и RFC 9112, опубликованным в июне 2022 года.
- Поведение gRPC описано по официальному Retry guide, проверенному 2026-07-18.
- Детали автоматических повторов `net/http.Transport` соответствуют Go 1.26.5.
- Вне scope: broker redelivery, workflow-level retries длиной в часы и настройка SDK конкретного облачного API.

## Ментальная модель

Retry — это новый независимый шанс выполнить ту же логическую операцию. Между попытками мир меняется: первый сервер мог успеть применить запрос, лидер мог смениться, capacity могла восстановиться, а deadline — почти закончиться.

Полезно мыслить двумя бюджетами:

- **time budget** ограничивает суммарные попытки, ожидания и backoff;
- **retry budget** ограничивает добавочную нагрузку относительно обычного трафика.

Инвариант: один логический вызов не должен порождать неограниченное число физических attempts, а все attempts вместе не должны пережить исходный [[20 Бэкенд/Дедлайны запросов и распространение отмены|deadline]].

## Как устроено

### Решение о повторе

Retry допустим, когда одновременно выполняются четыре условия:

1. ошибка классифицирована как transient, а не validation, authentication или устойчивый конфликт;
2. операция идемпотентна либо защищена дедупликацией;
3. request body можно воспроизвести byte-for-byte или заново построить по тому же контракту;
4. в общем бюджете осталось время на pause, новую попытку и возврат ответа.

RFC 9110 разрешает автоматически повторять idempotent request после потери соединения до получения ответа. Неидемпотентный request автоматически повторяют только при дополнительном знании, что его семантика безопасна. `net/http.Transport` в Go 1.26.5 ещё строже: retry при network error делается на ранее успешно использованном connection, для idempotent request и только без body либо при заданном `Request.GetBody`. Наличие `Idempotency-Key` или `X-Idempotency-Key` также участвует в классификации Transport.

Ошибка после отправки запроса часто оставляет unknown outcome. Сетевой reset не доказывает, что сервер не выполнил операцию. Поэтому transport-level retry и бизнес-дедупликация должны согласовываться с [[20 Бэкенд/Ключи идемпотентности и дедупликация запросов|контрактом idempotency key]].

### Backoff и jitter

Сначала вычисляют верхнюю границу паузы очередной попытки:

```text
cap_n = min(max_backoff, initial_backoff * multiplier^n)
```

Full Jitter выбирает фактическую паузу равномерно:

```text
sleep_n = random(0, cap_n)
```

Так средняя нагрузка размазывается по интервалу. «Чистый» exponential backoff без случайности сохраняет синхронность cohort: все клиенты стартовали вместе, получили ошибку вместе и повторят вместе.

Алгоритмы различаются. gRPC, например, применяет jitter `±20%` к вычисленному backoff. HTTP `Retry-After` принимает `HTTP-date` либо целое `delay-seconds`: клиент, который следует этому hint, не повторяет раньше указанного момента, а если ожидание не помещается в общий deadline, отказывается от retry. gRPC `grpc-retry-pushback-ms` задаёт точную задержку в миллисекундах; отрицательное или неразбираемое значение запрещает retry. Локальная политика не должна заменять server pushback более агрессивной паузой.

### Total deadline и attempt timeout

У каждого attempt может быть локальный timeout, но он вложен в общий deadline. Перед стартом клиент оценивает:

```text
remaining > planned_sleep + minimum_useful_attempt + response_reserve
```

Если условие ложно, попытка лишь увеличит нагрузку и всё равно не успеет вернуть полезный ответ. Backoff sleep должен прерываться cancellation, а не блокировать goroutine до конца таймера.

### Где размещать retry

Повтор лучше выполнять на одном уровне, который понимает семантику операции и наблюдает общий budget. Если gateway, service и SDK независимо делают по три попытки, один пользовательский запрос в худшем случае создаёт `3 × 3 × 3 = 27` downstream attempts. Ещё один уровень даст 81.

Retry budget или token-based throttling приостанавливает повторы, когда доля ошибок растёт. В gRPC retry throttling не разрешает новые retries, когда token count не превышает половину `maxTokens`; успешные RPC постепенно восстанавливают запас. Это отрицательная обратная связь, которой нет у простого `maxAttempts`.

После получения response headers gRPC считает RPC committed и больше не повторяет его. Без явной retry policy остаются только transparent retries для низкоуровневых случаев, когда библиотека уверена, что application logic сервера запрос не видела.

## Пример или трассировка

У клиента общий deadline 900 ms. Политика допускает три attempts, attempt timeout 250 ms, Full Jitter с `initial=100 ms`, `multiplier=2`, `max=400 ms`. Операция идемпотентна.

1. Attempt 1 завершается `UNAVAILABLE` через 120 ms. Для первого backoff cap равен 100 ms, генератор выбирает 60 ms.
2. Attempt 2 начинается в `t=180 ms` и упирается в свой timeout в `t=430 ms`.
3. Следующий cap равен 200 ms; выбран jitter 170 ms. В `t=600 ms` остаётся 300 ms общего бюджета.
4. Attempt 3 получает timeout 250 ms с резервом 50 ms на возврат ответа и завершается успешно через 110 ms, в `t=710 ms`.

Без jitter тысяча клиентов после attempt 1 снова пришла бы примерно в `t=220 ms`. С Full Jitter их вторые попытки распределились бы по интервалу `120..220 ms`. Если бы после второго backoff оставалось меньше минимально полезного времени, третью попытку следовало бы не запускать.

## Trade-offs

Больше attempts повышает вероятность пережить короткий сбой, но увеличивает tail latency, стоимость и нагрузку. Быстрый retry полезен при stale connection или смене endpoint; при перегруженном сервере он ухудшает ситуацию.

Full Jitter обычно уменьшает суммарную работу и пики, но отдельный клиент иногда получает короткую или длинную паузу. Equal и decorrelated jitter дают другую форму распределения. Выбор алгоритма менее важен, чем отказ от синхронного deterministic backoff и наличие cap.

Hedging запускает дополнительную попытку до ошибки первой и снижает tail latency на read-like операциях. Цена выше: конкурентная работа создаётся даже без явного сбоя. Hedging требует ещё более строгой идемпотентности и отдельного budget.

Fail-fast лучше retry, когда ошибка постоянна, budget почти исчерпан или downstream уже защищён открытым [[40 Распределённые системы/Circuit breaker|circuit breaker]].

## Типичные ошибки

### Повтор на каждом слое

- **Неверное предположение:** локальные три попытки дают небольшую надёжную страховку.
- **Симптом:** во время сбоя downstream получает десятки вызовов на один пользовательский запрос.
- **Причина:** числа attempts перемножаются по глубине цепочки.
- **Исправление:** выбрать один семантически осведомлённый слой, в остальных отключить либо жёстко ограничить retry.

### Exponential backoff без jitter

- **Неверное предположение:** растущая пауза сама разнесёт клиентов.
- **Симптом:** нагрузка приходит периодическими высокими волнами.
- **Причина:** одинаковое время старта и одинаковые интервалы сохраняют фазу клиентов.
- **Исправление:** добавить случайность и cap, проверить распределение под массовым отказом.

### Retry любой ошибки

- **Неверное предположение:** новая попытка может исправить `400`, `403` или детерминированный conflict.
- **Симптом:** лишняя latency без роста success rate.
- **Причина:** не разделены transient и permanent outcomes.
- **Исправление:** явный allowlist retryable status/errors и server pushback.

### Попытки имеют отдельные полные timeouts

- **Неверное предположение:** `3 × 500 ms` совместимо с клиентским deadline 800 ms.
- **Симптом:** поздние attempts выполняются после ухода клиента.
- **Причина:** нет total budget.
- **Исправление:** ограничить весь retry loop одним deadline и пересчитывать остаток перед каждой попыткой.

## Когда применять

Retry применяют к редким transient failures, когда операция воспроизводима и система имеет запас capacity. Политику проверяют fault injection и нагрузочным тестом: важны success-after-retry, attempts per call, added load, backoff distribution и доля вызовов, остановленных budget.

Для перегрузки retries должны сочетаться с [[20 Бэкенд/Rate limiting и quotas|rate limits]], server pushback и retry throttling. Если зависимость долго нездорова, breaker или fail-fast защищают лучше очередной попытки.
