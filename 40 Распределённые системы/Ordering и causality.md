# Ordering и causality

## TL;DR

В распределённом выполнении естественен не один глобальный порядок, а частичный порядок причинности (causality). Событие `a` happened-before `b`, если они упорядочены внутри одного процесса, `a` отправляет сообщение, полученное в `b`, либо между ними есть цепочка таких связей. Если нет пути ни `a -> b`, ни `b -> a`, события concurrent: это означает отсутствие известной причинной связи, а не физическую одновременность.

Lamport clock гарантирует `a -> b => L(a) < L(b)`, но обратное неверно: числа могут искусственно упорядочить независимые события. Vector clock сохраняет больше информации и позволяет отличить descendant от concurrent version, однако его metadata растёт с числом участников. Total order нужен там, где операции конфликтуют и должны иметь один результат; требовать его для независимых действий — платить coordination latency и availability без пользы.

## Область применимости

Заметка рассматривает порядок событий и версий в message-passing системе. Порядок доставки конкретного broker, serializability транзакций и linearizability объектов — соседние, но отдельные контракты. Формальная модель сверена с работами Lamport 1978 и Mattern 1988/1989; практическая versioning-схема — с Dynamo SOSP 2007. Источники проверены 2026-07-18.

## Ментальная модель

Нарисуйте для каждого процесса вертикальную линию, а для каждого сообщения — стрелку. Двигаться можно вниз по линии локального выполнения и вдоль стрелки send-to-receive. Если из `a` можно дойти в `b`, `a` потенциально повлияло на `b`. Если пути нет в обе стороны, внешний наблюдатель может нарисовать несколько допустимых временных раскладок, не меняя наблюдаемое выполнение.

Отсюда три разных требования:

- **Partial/causal order:** сохраняет только зависимости.
- **Per-key или per-partition order:** сериализует выбранную группу событий, но ничего не обещает между группами.
- **Total order:** ставит все события в одну последовательность, в том числе независимые.

Более сильный порядок не обязательно «правильнее». Он добавляет решения, которых не было в причинной структуре, и требует общей точки координации либо consensus.

## Как устроено

### Happened-before фиксирует минимальную причинную структуру

Lamport задаёт отношение `->` как наименьшее, удовлетворяющее трём правилам:

1. события одного процесса идут в локальном порядке;
2. send конкретного сообщения предшествует соответствующему receive;
3. отношение транзитивно.

Это строгий partial order: есть пары, которые нельзя сравнить. Он описывает потенциальное влияние в принятой модели событий. Внешняя связь, не попавшая в сообщения и local trace, например человек скопировал значение между двумя системами, для алгоритма невидима; metadata не может восстановить неизвестную причинность.

### Scalar logical clock сохраняет направление, но теряет concurrency

Каждый процесс увеличивает счётчик перед событием. Сообщение несёт текущее значение; receiver устанавливает clock больше и своего значения, и timestamp сообщения. Тогда clock condition выполняется: если `a -> b`, то `L(a) < L(b)`.

Но из `L(a) < L(b)` нельзя заключить `a -> b`. Два независимых процесса могут получить разные числа из-за своих локальных событий. Пара `(L, process_id)` даёт deterministic total order для очереди или tie-break, однако этот порядок искусственный. Он годится, если протоколу нужен общий выбор и все участники применяют одно правило, но не служит доказательством causal descendant.

### Vector clock различает descendant и concurrent version

Vector clock хранит по компоненте на участника. Процесс увеличивает свою компоненту; при receive берёт component-wise maximum с вектором сообщения, затем отмечает новое событие. Для двух timestamps `V` и `W`:

```text
V < W, если для всех i: V[i] <= W[i]
       и хотя бы для одного j: V[j] < W[j]
```

В классической модели это соответствует причинному предшествованию. Если ни `V < W`, ни `W < V`, версии concurrent. Система может сохранить обе как siblings и передать их предметному merge. Vector clock обнаруживает конфликт, но не знает, как объединить адрес, корзину или денежный остаток.

Цена — размер и lifecycle identity. При динамическом числе writers vector растёт, retired identities нужно безопасно сворачивать, а truncation способна потерять causal information. Dynamo использовал vector clocks и ограничивал рост metadata, принимая соответствующий компромисс. Version vector также должен относиться к ясной сущности: общий vector всей базы быстро создаст ложные зависимости, а слишком узкий не увидит конфликт нужного aggregate.

### Delivery order и causal order — разные контракты

TCP упорядочивает bytes одного соединения, но reconnect создаёт новую последовательность. Broker обычно сохраняет порядок внутри partition, пока producer/retry policy не вносит дополнительную семантику; между partitions общего порядка нет. Подробности этой границы разобраны в [[40 Распределённые системы/Очереди, streams, группы потребителей и DLQ|заметке об очередях и streams]].

Causal delivery требует не выдавать event раньше его dependencies. Если consumer получил `b` с зависимостью от `a`, а `a` задержалось, `b` приходится буферизовать или дочитать. Это сохраняет ментальную модель, но добавляет latency, metadata и риск queue buildup. Для независимых событий causal order не выбирает последовательность.

### Total order нужен для общего решения

Если две команды конкурируют за единственный ресурс, всем replicas нужен одинаковый порядок либо операция должна быть commutative. Leader/consensus log присваивает одну позицию и реплицирует решение. Глобальный sequencer упрощает state machine, но становится coordination boundary: межрегиональный RTT, quorum availability и head-of-line blocking входят в цену.

Альтернатива — разбить order domain по ключу. События одного заказа идут последовательно, разные заказы обрабатываются параллельно. Это безопасно только если инвариант также локален ключу; глобальный лимит нельзя защитить независимыми per-user logs без дополнительной координации.

## Пример или трассировка

Две replicas редактируют один профиль во время partition. Начальный vector `[A:0, B:0]`.

1. `A` записывает адрес `Paris`, увеличивает свою компоненту: `vA=[1,0]`.
2. Не видя эту запись, `B` записывает `Rome`: `vB=[0,1]`.
3. После healing vectors несравнимы: `vA` больше по `A`, `vB` — по `B`. Обновления concurrent. Scalar Lamport timestamps с tie-break выбрали бы один deterministic winner, но не доказали бы, что он потомок второго.
4. Приложение показывает конфликт или выполняет предметный merge. Допустим, пользователь подтверждает `Rome` после чтения обеих versions. Replica объединяет knowledge в `[1,1]` и создаёт новую запись `[1,2]`.
5. Теперь `[1,0] < [1,2]` и `[0,1] < [1,2]`: resolved version causally наследует обе ветви.

Наблюдаемый результат: metadata не разрешило конфликт автоматически, зато не позволило принять одну concurrent запись за stale descendant и молча потерять другую.

## Trade-offs

| Механизм | Выигрыш | Цена и ограничение |
| --- | --- | --- |
| Physical timestamp + LWW | Малое metadata, простой выбор winner | [[40 Распределённые системы/Clock skew и отсутствие глобального времени|Clock skew]], скрытая потеря concurrent update |
| Lamport clock | Компактный порядок, совместимый с causality | Не распознаёт concurrency; total tie-break искусственный |
| Vector clock | Точно различает causally ordered и concurrent события в модели | `O(writers)` metadata, сложный membership/truncation |
| Per-key sequencer/log | Понятный порядок при параллелизме между keys | Cross-key invariant требует новой coordination boundary |
| Global total order | Простая единая state machine | Quorum/leader latency, меньшая availability и bottleneck порядка |
| Commutative merge | Локальные операции без общего порядка | Подходит только при доказанной предметной алгебре и delete semantics |

## Типичные ошибки

- **Неверное предположение:** меньший timestamp означает причинно более раннее событие. **Симптом:** независимая версия ошибочно отбрасывается как stale. **Причина:** scalar order не даёт обратной импликации. **Исправление:** хранить causal metadata или сериализовать writers.
- **Неверное предположение:** concurrent означает «в одну наносекунду». **Симптом:** conflict policy зависит от физической близости времени. **Причина:** concurrency означает отсутствие causal path. **Исправление:** рассуждать через local order и сообщения.
- **Неверное предположение:** vector clock разрешает конфликт. **Симптом:** инфраструктурный tie-break теряет бизнес-изменение. **Причина:** vector только классифицировал versions. **Исправление:** задать semantic merge, user choice или единый owner.
- **Неверное предположение:** порядок одной partition глобален. **Симптом:** consumer видит `OrderShipped` до связанного `PaymentCaptured` из другого key. **Причина:** зависимости пересекли order domains. **Исправление:** совместный partition key, dependency metadata или workflow coordination.
- **Неверное предположение:** total order автоматически нужен всей системе. **Симптом:** независимые writes ждут один глобальный sequencer. **Причина:** coordination scope шире инварианта. **Исправление:** найти минимальный key/aggregate, внутри которого операции конфликтуют.

## Когда применять

Сначала перечисляют события и causal edges, затем формулируют инвариант. Если достаточно не показывать следствие раньше причины, нужен causal contract. Если конфликт ограничен одним aggregate, выбирают per-key order. Если разные keys совместно защищают один ресурс, требуется общий order/transaction domain либо переработанная commutative модель.

В telemetry сохраняют wall time вместе с trace/span relation, message ID, parent event, partition/offset или log index. Это не создаёт отсутствующую причинность, зато позволяет восстановить известные edges. При тестировании задерживают и переупорядочивают независимые сообщения, повторяют delivery и проверяют, что consumer не полагается на случайный arrival order.
