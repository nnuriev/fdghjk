# M.Tech — Go на 350к

## TL;DR

Запись объединяет опыт кандидата, широкий теоретический screening и две практические задачи. Первая задача проверяет конкурентную выдачу Outbox-записей нескольким worker pod’ам, lease, повторную обработку после падения и семантику at-least-once. Вторая представляет собой упрощённый System Design автоматизированной парковки: въезд, учёт вместимости, тарификация, отдельный платёжный терминал, льготное окно для выезда и повторная оплата.

Кандидат уверенно обсуждает знакомые производственные инструменты, но в фундаментальных вопросах смешивает соседние понятия. Особенно заметны ошибки в распределении Kafka partitions, stack/heap, window functions, partitioning и replication. В первой задаче решение уходит в центральный dispatcher и таблицу pod’ов, хотя условие естественно решается короткой транзакцией с `FOR UPDATE SKIP LOCKED` и сохраняемым lease. Во второй задаче появляется базовый happy path, но не оформляются state machine, идемпотентность платежа и подтверждение физического выезда.

### Наблюдаемые ошибки и точные поправки

- Кандидат определяет нормализацию как «снижение числа отношений за счёт дублирования». Это описание ближе к денормализации. Нормализация декомпозирует данные по зависимостям, чтобы уменьшить update/insert/delete anomalies; цена — дополнительные joins и более сложные read paths.
- Durability связывается только с replication. На уровне PostgreSQL она опирается на WAL, flush policy и recovery; replication расширяет failure tolerance, но не подменяет локальный commit contract.
- Названный «range index» по описанию похож на BRIN. BRIN хранит summaries ranges физических pages и выигрывает, когда значение коррелирует с physical order. Он не возвращает агрегатную функцию из индекса и не требует, чтобы колонка называлась временем.
- Window function ошибочно описана как функция, сворачивающая набор строк в одно значение. Aggregate без window действительно уменьшает число rows; window function вычисляет значение для каждой output row по её partition/frame.
- Pessimistic locking и optimistic version check объяснены близко к правильной модели. Для optimistic update нужен atomic predicate вроде `WHERE id=$1 AND version=$old`, проверка affected row count и явная retry/conflict policy.
- Partitioning смешивается с read/write replicas. Replication копирует одни данные; partitioning делит logical set; sharding размещает части по независимым узлам. Эти границы подробно зафиксированы в [[30 Данные/Партиционирование и шардирование|заметке о partitioning и sharding]] и [[30 Данные/Репликация данных|репликации]].

# Задача 1 — Outbox workers

## Нормализованное условие — `00:46:01–01:06:07`

Есть PostgreSQL-таблица с `id` и `payload`. Несколько одинаковых worker pod’ов периодически забирают batch записей, выполняют внешний effect и затем подтверждают обработку либо удаляют запись.

Нужно обеспечить два инварианта:

1. Два workers, одновременно запросившие batch, не получают одну запись в рамках одной попытки.
2. Если worker получил запись и умер до подтверждения, запись через ограниченное время снова доступна другому worker.

Claim и acknowledgement выполняются в разных транзакциях. Нельзя держать DB transaction открытой во время сетевой публикации или долгой обработки.

## Ход решения кандидата

Сначала предлагаются статусы `new/processing/done`, затем global counter с последним взятым ID. После вопроса о полном падении datacenter кандидат видит, что counter перескакивает необработанные rows и не даёт понять, когда их вернуть.

Дальше решение усложняется таблицей назначений `message → pod`, отдельной таблицей состояния pod’ов и центральным worker, который распределяет batches и переназначает их после timeout. Temporal idea появляется: хранить timestamp назначения и после истечения интервала отдавать работу другому pod. Но схема не доходит до атомарного SQL claim, добавляет coordinator как новую точку отказа и связывает correctness с эфемерным Kubernetes pod identity.

## Ментальная модель

Row проходит через повторяемый lease protocol:

```text
available → leased(token, until) → processed
              │
              └── until истёк → available для следующей попытки
```

Lease не утверждает, что старый worker умер. Он утверждает более узкое: после `locked_until` этот worker теряет право единолично подтверждать attempt. Поэтому claim выдаёт уникальный `lease_token`, а acknowledgement проверяет token. Это fencing: запоздавший worker не может закрыть уже переназначенную работу.

## Минимальная схема

```sql
CREATE TABLE outbox (
    id               uuid PRIMARY KEY,
    payload          jsonb NOT NULL,
    created_at       timestamptz NOT NULL DEFAULT now(),
    next_attempt_at  timestamptz NOT NULL DEFAULT now(),
    attempts         integer NOT NULL DEFAULT 0,
    locked_by        text,
    lease_token      uuid,
    locked_until     timestamptz,
    processed_at     timestamptz,
    last_error       text
);

CREATE INDEX outbox_ready_idx
    ON outbox (next_attempt_at, id)
    WHERE processed_at IS NULL;
```

`locked_by` полезен для диагностики, но correctness опирается на transaction, row lock, `locked_until` и `lease_token`. Хранить отдельную таблицу Kubernetes pods для базового claim не требуется.

## Atomic claim batch

В одной короткой транзакции worker выбирает незаблокированные ready rows и сразу записывает lease:

```sql
WITH picked AS (
    SELECT id
    FROM outbox
    WHERE processed_at IS NULL
      AND next_attempt_at <= now()
      AND (locked_until IS NULL OR locked_until < now())
    ORDER BY id
    FOR UPDATE SKIP LOCKED
    LIMIT $1
)
UPDATE outbox AS o
SET locked_by = $2,
    lease_token = $3,
    locked_until = now() + $4::interval,
    attempts = o.attempts + 1
FROM picked
WHERE o.id = picked.id
RETURNING o.id, o.payload, o.lease_token, o.attempts;
```

`SKIP LOCKED` намеренно даёт inconsistent subset и потому не подходит для обычного предметного чтения. Для queue-like table это нужное поведение: второй worker не ждёт rows первого, а берёт другие. После `COMMIT` row locks освобождены, а право попытки продолжает сохранённый lease.

Внешний effect выполняется вне транзакции. При успехе acknowledgement использует fencing token:

```sql
UPDATE outbox
SET processed_at = now(),
    locked_by = NULL,
    lease_token = NULL,
    locked_until = NULL
WHERE id = $1
  AND lease_token = $2
  AND processed_at IS NULL;
```

Ноль affected rows означает, что attempt уже потерял lease или запись завершена. Такой worker не имеет права молча считать acknowledgement успешным.

## Failure modes и семантика доставки

- Worker умер до внешнего effect: lease истечёт, row заберут повторно.
- Worker выполнил effect и умер до `processed_at`: row будет повторён. Получается at-least-once; consumer или внешний API должен дедуплицировать stable `event_id`.
- Worker завис дольше lease: heartbeat продлевает lease только пока worker ещё владеет `lease_token`, либо lease выбирают по bounded processing time. Слишком длинный lease замедляет recovery, слишком короткий создаёт лишние concurrent attempts.
- Постоянная ошибка: `attempts`, exponential backoff через `next_attempt_at`, max attempts и DLQ/manual review. Бесконечный tight retry превращает одну bad row в hot loop.
- Cleanup: processed rows удаляются или архивируются после retention, но не до того, как audit/reconciliation больше не нужны.
- Fairness: стабильный `ORDER BY` нужен, однако старые постоянно падающие rows не должны навсегда блокировать новые; backoff участвует в ordering.

## Что проверяет задача

Это не задача на знание одного SQL keyword. Интервьюер проверяет, видит ли кандидат границу транзакции, различает ли row lock и сохраняемый lease, умеет ли доказать retry после crash и честно назвать at-least-once. Центральный dispatcher допустим при дополнительных требованиях к scheduling, но здесь он увеличивает число moving parts без необходимости.

# Задача 2 — автоматизированная парковка

## Нормализованное условие — `01:06:07–01:26:10`

Есть участок с ограниченным числом мест, въездным и выездным шлагбаумами. Ручного оператора нужно заменить системой, которая:

- понимает, можно ли впустить машину;
- выдаёт ticket и фиксирует время въезда;
- не превышает вместимость;
- рассчитывает стоимость по тарифу;
- принимает наличную или карточную оплату в отдельном kiosk;
- даёт ограниченное время до выезда после оплаты;
- требует доплату, если grace period истёк;
- освобождает вместимость после фактического выезда.

В устном примере первые `15 минут` бесплатны, далее действует цена `200 рублей` за начатый час. Для стоянки `4 часа 40 минут` участники приходят к `1000 рублям`, но это не математическая гарантия из формулировки, а выбранная политика округления. На реальном проектировании правило нужно записать явно.

## Ход решения кандидата

Кандидат начинает с сервиса парковочных мест, ticket ID и `entered_at`. Для парковки без закреплённых spots он предлагает хранить только число свободных мест. На выезде добавляет billing/payment и QR либо физический kiosk, затем понимает, что оплата и barrier находятся в разных точках. Появляются `paid` status и payment timestamp.

Обсуждение grace period несколько раз меняет правило. Сначала любое превышение `15 минут` после оплаты означает доплату; затем замечено, что клиент мог уже оплатить округлённый час. ## Сначала контракт, потом сервисы

Перед diagram стоит уточнить:

- одна парковка или сеть lots;
- места свободные или закреплённые;
- ticket, license plate или оба идентификатора;
- тариф округляется вверх, поминутно или по календарным интервалам;
- free period и grace period — разные правила;
- допустима ли offline-работа устройств;
- требуется ли real-time display свободных мест;
- кто обрабатывает lost ticket, chargeback и ручное открытие.

Для одной парковки разумный старт — modular service и одна transactional DB. Разнос на микросервисы не улучшает correctness сам по себе; он добавит network failures в critical path шлагбаума.

## Минимальная модель данных

```text
ParkingLot(
  id, capacity, occupied, version
)

ParkingSession(
  id, lot_id, ticket_token,
  entered_at, exited_at,
  state,
  tariff_version,
  paid_through,
  exit_deadline,
  version
)

Payment(
  id, session_id, idempotency_key,
  amount, currency,
  provider_payment_id,
  status, created_at, confirmed_at
)

GateCommand(
  command_id, gate_id, session_id,
  kind, status, created_at, acknowledged_at
)
```

Если назначаются конкретные spots, вместо одного `occupied` появляется `ParkingSpot(id, lot_id, state, session_id, version)`. Для свободной рассадки достаточно счётчика, но понадобится reconciliation с entry/exit sensors: аппаратный сбой иначе навсегда создаст drift.

Главный capacity invariant:

```text
0 <= occupied <= capacity
```

Проверка и увеличение `occupied` должны быть одной atomic DB operation, например conditional `UPDATE ... WHERE occupied < capacity RETURNING ...`, в той же transaction, где создаётся session. Отдельные `SELECT free` и последующий `UPDATE` допускают два въезда на последнее место.

## State machine

```text
admitted → parked → payment_pending → paid → exit_requested → exited
                       │                 │
                       └→ payment_failed └→ grace_expired → payment_pending
```

State names могут отличаться, но переходы обязаны быть monotonic и идемпотентными. Payment success не равен физическому выезду; slot освобождается после exit sensor/подтверждённого crossing. Иначе оплативший и оставшийся внутри автомобиль увеличит фактическую загрузку выше счётчика.

## Сквозной happy path

1. Entry device создаёт `request_id` и просит admission.
2. Service одной transaction условно увеличивает `occupied` и создаёт `ParkingSession` с `entered_at`, `tariff_version` и случайным signed `ticket_token`.
3. После commit создаётся idempotent `GateCommand(open)`. Sensor подтверждает crossing; отсутствие подтверждения уходит в reconciliation, а не в слепое повторное создание session.
4. В kiosk ticket находит active session. Pricing вычисляет сумму по сохранённой tariff version.
5. Payment создаётся с idempotency key. Повтор после timeout возвращает существующий outcome, а unknown result сверяется с provider.
6. После confirmed payment session получает `paid_through` и `exit_deadline`. Эти поля фиксируют продуктовую политику лучше, чем один `paid_at`.
7. Exit gate проверяет session, создаёт idempotent open command и ждёт sensor crossing.
8. На подтверждённом crossing session становится `exited`, `exited_at` фиксируется, `occupied` уменьшается один раз.

Если машина приезжает после `exit_deadline`, pricing считает только новый unpaid interval относительно `paid_through`. Новый Payment имеет новый business key, но повтор одной попытки использует тот же idempotency key. Такой protocol не берёт второй раз деньги за уже покрытый период.

## Failure modes

- Последнее свободное место одновременно запрашивают два въезда: conditional update разрешает один admission.
- Ticket напечатан, barrier не открылся: session остаётся наблюдаемой; operator/reconciliation либо повторяет тот же command, либо отменяет admission и возвращает capacity.
- Barrier открылся, sensor не подтвердил въезд: нельзя бесконечно печатать новые tickets; нужен device event log и ручное разрешение ambiguous outcome.
- Provider списал деньги, ответ потерялся: новый charge не создаётся, пока status не сверится по idempotency/provider ID.
- Kiosk и exit gate видят разные состояния: payment и session transition фиксируются transactionally либо через надёжный event/outbox protocol.
- Exit command повторён: `command_id` делает device operation идемпотентной.
- Машина заплатила и не уехала: session переходит в `grace_expired`; capacity всё ещё занята.
- Clock drift устройств: authoritative time выдаёт backend; timestamp terminal’а не должен единолично определять деньги.
- Ticket потерян или скопирован: нужен recovery по license plate/operator и защита от двух active exits одной session.
- История растёт без границы: retention/архивирование допустимы только после закрытия финансовой сверки и audit window.

## Trade-offs

| Решение | Выигрыш | Цена и граница |
| --- | --- | --- |
| Общий `occupied` counter | Простая admission transaction | Не показывает конкретные spots, требует reconciliation с sensors |
| Таблица spots | Назначение места и точная диагностика | Больше contention и state transitions |
| Modular monolith | Простая transaction boundary для одной парковки | Ограниченная независимость масштабирования компонентов |
| Microservices | Отдельный lifecycle pricing/payment/device domains | Distributed consistency и больше failure paths у шлагбаума |
| Ticket token | Работает без account и телефона | Потеря/копирование требует recovery и fraud policy |
| License plate recognition | Удобный бесконтактный flow | Ошибки распознавания, privacy, ручной fallback |

## Что проверяет задача

Интервьюер оценивает не число нарисованных сервисов. Сильный сигнал — последовательное превращение физического процесса в invariants и state transitions: кто подтверждает въезд, где резервируется capacity, что означает «оплачено», когда можно выпустить автомобиль и как восстановиться после ambiguous hardware/payment outcome.

# Наблюдаемая оценка кандидата

## Сильные стороны

- Есть опыт с Go, PostgreSQL, Redis, Kafka, OpenAPI, Docker и Kubernetes; ответы опираются на реальные кейсы, а не только определения.
- В cache-кейсе замечен concurrent miss и применён `singleflight`.
- Кандидат знает основной набор concurrency primitives, race detector, tri-color GC и optimistic version check.
- В Outbox-задаче после подсказок появляется timeout/lease idea.
- В парковке быстро выделены session, capacity, ticket, pricing и payment; замечено физическое разделение kiosk и barrier.
- Кандидат не маскирует отсутствие опыта с нагрузочными тестами или тонкой настройкой Kubernetes.

## Зоны риска

- Несколько фундаментальных ответов звучат уверенно, но неверно: Kafka partition assignment, stack/heap, normalization, window functions и partitioning/replication.
- Называются primitives и продукты без точного invariant, который они защищают.
- В Outbox-задаче долго отсутствует atomic claim SQL; coordinator и pod registry добавляются раньше, чем доказана необходимость.
- Не формулируется at-least-once и duplicate window между effect и acknowledgement.
- В парковке happy path рисуется раньше требований; payment, gate command и physical crossing сначала склеены в одно событие.
- Tariff rounding и grace period меняются по ходу обсуждения без зафиксированного контракта.
- Failure handling, idempotency и reconciliation появляются только после прямых вопросов.

Telegram-публикация сообщает отказ, но запись не содержит recruiter feedback. Поэтому нельзя доказанно связать решение компании с одной конкретной ошибкой или блоком.

# План подготовки по этой записи

1. За пять минут исправить вслух пять границ: Kafka partitions, stack/heap, aggregate/window, replication/partitioning и `panic/recover`.
2. Написать claim batch через `FOR UPDATE SKIP LOCKED`, lease token и `UPDATE ... RETURNING`; затем разыграть crash до effect, после effect и после acknowledgement.
