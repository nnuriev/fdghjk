# VK Tech — собеседование Golang на 350к

## TL;DR

Это русскоязычное техническое интервью продолжительностью `01:22:14`: знакомство с продуктом электронного документооборота, короткий разговор об опыте и выборе Go, две задачи на чтение кода, code review конкурентного счётчика, блок по базам данных и индексам PostgreSQL, затем получасовая low-level/system-design задача про библиотеку. Итоговый бланк содержит `402` отмеченных пункта: заданные вопросы, задачи, раскрытия и вопросы кандидата. В подписи публикации указан оффер на `350к`.

Интервью проверяет не алгоритмы, а способность последовательно выводить поведение из инвариантов. В Go-секции это `nil` против empty slice, общий backing array, `len/cap`, значение указателя при передаче в функцию, nil map, data race, unsigned underflow и то, что `++` в Go является statement. В секции данных интервьюер идёт от Redis/PostgreSQL и hash collisions к выбору B-tree/Hash, составным индексам и `EXPLAIN`. В design-секции исходные пять операций превращаются в границы трёх сервисов, модель `title/edition/copy`, конкурентную аренду физического экземпляра, электронное бронирование по ISBN, синхронные RPC и асинхронную аналитику.

Кандидат хорошо рассуждает вслух, принимает подсказки и в основном приходит к ожидаемому направлению. Он верно объясняет aliasing slices, исправляет pointer-задачу через `*p = newP`, замечает гонку и unsigned underflow, различает equality/range use cases и собирает два составных индекса. Сильнее всего выглядит практический PostgreSQL-опыт: `EXPLAIN ANALYZE`, поиск `Seq Scan`, `tsvector` и GIN.

Пробелы проявляются на точности. В code review не сразу замечена неинициализированная map; первая версия блокировки оставляет check-then-act race и не защищает чтение; `sync.Map` предлагается как почти механическая замена, хотя compound increment/decrement от этого не становятся атомарными. Redis объявляется быстрее PostgreSQL по одной только асимптотике, бинарный Protocol Buffers — «сжатым», а gRPC — безусловно более производительным. В design-задаче не сформулированы атомарный захват экземпляра, idempotency, TTL брони, источник истины и сценарий частичного отказа между сервисами. Судя по заявленному офферу, эти пробелы не стали блокирующими; официальный feedback в записи отсутствует.

# Точный разбор экранных Go-задач

## Задача 1: полный код и ответ

Код на экране, дополненный package/import только для запуска:

```go
package main

import "fmt"

func main() {
	a := []int{0, 1, 2, 3}
	b := a

	b[1] = b[len(b)-1]
	b[len(b)-1] = 0
	b = b[:len(b)-1]
	b[0] = 4

	fmt.Println(a, b)
	fmt.Println(len(a), len(b), cap(a), cap(b))

	b = append(b, 7)
	fmt.Println(a, b)
}
```

Результат:

```text
[4 3 2 0] [4 3 2]
4 3 4 4
[4 3 2 7] [4 3 2 7]
```

Причинная цепочка:

1. `b := a` копирует slice descriptor, но оба descriptors указывают на один backing array.
2. Записи через `b` меняют этот массив, поэтому их видит `a`.
3. `b = b[:3]` меняет только `len(b)`; capacity остаётся `4`.
4. `append(b, 7)` требует длину `4`, она помещается в capacity `4`.
5. Элемент записывается в старый backing array по индексу `3`, где `a` раньше видел `0`.

Неверное предположение «после reslice хвост удалён» даёт симптом: ожидание нового массива или неизменного `a`. Причина — `len` ограничивает доступный view, но не уничтожает backing storage. Исправление, если aliasing нежелателен: перед `append` ограничить capacity `b = b[:len(b):len(b)]` либо сделать явную копию.

## Задача 2: почему `p = &newP` не работает

Pointer value, как и остальные аргументы, передаётся по значению. В `changePointer` параметр `p` — отдельная переменная с копией адреса `foo`. Присваивание `p = &newP` заменяет эту локальную копию; `bar` в `main` не меняется. `*p = newP` идёт по адресу и записывает `3` в сам `foo`.

`**int` понадобился бы, если контракт требует изменить именно pointer variable вызывающей стороны. Для текущей задачи это лишняя indirection: нужно изменить число, а не перенаправить `bar`.

## Code review: исправленная реализация

```go
package main

import "sync"

type BurgerCounter struct {
	mu      sync.RWMutex
	workers map[string]uint64
}

func NewBurgerCounter() *BurgerCounter {
	return &BurgerCounter{workers: make(map[string]uint64)}
}

func (c *BurgerCounter) Make(workerName string) uint64 {
	c.mu.Lock()
	defer c.mu.Unlock()

	c.workers[workerName]++
	return c.workers[workerName]
}

func (c *BurgerCounter) Reject(workerName string) (uint64, bool) {
	c.mu.Lock()
	defer c.mu.Unlock()

	count := c.workers[workerName]
	if count == 0 {
		return 0, false
	}

	count--
	c.workers[workerName] = count
	return count, true
}

func (c *BurgerCounter) Count(workerName string) uint64 {
	c.mu.RLock()
	defer c.mu.RUnlock()

	return c.workers[workerName]
}
```

Инвариант `count >= 0` проверяется и изменяется внутри одной critical section. Все обращения к map синхронизированы, включая чтение. Constructor гарантирует non-nil map. `sync.Map` здесь не улучшает модель: даже если lookup безопасен, условный decrement остаётся compound operation. Для очень высокой конкуренции можно исследовать sharding или `atomic.Uint64` с CAS loop, но это следующий trade-off, а не базовое исправление.

# Production-grade решение design-задачи

## Scope и ментальная модель

Одна и та же фраза «взять книгу» скрывает два разных command:

- `ReserveEdition(reader_id, isbn, idempotency_key)` — пользователь выбирает издание в электронном каталоге, система атомарно закрепляет любой доступный физический copy на ограниченное время;
- `CheckoutCopy(reader_id, copy_id, idempotency_key)` — сотрудник сканирует конкретный экземпляр и переводит reservation/copy в активную аренду.

Минимальные сущности:

| Сущность | Идентификатор | Назначение |
| --- | --- | --- |
| Reader | `reader_id` | Участник, eligibility, quota, status |
| Edition | `isbn`/`edition_id` | Название, автор, конкретное издание |
| Copy | `copy_id` | Физический экземпляр, филиал, condition, availability state |
| Reservation | `reservation_id` | Временное закрепление copy, `expires_at` |
| Rental | `rental_id` | Активная/завершённая аренда, `due_at`, `returned_at` |

ISBN не заменяет `copy_id`: несколько физических экземпляров одного издания имеют один ISBN, но могут одновременно быть свободны, арендованы, утеряны или находиться в разных филиалах.

## Граница консистентности

Рабочий baseline устроен так: Catalog/Inventory владеет `edition` и `copy`, а Rental capability — allocation/reservation/rental в той же PostgreSQL transaction boundary. Reader profile можно держать отдельно, но quota/eligibility, которые обязаны соблюдаться атомарно при checkout, должны иметь локально проверяемое представление или reservation protocol.

Критический invariant:

```text
Для одного copy_id существует не более одной строки
со статусом reserved или active.
```

Его нельзя оставлять только приложению. Один вариант PostgreSQL:

```sql
CREATE UNIQUE INDEX one_open_rental_per_copy
ON rentals (copy_id)
WHERE status IN ('reserved', 'active');
```

Выбор экземпляра должен происходить в transaction, например через подходящий indexed query с `FOR UPDATE SKIP LOCKED`, затем `INSERT` reservation. Unique partial index остаётся последней защитой от double allocation. При конфликте command получает детерминированный business result и может выбрать другой copy или вернуть `no_availability`.

## Основной поток бронирования

1. API аутентифицирует caller и принимает `reader_id`, `isbn`, филиал и idempotency key.
2. Rental capability находит предыдущий результат по idempotency key; повтор возвращает тот же `reservation_id`.
3. В transaction блокируется reader quota row или используется другой атомарный quota mechanism.
4. Выбирается один доступный `copy` нужного edition/branch с row lock.
5. Создаётся `reserved` row с `expires_at`; unique partial index запрещает double booking.
6. В той же transaction записывается outbox event.
7. После commit клиент получает reservation и expiry; publisher доставляет событие независимо от request.
8. Idempotent worker переводит просроченную reservation в `expired`, если checkout ещё не выполнен.

Checkout переводит `reserved → active`, фиксирует `rent_at` и вычисляет `due_at` из server-side policy. Return переводит `active → returned` и задаёт `returned_at`. Extension атомарно проверяет лимит продлений, overdue и отсутствие следующей reservation.

## Границы сервисов и trade-off

Три deployable services не обязательны. Для небольшой нагрузки модульный монолит с модулями `readers`, `catalog`, `rentals` сохраняет те же domain boundaries, но оставляет критический invariant внутри одной database transaction. Выделение сервисов оправдано, если появились отдельные команды, независимый scaling, разные release cadence или security boundaries.

Если Catalog и Rental физически разделены, synchronous `IsAvailable(copy)` не защищает от гонки. Нужна command `ReserveCopy` у владельца availability с атомарным state transition, а не read-check из чужого сервиса. При частичном отказе повтор идёт с тем же idempotency key; Saga/compensation освобождает reservation по expiry.

## Сильные стороны

- не заучивает только финальный ответ, а показывает ход мысли и быстро корректируется;
- уверенно держит aliasing, `len/cap` и pointer semantics;
- имеет практический опыт PostgreSQL diagnostics и full-text search;
- различает equality и range access patterns;
- в design-задаче постепенно отделяет relation аренды от readers/catalog;
- задаёт зрелые вопросы про ownership feature, качество требований, support и командные процессы.

## Пробелы и likely decision signals

- базовые дефекты кода ищутся не системно: compile → zero value → invariant → concurrency → API;
- `sync.Map` и mutex появляются раньше точного определения compound operation;
- database performance объясняется асимптотикой без выравнивания semantics и измерений;
- design flow строится как последовательность remote reads, а не как защищаемый invariant;
- ISBN, edition и physical copy разделяются только после подсказки;
- gRPC получает ярлык «быстрее», Protocol Buffers — «сжатие»;
- нет явных idempotency, timeout, retry и partial-failure semantics.

Заявленный оффер согласуется с профилем Middle+/Senior-: хорошие fundamentals, практический database опыт, сильная коммуникация и обучаемость при неполной глубине distributed invariants. Это inference по записи, а не официальный hiring feedback VK Tech.

# Как готовиться именно к такому интервью

1. Для каждого container начинать с zero value, ownership/aliasing, mutation и concurrency contract.
2. В code review идти порядком: компилируется ли → инициализировано ли → сохраняется ли invariant → есть ли race → какой API выражает failure.
3. На database comparison сначала уравнять semantics, затем обсуждать access path, cache/network и только потом complexity.
4. Composite indexes рисовать как ordered keys и проверять каждый predicate по leftmost prefix, затем оговаривать planner/skip scan.
5. В design-задаче различать title/edition/copy и писать один главный invariant до списка сервисов.
6. Любой read-check-write превращать в вопрос: какая transaction/constraint/owner делает его атомарным?
7. Для gRPC разводить RPC model, gRPC framework, HTTP/2 transport и protobuf encoding.
8. Для async path сразу называть event identity, delivery semantics, idempotent consumer, outbox и lag monitoring.
