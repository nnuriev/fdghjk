### Технически точная модель

В одной consumer group каждая partition назначается ровно одному member, поэтому при четырёх partitions пятый consumer не добавляет parallelism. Consumers из другой group получают собственное логическое чтение тех же records. Consumer держит текущую position в процессе, а durable group progress по стандартному механизму commit попадает через coordinator в compacted topic `__consumer_offsets`. Именно поэтому фраза «offset хранится у consumer» неполна: локально есть position/cache, но восстановление group после restart опирается на committed state.

### Корректная mental model scheduler

`G` хранит состояние goroutine, `M` соответствует OS thread, а `P` — runtime resource/context, необходимый M для выполнения user Go code. Число P определяется `GOMAXPROCS`. Runnable work чаще берётся из local P queue; scheduler периодически обращается к global queue, проверяет netpoll/timers и крадёт work у других P. При syscall P может отделиться от заблокированного M и продолжить выполнение на другом M.

Интервьюер верно вспомнил конкретный механизм fairness: в исходниках Go `1.26.5` `findRunnable` проверяет global queue при `pp.schedtick%61 == 0`, потому что две goroutines иначе могут постоянно respawn друг друга в local queue. Это полезная деталь исходного кода, но её нельзя обещать как стабильный API или language semantics.

### Сильные стороны

- Привязывает технологии к реальной industrial domain, а не перечисляет стек.
- Объясняет local autonomy фабрик и failure tolerance при потере центральной связи.

- Понимает topic/partition, per-partition ordering и idle consumers при избытке members.
- Различает REST и gRPC на уровне contracts/use cases, хотя определение REST могло быть точнее.
- Уверенно воспроизводит G/M/P, local/global queues, work stealing, netpoll и `sysmon`.
- Знает tri-color abstraction, write barrier и короткие STW transitions.
- Правильно отвечает на поведение `runtime.GOMAXPROCS(n)` при `n < 1`.
- Знает mutex fast path/slow path, основные channel states и runtime `hchan`.
- Верно называет основные ACID properties, isolation и index families.
- Предлагает production observability: Prometheus/Grafana, ELK, pprof, query analysis.
- Задаёт содержательные вопросы о legacy, доступах, команде и stored procedures.

### Пробелы и опасные формулировки

- Перепутал названия at-most-once/at-least-once/exactly-once.
- Exactly-once объяснил главным образом дополнительными ресурсами, не зафиксировав scope и protocol participants.
- Смешал consumer position и durable committed offsets.
- Kafka/RabbitMQ свёл к лозунгу про smart/dumb broker.
- REST описал преимущественно через HTTP methods/JSON, а не constraints и uniform resource interface.
- Во время вопроса о gRPC искал определение в браузере.
- P местами описывал как processor/core, хотя это runtime execution context.
- Scheduler implementation details подавал почти как стабильный контракт.
- Stack/heap вывел из fixed size/type rules вместо context-sensitive escape analysis.
- Отключение GC обсуждал без `GOMEMLIMIT` и soft-limit semantics.
- Oversized `GOMAXPROCS` объяснил «зависшими P с очередями» вместо scheduling/throttling costs.
- Mutex назвал полностью user-space, хотя slow path опирается на runtime semaphores/OS support.
- `UPDATE` назвал буквально `DELETE+INSERT`, не оговорив tuple-version mental model и HOT.
- Autovacuum смешал с ручным обслуживанием/backup.
- Race detector предложил для поиска memory leaks.
- Cache предложил раньше доказательства database bottleneck.
- Паттерны перечислял по именам, не всегда раскрывая invariant и failure recovery.

### Как отвечать сильнее

Для этого интервью полезен каркас из пяти шагов:

1. Зафиксировать scope и invariant.
2. Объяснить mechanism.
3. Проиграть failure sequence.
4. Назвать version/implementation boundary.
5. Показать production measurement.

Например: «Внутри одной Kafka consumer group partition назначается одному member; поэтому consumers сверх числа partitions idle. Current position живёт у consumer, а стандартный durable commit хранится через group coordinator в compacted `__consumer_offsets`. Commit до side effect рискует потерей, после — повтором, поэтому destination должна принимать stable idempotency key или offset и result в одной transaction».

Для Go scheduler: «P — runtime context, не CPU core. На Go `1.26.5` `findRunnable` раз в 61 tick проверяет global queue ради fairness, затем учитывает local/global work, netpoll, timers и stealing. Число 61 — деталь конкретного runtime source; production tuning делаю по latency/profile/load test, а не по этому числу».

Для memory incident: «Сначала разделяю RSS, runtime-managed memory, live heap и allocation rate. Снимаю heap/alloc profiles и goroutine stacks, проверяю cgroup/OOM, boundedness cache/map/queues. Race detector отвечает на другой вопрос — conflicting unsynchronized accesses».

## Эволюция и версии

| Версия | Изменение | Практический эффект |
| --- | --- | --- |
| Go `1.18` | roots вошли в формулу GC work для `GOGC` | приложения с большим числом goroutine stacks получили более точный heap target |
| Go `1.19` | добавлен soft runtime memory limit через `GOMEMLIMIT`/`debug.SetMemoryLimit` | можно ограничивать runtime-managed memory независимо от обычного `GOGC` target, оставляя headroom контейнеру |
| Go `1.25` | Linux default `GOMAXPROCS` стал учитывать cgroup CPU bandwidth limit и периодически обновляться | container workload реже получает лишнее throttling из-за host CPU count; manual override отключает automatic behavior |
| Go `1.26.5` | в `findRunnable` сохраняется fairness-проверка global queue при `schedtick%61==0` | полезно для понимания текущей реализации, но число не является language/API guarantee |
| PostgreSQL `18` | актуальная поддерживаемая ветка на дату проверки | isolation, WAL, MVCC, index families, HOT и vacuum ниже сверены именно с документацией 18 |
| Apache Kafka `4.1` | использован versioned client contract consumer groups/offsets | assignment и distinction position/committed offset привязаны к документации 4.1 |

Технические утверждения проверены для Go `1.26.5`, PostgreSQL `18`, Apache Kafka client documentation `4.1` и RabbitMQ `4.1` на `2026-07-18`.
