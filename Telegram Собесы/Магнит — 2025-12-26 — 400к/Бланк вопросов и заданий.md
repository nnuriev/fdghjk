# Магнит — собеседование Golang на 400к

## TL;DR

Это русскоязычное техническое интервью продолжительностью `01:46:47`: опыт кандидата, две задачи с кодом, Go core/runtime, PostgreSQL и Kafka. В подписи публикации указан результат «отказ». Запись относится к команде «Магнит Аптеки» внутри «Магнит ИТ»; один интервьюер ведёт основной опрос, второй задаёт уточнения по PostgreSQL и Kafka.

Практические задачи простые по формулировке, но показательные по исполнению. Слияние двух отсортированных `[]int` потребовало примерно 25 минут и множества прямых подсказок, хотя итоговый two-pointer algorithm получился корректным. Fan-in произвольного числа каналов кандидат решил существенно увереннее: одна forwarding goroutine на вход, `WaitGroup` и отдельный closer для результата. Однако в production-варианте не хватает cancellation: если downstream перестанет читать, workers могут утечь.

Слабое впечатление складывается не из одного незнакомого факта, а из повторяющегося паттерна. Кандидат часто начинает отвечать до того, как построил модель: строку называет Unicode symbols, heap — binary tree, `atomic` связывает со stop-the-world, `Serializable` — с буквальным последовательным выполнением, а Go 1.24 Swiss map — с группами по 16 и SIMD. Часть направления мысли верна, но детали либо не подтверждают тезис, либо прямо ему противоречат.

Сильнее выглядят end-to-end ownership по последнему проекту, итоговая конструкция fan-in, общая идея двух реализаций map, практическое использование `sync.Pool`, перечисление способов масштабирования PostgreSQL и знакомство с Inbox/Outbox. Слабее — самостоятельное решение базового алгоритма, precise semantics Go, различие stack/heap, фазы GC, multicolumn indexes, PostgreSQL isolation и независимость consumer groups Kafka.

Здесь особенно полезен видеоряд. На экране полностью видны условия и финальный код обеих задач; примеры подтверждают, что дубликаты при merge нужно сохранить. Во время PostgreSQL-блока кандидат открывает собственную Notion-страницу с заметками: она помогает понять, откуда взялись формулировки, но не доказывает их правильность.

# Практические задания и эталонный ход решения

## Merge: точная реконструкция условия

На экране записано:

```text
Смерджить 2 упорядоченных массива в 1 упорядоченный массив.

Дано:      [1, 2, 3, 4, 5], [4, 5, 6, 7, 8]
Результат: [1, 2, 3, 4, 4, 5, 5, 6, 7, 8]

func merge(a []int, b []int) []int
```

Из примера следуют два важных invariants: оба входа уже отсортированы по возрастанию, а одинаковые значения сохраняются с multiplicity. Произвольные длины интервьюер подтвердил устно.

### Модель решения

Держим `i` и `j` на первых ещё не перенесённых элементах. Пока оба индекса валидны, переносим меньший элемент; при равенстве можно переносить левый, потому что правый останется и попадёт на следующей итерации. Когда один input закончился, его сравнивать больше не с чем, поэтому целиком добавляем уже отсортированный хвост второго.

```go
func mergeSorted(a, b []int) []int {
	result := make([]int, 0, len(a)+len(b))
	i, j := 0, 0
	for i < len(a) && j < len(b) {
		if a[i] <= b[j] {
			result = append(result, a[i])
			i++
		} else {
			result = append(result, b[j])
			j++
		}
	}
	result = append(result, a[i:]...)
	result = append(result, b[j:]...)
	return result
}
```

Время — `O(len(a)+len(b))`: каждый элемент сравнивается и переносится не более константного числа раз. Дополнительная память результата — `O(len(a)+len(b))`. Предвыделение capacity не меняет асимптотику, но убирает промежуточные роста backing array.

### Что сделал кандидат

Финальный код корректен: три comparison branches, при равенстве append обоих элементов, затем два хвоста. Ветка равенства чуть сложнее необходимого, но сохраняет дубликаты. Проблема не в финальном коде, а в пути к нему: сначала map, затем идея `O(n²)`, затем более десяти минут на понимание двух indices и ещё примерно десять на boundaries. Интервьюеры фактически провели кандидата по алгоритму вопросами «что поставить в результат?» и «какой указатель сдвинуть?».

Отдельная ошибка в устной оценке: кандидат назвал время и память просто `O(n)`, не определив `n`, и предположил, что `make([]int, 0, len(a)+len(b))` меняет time complexity. Точная формулировка: `O(n+m)` времени, `O(n+m)` результата; preallocation уменьшает число allocations/copies, но не превращает линейный алгоритм в константный.

## Fan-in: точная реконструкция условия

На экране записано:

```go
// Смерджить произвольное количество каналов в 1 канал.
func merge(chList ...chan int) chan int
```

Тестовая заготовка создаёт два buffered channels capacity `2`, пишет в каждый `1` и `2`, закрывает, вызывает `merge(ch1, ch2)` и читает `for range` из результата. Значит output обязан закрыться только после завершения обоих inputs.

### Базовая модель решения из интервью

Итоговый вариант кандидата соответствует каноническому fan-in:

```go
func merge(chList ...chan int) chan int {
	out := make(chan int)
	var wg sync.WaitGroup

	for _, ch := range chList {
		wg.Add(1)
		go func(ch chan int) {
			defer wg.Done()
			for value := range ch {
				out <- value
			}
		}(ch)
	}

	go func() {
		wg.Wait()
		close(out)
	}()
	return out
}
```

`Add` выполняется до старта worker, а output закрывает только closer после `Wait`. Это и есть ключевой ownership invariant: ни один sender не может послать после close.

### Production-добавка: cancellation

Базовый вариант предполагает, что consumer дочитает output и каждый producer когда-нибудь закроет input. Если consumer уйдёт раньше, workers заблокируются на `out <- value`; если input никогда не закроется, `Wait` не завершится. Поэтому библиотечному варианту нужен cancellation, причём select должен покрывать и receive, и send. Проверенная generic-реализация лежит в [verified_tasks.go](Данные/verified_tasks.go).

Обсуждение loop capture было полезным, но historical boundary кандидат не знал. Начиная с language version Go 1.22, variables, объявленные loop clause, создаются заново для каждой iteration. Явная передача `ch` parameter остаётся хорошей: она документирует captured value и не зависит от чтения старого кода или версии модуля.

## Калибровка результата

По записи отказ понятен. Даже если не требовать знания runtime source, для Go backend уровня `400к` ожидаются самостоятельное линейное merge, точные channel edge cases, уверенное различие bytes/runes, нормальная модель stack/heap и прикладная PostgreSQL isolation. Здесь ошибки образуют цепочку: candidate угадывает термин, но не может вывести поведение из механизма.

При этом это не «полный провал». Fan-in решён, опыт не выдуманно общий, часть production patterns знакома. Ближайшая точка роста — не заучивать ещё больше названий. Нужно на каждой теме отвечать одинаковым каркасом: contract → mechanism → observable behavior → failure mode → trade-off. Тогда незнакомую деталь можно честно ограничить, не заполняя пробел ложной уверенностью.

# Модель хорошего ответа на ключевые темы

## Merge

«Два входа уже отсортированы, поэтому держу по индексу на первый необработанный элемент. Переношу меньший и двигаю только его индекс; когда один input закончится, дописываю хвост другого. Дубликаты сохраняю, как требует пример. Время `O(n+m)`, result memory `O(n+m)`. Сразу выделю capacity `n+m`, чтобы избежать промежуточных growth/copy».

## Fan-in

«Запускаю по forwarding goroutine на input, а их завершение считаю `WaitGroup`. Только отдельный closer после `Wait` закрывает output — это исключает send-after-close. Возвращаю receive-only channel. Если consumer может остановиться раньше или inputs ненадёжны, добавляю context и проверяю cancellation и на receive, и на send, иначе возможна goroutine leak».

## Slice

«Slice — descriptor над backing array: pointer, length и capacity в текущей реализации. Index — `O(1)`. Append без роста — `O(1)`, при growth — allocation и `O(n)` copy, поэтому amortized `O(1)`. Коэффициент growth — implementation detail; в Go 1.26.5 threshold в `nextslicecap` равен `256`, но production code не должен от него зависеть. Копирование descriptor сохраняет aliasing, а независимые данные требуют clone/copy».

## Strings

«String — immutable sequence of bytes и не обязан быть valid UTF-8. `len` и index работают с bytes. Rune — Unicode code point, alias `int32`; UTF-8 encoding valid rune занимает 1–4 bytes. `range` декодирует UTF-8 и даёт byte offsets, но даже rune не всегда равна пользовательскому символу: grapheme может состоять из нескольких code points».

## Stack и heap

«Stack goroutine начинается маленьким и растёт: runtime переносит used frames в больший contiguous block. Stack overflow — это превышение limit, а не первое исчерпание 2 KiB. Stack/heap placement решает escape analysis; возврат pointer часто заставляет value escape, но inlining/caller могут изменить решение. Heap здесь — область динамических allocations, а не binary heap data structure».

## PostgreSQL composite index

«Сначала фиксирую реальные queries: predicates, sort, projection и update rate. Для B-tree равенства на leading columns и range на первой следующей колонке ограничивают scan. Порядок условий в SQL не важен сам по себе; selectivity — один из inputs, но не универсальное правило. Проверяю candidates через `EXPLAIN (ANALYZE, BUFFERS)` на representative data. `INCLUDE` добавляю только для нужного index-only scan и учитываю visibility map/write amplification».

## Kafka consumer groups

«В classic group каждая partition назначена одному consumer; если consumers больше, лишние idle, если меньше — consumer читает несколько partitions. Порядок есть внутри partition, поэтому entity key должен стабильно попадать в неё. Разные groups имеют независимые offsets и lag; медленная group не двигает position другой, хотя может косвенно конкурировать за broker/network resources. Для external DB обещаю at-least-once плюс idempotent effect, если не могу атомарно связать effect и offset».

# Что говорит интервью о формате Магнита

1. Live coding занимает почти 40 минут из 1:46 и сильно влияет на итог: ценится не только финальный код, но самостоятельность движения.
2. Go спрашивают широко и с углублением в implementation versions: loop variables, old/new map, stack growth и GC phases.
3. Интервьюеры проверяют причинную модель повторными «почему?»; заученное определение без механизма быстро раскрывается.
4. PostgreSQL занимает примерно 14 минут и требует не списка терминов, а выбора index/isolation/lock под запрос.
5. Kafka-блок короткий, но проверяет partition assignment, processing guarantees и независимость groups.
6. Вопросы кандидата оставлены на последние четыре минуты, поэтому самые важные стоит приоритизировать заранее.
