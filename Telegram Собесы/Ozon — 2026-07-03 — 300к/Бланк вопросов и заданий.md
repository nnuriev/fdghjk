## TL;DR

Запись содержит знакомство с продуктом, разбор опыта, короткий технический этап и отдельное двухчасовое coding interview. Всего восстановлено девять самостоятельных заданий: `append` и общий backing array, `defer` с value/pointer receiver и closure, монотонный массив, открытый вопрос о масштабировании сервиса, конкурентная проверка URL с остановкой после двух успехов, потокобезопасный in-memory cache, декодирование вложенной строки, Top K frequent elements и разворот односвязного списка.

Кандидат уверенно решает базовые алгоритмические задачи после подсказок и пишет рабочий каркас cache. Основные пробелы проявляются не в синтаксисе, а в причинном объяснении: граница переаллокации `append`, момент вычисления receiver у `defer`, владение каналом, deadlock при `Wait` до чтения, точная остановка конкурентных запросов и методика выбора способа масштабирования. В задаче про `append` первоначальный ответ был неверным; в HTTP-задаче несколько проблем исправлены интервьюерами по ходу обсуждения.

Во время демонстрации экрана рядом с редактором почти постоянно открыт ChatGPT с загруженными скриншотами условий и готовыми ответами. В задаче `decodeString` поверх него появляется Telegram-сообщение с почти полной реализацией. Поэтому финальную корректность кода нельзя автоматически считать независимым решением кандидата. Это наблюдение по кадрам, а не вывод о причине заявленного отказа.

# Подробный разбор заданий

## 1. Общий backing array у slices — `00:22:32–00:31:01`

### Условие с экрана

```go
func main() {
    nums := []int{1, 2, 3}

    addNum(nums[0:2])
    fmt.Println(nums) // ?

    addNums(nums[0:2])
    fmt.Println(nums) // ?
}

func addNum(nums []int) {
    nums = append(nums, 4)
}

func addNums(nums []int) {
    nums = append(nums, 5, 6)
}
```

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/01-slices-condition-00-22-32.jpg]]

### Правильный результат

```text
[1 2 4]
[1 2 4]
```

`nums[0:2]` имеет `len=2`, `cap=3`. Первый `append` увеличивает длину до трёх, помещается в capacity и записывает `4` в третий элемент общего backing array. Локальный slice header не возвращается вызывающему коду, но запись в общий массив остаётся видимой.

Второй `append` должен получить длину четыре. Capacity исходного subslice равна трём, поэтому одна операция `append(nums, 5, 6)` сначала выбирает новый backing array и уже туда записывает оба значения. Она не обязана частично записывать `5` в старый массив перед переаллокацией. Внешний `nums` остаётся `[1 2 4]`.

Чтобы отсечь spare capacity, достаточно передать full slice expression `nums[0:2:2]`. Тогда `cap=2`, и любой непустой `append` выделит другой backing array. Механизм подробно разобран в [[60 Go/Массивы и слайсы|заметке о массивах и slices]].

### Ответ кандидата

Кандидат правильно объясняет общий массив после первого вызова, но сначала ожидает `[1 2 5]` после второго. Такой же неверный второй результат виден в ответе ChatGPT слева от условия. После вопроса интервьюера кандидат замечает, что итоговая длина четыре превышает capacity три, и исправляет ответ. Full slice expression предлагает открытая внешняя подсказка; считать этот follow-up самостоятельным ответом по записи нельзя.

## 2. `defer`: value receiver, pointer receiver и closure — `00:31:01–00:37:13`

### Условие с экрана

```go
type X struct {
    V int
}

func (x X) S() {
    fmt.Println(x.V)
}

func main() {
    x := X{123}
    defer x.S()
    x.V = 456
}
```

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/03-defer-condition-00-31-40.jpg]]

### Правильная модель

Результат — `123`. При выполнении `defer` вычисляются function value, receiver и аргументы. Value receiver содержит копию `x` с `V=123`; отложен сам вызов этой уже подготовленной функции. Несколько deferred calls выполняются в обратном порядке регистрации.

Если метод изменить на `func (x *X) S()`, сохраняется указатель на ту же переменную, и при фактическом вызове метод прочитает `456`. Для `defer func() { x.S() }()` внешний `defer` сохраняет closure, но `x.S()` выполняется только внутри closure во время возврата; value receiver будет скопирован в тот момент и тоже напечатает `456`.

Фраза «`defer` замораживает всё» опасна: она не объясняет разницу между параметром deferred call и переменной, которую closure читает позже. Точная семантика связана с [[60 Go/defer, panic и recover|моментом вычисления deferred call]] и [[60 Go/Функции, function values и замыкания|захватом переменных closure]].

### Ответ кандидата

Первый результат и LIFO кандидат называет правильно. Объяснение несколько раз сводится к «`defer` не видит изменения», из-за чего pointer receiver и closure приходится разбирать с интервьюером. На кадре виден параллельно открытый ответ ChatGPT с тем же различием.

## 3. Монотонный массив — `00:37:13–00:46:14`

### Условие с экрана

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/05-monotonic-condition-00-39-10.jpg]]

### Сильное решение

За один проход поддерживаются два инварианта: все уже просмотренные соседние пары допускают неубывание и все допускают невозрастание. Для каждой пары `a[i-1], a[i]`:

- если `a[i] < a[i-1]`, неубывание больше невозможно;
- если `a[i] > a[i-1]`, невозрастание больше невозможно;
- равенство не ломает ни одно направление;
- если оба флага стали `false`, можно завершиться раньше.

Время — `O(n)`, дополнительная память — `O(1)`. Пустой массив и массив из одного элемента монотонны vacuously: нет соседней пары, нарушающей условие. Альтернатива — пропустить равный префикс, установить направление по первой неравной паре и затем проверять один инвариант. Она использует тот же asymptotic cost, но требует аккуратнее обработать массив из равных элементов.

### Ответ кандидата

Кандидат выбирает корректную идею с `nonDecreasing` и `nonIncreasing`. В редакторе некоторое время перепутаны имена и условия, но итоговая логика после обсуждения соответствует задаче. Саму идею и полный вариант кода одновременно показывает ChatGPT, поэтому надёжно оценивать самостоятельность реализации нельзя.

## 4. Масштабирование сервиса B — `00:46:14–00:53:32`

### Условие с экрана

> Есть два сервиса A и B. Сервис A — внешний, влиять на него нельзя. Сервис B полностью находится в вашей ответственности. Бизнесовая читающая и пишущая нагрузка на B растёт вместе с бизнесом. Какие варианты и направления масштабирования, балансировки и оптимизации B существуют? Можно использовать любые технологии.

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/07-scaling-condition-00-46-40.jpg]]

### Сильная структура ответа

1. Сначала определить SLI и форму нагрузки: read/write ratio, RPS недостаточно без распределения стоимости запросов; нужны latency по квантилям, errors, saturation, CPU, memory, queue depth, connection pools и downstream latency.
2. Найти bottleneck профилированием и трассировкой: приложение, сериализация, network, lock contention, GC, база или внешний A. Масштабирование не исправляет дорогой запрос, если bottleneck остаётся в общей базе или A.
3. Снизить стоимость критического пути: алгоритм, batching, правильный контракт, connection reuse, query plan, ограниченная выборка данных.
4. Vertical scaling быстро покупает запас, но имеет предел экземпляра и может увеличить blast radius. Horizontal scaling требует stateless processing либо явного распределения state, load balancer, readiness и контролируемого rollout.
5. Cache полезен для повторных чтений при допустимой stale window. Нужны key design, TTL/invalidation, защита от stampede и план на cache miss.
6. Async queue сглаживает пики и отделяет принятие работы от исполнения, но добавляет lag, retry, duplicate delivery, ordering и backpressure.
7. База оптимизируется по реальным запросам: query plans, индексы, connection pool, read replicas, partitioning или sharding только после доказанного bottleneck. Индекс ускоряет подходящие чтения ценой write amplification, памяти и обслуживания.
8. На границе с A нужны deadlines, bounded concurrency, rate limiting, retry budget, circuit breaking и degraded mode. Нельзя позволить медленному A заполнить все goroutines и соединения B.
9. Перед ростом capacity задаётся overload behavior: bounded queues, load shedding и приоритеты. Затем решение подтверждается load test и наблюдением четырёх сигналов: latency, traffic, errors, saturation.

### Ответ кандидата

Кандидат называет load balancer, vertical/horizontal scaling, CPU, memory, RPS, cache, async и индексы. Начальная идея «балансировщик между A и B» не определяет, между какими replicas распределяется входящий трафик и почему это снимает bottleneck. Выбор horizontal/vertical объясняется в основном наличием денег и достижением потолка машины, а не типом state, bottleneck и SLO. В части базы он вспоминает selectivity, но не связывает индекс с конкретным predicate, порядком колонок, write cost и query plan.

## 5. HTTP: от последовательного цикла до отмены — `00:55:08–01:26:51`

### Базовое условие

Для каждого URL нужно сделать GET. Статус `200` печатается как `url - ok`; ошибка запроса или любой другой status code — как `url - not ok`.

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/08-http-condition-00-55-50.jpg]]

### Resource lifetime

При `err == nil` caller обязан закрыть `resp.Body`. Если `defer resp.Body.Close()` регистрируется прямо внутри длинного цикла `main`, все body остаются открыты до завершения `main`. Исправление — вынести одну итерацию в helper и использовать `defer` там либо закрывать body явно перед следующей итерацией.

Для повторного использования HTTP/1.x keep-alive соединения документация `net/http` требует прочитать body до EOF и закрыть его. Если тело потенциально велико и содержимое не нужно, безусловное полное чтение тоже опасно; выбирают ограниченный контракт ответа, timeout/limit либо осознанно отказываются от reuse. Эти связи подробно разобраны в [[60 Go/HTTP-клиент и Transport|заметке об HTTP client и Transport]].

### Channel lifecycle

У канала результатов должен быть один владелец закрытия. Рабочие goroutines отправляют значения, `main` читает. Отдельная coordinator goroutine выполняет `wg.Wait()` и затем `close(results)`. Тогда consumer может безопасно делать `for result := range results`.

Схема `wg.Wait()` в consumer перед чтением хрупка: worker может заблокироваться на send, не дойти до `Done`, а consumer ждёт `Wait`. Большой buffered channel лишь скрывает цикл ожидания до тех пор, пока число отправок не превысит capacity. Схема `defer close(results)` в `main` тоже не помогает: deferred close выполнится только после возврата, а `range` ждёт закрытия, чтобы этот возврат стал возможен.

### Остановка после двух успехов

Самая простая модель — один владелец счётчика: `main` принимает результаты, считает только успешные, после второго вызывает `cancel()` и больше не печатает success. Worker создаёт запрос через `http.NewRequestWithContext(ctx, ...)` и отправляет результат через:

```go
select {
case results <- result:
case <-ctx.Done():
}
```

Coordinator всё равно ждёт workers и закрывает канал. `cancel()` подаёт сигнал, но не ждёт физического завершения goroutines. Одновременно успевшие запросы могут завершиться успешно; требование «вывести ровно не больше двух» обеспечивает consumer, а не надежда, что сеть мгновенно остановится.

Atomic counter нужен только если владение счётчиком оставляют workers. `atomic.AddInt32` возвращает новое значение, но несколько goroutines могут получить `2`, `3`, `4`; отмена на `2` не откатывает уже завершённую работу. Для выдачи только двух permits потребуется CAS-loop либо отдельный bounded coordination channel. В этой задаче single-owner consumer проще и лучше объясняет инвариант.

Для большого списка нужен concurrency limit: без него программа одновременно создаёт столько goroutines и сетевых операций, сколько URL пришло на вход. Отмена и безопасная отправка защищают от [[60 Go/Goroutine и channel leaks|утечек goroutines]], но не заменяют backpressure.

### Ответ кандидата

Последовательный вариант кандидат пишет быстро. `defer` внутри цикла оставляет до подсказки интервьюера. В параллельном варианте использует `WaitGroup` и buffered channel, затем ставит `wg.Wait()` перед `range`, а закрытие канала откладывает до выхода из `main`; интервьюеры показывают deadlock/circular wait и подводят к coordinator goroutine.

Для остановки после двух успехов кандидат сначала предлагает `context` и atomic counter, но не передаёт context в `http.Get`. Затем инкрементирует counter в workers и отменяет при равенстве двум; интервьюеры отдельно показывают, что третья и четвёртая goroutines могут успеть отправить `ok`. Итоговая правка фильтрует значения по возвращённому `Add`, однако более ясная архитектура со счётчиком в consumer была озвучена интервьюером раньше и не реализована кандидатом.

## 6. Потокобезопасный in-memory cache — `01:26:51–01:35:05`

### Условие с экрана

```go
type Cache interface {
    Set(k, v string)
    Get(k string) (v string, ok bool)
}
```

Память считается бесконечной, удалять ключи не требуется.

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/11-cache-condition-01-27-30.jpg]]

### Сильное решение и trade-off

Базовая структура — `map[string]string` и lock. `Get` под `RLock`, `Set` под `Lock`; конструктор инициализирует map. Это linearizable относительно одной process-local структуры: каждый завершившийся `Set` виден последующему `Get`, если вызовы синхронизированы через тот же lock.

`RWMutex` не является автоматическим ускорением: при коротких операциях и низкой конкуренции обычный `Mutex` может быть дешевле. Выбор проверяют benchmark с реальным read/write ratio. `sync.Map` специализирован для write-once/read-many либо операций над непересекающимися группами ключей; обычная map с lock даёт type safety и позволяет атомарно поддерживать дополнительные инварианты. В production-задаче далее появились бы TTL, eviction, memory budget, stampede, метрики и lifecycle, но условие специально исключает эти части. Более широкий дизайн есть в [[50 Проектирование систем/Проектирование и реализация in-memory cache в Go|разборе in-memory cache]].

### Ответ кандидата

Кандидат реализует `map[string]string`, `sync.RWMutex`, `RLock/RUnlock` для чтения и `Lock/Unlock` для записи. Это соответствует условию. В обсуждении он верно говорит о параллельных readers, но характеристику `sync.Map` вспоминает неточно. На экране открыта готовая реализация ChatGPT ещё до завершения собственного кода.

## 7. Decode string — `01:35:05–01:55:59`

### Условие с экрана

Формат — `k[encoded_text]`, вложенность допустима, скобки корректны. Примеры:

```text
3[a]2[bc]      -> aaabcbc
3[a2[c]]       -> accaccacc
2[abc]3[cd]ef  -> abcabccdcdcdef
```

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/14-decode-condition-code-01-55-59.jpg]]

### Ментальная модель

При `[` текущая строка и число повторений образуют stack frame; текущая строка сбрасывается для вложенного фрагмента. При `]` frame снимается, вложенный фрагмент повторяется `k` раз и добавляется к предыдущему префиксу. Для многозначного числа используется `num = num*10 + digit`.

Нижняя граница времени уже равна размеру результата: декодированную строку надо материализовать. При аккуратном накоплении оценка — `O(len(input) + len(output))`, память — `O(depth + len(output))`. Наивные `result += piece` могут многократно копировать растущий префикс; `strings.Builder` либо стек builders уменьшает лишние перераспределения, но не отменяет стоимость самого output.

### Ответ кандидата и внешняя подсказка

Кандидат выбирает два stack: числа и предыдущие строки, и после исправления синтаксиса получает правильные ответы на трёх примерах. Интервьюер разрешает упрощение до однозначного `k`, хотя исходный формат этого явно не гарантирует.

В `01:40:00` поверх ChatGPT появляется Telegram-окно от «Никита Рулик» с почти полной реализацией `decodeString`: `numStack`, `strStack`, разбор цифры, обработка `[` и `]`. Это не скрытое предположение, а видимый элемент screen share:

## 8. Top K frequent elements — `01:55:59–02:08:36`

### Условие и решение

Нужно вернуть `k` значений с наибольшей частотой. Кандидат выбирает bucket grouping:

1. `freq[value]++` считает частоты;
2. `buckets[count]` хранит все значения с этой частотой;
3. buckets просматриваются от `len(nums)` к единице;
4. значения добавляются, пока не набрано `k`.

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/15-top-k-code-02-01-40.jpg]]

Частота не может превышать число элементов, поэтому диапазон buckets ограничен `0..n`. Время — `O(n)`, память — `O(n)`. Если целый bucket пересекает границу `k`, можно добавить его и вернуть `result[:k]`, но только после валидации `k` и при условии, что произвольный порядок ties разрешён. Порядок обхода Go map не определён, поэтому deterministic tie-break требует отдельного правила и обычно сортировки внутри группы либо другой структуры.

Эта задача не совпадает с [[Авито/Решения/Алгоритмы/Максимальные элементы в массиве — Top K|Top K максимальных значений]]: там ключ — величина элемента, здесь — его частота. Heap-решение за `O(n + u log k)` может быть практичнее, если диапазон частот не хочется материализовать или данные идут потоком; bucket-вариант выигрывает на статическом массиве с ограничением частоты `n`.

### Ответ кандидата

Основная идея верна. Кандидат долго выясняет, почему нужен `len(nums)+1`, и несколько раз путает индекс bucket с нулевой индексацией массива. Тест запускается после синтаксических исправлений. Границы `k`, ties и nondeterministic map iteration не обсуждаются.

## 9. Разворот односвязного списка — `02:08:36–02:20:41`

### Условие и инвариант

Для каждого `curr`:

1. сохранить `next := curr.Next`, иначе хвост будет потерян;
2. развернуть ребро `curr.Next = prev`;
3. сдвинуть `prev = curr`;
4. сдвинуть `curr = next`.

После каждой итерации `prev` — голова уже развёрнутого префикса, а `curr` — голова ещё не обработанного суффикса. Когда `curr == nil`, `prev` становится новой головой. Время `O(n)`, память `O(1)`. Пустой список возвращает `nil`, один узел — самого себя. Та же pointer-модель рассматривается в [[10 Основы CS/Связные списки, стек, очередь и двусторонняя очередь|заметке об односвязных списках]].

![[90 Вложения/Telegram Собесы/Ozon — 2026-07-03 — 300к/Кадры/16-list-condition-02-08-36.jpg]]

### Ответ кандидата

Сам reverse-loop кандидат восстанавливает правильно. После этого тратит заметное время не на алгоритм, а на создание примера списка и печать всей цепочки: первоначально пытается вывести только новую голову и не продвигает указатель в проверочном цикле. Это хороший пример различия между знанием шаблона и умением самостоятельно построить минимальный проверяемый тест.

## Общая оценка кандидата

### Что получилось

- Опыт BFF и fan-out интеграций изложен связно; кандидат отличает critical и optional downstream dependencies.
- В монотонном массиве, Top K frequent и reverse list выбраны подходящие `O(n)` идеи.
- Cache реализован минимально и по условию: типизированная map плюс `RWMutex`.
- После наводящих вопросов кандидат способен исправить ошибку и довести код до запуска.
- Он не скрывает использование AI и на словах правильно ограничивает доверие к сгенерированному коду.

### Где проявились пробелы

- Механизм часто заменяется запомненной формулой: «defer замораживает», «если денег много — vertical», «selective колонка — хороший индекс».
- Не сразу выделяется владелец concurrent state. Из-за этого lifecycle канала, `Wait`, close, counter и cancellation рассматриваются по отдельности, а не как один протокол.
- Успешный happy path формулируется раньше failure modes: body lifetime, blocked send, более двух одновременных success, invalid `k`, tie order.
- Заявленные метрики из опыта не защищены способом измерения: benchmark gRPC/REST и `30%` TTM остаются без baseline и confounders.
- Финальный код нескольких задач появляется при постоянно открытом ChatGPT; в `decodeString` видна ещё и готовая Telegram-подсказка. Поэтому скорость и полноту нельзя использовать как чистую оценку навыка.

### Уровень, который фактически проверяет запись

Набор хорошо различает кандидата, который знает популярные решения, и инженера, способного вывести поведение из инвариантов. Для уверенного Senior-ответа недостаточно назвать `RWMutex`, horizontal scaling или context: нужно связать primitive с ownership, resource lifetime, overload behavior и доказуемым условием завершения.

Заявленный отказ совместим с наблюдаемыми пробелами, но запись не доказывает причинную связь. Решение могло учитывать другие сигналы, неизвестные из видео.

Задачи не являются копиями комплекта Авито. Пересечение есть на уровне фундаментальных тем Go, concurrency и Top K, но конкретные условия и follow-up существенно иные. Ozon сильнее концентрируется на коротких coding patterns и постепенном усложнении одной HTTP-задачи; разбор Авито сочетает алгоритмический блок с отдельной Go-platform задачей.

## Как готовиться именно к такому формату

1. Для каждого Go primitive проговаривать не определение, а ownership и момент эффекта: кто владеет памятью, каналом, lock, body и cancel.
2. В coding-задаче сначала вслух назвать инвариант и failure modes, затем писать код.
3. Для concurrency отдельно доказать termination: кто закрывает канал, кто ждёт workers, что разблокирует send и что происходит после cancellation.
4. Для System Design начинать с SLI, workload и bottleneck; список технологий без условий выбора не демонстрирует инженерного решения.
5. После алгоритма проверять пустой вход, один элемент, invalid parameters, ties, overflow и размер результата.
6. Если используется AI, уметь независимо объяснить каждую строку и заметить его ошибку. В этой записи ChatGPT дал неверный второй результат в первой же задаче, что хорошо показывает риск слепого копирования.
