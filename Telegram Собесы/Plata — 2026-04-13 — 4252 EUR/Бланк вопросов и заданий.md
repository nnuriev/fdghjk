### Точное условие

```text
На входе есть отсортированные по возрастанию массивы чисел:
- suspects (подозреваемые)
- innocents (невиновные)

Необходимо из массива подозреваемых исключить всех невиновных.

Input:  suspects=[1,2,3,4,5], innocents=[2,4]
Output: [1,3,5]

Input:  suspects=[3,80,123,421,936], innocents=[80,936]
Output: [3,123,421]
```

Интервьюер отдельно подтвердил, что duplicates нет. Поэтому set semantics однозначна и можно использовать два указателя без отдельного счётчика частот.

### Решение кандидата

```go
func filter(suspects, innocents []int) []int {
	excluded := make(map[int]struct{}, len(innocents))
	for i := range innocents {
		excluded[innocents[i]] = struct{}{}
	}

	result := make([]int, 0, len(suspects))
	for i := range suspects {
		if _, exists := excluded[suspects[i]]; exists {
			continue
		}
		result = append(result, suspects[i])
	}
	return result
}
```

Решение корректно: `O(n+m)` expected time, `O(m)` auxiliary map space и до `O(n)` output. Capacity `len(suspects)` исключает growth allocations, но удерживает место под худший результат даже когда почти все элементы будут отброшены. Это trade-off между предсказуемым числом allocations и peak/retained memory.

### Решение, использующее sorted invariant

```go
func filterSorted(suspects, innocents []int) []int {
	result := make([]int, 0, len(suspects))
	j := 0

	for _, suspect := range suspects {
		for j < len(innocents) && innocents[j] < suspect {
			j++
		}
		if j < len(innocents) && innocents[j] == suspect {
			continue
		}
		result = append(result, suspect)
	}
	return result
}
```

Каждый pointer движется только вперёд, поэтому суммарное число сравнений линейно. Дополнительная структура на `m` элементов не нужна. Однако это решение корректно только при доказанной сортировке. Если вход произвольный, map-версия проще и надёжнее, чем предварительная сортировка, которая к тому же меняет порядок или требует copies.

Обе реализации и сценарий с duplicates проверены через официальный Go Playground на Go `1.26.5`; сохранённый вывод: `[1 3 5]`, `[1 3 5]`, `[1 3]`.

### Где кандидат ошибся в `append`

Кандидат правильно сказал, что результат `append` нужно сохранить, но объяснил это так, будто операция всегда создаёт копию и новую ссылку. Контракт другой: `append` возвращает slice с новым length; если прежней capacity хватает, он использует тот же backing array, иначе выделяет новый. Именно поэтому результат обязателен к присваиванию даже при reuse: исходная локальная копия descriptor по-прежнему содержит старый length.

Все значения в Go передаются по value. Для slice копируется descriptor, а не elements. Несколько descriptors могут указывать на один backing array. Поэтому `s[0] = 42` обычно видно caller, а локальный `s = append(s, 42)` сам по себе не меняет descriptor caller.

### Где кандидат ошибся в map keys

Сильная часть ответа — правило comparability и возможность использовать comparable struct. Ошибка — кандидат запретил channel. Channel values сравнимы: равны, если созданы одним `make` либо оба `nil`, поэтому `map[chan int]V` допустима. Не сравнимы slices, maps и functions; последние можно сравнивать только с `nil`, но не друг с другом.

`struct{}{}` имеет нулевой размер как value, однако фраза «значение в map бесплатно» слишком сильна. Общая память map зависит от implementation, metadata, load factor и layout групп. Начиная с Go 1.24 standard toolchain использует Swiss Tables, но это не меняет semantic contract map и не даёт права выводить точную стоимость entry без measurement на конкретной версии.

### Исходный код с экрана

```go
func main() {
	c := make(chan int, 1000)
	for i := 0; i < 100; i++ {
		go foo(c)
	}

	sum := 0
	for r := range c {
		sum += r
	}
	fmt.Println(sum)
}

func foo(c chan int) {
	r := rand.Int()
	for i := 0; i < r; i++ {
		c <- r
	}
}
```

Ключевой bug — у потока нет completion signal. `range` ждёт следующее значение до закрытия channel. Формально loops producers конечны, но каждый `r` может быть огромным, поэтому программа ещё и непрактична по объёму работы. После завершения всех producers main всё равно останется ждать receive, потому что channel открыт.

### Почему первый fix кандидата тоже блокировался

Кандидат добавил `WaitGroup`, затем поставил `wg.Wait(); close(c)` перед consumer loop. Если producers суммарно посылают больше capacity, они блокируются на send. Main ждёт producers; consumer ещё не запущен и не освобождает slots. Получается cycle ожидания.

Правильная схема:

```go
var wg sync.WaitGroup
values := make(chan int, 1000)

for i := 0; i < 100; i++ {
	wg.Go(func() {
		foo(values)
	})
}

go func() {
	wg.Wait()
	close(values)
}()

sum := 0
for value := range values {
	sum += value
}
```

Consumer работает одновременно с producers. Отдельная goroutine закрывает channel только после того, как все senders завершились. `range` сначала получает buffered values, затем выходит. Capacity теперь влияет на throughput и backpressure, но не является условием завершения.

`WaitGroup.Go` добавлен в Go `1.25`; на более ранних toolchains нужен `Add`/`Done`. Документация отдельно требует, чтобы переданная функция не panic-овала.

### Переполнение: важное исправление интервью

Интервьюер предположил panic при переполнении `sum`, кандидат предложил `math/big`. В Go обычное signed integer overflow в runtime arithmetic не вызывает panic. Результат остаётся значением signed integer и может стать отрицательным; compiler не вправе считать, что overflow невозможен. Поэтому:

- если допустима машинная арифметика по модулю разрядности, `int` технически продолжит работу;
- если overflow должен обнаруживаться, нужна явная checked arithmetic;
- если нужна точная сумма произвольного размера, подходит `math/big.Int`.

Finite deterministic equivalent с buffer `2` и `big.Int` проверен через официальный Go Playground на Go `1.26.5`; результат для repetitions `[3,2,1]` равен `4`.

### Повторный перевод: boundary неопределённого результата

Проблема возникает ещё до broker: client не знает, применил ли server первый request. Сильный contract:

1. client создаёт idempotency key один раз на logical transfer и повторяет его при transport retry;
2. server в transaction вставляет operation с unique key либо читает уже существующую;
3. money movement и operation state связаны одной consistency boundary;
4. повтор возвращает тот же committed result, а concurrent duplicate сериализуется constraint/lock;
5. downstream events несут stable operation/event ID, а consumers также идемпотентны.

Кандидат предложил event ID и deduplication consumer-ом — направление правильное, но оно не полностью закрывает HTTP/API boundary. Если два client requests успели создать две разные payment operations и два разных events, consumer уже не знает, что пользователь имел в виду один перевод. Нужен stable key логической операции до создания side effect.

### Delivery labels без магии

- At-most-once: сообщение/operation может потеряться, но повторная обработка исключается выбранным порядком acknowledgement.
- At-least-once: сообщение не теряется при оговорённой failure model, но может быть обработано повторно.
- Exactly-once: state transition наблюдается один раз только внутри явно описанного scope и при участии destination state.

Kafka transactions дают exactly-once read-process-write между Kafka topics, если offsets и output записываются одной transaction и consumers читают committed data. Для внешней PostgreSQL side effect нужна совместная граница: idempotent consumer с transaction и dedupe key, хранение offset рядом с output либо другой согласованный protocol.

### Transactional Outbox

Business change и outbox row записываются одной database transaction. Relay читает committed outbox rows и публикует в broker; при crash он может повторить publish, поэтому delivery обычно at-least-once. Event имеет stable ID, а consumer deduplicates или выполняет idempotent update. После подтверждённой публикации row помечается/удаляется согласно retention policy.

Кандидат восстановил этот механизм по сценарию, хотя сначала не был уверен в названии. Интервьюер отдельно отметил это как сильную прикладную часть ответа.

### Как отвечать сильнее

Для большинства вопросов здесь работает один каркас:

1. назвать invariant и scope;
2. описать mechanism;
3. показать failure sequence;
4. назвать trade-off и version boundary;
5. предложить measurement/test.

Например: «Channel закрывает сторона, которая знает, что sends завершены. `Wait` нельзя ставить перед draining, иначе заполненный buffer заблокирует producers. Поэтому отдельная goroutine ждёт producers и закрывает stream, а caller одновременно читает до close. `WaitGroup.Go` доступен с Go 1.25; раньше использую `Add`/`Done`. Проверяю с buffer 0/1, race detector и cancellation test».

Для медленного endpoint: «Сначала по trace доказываю слой задержки; затем смотрю pool/locks/query stats; только после этого `EXPLAIN (ANALYZE, BUFFERS)` и index design». Для повторного перевода: «Idempotency key рождается на client boundary один раз на logical operation, server фиксирует его unique constraint в той же transaction, а events и consumers сохраняют stable identity дальше по pipeline».

## Эволюция и версии

| Версия | Изменение | Практический эффект |
| --- | --- | --- |
| Go `1.18` | появился predeclared constraint `comparable` вместе с generics | type parameter можно ограничить типами с допустимыми equality operations |
| Go `1.20` | PGO появился как preview | build принимает CPU pprof через `-pgo` |
| Go `1.21` | PGO стал production-ready и `default.pgo` автоматически используется при build | production profile можно включить в обычный repeatable build flow |
| Go `1.22` | `range` по integer и per-iteration variables для соответствующей language version | проверенный test использует `for range repeat`; closure capture стал безопаснее в новых modules |
| Go `1.24` | standard built-in map перешёл на Swiss Tables | performance/layout изменились, но language semantics map не изменились |
| Go `1.25` | добавлен `sync.WaitGroup.Go` | код кандидата не компилируется на более раннем toolchain без `Add`/`Done` |
| Go `1.25` | Linux default `GOMAXPROCS` стал container-aware и обновляемым | manual `automaxprocs` часто больше не нужен при стандартных cgroup CPU limits, но manual override отключает automatic behavior |
| PostgreSQL `18` | native generation UUIDv4/UUIDv7 документирована в core | UUIDv7 можно выбирать ради time ordering/locality, сохраняя 128-bit distributed ID trade-offs |

Технические утверждения проверены для Go `1.26.5`, PostgreSQL `18`, Apache Kafka documentation `4.1` и Debezium `3.6` на `2026-07-18`.
