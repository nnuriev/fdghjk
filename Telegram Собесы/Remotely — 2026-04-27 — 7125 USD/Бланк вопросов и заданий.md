### Что можно восстановить о challenge

Задание было full-stack и выполнялось до встречи. В backend существовала candidate feature с HTTP handlers, business logic и database access; ожидалось возможное расширение другими features, например HR/matching. На frontend был React code с data fetching через `useEffect`. Больше требований запись не содержит, поэтому выдумывать domain model, endpoints или acceptance criteria нельзя.

Кандидат объясняет feature-first layout будущим ростом: candidate functionality находится в одном vertical slice, а database details отделены от logic. Он сознательно не копирует крупную reference structure целиком, потому что для ограниченного challenge множество layers и interfaces не окупаются. Если появится сложная логика, которую нужно тестировать изолированно, handlers можно оставить transport adapters, а behavior вынести в services за узкими interfaces.

### Оценка ответа

Это сильная часть интервью. Кандидат связывает structure не с модой, а с ожидаемым изменением и стоимостью тестирования. Хороший инвариант здесь: HTTP handler отвечает за transport contract, business code — за decision logic, repository/adapter — за внешнее состояние. Но physical folders не обязаны повторять эти понятия один к одному. Interface нужен у consumer в месте реальной вариативности или test seam; interface «на каждый struct» увеличивает indirection без нового contract.

Не хватило одного сквозного примера: какая зависимость была оставлена concrete, какой change потребовал бы interface и какой test стал бы проще. Тем не менее интервьюер называет backend solution `10/10`, то есть выбранная глубина соответствовала ожидаемому scope.

### Testing: где ответ просел

Кандидат говорит, что на frontend обычно мало logic, проверка присутствия component кажется ему странной, а внешний вид он чаще оценивает глазами. Тесты он считает полезными главным образом для calculations. Это слишком узкая модель.

Ценность frontend test определяется не количеством arithmetic, а риском пользовательского behavior. Полезные границы:

- unit test для pure transformation и validation;
- component/integration test для формы, permissions, loading/error/empty states, keyboard interaction и accessible name/role;
- browser/E2E test для критического journey через routing, API и real browser behavior;
- visual regression только там, где layout сам является contract.

Тест «component существует» действительно слаб, если проверяет implementation detail. Но запрос кнопки по accessible role/name, ввод данных и проверка наблюдаемого результата моделируют действие пользователя и одновременно ловят часть accessibility regressions. Ручной просмотр остаётся exploratory check, а не воспроизводимая защита от regression.

### `useEffect`, stale response и cancellation

Кандидат правильно понимает симптом: response старого page request может прийти позже и затереть новое state. `isMounted`/локальный `ignore` flag в cleanup защищает state update, но не отменяет network work. Для `fetch` можно создать `AbortController`, передать `signal` и вызвать `abort()` в cleanup:

```javascript
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/items?page=${page}`, { signal: controller.signal })
    .then(response => response.json())
    .then(items => setItems(items))
    .catch(error => {
      if (error.name !== "AbortError") throw error;
    });

  return () => controller.abort();
}, [page]);
```

Код — reference example, его нет в записи. Он проверен на соответствие React documentation и DOM/Fetch cancellation contract, но локально не исполнялся: браузерный проект в репозитории отсутствует.

Cancellation и ignore решают разные части проблемы. Abort просит underlying operation завершиться раньше и освобождает ресурсы; ignore гарантирует, что уже полученный или неотменяемый result не изменит stale UI. Data library/router может дополнительно дать deduplication, cache, retries и server rendering. Одного boolean `isMounted` недостаточно, если несколько запросов запускаются при changed dependency, пока component остаётся mounted.

### Оценка ответа

Причину race кандидат увидел и вспомнил, что запрос можно отменить, но не смог назвать API или показать protocol. Для full-stack роли это исправимый пробел. Гораздо заметнее отсутствие тестовой стратегии: ответ не различает behavior, accessibility, integration и E2E risks.

### Оценка ответа

Хорошо названы две разные задачи docs: воспроизвести operation и сохранить rationale. Слабость — надежда на дисциплину без enforcement. Run command можно проверять в CI; API/schema — генерировать или валидировать; decision rationale — хранить рядом с change в коротком ADR; owner и review trigger — указывать явно. Documentation не обязана повторять код, если отвечает на другой вопрос: prerequisites, invariant, failure recovery и «почему альтернатива отвергнута».

«Один день в лёгком sprint» помогает с локальной уборкой, но плохо управляет systemic debt. Нужна очередь с impact: incident risk, lead time, support cost, security, performance и blocked roadmap. Малое исправление встраивается в feature, крупное получает отдельный business case, success metric и staged migration. Если долг скрывать до полного переписывания, product услышит только большой срок без наблюдаемой отдачи.

## Ответы команды на момент записи

- Platform — один крупный Go web service, который обслуживает API и React application, плюс periodic jobs, event processing и небольшие AI services в monorepo.
- Старый event-based processing постепенно переносят в Temporal workflows; RabbitMQ пока остаётся.
- AI features используют OpenAI и ElevenLabs: из transcript/work history извлекают structured profile, есть voice agent для заполнения candidate profile. In-house models нет.
- Основная инфраструктура — GCP, managed Kubernetes и managed PostgreSQL; старый workload остаётся на Hetzner. Application developer обычно не обязан управлять cluster internals.
- Есть production, staging, local и новые branch preview environments. Упомянуты production database dumps для staging/local; sanitization в записи не обсуждается.
- Local setup требует Go, Git, Docker и PostgreSQL; RabbitMQ и Temporal добавляются для соответствующих flows. В README три шага, команда планирует свести запуск к одной command.
- Текущий data mapper по умолчанию делает `SELECT *`; на больших results это уже стало bottleneck, поэтому вводятся projections/custom SQL и обсуждается замена mapper.
- Полноценных migrations пока нет. Deployment проверяет compatibility binary с текущей schema, а команда планирует добавить versioned migrations.

### Технические границы ответов

Temporal и RabbitMQ пересекаются только в части orchestration use cases, но не взаимозаменяемы по определению. Workflow engine хранит event history и восстанавливает durable execution через deterministic replay; broker маршрутизирует независимые messages consumers и имеет собственные ordering, acknowledgement, redelivery и fan-out semantics. Перед удалением RabbitMQ нужно перечислить каждый use case: command queue, pub/sub, delayed retry, long-running saga, external integration — и доказать, что новый primitive сохраняет нужные guarantees.

Schema compatibility check полезен как guardrail, но не заменяет migration. Он может остановить несовместимый binary, однако не создаёт column/index, не выполняет backfill и не координирует mixed-version rollout. Для zero-downtime change нужен expand/contract: добавить backward-compatible schema, развернуть dual-compatible code, backfill/validate, переключить reads/writes и только потом удалить старое.

`SELECT *` не является ошибкой само по себе. Проблема появляется, когда contract требует малую projection, а query читает wide rows, передаёт лишние bytes, декодирует ненужные значения или лишает planner index-only path. Решение подтверждается `EXPLAIN (ANALYZE, BUFFERS)` и метриками, а не запретом звёздочки как style rule. В записи у команды есть наблюдаемый slowdown на выросшем volume, поэтому переход к explicit projections обоснован.

Production dump в development — отдельная граница риска. Запись не говорит, содержатся ли там реальные candidate PII и выполняется ли masking. Если данные не синтетические, нужны documented lawful purpose, minimization, masking/tokenization, short retention, encryption, audit и least-privilege access; удобство foreign keys не оправдывает распространение production identities по laptops и preview environments.
