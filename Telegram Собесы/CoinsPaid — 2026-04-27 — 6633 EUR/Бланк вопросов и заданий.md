### Эталонная модель

`G` — goroutine и её execution state; `M` — OS thread; `P` — runtime resource, необходимый M для выполнения user Go code. Число P равно `GOMAXPROCS`. Планировщик сопоставляет runnable G с M, у которого есть P. При блокирующем syscall M может отдать P, чтобы другой M продолжил выполнять Go code.

У P есть local run queue: частые enqueue/dequeue не проходят через один общий lock и лучше масштабируются. Global queue нужна как общий reservoir и для fairness; work stealing переносит работу между P. Network poller возвращает готовые goroutines после asynchronous network I/O. Это implementation model, а не API contract: точные квоты, размеры очередей и порядок выборки могут меняться между версиями runtime.

Кандидат уверенно называет G/M/P и три источника runnable work. Объяснение local queue через снижение contention верное, хотя фраза «у каждого потока своя очередь» неточна: очередь привязана к P, не навсегда к M. Начальный user stack действительно мал — runtime documentation приводит `2K` как пример — и динамически растёт или сжимается.

## Stack, heap, escape analysis и версии — `00:11:22–00:17:59`

14. Как compiler понимает, что slice не растёт и не покидает функцию?
15. Чем stack allocation практически выгоднее heap allocation?
16. Всегда ли чтение со stack быстрее чтения с heap?
17. Как команда выбирает версию Go и разумно ли всегда отставать на один major release?

### Что важно ответить точнее

Escape analysis — решение compiler о lifetime и reachability, а не правило вида «pointer означает heap». Значение может оставаться на goroutine stack, если compiler докажет, что ссылки не переживут frame и размещение безопасно; может уйти на heap из-за return, closure, interface conversion или другого пути escape. Конкретный результат — implementation detail, его проверяют compiler diagnostics и benchmark/profile, а не угадывают по синтаксису.

Slice — descriptor над backing array. Descriptor и backing storage могут получить разные места размещения; `make([]T, n)` не гарантирует heap. Фиксированная длина, bounded size и отсутствие escape помогают compiler оставить backing array на stack, но это не свойство типа `[]T`.

Главная выгода stack allocation — дешёвое lifetime management и отсутствие отдельного heap object, который должен учитывать garbage collector. Утверждение «stack memory читается быстрее» само по себе неверно: CPU обращается к virtual addresses и cache lines, а не к семантическим категориям Go. Locality зависит от layout и access pattern. Heap array с последовательным обходом может иметь отличную locality; разрозненные pointers на stack — плохую. Кроме того, goroutine stacks динамически растут и перемещаются.

Политика «не брать `.0` немедленно» может быть частью controlled rollout, но «предыдущая версия всегда стабильнее» — не инвариант. На `2026-07-18` актуальны Go `1.26.5` и поддерживаемая ветка `1.25.x`; security fixes выпускаются только для двух последних major lines. Решение должно учитывать supported window, конкретные release notes, тесты, canary и rollback. Отставание не должно оставлять сервис на неподдерживаемой ветке.

### Точные ответы

- `int8/16/32/64` и `uint8/16/32/64` имеют фиксированную ширину; `int` и `uint` — либо 32, либо 64 bits и одинаковы по размеру на конкретной реализации. `int` не становится 8- или 16-bit на экзотической архитектуре по правилам спецификации.
- `int64` хранит signed two's-complement value: от `-2^63` до `2^63-1`, то есть от `-9223372036854775808` до `9223372036854775807`. Кандидат путает его с почти полным диапазоном `uint64`.
- У array length входит в type: `[16]byte` и `[32]byte` — разные value types. Array подходит, когда размер является частью protocol/invariant, нужна value semantics/comparability или API действительно требует фиксированный buffer.
- Slice задаёт изменяемое window над underlying array и обычно удобнее для collections. `make([]T, len, cap)` позволяет заранее выделить capacity, но append сверх cap создаст новый backing array и скопирует элементы.
- Array не гарантирует stack, slice не гарантирует heap. Placement решает compiler. Выбор между ними нужно делать прежде всего по semantics, а allocation подтверждать измерением.

Ответ кандидата показывает базовое различие fixed/dynamic, но слишком быстро связывает array со stack и меньшей latency. Интервьюеры несколько раз возвращают его к механизму; правильная причинная цепочка — value shape и lifetime → решение escape analysis → число heap objects/GC work → измеряемая latency, а не «array быстрее по определению».

### Инварианты языка

Map key должен поддерживать `==` и `!=`, потому что hash только выбирает небольшое множество candidates, а равенство подтверждает identity. Одинаковые keys обязаны давать один hash в рамках операции; разные keys могут collide. Поэтому ожидаемая сложность близка к `O(1)` при хорошем distribution, но спецификация не обещает worst-case `O(1)` и вообще не фиксирует runtime layout.

### До Go 1.24

В Go 1.23 runtime выбирал основной bucket по низким bits hash. Bucket содержал восемь `tophash` bytes, затем плотные arrays keys и values, а в конце pointer на overflow bucket. Lookup сравнивал `tophash`, затем полноценный key. При переполнении проходил overflow chain. Именно эту реализацию описывают интервьюеры и кандидат; для Go `1.23.12` ответ корректен по существу.

### Начиная с Go 1.24

Go 1.24 полностью заменил builtin map на Swiss Table implementation. В Go `1.26.5`:

- hash разделяется на `H1` — upper 57 bits — и `H2` — lower 7 bits;
- group содержит восемь key/value slots и 8-byte control word;
- control byte помечает empty/deleted/used slot, а для occupied slot хранит `H2`;
- lookup одновременно отбирает вероятные slots по control word, затем сравнивает полный key;
- collision переходит к следующей group по probe sequence — отдельной overflow bucket chain больше нет;
- крупная map может состоять из нескольких tables, выбранных через directory по upper hash bits.

Восемь slots остались, поэтому часть старого ответа звучит правдоподобно и после миграции. Диагностический признак устаревания — именно `overflow buckets`. На собеседовании стоит сначала назвать версию, затем дать language invariants и только после этого — layout конкретного runtime.

# Практические задачи

## Задача 1. CPU-bound worker pool — `01:04:45–01:08:43`

### Условие

Есть очень большой array чисел. Для каждого элемента нужно вызвать независимую pure mathematical function; один вызов занимает несколько минут CPU time. Порядок обработки и вывода результатов не важен. Как организовать параллельное вычисление, сколько workers создать и как не оставить часть CPU idle при неодинаковой длительности items?

### Ожидаемое решение

Число workers берут около доступного parallelism, обычно `runtime.GOMAXPROCS(0)`, а затем подтверждают benchmark. Static split на N chunks плох при skew: worker с лёгким chunk закончит рано, пока тяжёлый удерживает остальную работу. Dynamic pool читает отдельные jobs из общего bounded channel; освободившийся worker сразу берёт следующий item.

`WaitGroup` учитывает workers, отдельная goroutine закрывает `results` после их завершения, а producer один раз закрывает `jobs`. Только owner закрывает channel. Cancellation должен останавливать и producer, и workers; output consumer не должен заставить workers навсегда зависнуть.

```go
package main

import (
	"context"
	"fmt"
	"runtime"
	"sync"
)

type result struct {
	input int
	value int
}

func calculate(input int) int {
	return input * input // Placeholder for the expensive pure function.
}

func main() {
	ctx := context.Background()
	inputs := []int{1, 2, 3, 4}
	jobs := make(chan int)
	results := make(chan result)

	workers := runtime.GOMAXPROCS(0)
	var workerWG sync.WaitGroup
	workerWG.Add(workers)
	for range workers {
		go func() {
			defer workerWG.Done()
			for input := range jobs {
				computed := result{input: input, value: calculate(input)}
				select {
				case results <- computed:
				case <-ctx.Done():
					return
				}
			}
		}()
	}

	go func() {
		defer close(jobs)
		for _, input := range inputs {
			select {
			case jobs <- input:
			case <-ctx.Done():
				return
			}
		}
	}()

	go func() {
		workerWG.Wait()
		close(results)
	}()

	for item := range results {
		fmt.Printf("%d -> %d\n", item.input, item.value)
	}
}
```

Reference implementation отсутствует в видео. Код проверен официальным Go Playground на Go `1.26.5` (`2026-07-18`); `vet` ошибок не вернул. Реальная expensive function, memory footprint и error policy остаются частью production design.

### Оценка ответа

Кандидат быстро называет goroutines, `WaitGroup` и число около CPU cores. После уточнения замечает проблему static chunks и приходит к jobs channel/worker pool. Это правильное решение. Не хватило явного ownership channels, cancellation и того, что `GOMAXPROCS` ограничивает одновременно исполняющийся Go code, но не запрещает создать больше goroutines; «больше workers всегда медленнее» тоже не универсально при неоднородном workload или blocking inside function.

## Задача 2. I/O-bound HTTP fan-out — `01:08:43–01:21:01`

### Изменённое условие

Вместо CPU function каждый item требует HTTP request к очень мощному внешнему service. Bandwidth и capacity удалённой стороны условно не ограничивают. Клиент работает на обычной Linux machine. Сколько simultaneous requests можно и нужно отправлять? Почему число не равно `GOMAXPROCS` и во что упрётся клиент?

### Модель решения

Пока goroutine ждёт network I/O, P может выполнять другую G, поэтому concurrency может быть намного выше CPU parallelism. Но универсального числа `100`, `1024` или `10 000` нет. Bound выводят из latency, target throughput и измеренных limits:

`in-flight ≈ target requests/second × service time`.

Затем проверяют:

- `RLIMIT_NOFILE` и уже занятые file descriptors;
- ephemeral port range для множества новых TCP connections к одной destination;
- connection reuse, HTTP/2 multiplexing, `MaxConnsPerHost` и idle pool;
- memory на goroutine, request/response buffers и TLS state;
- CPU на TLS, encoding и response processing;
- timeouts, cancellation, retry amplification и response-body close/drain;
- queue length, backpressure и downstream rate limit.

На Linux default ephemeral range из kernel documentation — `32768–60999`, но это не «лимит goroutines»: reuse/HTTP2, несколько destination IP, TIME_WAIT и local addresses меняют связь между requests и ports. `RLIMIT_NOFILE` также конфигурируется на процесс. Поэтому порядок «десятки тысяч» возможен, но не является ответом без load test и transport configuration.

Практически создают один reused `http.Client`/`Transport`, выставляют request deadline, bounded semaphore или workers, измеряют p95/p99 latency, errors, open FDs, connections, goroutines, heap и CPU, затем увеличивают concurrency до saturation point. Unbounded goroutine-per-item превращает входную очередь в память процесса и убирает backpressure.

### Оценка ответа

Кандидат сначала переносит CPU-bound число workers на HTTP, затем понимает, что ожидание позволяет создать больше goroutines. Он называет connections, ports, descriptors и в итоге порядок около десятков тысяч. Направление верное, но конкретное число кандидат всё ещё угадывает. Сильный Senior-ответ начинается с required throughput/latency и protocol reuse, а заканчивается измеренным bound и overload behavior.

## Kubernetes, PID namespaces и scaling — `01:25:27–01:31:24`

48. Есть ли практический опыт с Kubernetes?
49. Если получить shell на worker node и выполнить `ps`/`top`, будет ли виден process приложения из container?
50. За счёт чего container изолирован, если использует kernel host OS?
51. Что должен учитывать service, чтобы его можно было масштабировать горизонтально?

Container — не отдельная VM. На Linux это обычные host processes, изолированные namespaces/cgroups и другими механизмами runtime. Process в ancestor/root PID namespace видит processes descendant namespaces, поэтому с host node container process обычно виден под host PID, subject to permissions и выбранному `/proc`. Обратно container process не видит host/ancestor processes. Внутри container PID может отличаться и часто начинается с `1`.

Кандидат правильно вспоминает namespaces, но перед этим колеблется, не противоречит ли host visibility изоляции. Не противоречит: visibility асимметрична по hierarchy PID namespaces. Pod тоже не «полноценный server с собственной OS»; это logical host над одним или несколькими containers с shared network/storage context.

Для horizontal scaling кандидат называет stateless instances, external session/state store, load balancing и idempotent operations. Это хороший базовый ответ. Дополнить нужно distributed ownership: leases/locks для singleton jobs, partitioning consumers, cache invalidation, connection pool budget на все replicas, graceful shutdown/readiness и совместимость mixed versions. «Всё состояние вынести в центральную БД» решает routing, но может создать общий bottleneck.

## PostgreSQL: isolation, locks, deadlocks и WAL — `01:31:24–01:41:21`

52. Почему переход с `SERIALIZABLE` на `REPEATABLE READ` снизил число conflicts?
53. Какие способы pessimistic и optimistic concurrency control доступны?
54. Когда и как применять `SELECT ... FOR UPDATE`?
55. Что произойдёт, если две транзакции блокируют несколько rows в разном порядке?
56. Как уменьшить вероятность deadlock и что делать после его обнаружения?
57. Как реализовать optimistic locking через version column?
58. Как понять, что conditional `UPDATE` ничего не изменил, и как retry сохранить semantics?
59. Почему неподходящий index особенно опасен для locking read в MySQL/InnoDB?
60. Что такое WAL и какие гарантии он даёт?

### Isolation нельзя снижать только ради меньшего числа ошибок

В PostgreSQL 18 `REPEATABLE READ` даёт stable snapshot, но допускает serialization anomaly; `SERIALIZABLE` добавляет tracking read/write dependencies и abort одной transaction, если нельзя доказать serial order. Поэтому снижение уровня действительно может убрать часть `40001`, но меняет correctness guarantee. Оно безопасно только после анализа invariants и конкретной anomaly, которую теперь предотвращает explicit lock, constraint или atomic statement.

Кандидат говорит, что повышенная строгость была избыточна, но не называет business invariant и не показывает anomaly test. Без этого optimization недоказана.

### Pessimistic и optimistic paths

`SELECT ... FOR UPDATE` уместен, когда после чтения текущего state та же transaction принимает решение и обновляет row. Locks удерживаются до transaction end, поэтому порядок захвата должен быть deterministic, transaction — короткой, а external I/O — вне lock window. PostgreSQL автоматически обнаруживает deadlock и aborts одну transaction; application обязана retry whole transaction с bounded backoff, а не продолжать с частичным state.

Optimistic update выглядит так:

```sql
UPDATE accounts
SET state = $1,
    version = version + 1
WHERE id = $2
  AND version = $3
RETURNING version;
```

Ноль returned rows означает conflict или отсутствие entity; application перечитывает current state, заново применяет domain decision и повторяет bounded number of times. Нельзя механически повторить старый write, если решение зависело от устаревших данных.

### Важная граница MySQL/PostgreSQL

Уточнение интервьюера про плохой index относится к MySQL/InnoDB: locking read обычно ставит locks на scanned index records/ranges; без suitable index full scan может заблокировать все rows и inserts. В PostgreSQL `SELECT FOR UPDATE` row-locks возвращённые rows, а плохой plan прежде всего удлиняет scan и lock holding; переносить InnoDB next-key/gap semantics на PostgreSQL нельзя.

### WAL

Кандидат верно называет write-ahead rule и crash recovery, но формулировка «WAL восстанавливает consistency, если она нарушилась» слишком широкая. PostgreSQL сначала flushes WAL records, описывающие page changes, и только потом соответствующие data pages могут попасть на disk. После crash REDO повторяет committed changes, которые ещё не дошли до data files. WAL обеспечивает durability/recovery при корректной storage configuration, но не чинит business invariant, lost update или ошибочный SQL.

## Transactional Outbox — `01:41:21–01:44:40`

61. Зачем нужен Transactional Outbox?
62. Какие две операции он связывает одной database transaction?
63. Как Outbox соотносится с eventual consistency и почему не даёт exactly-once end-to-end?

Outbox решает dual-write между изменением business state и намерением опубликовать event. Service в одной local transaction записывает domain change и outbox row. Отдельный relay читает committed outbox и publishes message. Так не возникает состояния «DB commit прошёл, process упал до publish, событие потеряно».

Но relay может publish и упасть до пометки row как sent; событие придёт повторно. Нужны stable event ID, publisher confirmation, retry и idempotent consumer/inbox/unique constraint. Outbox даёт atomic capture намерения, а не exactly-once processing всей distributed chain. Он является одним из способов получить eventual propagation, но не «реализацией BASE» и не заменяет saga/compensation.

Кандидат после уточнения формулирует правильную пару — database state и message publication. До этого смешивает Outbox, retries, rollback и compensating transactions. Это типичная ошибка: паттерн имеет узкую failure window, и ответ нужно начинать именно с неё.

## Задача 3. Платёжный consumer и внешний HTTP-сервис — `01:44:40–01:50:18`

### Условие

Наш service получает из message queue command «перевести сумму с account A на account B» и вызывает внешний, не принадлежащий нам payment service по HTTP. Какие основные риски возникают и как защититься?

### Failure model

1. Message может быть redelivered: consumer упал или потерял connection до ACK.
2. HTTP request может не дойти до provider.
3. Provider может выполнить transfer, но response потеряется; timeout не говорит, был ли effect.
4. Process может упасть после remote success, но до local durable record и queue ACK.
5. Две delivery одного command могут выполняться одновременно.
6. Transient error, permanent validation error и poison message требуют разных retry policies.
7. Out-of-order commands или delayed retry могут нарушить account-level invariant.

### Надёжный protocol

- Producer выдаёт immutable `operation_id`; consumer хранит его под `UNIQUE` constraint.
- Внешний provider обязан принимать тот же idempotency key или иметь status lookup по operation ID. Если ни того, ни другого нет, локальный service не способен устранить ambiguous outcome: повтор может задвоить transfer, отказ от повтора — потерять его.
- Consumer поддерживает durable state machine: `received → executing → succeeded | permanently_failed | unknown`.
- Повтор после timeout сначала проверяет provider status или вызывает тот же idempotent request, а не создаёт новую operation.
- ACK отправляется только после durable state, достаточного для безопасного восстановления. При crash до ACK broker redelivers, а unique key/idempotency делает replay безопасным.
- Retry применяется только к transient classes, с exponential backoff, jitter и limit. Permanent error получает terminal state; poison message — controlled DLQ. Immediate `NACK requeue` loop сжигает CPU/network.
- Success/failure event в broker публикуется через local Outbox. Сам Outbox не делает внешний HTTP transfer exactly once.
- Reconciler периодически проверяет `unknown`/stuck operations и сравнивает ledger с provider.
- Metrics и audit log связывают message ID, operation ID, provider request ID, state transition и attempt.

Consumer acknowledgements и publisher confirms закрывают разные network legs и не знают друг о друге. Manual ACK предотвращает удаление delivery до обработки, но требует готовности к duplicates. Publisher confirm доказывает принятие publish broker, а не выполнение consumer или transfer provider.

### Оценка ответа

Кандидат быстро называет loss/duplicates, network failure, idempotency key, retry и ACK/NACK — ядро проблемы он видит. Однако не разбирает самый сложный window: provider выполнил transfer, response потерян. Также снова упоминает Outbox как универсальную защиту, хотя он нужен для local DB→broker handoff, а не для remote HTTP side effect.

Мысль про PCI DSS уместна только при реальной обработке cardholder data. В условии передаются account IDs и amount; нельзя подменять reliability question требованиями к PAN/CVV. Интервьюеры сами отмечают, что кандидат сначала ушёл в сторону, но положительно оценивают awareness о payment data.
