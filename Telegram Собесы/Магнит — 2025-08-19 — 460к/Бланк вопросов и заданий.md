# Магнит — собеседование Golang на 460к

## TL;DR

Это русскоязычное интервью продолжительностью `01:39:39`. Первые 13 минут посвящены опыту кандидата и продукту Partner Billing. Затем идут Go basics, две задачи на slices, map internals и live coding бинарного поиска. Вторая половина включает B-tree, составные/частичные/covering indexes, ACID, конкурентное изменение баланса, `FOR SHARE`, CAP, 17-минутный system design выдачи миллионов уникальных промокодов и паттерны распределённых транзакций. Итоговый бланк содержит `544` отмеченных пункта. В подписи указан оффер на `460к`; официального feedback в записи нет.

Формат необычный: технический интервьюер заболел, а заменивший его people manager прямо говорит, что не разрабатывал около четырёх лет и идёт по чек-листу. Это объясняет несколько неточных формулировок и отсутствие глубоких уточнений. Для подготовки запись всё же полезна: она показывает, как проверяют причинную модель slices, способность писать алгоритм без IDE, практический PostgreSQL и умение удерживать correctness-инварианты в design-задаче.

Кандидат силён в production-историях, открыто проговаривает неопределённость и находит разумные направления: общий backing array, hash table, условный `UPDATE` баланса, Redis/Kafka, 2PC и Saga. Но точность заметно проседает. Он называет устаревший коэффициент роста slice контрактом, описывает старую реализацию map как текущую, путает B-tree с binary tree, выбирает первый столбец composite index по одной только уникальности, а Serializable — с row mutex. Бинарный поиск проходит только потому, что цель сразу попадает в первый `mid`; финальная функция может зациклиться и паникует на пустом входе.

В system design кандидат хорошо замечает проблему «pod зарезервировал пачку кодов и умер», но перенос миллионов кодов в Kafka не решает главного: сообщение может быть доставлено повторно, а consume и запись назначения не образуют одну транзакцию. Production-grade граница консистентности проще: таблица назначений с `UNIQUE(phone)` и `UNIQUE(promo_code)`, атомарный claim после подтверждения телефона, idempotency key, outbox для SMS и retryable state machine. Kafka может масштабировать асинхронную доставку, но не заменяет database constraints.

# Точный разбор Go-задач

## Задача 1: общий backing array

Код с экрана:

```go
func main() {
	weekTempArr := [7]int{1, 2, 3, 4, 5, 6, 7}
	workDaysSlice := weekTempArr[:5]
	weekendSlice := weekTempArr[5:]
	weekTempSlice := weekTempArr[:]
	weekTempSlice[0] = 0

	fmt.Println(workDaysSlice, len(workDaysSlice), cap(workDaysSlice))
	fmt.Println(weekendSlice, len(weekendSlice), cap(weekendSlice))
	fmt.Println(weekTempSlice, len(weekTempSlice), cap(weekTempSlice))
}
```

Все slices ссылаются на `weekTempArr`. Capacity считается от начала конкретного slice до конца массива: `7`, `2` и `7`. Запись через `weekTempSlice` меняет сам массив и поэтому видна через `workDaysSlice`.

## Задача 2: descriptor скопирован, элементы общие

Финальный код с экрана:

```go
type A struct {
	V int
}

func main() {
	a := make([]A, 0, 2)
	a = append(a, A{V: 1}, A{V: 2}, A{V: 3})
	test(a)
	println(a[0].V)
}

func test(a []A) {
	a = append(a, A{V: 4})
	a[0].V = 999
}
```

В наблюдаемой Go-среде append трёх значений к capacity 2 даёт slice длины 3 с запасом ещё на один элемент. Параметр получает копию descriptor. Локальный append меняет только его `len`, но остаётся в общем массиве; затем запись в `a[0]` видна caller-у. Если бы append потребовал reallocation, `999` попал бы уже в новый массив. Поэтому production API не должен случайно зависеть от spare capacity.

## Бинарный поиск: ошибка инварианта

Смысл финального кода кандидата:

```go
func binarySearch(arr []int, target int) int {
	leftIdx, rightIdx := 0, len(arr)-1

	for rightIdx-leftIdx > 1 {
		midIdx := (rightIdx - leftIdx) / 2
		if arr[midIdx] == target {
			return midIdx
		}
		if arr[midIdx] > target {
			rightIdx = midIdx - 1
		} else {
			leftIdx = midIdx + 1
		}
	}

	if arr[leftIdx] == target {
		return leftIdx
	}
	if arr[rightIdx] == target {
		return rightIdx
	}
	return -1
}
```

Ошибка №1: `midIdx` — длина половины интервала, а не индекс внутри исходного slice. Нужен `leftIdx + (rightIdx-leftIdx)/2`. Ошибка №2: loop condition и endpoint checks взяты из варианта, где boundaries остаются candidates, а updates — из варианта, который исключает `mid`. Ошибка №3: empty input не обработан.

Простой корректный invariant: target, если существует, всегда находится в inclusive interval `[left,right]`. Каждая итерация либо возвращает match, либо уменьшает interval:

```go
func binarySearch(arr []int, target int) int {
	left, right := 0, len(arr)-1
	for left <= right {
		mid := left + (right-left)/2
		switch {
		case arr[mid] < target:
			left = mid + 1
		case arr[mid] > target:
			right = mid - 1
		default:
			return mid
		}
	}
	return -1
}
```

Проверены cases: empty, one element hit/miss, first, middle, last, below range, above range и duplicates. Для production code при подходящем контракте предпочтительнее standard library `slices.BinarySearch`: ручная реализация на собеседовании нужна для демонстрации invariant, а не как рекомендация заново писать primitive.

# Production-grade решение design-задачи

## Scope и инварианты

Сначала нужно определить, где сделать неделимой операцию; выбор хранилища для миллионов строк идёт следом:

1. подтверждённый phone получает не более одного promo;
2. promo назначается не более чем одному phone;
3. повтор того же request возвращает прежний результат;
4. SMS можно повторять, не создавая новое assignment;
5. после crash ни подтверждённое assignment, ни намерение отправить SMS не теряются.

Kafka, Redis и batching сами по себе этих свойств не дают. Граница consistency должна включать authoritative assignment и outbox record в одной local database transaction.

## Минимальная модель данных

```sql
CREATE TABLE promo_code (
    code        text PRIMARY KEY,
    assigned_to text UNIQUE,
    assigned_at timestamptz
);

CREATE TABLE registration (
    phone             text PRIMARY KEY,
    verification_state text NOT NULL,
    promo_code        text UNIQUE REFERENCES promo_code(code),
    assigned_at       timestamptz,
    CHECK (
        (promo_code IS NULL AND assigned_at IS NULL)
        OR
        (promo_code IS NOT NULL AND assigned_at IS NOT NULL)
    )
);

CREATE TABLE outbox (
    event_id       uuid PRIMARY KEY,
    aggregate_key  text NOT NULL,
    event_type     text NOT NULL,
    payload        jsonb NOT NULL,
    created_at     timestamptz NOT NULL,
    published_at   timestamptz
);
```

`phone` хранится в нормализованной форме. Для реальной системы raw PII, display form, encryption/tokenization, retention и access audit проектируются отдельно. `verification_state` — state machine, а не boolean: это позволяет различить created, challenged, verified, locked/expired и не повторять уже принятый OTP.

## Основной поток

1. `POST /registrations` создаёт/читает registration по normalized phone и возвращает challenge; повтор с тем же idempotency key не создаёт новый logical attempt.
2. OTP проверяется с TTL, single-use semantics и bounded attempts. До verification промокод не резервируется.
3. После successful verification начинается database transaction.
4. Existing `registration.promo_code` возвращается как idempotent result.
5. Иначе выбирается одна свободная row:

```sql
SELECT code
FROM promo_code
WHERE assigned_to IS NULL
FOR UPDATE SKIP LOCKED
LIMIT 1;
```

6. Выбранный code помечается phone, registration получает тот же code, в outbox вставляется `PromoAssigned`. Всё commit-ится вместе.
7. Outbox relay публикует событие. SMS worker отправляет message, записывает provider message ID/status и повторяет transient failures с backoff.
8. Повторный API request читает уже committed assignment; повторная доставка event не выдаёт новый code.

`SKIP LOCKED` повышает concurrency claim-ов, но возвращает intentionally inconsistent view заблокированных rows; здесь это допустимо, потому что нужен любой свободный code. Порядок выбора и index на predicate свободного stock нужно проверить на реальном объёме. Если одна большая table становится узким местом, следующий шаг — partitioned ownership/ranges или предварительно распределённые shards, но global constraints/ownership protocol должны остаться явными.

## Failure matrix

| Failure | Что остаётся | Восстановление |
| --- | --- | --- |
| Crash до database commit | Assignment и outbox отсутствуют | Request безопасно повторяется |
| Crash после commit до publish | Assignment и outbox durable | Relay найдёт unpublished row |
| Kafka redelivery | То же event приходит снова | Consumer deduplicates по `event_id`/aggregate state |
| SMS timeout с неизвестным outcome | Assignment не меняется | Проверка provider status или повтор с provider idempotency key |
| Первый SMS provider недоступен | Outbox сохраняет намерение | Retry/backpressure, затем контролируемый fallback |
| Два requests одного phone | Конфликтуют по primary/unique key | Один commit, второй читает existing assignment |
| Два workers выбрали один code | Row lock/unique constraint сериализует claim | Loser выбирает другой code |
| Stock исчерпан | Ни одна row не выбрана | Явный business outcome, alert и admission control |

## Почему вариант из записи недостаточен

Пулы в памяти сокращают latency Redis calls, но превращают каждый server в lease holder. Теперь нужно durable знать, какие codes только зарезервированы, отличать медленный server от умершего, не возвращать code до истечения lease и fencing-ом не позволять старому owner выдать уже reclaimed code. Для исходных 20k RPS это не первая оптимизация без измеренного bottleneck.

Kafka partitions дают ordered log внутри partition и распределяют records по consumers, но обычный consumer может обработать record, записать assignment и упасть до offset commit. Record придёт повторно. Kafka exactly-once помогает для Kafka-to-Kafka processing при правильной transactional configuration; оно не превращает SMS provider и произвольную database transaction в один atomic resource. Финальную защиту дают idempotency и database uniqueness.

- Есть реальные production stories с наблюдаемым результатом, а не только перечисление технологий.
- Кандидат не маскирует незнание и способен развивать решение вместе с интервьюером.
- В slices правильно замечает общий backing array и различие между изменением descriptor и element.
- В задаче баланса быстро приходит к нескольким слоям защиты: row lock, `CHECK` и условная запись.
- В design-задаче самостоятельно обнаруживает наиболее неприятный failure mode in-memory batching: server забрал codes и умер.
- Вопросы о on-call, observability, test ownership, autonomy и internal mobility хорошо проверяют реальную инженерную среду.

## Пробелы и likely decision signals

- Language/runtime contract не отделён от версии implementation. Особенно заметно на slice growth и map.
- Алгоритмическое решение не сопровождается invariant и минимальным набором tests. Для Senior это сильный отрицательный сигнал даже при правильном output на одном input.
- PostgreSQL объясняется через эвристики: «уникальное поле первым», «примерно половина rows», «Serializable как mutex». На уровне Senior нужны planner cost, MVCC и anomaly model.
- В CAP воспроизводится фраза «любые два из трёх» без условий theorem.
- В system design масштабирование начинается раньше correctness. Нет authoritative schema, uniqueness constraints, idempotency и atomic handoff to SMS.
- 2PC описан как последовательность «создать, потом подтвердить», без durable prepared state и coordinator failure.

Заявленный оффер показывает, что конкретная команда сочла общий уровень достаточным. Это нельзя превращать в универсальную hiring bar: интервью проводил заменивший технического специалиста manager, а официального rubric/feedback в записи нет.

# Как готовиться к такому интервью

1. Уметь рисовать slice как `pointer/len/cap` и вручную вычислять aliasing после каждого `append`.
2. На вопрос об internals сначала назвать stable language guarantees, затем конкретную версию runtime.
3. Любой binary search начинать с выбранного interval invariant и прогонять empty/one/edges/miss до запуска.
4. Composite index проектировать от реальных `WHERE/ORDER BY`, а не по абстрактной «уникальности».
5. Isolation level объяснять через snapshots/anomalies/retries, row locks — отдельным механизмом.
6. В balance task первым рассмотреть one-statement conditional update и uniqueness/idempotency.
7. CAP раскрывать только в сценарии network partition и не путать с sharding.
8. Design начинать с invariants и failure matrix; brokers/caches добавлять после source of truth.
9. Для external side effects всегда проговаривать crash between local commit and publish/send.
10. 2PC, Saga и outbox различать по решаемой проблеме, а не перечислять как взаимозаменяемые buzzwords.

# Сопоставление с другими собеседованиями и папкой «Авито»

По форме это не «ещё одно алгоритмическое Авито». В [[Telegram Собесы/Авито — 2026-07-17/Бланк вопросов и заданий|Авито 2026-07-17]] platform и algorithms разведены по разным этапам и оцениваются глубже. В текущем Магните один интервью смешивает Go basics, очень простой live coding, PostgreSQL, CAP и system design; алгоритмическая задача лишь проверяет базовый invariant.

С [[Telegram Собесы/Авито — 2026-04-20 — 470к/Бланк вопросов и заданий|Авито 2026-04-20]] пересекаются delivery guarantees, rate limiting, PostgreSQL и design trade-offs. Но там больше внимания к platform reasoning, а здесь — к чтению короткого Go-кода и широкому чек-листу.

С [[Telegram Собесы/Магнит — 2025-12-26 — 400к/Бланк вопросов и заданий|Магнит 2025-12-26]] совпадают slices, Go runtime, PostgreSQL и Kafka. При этом более поздняя запись технически плотнее в Go/concurrency, а эта даёт более длинный system design и отдельный блок CAP/2PC/Saga. Это подтверждает, что даже внутри одной компании набор сильно зависит от команды и интервьюера.

Материалы [[Авито/roadmap|папки «Авито»]] полезны для отдельных навыков — algorithms и System Design, — но задания не являются буквальными копиями. Ближайший аналог design-блока — не конкретная задача Spotify/Uber, а общий способ работы: сначала requirements/invariants, затем capacity, data model, failure modes и только после этого components.
