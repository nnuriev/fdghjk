## TL;DR

Это русскоязычное интервью продолжительностью `00:49:36`: опыт кандидата, две устные практические задачи, transports и protocols, массивы и pointers, XML/SOAP, доступ к базам из Go, процессы разработки и подробный рассказ о системе продажи автозапчастей. Согласно подписи публикации кандидат получил оффер `3150 USD`.

Две задачи определяют весь технический профиль встречи. В первой нужно обработать CSV-файл размером `16 GiB` восемью goroutines и в любой момент видеть общее число успешно обработанных records. Во второй — показать пользователю далёкую страницу таблицы на `200–300 млн` строк. Первая проверяет различие между дешёвым финальным reduce и live counter через atomic; вторая — понимание того, почему большой `OFFSET` не масштабируется и как устроить seek/keyset pagination по стабильному индексированному ключу.

Live coding не было: кандидат отвечал устно, экран не демонстрировал. В задаче с CSV он сначала предложил корректные локальные counters и сумму после `WaitGroup`, а после требования «видеть результат в любой момент» — общий atomic counter. В задаче пагинации он не объяснил причину деградации `OFFSET`, но дошёл до cursor по `id` или timestamp; интервьюер добавил обязательный индекс. На вопрос о массиве pointers и pointer на массив кандидат ответил неверно, смешав Go с C-style pointer arithmetic и динамическим slice.

Есть неточность и в вопросах интервьюера. `proto2` назван устаревшим так, будто он больше не поддерживается. На `2026-07-18` официальная документация Protobuf по-прежнему поддерживает `proto2`, `proto3`, Edition 2023 и Edition 2024; при этом для новых gRPC service definitions рекомендуются `proto3` или Edition 2023. Это рекомендация для новых схем, а не исчезновение совместимости с `proto2`.

Из продуктового блока уже получается заготовка System Design: более `50 млн` уникальных артикулов и свыше `200 млн` offers по складам, ежедневные `30–50` supplier price lists, exchange rates, delivery cost, индивидуальные скидки, большие выгрузки для клиентов, B2B и retail, supplier APIs/email/1C и несколько конкурирующих источников order status. Главный Senior+-вопрос здесь — не «как переписать PHP на Go», а где держать raw offers, derived prices и materialized exports, как версионировать расчёт и как детерминированно разрешать status conflicts.

# Практические задания и эталонный ход решения

## CSV: точная реконструкция условия

Наблюдаемые constraints:

- один CSV-файл размером `16 GiB`;
- допускается разделение на восемь chunks;
- восемь goroutines обрабатывают chunks параллельно;
- повреждённые records не входят в success count;
- сначала нужен финальный total;
- затем интервьюер усиливает условие: total должен быть доступен во время обработки;
- фраза «что-то упало» не уточняет worker error это или crash всего процесса.

### Модель ответа

Если нужен только финал, лучший базовый вариант — local subtotal у каждого worker и один reduce после `WaitGroup`: workers не спорят за cache line, synchronization происходит один раз. Если нужен точный текущий count, успешный record делает `atomic.Uint64.Add(1)`, observer — `Load()`. Mutex тоже корректен, но heavier и не даёт преимущества для одной независимой integer operation.

Если «упал» весь process, atomic не спасает: его значение было только в RAM. Для restart нужен внешний checkpoint с identity входного файла, completed chunk/range, последним durable offset и idempotent sink. Если упал один worker контролируемой ошибкой, coordinator отменяет остальных, ждёт завершения и возвращает partial snapshot вместе с error; продолжать молча допустимо только для заранее классифицированной invalid row.

Нельзя произвольно делить настоящий CSV по newline: quoted field может содержать newline. Либо условие гарантирует record-aligned chunks, либо один `encoding/csv.Reader` stream-ит records в bounded worker pool, либо сначала строится безопасный index record boundaries.

Полный минимальный пример сохранён в `Данные/verified_csv_counter.go`. Он проверен официальным Go Playground на Go `1.26.5`; `go vet` не выдал ошибок, ожидаемый output — `5`:

```go
type Progress struct {
    success atomic.Uint64
}

func (p *Progress) Success() uint64 {
    return p.success.Load()
}

// В каждом worker после успешной validation:
p.success.Add(1)
```

Пример сознательно предполагает record-aligned `io.Reader` chunks и отдельно комментирует отсутствие durability.

## Pagination: точная реконструкция условия

Наблюдаемые constraints:

- таблица примерно `200–300 млн` rows;
- UI показывает pages;
- пользователь может вручную ввести очень далёкий page number;
- database названа PostgreSQL лишь как пример, механизм считают общим;
- интервьюер ожидает объяснение большого `OFFSET`, затем cursor/key и index.

### Модель ответа

Для последовательного перехода заменяем offset на seek по стабильному unique order:

```sql
CREATE INDEX offers_created_id_idx ON offers (created_at, id);

SELECT id, created_at, payload
FROM offers
WHERE (created_at, id) > ($1, $2)
ORDER BY created_at, id
LIMIT $3;
```

Tuple `(created_at, id)` устраняет неоднозначность одинакового timestamp. PostgreSQL может начать index scan около anchor вместо вычисления миллионов skipped rows. Для tenant/filter индекс должен начинаться с equality-prefix, например `(tenant_id, created_at, id)`.

У seek есть жёсткое ограничение: он не даёт мгновенный random jump на page `3 000 000`, потому что для неё неизвестен anchor. Здесь нужно изменить UI на next/previous, хранить sparse page anchors/materialized result, либо принять дорогой/приблизительный поиск boundary. Это отдельное product requirement, а не «ещё один индекс».

## Массив pointers против pointer на массив

```go
var elements [3]*Item // array из трёх pointers; каждый может быть nil
var whole *[3]Item    // один pointer на array; сам может быть nil
```

Assignment `elements2 := elements` копирует три pointer values. Assignment `whole2 := whole` копирует один pointer, поэтому оба указывают на тот же array. Array fixed-size; `append` и изменение length относятся к slice. Go не даёт обычной pointer arithmetic по элементам array.

# Модель хорошего ответа на ключевые темы

## Aggregation и live progress

Сначала отделяю requirement финального результата от live observability. Для финала каждый worker держит local count, затем coordinator суммирует восемь результатов. Для exact live count workers инкрементируют typed atomic, reader делает `Load`. Если per-row atomic contention заметен, можно batch-ить increments, но тогда snapshot между flushes approximate. Process crash требует durable checkpoint; synchronization primitive внутри process ничего не сохраняет на disk.

## Deep pagination

`OFFSET` прост, но server всё равно вычисляет skipped rows, поэтому cost растёт с глубиной. Seek pagination хранит последний unique ordered tuple и делает inequality predicate с matching B-tree index. Она стабильно даёт next/previous при mutations, но не решает прямой переход к page N без заранее известного anchor. Если random jump обязателен, проектируем sparse anchors/materialized snapshot или честно меняем SLA/UX.

## Protobuf versions

Не смешиваю gRPC и Protobuf. gRPC задаёт RPC service/lifecycle и по умолчанию использует Protobuf как IDL/message format. `proto2` и `proto3` — legacy syntax revisions; Editions постепенно заменяют их как модель эволюции language features. На `2026-07-18` proto2 поддерживается, но official proto2 guide рекомендует proto3 или Edition 2023 для новых gRPC services. Compatibility определяется field numbers/types/presence и rollout, а не только первой строкой `.proto`.

## Data-heavy pricing system

Разделяю canonical catalog, supplier offers, raw imports, exchange rates, pricing policies, client-specific overrides и materialized views/exports. Каждая derived price несёт calculation version и source versions, иначе её нельзя объяснить или воспроизвести. Updates идут idempotent batches; publishing — atomic после validation. Для order statuses храню immutable source events и детерминированно строю current state через explicit state machine, source sequence/priority и reconciliation, а не «последний пришедший wins».
