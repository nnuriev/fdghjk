# X5 — собеседование Golang на 300к

## TL;DR

Это русскоязычное техническое интервью продолжительностью `01:17:39`: короткий разбор опыта, плотный опрос по HTTP и Go runtime, PostgreSQL, выбор хранилища, DDD/Clean Architecture и одна live-coding задача на LRU cache. В подписи публикации указан результат «отказ».

Интервью устроено как последовательное углубление. Почти каждый стартовый вопрос получает проверочное уточнение: не просто «что такое map», а можно ли писать в `var m map`, безопасны ли конкурентные чтения и когда нужен `sync.Map`; не просто «уровни изоляции», а действительно ли `Serializable` запускает транзакции по одной; не просто «сделайте LRU», а почему операции должны быть `O(1)`, как обеспечить thread safety и что делать с contention.

Кандидат узнаёт большинство терминов и иногда приходит к верной идее после подсказки. Сильнее выглядят практический опыт миграции Python-монолита на Go, implicit interface satisfaction, основной aliasing hazard у slices, старая `bucket`-модель map, общая идея tri-color GC и итоговый выбор `map + doubly linked list` для LRU. Он также правильно вспоминает PostgreSQL `Read Committed` по умолчанию и после объяснения понимает keyset pagination.

Главная проблема — отсутствие точной причинной модели. HTTP/2 связывается с «поддержкой HTTPS», `new(map)` принимается за инициализацию, конкурентные чтения map объявляются небезопасными, `sync.Map` выбирается по порогу в пять goroutine, mutex starvation объясняется scheduler quantum, `Serializable` — буквальным выполнением транзакций по одной. В архитектурной секции кандидат смешивает bounded context с ubiquitous language и не формулирует Dependency Rule.

На live coding виден тот же паттерн. До комбинации hash map и двусвязного списка кандидат дошёл только после длинной серии подсказок. Базовая однопоточная идея почти собрана, но финальный thread-safe вариант не компилируется, использует неинициализированный pointer на `RWMutex` и пытается менять список под `RLock`. Код не запускался и не тестировался. Поэтому отказ выглядит согласованным с наблюдаемым уровнем, но это вывод по записи, а не официальный feedback X5.

# Разбор live-coding решения

## Ожидаемая ментальная модель

LRU хранит два независимых вида информации, которые должны меняться атомарно:

- `map[key] -> node` даёт average `O(1)` поиск;
- doubly linked list хранит полный recency order: front — MRU, back — LRU;
- каждый node содержит и `key`, и `value`, чтобы при eviction удалить тот же key из map;
- `Get(hit)` и `Put(existing)` двигают node к front;
- `Put(new)` при заполненном cache удаляет back из обеих структур, затем вставляет новый node в front.

Инвариант: каждый key из map указывает ровно на один node списка, каждый node списка имеет ровно одну запись map, а их размеры совпадают. Именно поэтому один lock вокруг обеих структур — корректная исходная concurrency design.

## Что написал кандидат

По кадрам финальная версия восстанавливается в следующем существенном виде; это не исправленный код, а фиксация наблюдаемого результата:

```go
import (
    "container/list"
)

type entry struct {
    key   string
    value string
}

type LRUCache struct {
    cap   int
    cache map[string]*list.Element
    list  *list.List
    mu    *sync.RWMutex
}

func (c *LRUCache) Get(key string) string {
    c.mu.RLock()
    defer c.mu.RUnlock()
    if el, ok := c.cache[key]; ok {
        c.list.MoveToFront(el)
        return el.Value.(*entry).value
    }
    return ""
}

func (c *LRUCache) Put(key string, value string) {
    c.mutex.Lock()
    defer c.mutex.Unlock()
    // update, eviction и PushFront...
}
```

До concurrency-изменения кандидат собрал правильное ядро: update value + `MoveToFront`, eviction через `Back` + delete key + `Remove`, insertion через `PushFront` + запись element в map. Но финальный результат не является рабочим решением.

## Дефекты финальной реализации

| Дефект | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| Нет import `sync` | compile error `undefined: sync` | На кадре import block содержит только `container/list` | Добавить `sync` |
| Поле `mu`, вызов `c.mutex` | compile error: field/method отсутствует | Несогласованное имя при правке | Использовать одно поле `mu` |
| `mu *sync.RWMutex` не инициализирован | panic на `RLock/Lock` | Конструктора нет, zero value pointer равен nil | Хранить `sync.Mutex` value или явно создавать pointer |
| `Get` меняет list под `RLock` | data race/corruption при concurrent hits | `MoveToFront` — write | Использовать exclusive `Lock` |
| Нет constructor | panic при доступе к nil list, panic при записи в nil map | `list` и `cache` не созданы | `list.New()` и `make(map...)` |
| Capacity `0` не обработана | `Back()` вернёт nil, последующий доступ/Remove panics | Eviction запускается на пустом cache | Для zero capacity сделать `Put` no-op либо отклонить capacity |
| `Get` возвращает только string | miss неотличим от cached `""` | Слабый API исходной заготовки | Если можно менять API, вернуть `(string, bool)` |
| Нет запуска/tests | ошибки остались на экране | Решение не прошло feedback loop compiler/test | Сначала compile, затем table tests и `-race` |

## Проверенный вариант

Этот вариант сохраняет `O(1)` expected operations, обрабатывает zero capacity, различает miss и cached empty string и защищает общий инвариант одним value `sync.Mutex`:

```go
package main

import (
    "container/list"
    "fmt"
    "sync"
)

type entry struct {
    key   string
    value string
}

type LRUCache struct {
    mu       sync.Mutex
    capacity int
    items    map[string]*list.Element
    order    *list.List
}

func NewLRUCache(capacity int) *LRUCache {
    if capacity < 0 {
        panic("negative capacity")
    }
    return &LRUCache{
        capacity: capacity,
        items:    make(map[string]*list.Element, capacity),
        order:    list.New(),
    }
}

func (c *LRUCache) Get(key string) (string, bool) {
    c.mu.Lock()
    defer c.mu.Unlock()

    elem, ok := c.items[key]
    if !ok {
        return "", false
    }
    c.order.MoveToFront(elem)
    return elem.Value.(*entry).value, true
}

func (c *LRUCache) Put(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()

    if c.capacity == 0 {
        return
    }
    if elem, ok := c.items[key]; ok {
        elem.Value.(*entry).value = value
        c.order.MoveToFront(elem)
        return
    }
    if c.order.Len() == c.capacity {
        victim := c.order.Back()
        delete(c.items, victim.Value.(*entry).key)
        c.order.Remove(victim)
    }
    elem := c.order.PushFront(&entry{key: key, value: value})
    c.items[key] = elem
}

func main() {
    c := NewLRUCache(2)
    c.Put("a", "1")
    c.Put("b", "2")
    fmt.Println(c.Get("a"))
    c.Put("c", "3")
    fmt.Println(c.Get("b"))
    fmt.Println(c.Get("c"))
    c.Put("a", "10")
    fmt.Println(c.Get("a"))
}
```

Проверено официальным Go Playground `2026-07-19`, current toolchain Go `1.26.5`: compile errors отсутствуют, `vet` errors отсутствуют, output:

```text
1 true
 false
3 true
10 true
```

Пустая строка во второй строке — zero value для miss, `false` подтверждает вытеснение `b` после того, как `Get("a")` сделал `a` самым свежим.

## Что ожидалось от ответа про high contention

Один exact LRU по определению изменяет глобальный recency order почти на каждом hit. Поэтому заменить Mutex на channel недостаточно: actor всё равно сериализует state transitions. Shard только map тоже недостаточно, если все операции затем берут lock одного списка.

На практике разумная последовательность такая:

1. сначала один `Mutex`, correctness tests, race test и benchmark;
2. если lock действительно bottleneck — несколько полностью независимых shards, каждый со своей map, list, lock и частью capacity;
3. принять, что eviction становится LRU внутри shard, а не точным global LRU;
4. при высокой read rate рассмотреть policy с менее частым обновлением recency: segmented LRU, 2Q, sampled/approximate LRU или TinyLFU-подобную admission policy;
5. измерять hit ratio и tail latency: уменьшение lock wait бесполезно, если policy резко ухудшила hit rate.

## Сильные стороны

- кандидат не скрывает пробелы и иногда сам исправляет ответ, как в случае comparable struct;
- знает направление нескольких механизмов: implicit interfaces, slice descriptor, old map buckets, tri-color marking;
- после объяснения способен усвоить keyset pagination и map/list LRU design;
- update/eviction/insert ядро LRU к концу было логически близко к правильному однопоточному варианту;
- опыт похож на реальную backend-разработку с migration, integrations, messaging и observability.

## Слабые стороны и likely decision signals

- четыре с половиной года Go experience не подтверждаются точностью базовой semantics: nil map, concurrent reads, string bytes, GMP;
- runtime terms воспроизводятся отдельно, но кандидат не выводит behavior из mechanism;
- на уточнениях часто появляется уверенная догадка вместо проверки assumptions;
- database reasoning начинается с готовых ярлыков — replica/index/NoSQL — до query plan и consistency requirement;
- live coding требует почти полного наведения на data structures, а compiler feedback loop не используется;
- concurrency добавляется механически: `RWMutex` выбирается из-за слова «читать», хотя `Get` изменяет list;
- architecture vocabulary не опирается на boundaries, invariants и direction of dependencies.

Официальный feedback в записи отсутствует. Связь этих наблюдений с отказом — обоснованная inference, а не факт, сообщённый X5.

# Как готовиться именно к такому интервью

1. Отвечать цепочкой `contract → mechanism → failure mode → diagnostic → trade-off`, а не списком терминов.
2. Для каждого Go container знать zero value, aliasing, concurrency contract и version boundary implementation.
3. Развести scheduler fairness, mutex fairness и memory visibility: это три разные модели.
4. Держать двухсессионные PostgreSQL examples для RC/RR/Serializable и один `EXPLAIN` deep-pagination example.
5. Перед кодом проговаривать invariants и edge cases, затем сразу запускать compiler/tests.
6. Для LRU за минуту выводить `map + doubly linked list`, а для thread safety замечать, что `Get` mutates recency.
7. Architecture terms связывать с решениями: bounded context — model boundary, ubiquitous language — язык внутри неё, Dependency Rule — направление source dependencies.

# Сопоставление с материалами репозитория

Здесь нет platform/system-design секции масштаба Авито. Это скорее проверка глубины Go core/runtime плюс одна LLD-задача. Большая часть теории уже покрыта предметными заметками vault; запись добавляет не новые темы, а точную последовательность follow-up и их сочетание с LRU implementation.

| Блок интервью | Покрытие в vault | Что добавляет запись |
| --- | --- | --- |

| Interfaces/slices/strings/map | Полное | Быстрые уточнения zero value, aliasing, concurrent reads |
| `sync.Map` | Частичное внутри sync/map материалов | Важная граница Go 1.23 → 1.24: read/dirty → hash-trie |
