### Эталонная модель

В Linux thread — schedulable task, а process удобнее понимать как resource container: address space, file descriptor table, signal dispositions, credentials и другие attributes могут разделяться между tasks. `clone(2)` позволяет выбирать sharing flags; `CLONE_VM` делает memory mappings общими. Поэтому фраза «thread всегда вложен в process» полезна как прикладная модель POSIX application, но kernel primitive устроен гибче.

Threads одного POSIX process разделяют address space и open file descriptors, но имеют собственные stack, registers, scheduling state, signal mask и thread-local storage. У разных processes address spaces изолированы по умолчанию; shared memory возможно создать явно. Число `8 MB`, названное в интервью для stack thread, не является Linux-инвариантом: default зависит от implementation, `RLIMIT_STACK`, attributes `pthread_create` и platform.

Async I/O полезен, когда много операций большую часть времени ждут network/disk/timer и их state можно multiplex на ограниченном числе threads. CPU-bound code выигрывает от parallel execution примерно до доступного CPU budget; дальше растут scheduling и synchronization overhead. Processes дают fault/address-space isolation и возможность независимого lifecycle/resource limits, но требуют IPC и дороже по memory/coordination. Это не три взаимоисключающих архитектуры: server может иметь несколько processes, внутри каждого — threads, а внутри event loop — asynchronous operations.

Кандидат правильно различает I/O-bound и CPU-bound work, упоминает GIL и распределение processes по machines. Ответ проседает на isolation и слишком быстро сводит процессы к «threads без общей памяти». На Senior-уровне ожидается причинная цепочка `sharing boundary → synchronization/IPC cost → failure isolation → workload fit`.

### G/M/P без распространённых подмен

`G` хранит state goroutine, `M` представляет OS thread, `P` — runtime resource, необходимый M для выполнения user Go code. Число P соответствует `GOMAXPROCS`. Runnable G обычно лежат в local run queue P, а не «навсегда у thread». Work stealing позволяет idle P забрать часть runnable work у другого P; global queue и другие sources помогают fairness.

`GOMAXPROCS` задаёт maximum number of CPUs, исполняющих Go code одновременно, а не maximum number of M. Runtime может иметь больше threads: часть блокируется в syscall, обслуживает runtime или ждёт P. Начиная с Go 1.25 default на Linux также учитывает cgroup CPU quota; в Go 1.26 runtime может периодически обновлять default. На собеседовании достаточно назвать invariant parallelism limit и отдельно сказать, что точное поведение default version-dependent.

Начальный goroutine stack мал и растёт динамически; runtime documentation Go 1.26 приводит `2K` как пример, а не language guarantee. Фраза кандидата про `4 KB` не должна становиться заученным числом. Goroutine имеет descriptor, stack, scheduler state и может удерживать references/resources; «можно бездумно создать десятки тысяч» — риск leaks, memory pressure и unbounded downstream load.

### Race, deadlock и правильный выбор primitive

Data race — concurrent access к одной memory location, где хотя бы один access — write и нет корректной synchronization. Lost update — один из возможных логических симптомов, но race может проявиться иначе; conversely, logical race возможно получить и без data race, если отдельные synchronized operations не образуют atomic business transition.

Deadlock prevention нельзя свести к совету «писать аккуратно». Нужны проверяемые invariants: единый global lock order, короткие critical sections, отсутствие network I/O под lock, минимальное число одновременно удерживаемых locks, ownership model и cancellation для blocking operations. Для диагностики нужны goroutine dump, mutex/block profiles и trace; `TryLock` существует, но documentation прямо называет корректные случаи редкими.

- `Mutex` защищает invariant над shared state.
- `RWMutex` допускает нескольких readers или одного writer; выигрывает не автоматически, а при измеримо read-heavy access и достаточно длинных critical sections.
- `atomic` подходит для узкого состояния, которое действительно выражается одной поддерживаемой operation/CAS loop. Go atomics sequentially consistent, но «одна CPU instruction» не является контрактом и contention может сделать CAS loop дорогим.
- `WaitGroup` ждёт завершения набора tasks. Он не ограничивает concurrency: запуск `1000` goroutines с `WaitGroup` остаётся unbounded fan-out.
- Buffered channel может быть semaphore, channel ownership — способом передавать state. Для обычной mutual exclusion `Mutex` обычно яснее.
- `Once` публикует результат однократной initialization; его нельзя копировать после начала использования.

### Задача 1. Границы срезов

Код с экрана:

```go
func slicesExample() {
	x := []string{"a", "b", "c", "d"}
	y := x[:2]
	z := x[1:]
	d := x[1:3]
	e := x[:]

	fmt.Println("x:", x)
	fmt.Println("y:", y)
	fmt.Println("z:", z)
	fmt.Println("d:", d)
	fmt.Println("e:", e)
}
```

Результат:

```text
x: [a b c d]
y: [a b]
z: [b c d]
d: [b c]
e: [a b c d]
```

Здесь проверяют не memory mutation, а half-open interval `[low:high]`, default bounds и отсутствие copy при slicing.

### Задача 2. Изменения через разные views

```go
func slicesSharedMem() {
	x := []string{"a", "b", "c", "d"}
	y := x[:2]
	z := x[1:]

	x[1] = "y"
	y[0] = "x"
	z[1] = "z"

	fmt.Println("x:", x)
	fmt.Println("y:", y)
	fmt.Println("z:", z)
}
```

Результат:

```text
x: [x y z d]
y: [x y]
z: [y z d]
```

Все descriptors смотрят на один array. `z[1]` — это `x[2]`; `y[0]` — `x[0]`. Кандидат правильно восстанавливает общий storage.

### Задача 3. `append` перезаписывает элемент исходного slice

```go
func slicesSharedMemAppend() {
	x := []string{"a", "b", "c", "d"} // len=4, cap=4
	y := x[:2]                          // len=2, cap=4

	fmt.Println(cap(x), cap(y))
	y = append(y, "z")
	fmt.Println("x:", x)
	fmt.Println("y:", y)
}
```

Результат:

```text
4 4
x: [a b z d]
y: [a b z]
```

`append` видит свободный slot внутри capacity `y` и пишет в `x[2]`. Это не «изменение длины x»: `len(x)` остаётся `4`; меняется элемент общего array.

### Как разорвать или ограничить aliasing

Полная независимость требует copy:

```go
y := append([]string(nil), x[:2]...)
```

Если нужно только запретить будущему `append` писать в хвост исходного array, можно ограничить capacity full slice expression:

```go
y := x[:2:2]
```

Тогда `append(y, "z")` выделит другой array. Но существующие `y[0]` и `y[1]` всё ещё alias с `x`; full slice expression не является copy. Именно это различие часто проверяют дополнительным вопросом.

### Struct и interface

Struct — value, состоящее из ordered fields. Methods объявляются отдельно с receiver. Embedding продвигает fields/methods и помогает composition, но не вводит subtype inheritance. Кандидат верно отказывается называть embedding наследованием.

Утверждение «fields можно ставить в любом порядке, compiler сам всё выровняет» опасно. Source order сохраняется, а alignment может вставить padding; перестановка fields иногда заметно меняет `unsafe.Sizeof`. Оптимизировать layout стоит только при доказанной цене memory/cache, сохраняя readability и API constraints.

Zero value формируется рекурсивно: numeric `0`, `false`, empty string, `nil` для pointer/function/interface/slice/map/channel, а struct получает zero value каждого field. Nil map можно читать, но запись panic; nil slice можно `append`; операции с nil channel имеют особую blocking semantics.

Interface задаёт behavioral contract через method set. Для decoupling важнее language contract, чем runtime-представление «два слова». Узкий interface обычно объявляет consumer. Нельзя забывать typed nil: interface value ненулевой, если dynamic type присутствует, даже когда dynamic pointer равен `nil`.

### Задача 4. Shadowing

```go
func shadowIfExample() {
	x := 10
	if x > 5 {
		x, y := 5, 20
		fmt.Println(x, y)
	}
	fmt.Println(x)
}
```

Результат:

```text
5 20
10
```

`:=` объявляет `x` и `y` в inner block. Внешний `x` не переиспользуется: declaration находится в другом lexical scope. После `if` inner variables исчезают, поэтому печатается исходное `10`.

### Closure и граница Go 1.22

Closure захватывает variables из lexical environment, а не автоматически immutable snapshots. До Go 1.22 variables, объявленные `for`/`range`, создавались один раз и переиспользовались; asynchronous closures могли увидеть одно позднее value. Начиная с language version Go 1.22 каждая iteration получает новые variables, если они объявлены самим loop через `:=`.

```go
for _, v := range values {
	go func() { fmt.Println(v) }()
}
```

Для module с `go 1.22` или новее классическая ловушка с `v` устранена. Но если `v` объявлена заранее и loop использует assignment, она остаётся общей:

```go
var v string
for _, v = range values {
	go func() { fmt.Println(v) }()
}
```

Кроме того, любые другие captured shared variables по-прежнему требуют synchronization. Поэтому ответ «все goroutines напечатают последнее значение» без версии Go и формы loop сегодня неполон.

First-class function можно сохранить в variable, передать argument, вернуть result и замкнуть над state. Практический эффект — callbacks, middleware, dependency injection без inheritance и functions higher order.

### Задача 5. `defer` и момент вычисления arguments

Код с экрана:

```go
func deferExample() int {
	a := 10
	defer func(val int) {
		fmt.Println("first:", val)
	}(a)

	a = 20
	defer func(val int) {
		fmt.Println("second:", val)
	}(a)

	a = 30
	fmt.Println("exiting:", a)
	return a
}
```

Результат:

```text
exiting: 30
second: 20
first: 10
```

Arguments сохраняются в момент выполнения каждого `defer`, а calls запускаются LIFO непосредственно перед возвратом surrounding function. При panic они выполняются во время stack unwinding, но `os.Exit` завершает process без deferred calls.

### Context и transaction boundary

`context.Context` переносит cancellation, deadline и request-scoped values через API boundary. Его передают первым parameter, не сохраняют в struct и не используют как необязательный parameter bag. `WithCancel`, `WithTimeout` и `WithDeadline` возвращают `CancelFunc`; caller обязан вызвать его, чтобы освободить связанные resources, когда operation завершена раньше.

Кандидат рассказывает о platform abstraction, где transaction извлекается из context, а repository transparently использует её вместо обычного connection pool. Такой design возможен как infrastructure compromise, но у него есть цена: dependency и transaction scope становятся неявными, type system не показывает requirement, а nested calls труднее анализировать и тестировать.

Более явные alternatives:

- application service открывает `Tx`, передаёт typed transaction repositories в closure и делает commit/rollback;
- Unit of Work создаёт repositories, связанные с одной transaction;
- repository предоставляет atomic domain operation вместо того, чтобы orchestration layer вручную координировал tables.

Если transaction всё же переносится через context, domain code не должен знать storage-specific key; доступ изолируют внутри adapter, а отсутствие transaction должно иметь явную semantics. `Context.Value` documentation разрешает только request-scoped data, проходящие через processes/API, и не рекомендует заменять им function parameters.

### Channel matrix

| Состояние | Send | Receive | `close` |
| --- | --- | --- | --- |
| Unbuffered open | ждёт matching receiver | ждёт matching sender | закрывает channel |
| Buffered open | ждёт только при full buffer | ждёт при empty buffer | закрывает channel |
| Closed | panic | buffered values, затем zero value и `ok=false` | повторный close — panic |
| Nil | блокируется навсегда | блокируется навсегда | panic |

Обычно закрывает sender/owner, потому что только он знает, что новых values больше не будет. Закрытие — broadcast о завершении stream, не способ «освободить память».

`select` сначала вычисляет channel operands и send values, затем выбирает один ready communication псевдослучайно и равномерно. `default` выполняется только когда ни один communication case не готов. Case с nil channel disabled, что позволяет динамически включать и выключать branches.

`make` инициализирует runtime representation slice/map/channel и возвращает value типа `T`; `new(T)` выделяет zeroed `T` и возвращает `*T`. `new(map[K]V)` даёт pointer на nil map и почти никогда не нужен.

В Go `switch` не проваливается в следующий case автоматически. `fallthrough` безусловно запускает body следующего case, не проверяя его condition. Unlabeled `break` завершает innermost `for`, `switch` или `select`.

### Индексы и реальные production-сигналы

Кандидат приводит полезные симптомы: missing index, расхождение planner estimates с actual rows, accumulated dead tuples и необходимость пересмотреть autovacuum. Это хороший production vocabulary, но causal chain стоит проговаривать полностью:

`data distribution changed → statistics no longer model selectivity → planner chooses wrong join/scan → EXPLAIN ANALYZE shows estimate error → ANALYZE/statistics target/schema/query correction → повторное измерение`.

MVCC оставляет старые row versions; `VACUUM` освобождает reusable space, обновляет visibility map и защищает от transaction ID wraparound, а `ANALYZE` собирает planner statistics. «Bloat всегда лечится VACUUM» слишком грубо: обычный `VACUUM` обычно не возвращает место OS и не устраняет все формы index/table bloat; сначала нужно определить причину и lifecycle tuples.

Самая серьёзная ошибка в этом блоке — рассказ о hash index для range query. В PostgreSQL 18 hash index поддерживает только simple equality `=`. Для `<`, `<=`, `>`, `>=`, `BETWEEN`, prefix и ordered scan обычно рассматривают B-tree. GIN полезен для composite values вроде arrays/JSONB/full-text tokens, GiST/SP-GiST — для operator classes и search spaces, BRIN — для больших physically correlated tables.

Index при `INSERT/UPDATE/DELETE` поддерживается инкрементально; он не «полностью rebuild каждую запись». Цена — дополнительные page changes, WAL, I/O, locks/contention, storage и vacuum work. Выбор index начинается с workload и plans, а не с желания индексировать каждый filter column.

Multicolumn B-tree наиболее эффективен, когда query ограничивает leading columns. PostgreSQL 18 умеет skip scan в подходящих distributions, поэтому правило «работает только слева направо» — полезная эвристика, но не абсолютный запрет. Нужны `EXPLAIN (ANALYZE, BUFFERS)`, реальные parameter distributions и статистика.

### Scaling без смешения механизмов

- Vertical scaling увеличивает resources одного instance, но сохраняет single-node ceilings и failure domain.
- Declarative partitioning делит logical table внутри PostgreSQL. Partition pruning исключает partitions по bounds и predicates; само наличие partitions не ускоряет любой query.
- Replication создаёт copies для HA и/или read scaling. Async replica может lag, поэтому read-after-write и failover требуют явных consistency/RPO/RTO решений.
- Sharding распределяет data по independent database instances. Он увеличивает aggregate capacity, но усложняет shard key, routing, rebalancing, unique constraints, transactions и joins across shards.

Partitioning по region разумно, если почти все requests содержат region и распределение нагрузки не создаёт hot/skewed partition. Cross-region queries прочитают несколько partitions; global uniqueness, foreign keys и operational changes могут стать сложнее. Core PostgreSQL предоставляет partitioning и replication primitives, но не transparent automatic application sharding.

В рассказе кандидата replication topology — один primary и две replicas в разных data centers, а failover скрыт internal platform. Для Senior-ответа важно не угадывать platform internals, а назвать вопросы: synchronous или asynchronous replication, кто выбирает leader, как исключается split brain, как fencing защищает старый primary, какой допустим data loss и куда направляются stale reads.

## Как выглядел бы сильный короткий ответ на всё интервью

1. Сначала назвать boundary и workload: shared memory или isolation, I/O-bound или CPU-bound, required parallelism и failure domain.
2. Для Go scheduler зафиксировать `G runnable work`, `M OS thread`, `P execution resource`; `GOMAXPROCS = число P`, а не лимит threads.
3. Для concurrency назвать invariant и owner state, затем primitive; отдельно обеспечить bounded concurrency, cancellation и join.
4. Для slice всегда рисовать descriptor и backing array; отличать `len`, `cap`, mutation, reallocating `append`, cap clamp и copy.
5. Для closure/defer/scope указывать точный момент binding/evaluation и language version.
6. Для database начинать с query plan и workload, а scaling mechanism выбирать по bottleneck и consistency requirements.
7. В спорных implementation details явно говорить: «это поведение конкретной версии, не language contract».

# Эволюция и версии

- Go `1.21` и старше → loop variables, объявленные loop, переиспользовались → asynchronous closures могли увидеть позднее value → требовался explicit copy `v := v` или argument → [Go 1.22 Release Notes](https://go.dev/doc/go1.22).
- Начиная с language version Go `1.22` → каждая iteration создаёт новые loop variables при declaration через `:=` → классическая capture bug устранена, но заранее объявленная variable с `=` остаётся общей → проверено по specification Go `1.26` и release notes.
- Go `1.25` → default `GOMAXPROCS` на Linux начал учитывать cgroup CPU quota → container может получить меньший default, чем число logical CPUs → runtime settings нельзя объяснять только host CPU count.
- Go `1.26` → runtime может обновлять default `GOMAXPROCS` по изменениям environment → hard-coded startup intuition ещё менее надёжна → проверено по `runtime` `go1.26.5`.
- PostgreSQL `18` → multicolumn B-tree documentation явно описывает planner skip scan → простое правило leftmost prefix остаётся хорошей starting heuristic, но не абсолютным описанием plan choices.
