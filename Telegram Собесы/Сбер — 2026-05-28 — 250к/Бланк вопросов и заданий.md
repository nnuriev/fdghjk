# Эталонный разбор практических задач

## Задача 1. Goroutines без ожидания

Исходный код на экране:

```go
func main() {
	for _, val := range []int{1, 2, 3} {
		go println(val)
	}
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/01-goroutine-original-00-06-30.jpg]]

Ни одно конкретное множество строк не гарантировано. `main` может завершиться раньше всех новых goroutines, поэтому допустим пустой вывод. В зависимости от scheduling может появиться часть или все значения, а порядок не определён.

Исправление кандидата:

```go
func main() {
	wg := sync.WaitGroup{}
	for _, val := range []int{1, 2, 3} {
		wg.Add(1)
		go func() {
			defer wg.Done()
			println(val)
		}()
	}
	wg.Wait()
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/02-goroutine-waitgroup-00-07-30.jpg]]

`Add` расположен правильно: counter увеличивается до того, как goroutine потенциально вызовет `Done`. `Wait` гарантирует завершение всех трёх функций, но не порядок печати.

На записи упоминают возможную ошибку closure. Для выбранного редактором Go `1.23` её здесь нет: `val` объявлена через `:=`, а с Go `1.22` каждая итерация создаёт собственную переменную. В старой language version переменная была общей, и для надёжности значение передавали параметром `go func(v int) { ... }(val)`.

## Задача 2. `range` по строке

```go
func main() {
	text := "гошка!"
	for i := range text {
		println(i)
	}
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/03-string-range-00-16-25.jpg]]

Результат:

```text
0
2
4
6
8
10
```

Каждая кириллическая буква занимает в UTF-8 два bytes, `!` — один. Единственная iteration variable получает не порядковый номер rune, а byte index начала очередного decoded code point.

Значение `string` — immutable sequence of bytes. Оно может содержать произвольные bytes, включая invalid UTF-8. Исходный текст Go записан в UTF-8, но escapes позволяют собрать string literal с произвольными bytes; runtime string не имеет encoding tag. `range` пытается декодировать UTF-8 и для invalid sequence выдаёт `utf8.RuneError`, продвигаясь на один byte.

Число code points можно получить через `len([]rune(text))` или `utf8.RuneCountInString(text)`. Первый вариант материализует slice runes, второй только считает. Ни один из них не считает grapheme clusters: emoji sequence, combining mark и один видимый символ могут состоять из нескольких runes.

## Задача 3. Slice, общий array и reslicing

```go
func main() {
	slice := make([]int, 0, 1000)
	slice = append(slice, 1, 2, 3)
	fmt.Println(slice)
	process(slice)
	fmt.Println(slice)
	fmt.Println(slice[:6])
}

func process(slice []int) {
	for index := range slice {
		slice[index] = 0
	}
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/04-slice-task-original-00-18-50.jpg]]

Точный вывод:

```text
[1 2 3]
[0 0 0]
[0 0 0 0 0 0]
```

При передаче в `process` копируется descriptor, но оба descriptors указывают на один underlying array. Поэтому присваивания элементам видны caller-у. Length caller-ского descriptor остаётся `3`: callee не может изменить его, не вернув новый slice или не получив pointer на descriptor.

Reslice `slice[:6]` допустим, потому что upper bound не превышает capacity `1000`. Новые видимые элементы имеют zero value `0`; они уже находились в выделенном backing array. Итоговые комментарии в editor подтверждают этот разбор:

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/05-slice-task-corrected-00-24-10.jpg]]

Спецификация не обещает рост capacity «в два раза», «на 25%» или по любому другому фиксированному правилу. Она гарантирует только: при недостаточной capacity `append` выделит достаточно большой новый array, иначе переиспользует прежний.

## Задача 4. `range` по nil map

```go
func main() {
	var m map[string]struct{}
	for k := range m {
		println(k)
	}
	println("Всё!")
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/06-nil-map-range-00-25-40.jpg]]

Цикл выполняет `0` итераций, затем печатается `Всё!`. Nil map разрешено читать и обходить: lookup возвращает zero value, comma-ok сообщает `false`, `len` равен `0`, `delete` безопасен. Запись запрещена и вызывает panic `assignment to entry in nil map`.

Это отличается от nil channel: send и receive на нём блокируются навсегда, а не ведут себя как пустая конечная коллекция.

## Задача 5. Выбор hash function для demo map

На экране дан псевдокод:

```go
func hash1(a string) int {
	return math.Random()
}

func hash2(a string) int {
	return 1
}
```

![[90 Вложения/Telegram Собесы/Сбер — 2026-05-28 — 250к/Кадры/07-hash-choice-00-27-10.jpg]]

Для корректного demo из этих двух нужна `hash2`. Hash одинакового key обязан быть воспроизводимым в пределах структуры: иначе insert поместит entry в одно место, а последующий lookup начнёт искать другое. `hash1` нарушает correctness независимо от скорости.

Константная `hash2` создаёт collision для всех keys. Она всё ещё может быть корректной, если реализация сравнивает keys и разрешает collisions chaining-ом или open addressing, но операции деградируют вплоть до линейных. Это хороший учебный контрпример, а не пригодная production hash function. Видимый `math.Random()` не является реальным API стандартного пакета `math`, поэтому фрагмент следует читать как псевдокод.

# Ключевые технические пояснения

## Почему goroutines дешевле threads

Goroutine — функция, которую Go runtime планирует поверх набора OS threads. Начальный stack мал и может расти, переключение часто остаётся внутри runtime, а blocking одной G не обязательно блокирует остальные: runtime запускает runnable work на другом M/P. Но goroutine не «работает без thread» — машинные инструкции всё равно исполняет OS thread.

У OS thread есть kernel-visible state, stack reservation, scheduler accounting и более дорогие creation/context-switch paths. Это не означает, что любая atomic operation выполняется «за один такт», а mutex всегда медленнее atomic. Atomic подходит для локального изменения одного значения или специально спроектированного lock-free protocol; mutex проще и часто дешевле для составного invariant, нескольких полей, ожидания или редкой contention.

## Map: ответ зависит от версии

На экране выбран Go `1.23`. Для этой версии уместен классический разговор о buckets по восемь slots, `tophash`, overflow buckets и постепенной evacuation при росте. Даже тогда число `8` не было гарантией спецификации.

Начиная с Go `1.24` runtime map заменена реализацией на основе Swiss Tables. Поэтому заученный ответ «в каждой map всегда buckets по восемь элементов» сейчас неверен. Стабильный уровень ответа: map использует hash, хранит и сравнивает keys, разрешает collisions и динамически меняет storage; конкретные groups, control bytes и growth policy зависят от версии runtime.

Спецификация также не обещает worst-case `O(1)`. Хорошее распределение hash даёт ожидаемый constant-time lookup, но collisions, probing, рост, adversarial input и cache effects меняют реальную стоимость.

## Где ответ был неточным или незавершённым

- Объяснял цену OS threads общими словами и не отделял kernel scheduling, stack reservation, syscall paths и runtime multiplexing.
- Считал closure над `val` потенциальной ошибкой, хотя editor работал на Go `1.23`.
- Сводил отличие buffered channel в основном к наличию queue, но не сформулировал точные readiness conditions и ownership закрытия.
- Утверждал, что atomic быстрее mutex почти по определению и выполняется «за один такт»; это не универсальная гарантия ни CPU, ни Go.
- Сказал, что строки «по спецификации UTF-8». Верно лишь то, что string хранит bytes, а `range` интерпретирует их как UTF-8; invalid bytes допустимы.
- Сначала принял index `range` строки за номер rune, а не byte offset.
- Назвал фиксированный коэффициент роста slice, которого спецификация не гарантирует.
- В slice-задаче сначала забыл, что копия descriptor сохраняет общий underlying array.
- Сначала сказал, что nil map нельзя читать; чтение, `range`, `len` и `delete` безопасны, запрещена только запись.
- Смешивал stable semantics map с implementation details. Для Go `1.23` bucket-модель была уместна, но для Go `1.24+` она уже историческая.
- Не сразу сформулировал обязательную детерминированность hash для одного key и корректный выбор константной функции в demo.

## Что фактически проверялось

Интервью построено не на количестве API, а на способности предсказать программу из нескольких invariants:

- process не ждёт goroutines автоматически;
- channel operation завершается только при выполнении readiness condition;
- string index — byte offset, rune — decoded code point;
- slice descriptor копируется, elements underlying array — нет;
- nil map читаема, но не принимает writes;
- hash обязан стабильно вести key к области поиска, а collisions всё равно требуют сравнения keys.

Кандидат чаще доходил до верного результата после подсказки или запуска. Для уверенного Senior-ответа ожидалось раньше отделять спецификацию от runtime implementation и объяснять результат через invariant, а не через запомненный layout.
