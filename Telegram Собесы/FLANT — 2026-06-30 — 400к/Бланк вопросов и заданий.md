# FLANT — Golang на 400к

## TL;DR

Это собеседование на позицию технического лидера в новую команду GPU Manager платформы Deckhouse. Проверка заметно смещена в сторону инфраструктуры: Linux, сети, Kubernetes, контейнерная изоляция, затем Go, проектирование и три короткие задачи на чтение кода. Алгоритмической секции и полноценного system design нет.

Восстановлено `105` самостоятельных вопросов и follow-up вопросов и три практические задачи: общий backing array нескольких slices, передача указателя по значению и `nil` внутри interface. Условия всех задач подтверждены кадрами. Для первой задачи отдельно проведена граница между гарантией спецификации Go и поведением runtime Go `1.26.5`.

Кандидат уверенно описал назначение StatefulSet, в общих чертах — архитектуру Kubernetes, журналирование файловой системы и dependency inversion. Основные пробелы: состояния процесса и виртуальная память, `SIGKILL`, удалённый открытый файл, DNS и TLS, namespaces/cgroups, admission webhooks, диагностика Pending Pod, гарантии сложности Go map и точная семантика указателей и interfaces. Кандидат называл свой уровень Go Senior, но вторую и третью базовые задачи решил неверно; в первой пришёл к правильному результату только после долгого и неустойчивого рассуждения.

В подписи Telegram указан результат «отказ». Запись не содержит внутреннего письменного фидбэка FLANT, поэтому наблюдаемые ошибки нельзя объявлять официальной причиной решения.

# Эталонный разбор технического блока

## Linux

### Zombie и состояния процесса

Zombie — уже завершившийся child process, для которого kernel сохранил минимальную запись: PID, exit status и accounting data. Он больше не исполняет код и не потребляет обычную память процесса, но остаётся, пока parent не вызовет `wait(2)`/`waitpid(2)` или пока механизм reparenting не позволит забрать статус другому процессу. В `/proc/<pid>/stat` это состояние `Z`.

Состояния надо объяснять не списком названий, а через возможность исполнения и ожидание: `R` — executing или runnable; `S` — interruptible sleep; `D` — uninterruptible sleep, часто ожидание I/O; `T` — остановлен job-control signal; `t` — остановлен tracer; `Z` — завершён и не reaped; `X` — dead; `I` — idle kernel thread. Точный набор символов относится к Linux и фиксируется интерфейсом `/proc`.

Кандидат zombie не знал, а из состояний неуверенно назвал running и sleeping. Для роли, где требуется разбирать проблемы Linux и контейнерного runtime, это существенный пробел в базовой диагностической модели.

### VIRT и RES

`VIRT` — размер виртуального address space процесса: отображённые файлы и библиотеки, anonymous mappings, reserved address ranges, shared mappings и другие регионы. Это не объём «диска, выделенного процессу». `RES`/RSS — число страниц процесса, которые сейчас resident в физической памяти; shared pages могут одновременно учитываться в RSS нескольких процессов, поэтому RSS нельзя бездумно складывать.

Кандидат верно вспомнил изоляцию адресных пространств, но связал virtual memory с выделенным местом на диске. Этого недостаточно: сильный ответ должен отделить virtual addresses, page tables, resident pages и optional swap-backed pages.

### IPC и signals

Для inter-process communication подходят pipes/FIFOs, Unix domain sockets, TCP/UDP sockets, shared memory с отдельной синхронизацией, POSIX/System V message queues, signals и более высокоуровневые хранилища или brokers. Выбор определяется локальностью, framing, объёмом данных, latency, backpressure, crash semantics и требованиями к persistence.

`SIGTERM` по умолчанию завершает процесс, но приложение может установить handler, выполнить graceful shutdown или проигнорировать signal. `SIGKILL` немедленно обрабатывается kernel и не может быть caught, blocked или ignored. Кандидат различил «просьбу завершиться» и принудительное завершение, но предположил, что `SIGKILL` можно перехватить; это прямо противоречит `signal(7)`.

### Journal filesystem

На примере ext4/JBD2 journal защищает от частично применённых metadata updates. Transaction сначала попадает в journal; наличие валидного commit record позволяет replay-ить её после crash, а незавершённая transaction отбрасывается. В default `data=ordered` journal в основном относится к metadata: это не обещание, что любое пользовательское содержимое после внезапного питания будет именно последней записанной версией. `data=journal` сильнее, но дороже.

Кандидат передал общий смысл — восстановление консистентного состояния без полного сканирования — и это близко к правильному ответу. Не хватило границы между metadata consistency и durability file data.

### Удалённый файл продолжает занимать место

`unlink` удаляет directory entry. Если link count стал нулём, но процесс всё ещё держит открытый file descriptor, inode и blocks освобождаются только после закрытия последнего descriptor. Поэтому `du`, обходящий имена, файл не видит, а `df`, смотрящий на blocks filesystem, показывает место занятым.

Практическая цепочка:

1. подтвердить расхождение `df` и `du` на том же mount;
2. найти deleted-but-open descriptors через `lsof +L1` или ссылки `/proc/<pid>/fd/*` с суффиксом `(deleted)`;
3. определить владельца и назначение descriptor;
4. штатно попросить процесс переоткрыть log либо перезапустить его;
5. принудительное обнуление через `/proc/<pid>/fd/<fd>` применять только после проверки, что формат и позиция записи не приведут к потере нужных данных или повторному росту.

Кандидат не нашёл механизм и начал искать большой файл по каталогам, хотя pathname уже отсутствует. После объяснения интервьюера согласился с причиной.

## Сети

### DNS

Обычный stub resolver обращается к configured recursive resolver. Тот проверяет cache и при необходимости проходит referrals от root к TLD и authoritative servers. Ответ может содержать `A`, `AAAA`, `CNAME` и другие записи; caching ограничен TTL и negative-caching правилами.

DNS использует и UDP, и TCP, обычно на порту `53`. RFC 1035 определяет оба transports; TCP нужен не только для zone transfer. Современный general-purpose DNS implementation обязан поддерживать TCP, а переход может понадобиться при truncation, больших ответах и других условиях. Кандидат описал иерархический поиск в общих чертах, но сказал, что TCP использовать нельзя, и не назвал порт `53`.

### MAC address

MAC address относится к network interface и служит адресом доставки frame на local link. Чтобы отправить IPv4 packet следующему hop, узел обычно разрешает его link-layer address через ARP; router на каждом routed hop формирует новый L2 header. Поэтому MAC адрес remote web server не проходит через весь интернет вместе с packet. Duplicate MAC в одном broadcast domain ломает корректное learning/neighbor resolution; OUI может указывать на организацию, которой выделен prefix, но не доказывает фактического производителя устройства.

Ответ кандидата был в основном верным: interface, data-link layer, конфликт в одной сети и OUI. Не прозвучали local-scope, next hop и ARP/NDP.

### TLS 1.3

Сначала устанавливается transport connection, обычно TCP; его SYN/SYN-ACK/ACK не является TLS handshake. Затем `ClientHello` и `ServerHello` согласуют параметры и key shares. Стороны выводят shared secrets и symmetric traffic keys. Certificate chain и signature доказывают владение private key и связывают server identity с доверенным issuer; application data защищается symmetric AEAD, а не шифруется certificate public key на каждый packet.

Кандидат смешал TCP handshake с TLS и заявил, что traffic расшифровывается открытым ключом certificate. Это неверная причинная модель: public-key operations аутентифицируют и участвуют в установлении секрета, но bulk traffic шифруется симметричными ключами.

## Kubernetes и containers

### Компоненты cluster

Control plane включает kube-apiserver, etcd, scheduler и controller-manager; cloud-controller-manager присутствует при соответствующей интеграции. Node components — kubelet, container runtime и kube-proxy, если network implementation его использует. API server — центральный API boundary, etcd хранит authoritative cluster state, scheduler выбирает Node для unscheduled Pod, controllers непрерывно приближают фактическое состояние к desired state.

Кандидат назвал большинство ключевых компонентов, хотя путал названия и роли. Это можно считать базовым знанием архитектуры, но не уверенным уровнем для самостоятельной эксплуатации control plane.

### Admission webhooks

Mutating admission webhook может изменить object после authentication/authorization и до persistence; validating webhook разрешает или отклоняет уже получившийся object. Mutation идёт до validation, поскольку validators должны видеть итоговый object. Webhook — внешний HTTP callback admission controller, а не произвольный hook внутри kubelet.

Кандидат ни mutating, ни validating webhook не объяснил.

### Namespaces и cgroups

Container — не отдельный тип kernel object. Namespaces изолируют представления ресурсов: PID, mount, network, IPC, UTS, user, cgroup и time. Cgroups v2 организуют процессы в hierarchy и учитывают/ограничивают CPU, memory, I/O, pids и другие ресурсы через controllers. Filesystem layers, capabilities, seccomp и LSM дополняют, но не заменяют эту пару механизмов.

Кандидат не назвал ни namespaces, ни cgroups, хотя роль напрямую связана с Kubernetes и container runtime.

### Deployment и StatefulSet

Deployment управляет interchangeable replicas через ReplicaSet и подходит, когда экземплярам не нужна стабильная identity. StatefulSet даёт каждому Pod sticky identity, ordinal name, ordered deployment/scaling и возможность собственного persistent volume claim по template. Это не означает, что StatefulSet автоматически делает приложение stateful-safe: репликация данных, quorum, backup и failover остаются обязанностью приложения или operator.

Это был один из лучших ответов кандидата: он назвал стабильные identifiers, порядок и storage. Не хватило оговорки, что сохранение данных обеспечивается PVC/storage system, а не самим controller.

### Pod не становится Running

Сначала надо зафиксировать phase и container states, затем прочитать Conditions и Events: `kubectl get pod -o wide`, `kubectl describe pod <name>`, при необходимости события namespace. `Pending` часто означает, что scheduler не нашёл Node из-за resources, selectors/affinity, taints/tolerations или unbound PVC. После scheduling container может оставаться `Waiting` из-за image pull, missing Secret/ConfigMap, volume mount, admission/runtime/network setup. Логи приложения могут отсутствовать, потому что process ещё не стартовал.

Кандидат перечислил часть возможных причин, но не знал основной точки входа — `describe` и Events; её подсказал интервьюер.

## Go и проектирование

### Map lookup, Swiss Table и collisions

Спецификация Go определяет семантику map, но не даёт асимптотической гарантии `O(1)`. Практическое ожидание для качественного распределения — среднее constant-time поведение; worst case не является обещанным `O(1)` API contract.

Начиная с Go `1.24`, runtime использует дизайн Swiss Table. В Go `1.26.5` группа содержит `8` slots и `8`-byte control word; одинаковые `h2` bits сравниваются сразу, затем проверяются ключи candidate slots. Официальная статья Go называет группы по `16` элементов возможным будущим направлением, а не текущей реализацией. Поэтому ответ кандидата «теперь гарантированно O(1), в группе 16 элементов за одну SIMD-инструкцию» содержит сразу две ошибки: гарантии нет, текущая группа не 16.

Collision неизбежен для finite hash output и не означает дефект сам по себе. Нужно равномерное распределение и корректное разрешение совпадений через проверку полного ключа и probing/table structure. Cryptographic hash designed как one-way function: «прогнать результат обратно» нельзя; исходное значение иногда угадывают перебором малого domain, но это не inverse operation.

### OOP, SOLID и Clean Architecture

Go поддерживает encapsulation через packages и unexported names, polymorphism через interfaces и reuse через composition/embedding, но не class inheritance. Поэтому ответ зависит от определения OOP; полезнее назвать доступные механизмы и ограничения, чем спорить о ярлыке.

Liskov Substitution Principle требует, чтобы subtype/implementation сохранял поведенческий contract ожидаемой abstraction: не усиливал preconditions, не ослаблял postconditions и не нарушал invariants/observable expectations клиента. Кандидат выбрал LSP, но объяснение осталось абстрактным и смешалось с Single Responsibility Principle.

В Clean Architecture source-code dependencies направлены к policy/business rules, а детали I/O и frameworks зависят от abstractions внутреннего слоя. В Go эти boundaries часто выражаются маленькими interfaces, определёнными со стороны consumer. Здесь ответ кандидата был в целом верным.

### Event-driven architecture в Kubernetes

Event-driven architecture не обязана иметь Kafka: главное — компоненты публикуют факты или изменения состояния, а consumers реагируют асинхронно. В Kubernetes informer/watch получает изменения API objects, controller помещает keys в workqueue, reconcile loop сравнивает desired и actual state и делает идемпотентные действия. Это событийный trigger поверх state-based reconciliation, а не простая «рассылка команды всем контейнерам».

Кандидат свёл понятие к Kafka и рассылке событий нескольким контейнерам и не смог привести Kubernetes mechanism. Для сильного ответа нужны delivery semantics, ordering, duplicate handling, idempotency и distinction event notification/event-carried state transfer.

# Практические задачи

## Задача 1 — slices и общий backing array

### Условие с экрана

```go
package main

import "fmt"

func main() {
    x := []int{}
    x = append(x, 0)
    x = append(x, 1)
    x = append(x, 2)
    y := append(x, 3)
    z := append(x, 4)
    fmt.Println(y, z) // ?
}
```

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/01-slices-condition-00-53-35.jpg]]

### Ответ для Go 1.26.5

```text
[0 1 2 4] [0 1 2 4]
```

В runtime Go `1.26.5` первые append приводят `x` к `len=3`, `cap=4`. `y := append(x, 3)` переиспользует backing array и записывает `3` в slot с index `3`. Исходный header `x` по-прежнему имеет `len=3`, `cap=4`; поэтому `z := append(x, 4)` снова пишет в тот же slot и заменяет `3` на `4`. И `y`, и `z` имеют headers длины четыре, ссылающиеся на один array, поэтому оба видят `4`.

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/03-slices-final-01-01-57.jpg]]

### Важная граница гарантии

Спецификация гарантирует: если новая длина помещается в capacity, `append` переиспользует underlying array; иначе выделяет новый array. Но стратегия роста capacity не стандартизована. Если другая conforming implementation оставила бы после третьего append `cap=3`, оба следующих append могли бы выделить разные arrays, и конкретный вывод изменился бы. Поэтому корректный полный ответ звучит так: «для Go `1.26.5` — два раза `[0 1 2 4]`; спецификация не фиксирует именно этот capacity history».

Локально пример не запускался: Go toolchain в среде отсутствует. Вывод проверен по алгоритму `nextslicecap` на фиксированном tag `go1.26.5` и по семантике `append` из спецификации.

### Как отвечал кандидат

Кандидат долго колебался, предполагал новые arrays и разные варианты результата, затем с помощью вопросов интервьюера пришёл к общему backing array и записал правильный вывод. Финальный результат верен, но механизм не был устойчивым: без знания текущей capacity policy невозможно обосновать, почему после трёх append осталось место ровно для одного элемента.

## Задача 2 — передача указателя по значению

### Условие с экрана

```go
package main

import "fmt"

func main() {
    v := 5
    p := &v
    fmt.Println(*p)
    changePointer(p)
    fmt.Println(*p)
}

func changePointer(p *int) {
    v := 3
    p = &v
}
```

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/04-pointer-condition-01-02-30.jpg]]

### Ответ

```text
5
5
```

Все arguments в Go передаются как values. Вызов копирует pointer value в локальную переменную `p` функции. `p = &v` меняет только эту локальную копию; caller variable `p` продолжает хранить адрес исходной `v == 5`. Локальная `v == 3` может escape to heap, но это не связывает новый адрес с caller.

Чтобы изменить значение по адресу, достаточно:

```go
func changePointer(p *int) {
    v := 3
    *p = v
}
```

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/05-pointer-correction-01-04-17.jpg]]

Чтобы изменить сам pointer caller, пришлось бы вернуть новый `*int` или передать `**int`. Кандидат ответил `5`, затем `3`, потому что счёл передачу pointer отсутствием копирования. Интервьюер показал различие между `p = &v` и `*p = v`.

## Задача 3 — `nil` interface и typed nil

### Условие с экрана

```go
package main

import "fmt"

type errorString struct {
    s string
}

func (e errorString) Error() string {
    return e.s
}

func checkErr(err error) {
    fmt.Println(err == nil)
}

func main() {
    var e1 error
    checkErr(e1) // ?

    var e *errorString
    checkErr(e) // ?

    e = &errorString{}
    checkErr(e) // ?

    e = nil
    checkErr(e) // ?
}
```

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/06-interface-condition-01-04-50.jpg]]

### Ответ

```text
true
false
false
false
```

Interface value можно мысленно представить парой `(dynamic type, dynamic value)`. Он равен `nil`, только когда обе части отсутствуют.

| Вызов | Состояние переданного `error` | `err == nil` |
| --- | --- | --- |
| `checkErr(e1)` | нет dynamic type, нет value | `true` |
| первый `checkErr(e)` | type `*errorString`, value `nil` | `false` |
| после `e = &errorString{}` | type `*errorString`, value — non-nil pointer | `false` |
| после `e = nil` | type `*errorString`, value `nil` | `false` |

Последнее присваивание обнуляет переменную конкретного типа `*errorString`. При передаче в `checkErr` снова выполняется conversion к `error`, и dynamic type восстанавливается как `*errorString`. До conversion выражение `e == nil` было бы `true`.

![[90 Вложения/Telegram Собесы/FLANT — 2026-06-30 — 400к/Кадры/08-interface-candidate-answers-01-09-00.jpg]]

Кандидат проставил `true` как минимум для первых трёх вызовов и до конца путал наличие zero-value struct, pointer value и interface pair. Правильным был только первый ответ; финальный typed-nil случай также не был устойчиво объяснён.
