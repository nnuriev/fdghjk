## Concurrency и execution — `00:08:43–00:14:50`

8. Чем OS process отличается от thread?
9. Как обычно предотвращать race conditions при доступе к shared state?
10. Каков ваш процесс диагностики уже наблюдаемой race condition?
11. В чём различие synchronous и asynchronous execution?
12. Что такое immutability?
13. Чем immutability помогает в concurrent program?

### Эталонная глубина

Process — контейнер ресурсов и address space; в одном process может выполняться несколько threads. POSIX threads разделяют global memory, heap и process-wide resources вроде open file descriptors, но имеют собственные stack, thread ID и register state. Поэтому общение внутри процесса дешевле прямого IPC, а ошибка записи в shared memory требует synchronization и может повредить весь process.

Race condition шире data race: результат зависит от нежелательного interleaving. Data race в модели Go — конкурентные конфликтующие доступы к одной memory location, где хотя бы один доступ является write и нет требуемого happens-before ordering. Защита выбирается по ownership: confinement/immutability, channel handoff, mutex, atomic или redesign состояния.

Для диагностики мало «посмотреть логи». Нужны воспроизводимый workload, `go test -race` или binary с `-race`, stack traces конфликтующих доступов, stress-тест и проверка synchronization protocol. Race detector динамический: он видит только исполненные paths. `pprof` и tracing помогают исследовать CPU, allocation, blocking и execution, но сами по себе не доказывают data race.

Asynchronous означает, что инициатор не обязан ждать завершения операции в том же control flow. Такая операция может выполняться конкурентно даже на одном core; parallelism требует одновременного выполнения. Immutability убирает конфликтующие writes к конкретному immutable value, но не исключает races в публикации reference, lifecycle bugs, lost messages, ordering problems и deadlocks.

### Оценка ответа

Кандидат верно назвал isolation process, shared memory threads, IPC, mutex/RWMutex, atomics и channels. Хорошо начал диагностику с воспроизведения и упомянул `-race`. Формулировка asynchronous как «independently or in parallel» оставила смешанными concurrency и parallelism. Ответ про immutability правильный для shared immutable data, но слишком широко звучал как отсутствие race conditions вообще.

## Database indexes, testing и performance — `00:14:50–00:19:01`

14. Что такое database index?
15. Когда индекс помогает производительности?
16. Каковы издержки и trade-offs индекса?
17. Чем unit test отличается от integration test?
18. Как решить, где нужен unit, а где integration test?
19. Что проверять, если production system внезапно стал медленным?

### Эталонная глубина

Индекс — отдельная структура доступа, которая меняет путь поиска. B-tree обычно применим к equality/range и может удовлетворять совместимый `ORDER BY`; другие index types решают другие классы запросов. Цена — storage, дополнительные writes, WAL/maintenance, vacuum pressure и ошибки выбора planner. Индекс оправдан не названием колонки, а конкретным query shape и измеренным plan: predicate, join, ordering, selectivity, returned fraction и update rate.

Unit/integration — не «быстрый и медленный» как формальное определение. Unit изолирует небольшую decision logic и позволяет точно локализовать failure. Integration проверяет реальный контракт нескольких компонентов: SQL/schema, broker protocol, serialization, network boundary. Рабочая стратегия соединяет много дешёвых deterministic tests с меньшим числом реалистичных boundary tests.

При slowdown сначала нужно локализовать ресурс и слой, сравнив baseline с текущим состоянием: traffic/error/latency, saturation CPU/memory/disk/network, goroutine/block/mutex profiles, traces, dependency latency, connection pools, GC и query plans через `EXPLAIN (ANALYZE, BUFFERS)`. Profiling без вопроса «какая метрика изменилась» часто даёт много данных, но не причину.

### Оценка ответа

Кандидат правильно назвал `WHERE`, `JOIN`, `ORDER BY`, storage и write amplification. В testing ответе отделил business logic от database/API/message boundaries. В performance-диагностике перечислил metrics, logs, traces, `pprof` и `EXPLAIN ANALYZE`; для короткого breadth-вопроса это сильный ответ. Не хватило только последовательности локализации и сравнения с baseline/change events.

## Cryptography и blockchain — `00:19:01–00:23:13`

20. Чем encoding отличается от encryption?
21. На какие две большие группы делят encryption algorithms?
22. Чем symmetric encryption отличается от asymmetric?
23. Что такое hashing и чем он отличается от encryption?
24. На какие две модели состояния можно грубо разделить blockchain systems по структуре транзакций?
25. Чем UTXO model отличается от account-based model?
26. Что такое Bitcoin address?

### Эталонная глубина

Encoding меняет representation ради хранения/передачи и не требует secret. Encryption обеспечивает confidentiality относительно key и рассчитана на обратимое decryption. Symmetric scheme использует shared secret, public-key scheme — связанную пару public/private keys; production protocols обычно комбинируют public-key key establishment/signatures с быстрой authenticated symmetric encryption.

Cryptographic hash отображает сообщение в digest фиксированной длины и применяется, например, для integrity construction и signatures. «Односторонний» не означает encryption без обратной функции: digest не хранит достаточно информации для общего восстановления input. Отдельная важная граница — passwords нельзя безопасно хранить просто через быстрый SHA-256. Нужны unique salt, password hashing scheme/KDF и cost factor; NIST SP 800-63B-4 требует salted hashing подходящей password hashing scheme.

В UTXO model transaction consumes конкретные предыдущие outputs и создаёт новые; состояние — набор unspent outputs. В account model состояние хранит balances/nonces/accounts и применяет transitions к ним. Это влияет на parallel validation, state access и transaction construction, но не делит абсолютно все ledger designs без исключений.

Bitcoin address — человекочитаемое кодирование destination/payment condition с network/version и checksum/error-detection semantics. Legacy P2PKH действительно кодирует public-key hash, но общее определение «уникальный identifier, derived from user's public key» слишком узко: P2SH и современные SegWit/Taproot destinations кодируют другие witness/script programs, а address рекомендуется не переиспользовать.

### Оценка ответа

Кандидат уверенно отличил encoding, encryption, symmetric/asymmetric и hashing. UTXO/account и Base58Check/Bech32 назвал правильно. Два упрощения требуют поправки: password storage нельзя сводить к обычному hash, а Bitcoin address не всегда hash public key. Для general backend эти ответы достаточны; для Bitcoin-centric компании Senior-уровень ожидает различать address types и payment conditions.

## Go — `00:23:13–00:28:20`

27. Как Go управляет памятью?
28. Чем array отличается от slice?
29. Что такое interface и как type его реализует?
30. Что такое zero value и каковы значения для основных групп типов?
31. Чем buffered channel отличается от unbuffered?
32. Для чего используется `defer` и в каком порядке выполняются deferred calls?

### Эталонная глубина

Go implementation выбирает stack или heap по lifetime и escape analysis; адрес локальной переменной может безопасно пережить функцию, если значение перемещено в подходящее storage. Standard toolchain использует tracing, concurrent, non-moving mark-sweep GC для heap objects. Большая часть tracing выполняется concurrent с application, но короткие stop-the-world phases остаются. Stack allocation дешевле не потому, что programmer вручную вызывает free, а потому что compiler может связать lifetime со scope/stack growth.

Array `[N]T` содержит фиксированное число элементов, и `N` входит в type. Slice `[]T` — descriptor над backing array с pointer, length и capacity. `append` может переиспользовать текущий array или выделить новый, поэтому aliasing и capacity важнее фразы «динамический массив».

Interface задаёт type set/method set; implementation implicit. Точность требует учитывать method sets `T` и `*T`, typed nil внутри non-nil interface и стоимость слишком широкого interface. Полиморфизм получается через behavior contract, а не inheritance declaration.

Zero values: `false`, numeric `0`, empty string `""`; `nil` для pointer, function, interface, slice, channel и map; arrays/structs zero-initialize рекурсивно. Полезность зависит от type design: nil slice можно читать и append, nil map читать можно, но write panic; nil channel блокируется навсегда.

Unbuffered channel синхронизирует send с готовым receive. Buffered channel допускает send, пока есть место, и receive, пока есть элемент. Buffer — часть backpressure/lifecycle protocol, а не универсальный способ «ускорить» concurrency.

При выполнении `defer` function value и arguments вычисляются сразу, вызов откладывается до return окружающей function; calls выполняются LIFO. Deferred functions могут читать/менять named result. Они не выполнятся при безусловном завершении process через `os.Exit`.

### Оценка ответа

Все шесть тем кандидат узнал и объяснил по существу. Особенно хороши slice descriptor, implicit interface implementation и blocking rules channels. Memory management ответ остался обзорным: heap/stack и concurrent non-moving GC без escape analysis и cost model. Zero values перечислены не полностью. `defer` по неясной транскрипции нельзя надёжно оценить по слову о порядке, поэтому итоговая оценка опирается только на точно слышимое назначение и cleanup examples.
