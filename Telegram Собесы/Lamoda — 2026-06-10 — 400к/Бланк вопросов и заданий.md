# Эталонный разбор практических задач

## Задача 1. Interface с nil pointer

Восстановленный смысл исходного кода:

```go
type Seller interface {
	GetID() int64
}

type Lamoda struct{}

func (Lamoda) GetID() int64 { return 1 }

func main() {
	var seller Seller
	if seller == nil {
		fmt.Println("nil interface")
	}

	var lamoda *Lamoda
	if lamoda == nil {
		fmt.Println("nil struct")
	}

	seller = lamoda
	if seller == nil {
		fmt.Println("nil assignment")
	}

	l := seller.(*Lamoda)
	_ = l
}
```

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/01-interface-typed-nil-00-04-50.jpg]]

Печатаются первые две строки. До присваивания `seller` концептуально содержит `(dynamic type=nil, dynamic value=nil)`. После `seller = lamoda` он содержит `(*Lamoda, nil)`, поэтому сам interface не равен `nil`. Assertion на `*Lamoda` успешен и возвращает nil pointer; panic не возникает, потому что dynamic type совпал.

`Lamoda` реализует `Seller`, потому что method set `Lamoda` содержит value-receiver `GetID`. `*Lamoda` тоже реализует его: method set pointer type включает методы с receivers `T` и `*T`. Обратное неверно: method с receiver `*T` не входит в method set `T`, хотя компилятор разрешает вызвать его на addressable value как `(&v).M()`.

Если вызвать `seller.GetID()` в текущем состоянии, произойдёт panic: для value receiver нужно разыменовать nil `*Lamoda`, чтобы получить значение `Lamoda`. Наличие метода в method set не делает nil pointer допустимым receiver для value method.

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/03-order-options-errgroup-attempt-00-23-00.jpg]]

Минимальный all-or-nothing вариант:

```go
func BuildOrderOptions(ctx context.Context, orderNr string) (OrderOptions, error) {
	g, requestCtx := errgroup.WithContext(ctx)

	var quantity, deliveryDate, paymentMethod string

	g.Go(func() error {
		value, err := AvailableQuantity(requestCtx, orderNr)
		if err == nil {
			quantity = value
		}
		return err
	})
	g.Go(func() error {
		value, err := DeliveryDate(requestCtx, orderNr)
		if err == nil {
			deliveryDate = value
		}
		return err
	})
	g.Go(func() error {
		value, err := AvailablePaymentMethod(requestCtx, orderNr)
		if err == nil {
			paymentMethod = value
		}
		return err
	})

	if err := g.Wait(); err != nil {
		return OrderOptions{}, err
	}
	return OrderOptions{
		availableQuantity:      quantity,
		deliveryDate:           deliveryDate,
		availablePaymentMethod: paymentMethod,
	}, nil
}
```

Каждая goroutine пишет в отдельную переменную, а чтение происходит после `Wait`, поэтому между этими accesses есть synchronization. Derived context отменяется при первой non-nil error, но cancellation кооперативна: функцию нельзя «убить», она должна передать этот context в HTTP request и обработать `Done`.

`Wait` возвращает первую non-nil error, но только после завершения всех функций. Поэтому sibling requests должны действительно реагировать на context. Иначе ошибочный запрос завершится быстро, а `Wait` всё равно будет ждать зависшую операцию.

В `fetchUrl` отдельно нужны:

```go
resp, err := http.DefaultClient.Do(req)
if err != nil {
	return "", err
}
defer resp.Body.Close()

if resp.StatusCode < 200 || resp.StatusCode >= 300 {
	return "", fmt.Errorf("unexpected HTTP status: %s", resp.Status)
}

body, err := io.ReadAll(io.LimitReader(resp.Body, maxResponseBytes+1))
```

Точный limit и timeout — часть service contract, а не универсальная константа. Request context может ограничивать весь вызов, но это должно быть гарантировано caller-ом; reusable client и transport лучше конфигурировать явно.

## GMP без мифов

G — goroutine, M — OS thread, P — runtime resource, необходимый M для исполнения Go-кода. `GOMAXPROCS` ограничивает число CPUs, которые могут одновременно исполнять Go-код; это не число goroutines и не жёсткий лимит OS threads.

При `GOMAXPROCS=1` сохраняется concurrency: одна G может заблокироваться, после чего P исполняет другую. Одновременного parallel execution обычного Go-кода на нескольких P нет. Для known blocking syscall runtime может отделить P от заблокированного M и передать P другому M; readiness сетевых descriptors обычно обслуживает netpoller.

В Go `1.26.5` local run queue содержит `256` slots. При переполнении `runqputslow` переносит половину локальной очереди вместе с новой G в global queue; `runqsteal` крадёт примерно половину доступной работы. Это текущая implementation detail, а не основание проектировать application protocol вокруг числа `256`.

## Задача 3. Закрытие channel и concurrent map

### Unbuffered channel

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/04-unbuffered-channel-close-00-31-40.jpg]]

Sender блокируется на `ch <- 1`: unbuffered send не завершён без соответствующего receive. Через `500 ms` main вызывает `close(ch)`. Заблокированный sender просыпается с panic `send on closed channel`. Чтение из закрытого channel само по себе безопасно; ошибка кандидата состояла в выборе строки с `for range` как источника panic.

Buffer capacity `1` позволяет единственному send завершиться до `close`, поэтому `range` получает `1` и заканчивается:

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/05-buffered-channel-fix-00-32-40.jpg]]

Но правильный protocol не должен доказываться задержкой. Например, producer может сам закрыть channel:

```go
ch := make(chan int)
go func() {
	defer close(ch)
	ch <- 1
}()

for value := range ch {
	fmt.Println(value)
}
```

### Concurrent map

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/06-concurrent-map-00-34-00.jpg]]

Здесь одна goroutine бесконечно читает `m["a"]`, а другая записывает `m["a"] = 2`. У программы есть data race, а runtime обычно завершит процесс сообщением `fatal error: concurrent map read and map write`. Это не recoverable panic: `recover` не превращает этот дефект в допустимый control flow. Даже момент fatal error и последовательность напечатанных `1`/`2` не гарантированы.

Обычный `Mutex` — разумный default. `RWMutex` выигрывает не от самого наличия reads, а при достаточно длинных/частых параллельных чтениях и редких writes; при write-heavy workload его bookkeeping может быть дороже. `sync.Map` рассчитан на специальные patterns, например write-once/read-many или работу с disjoint key sets.

`panic` уместен для действительно исключительного нарушения внутреннего invariant или невозможности продолжить initialization. Ожидаемые I/O, validation и network failures должны возвращаться как `error`. Обычный panic можно перехватить только `defer recover()` в той же goroutine, стек которой раскручивается.

## Race detector: что доказывает, а что нет

Data race — concurrent conflicting accesses к одной memory location, где хотя бы один access — write и между ними нет требуемой synchronization. Race condition шире: например, два atomic шага «проверить остаток → списать» могут быть каждый race-free, но их composition нарушает invariant из-за порядка выполнения.

Race detector запускается через `go test -race`, `go run -race` или `go build -race`. Он сообщает стеки конфликтующих accesses и создания goroutines. Отчёт нельзя игнорировать как «ложную тревогу»: даже если race кажется benign, она остаётся нарушением модели памяти. Отсутствие отчёта не является доказательством — инструмент видит только runtime paths, которые реально исполнились. Документация Go `1.26.5` оценивает типичный overhead как `2–20×` по времени и `5–10×` по памяти.

## Задача 4. Semaphore, ранний return и утечка

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/07-semaphore-leak-00-47-30.jpg]]

Код создаёт все `count` goroutines сразу. Каждая пытается отправить token в buffered channel ёмкости `3`, выполняет `do`, отправляет result в unbuffered `data` и только затем освобождает token. Пока receiver читает ровно `count` results, успех возможен.

Failure path ломает lifecycle:

1. receiver получает result с error и немедленно возвращается;
2. другие активные goroutines блокируются на `data <- result`, потому что receive больше не будет;
3. blocked send расположен раньше `<-sem`, поэтому tokens остаются заняты;
4. остальные goroutines навсегда блокируются на `sem <- true`.

Это leak, а не обязательный deadlock всего process: request handler может вернуть управление, пока недостижимые снаружи goroutines остаются живы. При повторении request растут goroutine count, retained memory и число blocked stacks.

Для небольшого фиксированного fan-out можно применить `errgroup.WithContext` и `SetLimit`. Для `10 000` или потока jobs лучше настоящий worker pool: создать ровно `N` workers, подавать jobs через channel, делать каждый send/receive context-aware и закрывать result channel только после завершения workers. В обоих вариантах invariant один: функция не возвращает, пока все запущенные ею goroutines либо завершились, либо получили cancellation и дошли до завершения.

Диагностика:

- график `runtime.NumGoroutine` под повторяемой нагрузкой;
- `goroutine` profile со стеками, застывшими на `chan send` и `chan send` в semaphore;
- `goroutineleak` profile, перечисленный в `runtime/pprof` Go `1.26.5`;
- block profile для времени ожидания synchronization primitives;
- heap profile для retained objects.

Высокое число goroutines может быть нормальным. Признак leak — монотонный рост после окончания одинаковых операций и повторяющиеся stacks без lifecycle progress.

## SQL-задача

### Схема и данные

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/08-sql-schema-00-56-20.jpg]]

`customer(id, email, country)` содержит четыре строки, а `cart_item(id, customer_id, title, amount, price)` — товары только для customers `1` и `2`. Foreign key в условии не объявлен, хотя production schema обычно должна закреплять связь явно.

### 1. Все customers и их cart items

```sql
SELECT c.id, c.email, i.title, i.amount
FROM customer AS c
LEFT JOIN cart_item AS i ON i.customer_id = c.id
ORDER BY c.id, i.id;
```

`LEFT JOIN` сохраняет customers `3` и `4`; `title` и `amount` для них будут `NULL`. `RIGHT JOIN cart_item` в показанном направлении сохранял бы все items, а не всех customers. На записи кандидат сначала выбрал правильный `LEFT JOIN`, затем интервьюер попросил отдельно рассмотреть `RIGHT JOIN`; последний вариант остался в editor и не должен читаться как финальная рекомендация.

### 2. Top-10 по общей стоимости

```sql
SELECT c.id, c.email, SUM(i.price * i.amount) AS total
FROM customer AS c
JOIN cart_item AS i ON i.customer_id = c.id
GROUP BY c.id, c.email
ORDER BY total DESC
LIMIT 10;
```

Здесь нужен `price * amount`: стоимость шести единиц по `100` равна `600`, а не `100`. `JOIN` проще `LEFT JOIN ... HAVING SUM(...) IS NOT NULL`, поскольку пустые корзины в top всё равно не входят. В PostgreSQL output alias `total` разрешён в `ORDER BY`, потому что сортировка логически идёт после вычисления select list.

### 3. Customers из России с корзиной не менее 1000 рублей

```sql
SELECT c.id, c.email, SUM(i.price * i.amount) AS total
FROM customer AS c
JOIN cart_item AS i ON i.customer_id = c.id
WHERE c.country = 'ru'
GROUP BY c.id, c.email
HAVING SUM(i.price * i.amount) >= 1000
ORDER BY total DESC;
```

`WHERE` фильтрует исходные rows до aggregation, `HAVING` — сформированные groups. На данных из условия customer `1` имеет `700`, customer `2` — `195`, поэтому результат пуст. Оставшиеся от предыдущей задачи `LIMIT 10` не нужны.

## Индексы PostgreSQL

PostgreSQL 18 поддерживает B-tree, Hash, GiST, SP-GiST, GIN и BRIN. B-tree — default и обслуживает equality, range и ordered scans. Hash хранит hash code и поддерживает только simple equality `=`. GiST/SP-GiST — frameworks для operator classes и пространственных/search structures, GIN удобен для составных значений вроде arrays и full-text tokens, BRIN хранит summaries диапазонов blocks и особенно полезен при корреляции значения с физическим порядком rows.

Обычный `CREATE INDEX` блокирует writes, но не reads. `CONCURRENTLY` не блокирует inserts/updates/deletes, зато делает две scans, ждёт старые transactions, создаёт больше CPU/I/O load, дольше работает и при failure может оставить `INVALID` index. Его нельзя выполнять внутри transaction block.

На кадре видны две мини-задачи:

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/10-sql-indexes-01-18-30.jpg]]

Hash index на `age` не помогает `age > 18`, но подходит для `age = 18` — если planner считает Index Scan дешевле sequential scan.

Для B-tree `(sku, country, customer_id)` predicate по `sku` и `country` использует leading prefix эффективно; третье поле необязательно. Predicate только по `country` и `customer_id` классически считался неподходящим из-за пропуска `sku`. В PostgreSQL 18 planner может применить skip scan, если distinct values первого столбца достаточно мало; поэтому точный ответ — «обычно неэффективно, но использование возможно и проверяется `EXPLAIN (ANALYZE, BUFFERS)`», а не безусловное «индекс никогда не используется».

## Списание баланса

Исходный pseudocode:

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/11-balance-original-01-20-00.jpg]]

`REAL` в PostgreSQL и `float32` в Go — inexact floating-point types. Для денег нужны integer minor units с явной currency либо `NUMERIC(precision, scale)` и точное decimal-представление на стороне приложения.

Главная concurrency-ошибка — stale check. Две transactions могут обе прочитать `100` и обе решить, что списание `80` допустимо. Первая обновит row до `20`; вторая под `READ COMMITTED` дождётся row lock, затем выполнит `balance = balance - 80` над новой версией и получит `-60`. Это не классический lost update, потому что expression использует актуальный database value; нарушен invariant `balance >= 0`, так как проверка была отдельным statement.

Atomic conditional update:

```sql
UPDATE users
SET balance = balance - $2
WHERE user_id = $1
  AND balance >= $2
RETURNING balance;
```

![[90 Вложения/Telegram Собесы/Lamoda — 2026-06-10 — 400к/Кадры/12-balance-atomic-update-01-26-40.jpg]]

Под `READ COMMITTED` второй updater ждёт первого, затем PostgreSQL повторно проверяет `WHERE` на обновлённой row version. Если balance уже недостаточен, row не обновляется. `RETURNING` отдаёт новый balance без дополнительного `SELECT`. Zero rows нужно превратить в domain outcome; если важно различить unknown user и insufficient funds, потребуется отдельная договорённость или дополнительная проверка.

`UPDATE`, не меняющий key columns, берёт row-level lock режима `FOR NO KEY UPDATE`, а не абстрактный «exclusive row lock». Альтернатива с несколькими statements — `SELECT ... FOR UPDATE`, затем check и update в одной transaction. При блокировке нескольких accounts их нужно брать в стабильном порядке, иначе возможен deadlock.

`REPEATABLE READ` и `SERIALIZABLE` могут завершить конфликтующую transaction serialization failure. Это не автоматическое исправление: приложение обязано отменить и повторить transaction целиком. Для одного локального invariant conditional `UPDATE` под `READ COMMITTED` проще и дешевле; `SERIALIZABLE` нужен, когда invariant зависит от более широкого набора rows и predicates.

# Оценка ответов кандидата

## Что получилось

- Узнал typed nil и правильно ожидал, что interface после присваивания nil pointer не равен `nil`.
- Предложил parallel fan-out для независимых HTTP calls и вспомнил `errgroup`/context cancellation.
- Знал базовую модель GMP, local/global queues и work stealing.
- Понял blocking unbuffered send и исправил конкретный пример buffer-ом.
- Сразу распознал concurrent map access как проблему и назвал mutex/`sync.Map`.
- Выбрал `LEFT JOIN`, aggregation, `HAVING` и conditional `UPDATE` с predicate по balance.
- Отметил, что floating-point representation не подходит для денег.

## Где ответ был неточным или незавершённым

- Путал method-call sugar с interface satisfaction: addressable `T` может вызвать `(*T).M`, но это не добавляет method в method set `T`.
- Не завершил `errgroup`-решение: два запроса остались последовательными, а код содержал undefined `err`.
- Не сформулировал, что `Wait` ждёт все goroutines и cancellation не принудительна.
- В channel-задаче указал panic на receive, хотя panic возникает у blocked sender после `close`.
- Назвал runtime fatal error «фатальной panic» и сначала предложил panic для внешней сетевой ошибки — это обратный practical rule.
- Не развёл data race и race condition и не объяснил dynamic coverage race detector.
- Выбирал `RWMutex` по правилу «есть reads и writes», не оценивая ratio, contention и critical-section cost.
- Утечку goroutines увидел только после нескольких подсказок и сначала сосредоточился на частичном result.
- В SQL пропустил умножение на `amount` и сортировку top-10 до подсказки.
- Не назвал точные trade-offs `CREATE INDEX CONCURRENTLY` и row-lock mode.

## Что могло проверяться на уровне Senior

Интервьюеры последовательно переводили вопрос от результата программы к invariant и lifecycle:

- не только «что выведется», но почему method set и dynamic type дают такой результат;
- не только «добавить goroutines», но кто отменяет requests и кто ждёт их завершения;
- не только «поставить semaphore», но сколько goroutines создано и кто их освобождает на error path;
- не только «создать index», но какие locks, scans и planner conditions стоят за ним;
- не только «повысить isolation», но какой single statement закрепляет business invariant.

Именно переход от узнавания конструкции к доказательству завершения и сохранения invariant оказался самым слабым местом кандидата.
