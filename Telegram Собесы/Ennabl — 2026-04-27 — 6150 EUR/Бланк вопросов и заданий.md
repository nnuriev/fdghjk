### Восстановленная ментальная модель

Клиент обращается через `FE → GW`. Gateway оркестрирует риск-проверки, CAPTCHA и вызовы `Security`/`2FA`. Контур 2FA проверяет применимость phone number, инициирует SMS через каскад providers и сверяет введённый code. Security ограничивает число попыток и вводит cooldown примерно `10–15` минут; GeoIP/IP signals добавлены после инцидента. После успешной проверки регистрационный flow выдаёт временный token/session и продолжает создание пользователя.

Такое описание полезно, но на Senior+ уровне его стоит превратить в явную state machine, например:

`NEW → RISK_PASSED → OTP_ISSUED → VERIFIED → REGISTERED`

У каждого перехода нужны TTL, attempt budget, idempotency key и устойчивый результат повторного запроса. Иначе retries клиента или provider могут дважды отправить SMS, две конкурентные проверки могут по-разному изменить счётчик, а истёкший code — оказаться пригодным после нового выпуска. Ограничения нужны не только на phone number: abuse обычно требует сочетать phone, IP/subnet, device/session, account и provider budget, иначе атакующий меняет один признак.

### Failure story: что именно показал ответ

Команда сочла legacy CAPTCHA/GeoIP integration слишком дорогой для первого релиза и выпустила flow без неё. В течение первого часа endpoint начали использовать для массовой отправки SMS; его закрыли и добавили защитные проверки. Это хороший ответ не потому, что решение было безошибочным, а потому, что видна причинная цепочка:

`неверное предположение «attempt limit по номеру пока достаточен» → резкий SMS traffic/cost → публичный endpoint позволяет менять номера и инициировать платные сообщения → закрыть endpoint, добавить multi-dimensional rate limits, bot/risk checks и наблюдаемость`.

Не хватило конкретных operational деталей: какая метрика сработала, как ограничили ущерб у providers, были ли idempotent sends, как выполняли replay/reconciliation и какое SLO установили после инцидента. CAPTCHA также не должна быть единственной границей защиты: её можно обходить, а ложные срабатывания ухудшают legitimate conversion.

### Ход решения кандидата

Кандидат вводит `CarSize(Enum)`, dictionary свободных мест и `active_tickets: token → spot_size`. Для каждого размера вручную проверяет подходящие места от меньшего к большему. При парковке уменьшает counter, генерирует `uuid4` и сохраняет ticket. При возврате проверяет token, удаляет ticket и увеличивает counter.

### Что получилось хорошо

1. Модель минимальна: availability отвечает на вопрос о capacity, ticket хранит информацию для обратной операции.
2. Smallest-fit policy сохраняет scarce large spots для больших машин.
3. Неизвестный или повторно использованный token не увеличивает capacity второй раз: `return_car` возвращает `False`.
4. Кандидат сам замечает hard-coded capacities, persistence, concurrency и повторяющиеся branches.
5. Обобщение по ordered sizes сохраняет правило выбора и уменьшает число мест, которые надо менять при добавлении размера.

### Ошибки и недостающие инварианты

На финальном кадре после `self.active_tickets[token] = spot` сразу начинается `return_car`; `return token` отсутствует. Поэтому API обещает parking token, но успешный Python-call неявно возвращает `None`. Аннотация `-> str` также противоречит ветке «парковка заполнена», которая возвращает `None`; корректнее `Optional[str]` или `str | None` в Python 3.10+.

Синтаксис в Notes местами повреждён и не запускался, но missing return — именно semantic defect. Его можно поймать одним contract test: успешный `park_car` должен вернуть непустой token, после чего `return_car(token)` один раз возвращает `True`, а второй — `False`.

Для корректности полезно явно зафиксировать инварианты:

- `0 ≤ available[size] ≤ capacity[size]`;
- каждый active token соответствует ровно одной занятой reservation;
- `capacity[size] - available[size]` равно числу active tickets этого spot size;
- при наличии idempotency key повтор одного запроса не создаёт второй active token;
- `park_car` атомарно либо уменьшает availability и создаёт ticket, либо не меняет состояние;
- `return_car` атомарно либо удаляет active ticket и освобождает место, либо ничего не меняет.

Без atomicity два потока могут одновременно увидеть последнее свободное место, оба уменьшить counter и выдать два tokens. Lock должен охватывать весь check-and-modify, а не только отдельный dictionary access.

### Минимальный исправленный вариант

```python
from enum import Enum
from threading import RLock
from typing import Dict, Optional, Tuple
from uuid import uuid4


class CarSize(Enum):
    SMALL = "small"
    MEDIUM = "medium"
    LARGE = "large"


class ValetParking:
    _fit_order: Dict[CarSize, Tuple[CarSize, ...]] = {
        CarSize.SMALL: (CarSize.SMALL, CarSize.MEDIUM, CarSize.LARGE),
        CarSize.MEDIUM: (CarSize.MEDIUM, CarSize.LARGE),
        CarSize.LARGE: (CarSize.LARGE,),
    }

    def __init__(self, small: int, medium: int, large: int) -> None:
        if min(small, medium, large) < 0:
            raise ValueError("capacity must be non-negative")
        self._available = {
            CarSize.SMALL: small,
            CarSize.MEDIUM: medium,
            CarSize.LARGE: large,
        }
        self._tickets: Dict[str, CarSize] = {}
        self._lock = RLock()

    def park_car(self, car_size: CarSize) -> Optional[str]:
        with self._lock:
            for spot_size in self._fit_order[car_size]:
                if self._available[spot_size] == 0:
                    continue
                token = str(uuid4())
                self._available[spot_size] -= 1
                self._tickets[token] = spot_size
                return token
            return None

    def return_car(self, token: str) -> bool:
        with self._lock:
            spot_size = self._tickets.pop(token, None)
            if spot_size is None:
                return False
            self._available[spot_size] += 1
            return True
```

Проверенный сценарий на Python `3.9.6`: по одному месту каждого типа последовательно принимаются `S`, `M`, `L`; четвёртая small car получает `None`; возврат medium token освобождает `M`; следующая small car занимает `M`; повторный возврат того же token даёт `False`. Наблюдаемый итог теста — `ok`.

Explicit compatibility table лучше неявного сравнения числовых enum: если позже появится `ELECTRIC_COMPACT`, handicap spot или truck, физическая совместимость перестанет быть простым отношением `spot_size >= car_size`.

### In-memory lock и distributed correctness

`RLock` защищает только один Python process. GIL не делает составной протокол «проверить counter → уменьшить → записать ticket» бизнес-атомарным и не синхронизирует несколько workers/hosts. Если сервис имеет replicas, source of truth должен выполнять reservation атомарно.

Один вариант — хранить отдельные `parking_spot` rows и `reservation` rows. В transaction выбирается свободное совместимое место в порядке smallest fit с row-level locking, затем создаётся active reservation; возврат меняет её состояние только из `ACTIVE`. Другой вариант — counters per size с conditional update `available > 0` и unique active token, но тогда сложнее доказать связь с конкретным физическим местом. При высокой конкуренции полезно заранее определить retry/deadlock policy и fairness, а не бесконечно повторять transaction.

Просто «загрузить active tickets после restart и пересчитать counters», как предложено в интервью, восстанавливает один process, но не устраняет race между replicas и crash window между двумя несвязанными writes. Нужна одна durable transaction либо журналируемая state transition с idempotent recovery.

### Оценка решения

Для 25-минутного design exercise кандидат выбрал адекватные structures и во время review назвал почти все направления развития. Слабое место — точность базового API contract: missing return должен быть замечен до разговора о database и extensibility. Второй пробел — слово «concurrency» осталось предложением добавить methods/locks без формулировки critical section и без перехода от process-local lock к distributed transaction.
