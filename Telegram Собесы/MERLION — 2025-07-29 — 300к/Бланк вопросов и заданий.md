# MERLION (Ситилинк) — собеседование Golang на 300к

## TL;DR

Это русскоязычное интервью продолжительностью `01:13:13`. Алгоритмов и system design в привычном смысле здесь почти нет. Вместо них идёт широкий устный опрос по инженерной базе: опыт, паттерны, SOLID и LSP, Go composition, тестирование, Docker, shell, cron, Git, ACID, PostgreSQL indexes, диагностика двухсекундного запроса, generics, reflection и финальное code review на concurrency. В подписи Telegram указан отказ и компенсация `300к`; официальной причины отказа в записи нет.

Главная практическая задача — ревью Go-handler, который параллельно получает описание и цену товаров. В исходнике нет ожидания горутин, две горутины пишут в один `resultErr`, одна теряет входной `ctx`, другая вообще не умеет принимать context. После этого код дважды добавляет один и тот же товар в response: отдельный элемент с `Info` и отдельный с `Price`. Есть и ошибки моделирования: `CreatedAt uint8`, деньги в `float64`, три варианта ID (`int`, `int32`, `ProductId`), лишние pointer states и неоговорённое поведение для частично найденного товара.

Запись также содержит важный контекст оценки кандидата: на его экране во время интервью открыты Grok 3 и ChatGPT 4.5. Он читает ответы Grok в блоках про patterns и indexes, заранее просит ChatGPT решать присланные скриншоты, а во время code review рядом с Yandex Code появляется предложенная моделью реализация с `WaitGroup` и `Mutex`. Это наблюдаемый факт из видео, а не предположение. Однако интервьюер его не комментирует, поэтому связывать использование AI с отказом как установленную причину нельзя.

Для подготовки важнее всего вот что: здесь проверяют не память на определения, а способность удерживать причинную модель. «Decorator соблюдает SOLID», «generic всегда такой же быстрый», «pointer обязательно лежит в heap» и «любой доступ через reflection очень медленный» — недостаточно точные формулы. Сильный ответ оговаривает контракт, текущую реализацию, условия измерения и ближайшую альтернативу.

# Разбор ключевых ответов

## 1. Cache decorator: важен не ярлык, а граница политики

Интервьюер ожидает Decorator и связывает его с SRP/OCP. Это разумный вариант: `CachedStorage` реализует тот же read contract, сначала проверяет Redis, а при miss делегирует вызов database storage. Caller не знает, кэширован ли storage, а policy не смешивается с SQL.

Но из этого не следует, что cache внутри repository всегда «нарушает SOLID». Если repository contract уже обещает materialized read model с заданной freshness, cache может быть частью одной ответственности. Решение выбирают по трём вопросам: кто владеет freshness contract, где проходят write/invalidation paths и можно ли протестировать отказ Redis независимо. Pattern name помогает обсуждать форму, но не заменяет эти условия.

Минимальный production flow:

1. read cache;
2. на hit вернуть value и записать hit/age metrics;
3. на miss выполнить source-of-truth read;
4. coalesce concurrent misses одного key;
5. положить value с TTL и schema version;
6. на Redis failure выполнить controlled bypass, если business contract это допускает;
7. на write сначала завершить database transaction, затем надёжно invalidate/update cache;
8. явно определить допустимое stale window.

## 2. SOLID в Go: LSP проверяет поведение, а не embedding

Go не имеет class inheritance. Embedded field даёт promoted fields/methods, но enclosing struct остаётся отдельным named type. `Person` с embedded `Human` нельзя передать как `Human`; нужно выбрать `p.Human` или определить interface поведения. Поэтому классическую иерархию «base/derived» нельзя механически переносить на structs.

LSP при этом применим. Если handler принимает `Storage`, каждая реализация должна сохранять согласованный observable contract: значение, категории errors, cancellation, ordering, side effects и consistency. Реализация, которая игнорирует `ctx`, возвращает partial data как success или делает retries с непредсказуемыми side effects, может нарушать substitutability даже при полном совпадении method set.

## 3. Docker: `scratch` — специальный пустой base

Правильный ответ на задачу — multi-stage build. Builder содержит Go toolchain и modules; runtime stage получает только binary и необходимые runtime files. Вопреки реплике интервьюера, Docker официально показывает `FROM scratch` как base второго stage.

```dockerfile
# syntax=docker/dockerfile:1
FROM golang:1.26 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -trimpath -o /out/app ./cmd/app

FROM scratch
COPY --from=build /out/app /app
ENTRYPOINT ["/app"]
```

Такой вариант подходит не всегда. HTTPS может требовать CA bundle, timezone operations — tzdata, а CGO-linked binary — dynamic libraries. Ближайшая альтернатива — distroless runtime: он чуть больше, зато проще для certificates, non-root identity и диагностики. Выбор определяется dependencies и operating model, а не только размером image.

## 4. Диагностика latency: сначала разрез времени

Кандидат сразу идёт в `EXPLAIN`, index, replication и sharding. Интервьюер постепенно подводит к connection pool exhaustion. Сильнее начать с trace и сначала разрезать общее время запроса:

```text
request total
├── queue / middleware
├── pool acquire
├── network + PostgreSQL execution
├── row decoding
└── response serialization
```

Если длинный span — pool acquire, ускорение SQL не устранит очередь. Если server-side execution быстрое, а client span длинный, проверяют network, TLS/proxy, decoding и payload. Если медленно внутри PostgreSQL, только тогда сопоставляют `EXPLAIN (ANALYZE, BUFFERS, WAL, SETTINGS)` с locks, waits, row estimates, I/O и bloat. Replication или sharding появляются после доказанного resource bottleneck и read/write pattern, а не как универсальная оптимизация.

## 5. Generics: скорость не является language guarantee

Формула интервьюера «compiler создаёт отдельную typed function, поэтому скорость всегда одинаковая» слишком категорична. Go specification задаёт type semantics, но не равенство machine code или latency. Реализация Go 1.18 ввела dictionaries и gcshape stenciling: несколько concrete types могут использовать одну shape instantiation, а call получает dictionary с type-dependent данными. Optimizer и inliner продолжают меняться между releases.

Для простого `a+b` generic и concrete versions часто дают одинаковый или близкий код, но это нужно проверять benchmark и assembly на целевой Go version/GOOS/GOARCH. Практическое правило: выбирать generics ради type-safe reuse, а performance claim подтверждать измерением. Если набор типов закрыт и логика различается, type switch может быть понятнее; reflection оставляют для genuinely dynamic schema.

## 6. Code review: полный список дефектов

| Приоритет | Дефект | Наблюдаемый риск | Исправление |
| --- | --- | --- | --- |
| P0 | Нет ожидания goroutines | Empty/partial response, races | `WaitGroup` или `errgroup`, затем читать results |
| P0 | Общий `resultErr` | Write/write и read/write races, потеря error | Separate slots или `errgroup`; определить aggregation contract |
| P0 | `context.Background()` | Работа продолжается после cancel/deadline | Передавать incoming/derived `ctx` |
| P0 | `Products` без context | Call нельзя отменить | Изменить storage interface |
| P0 | Два append на один ID | Два неполных товара вместо одного | Собрать обе map entries, append один `Product` |
| P1 | Missing entry молча игнорируется | Partial data выглядит успешной | Явный partial/not-found contract |
| P1 | `CreatedAt uint8` | 256 состояний вместо времени | `time.Time` или явно документированный epoch type |
| P1 | `Price float64` | Ошибки двоичного представления денег | Minor units или decimal |
| P1 | `int32` ↔ `int` по всему handler | Noise, overflow risk, размытая граница | Нормализовать ID один раз на transport boundary |
| P2 | Pointer-heavy DTO | Лишние nil states и возможные allocations | Values для небольших обязательных полей; измерить |
| P2 | Нет preallocation | Лишний рост response slice | `make([]Product, 0, len(ids))` |
| P2 | Нет validation/limit | Nil panic, слишком большой batch | Проверить request и upper bound |
| P2 | Bare errors | Непонятно, какой dependency упал | Wrap operation context |
| P2 | Нет тестов | Регрессии ordering/missing/cancel | Table tests, cancellation test, `-race` |

## 7. Исправленная реализация

Ниже выбран строгий contract: оба источника обязательны; response сохраняет порядок запроса; при отсутствующем `Info` или `Price` handler возвращает error. Если продукт допускает partial response, это решение нужно изменить явно, а не молча пропускать поля.

```go
package product

import (
	"context"
	"errors"
	"fmt"
	"sync"
	"time"
)

type ProductInfo struct {
	ProductID int
	CreatedAt time.Time
}

type ProductPrice struct {
	ProductID  int
	PriceMinor int64
}

type Product struct {
	ID    int
	Info  ProductInfo
	Price ProductPrice
}

type Storage interface {
	Products(ctx context.Context, productIDs []int) (map[int]ProductInfo, error)
	Prices(ctx context.Context, productIDs []int) (map[int]ProductPrice, error)
}

type GetRequest struct {
	IDs []int
}

type GetResponse struct {
	Products []Product
}

type Server struct {
	storage Storage
}

func (s *Server) Get(ctx context.Context, req *GetRequest) (*GetResponse, error) {
	if req == nil {
		return nil, errors.New("nil request")
	}

	ctx, cancel := context.WithCancelCause(ctx)
	defer cancel(nil)

	var (
		wg       sync.WaitGroup
		infos    map[int]ProductInfo
		prices   map[int]ProductPrice
		infoErr  error
		priceErr error
	)

	wg.Add(2)
	go func() {
		defer wg.Done()
		infos, infoErr = s.storage.Products(ctx, req.IDs)
		if infoErr != nil {
			cancel(infoErr)
		}
	}()
	go func() {
		defer wg.Done()
		prices, priceErr = s.storage.Prices(ctx, req.IDs)
		if priceErr != nil {
			cancel(priceErr)
		}
	}()
	wg.Wait()

	if err := errors.Join(infoErr, priceErr); err != nil {
		return nil, fmt.Errorf("load products: %w", err)
	}

	result := make([]Product, 0, len(req.IDs))
	for _, id := range req.IDs {
		info, infoOK := infos[id]
		price, priceOK := prices[id]
		if !infoOK || !priceOK {
			return nil, fmt.Errorf(
				"incomplete product %d: info=%t price=%t",
				id, infoOK, priceOK,
			)
		}
		result = append(result, Product{ID: id, Info: info, Price: price})
	}

	return &GetResponse{Products: result}, nil
}
```

Здесь горутины записывают в разные result slots. После `wg.Wait()` их чтение синхронизировано, поэтому общий mutex не нужен. `cancel` помогает остановить sibling request, если storage соблюдает context. `errors.Join` сохраняет обе ошибки, если они успели возникнуть. Если нужен только первый error и automatic sibling cancellation, короче использовать `errgroup.WithContext`; `Wait` всё равно ждёт все functions, но возвращает первый ненулевой error.

## 8. Оценка ответов кандидата

| Блок | Что получилось | Что просело | Как усилить |
| --- | --- | --- | --- |
| Experience | Называет реальные продукты и stack | Мало измеримого результата и failure analysis | Подготовить 2 истории с baseline, decision, metric и trade-off |
| Patterns | Вспоминает Singleton | Не объясняет hidden dependency и test isolation | Учить patterns через проблему и contract, не список названий |
| Cache task | Понимает, что нужен отдельный слой | Долго угадывает Observer/Decorator | Нарисовать read/write path, invalidation и failure policy |
| SOLID/LSP | Связывает Go с interfaces | Путает embedding, inheritance и substitutability | Отработать assignability и behavioral contract |
| Testing | Знает `testing` и GoMock | Не знает состояние fork | Pin tools и понимать, где mock хуже integration test |
| Docker | Отличает image от container | Не знает multi-stage build | Собрать minimal Go image и объяснить CGO/certificates |
| Shell/Git | Узнаёт cron, merge/rebase | Не воспроизводит команды уверенно | Небольшая практика `find`, crontab, interactive rebase |

| Generics | Знает type switch/reflection | Принимает категоричный performance claim | Разделять spec/implementation, benchmark before claim |
| Code review | Замечает потерю context и позже no-wait | Пропускает races и duplicate products; опирается на AI | Review по invariants: lifecycle → synchronization → errors → data contract |

## 9. Использование AI во время интервью

По записи можно утверждать только следующее:

- на экране кандидата одновременно с видеовстречей открыты Grok 3 и ChatGPT 4.5;
- Grok показывает ответы по темам, которые в этот момент обсуждаются, включая patterns и B-tree/hash indexes;
- ChatGPT заранее получает инструкцию решать live-coding screenshots и выдавать готовый код;
- во время code review ChatGPT показывает исправление с `WaitGroup` и `Mutex`, а кандидат параллельно добавляет `WaitGroup` в Yandex Code;
- позже кандидат отправляет модели вопрос о pointers/values.

Нельзя утверждать по записи:

- что интервьюер видел второй экран;
- что использование AI было разрешено или запрещено правилами конкретного интервью;
- что каждый ответ кандидата взят из модели;
- что отказ произошёл именно из-за AI.

Перед своим интервью стоит заранее уточнить, разрешены ли docs, search и AI. Если AI запрещён, скрытая помощь — не только этический риск, но и плохой способ показать собственную reasoning chain. Если разрешён, нужно уметь проверить предложенный код: показанная ChatGPT версия добавляет mutex, но не исправляет отсутствие context у `Products`, duplicate response elements, money/time types и contract missing data.
