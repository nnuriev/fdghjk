# Авито — собеседование на 470к

## TL;DR

Это русскоязычное техническое интервью продолжительностью `01:50:01`. Внутри одной записи действительно совмещены platform/backend design, Go concurrency, PostgreSQL и алгоритмы. Сначала кандидат сравнивает синхронный RPC, очередь и Long Polling, затем проектирует producer–broker–consumer с разными гарантиями обработки, проводит code review кэша, рисует распределённый rate limiter, пишет `merge N channels`, разбирает деградацию PostgreSQL и online data fix, а в конце решает merge intervals.

Главная особенность собеседования — интервьюеры почти не принимают ответ по названию технологии. Они несколько раз уменьшают задачу до инварианта: где именно зафиксирован request, какая операция атомарна, что произойдёт между двумя writes, как не превысить общий RPS при нескольких replicas, почему прежний plan стал плохим, и можно ли безопасно обработать `10 млн` строк в живой таблице на `1 млрд` строк.

Кандидат получает оффер и хорошо показывает ход мысли: уточняет условия, признаёт отсутствие коммерческого опыта с ClickHouse, восстанавливает алгоритм merge intervals и замечает часть ошибок кэша. При этом в записи остаются серьёзные пробелы. Оба набранных Go-решения не компилируются; схема rate limiter выполняет read и increment неатомарно; `exactly once` сначала сводится к наличию idempotency key; Long Polling ошибочно связывается с обязательным созданием нового connection; определения normalization и denormalization сначала перепутаны.

### Что здесь проверяют

Выбор нельзя делать по правилу «фоновые задачи — Kafka, данные для frontend — HTTP». Сначала фиксируют contract:

- нужен ли ответ в пределах одного request deadline;
- допустимо ли принять работу, но выполнить позже;
- где находится durability boundary;
- нужны ли ordering, replay, backpressure и независимое масштабирование consumer;
- сколько стоит эксплуатация broker и какая failure domain появляется;
- как caller повторяет request и как система различает retry от новой business operation.

`202 Accepted` сам по себе не делает операцию надёжной. Если handler вернул success, а task жил только в process memory, crash потеряет работу. Нужна durable state machine: запись task/outbox в database до ответа, либо подтверждённая запись в broker. И HTTP, и broker могут быть synchronous на transport boundary; различие создают ownership, durability, buffering, replay и processing model.

Long Polling удерживает request до event или timeout, после ответа client обычно немедленно посылает следующий request. Persistent HTTP connection можно переиспользовать: тезис кандидата «на каждый poll обязательно новый connection» неверен. WebSocket создаёт message-oriented bidirectional channel после upgrade; Long Polling остаётся последовательностью обычных HTTP request/response и проще проходит через существующие HTTP infrastructure, auth, observability и restrictive networks. HTTP/3 — mapping HTTP semantics поверх QUIC; он не превращает request/response в WebSocket и не отменяет application semantics Long Polling.

### Ментальная модель guarantees

Guarantee следует описывать относительно наблюдаемого эффекта, а не одним словом:

| Contract | Порядок действий | Failure mode |
| --- | --- | --- |
| At-most-once processing | commit/ack до обработки либо без retry | сообщение можно потерять, duplicate processing не возникает |
| At-least-once delivery | обработка, затем ack; timeout вызывает retry | сообщение не теряется при поддерживаемых failures, но side effect может повториться |
| Effectively-once effect | at-least-once delivery + atomic idempotency claim/effect | duplicate delivery остаётся, business effect применяется один раз |
| Kafka EOS | read Kafka → transactionally write Kafka + offsets → read committed | atomicity ограничена Kafka transaction boundary; внешняя система должна сотрудничать |

Наличие `event_id` ещё не даёт exactly-once. Небезопасная последовательность `SELECT id → side effect → INSERT id` имеет два окна: concurrent consumers оба увидят отсутствие key, а crash между effect и отметкой приведёт к повтору. Надёжная реализация использует unique constraint/conditional write в той же transaction, inbox state machine или делает сам business transition идемпотентным.

Кандидат верно назвал три режима и предложил idempotency key, но смешал «доставка больше одного раза, обработка один раз» с общей гарантией exactly-once. Сильный ответ должен сразу назвать boundary и crash points.

### ClickHouse: где ответ кандидата остановился слишком рано

Предложение хранить dedup table в PostgreSQL, а затем переливать данные в ClickHouse убирает дорогой point lookup из ClickHouse, но создаёт dual-write. Если process упал после отметки в PostgreSQL и до insert в ClickHouse, event потеряется; если порядок обратный, возможен duplicate. Нужны durable outbox/status, повторяемая delivery state machine и reconciliation.

Для idempotent retry одного и того же insert ClickHouse поддерживает deduplication token/hash в MergeTree-family tables, но это не универсальная unique constraint по произвольному event ID. `ReplacingMergeTree` удаляет logical duplicates по sorting key во время background merge; до merge запрос может видеть обе версии, поэтому нужны `FINAL` или `argMax`-pattern с их ценой. Выбор зависит от того, требуется ли ingest retry safety, latest-state model или строгая uniqueness на момент business write.

### Код с экрана и полный review

```go
var cache = make(map[string]string)

func GetOrCreate(key, value string) string {
	var m sync.Mutex
	m.Lock()
	value = cache[key]
	m.Unlock()
	if value != "" {
		return value
	}
	m.Lock()
	cache[key] = value
	m.Unlock()
	return value
}

func Get(key string) string {
	var m sync.Mutex
	m.Lock()
	v := cache[key]
	m.Unlock()
	return v
}
```

Ошибки здесь независимы друг от друга:

1. Каждый call создаёт собственный mutex. Два goroutine блокируют разные locks, поэтому общей mutual exclusion нет.
2. Lookup перезаписывает parameter `value`. На miss функция затем кладёт в map пустую строку, а не переданный fallback.
3. Проверка `value != ""` делает пустую строку неотличимой от отсутствующего key. Нужен `v, ok := cache[key]`.
4. Read и последующий write разделены: несколько caller могут одновременно пройти miss. Для дешёвого готового value это приводит к лишней работе; для фабрики — к duplicate side effects/stampede.
5. Global mutable state осложняет lifecycle, tests, capacity policy и invalidation.
6. `80/20` не доказывает преимущество `RWMutex`: оно зависит от contention и длины critical section.

Минимальное исправление для уже вычисленного fallback:

```go
type Cache struct {
	mu   sync.RWMutex
	data map[string]string
}

func (c *Cache) GetOrCreate(key, fallback string) string {
	c.mu.RLock()
	v, ok := c.data[key]
	c.mu.RUnlock()
	if ok {
		return v
	}

	c.mu.Lock()
	defer c.mu.Unlock()
	if v, ok := c.data[key]; ok {
		return v
	}
	c.data[key] = fallback
	return fallback
}
```

Если value строится медленно, держать global write lock во время network call тоже плохо. Тогда нужны per-key coordination или `singleflight`, timeout/cancellation, negative caching и явная политика cache stampede.

Документация Go 1.26.5 прямо называет `sync.Map` specialized type. Основные оптимизированные случаи: key записывается один раз и много читается либо goroutines работают с непересекающимися sets ключей. Для обычного typed state `map + lock` сохраняет type safety и позволяет защищать дополнительные invariants. Формулировка кандидата «на больших core counts sync.Map lock-free и поэтому выигрывает» не является контрактом API.

### Что нарисовал кандидат

Схема содержит load balancer, несколько pods, отдельный rate-limiter service, PostgreSQL с policies и sharded/replicated Redis. Ключ постепенно превращается в `timestamp:service`; request path читает policy, считает requests за последнюю секунду, сравнивает `count + 1` с RPS, записывает новый request и возвращает `429` при превышении. Для очистки предложен TTL.

Это хорошая стартовая декомпозиция, но не законченная design:

- `read count → decide → write` неатомарен;
- policy read из PostgreSQL на каждый request создаёт лишнюю latency/failure dependency;
- хранение каждой заявки и scan последней секунды дорого по memory и CPU;
- fixed-second aggregation имеет boundary burst;
- asynchronous replica не подходит для admission decision со строгим лимитом;
- один popular service key упирается в один Redis shard;
- равномерное распределение load balancer не гарантирует строгий per-pod rate;
- не выбран contract при Redis outage и partition.

### Production-grade вариант

Сначала нужно зафиксировать SLO, например: «не более `R` успешно допущенных requests на service за любой rolling interval `1 s`, burst не более `B`, решение global по region, при недоступности Redis — fail-closed для платного downstream». После этого алгоритм становится следствием contract.

Для общего API чаще подходит token bucket: state `{tokens, last_refill}`, server-side atomic script вычисляет refill, проверяет token и списывает его. Burst задаётся capacity, средняя скорость — refill rate. Для строгого rolling window нужен sorted set/log: атомарно удалить события старше cutoff, посчитать cardinality, при наличии quota добавить request ID и обновить TTL. Это точнее, но memory равна числу событий в window. Fixed window дешевле всего, но имеет boundary burst.

Redis script/Function должен атомарно выполнить `prune/refill → check → update → TTL → return {allowed, remaining, retry_after}`. Решение читается с primary/leader. При multi-region или failover строгая global quota конфликтует с availability: во время partition независимые regions не могут одновременно сохранять hard upper bound без coordination. Возможные compromises — region budgets, leased quotas и явно ограниченный overshoot.

### Решение кандидата не компилируется

Общая схема fan-in у кандидата верная: по worker на input, `WaitGroup`, отдельный closer. Но output создаётся как receive-only channel:

```go
result := make(<-chan int)
```

После этого и `result <- val`, и `close(result)` запрещены type system. Проверка официальным Go Playground на Go 1.26.5 дала:

```text
invalid operation: cannot send to receive-only channel <-chan int result
invalid operation: cannot close receive-only channel result
```

Нужно создать bidirectional channel и сузить direction только в return type. Кроме того, исходный вариант течёт, если downstream прекращает чтение: workers навсегда блокируются на send, `Wait` не завершается, output не закрывается.

Проверенный reference variant:

```go
func merge[T any](ctx context.Context, inputs ...<-chan T) <-chan T {
	out := make(chan T)

	var wg sync.WaitGroup
	wg.Add(len(inputs))
	for _, input := range inputs {
		go func() {
			defer wg.Done()
			for {
				var v T
				var ok bool
				select {
				case <-ctx.Done():
					return
				case v, ok = <-input:
					if !ok {
						return
					}
				}

				select {
				case <-ctx.Done():
					return
				case out <- v:
				}
			}
		}()
	}

	go func() {
		wg.Wait()
		close(out)
	}()
	return out
}
```

Он скомпилирован и выполнен на Go 1.26.5. Два `select` нужны, чтобы cancellation могла остановить и ожидание input, и блокированный send. Порядок values внутри одного input сохраняется, но общий порядок разных channels nondeterministic.

Без `WaitGroup` workers могут отправлять по одному token в buffered `done := make(chan struct{}, len(inputs))`, а closer получить ровно `len(inputs)` tokens и закрыть output. Atomic counter сам по себе недостаточен: goroutine, которая видит zero, должна иметь корректный happens-before относительно всех sends и владеть закрытием.

Строка `channel := channel` была нужна для типичной closure bug при старой semantics loop variables. Начиная с language version Go 1.22 каждая iteration создаёт новые variables для объявленных loop variables, поэтому здесь она избыточна, хотя безвредна.

### Диагностика деградации

![[90 Вложения/Telegram Собесы/Авито — 2026-04-20 — 470к/Кадры/06-slow-query-01-21-40.jpg]]

Правильный порядок — `сохранить сервис → локализовать bottleneck → объяснить изменение → исправить root cause`.

1. Проверить blast radius и при необходимости применить bounded mitigation: load shedding, timeout, rollback, cache/read replica либо временное снижение concurrency.
2. Сравнить normalized query в `pg_stat_statements`: calls, mean/total time, rows, block reads, temp spill; посмотреть wait events, locks, CPU, I/O и cache hit.
3. Снять plan с representative parameters. `EXPLAIN (ANALYZE, BUFFERS, WAL, SETTINGS)` действительно исполняет statement, поэтому на production нужен безопасный query, read-only transaction/replica либо ограниченная копия.
4. Сравнить planned и actual rows на каждом node. Большое расхождение указывает на stale/too-coarse statistics, skew или correlation, но не доказывает единственную причину.
5. Исправить: `ANALYZE`, per-column statistics target, `CREATE STATISTICS` для correlation/MCV/ndistinct, sargable predicate, более подходящий index, query rewrite или autovacuum/analyze tuning.

Точнее: изменившаяся distribution и ошибочная cardinality estimate заставили planner выбрать плохой plan; сам index мог остаться исправным.

### VACUUM без двух распространённых ошибок

![[90 Вложения/Telegram Собесы/Авито — 2026-04-20 — 470к/Кадры/07-vacuum-01-20-00.jpg]]

Обычный `VACUUM` помечает пространство dead row versions доступным для повторного использования, обслуживает visibility map и выполняет freeze против XID wraparound. Обычно он не уменьшает файл и не возвращает space OS. `VACUUM FULL` переписывает table, может физически сжать файл, требует дополнительное место на rewrite и `ACCESS EXCLUSIVE` lock.

Нужно различать два типа statistics. `VACUUM` обновляет приблизительные `reltuples/relpages`; `ANALYZE` собирает distribution statistics в `pg_statistic`. Поэтому фразы «VACUUM вообще не влияет на statistics» и «обычного VACUUM достаточно для селективности planner» обе неточны.

### Normalization, projection и TOAST

Normalization разносит facts по relations согласно functional dependencies и уменьшает redundancy/update anomalies. Denormalization намеренно дублирует или prejoins state ради чтения, принимая цену consistency, write amplification и reconciliation. В интервью кандидат сначала назвал складывание всех данных в одну строку normalization, то есть перепутал направления, затем исправился на примере отдельной таблицы адресов.

Отдельный `address_id` полезен, если address — самостоятельная entity, переиспользуется, имеет lifecycle и требует referential integrity. Но совпадающая строка не всегда означает один адрес: history, apartment, geocoding precision и ownership могут различаться.

Утверждение «row database всё равно читает все columns» слишком широкое. PostgreSQL читает heap pages и tuple headers с page granularity, поэтому narrow projection не превращает storage в column store. Но executor не обязан materialize/return unused expressions, index-only scan может обойти heap при выполнении условий visibility/coverage, а большие TOAST values хранятся out-of-line и подтягиваются только при выборе/использовании. `SELECT` только нужных columns всё равно имеет смысл.

### Надёжный rollout

1. Сначала остановить источник плохих writes: исправить code path или выпустить backward-compatible dual logic.
2. Зафиксировать predicate и invariant результата. Повторный запуск не должен менять уже исправленную row.

6. Задать `lock_timeout`, `statement_timeout`, retry с jitter и pause switch. Throttle по request latency, DB CPU/I/O, WAL bytes, replica lag, locks и autovacuum pressure.

### Инвариант и оценка

После сортировки по `start`, а затем по `end`, output содержит непересекающиеся интервалы, полностью покрывающие обработанный prefix. У текущего interval нет шанса пересечь что-то раньше последнего output interval: последний имеет самый правый relevant boundary. Для closed intervals новый interval начинается отдельно, если `last.end < current.start`; иначе `last.end = max(last.end, current.end)`.

Sorting занимает `O(n log n)`, scan — `O(n)`, итог — `O(n log n)`. Result в худшем случае `O(n)`. Sorting input in place экономит copy, но меняет caller-owned порядок; contract должен это разрешать.

Кандидат сам дошёл до корректного общего overlap predicate для closed intervals:

```text
max(a.start, b.start) <= min(a.end, b.end)
```

и корректных merged bounds `min(starts), max(ends)`. Однако код с финального кадра не компилируется:

![[90 Вложения/Telegram Собесы/Авито — 2026-04-20 — 470к/Кадры/10-merge-intervals-code-01-49-10.jpg]]

- `func mergeMeetings(input [][]int)` не объявляет return type, но возвращает `result`;
- импортирован только `fmt`, хотя вызывается `sort.Slice`;
- встроенные `min`/`max` требуют language version Go 1.21+;
- main не вызывает решение, поэтому пример не проверен;
- unmerged inner slices переиспользуют storage caller, а sorting меняет input.

Go Playground 1.26.5 подтвердил первые две compiler errors. Проверенный reference variant копирует input и явно проверяет contract:

```go
func mergeMeetings(input [][]int) [][]int {
	if len(input) == 0 {
		return nil
	}

	intervals := make([][2]int, 0, len(input))
	for _, p := range input {
		if len(p) != 2 || p[0] > p[1] {
			panic("invalid interval")
		}
		intervals = append(intervals, [2]int{p[0], p[1]})
	}

	sort.Slice(intervals, func(i, j int) bool {
		if intervals[i][0] == intervals[j][0] {
			return intervals[i][1] < intervals[j][1]
		}
		return intervals[i][0] < intervals[j][0]
	})

	out := make([][]int, 0, len(intervals))
	for _, cur := range intervals {
		if len(out) == 0 || out[len(out)-1][1] < cur[0] {
			out = append(out, []int{cur[0], cur[1]})
			continue
		}
		if cur[1] > out[len(out)-1][1] {
			out[len(out)-1][1] = cur[1]
		}
	}
	return out
}
```

Для input `[[1,3],[2,6],[8,10],[15,18]]` проверенный output — `[[1 6] [8 10] [15 18]]`.

## Итоговая оценка ответов кандидата

### Что получилось хорошо

- Кандидат рассуждает вслух и уточняет неизвестные requirements вместо мгновенного выбора technology.
- Правильно называет decoupling producer/consumer, at-least-once duplicates и idempotency key как часть решения.
- В code review замечает локальный mutex, необходимость struct, `RWMutex` и comma-ok lookup.
- В rate limiter выделяет enforcement point, durable/config state, Redis sharding, TTL и несколько алгоритмов.
- В backfill предлагает batches, transaction boundaries, time index и сначала исправить новые writes.
- В merge intervals восстанавливает `O(n log n)` решение и сам исправляет overlap predicate при ручной трассировке.

### Что стоило исправить до следующего интервью

- Всегда называть boundary guarantee и разбирать crash между каждой парой operations.
- Не утверждать, что Long Polling создаёт новый transport connection на каждый request; HTTP keep-alive и HTTP/3 connection reuse отделены от request lifecycle.
- Не называть `sync.Map` универсально lock-free или лучшим «с большого числа cores»; выбирать по workload и benchmark.
- В system design раньше фиксировать quota dimension, burst, failure policy и atomic state transition.
- При live coding запускать compiler раньше. Оба решения имели structural errors, которые выявляет один Run.
- В PostgreSQL связывать estimates, statistics и plan choice причинно, а не говорить «index испортился».
- Не путать normalization с denormalization и не переносить OLTP row-store heuristics на все storage/queries.
- В online migration обязательно произнести idempotency, concurrent update protection, checkpoint, throttling и validation.
