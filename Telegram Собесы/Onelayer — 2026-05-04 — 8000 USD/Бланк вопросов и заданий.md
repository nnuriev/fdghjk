package main

import "fmt"

type Queue struct {
	// TODO: implement me
}

func NewQueue(size int) Queue {
	// TODO: implement me
	return Queue{}
}

func (q Queue) Push(v int) {

}

	panic("implement me!")
}

func main() {
	q := NewQueue(3)
	q.Push(1)
	q.Push(2)
	q.Push(3)
	fmt.Println(q.Pop()) // expected 1
	fmt.Println(q.Pop()) // expected 2
	fmt.Println(q.Pop()) // expected 3

Контракт был проговорён явно:

- `size` задаёт границу;
- `Push` в полную очередь блокирует вызывающую goroutine до чужого `Pop`;
- `Pop` из пустой очереди блокирует вызывающую goroutine до чужого `Push`;
- разрешена вся standard library, запрещены поиск, AI и внешние подсказки.

## Базовое решение кандидата

```go
type Queue struct {
	ints chan int
}

func NewQueue(size int) Queue {
	ints := make(chan int, size)
	return Queue{ints: ints}
}

func (q Queue) Push(v int) {
	q.ints <- v
}

func (q Queue) Pop() int {
	return <-q.ints
}
```

![[90 Вложения/Telegram Собесы/Onelayer — 2026-05-04 — 8000 USD/Кадры/00-54-00-channel-решение.jpg]]

Это минимальное и корректное решение поставленной версии задачи. Buffered channel уже содержит очередь, capacity, thread-safe synchronization и нужный backpressure: send разрешён, пока в buffer есть место; receive — пока есть элемент. Отдельные goroutines внутри методов, mutex или condition variables не нужны. Кадр показывает вывод `1`, `2`, `3`; независимый запуск в официальном Go Playground дал тот же результат.

Значимый нюанс: FIFO гарантирует порядок значений внутри channel. Если несколько goroutines конкурентно начинают send, API не обещает прикладному коду детерминированный порядок между этими конкурентными вызовами. Это не ломает очередь, но мешает приписывать ей более сильную гарантию «по wall-clock времени вызова».

## Follow-up: отменяемый `Push`

Интервьюер предложил изменить API так, чтобы бесконечно ожидающий вызов можно было отменить. Кандидат правильно выбрал `context.Context` первым параметром, но путь к `select` прошёл через две ошибки.

### Попытка 1: ждать cancellation до send

На первом кадре receive из `ctx.Done()` и send существуют последовательно. В такой структуре функция либо бессмысленно ждёт отмены до попытки положить значение, либо после отдельной проверки всё равно входит в неотменяемый blocking send.

![[90 Вложения/Telegram Собесы/Onelayer — 2026-05-04 — 8000 USD/Кадры/00-56-00-первая-попытка-context.jpg]]

Неверное предположение: «если проверить context перед send, этого достаточно». Симптом: cancellation, произошедший после проверки, не разблокирует `q.ints <- v`. Причина: в момент блокировки код больше не выполняет receive из `ctx.Done()`. Исправление: обе communication operations должны конкурировать внутри одного `select`.

### Попытка 2: `default`, затем блокирующий send

```go
func (q Queue) Push(ctx context.Context, v int) {
	select {
	case <-ctx.Done():
		return
	default:
		q.ints <- v
	}
}
```

![[90 Вложения/Telegram Собесы/Onelayer — 2026-05-04 — 8000 USD/Кадры/00-58-00-ошибка-default.jpg]]

`default` делает только сам выбор неблокирующим. Если context ещё не отменён, `select` немедленно выходит в `default`, после чего обычный send может блокироваться сколько угодно. Поздняя отмена уже не наблюдается. Это классическая ошибка `select`: нужно ждать готовность самой операции, а не выбирать ветку, внутри которой она потом выполняется.

### Итог кандидата

```go
func (q Queue) Push(ctx context.Context, v int) {
	select {
	case <-ctx.Done():
		return
	case q.ints <- v:
	}
}
```

![[90 Вложения/Telegram Собесы/Onelayer — 2026-05-04 — 8000 USD/Кадры/01-00-00-cancellable-send.jpg]]

Теперь `select` блокируется, пока возможен хотя бы один исход: context отменён или в очереди появилось место. Вторая ветка не означает «сначала выбрать, затем попытаться отправить» — send является самой communication operation, готовность которой участвует в выборе.

Решение выполняет озвученный контракт, но API теряет результат: `return` не сообщает вызывающему коду, был элемент помещён или операция отменена. Практическая версия должна возвращать `error`:

```go
func (q Queue) Push(ctx context.Context, v int) error {
	select {
	case <-ctx.Done():
		return ctx.Err()
	case q.ints <- v:
		return nil
	}
}
```

Если нужен симметричный отменяемый `Pop`:

```go
func (q Queue) Pop(ctx context.Context) (int, error) {
	select {
	case <-ctx.Done():
		return 0, ctx.Err()
	case v := <-q.ints:
		return v, nil
	}
}
```

Этот усиленный вариант проверен в официальном Go Playground: при capacity `1`, уже занятой первым элементом, отменённый второй `Push` вернул `context.Canceled`, а последующий `Pop` сохранил первый элемент; наблюдаемый вывод — `true 1`.

## Почему cancellation не имеет абсолютного приоритета

Если `ctx.Done()` уже закрыт и send одновременно готов, спецификация `select` выбирает одну из готовых communication operations uniform pseudo-random. Поэтому формулировка «после отмены значение точно не будет помещено» для приведённого кода слишком сильная. Его гарантия другая: заблокированный вызов получает возможность завершиться по cancellation; когда оба исхода уже возможны, допустим любой.

Если бизнес-контракт требует жёстко не начинать операцию после замеченной отмены, можно сделать предварительную проверку context, но между ней и следующим `select` всё равно остаётся race изменения состояния. Для truly transactional semantics нужны более сильный protocol и явное определение точки commit, а не только приоритет ветки.

## Edge cases, которые логично обсудить

Эти вопросы в записи не задавались; это ближайшие follow-up, которыми проверяется полнота production-модели.

- `size == 0` создаёт unbuffered channel: это synchronous handoff, а не очередь с накоплением.
- `size < 0` вызывает panic в `make(chan int, size)`; constructor может валидировать аргумент и возвращать ошибку.
- zero value `Queue{}` содержит nil channel; `Push` и `Pop` блокируются навсегда. Если нужна useful zero value, API придётся проектировать иначе.
- Копирование `Queue` с value receiver копирует channel descriptor, но все копии ссылаются на один channel. Для текущей структуры это безопасно; добавление mutex или mutable metadata потребует пересмотреть copy semantics и receivers.
- Контракт не определяет `Close`, поведение оставшихся readers и ошибку send после close. Без ownership-модели добавлять публичный `Close` опасно: concurrent close/send способен вызвать panic.
- `Pop(ctx)` с результатом `(int, error)` должен отличать реальный нулевой элемент от отмены.
- `context.Background().Done()` возвращает nil; соответствующий `select case` отключён, а channel operation продолжает работать обычным образом.
- Для generic queue в Go `1.18+` достаточно заменить `int` на type parameter `T`; это не меняет synchronization semantics.

## Как проверить решение без ненадёжных sleeps

1. FIFO: последовательно положить `1, 2, 3`, получить ровно `1, 2, 3`.
2. Full queue: заполнить capacity, запустить ещё один `Push`, доказать через отдельный completion channel, что он не завершился, затем выполнить `Pop` и дождаться его завершения.
3. Empty queue: симметрично запустить `Pop`, затем `Push` и проверить полученное значение.
4. Cancellation: заполнить очередь, отменить context заблокированного `Push`, получить `context.Canceled` и убедиться, что исходный элемент не потерян.
5. Race: прогнать concurrent producers/consumers с `go test -race` в доступной Go-среде.

В самом репозитории Go toolchain недоступен, поэтому локальные тесты не запускались и инструмент не устанавливался. Два минимальных случая выполнены через официальный Go Playground; race detector там не проверялся.

# Разбор проектного ответа

## Что было сильным

- Кандидат выбрал проект с несколькими слоями сложности: data ownership, schema selection, межкомандная координация, initial backfill, asynchronous updates и migration without downtime.
- По уточнениям смог назвать Kafka, полную запись вместо diff и PostgreSQL.
- Связал работу с наблюдаемым бизнес-результатом: меньше ручной маршрутизации и примерно `20%` снижения соответствующей нагрузки.
- Не пытался строить queue вручную поверх slice и mutex, а узнал готовую channel semantics.
- После объяснения ошибки с `default` понял, что send должен быть самостоятельным `case`.

## Где ответу не хватило глубины

- В проекте миграции не раскрыты idempotency, duplicate/out-of-order events, schema evolution, retries, poison messages, reconciliation и способ доказать consistency после cutover.
- «Событие содержит всю запись, а мы сравниваем» не объясняет versioning и защиту от перезаписи нового состояния старым событием.
- PostgreSQL назван «основной базой команды», но не разобраны workload и критерии выбора.
- В техническом конфликте кандидат не привёл реальную ситуацию, не сформулировал decision criteria и способ дешёвой проверки спорного решения.
- Слово «mutex внутри channel» передаёт интуицию thread safety, но это слабая модель. Контракт задают channel communication, blocking rules и happens-before guarantees; внутренняя реализация runtime не нужна для объяснения корректности.
- Cancellation API без возвращаемого результата скрывает, произошёл enqueue или отмена.
