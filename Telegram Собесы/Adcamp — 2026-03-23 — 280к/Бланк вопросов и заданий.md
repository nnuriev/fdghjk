# Adcamp — собеседование Golang с оффером 280к

## TL;DR

Это русскоязычное интервью продолжительностью `00:42:25`: около восьми минут занимает опыт кандидата, девятнадцать минут — теоретический Go-блок, остальное — рассказ о продукте и вопросы кандидата. Live coding, алгоритмической задачи, code review, SQL и System Design здесь нет. Техническая часть проверяет goroutines и G/M/P, конкурентность, channels, stack/heap и GC, `sync.Pool`, экспериментальные arenas, interfaces, closures, `context`, distributed tracing и виды тестирования.

Блок короткий, но местами обманчиво глубокий. Интервьюер просит перечислить состояния channel, а следом спрашивает про `sync.Pool` во время GC, отключение GC, третью модель размещения памяти, сравнимость interface values, typed nil, method sets, cancellation cause и сквозной trace через микросервисы. Здесь мало заучить отдельные фразы: после определения нужен механизм, затем граничный случай и практическое следствие.

Кандидат получил оффер согласно подписи публикации. Большую часть терминов он узнал, но несколько ответов были неточными: назвал channel «буферизованной кольцевой очередью» вообще, смешал concurrency с борьбой за один ресурс, описал `sync.Pool` как обычную достижимую память, перепутал правило method sets и представил arenas как стабильную третью область памяти. В самой беседе также прозвучало неверное версионное утверждение: пакет `arena` не стал обычной частью стандартной библиотеки Go 1.24 и не был затем удалён в Go 1.25. На Go `1.26.5` исходники по-прежнему защищены build constraint `goexperiment.arenas`; API экспериментальный, а исходное предложение по arenas отложено на неопределённый срок.

Без видеоряда легко упустить важный контекст. В начале техблока кандидат открывает индекс собственных заметок; во время вопросов о channels на экране видна таблица операций над open, closed и nil channel; в блоке о памяти он ищет упомянутую интервьюером статью и открывает заметку `Heap или Stack`. Это наблюдение о содержимом экрана, а не вывод о допустимости или причине использования подсказок.

# Модель хорошего ответа на ключевые вопросы

## Goroutine, concurrency и scheduler

Короткий ответ: goroutine — независимо планируемое Go runtime вычисление с динамически растущим stack. Scheduler multiplex-ит goroutines `G` на OS threads `M`; право исполнять Go code представляет `P`, число которых связано с `GOMAXPROCS`. Concurrency описывает композицию нескольких independently progressing tasks, parallelism — их одновременное выполнение. Поэтому много I/O goroutines полезны и на небольшом числе P, а CPU speedup требует доступных P/cores и отсутствия dominating contention.

Точное продолжение: blocking runtime operation обычно park-ит G и освобождает M/P для другой работы; blocking syscall может отделить M от P, после чего P продолжит работу с другим M. Это не отменяет OS scheduler, syscalls и context switching. Стоимость goroutine ниже thread, но leak тысячи goroutines всё равно удерживает stacks, timers, channel references и другие resources.

## Channels

Короткий ответ: unbuffered channel синхронизирует sender и receiver rendezvous-операцией; buffered channel допускает до `cap(ch)` неполученных values. Send в closed channel и close закрытого или nil channel panic. Send и receive для nil channel блокируются навсегда. После close receiver сначала drain-ит buffer, затем получает zero value с `ok=false`.

Практическое правило: close принадлежит стороне, которая знает, что sends закончились. Buffer задаёт bounded queue и backpressure policy, но не лечит overload. Каждый pipeline должен иметь явные ownership, termination и cancellation paths; иначе downstream exit оставит upstream на blocking send.

## Memory, GC и `sync.Pool`

Короткий ответ: compiler escape analysis решает, может ли allocation жить на goroutine stack; объект, чей lifetime выходит за stack frame или который compiler не может безопасно разместить там, попадает в heap. GC Go трассирует достижимый object graph и делает недостижимую heap memory доступной allocator. `sync.Pool` уменьшает allocation pressure для временных объектов, но не гарантирует retention: item может исчезнуть автоматически, а `Get` вправе не вернуть ранее положенный value.

Практическое правило: сначала benchmark и profiles, затем оптимизация. `GOGC=off` не является обычным способом ускорить service; при долгой работе live/garbage heap растёт, а configured memory limit может всё равно инициировать collections. Arenas на Go `1.26.5` остаются experiment и не должны описываться как стабильная третья область памяти общего назначения.

## Interfaces

Короткий ответ: interface value концептуально хранит dynamic type и dynamic value. Оно равно `nil`, только когда отсутствуют оба; interface с typed nil pointer не nil. Сравнение interface values допустимо лишь пока dynamic values comparable, иначе runtime panic. Method set `T` содержит methods с receiver `T`; method set `*T` — methods с receiver `T` и `*T`. Следовательно, pointer-only required method реализует interface только для `*T`.

Практическое правило: interface вводят на стороне consumer вокруг нужного поведения. Dependency Inversion — зависимость high-level policy и low-level details от abstraction; dependency injection — способ передать конкретную реализацию. Ни один из принципов не требует interface для каждого struct.

## `context` и tracing

Короткий ответ: `Context` переносит deadline, cancellation и request-scoped values через call chain. Производная функция возвращает cancel; её нужно вызвать, чтобы освободить связанные resources. `WithCancelCause` позволяет сохранить domain-relevant причину, которую читает `context.Cause`. Для distributed trace trace/span context нужно extract-ить из входящего carrier, использовать как parent и inject-ить в исходящий carrier; TraceID объединяет spans, SpanID идентифицирует конкретную операцию.

Практическое правило: cancellation cooperative — каждый blocking boundary должен её проверять. OpenTelemetry создаёт и экспортирует telemetry, а Jaeger может хранить и показывать traces; без propagation одна пользовательская операция распадётся на несвязанные локальные spans.
