### Sizes: отвечать нужно слоями

Спецификация задаёт точные sizes для `int8/16/32/64`, `uint8/16/32/64`, `float32/64`, complex types и aliases `byte`/`rune`. Размеры `int`, `uint`, `uintptr`, pointer, `bool`, string descriptor, slice descriptor и interface representation зависят от implementation и architecture. Поэтому сильный ответ выглядит так:

1. назвать semantic contract;
2. отдельно привести типичную реализацию `gc` на `amd64`;
3. проверить конкретный binary через `unsafe.Sizeof`, `unsafe.Alignof`, `GOARCH` и, если layout критичен, compile-time assertions/tests.

Для стандартного `gc` на `amd64` pointer обычно занимает `8` bytes, string value — `16` bytes, а `struct{ S string }` без дополнительных полей также `16` bytes. Но backing bytes находятся отдельно: `unsafe.Sizeof(s)` не измеряет retained memory строки. `bool` в обычной реализации занимает byte, однако language guarantee формулируется не этим числом.

### Строка: bytes, runes и пользовательские символы

По specification string — immutable sequence of bytes. `len("Привет")` возвращает число bytes UTF-8, а не число русских букв. `range` декодирует UTF-8 code points и выдаёт byte offset; `utf8.RuneCountInString` считает runes. Ни один из этих способов не считает grapheme clusters: комбинация base character + combining mark или emoji sequence может восприниматься пользователем как один символ и состоять из нескольких runes.

Кандидат правильно вспомнил `utf8.RuneCountInString` и различил bytes/runes, но определил string как «массив bytes со `len`». Это полезная implementation model, однако array является value type с length в type, а string — отдельный immutable type; внутренний descriptor не является публичным контрактом.

### Arrays, slices и comparability

Array values self-contained и копируются целиком при обычной передаче по value. Slice — descriptor диапазона underlying array; копирование descriptor не копирует elements. Array сравним, если его element type comparable; lengths должны совпасть уже на уровне type. Slice нельзя сравнивать с другим slice даже одинакового типа, но можно сравнить с `nil`.

Map key должен быть comparable. Interface сам синтаксически допустим как key, но hash/equality выполняются по dynamic type. Если туда положить slice, map или function, программа получит run-time panic. Кандидат этот edge case назвал верно.

### Map: важная версионная граница

Начиная с Go `1.24`, built-in map стандартного toolchain основан на Swiss Tables. В текущей реализации Go `1.26.5` group содержит `8` slots и control word; кандидат говорил о `16` elements, что не соответствует текущему source. До Go 1.24 действовала другая bucket-based implementation. Ни старые buckets, ни новые groups не следует выдавать за Go language guarantee: application code обязан зависеть от semantics map, а не от runtime layout.

### G/M/P и stack/heap без ложной точности

`G` хранит состояние gorутины, `M` представляет OS thread, `P` — resources и право исполнять user Go code. Runnable goroutines обычно находятся в local queue P; idle P может steal часть work у другого P. Runtime создаёт и паркует M по мере blocking и scheduling needs. Детали fairness, preemption intervals и queue polling — implementation details и меняются между releases.

Go не даёт разработчику direct stack/heap placement syntax. Compiler делает escape analysis и сохраняет value на stack, если может доказать безопасный lifetime; иначе allocation попадает в heap. Возврат `&local` безопасен: compiler продлевает lifetime, а не оставляет dangling pointer. Но правило «любой возвращённый pointer всегда heap» слишком сильное после inlining и optimization. Проверять конкретную сборку нужно через `go build -gcflags=-m=3`, а real allocation pressure — profiles/benchmarks.

Кандидат правильно назвал escape analysis и направление команды, но привязал heap к несуществующему универсальному size threshold `10/20 MB`. Большой local value действительно может быть вынесен из stack из-за implementation limits, однако число не является языковым invariant и не должно быть частью заученного ответа.

### Channel state table

| Состояние | Send | Receive | `close` |
| --- | --- | --- | --- |
| `nil` | блокируется навсегда | блокируется навсегда | panic |
| open unbuffered | ждёт matching receive | ждёт matching send | success владельца |
| open buffered | ждёт free slot при полном buffer | ждёт value при пустом buffer | success владельца |
| closed | panic | buffered values, затем zero value и `ok=false` | panic |

Сам по себе blocking operation не означает runtime deadlock: panic `all goroutines are asleep` возникает, когда program не может продолжить работу. Поэтому фраза «запись в полный channel — deadlock» верна только если никто больше не способен освободить slot или отменить operation.

`sync.Map` documentation выделяет два характерных workload: key пишется один раз и читается много раз либо goroutines работают с disjoint key sets. В остальных случаях обычная typed map с отдельным mutex часто даёт более ясные invariants. Получение `any` и type assertion — не то же самое, что обязательный reflection call.

### Как развивалось решение кандидата

Сначала кандидат предложил generator `[]string → chan string` и фиксированное число workers. Затем заменил его на `errgroup.WithContext` + `SetLimit`, запустил одну errgroup task на поисковик, результаты направил в unbuffered channel, а minimum стал считать в отдельной goroutine.

Идея bounded fan-out и cancellation была правильной. Проблема появилась на границе ownership: `errgroup.Wait()` ждёт только producer tasks, но не отдельный aggregator. Наблюдаемая финальная версия по смыслу выглядела так:

```go
g, ectx := errgroup.WithContext(ctx)
g.SetLimit(connMaxLimit)

results := make(chan elem)
var best elem

for _, searcher := range searchers {
	g.Go(func() error {
		d, err := testSearcher(ectx, searcher)
		if err != nil {
			return err
		}
		select {
		case <-ctx.Done():
			return errors.New("context canceled")
		case results <- elem{name: searcher, respTime: d}:
			return nil
		}
	})
}

go func() {
	for r := range results {
		if best.respTime == 0 || r.respTime < best.respTime {
			best = r
		}
	}
}()

if err := g.Wait(); err != nil {
	close(results)
	return "", 0, err
}
return best.name, best.respTime, nil
```

### Полный code review

1. На success path `results` не закрывается. Aggregator навсегда остаётся в `range results`: это goroutine leak.
2. `g.Wait()` не ждёт aggregator. Main goroutine читает `best`, пока aggregator может его менять: это data race и возможный неполный minimum.
3. На error path channel закрывается после завершения producers, но aggregator всё равно не join-ится перед чтением/return.
4. В финальном кадре остаётся `var maxRespTime time.Duration`, которая не используется. Go compiler отвергнет файл.
5. В видимой import-секции были только `context` и `time`, хотя код использует `errors` и `errgroup`. Без добавленных imports код не компилируется.
6. Derived `ectx` передаётся в probe, но send выбирает parent `ctx.Done()`. После ошибки sibling derived context уже отменён, а parent может оставаться active; cancellation branch должна слушать `ectx.Done()` и возвращать `ectx.Err()`.
7. Проверка `best.respTime == 0` смешивает «результата ещё нет» с допустимой zero duration. Нужен отдельный boolean.
8. Contract для empty input, invalid limit и tie не определён.
9. Названия `maxElem`/`maxRespTime` противоречат задаче minimum и помогли допустить логическую ошибку `>` до подсказки интервьюера.
10. Closure над `searcher` является bug только для package с language version до Go 1.22. Для модуля `go 1.22+` iteration variable создаётся заново. На интервью это следовало оговорить, а не безусловно объявлять код ошибочным.

### Проверенное решение с worker pool

Ниже вариант без внешних dependencies. Он создаёт не более `limit` workers, включает producer и workers в один lifecycle, отменяет группу с первой причиной ошибки, защищает aggregate mutex и не возвращается до завершения всех созданных goroutines.

```go
type probeFunc func(context.Context, string) (time.Duration, error)

type probeResult struct {
	name     string
	duration time.Duration
}

func getFastestSearcher(
	ctx context.Context,
	searchers []string,
	limit int,
	probe probeFunc,
) (string, time.Duration, error) {
	if len(searchers) == 0 {
		return "", 0, errors.New("empty searcher list")
	}
	if limit <= 0 {
		return "", 0, errors.New("concurrency limit must be positive")
	}
	if probe == nil {
		return "", 0, errors.New("nil probe")
	}
	if limit > len(searchers) {
		limit = len(searchers)
	}

	workerCtx, cancel := context.WithCancelCause(ctx)
	defer cancel(nil)

	jobs := make(chan string)
	var wg sync.WaitGroup
	var mu sync.Mutex
	var best probeResult
	haveBest := false

	wg.Add(1)
	go func() {
		defer wg.Done()
		defer close(jobs)
		for _, name := range searchers {
			select {
			case jobs <- name:
			case <-workerCtx.Done():
				return
			}
		}
	}()

	wg.Add(limit)
	for range limit {
		go func() {
			defer wg.Done()
			for {
				select {
				case <-workerCtx.Done():
					return
				case name, ok := <-jobs:
					if !ok {
						return
					}
					d, err := probe(workerCtx, name)
					if err != nil {
						cancel(fmt.Errorf("probe %q: %w", name, err))
						return
					}

					mu.Lock()
					if !haveBest || d < best.duration ||
						(d == best.duration && name < best.name) {
						best = probeResult{name: name, duration: d}
						haveBest = true
					}
					mu.Unlock()
				}
			}
		}()
	}

	wg.Wait()
	if err := context.Cause(workerCtx); err != nil {
		return "", 0, err
	}
	return best.name, best.duration, nil
}
```

Код собран и выполнен через официальный Go Playground на Go `1.26.5`: для `google=120ms`, `yandex=80ms`, `mail=250ms`, `limit=2` получен результат `yandex 80ms <nil>`. Полная проверенная версия сохранена в `Данные/verified_solution.go`. Локального Go toolchain в рабочей среде нет, поэтому `go test -race` здесь не запускался.

### Диагностика «добавили index, быстрее не стало»

Правильный порядок начинается не с partitioning:

1. зафиксировать exact query, parameters, latency distribution и returned row count;
2. снять `EXPLAIN (ANALYZE, BUFFERS, WAL)` в безопасной среде и сравнить estimates с actual;
3. проверить predicate/operator class, implicit casts, functions, collation и leading columns composite index;
4. проверить statistics, data skew и extended statistics для correlated columns;
5. понять, сколько table надо прочитать: при низкой selectivity sequential scan может быть оптимальным;
6. проверить joins, sorts, temp spills, heap fetches, visibility map, bloat и I/O;
7. только после найденного bottleneck выбирать index redesign, query rewrite, partitioning, caching или sharding.

Кандидат назвал selectivity, partial index, `EXPLAIN ANALYZE` и partitioning — это хорошие пункты. Но фраза «таблица большая, значит напрашивается partitioning» слишком ранняя: pruning работает только когда predicates совместимы с partition bounds. Также `EXPLAIN ANALYZE` реально выполняет statement; для mutating query его нельзя бездумно запускать в production.

### PostgreSQL как work queue

Pattern `FOR UPDATE SKIP LOCKED` позволяет нескольким consumers пропускать rows, уже захваченные другими transactions. Он специально уместен для queue-like table, хотя даёт inconsistent view и не подходит для общего чтения.

Есть два основных lifecycle:

- обработать task внутри короткой transaction, если работа локальна и быстра;
- атомарно перевести row `pending → processing`, записать owner/lease/attempt, commit, выполнить внешнюю работу, затем завершить `done`; expired leases возвращает reaper.

Второй вариант переживает долгую работу и crash, но требует idempotency: worker может выполнить external side effect и упасть до отметки `done`. Optimistic version column защищает concurrent update, но сама по себе не распределяет claims так удобно, как row lock/conditional state transition.

### Consumer group и ordering

При шести partitions и шести active consumers одной group каждый обычно получает по одной partition. После падения двух members coordinator инициирует rebalance и переназначает их partitions. При четырёх consumers и шести partitions двое получат по две partitions, двое — по одной, хотя exact assignment зависит от assignor и предыдущего ownership.

Kafka сохраняет offset order внутри одной partition, а не всего topic. Чтобы события одного promo code шли последовательно, producer использует стабильный entity key. Но добавление partitions может изменить mapping hash→partition для новых records, поэтому migration требует отдельного плана, если ordering должен продолжаться через изменение partition count.

Кандидат сначала не смог описать rebalance и сказал, что «так обычно не делают». Это слабый ответ: failure/reassignment — базовый consumer-group scenario, независимо от нормального steady-state ratio.

### Replay, retry и DLQ — три разные операции

- Replay повторно читает сохранённый log range: новая group с нужным `auto.offset.reset`, explicit `seek`, reset committed offsets или отдельный reprocessing job.
- Retry повторяет временно неуспешную обработку с ограничением attempts и backoff, не блокируя partition навсегда.
- DLQ изолирует poison message для анализа/ручного или отдельного recovery flow и сохраняет error context.

В вопросе нужно было перечитать исторический диапазон после исправления consumer logic. Ответ кандидата про DLQ полезен для будущих failures, но не решает replay уже committed records. Сильный ответ сначала фиксирует offsets/time range и создаёт isolated reprocessing consumer, затем делает downstream effect idempotent, мониторит lag/errors и только после reconciliation переключает основной flow.

Kafka Connect имеет explicit DLQ settings, но для обычного custom consumer DLQ/retry topics — application pattern. Нельзя отвечать так, будто core broker автоматически отправляет все failed records в DLQ.

### Пробелы и опасные формулировки

- Implementation size выдаётся как почти универсальный факт без `GOARCH`/compiler boundary.
- String называется массивом, а размер descriptor смешивается с размером данных.
- Для Swiss Table названо `16` slots вместо текущих `8`.
- Scheduler и heap placement объясняются фиксированными числами, которых language contract не даёт.
- `sync.Map` связывается с reflection, хотя обычный `Load` возвращает `any` и требует assertion.
- Live-coding решение не проходит compile/lifecycle/race review.
- Переменные с `max` привели к поиску maximum вместо minimum до подсказки.
- PostgreSQL partitioning/sharding предлагаются до доказательства bottleneck.
- Rebalance после падения consumers не восстановлен.
- Replay неправильной обработки подменён DLQ.

### Как отвечать сильнее

У ответа должен быть повторяемый каркас:

1. contract или invariant;
2. concrete mechanism;
3. failure/cancellation boundary;
4. version/implementation boundary;
5. способ проверить предположение.

Например, не «строка весит 16 bytes», а: «По specification string — immutable bytes. В standard `gc` на `amd64` value обычно descriptor из pointer и length, поэтому `unsafe.Sizeof` даёт 16 bytes; backing bytes не входят, и это implementation detail». Не «закрою channel после `Wait`», а: «channel закрывает goroutine, которая знает, что producers завершены; consumer обязательно join-ится до чтения aggregate и return».
