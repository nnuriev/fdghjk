# Правила для агентов

Этот файл действует для всего репозитория. Его задача — сохранять структуру vault, доказательность материалов и явную привязку версионных утверждений к первоисточникам.

## Область работы

- Пишите технические материалы по-русски для уровня Middle–Senior; архитектурные разборы углубляйте до Senior+.
- При первом упоминании термина сохраняйте общепринятый английский вариант, если он помогает искать документацию и исходный код.
- Не создавайте предметную заметку без явного запроса пользователя. Не материализуйте списком все отсутствующие ссылки из карт.
- Начинайте предметную заметку только из конкретной ссылки в `План заметок` и используйте соответствующий шаблон из `99 Шаблоны`.
- После завершения перенесите ссылку в `Готовые заметки`; до завершения она остаётся в плане.

## Структура заметки

- Сохраняйте только поля frontmatter `aliases`, `tags`, `уровень`, `статус`.
- Допустимые уровни: `middle`, `senior`, `senior+`.
- Допустимые статусы: `черновик`, `проверено`, `требует обновления`.
- Используйте русские иерархические теги. Не дублируйте тегами уровень, статус или полный путь папки.
- Сохраняйте разделы выбранного шаблона: TL;DR, ментальную модель, устройство, trade-offs, ошибки, применение, связанные темы и источники.
- Wiki-links указывайте полным путём от корня без расширения `.md`. Вне `План заметок` ссылайтесь только на существующие страницы.

## Источники и проверяемость

- Подбирайте первичный источник под утверждение: спецификацию, стандарт, RFC, официальную документацию, release notes, proposal, migration guide, исходный код или тесты.
- Вторичные статьи используйте только как дополнительное объяснение и явно отделяйте их выводы от первичных данных.
- Для исходного кода давайте permalink на tag или commit, конкретный файл и по возможности символ или диапазон строк. Не используйте плавающую ветку `main` или `master` как доказательство версионного поведения.
- Для Go предпочитайте Go specification, memory model, release notes, proposals, документацию стандартной библиотеки и исходники Go на фиксированном tag/commit.
- Для каждого версионно-зависимого тезиса указывайте точную версию, платформу при необходимости и дату проверки.
- Если первичного подтверждения нет, не выдавайте предположение за факт и не присваивайте заметке статус `проверено`.

Формат записи источника:

```text
[Название](URL) — организация или репозиторий, версия или commit/tag, проверено YYYY-MM-DD.
```

## Эволюция версий

- Слово «сейчас» всегда сопровождайте точной версией и датой проверки.
- Добавляйте раздел `Эволюция и версии`, когда поведение, API, гарантии или рекомендации действительно менялись.
- Сравнивайте подтверждённые состояния в формате `версия или период → было → стало → практический эффект → источник`.
- Когда различие лучше видно в коде, приводите два минимальных примера: «до» и «начиная с» конкретной версии.
- Не выдумывайте историческую пару и не добавляйте искусственную эволюцию для фундаментальной неизменной концепции.
- Если поддерживаемые версии расходятся по поведению, опишите обе и дайте правило выбора.

## Код и примеры

- Делайте пример минимальным, воспроизводимым и связанным с тезисом.
- Проверяйте компиляцию, тест, benchmark или наблюдаемый вывод, когда подходящая среда доступна.
- Если проверить код невозможно, явно помечайте его как непроверенный и не ставьте статус `проверено` только на основании такого примера.
- Для Go указывайте версию Go и значимые GOOS/GOARCH. Внешний Go Playground допустим только как необязательный проверенный permalink.
- Не создавайте локальные labs или каталог Playground без отдельного запроса.

## Критерии статуса

- `черновик`: структура или существенные утверждения ещё не проверены.
- `проверено`: ключевые утверждения подтверждены первичными источниками, версионная область указана, а исполняемые примеры проверены при доступной среде.
- `требует обновления`: источник, версия, API или практическая рекомендация могли устареть либо обнаружено противоречие.

## Ограничения

- Не устанавливайте Obsidian, Go, community plugins или другие инструменты без отдельного запроса.
- Не добавляйте карточки, self-check, spaced repetition, готовые учебные материалы, CI, remote, публикацию или лицензию без отдельного запроса.
- Не меняйте нумерованную структуру каталогов, frontmatter-контракт или шаблоны как стабильный интерфейс без явного согласования.
- Сохраняйте Markdown читаемым вне Obsidian и не используйте синтаксис сторонних плагинов.
