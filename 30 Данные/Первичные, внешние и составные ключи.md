# Первичные, внешние и составные ключи

## TL;DR

Ключ отвечает за identity факта. Candidate key минимально и однозначно определяет строку; один из candidate keys объявляют `PRIMARY KEY`, остальные сохраняют через `UNIQUE`. `FOREIGN KEY` требует, чтобы ссылка совпадала с допустимым ключом другой строки, а composite key выражает identity из нескольких атрибутов.

Технический `id` удобен, но не отменяет бизнес-уникальность. Для multi-tenant схемы составной ключ `(tenant_id, object_id)` часто полезнее глобального `object_id`: он позволяет самой базе запретить ссылку между арендаторами.

## Область применимости

- Термины candidate key, superkey и referential integrity относятся к реляционной теории; синтаксис и детали ниже относятся к PostgreSQL 18.4.
- В PostgreSQL `PRIMARY KEY` означает `UNIQUE` плюс `NOT NULL` и автоматически создаёт unique B-tree index. У таблицы один primary key, но несколько unique constraints.
- По умолчанию `UNIQUE` допускает несколько `NULL`; `NULLS NOT DISTINCT` меняет это поведение. Внешний ключ с любым `NULL` при стандартном `MATCH SIMPLE` обычно не проверяет referenced row.
- Вне scope: temporal keys `WITHOUT OVERLAPS` и `PERIOD`, добавленные в PostgreSQL 18, а также глобальные ключи поверх нескольких shards.
- Пример статически проверен по грамматике PostgreSQL 18 и не исполнялся локально.

## Ментальная модель

Ключ похож на координату факта. Primary key выбирает канонический адрес внутри отношения. Foreign key хранит такую координату в другом отношении и просит СУБД доказать, что цель существует.

Составной ключ не означает «несколько независимых ключей». `(tenant_id, user_id)` сравнивается как единая координата. Это различие критично: два отдельных foreign keys доказали бы лишь существование tenant и user по отдельности, но не принадлежность пользователя этому tenant.

## Как устроено

### Candidate, primary и alternate keys

Superkey однозначно определяет строку, но может содержать лишние атрибуты. Candidate key минимален: удаление любой его части разрушает уникальность. Из нескольких candidate keys проектировщик выбирает primary key; выбор влияет на внешние ссылки, ширину вторичных индексов и удобство миграций, но не делает остальные candidate keys менее обязательными.

Natural key состоит из значений предметной области: ISO-кода, email внутри tenant, пары `(order_id, line_no)`. Surrogate key генерируется системой и обычно не несёт бизнес-смысла. Частый компромисс: короткий surrogate primary key плюс `UNIQUE` на стабильном natural key.

Identity column генерирует значение последовательностью, но сама по себе не гарантирует уникальность. Нужен `PRIMARY KEY` или `UNIQUE`. Значения sequence не откатываются вместе с транзакцией, поэтому пропуски нормальны и не доказывают потерю строк.

### Referential integrity

PostgreSQL разрешает FK ссылаться на primary key, unique constraint либо на столбцы нечастичного (`non-partial`) unique index. Типы и operator classes должны позволять сравнение. СУБД проверяет insert/update дочерней строки и защищает родительскую строку от несовместимого update/delete.

Действие задаётся явно:

- `NO ACTION` оставляет проверку ограничению и допускает deferred-проверку, если constraint объявлен deferrable;
- `RESTRICT` запрещает действие немедленно;
- `CASCADE` переносит update/delete на дочерние строки;
- `SET NULL` и `SET DEFAULT` меняют ссылочные столбцы, если остальные ограничения это допускают.

PostgreSQL не создаёт индекс на referencing columns автоматически. Индекс часто нужен для joins и для быстрой проверки удаления/изменения parent row, но его форма зависит от workload.

### Ширина составного ключа

Composite primary key естественно выражает слабую сущность, например строку заказа `(order_id, line_no)`. Однако каждый FK переносит все его столбцы, а каждый referencing index становится шире. Отдельный surrogate может сократить ссылки, но тогда `UNIQUE (order_id, line_no)` всё равно нужен, иначе бизнес-identity потеряна.

## Пример или трассировка

Схема запрещает посту ссылаться на пользователя другого tenant:

```sql
CREATE TABLE tenant (
    tenant_id bigint PRIMARY KEY
);

CREATE TABLE app_user (
    tenant_id bigint NOT NULL REFERENCES tenant ON DELETE CASCADE,
    user_id bigint NOT NULL,
    email text NOT NULL,
    PRIMARY KEY (tenant_id, user_id),
    UNIQUE (tenant_id, email)
);

CREATE TABLE post (
    tenant_id bigint NOT NULL REFERENCES tenant ON DELETE CASCADE,
    post_id bigint NOT NULL,
    author_id bigint NOT NULL,
    body text NOT NULL,
    PRIMARY KEY (tenant_id, post_id),
    FOREIGN KEY (tenant_id, author_id)
        REFERENCES app_user (tenant_id, user_id)
);

INSERT INTO tenant VALUES (1), (2);
INSERT INTO app_user VALUES
    (1, 10, 'a@t1.test'),
    (2, 20, 'b@t2.test');

INSERT INTO post VALUES (1, 100, 10, 'ok');
INSERT INTO post VALUES (1, 101, 20, 'cross-tenant');
```

Первая вставка проходит. Вторая должна завершиться `foreign_key_violation`: пары `(1, 20)` в `app_user` нет, хотя tenant `1` и user id `20` по отдельности существуют. Два независимых foreign keys этого инварианта не выразили бы.

## Trade-offs

- Natural key сразу запрещает дубликат бизнес-сущности и делает некоторые joins понятнее. Изменяемое или длинное значение раздувает все ссылки и превращает rename в каскадную запись.
- Surrogate key узок и стабилен. Без отдельного unique constraint он разрешает семантические дубликаты.
- Composite key кодирует область identity и помогает tenant isolation. Цена: более широкие foreign keys, индексы и API-параметры.
- `ON DELETE CASCADE` удобно выражает общий lifecycle. Ошибка в parent-delete получает большой blast radius; для независимо значимых данных лучше `RESTRICT` или soft-delete с явной политикой.
- Nullable FK моделирует необязательную связь. Если `NULL` на самом деле означает несколько состояний вроде «ещё не назначено» и «назначение удалено», одного столбца недостаточно.

## Типичные ошибки

- Неверное предположение: auto-increment гарантирует identity. Симптом: две строки с разными `id` описывают одного пользователя. Причина: sequence генерирует числа, но не знает business key. Исправление: добавить `UNIQUE` на business identity и определить политику canonicalization.
- Неверное предположение: два одиночных FK равны одному составному. Симптом: появляется cross-tenant ссылка. Причина: ограничения доказывают два независимых существования, а не существование пары. Исправление: использовать composite candidate key и composite FK с тем же порядком столбцов.
- Неверное предположение: FK автоматически индексирует child columns. Симптом: удаление parent row или join долго сканирует большую child table. Причина: PostgreSQL создаёт индекс для referenced uniqueness, но не для referencing side. Исправление: добавить индекс по измеренному workload, обычно с FK columns в начале.
- Неверное предположение: `UNIQUE` запрещает второй `NULL`. Симптом: таблица принимает несколько «неизвестных» значений. Причина: по умолчанию PostgreSQL считает null values distinct. Исправление: `UNIQUE NULLS NOT DISTINCT`, `NOT NULL` или partial unique index с нужной семантикой.
- Неверное предположение: cascade всегда упрощает удаление. Симптом: одно удаление блокирует много строк или стирает самостоятельные данные. Причина: lifecycle ребёнка ошибочно привязан к parent. Исправление: пересмотреть ownership, использовать `RESTRICT` и явный процесс удаления.

## Когда применять

Для каждой таблицы выпишите все candidate keys до выбора primary key. Выбирайте composite key, когда область identity входит в сам факт: line number внутри order, user id внутри tenant, version внутри document. Добавляйте surrogate key, когда он сокращает многочисленные ссылки или стабилизирует изменяемый natural key, но сохраняйте business uniqueness.

Foreign key нужен там, где ссылка и цель меняются в одной базе и referential integrity входит в контракт. Если цель находится в другом сервисе или shard, обычный FK не охватывает границу; потребуются ownership, repair и reconciliation, а локальная схема всё равно должна запрещать внутренне невозможные состояния.
