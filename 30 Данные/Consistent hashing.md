# Consistent hashing

## TL;DR

Consistent hashing — семейство способов назначать ключи узлам так, чтобы изменение состава переносило только часть ключей, а не почти весь набор, как при `hash(key) mod node_count`. Классическое представление — круговое hash space: ключ принадлежит первому token по часовой стрелке. Virtual nodes (vnodes) дают одному физическому узлу много tokens и делают долю владения и перенос более дробными.

Это механизм placement, а не готовая распределённая база. Он не обеспечивает репликацию, согласованную смену topology, равномерность реального workload или лечение hot key. Для этого нужны отдельные правила preference list, membership, [[30 Данные/Rebalancing данных|rebalancing]], routing и repair.

## Область применимости и версии

Ментальная модель основана на исходной работе Karger и соавторов 1997 года и на реализации идей кольца в Dynamo 2007. Термины token и vnode сопоставлены с Apache Cassandra 5.0.8; проверено 2026-07-18. Конкретная СУБД может использовать иной partitioner, каталог диапазонов или rendezvous hashing, даже если в документации употребляется слово «consistent».

## Ментальная модель

Обычное `hash(key) mod n` смешивает две задачи: получить равномерное число и превратить его в owner. После перехода `n -> n+1` остаток деления меняется для большинства значений, поэтому cache или storage вынуждены массово перемещать данные.

Consistent hashing фиксирует hash space независимо от числа узлов. Узлы занимают позиции или получают ranges внутри этого пространства. При добавлении owner меняются только границы рядом с его новой долей. Главный инвариант — **один и тот же topology snapshot однозначно отображает любой hash в одного первичного владельца и, при репликации, в определённый preference list**.

Полезно разделять три термина:

- **Hash range** — непрерывный интервал значений хеша. Его можно хранить в обычной отсортированной таблице; круг для этого не обязателен.
- **Ring** — hash space с замыканием максимального значения на минимальное. В классической схеме token узла владеет интервалом от предыдущего token, не включая его, до своего token включительно.
- **Virtual node (vnode)** — логическая позиция/token или небольшой range, назначенный физическому узлу. Один сервер владеет многими несмежными vnodes.

Ring задаёт геометрию поиска owner, vnode — гранулярность размещения. Смешивать их опасно: «у нас ring» ещё не говорит, сколько ranges можно независимо переносить и насколько ровно распределена ёмкость.

## Как устроено

### От ключа к owner

1. Стабильная hash function отображает partition key в фиксированное пространство, например `0..2^m-1`.
2. Отсортированный token map находит первый token не меньше хеша; после последнего поиск замыкается на первый.
3. Физический узел, которому назначен token, становится primary owner соответствующего интервала.
4. Для [[30 Данные/Репликация данных|репликации]] строится preference list следующих подходящих физических узлов с учётом rack/zone. Несколько vnodes одного сервера нельзя ошибочно считать независимыми репликами.

При одном token на физический узел случайное размещение даёт большой разброс размеров соседних интервалов. Dynamo описывает virtual nodes как способ распределить владение по множеству позиций, принимать при уходе узла небольшие части от разных neighbors и учитывать неодинаковую capacity числом tokens. Чем больше независимых ranges, тем ближе фактическая доля к целевой при равномерных хешах.

Цена vnodes — больше metadata и связей между узлами. Документация Cassandra 5.0.8 отмечает, что каждый дополнительный token расширяет число соседей для maintenance, усложняет repair и ухудшает range scans. Поэтому «как можно больше vnodes» — не универсальная оптимизация; современные конфигурации часто предпочитают меньшее число хорошо рассчитанных tokens.

### Что именно минимизируется

При добавлении равноправного узла идеальная целевая доля нового участника — примерно `1/(n+1)` hash space. Именно эту долю и нужно скопировать ему в сбалансированном состоянии; consistent hashing избегает лишнего remapping остальных долей. Это ожидание при равномерном распределении и достаточно мелких ranges, а не обещание для конкретной случайной расстановки.

Удаление симметрично передаёт его ranges следующим owners. Но mapping — лишь желаемое владение. Байты ещё нужно перенести, новые записи — не потерять, а routers — переключить на одну topology epoch. Эту часть выполняет отдельный migration protocol.

### Чего алгоритм не решает

- Хороший hash распределяет разные ключи, но один популярный ключ остаётся одним hash point.
- Равная доля hash space не означает равные bytes или QPS, если ключи весят и запрашиваются по-разному.
- Последовательность owners на ring не гарантирует независимые failure domains; placement policy должна исключать копии на одном хосте, rack или zone.
- Две стороны сети могут знать разные token maps. Без согласованного membership они направят запись разным owners.
- Range queries по исходному ключу теряют локальность, потому что hash уничтожает порядок.

## Пример или трассировка

Пусть hash space — кольцо `0..99`, а физические узлы имеют по одному token:

| Token | Узел | Владеет range |
|---:|---|---|
| `20` | `A` | `(80, 20]`, включая замыкание через `99 -> 0` |
| `50` | `B` | `(20, 50]` |
| `80` | `C` | `(50, 80]` |

Поэтому `h(k)=10` идёт в `A`, `35` — в `B`, `60` и `70` — в `C`, `90` — снова в `A`.

Добавим узел `D` с token `65`:

1. До изменения весь `(50,80]` принадлежал `C`.
2. После изменения `D` владеет `(50,65]`, а `C` сохраняет `(65,80]`.
3. Ключ с hash `60` переносится `C -> D`.
4. Ключи `10`, `35`, `70`, `90` остаются у прежних owners.

Наблюдаемый результат: изменился только один соседний интервал, а не четыре из пяти показанных ключей. Для сравнения, переход `hash mod 3 -> hash mod 4` пересчитывает owner независимо для каждого ключа и не сохраняет такое локальное свойство.

Теперь дадим каждому серверу по 64 vnodes. Новый узел получает маленькие intervals по всему ring, а не один большой соседний кусок. Перенос можно распараллелить между несколькими donors и точнее приблизить долю `25%`, но metadata, streaming connections и repair ranges становятся многочисленнее.

## Trade-offs

| Подход | Когда выигрывает | Цена |
|---|---|---|
| Один token на узел | Очень простой token map | Высокий variance долей, крупный перенос |
| Много vnodes | Гранулярный rebalance, capacity weighting | Больше metadata, neighbors и repair overhead |
| Явные hash ranges в каталоге | Точный контроль границ и placement | Нужен надёжный metadata service и планировщик ranges |
| `hash mod n` | Фиксированный состав, stateless распределение без миграции | Массовый remapping при смене `n` |
| Range по исходному ключу | Диапазонная локальность | Горячий хвост и сложнее равномерность |

Rendezvous hashing — близкая альтернатива: для ключа вычисляют score каждого кандидата и выбирают лучший. Она тоже минимизирует remapping и удобна при небольшом списке узлов, но наивный lookup проверяет всех кандидатов. Каталог явных ranges лучше, когда система хочет сознательно делить тяжёлые ranges по наблюдаемой нагрузке, а не полагаться только на статистику хеша.

## Типичные ошибки

- **Неверное предположение:** consistent hashing автоматически балансирует QPS. **Симптом:** узлы владеют равной долей tokens, но один перегружен. **Причина:** workload skew и разный вес ключей. **Исправление:** балансировать по bytes/QPS/CPU, а [[30 Данные/Hot partitions и hot keys|hot keys]] обрабатывать отдельно.
- **Неверное предположение:** token и physical node — одно и то же. **Симптом:** preference list считает два vnodes одного сервера двумя репликами. **Причина:** смешаны логическое и физическое размещение. **Исправление:** строить replicas по distinct hosts и failure domains.
- **Неверное предположение:** смена token map завершает rebalance. **Симптом:** новый owner отвечает `not found` или старый принимает потерянные записи. **Причина:** metadata переключили раньше streaming/catch-up. **Исправление:** разделить desired ownership, движение данных, cutover и cleanup.
- **Неверное предположение:** больше vnodes всегда лучше. **Симптом:** repair и topology changes создают тысячи мелких операций и связей. **Причина:** гранулярность имеет эксплуатационную цену. **Исправление:** выбирать число tokens по измеренному balance и maintenance cost конкретной версии СУБД.
- **Неверное предположение:** изменение hash function совместимо. **Симптом:** после деплоя разные клиенты адресуют один ключ разным узлам. **Причина:** partition function — часть persistent layout. **Исправление:** версионировать алгоритм и мигрировать данные/routing как schema change.

## Когда применять

Consistent hashing полезен для распределённых caches, object/key-value storage и других систем, где состав owners меняется, а range locality исходного ключа не нужна. До выбора алгоритма надо определить, кто публикует membership, как clients получают epoch, сколько данных допустимо переносить одновременно и как репликация учитывает зоны.

Проверяют не только равномерность tokens. Нужны гистограммы bytes, ключей, QPS и latency на vnode/range, симуляция добавления и потери узла, а также максимальная доля, которую один физический failure заставит перестроить. Если workload содержит большие tenants или популярные ключи, алгоритм placement дополняют split/merge и прикладным разбиением, а не пытаются исправить всё числом tokens.
