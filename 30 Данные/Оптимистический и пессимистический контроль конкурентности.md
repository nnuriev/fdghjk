# Оптимистический и пессимистический контроль конкурентности

## TL;DR

Optimistic concurrency control (OCC) разрешает конкурентно читать и вычислять, а перед записью проверяет, что read version не изменилась. При конфликте операция aborts или возвращает «не обновлено» и повторяется. Pessimistic concurrency control (PCC) сначала берёт lock, заставляя contender ждать, ошибаться через `NOWAIT` или пропустить row через `SKIP LOCKED`.

Выбор зависит от conflict probability и цены уже выполненной работы. OCC выигрывает при редких конфликтах и коротком retryable computation. PCC лучше для hot rows, где поздний abort дорог и критическую секцию можно держать короткой. Оба подхода требуют явной transaction boundary и не включают external side effects.

## Область применимости

- Application-level OCC показан через explicit `version bigint`; PCC через PostgreSQL 18.4 `SELECT ... FOR UPDATE`.
- PostgreSQL MVCC и Serializable SSI используют оптимистические идеи внутри СУБД, но это не тождественно version-column protocol: они обнаруживают другие классы conflicts.
- `xmin` идентифицирует creating transaction row version, но transaction IDs 32-bit и wrap around; документация не советует полагаться на их долгосрочную уникальность. Для API/ORM token лучше собственный version.
- `SKIP LOCKED` даёт inconsistent view и применим к queue-like tables, а не как общий способ «не ждать» при business invariant.
- Пример статически проверен, но не исполнялся двумя sessions.

## Ментальная модель

OCC задаёт вопрос на выходе: «То, на основании чего я решил писать, всё ещё актуально?» Проверка и write должны быть одним atomic statement или частью serializable commit validation. Read-compute-write без atomic comparison остаётся check-then-act race.

PCC задаёт вопрос на входе: «Можно ли зарезервировать право менять этот объект, пока я принимаю решение?» Lock переносит conflict раньше и заменяет wasted retry ожиданием. Если под lock выполняется HTTP или пользовательский ввод, ожидание превращается в очередь без управляемой длины.

## Как устроено

### Optimistic protocol

Классическая схема имеет read, validation и write phases. В SQL application обычно читает `(state, version)`, вычисляет новый state и выполняет:

```sql
UPDATE t
SET state = :new_state, version = version + 1
WHERE id = :id AND version = :observed_version;
```

Affected row count `1` означает успешную validation+write. `0` означает conflict, deletion либо failed business predicate; эти исходы надо различить по контракту. `RETURNING` получает новое значение без отдельного race-prone `SELECT`.

Version token должен меняться при каждой записи, способной инвалидировать решение. Если два поля обновляются разными code paths, но один path забывает increment, OCC даёт ложный success. Whole-row version проще доказывать; field-level versions повышают concurrency ценой сложного merge semantics.

### Pessimistic protocol

`SELECT ... FOR UPDATE` блокирует выбранные rows до transaction end. Concurrent locker ждёт, `NOWAIT` сразу получает error, `SKIP LOCKED` пропускает занятые rows. Lock strength можно ослабить до `FOR NO KEY UPDATE`, `FOR SHARE` или `FOR KEY SHARE` под точную conflict matrix.

После ожидания на Read Committed PostgreSQL возвращает/блокирует current row version и повторно проверяет predicate. На Repeatable Read/Serializable изменение после snapshot может дать `40001`. Поэтому код всё равно проверяет state после получения lock.

Пессимистическая схема защищает только locked set. Predicate «нет другой брони на этот interval» нельзя надёжно закрыть `FOR UPDATE`, если совпадающей row ещё нет. Здесь нужны unique/exclusion constraint, guard row, table/range protocol или Serializable.

### Сравнение failure semantics

OCC превращает conflict в conditional miss или serialization failure. Retry должен повторить read и business decision. PCC превращает conflict в wait, timeout, nowait error или deadlock victim. Retry после deadlock тоже охватывает всю transaction.

Слепой retry опасен для обоих подходов, если внутри уже случился external effect. Stable operation ID, outbox и query-by-id отделяют unknown outcome от concurrency retry.

## Пример или трассировка

Остался один экземпляр товара:

```sql
CREATE TABLE inventory (
    sku text PRIMARY KEY,
    stock integer NOT NULL CHECK (stock >= 0),
    version bigint NOT NULL
);
INSERT INTO inventory VALUES ('book', 1, 7);
```

Два optimistic clients читают `(stock = 1, version = 7)`. Client A первым выполняет:

```sql
UPDATE inventory
SET stock = stock - 1,
    version = version + 1
WHERE sku = 'book'
  AND stock > 0
  AND version = 7
RETURNING stock, version;
```

Client A получает `(0, 8)`. Client B выполняет тот же statement со старой version `7` и получает zero rows. Он не должен считать заказ зарезервированным; после перечитывания видит stock `0`. Atomic `WHERE` предотвратил oversell без предварительного lock.

Пессимистический вариант после сброса stock:

```text
Session A                                      Session B
BEGIN;                                         BEGIN;
SELECT stock FROM inventory                    SELECT stock FROM inventory
WHERE sku='book' FOR UPDATE;  -- 1             WHERE sku='book' FOR UPDATE;  -- waits
UPDATE inventory SET stock=0, version=9
WHERE sku='book';
COMMIT;                                        -- unblocks, sees stock=0
                                                ROLLBACK;
```

Session B ждёт, затем обязана повторно проверить stock. Lock гарантирует one-at-a-time работу с этой row, но не резервирует внешнюю оплату: remote call внутри lock сделал бы критическую секцию зависимой от сети.

## Trade-offs

- OCC не удерживает lock во время application computation и хорошо масштабируется при редких conflicts. При hot key многие workers делают одинаковую работу, а победит один; retry storm повышает tail latency.
- PCC предотвращает wasted computation и задаёт очередь на row lock. Он удерживает connection/transaction, ограничивает throughput hot key и добавляет deadlocks при нескольких objects.
- Whole-row version обнаруживает любой concurrent update. Это безопасный default, но конфликтует даже при независимых fields. Field-level merge требует явной commutativity и invariant analysis.
- `NOWAIT` даёт быстрый backpressure и предсказуемый latency. Клиент должен отличать contention от отсутствия row и выбрать retry/409/queue.
- `SKIP LOCKED` распределяет jobs между workers. Пропущенный hot job может голодать; нужны lease/retry/ordering semantics и отдельный reconciliation.
- Serializable SSI защищает прочитанные predicates без ручного version на каждой row. Он возвращает `40001` позже и требует transaction-wide retry; внешняя API conflict semantics всё равно может требовать version/ETag.

## Типичные ошибки

- Неверное предположение: read version, затем обычный UPDATE достаточно для OCC. Симптом: оба клиента сообщают success, последний overwrite теряет изменение. Причина: validation отделена от write. Исправление: version и business predicate в одном `UPDATE ... WHERE`, проверка affected rows.
- Неверное предположение: любой zero-row UPDATE означает version conflict. Симптом: клиент бесконечно retry удалённый объект или out-of-stock. Причина: в одном predicate смешаны existence, version и domain precondition. Исправление: определить outcome contract и после miss безопасно различить причины либо вернуть единый domain conflict.
- Неверное предположение: `xmin` служит вечным ETag. Симптом: token нестабилен после maintenance/migration или предположение о глобальной уникальности ломается. Причина: это 32-bit transaction ID row version, а не application version. Исправление: явный monotonic version/UUID token с собственным контрактом.
- Неверное предположение: lock строки защищает predicate отсутствия. Симптом: две transactions вставляют overlapping booking. Причина: отсутствующую row нельзя заблокировать `FOR UPDATE`. Исправление: exclusion/unique constraint, materialized guard или Serializable.
- Неверное предположение: пессимистическая transaction может держать lock во время remote call. Симптом: pool и lock queue растут вслед за latency внешнего сервиса. Причина: lock lifetime охватил ненадёжную операцию. Исправление: разделить reservation state и external workflow, commit быстро, продолжить идемпотентно.
- Неверное предположение: `SKIP LOCKED` возвращает полный consistent result. Симптом: отчёт недосчитывает rows или job голодает. Причина: locked rows намеренно пропущены. Исправление: применять только к queue consumption с recovery, не к общему чтению.

## Когда применять

Выбирайте OCC, если conflict rate мала, computation коротко повторяется, а клиент уже имеет version/ETag. Выбирайте PCC для короткой операции над hot row, когда раннее ожидание дешевле позднего abort. Для нескольких locks зафиксируйте total order.

Если invariant охватывает набор rows или отсутствие row, поднимитесь от row-level техники к constraint или Serializable. Проверяйте throughput вместе с conflict rate, retries per success, lock wait p95, deadlocks и возрастом transactions.
