# Индексы и цена чтения и записи

## TL;DR

Индекс хранит дополнительное представление ключей, по которому PostgreSQL может найти небольшой диапазон строк, выдать совместимый порядок или проверить uniqueness без полного сканирования heap. Он ускоряет только те чтения, для которых predicate, operators, ordering и selectivity подходят access method.

Каждый индекс оплачивается на записи: новые entries, page splits, WAL, cache pressure, vacuum и иногда потеря HOT update. Поэтому вопрос звучит не «какой индекс добавить», а «какой измеренный read path окупает постоянную write amplification».

## Область применимости

- Детали относятся к PostgreSQL 18.4. В нём heap и secondary indexes хранятся отдельно; B-tree используется по умолчанию.
- Рассматриваются B-tree, multicolumn, partial, expression и covering indexes. GiST, SP-GiST, GIN, BRIN и hash упомянуты только как граница выбора.
- PostgreSQL 18 добавил B-tree skip scan. Он расширяет число применимых запросов, но не отменяет ценность equality constraints на leading columns.
- Вне scope: устройство страниц B+tree, LSM compaction, distributed secondary indexes и automatic index advisors.
- Пример статически сверен с PostgreSQL 18, но не исполнялся; конкретный plan зависит от данных, statistics, cache и cost settings.

## Ментальная модель

Heap содержит версии строк, а индекс содержит отсортированные или специализированные указатели на них. Поиск по индексу состоит из двух возможных частей: найти entries, затем посетить heap tuples и проверить visibility. Поэтому «все нужные столбцы есть в индексе» ещё не гарантирует index-only scan: visibility map должна подтвердить, что heap page целиком видима snapshot.

Индекс похож на сохранённый ответ не на один query, а на семейство порядков. B-tree `(tenant_id, created_at DESC)` хорошо отвечает на «последние события tenant». Он не обязан помогать `lower(message)`, поиску подстроки или выборке половины таблицы.

## Как устроено

### Выбор access method

B-tree поддерживает equality, ranges и упорядоченную выдачу для operator class. Hash ориентирован на equality. GIN полезен для composite values с множеством элементов, GiST/SP-GiST дают extensible search trees, BRIN хранит summaries диапазонов страниц и выигрывает на большой физически коррелированной таблице.

Planner сравнивает альтернативы. Seq Scan дешёво читает много соседних pages. Index Scan хорош для малого результата, но случайные heap visits становятся дорогими при низкой selectivity. Bitmap Index Scan собирает tuple locations, а Bitmap Heap Scan группирует heap access по pages; это промежуточный режим.

### Multicolumn order

Для B-tree equality constraints на leading columns и range на следующем столбце обычно ограничивают непрерывный диапазон scan. Остальные условия можно проверить в index, но они не всегда уменьшают прочитанную часть. PostgreSQL 18 способен применить skip scan при пропущенном leading column, если distinct values впереди мало и повторные searches дешевле полного scan.

Порядок выбирают по query shape, а не по мифу «самый selective первым». Для `WHERE tenant_id = ? ORDER BY created_at DESC LIMIT 50` индекс `(tenant_id, created_at DESC)` одновременно ограничивает tenant и отдаёт нужный порядок. Индекс `(created_at, tenant_id)` для этого workload обычно хуже.

### Partial, expression и covering indexes

Partial index содержит только строки, удовлетворяющие predicate. Он меньше и дешевле обновляется, но planner должен доказать, что query predicate влечёт index predicate. Parameterized условие не всегда даёт такое доказательство на этапе planning.

Expression index хранит результат immutable expression, например `lower(email)`. Query должен использовать совместимое выражение и operator class.

`INCLUDE` кладёт payload columns в leaf tuples, не включая их в search key и uniqueness. Это допускает index-only scan, если query читает только хранимые столбцы и visibility map позволяет не ходить в heap. Wide payload раздувает индекс, отключает B-tree deduplication для такого индекса и способен превысить предел index tuple.

### Цена записи

`INSERT` добавляет heap tuple и entries во все подходящие indexes. `DELETE` оставляет версии, которые позже очищает vacuum. `UPDATE` в MVCC создаёт новую row version; если меняется indexed column, соответствующие indexes получают новые entries.

HOT (heap-only tuple) может избежать новых index entries, когда update не меняет ни один индексируемый столбец и на heap page хватает места. Лишний индекс на часто меняющемся поле закрывает эту оптимизацию. Более низкий table `fillfactor` оставляет место на page, повышая вероятность HOT ценой большей таблицы.

## Пример или трассировка

Частый query читает два последних активных события tenant:

```sql
CREATE TABLE event (
    event_id bigint PRIMARY KEY,
    tenant_id bigint NOT NULL,
    created_at timestamptz NOT NULL,
    status text NOT NULL,
    deleted_at timestamptz
);

INSERT INTO event VALUES
    (1, 7, '2026-07-18 10:00:00+04', 'queued', NULL),
    (2, 7, '2026-07-18 10:01:00+04', 'done', NULL),
    (3, 7, '2026-07-18 10:02:00+04', 'failed', '2026-07-18 10:03:00+04'),
    (4, 7, '2026-07-18 10:04:00+04', 'queued', NULL),
    (5, 8, '2026-07-18 10:05:00+04', 'done', NULL);

CREATE INDEX event_active_tenant_time_idx
ON event (tenant_id, created_at DESC)
INCLUDE (status)
WHERE deleted_at IS NULL;

SELECT created_at, status
FROM event
WHERE tenant_id = 7
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 2;
```

Ожидаются `(10:04, queued)` и `(10:01, done)`. Entry для deleted event `3` отсутствует, tenant `8` лежит вне нужного key range, а порядок совпадает с B-tree. На крупной таблице planner может выбрать Index Scan или Index Only Scan. На пяти строках Seq Scan вполне разумен; это не доказывает бесполезность индекса.

Если добавить `UPDATE event SET status = 'done' WHERE event_id = 4`, covering index тоже надо изменить, потому что `status` хранится как payload. Без `INCLUDE (status)` такой update мог бы пройти HOT при достаточном месте на heap page и при отсутствии других индексов на `status`.

## Trade-offs

- Один composite index поддерживает filter и ordering конкретного query. Несколько одиночных indexes гибче комбинируются bitmap scans, но не дают общий order и требуют нескольких structures.
- Covering index сокращает heap reads на mostly-static данных. На горячей таблице visibility bits часто сбрасываются, payload увеличивает write cost, а выигрыш index-only scan исчезает.
- Partial index экономит место и writes для холодного большинства. Он не помогает query, из которого planner не выводит predicate, и требует отдельного плана для complementary rows.
- Expression index ускоряет normalized lookup. Цена: expression вычисляется при записи, а несовпадающий query syntax/operator может не использовать структуру.
- Unique index одновременно ускоряет доступ и охраняет invariant. Обычный performance index следует удалять, если production statistics показывают, что его чтения не покрывают обслуживание.

## Типичные ошибки

- Неверное предположение: индекс всегда быстрее Seq Scan. Симптом: planner игнорирует индекс при большой выборке. Причина: heap random I/O и обработка index entries дороже последовательного чтения многих pages. Исправление: оценить selectivity, actual buffers и объём результата.
- Неверное предположение: порядок столбцов composite index несущественен. Симптом: scan читает почти весь индекс или сортирует результат. Причина: key order не совпал с equality/range/order query. Исправление: проектировать индекс по точному predicate и `ORDER BY`, проверить альтернативы планом.
- Неверное предположение: `INCLUDE` бесплатно превращает query в index-only. Симптом: `Heap Fetches` остаётся большим, writes замедляются. Причина: visibility map не all-visible, payload раздул индекс и обновляется. Исправление: измерить heap fetches и update profile; оставить только нужные узкие columns.
- Неверное предположение: FK или часто встречающийся столбец автоматически требует индекс. Симптом: много почти неиспользуемых structures и дорогие writes. Причина: частота значения не равна частоте selective query. Исправление: начать с workload и referential checks, затем подтвердить `pg_stat_user_indexes` и plans.
- Неверное предположение: partial index работает с любым логически похожим predicate. Симптом: prepared query получает Seq Scan. Причина: planner не доказал implication при parameter или иной форме expression. Исправление: сверить predicate, generic/custom plan и использовать структуру, доказуемую на planning stage.
- Неверное предположение: update столбца, который не входит в индексы, не касается индексов. Симптом: мало HOT updates. Причина: столбец всё же попал в `INCLUDE`, expression или другой индекс либо на page нет места. Исправление: проверить все index definitions, `n_tup_hot_upd` и `fillfactor`.

## Когда применять

Снимите production query shape: predicates, expected rows, ordering, projection, frequency и latency SLO. Добавляйте минимальный индекс, который заметно сокращает visited pages или убирает дорогую сортировку. Затем измерьте writes, WAL, cache residency и HOT ratio.

Проверяйте индекс после изменения distribution и версии PostgreSQL. [[30 Данные/Планы выполнения SQL-запросов|План]] может измениться без DDL, потому что costs и statistics зависят от данных. Не фиксируйте вывод «индекс используется» без representative volume.
