# Миграции схемы данных

## TL;DR

Online schema migration — протокол совместимости между несколькими версиями приложения, схемы и данных. DDL-команда может выполниться за миллисекунды и всё равно мгновенно сломать старый процесс; наоборот, логически совместимое изменение может долго сканировать таблицу и блокировать workload.

Надёжный базовый приём — expand/contract: сначала добавить новое представление, развернуть код, который понимает оба формата, синхронизировать и проверить данные, переключить источник истины, дождаться окончания rollback window и только потом удалить старое. Backfill, dual write/read и constraint validation — отдельные фазы с собственными критериями завершения.

## Область применимости и версии

Основной пример — реляционная схема и rolling deployment. Физическое поведение DDL сверено для PostgreSQL 18.4, tag `REL_18_4`; многошаговая модель online schema change — с F1 из статьи PVLDB 2013. Проверено 2026-07-18. Возможность online index build, тип блокировки и наличие transactional DDL зависят от СУБД и версии: план нельзя переносить только по синтаксическому сходству.

## Ментальная модель

Схема — контракт, а migration временно создаёт несколько его участников:

| | Старая схема/данные | Новая схема/данные |
|---|---|---|
| Старый код | Должен продолжать работать до вывода из эксплуатации | Не обязан понимать новый элемент, но не должен его разрушать |
| Переходный код | Читает старое, умеет поддерживать новое | Понимает оба формата и наблюдает расхождения |
| Новый код | Нужен rollback path или явная несовместимая граница | Работает после cutover |

Вопрос «совместимо ли изменение» всегда имеет направление:

- **read compatibility:** может ли каждая живая версия прочитать строку;
- **write compatibility:** не теряет ли она неизвестные поля/индексы и поддерживает ли нужные представления;
- **operational safety:** какие locks, scans, rewrites, WAL и replica lag вызовет DDL/data change;
- **rollback compatibility:** останется ли старое представление актуальным после переключения.

Поэтому release приложения и migration schema — одна state machine. Независимые пайплайны безопасны только тогда, когда их допустимые состояния явно совместимы.

## Как устроено

### Expand/contract

Универсальный скелет выглядит так:

1. **Expand.** Добавить nullable column, новую table/index или ослабленное constraint так, чтобы старый код продолжал работать.
2. **Совместимый reader/writer.** Развернуть версию, умеющую видеть оба представления. Пока старые instances живы, новое поле не должно быть единственным обязательным входом.
3. **Поддержание новых записей.** После вывода старых writers включить запись в оба представления либо single-write + CDC/trigger. Если targets в одной базе, один transaction обычно безопаснее двух независимых commits.
4. **[[30 Данные/Online backfill|Backfill]].** Заполнить историю небольшими идемпотентными batches с защитой от перезаписи свежих значений.
5. **Validate.** Проверить completeness, semantic equality, constraints и lag, а не только отсутствие job errors.
6. **Cutover read/source of truth.** Сначала shadow-сравнение, затем постепенное чтение нового. Отдельно определить, с какого момента новые writes разрешены только в новом формате.
7. **Contract.** После rollback window остановить поддержку старого формата, сделать ограничения строгими и удалить legacy element отдельным release.

Каждая фаза имеет предусловие и rollback. До смены источника истины откат приложения обычно возвращает чтение к старому представлению. После того как новое приняло уникальные записи, такой откат безопасен только при reverse sync; иначе «rollback» теряет данные.

### Состояния элемента схемы

F1 показывает, почему простой переход `absent -> public` небезопасен при одновременной работе разных schema versions. При добавлении индекса старая версия может удалить строку, не зная, что должна удалить и index entry; новая — успеть создать entry. Получится dangling index record.

В F1 элемент проходит промежуточные состояния: для добавления индекса `absent -> delete-only -> write-only -> backfill -> public`. В `write-only` новые writes уже поддерживают индекс, но query planner ещё его не читает; backfill закрывает исторический gap. Удаление проходит обратные состояния. Конкретные имена принадлежат F1, а общий вывод переносим: **сначала все writers должны поддерживать структуру, потом история заполняется, и лишь затем readers получают право на неё полагаться**.

### Логическая совместимость и физическая цена DDL

В PostgreSQL 18.4 `ALTER TABLE` по умолчанию берёт `ACCESS EXCLUSIVE`, если для subcommand не указано слабее; при нескольких subcommands берётся самый сильный требуемый lock. Поэтому короткий catalog change может долго ждать старую транзакцию, а затем заблокировать новые запросы. Нужны ограниченный `lock_timeout`, наблюдение и повтор, а не бесконечное ожидание в deployment.

Несколько важных границ PostgreSQL 18.4:

- `ADD COLUMN` с constant non-volatile default не переписывает каждую строку: значение сохраняется в metadata и материализуется при последующей перезаписи. Volatile default требует вычисления для строк и физической работы.
- `CHECK` или foreign key можно добавить как `NOT VALID`: новые/изменённые строки уже проверяются, а исторический scan переносится в `VALIDATE CONSTRAINT`, который использует более слабый `SHARE UPDATE EXCLUSIVE` lock.
- `UNIQUE` нельзя просто сделать `NOT VALID`; один online-путь — `CREATE UNIQUE INDEX CONCURRENTLY`, затем attach через `ADD ... USING INDEX`, с отдельной обработкой failure invalid index.
- `SET NOT NULL` обычно должен доказать отсутствие `NULL`; заранее validated `CHECK (column IS NOT NULL)` может позволить избежать полного подтверждающего scan для самого `SET NOT NULL`.

«Metadata-only» не означает «без риска»: lock acquisition и несовместимый старый SQL остаются. И наоборот, долгий validation scan может быть логически безопасен, если throttled и не держит блокировку, останавливающую writes.

### Инварианты orchestration

- В каждый момент известны разрешённые пары `app version × schema state`.
- Новый обязательный элемент не читается как единственный источник до completeness validation.
- Старый элемент не удаляется, пока нет живых readers/writers и не истёк rollback window.
- Backfill не перезаписывает concurrent user update; live write path имеет приоритет.
- Migration lease/fencing не позволяет двум orchestrators одновременно перевести схему в несовместимые состояния.

## Пример или трассировка

Нужно заменить `users.name` на `users.display_name`. Прямой `RENAME COLUMN` мгновенно сломает старый binary, который всё ещё выполняет `SELECT name`.

Безопасная последовательность:

1. **S0:** `name` — источник истины. Добавляем nullable `display_name`; старый код не замечает column.
2. **S1:** разворачиваем reader `COALESCE(display_name, name)`, но writer пока пишет `name`. После полного rollout все процессы умеют видеть новое поле.
3. **S2:** разворачиваем writer, который в одном DB transaction обновляет `name` и `display_name`. Старых writers больше нет; rollback к S1 безопасен, потому что `name` всё ещё актуален.
4. **S3:** backfill выполняет условное заполнение только там, где `display_name IS NULL`; batches checkpointed и throttled. Затем сравниваем counts и значения.
5. **S4:** reader возвращает `display_name`; shadow check сравнивает его с `name`. Источник истины переключён, но writer пока поддерживает оба поля для rollback.
6. **S5:** после rollback window writer перестаёт обновлять `name`. Возврат к S1 теперь уже небезопасен без reverse backfill — это явная граница.
7. **S6:** в отдельном release удаляем `name` после проверки отсутствия SQL-зависимостей.

Наблюдаемый результат: на шагах S0–S4 старая версия может быть возвращена без потери новых изменений. В каждый момент хотя бы одно поле заполнено, а reader имеет fallback. Прямой rename не имеет такого совместимого промежуточного состояния.

Если во время S3 процесс обновляет пользователя, dual writer заполняет оба поля свежим значением. Backfill с условием `display_name IS NULL` не затрёт его историческим snapshot. Если миграция преобразует значение, одной проверки `NULL` мало: нужен source version/updated_at и compare-and-set.

## Trade-offs

| Выбор | Преимущество | Цена |
|---|---|---|
| Expand/contract | Rolling deploy и контролируемый rollback | Больше фаз, временное дублирование |
| Прямой breaking DDL | Короткий migration plan | Требует остановки/атомарного deploy всех clients |
| App dual write в одной транзакции | Согласованность двух columns/tables | Временная сложность writer и write amplification |
| Trigger/CDC | Покрывает старых writers или отделяет target | Скрытая логика, lag, отдельная эксплуатация |
| `NOT VALID` + validate | Короткая критическая DDL-фаза | До завершения history ещё не доказана |
| Offline migration | Простая граница состояния | Downtime, часто неприемлемый на больших данных |

Expand/contract не всегда нужен. Маленькая внутренняя таблица с одним остановленным client может безопасно пройти breaking migration в maintenance window. Многофазность оправдана, когда есть rolling instances, внешние consumers, большой scan или строгий SLO.

## Типичные ошибки

- **Неверное предположение:** transactional DDL делает rollout совместимым. **Симптом:** DDL commit успешен, старые instances падают на SQL. **Причина:** атомарность catalog change не обновляет application binaries. **Исправление:** compatibility matrix и expand/contract.
- **Неверное предположение:** nullable add завершает migration. **Симптом:** новый reader получает `NULL` на старых rows. **Причина:** history не backfilled и не validated. **Исправление:** fallback, resumable backfill и отдельный completeness gate.
- **Неверное предположение:** backfill безопасно заполняет всё без условия. **Симптом:** свежий user update заменён старым значением. **Причина:** snapshot race. **Исправление:** compare-and-set по source version или skip изменившихся строк.
- **Неверное предположение:** rollback всегда означает развернуть старый binary. **Симптом:** изменения, записанные только в новое поле, исчезают для старого кода. **Причина:** источник истины уже переключён. **Исправление:** отмечать irreversible boundary и держать reverse sync до окончания окна.
- **Неверное предположение:** online DDL не блокирует. **Симптом:** production queries копятся за `ALTER TABLE`. **Причина:** ожидание несовместимого lock или scan/rewrite. **Исправление:** изучить точный lock версии, задать timeout, репетировать на объёме и отделить validation.

## Когда применять

Любое изменение, при котором одновременно живут две версии приложения, нужно проектировать как protocol, даже если SQL состоит из одной строки. Для каждого шага записывают compatibility, source of truth, критерий готовности, мониторинг и rollback. Особенно это важно для rename/drop, нового `NOT NULL`, изменения типа, нового индекса и переноса данных между таблицами/хранилищами.

Перед contract проверяют не только deployment dashboard: нет ли старых jobs, ad-hoc clients, ETL, views, prepared statements и replica readers. Дешевле подержать legacy column ещё один release, чем обнаружить скрытого consumer после необратимого удаления.
