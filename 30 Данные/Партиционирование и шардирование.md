# Партиционирование и шардирование

## TL;DR

Партиционирование (partitioning) делит логический набор данных на непересекающиеся части по функции от ключа. Шардирование (sharding) добавляет к этому физическое распределение частей между независимыми узлами или группами реплик. Поэтому PostgreSQL-таблица, разбитая по месяцам на одном сервере, партиционирована, но не шардирована.

Удачный partition/shard key одновременно даёт однозначный routing, равномерную нагрузку и совместное размещение данных, которые нужны одной транзакции или запросу. Эти цели конфликтуют: равномерный hash уничтожает диапазонную локальность, а естественный последовательный диапазон создаёт горячий хвост. Выбор ключа — это выбор будущих локальных и распределённых операций.

## Область применимости и версии

Рассматривается горизонтальное деление строк или ключей. Вертикальное разнесение столбцов и функциональная декомпозиция сервисов — соседние, но другие решения. Конкретные утверждения о declarative partitioning относятся к PostgreSQL 18.4 (`REL_18_4`), о tablets — к Bigtable из статьи OSDI 2006, о directory placement — к Spanner из статьи OSDI 2012, о partition key — к Apache Cassandra 5.0.8; проверено 2026-07-18.

## Ментальная модель

Партиционирование задаётся тремя функциями:

1. `partition(key) -> logical part` — к какой части относится запись;
2. `owner(part, epoch) -> node/group` — кто владеет частью в этой версии topology;
3. `route(request) -> owners` — куда отправить конкретную операцию.

Первая функция определяет границы данных. Вторая превращает части в shards и может меняться при [[30 Данные/Rebalancing данных|rebalancing]]. Третья обязана видеть согласованную версию карты владения, иначе клиент пишет старому владельцу, пока новый уже обслуживает диапазон.

Изоляция частей даёт масштабирование только для операций, которые можно выполнить локально. Запрос без shard key становится scatter/gather: его нужно отправить многим shards, дождаться ответов и глобально объединить сортировку, limit или aggregation. Транзакция, уникальность или foreign key через несколько shards требуют отдельного распределённого протокола либо ослабления инварианта.

## Как устроено

### Partitioning не равно sharding

PostgreSQL declarative partitioning представляет одну логическую таблицу набором физических partitions. Корневая partitioned table сама строк не хранит; `RANGE`, `LIST` или `HASH`-правило направляет строку в leaf partition, а partition pruning исключает заведомо ненужные части из плана. Все части при этом могут находиться в одном экземпляре PostgreSQL и разделять его CPU, память, журнал и failure domain.

Sharding размещает части на разных узлах или группах реплик. Это добавляет:

- каталог placement и его версии;
- сетевой routing;
- распределённое выполнение запросов и транзакций;
- перенос владения и байтов при изменении состава;
- обработку частичных отказов, когда одни shards доступны, а другие нет.

Репликация ортогональна этому делению: shard отвечает за подмножество ключей, а [[30 Данные/Репликация данных|replica group]] хранит несколько копий одного shard.

### Стратегии границ

**Range partitioning** сохраняет порядок. Bigtable хранит строки лексикографически по row key и динамически делит диапазоны на tablets. Это удобно для сканов по префиксу или времени, но монотонно растущий ключ направляет новые записи в один крайний диапазон.

**List partitioning** явно сопоставляет категории частям: регион, тип клиента, tenant tier. Оно хорошо отражает предметные границы, но требует политики для нового или неизвестного значения и может дать сильный skew.

**Hash partitioning** сначала равномерно отображает ключ в hash space, затем делит пространство. Оно обычно лучше распределяет независимые ключи, но теряет естественный порядок: диапазонный запрос по исходному ключу превращается в fan-out. [[30 Данные/Consistent hashing|Consistent hashing]] решает более узкий вопрос — как менять owners с меньшим remapping при изменении узлов.

Практические системы комбинируют стратегии. Например, `tenant_id` выбирает shard, а время — локальную partition внутри shard. Тогда tenant-транзакции локальны, исторические partitions можно обслуживать отдельно, но один очень крупный tenant всё равно становится [[30 Данные/Hot partitions и hot keys|hot partition]].

### Ключ определяет границу атомарности

В Cassandra 5.0.8 partition key вычисляет token и набор реплик; строки с одним partition key размещаются вместе, а clustering columns упорядочивают их внутри partition. Это делает операции внутри partition естественно локальными, но слишком широкая partition растёт без границы, а слишком популярная — упирается в один replica set.

В Spanner directory — непрерывный диапазон ключей с общим placement. Перенос directory между Paxos groups позволяет менять физическое размещение без смены логического ключа. Общий принцип тот же: граница placement должна совпадать с самой частой границей локальной работы, но оставаться достаточно мелкой для балансировки.

### Инварианты маршрутизации

- Каждая допустимая запись принадлежит ровно одной логической partition в данной schema version: без overlap и gap, кроме явной default partition.
- Router и storage согласны о topology epoch либо старый owner умеет перенаправить/форвардить запрос.
- Изменение partition function версионируется; нельзя одновременно считать ключ по старой и новой формуле без migration protocol.
- Глобальный инвариант либо реализован поверх всех shards, либо явно ослаблен до локального. «Наверное, связанные строки окажутся рядом» — не контракт.

## Пример или трассировка

Есть таблица `orders(tenant_id, created_at, order_id, amount)`.

### Только partitioning

В одном PostgreSQL 18.4 она разбита по `created_at` на месячные partitions `orders_2026_06` и `orders_2026_07`.

1. Запрос за июль содержит границы `created_at`.
2. Planner отсекает июньскую partition.
3. Июльский index scan читает меньше страниц.
4. Отказ единственного экземпляра делает недоступными обе partitions.

Наблюдаемый результат: pruning ускорил запрос и упростил обслуживание истории, но не добавил ни одного независимого вычислительного или отказного домена.

### Sharding

Теперь routing задаётся `hash(tenant_id) mod 2`: tenants `42` и `77` попали на разные shards, у каждого своя replica group.

1. `GET /tenants/42/orders` содержит shard key и идёт в один shard.
2. Транзакция «создать заказ и увеличить счётчик tenant 42» остаётся локальной, если обе строки co-located по `tenant_id`.
3. Запрос «топ-10 заказов по всем tenants» идёт в оба shards; каждый отдаёт локальный top-10, coordinator сливает результаты и снова применяет global top-10.

Наблюдаемый результат: tenant-запрос масштабируется добавлением shards, а глобальный запрос приобрёл fan-out, сетевой merge и частичную недоступность. Если `tenant 42` создаёт половину всех заказов, hash распределяет tenants, но не нагрузку внутри одного tenant.

## Trade-offs

| Выбор | Сильная сторона | Ограничение |
|---|---|---|
| Range | Локальные range scans, естественная сортировка | Горячий хвост и неравные диапазоны |
| List | Понятное предметное размещение и compliance | Ручная эволюция списка, skew категорий |
| Hash | Равномернее независимые ключи | Потеря локальности, fan-out диапазонных запросов |
| Крупные partitions | Меньше metadata и routing overhead | Грубая балансировка, долгий перенос, hot partition |
| Мелкие partitions | Точное размещение и параллельность | Больше metadata, файлов, фоновой работы и coordination |
| Co-location по tenant | Локальные joins и транзакции tenant | «Кит»-tenant ограничен ёмкостью одного shard |

Ближайшая альтернатива sharding — scale-up и read replicas. Они сохраняют простые глобальные транзакции и запросы, поэтому часто дешевле до достижения реального предела одного write domain. Sharding оправдан, когда ограничение подтверждено ёмкостью, write throughput, географией или изоляцией tenants, а не просто ростом таблицы.

## Типичные ошибки

- **Неверное предположение:** declarative partitions автоматически распределяют нагрузку между серверами. **Симптом:** CPU и WAL одного узла остаются пределом. **Причина:** изменилось физическое устройство таблицы, но не placement. **Исправление:** отдельно определить, нужен ли sharding, и кто хранит каталог owners.
- **Неверное предположение:** высокая кардинальность ключа гарантирует равномерность. **Симптом:** один shard перегружен. **Причина:** важна частота и вес значений, а не только их число. **Исправление:** измерять распределение bytes, QPS и CPU на реальном workload.
- **Неверное предположение:** любой запрос можно «просто распараллелить». **Симптом:** p99 растёт с числом shards, `LIMIT` или aggregation дают неверный результат. **Причина:** fan-out ждёт хвост и требует глобального merge. **Исправление:** проектировать двухфазное выполнение и bounded fan-out либо материализовать отдельный read model.
- **Неверное предположение:** hash решит hot key. **Симптом:** один replica set остаётся пределом после добавления узлов. **Причина:** один ключ всегда хешируется в одну logical partition. **Исправление:** менять предметную гранулярность ключа, агрегировать или кэшировать нагрузку.
- **Неверное предположение:** shard key можно безболезненно заменить. **Симптом:** дубли, пропуски и записи на старом узле во время переноса. **Причина:** меняются и данные, и routing. **Исправление:** использовать версионированный migration protocol с backfill, проверкой и явной точкой cutover.

## Когда применять

Начинают с top queries и write paths: какие поля всегда известны в запросе, какие строки должны быть атомарны, где нужны диапазоны, каков skew и насколько дорог fan-out. После этого выбирают partition function и только затем число частей и placement.

Хороший проект оставляет запас на рост: logical partitions мельче физического узла, owners можно менять без смены ключа, а каталог topology наблюдаем и версионирован. На эксплуатации нужны метрики размера, QPS, latency и фоновой работы по каждой partition, а также лимиты на fan-out. Среднее по кластеру скрывает именно тот локальный предел, ради которого деление и вводилось.
