# LSM-tree

## TL;DR

- **Log-Structured Merge-Tree (LSM-tree)** принимает запись в память и последовательный [[30 Данные/Write-ahead log (WAL)|WAL]], затем сбрасывает отсортированные неизменяемые файлы (**SST, Sorted String Table**) и объединяет их в фоне.
- Новая версия ключа не перезаписывает старую на месте. На чтении система находит все потенциальные версии и выбирает самую новую, видимую для snapshot; удаление сначала представлено tombstone.
- Такой дизайн превращает случайные мелкие записи в крупные последовательные, но переносит цену в чтение, [[30 Данные/Compaction в LSM-хранилищах|compaction]], временное место и tail latency.
- WAL отвечает за восстановление ещё не сброшенной memtable, а MANIFEST/Version metadata — за то, какой набор SST-файлов образует согласованную версию базы.
- «LSM-tree» — семейство. Число уровней, overlapping runs, filters, compaction policy и гарантии API нужно уточнять для конкретного движка.

## Область и версии

Базовая идея сопоставлена с исходной статьёй O’Neil et al. 1996 года. Современный многослойный вариант разобран на RocksDB 11.1.1, tag `v11.1.1`, commit `6cdeb9d9d0630763327f512e6255cab33f6834e7`; официальная wiki зафиксирована на commit `98273e83a8c7897bdbcd01ffbaef39571ba949ee`. Проверено 2026-07-18.

Речь идёт о внутренней организации локального storage engine. Репликация, consensus и межузловая согласованность поверх LSM — отдельные уровни системы.

## Ментальная модель

У изменяемого B+tree запись находит конкретную страницу и меняет её. LSM-tree не пытается сразу положить значение в окончательное место. Он последовательно накапливает новые версии, а порядок восстанавливает пакетно:

```text
Write → WAL → mutable memtable
                  ↓ freeze
             immutable memtable
                  ↓ flush
              L0 SST files
                  ↓ compaction
             L1 → L2 → ...
```

Каждый SST отсортирован и после создания не меняется. Новая версия ключа определяется не положением файла, а внутренним порядком, обычно включающим user key, sequence number и тип записи (`value` или `deletion`). Это позволяет одновременно читать старые immutable файлы и строить новую версию без блокировки читателей.

Три независимых вида состояния важны для recovery:

- **WAL** содержит изменения, которые могли не попасть в SST;
- **SST** содержит immutable данные;
- **MANIFEST/Version metadata** задаёт, какие SST входят в текущую согласованную версию.

## Как проходит запись

1. Движок назначает операции sequence number или другой порядок версий.
2. Batch добавляется в WAL. Для синхронной durability WAL доводится до устойчивого носителя до успешного ответа.
3. Те же операции добавляются в mutable memtable.
4. После достижения лимита memtable замораживается; новая memtable продолжает принимать запись.
5. Immutable memtable сортированно выгружается в новый SST, обычно на level 0.
6. После создания файла Version metadata атомарно публикует его. WAL можно удалить только когда все column families, зависевшие от него, устойчиво представлены в SST.

В RocksDB обычный write path пишет WAL до вставки в memtable. Это порядок внутри процесса; durability ответа дополнительно зависит от `WriteOptions.sync` и настроек файловой системы/устройства.

Если memtable заполняются быстрее, чем flush и compaction освобождают место, движок замедляет или останавливает новые записи. Это backpressure, а не случайная неисправность: без него число runs и расход диска росли бы без границы.

## Как проходит чтение

### Точечный lookup

Движок проверяет mutable и immutable memtables, затем подходящие SST от новых к старым. Индекс блока указывает, где искать внутри SST; Bloom filter может доказать отсутствие ключа и не читать data block. Положительный ответ filter не доказывает наличие: false positive всё равно вызывает чтение.

В level 0 файлы обычно перекрываются по диапазонам, поэтому один key может потребовать проверки нескольких SST. На ненулевом уровне leveled layout обычно образует один логический sorted run, разбитый на файлы без взаимного перекрытия: для key подходит не более одного файла уровня. Tiered layout допускает несколько runs и увеличивает число кандидатов.

Из найденных внутренних записей выбирается самая новая версия, видимая для snapshot. Tombstone означает «ключ удалён начиная с этой версии», но старое значение нельзя физически выбросить, пока оно может понадобиться старому snapshot или всё ещё встречается глубже.

### Range scan

Для диапазона движок открывает итераторы memtables и всех пересекающихся runs, затем выполняет k-way merge по ключу и sequence number. Filters, полезные для отсутствующего точечного key, не устраняют саму необходимость слить диапазоны. Чем больше runs, тем выше CPU, I/O и heap итератора.

## Сквозной пример

Пусть memtable вмещает четыре операции. Последовательно выполняются:

```text
seq=1: put(a, 10)
seq=2: put(b, 20)
seq=3: put(a, 11)
seq=4: delete(b)
```

WAL и memtable содержат четыре внутренние записи, хотя логически жив только `a=11`:

```text
a@3 = 11
a@1 = 10
b@4 = DEL
b@2 = 20
```

После freeze и flush создаётся `SST-7`, а MANIFEST публикует новую версию с этим файлом.

Наблюдаемый результат:

- чтение `a` при snapshot `seq=4` возвращает `11`, потому что `a@3` новее `a@1`;
- чтение `b` возвращает `not found`, потому что `b@4=DEL` скрывает `b@2`;
- snapshot `seq=2`, если он всё ещё открыт, видит `a=10` и `b=20`, поэтому старые версии пока нельзя удалить;
- crash после durable WAL, но до публикации `SST-7`, приводит к replay WAL и восстановлению memtable;
- crash после публикации `SST-7` восстанавливает набор файлов по MANIFEST; незавершённый файл, не включённый в версию, не становится частью логической базы.

Поздний compaction может оставить только `a@3=11` и безопасно убрать старые записи, когда нет snapshot и более глубоких версий, для которых tombstone ещё нужен.

## Эволюция и версии

В статье 1996 года базовая модель описана через небольшой компонент `C0` в памяти и большой `C1` на диске с rolling merge. Современные движки вроде RocksDB сохранили принцип пакетного переноса отсортированных данных, но добавили WAL, несколько memtables, много SST, уровни, block cache, filters, snapshots и разные compaction policies.

Практический эффект: выводы «LSM всегда имеет два уровня» или «LSM всегда использует один конкретный compaction» неверны. Статья задаёт семейство и cost model; конкретный layout нужно проверять по версии движка.

## Trade-offs

### LSM-tree против изменяемого B+tree

| Свойство | LSM-tree | B+tree/B+tree-подобный индекс |
|---|---|---|
| Мелкая запись | WAL + память, затем пакетный flush | Изменение целевой страницы, иногда split |
| Физический паттерн записи | Крупные последовательные файлы и compaction | Случайные страничные обновления и WAL |
| Точечное чтение | Несколько компонентов; filters/cache критичны | Один путь от корня к листу |
| Диапазон | Merge нескольких runs | Последовательные листья после одного спуска |
| Фоновая цена | Обязательный compaction | Нет LSM-compaction, но возможны vacuum/rebuild |
| Пространство | Старые версии, tombstones, временные outputs | Свободное место страниц и MVCC-версии зависят от движка |

LSM не «делает запись бесплатной». Он откладывает и группирует сортировку, создавая [[30 Данные/Read и write amplification|write, read и space amplification]]. Выигрыш появляется, если последовательная пакетная работа дешевле случайных обновлений и система располагает I/O-бюджетом для compaction.

### Leveled против tiered

Leveled compaction обычно уменьшает число runs для чтения и space amplification, но переписывает данные при продвижении по уровням. Tiered compaction дольше сохраняет несколько runs: запись дешевле, чтение и временное место дороже. Подробный механизм — в [[30 Данные/Compaction в LSM-хранилищах|заметке о compaction]].

## Типичные ошибки

### Считать memtable durable

- **Неверное предположение:** успешная вставка в память уже переживёт выключение.
- **Симптом:** после crash исчезает недавно подтверждённая запись.
- **Причина:** memtable volatile, а WAL не был синхронизирован до ответа.
- **Исправление:** связать обещание API с точной политикой WAL sync и проверить её тестом аварийного отключения.

### Искать ключ только в одном уровне

- **Неверное предположение:** более глубокий SST всегда содержит окончательное значение.
- **Симптом:** чтение возвращает старую версию или «воскрешает» удалённый ключ.
- **Причина:** новая версия или tombstone остаётся в memtable/L0, пока compaction не перенёс её ниже.
- **Исправление:** искать все потенциальные runs и разрешать версии по sequence number/snapshot.

### Удалять tombstone сразу

- **Неверное предположение:** tombstone больше не содержит данных, значит его можно выбросить.
- **Симптом:** после compaction появляется старое значение.
- **Причина:** значение сохранилось в непросмотренном нижнем уровне, а маркер, который его скрывал, исчез.
- **Исправление:** удалять tombstone только при доказательстве отсутствия старых версий ниже и потребителей старого snapshot.

### Считать Bloom filter ускорителем любого чтения

- **Неверное предположение:** filter решает и point lookup, и большой range scan.
- **Симптом:** диапазоны остаются дорогими, несмотря на память для filters.
- **Причина:** filter хорошо исключает SST для конкретного отсутствующего key, но диапазон всё равно пересекает runs.
- **Исправление:** отдельно моделировать point misses, point hits и range scans.

### Игнорировать compaction debt

- **Неверное предположение:** фоновая работа когда-нибудь догонит сама.
- **Симптом:** растут L0 files, read latency и место, затем начинаются write stalls.
- **Причина:** sustained ingest превышает доступную скорость flush/compaction.
- **Исправление:** ограничить ingest или дать compaction I/O/CPU, настроить thresholds и держать запас диска.

## Когда применять

LSM-подобный движок подходит, когда доминирует большой поток вставок/обновлений, допустима фоновая перегруппировка, а hardware выгодно обслуживает последовательную запись. До выбора нужно оценить:

1. долю point hits, point misses и диапазонов;
2. максимальное число runs, которое попадёт в read path;
3. write/space budget compaction и запас диска;
4. допустимый tail latency во время flush и compaction;
5. срок жизни snapshots и tombstones;
6. точную durability семантику WAL и MANIFEST.
