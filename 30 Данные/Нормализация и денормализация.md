# Нормализация и денормализация

## TL;DR

Нормализация раскладывает отношение так, чтобы один независимый факт записывался в одном месте. Она снижает update, insert и delete anomalies и позволяет ограничениям защищать источник истины. Главный инструмент рассуждения: functional dependency `X -> Y`, где одинаковый `X` требует одинаковый `Y`.

Денормализация намеренно хранит производное или повторённое значение ради конкретного пути чтения. Это решение полно только вместе с владельцем истины, протоколом обновления, допустимым lag и repair. «Меньше joins» без этих четырёх пунктов означает скрытую потерю согласованности.

## Область применимости

- Рассматриваются 1NF, 2NF, 3NF, BCNF и граница 4NF на уровне, достаточном для OLTP-проектирования.
- Нормальные формы принадлежат реляционной теории и не зависят от PostgreSQL. DDL-пример соответствует PostgreSQL 18.4.
- PostgreSQL хранит arrays и `jsonb`, поэтому «atomic value» нельзя свести к физической неделимости. Практический тест 1NF: значение имеет один тип и семантику, а повторяющаяся группа не спрятана от нужных relational operations.
- Вне scope: 5NF/6NF, dependency inference algorithms и dimensional star schemas для OLAP.
- Пример не исполнялся локально; синтаксис и ожидаемые constraints сверены с документацией PostgreSQL 18.

## Ментальная модель

Одна ячейка знания должна иметь один адрес изменения. Если имя товара повторено в десяти тысячах строк заказа, команда «переименовать товар» не знает, какой набор копий образует один факт. Пропущенная строка оставит противоречие.

Functional dependency делает проблему точной. Запись `product_id -> product_name` говорит: две строки с одинаковым `product_id` не могут расходиться по `product_name`. Если determinant не служит ключом текущего отношения, одна сущность размазана по нескольким строкам. Декомпозиция даёт этому факту собственное отношение.

## Как устроено

### Зависимости и аномалии

Пусть исходная таблица имеет ключ `(order_id, product_id)` и столбцы:

```text
order_id, customer_id, customer_email, product_id, product_name, quantity
```

Зависимости предметной области:

```text
order_id -> customer_id
customer_id -> customer_email
product_id -> product_name
(order_id, product_id) -> quantity
```

Отсюда следуют три сбоя. Update anomaly требует менять `product_name` во всех заказах. Insert anomaly не даёт завести товар до первого заказа. Delete anomaly удаляет последнее известное имя товара вместе с последней строкой заказа.

### Что проверяют формы

**1NF** требует relation-shaped данные без повторяющихся групп вроде `product_1`, `product_2`, `product_3`. Каждый кортеж имеет одинаковый набор атрибутов, а операции работают со значениями доменов, не разбирая склеенную строку.

**2NF** касается composite candidate key: non-prime attribute не должен зависеть от части ключа. В примере `product_name` зависит только от `product_id`, поэтому его выносят в `product`.

**3NF** допускает нетривиальную dependency `X -> A`, только если `X` служит superkey либо `A` входит хотя бы в один candidate key (то есть это prime attribute). Практически она убирает transitive dependency non-key attribute от ключа через другой non-key attribute. `order_id -> customer_id -> customer_email` требует разделить `customer` и `customer_order`.

**BCNF** формулируется строже: для каждой нетривиальной functional dependency `X -> Y` determinant `X` должен быть superkey. Иногда decomposition в BCNF теряет возможность проверять dependency одним constraint; тогда 3NF с dependency preservation может быть практичнее.

**4NF** разбирает независимые multivalued dependencies. Если у курса независимо несколько преподавателей и несколько учебников, одна таблица создаст их Cartesian product. Два отношения `course_teacher` и `course_book` сохранят факты без ложных комбинаций.

### Lossless join и dependency preservation

Разделить таблицу мало. Декомпозиция должна быть lossless: natural join частей восстанавливает ровно допустимые исходные кортежи, без потери и без spurious tuples. Желательно также сохранить dependencies так, чтобы каждую можно было проверить локальными keys/constraints, без join при каждой записи.

### Осознанная денормализация

Повторение оправдано, если измеренный read path дороже протокола поддержки копии. Варианты различаются по consistency:

- stored generated column вычисляется из той же строки;
- materialized view обновляется по команде и имеет явный staleness;
- summary table поддерживается транзакцией или trigger в той же базе;
- read model через CDC обновляется асинхронно и требует idempotency, checkpoint и reconciliation.

Исторический snapshot, например цена в строке заказа, не всегда денормализация. Если он описывает значение в момент события, это самостоятельный факт с другим временем действия.

## Пример или трассировка

Нормализованная схема для приведённых зависимостей:

```sql
CREATE TABLE customer (
    customer_id bigint PRIMARY KEY,
    email text NOT NULL UNIQUE
);

CREATE TABLE product (
    product_id bigint PRIMARY KEY,
    name text NOT NULL
);

CREATE TABLE customer_order (
    order_id bigint PRIMARY KEY,
    customer_id bigint NOT NULL REFERENCES customer
);

CREATE TABLE order_line (
    order_id bigint NOT NULL REFERENCES customer_order ON DELETE CASCADE,
    product_id bigint NOT NULL REFERENCES product,
    quantity integer NOT NULL CHECK (quantity > 0),
    PRIMARY KEY (order_id, product_id)
);

INSERT INTO customer VALUES (7, 'a@example.test');
INSERT INTO product VALUES (10, 'old name');
INSERT INTO customer_order VALUES (100, 7), (101, 7);
INSERT INTO order_line VALUES (100, 10, 1), (101, 10, 2);

UPDATE product SET name = 'new name' WHERE product_id = 10;

SELECT ol.order_id, p.name, ol.quantity
FROM order_line AS ol
JOIN product AS p USING (product_id)
ORDER BY ol.order_id;
```

Ожидаются две строки с одинаковым `name = 'new name'` и quantities `1`, `2`. Факт имени изменён одной строкой. Если отчёту нужен заранее собранный `product_name`, его копия должна ссылаться на `product` как источник истины и иметь измеримый способ обнаружить и исправить drift.

## Trade-offs

- 3NF/BCNF уменьшают число мест записи и делают ограничения локальными. Чтение агрегата часто требует joins; правильные [[30 Данные/Индексы и цена чтения и записи|индексы]] обычно дешевле постоянной синхронизации копий.
- BCNF сильнее удаляет redundancy. 3NF иногда сохраняет dependency, которую после BCNF пришлось бы проверять join или trigger.
- JSON/array удерживает агрегат в одной строке и может совпадать с transaction boundary. Частичные обновления, foreign keys к элементам и ad hoc аналитика становятся сложнее.
- Синхронная денормализация даёт read-after-write внутри commit boundary, но увеличивает write amplification и contention. Асинхронная снижает latency основной записи, зато читатель должен переносить lag и duplicate/out-of-order delivery.
- Полная нормализация не лечит плохую семантику. Если два атрибута с одинаковым именем описывают разные моменты времени, их сначала надо развести как факты.

## Типичные ошибки

- Неверное предположение: каждая повторяющаяся строка нарушает нормальную форму. Симптом: историческую цену заменяют ссылкой на текущую и переписывают прошлые заказы. Причина: не различены текущий и событийный факты. Исправление: задать valid time и ownership каждого значения до анализа dependencies.
- Неверное предположение: surrogate primary key автоматически даёт 3NF. Симптом: таблица с `id` продолжает повторять customer email и product name. Причина: functional dependencies между business attributes никуда не исчезли. Исправление: анализировать candidate keys и determinants, а не выбранный технический PK.
- Неверное предположение: decomposition всегда безопасна. Симптом: join создаёт комбинации, которых не было, или constraint больше нельзя проверить локально. Причина: не проверены lossless join и dependency preservation. Исправление: доказать свойства декомпозиции на dependencies и candidate keys.
- Неверное предположение: join сам по себе медленный. Симптом: данные копируют до измерения, writes дорожают, копии расходятся. Причина: physical plan и индексы не исследованы. Исправление: измерить `EXPLAIN (ANALYZE, BUFFERS)`, исправить доступ, затем сравнить с полной ценой read model.
- Неверное предположение: денормализованное поле обновится «как-нибудь». Симптом: API показывает старый total после успешной оплаты. Причина: нет owner, ordering, lag budget и repair. Исправление: записать consistency contract, сделать обновление идемпотентным и добавить reconciliation.

## Когда применять

Для OLTP начинайте с 3NF или BCNF, опираясь на реальные dependencies и constraints. Не проводите нормализацию механически по названиям столбцов: одна и та же строка может содержать snapshots, события и текущие состояния с разной семантикой.

Денормализуйте после измерения конкретного запроса. Решение должно назвать source of truth, atomicity boundary, максимальный lag, поведение при сбое и repair job. Если эти пункты слишком дороги, join остаётся более простым и надёжным выбором.
