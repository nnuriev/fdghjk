# Read и write quorums

## TL;DR

Для `N` реплик write quorum `W` — сколько подтверждений нужно записи, read quorum `R` — сколько ответов участвует в чтении. При одном фиксированном наборе реплик условие `R + W > N` гарантирует, что read set пересечётся с acknowledged write set хотя бы в одной реплике.

Это лишь геометрия множеств. Она не доказывает linearizability: пересечение бесполезно, если чтение не распознаёт новую версию, concurrent writes не упорядочены, topology изменила набор, sloppy quorum использовал substitutes или ответ выбран по некорректным часам. Строгая гарантия требует протокола версий, членства и завершения операции, а не только трёх чисел.

## Область применимости и версии

Модель `N/R/W` описана для Dynamo 2007 и используется consistency levels Apache Cassandra 5.0.8; проверено 2026-07-18. В разных системах `W=QUORUM` может означать durable log, in-memory acknowledgement, quorum текущей конфигурации или локального datacenter. Перед переносом формулы нужно прочитать точный контракт продукта.

## Ментальная модель

Пусть все операции относятся к одному ключу и одному **фиксированному** universe из `N` distinct physical replicas. Запись версии `v` подтверждена множеством `Wset`, чтение получает ответы от `Rset`.

По принципу включения-исключения:

```text
|Rset ∩ Wset| = |Rset| + |Wset| - |Rset ∪ Wset|
              >= R + W - N
```

Следовательно, `R + W > N` даёт минимум одну общую реплику. При `R + W = N` множества могут быть непересекающимися. Для пересечения любых двух write quorums отдельно нужно `2W > N`; оно важно, если writes должны видеть/упорядочивать друг друга.

Общая реплика — только носитель свидетельства. Протокол ещё должен спросить её, получить ответ, понять, что `v` новее альтернатив, не вернуть результат до нужного числа ответов и при необходимости исправить stale copies.

## Как устроено

### Tunable consistency

Изменяя `R` и `W`, систему сдвигают между latency, availability и свежестью:

- маленький `W` отвечает после быстрых replicas и терпит больше недоступных узлов, но оставляет меньше подтверждённых копий;
- большой `W` повышает стоимость и вероятность тайм-аута записи;
- маленький `R` ускоряет чтение, но легче попадает в stale replica;
- большой `R` видит больше версий, зато ждёт более медленный хвост и создаёт read amplification.

В Dynamo координатор рассылает операцию по preference list и ждёт `W` или `R`. При отказах sloppy quorum может выбрать следующие здоровые узлы за пределами первых `N`, а hinted handoff позже передаёт данные исходным owners. Это повышает availability, но нарушает предпосылку одной фиксированной вселенной для простого доказательства пересечения.

Cassandra 5.0.8 отправляет mutation всем естественным replicas, а consistency level определяет, скольких acknowledgements ждать; чтение аналогично ждёт заданный набор ответов. Документация формулирует `W + R > RF` как условие перекрытия для сильного read-after-write в описанной конфигурации. Это продуктовый контракт при его topology, timestamp и reconciliation rules, а не общее доказательство linearizability любой leaderless-базы.

### Почему пересечение не равно linearizability

Linearizability требует, чтобы каждая завершённая операция выглядела атомарной в едином порядке, совместимом с реальным временем. `R + W > N` не покрывает как минимум следующие условия:

1. **Одинаковый replica set.** Sloppy quorum или незавершённый rebalance дают разные множества кандидатов.
2. **Сравнимые версии.** Один участник видел `v2`, но coordinator может выбрать `v1`, если timestamps повреждены или merge rule не соответствует причинности.
3. **Concurrent writes.** Две записи могут получить по `W` acknowledgements и образовать siblings; одно пересечение не назначает им порядок.
4. **Durability.** Acknowledgement из volatile memory исчезнет вместе с несколькими процессами.
5. **Membership.** Старое и новое `N` без joint transition могут оба собрать «кворум» в разных конфигурациях.
6. **Operation protocol.** Обычный read не превращает read-modify-write в compare-and-set; lost update остаётся возможным.

Для linearizable register нужны дополнительные правила — например, consensus leader, quorum read с монотонными logical versions и write-back либо алгоритм наподобие ABD в его полной модели. Название `QUORUM` само этих правил не добавляет.

### Partial failures и тайм-аут

При `N=3, W=2` запись доступна, пока достижимы две подходящие replicas; при `R=2` чтение имеет тот же предел. `W=3` не переживает одну недоступную replica, хотя остальные две могли безопасно хранить значение. Тайм-аут после частичных acknowledgements даёт **неизвестный исход**: клиент не знает, достигнут ли `W`. Повтор должен быть идемпотентным или использовать стабильный operation ID.

Quorum определяет число ответов, но placement определяет, какие отказы независимы. Три replicas в одном rack не дают zone tolerance, а `LOCAL_QUORUM` сознательно меняет межрегиональный контракт ради latency.

## Пример или трассировка

### Строгое пересечение

Для ключа `k` replicas — `A`, `B`, `C`, то есть `N=3`; выбраны `W=2`, `R=2`.

1. Запись `v2` подтверждают `A` и `B`; `C` пока хранит `v1`.
2. Самый неблагоприятный read set из двух — `B` и `C` или `A` и `C`.
3. В обоих случаях есть хотя бы одна replica с `v2`, потому что `R + W - N = 2 + 2 - 3 = 1`.
4. Если coordinator корректно сравнивает версии, он возвращает `v2` и может запустить read repair для `C`.

Наблюдаемый результат: чтение после завершённой записи не может опросить две replicas, обе не участвовавшие в записи; таких replicas всего одна.

При `R=1` получаем `1 + 2 = 3`, не строго больше `N`. Чтение только с `C` возвращает `v1`. Наблюдаемый результат меняется ровно там, где исчезло гарантированное пересечение.

### Sloppy quorum ломает предпосылку, а не арифметику

Номинально owners всё ещё `A`, `B`, `C`, `N=3`, `R=W=2`. Во время partition запись `v2` подтверждают достижимая `A` и временная substitute `D`. Позднее coordinator в другой части сети читает старые `B` и `C`.

Фактические множества `{A,D}` и `{B,C}` не пересекаются, хотя численно `2+2>3`. Наблюдаемый результат: возможно чтение `v1` до hinted handoff/repair. Ошибка не в формуле; фактический universe стал больше номинального `N`.

## Trade-offs

| Конфигурация при `N=3` | Поведение | Ограничение |
|---|---|---|
| `R=1, W=3` | При fixed set любая owner имеет подтверждённую версию | Write недоступна при одном отказе |
| `R=2, W=2` | Пересечение при fixed set, симметричный budget | И reads, и writes ждут большинство |
| `R=3, W=1` | Read видит все доступные версии | Read недоступен при одном отказе; write имеет мало подтверждённых копий |
| `R=1, W=1` | Минимальная latency и высокая доступность | Stale reads и большой conflict/repair window |

Ближайшая альтернатива tunable leaderless quorum — leader-based consensus. Он дороже координацией большинства на write path, зато даёт единую точку порядка; linearizable read достигается чтением у актуального лидера с подтверждёнными полномочиями или отдельным quorum/read-index protocol. Выбор зависит не от желания «strong consistency вообще», а от инвариантов конкретной операции.

## Типичные ошибки

- **Неверное предположение:** `R + W > N` автоматически даёт linearizability. **Симптом:** после успешной записи читается старое значение или concurrent writes расходятся. **Причина:** выполнена только предпосылка пересечения. **Исправление:** проверить membership, version order, write conflicts, read selection и real-time guarantee продукта.
- **Неверное предположение:** `QUORUM` одинаков во всех datacenters. **Симптом:** локальный quorum подтверждён, удалённый регион ещё не видит запись. **Причина:** consistency level ограничен локальным replica subset. **Исправление:** явно документировать failure scope каждого CL.
- **Неверное предположение:** timeout означает failure записи. **Симптом:** retry дважды применяет increment или платёж. **Причина:** часть replicas могла собрать достаточное число acknowledgements после client timeout. **Исправление:** идемпотентный operation ID и чтение статуса/верификация результата.
- **Неверное предположение:** `N=3` означает три независимых отказа. **Симптом:** один rack outage теряет quorum. **Причина:** placement не учитывал failure domains. **Исправление:** разносить replicas по racks/zones и тестировать именно общий отказ.
- **Неверное предположение:** увеличение `R` бесплатно улучшает свежесть. **Симптом:** read p99 и bandwidth резко растут. **Причина:** ответ ждёт медленную replica и сравнивает больше данных. **Исправление:** измерять tail latency, использовать подходящий CL и плановый anti-entropy repair.

## Когда применять

Tunable quorums полезны, когда операции имеют разные требования: например, telemetry допускает `W=1`, а изменение конфигурации требует пересечения и проверки версии. Выбор делают для каждой операции из допустимых stale window, latency, числа одновременно недоступных failure domains и цены конфликта.

В design review числа записывают вместе с предпосылками: `N` distinct hosts в каких zones, strict или sloppy quorum, что значит durable acknowledgement, какая version/reconciliation policy, как проходит topology transition. Только такая формулировка позволяет предсказать поведение при partition; `RF=3, QUORUM` без контекста — название настройки, а не доказательство гарантии.
