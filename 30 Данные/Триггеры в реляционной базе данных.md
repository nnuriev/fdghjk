# Триггеры в реляционной базе данных

## TL;DR

Trigger связывает изменение relation с автоматически вызываемой database function. В PostgreSQL 18.4 выбор задают три оси: момент `BEFORE`/`AFTER`/`INSTEAD OF`, granularity `ROW`/`STATEMENT` и событие `INSERT`/`UPDATE`/`DELETE`/`TRUNCATE`. Row trigger видит `OLD`/`NEW`; transition tables дают `AFTER` trigger набор строк, изменённых statement целиком.

Trigger выполняется внутри транзакции statement: его database effects коммитятся и откатываются вместе с вызывающей операцией. Это делает trigger сильным инструментом для локального инварианта или transactional audit/outbox, но не превращает внешний HTTP-вызов в часть SQL-транзакции. Главная цена — скрытый control flow: write latency, блокировки, рекурсия и дополнительные записи появляются вне application code. Поэтому constraint, generated column, явная stored procedure или application transaction часто понятнее.

## Область применимости

- Поведение и синтаксис относятся к PostgreSQL 18.4, выпущенному 2026-05-14 и проверенному 2026-07-18.
- Рассматриваются DML triggers на tables/views. Event triggers для DDL вне scope.
- Пример статически сверен с документацией PostgreSQL 18.4. Локального PostgreSQL runtime в workspace нет, поэтому пример не исполнялся; заметка сохраняет статус `черновик`.

## Ментальная модель

Обычный write path виден в application:

```text
application statement -> constraints -> changed rows -> commit
```

Trigger вставляет дополнительный database-owned control flow:

```text
statement
  -> BEFORE STATEMENT
  -> [BEFORE ROW -> row operation] × N
  -> [AFTER ROW] × N
  -> AFTER STATEMENT
  -> commit / rollback
```

Порядок упрощён, но подчёркивает главное: trigger не background job. Пока он работает, исходный statement не завершён, а его locks и transaction остаются активны. `AFTER` означает «после изменения строки/statement», а не «после commit».

## Как устроено

### BEFORE, AFTER и INSTEAD OF

`BEFORE ROW` вызывается перед операцией над конкретной строкой. Для `INSERT`/`UPDATE` функция может изменить `NEW` и вернуть её; возврат `NULL` пропускает операцию для этой строки. Такой trigger подходит для локальной нормализации или проверки, но built-in constraint/default/generated column предпочтительнее, если выражает тот же инвариант декларативно.

`AFTER ROW` видит окончательное состояние строки и эффекты других triggers. Его return value игнорируется. Row-level `AFTER` events ставятся в очередь и выполняются в конце statement; constraint trigger может быть deferred до конца transaction. `AFTER` удобен для записи audit/outbox, но всё ещё увеличивает latency и способен отменить statement исключением.

`INSTEAD OF` доступен для views и только на row level. Trigger сам выполняет нужные changes в underlying tables и возвращает row, если операция считается выполненной. Он позволяет сделать complex view обновляемым, но переносит semantics write в функцию: permissions, affected-row count и conflicts должны быть определены явно.

### ROW и STATEMENT

`FOR EACH ROW` вызывает функцию для каждой затронутой строки. Update миллиона строк означает до миллиона invocations, поэтому latency и lock duration способны вырасти нелинейно относительно ожиданий application.

`FOR EACH STATEMENT` вызывает функцию один раз, даже если statement не изменил ни одной строки. Он дешевле по числу calls, но по умолчанию не видит отдельные rows. Для set-oriented обработки `AFTER` trigger может запросить transition tables через `REFERENCING OLD TABLE` и `NEW TABLE`.

`TRUNCATE` поддерживает только statement-level triggers. `MERGE` не имеет отдельного типа trigger: в зависимости от actions срабатывают triggers `INSERT`, `UPDATE` и `DELETE`; statement-level triggers могут сработать для action, указанного в `MERGE`, даже если ни одна строка по нему фактически не изменилась.

### OLD, NEW и transition tables

В PL/pgSQL row trigger получает records:

- `INSERT`: `NEW` определена, `OLD` равна `NULL`;
- `UPDATE`: доступны и `OLD`, и `NEW`;
- `DELETE`: `OLD` определена, `NEW` равна `NULL`.

Statement trigger не имеет одной `OLD`/`NEW` row. Transition relations содержат before/after images всего statement и доступны только при поддерживаемой комбинации `AFTER` trigger и plain table. В PostgreSQL 18 transition tables нельзя запросить для foreign table, constraint trigger или column-specific `UPDATE OF ...`; ограничения надо сверять с `CREATE TRIGGER`.

Set-oriented `INSERT ... SELECT FROM new_rows` обычно эффективнее N row-level inserts. Но transition table удерживает набор changed rows до выполнения trigger, поэтому большой batch всё равно требует capacity и наблюдения.

### Транзакционность и видимость

Trigger function выполняет SQL в той же transaction и под тем же snapshot/locking contract, с нюансами isolation level и command visibility. Если trigger выбрасывает необработанное исключение, вызывающий statement не завершается успешно; без savepoint/обработки transaction становится aborted. Если внешняя transaction откатывается, изменения audit/outbox table, сделанные trigger, тоже откатываются. Это следует из [[30 Данные/Транзакции и ACID|общей transaction boundary]], а не из специальной «надёжности trigger».

Именно поэтому trigger может атомарно добавить row в transactional outbox рядом с business row. Publisher после commit доставит event асинхронно по схеме [[40 Распределённые системы/Transactional outbox и Change Data Capture|transactional outbox и CDC]]. Делать сетевой вызов непосредственно из trigger нельзя считать атомарным: remote side effect может выполниться, после чего SQL transaction откатится, или trigger зависнет на сети, удерживая locks.

### Порядок и рекурсия

Если для одного relation/event/timing определено несколько triggers, PostgreSQL запускает их в алфавитном порядке имён. Опора на имена как на сложную orchestration схему хрупка: связь скрыта и плохо меняется.

SQL внутри trigger может снова активировать triggers, включая тот же самый. PostgreSQL не ограничивает число cascade levels; остановить бесконечную recursion обязан автор. Проверка `pg_trigger_depth()` иногда выглядит быстрым guard, но маскирует coupling и меняет поведение при другом call graph. Надёжнее не писать обратно в тот же transition без необходимости, использовать `WHEN`, проверять фактическое изменение через `IS DISTINCT FROM` и проектировать идемпотентный invariant.

## Пример или трассировка

Нужно одним set-oriented действием записать audit изменений balance. Пример использует transition tables:

```sql
CREATE TABLE account (
    account_id bigint PRIMARY KEY,
    balance_cents bigint NOT NULL CHECK (balance_cents >= 0)
);

CREATE TABLE account_balance_audit (
    account_id bigint NOT NULL,
    old_balance_cents bigint NOT NULL,
    new_balance_cents bigint NOT NULL,
    changed_at timestamptz NOT NULL
);

CREATE FUNCTION audit_balance_changes()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
    INSERT INTO account_balance_audit (
        account_id,
        old_balance_cents,
        new_balance_cents,
        changed_at
    )
    SELECT n.account_id,
           o.balance_cents,
           n.balance_cents,
           transaction_timestamp()
    FROM old_rows AS o
    JOIN new_rows AS n USING (account_id)
    WHERE o.balance_cents IS DISTINCT FROM n.balance_cents;

    RETURN NULL;
END;
$$;

CREATE TRIGGER account_balance_audit_trg
AFTER UPDATE ON account
REFERENCING OLD TABLE AS old_rows NEW TABLE AS new_rows
FOR EACH STATEMENT
EXECUTE FUNCTION audit_balance_changes();
```

Трассировка для `UPDATE account SET balance_cents = balance_cents + 100 WHERE account_id IN (1, 2)`:

1. PostgreSQL изменяет подходящие rows и собирает их before/after images.
2. Trigger function вызывается один раз, а не два.
3. `INSERT ... SELECT` добавляет audit rows только там, где значение действительно поменялось.
4. Если последующая операция в той же transaction вызывает rollback, откатываются и balances, и audit rows.

`CHECK (balance_cents >= 0)` оставлен constraint-ом, а не trigger-ом: database умеет выразить invariant декларативно. Trigger отвечает только за derived audit write.

## Trade-offs

- **Constraint/generated column/default:** лучше видны planner-у и schema reader, обычно проще для локального invariant. Trigger нужен, когда declarative mechanism не выражает set-oriented side effect.
- **Application transaction:** control flow заметен в коде и проще тестируется end-to-end, но каждый writer обязан соблюдать protocol. Trigger охватывает все SQL clients, включая ad-hoc writes и migrations.
- **Stored procedure:** caller явно выбирает enriched write path; прямой DML всё ещё может его обойти. Trigger нельзя забыть вызвать, зато труднее заметить.
- **Transactional outbox trigger:** атомарно захватывает изменения от legacy writers. Application-managed outbox лучше связывает event schema с use case и избегает generic row diffs.
- **CDC:** не добавляет work в business statement и видит committed log changes. Цена — lag, schema mapping и отсутствие произвольной business context, если её не записали в source.
- **ROW trigger:** естественно работает с одной строкой. **STATEMENT + transition tables:** уменьшает function calls и поддерживает set operations, но требует иной логики и больше памяти на batch.

## Типичные ошибки

- **Неверное предположение:** `AFTER` выполняется после commit. **Симптом:** trigger ждёт network service и держит transaction/locks. **Причина:** `AFTER` всё ещё часть statement/transaction. **Исправление:** записывать outbox локально, внешнюю доставку делать после commit.
- **Неверное предположение:** row trigger вызывается один раз на SQL statement. **Симптом:** bulk update резко замедляется и создаёт миллионы audit rows. **Причина:** `FOR EACH ROW` масштабируется по числу affected rows. **Исправление:** transition table/statement trigger или отдельный batch path.
- **Неверное предположение:** порядок нескольких triggers несущественен. **Симптом:** один trigger видит неожиданно изменённый `NEW`. **Причина:** `BEFORE` triggers передают результат дальше в name order. **Исправление:** объединить зависимую логику либо явно задокументировать и протестировать ordering.
- **Неверное предположение:** recursion остановится сама. **Симптом:** stack exhaustion, repeated writes или lock amplification. **Причина:** trigger пишет в relation, которое снова активирует тот же path. **Исправление:** изменить data flow, добавить строгий guard/invariant и тестировать cascade.
- **Неверное предположение:** trigger виден application tracing автоматически. **Симптом:** p99 write latency растёт, а в service trace нет отдельного span. **Причина:** work выполняется внутри database statement. **Исправление:** `pg_stat_statements`, lock/WAL metrics, structured audit и migration review каждого trigger.
- **Неверное предположение:** отключение triggers безопасно для bulk load. **Симптом:** после загрузки нарушены audit/derived invariants. **Причина:** trigger был частью correctness contract. **Исправление:** задать эквивалентный bulk protocol, validate/reconcile перед публикацией и ограничить privileges.

## Когда применять

Trigger оправдан, когда правило должно действовать для всех database writers и требует локального transactional side effect: поддержать audit row, materialized auxiliary relation или outbox из legacy table. Сначала проверяют, нельзя ли выразить invariant constraint-ом, foreign key, generated column или явной command procedure.

До production фиксируют timing/granularity, affected-row budget, recursion graph, lock/WAL cost, failure semantics, migration/disable protocol и observability. Trigger, который нельзя обнаружить из schema и связать с owner/runbook, превращает локальную удобную функцию в системный hidden side effect.
