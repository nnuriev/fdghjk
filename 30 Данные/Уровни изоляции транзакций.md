# Уровни изоляции транзакций

## TL;DR

Уровень изоляции задаёт множество допустимых concurrent histories, а не общий рейтинг «безопасности». В PostgreSQL 18.4 Read Committed даёт новый snapshot каждому statement, Repeatable Read даёт transaction-level snapshot isolation, Serializable добавляет SSI-проверку read/write dependencies и aborts histories без serial order.

Названия нельзя переносить между СУБД без проверки реализации. PostgreSQL принимает `READ UNCOMMITTED`, но выполняет его как Read Committed; его Repeatable Read предотвращает phantom reads, хотя SQL standard этого не требует, и всё же допускает serialization anomalies вроде [[30 Данные/Аномалии изоляции и write skew|write skew]].

## Область применимости

- Стандартные названия и phenomena сопоставлены с реализацией PostgreSQL 18.4, проверенной 2026-07-18.
- Serializable означает equivalence некоторому serial execution для успешно committed transactions. Это не strict serializability: порядок serial execution не обязан совпадать с real-time order, а sequences и внешние effects имеют отдельную семантику.
- PostgreSQL Repeatable Read реализует Snapshot Isolation. PostgreSQL Serializable реализует Serializable Snapshot Isolation (SSI), а не strict two-phase locking.
- Вне scope: distributed isolation, linearizability, replica reads и vendor-specific implementations других СУБД.
- Двухсессионный пример не исполнялся локально; ожидаемые snapshots и ошибка взяты из документации PostgreSQL 18.

## Ментальная модель

Transaction читает не «самую свежую базу», а допустимый snapshot. Уровень отвечает на два вопроса:

1. Когда snapshot фиксируется и может ли измениться между statements?
2. Какие conflicts заставят ждать, перечитать row version или abort transaction?

Snapshot сам по себе даёт согласованное чтение прошлого, но не доказывает serializability. Две transactions могут прочитать один snapshot, изменить разные строки и вместе создать состояние, которого не дал бы ни один последовательный порядок.

## Как устроено

### Минимум SQL и фактическое поведение PostgreSQL

| Запрошенный уровень | Snapshot в PostgreSQL 18.4 | Dirty read | Non-repeatable read | Phantom read | Serialization anomaly |
| --- | --- | --- | --- | --- | --- |
| Read Uncommitted | Как Read Committed | Нет | Возможен | Возможен | Возможна |
| Read Committed | Новый на каждый statement | Нет | Возможен | Возможен | Возможна |
| Repeatable Read | Один с первого non-control statement | Нет | Нет | Нет, гарантия сильнее минимума standard | Возможна |
| Serializable | Snapshot плюс SSI dependency checks | Нет | Нет | Нет | Нет у committed transactions; конфликтующая transaction получает `40001` |

### Read Committed

Это default. Обычный `SELECT` видит commits до начала statement и собственные предыдущие writes. Следующий `SELECT` в той же transaction получает новый snapshot.

`UPDATE`, `DELETE` и locking `SELECT` сначала находят rows по snapshot statement. Если row уже изменила concurrent transaction, команда ждёт её исхода. После чужого commit PostgreSQL берёт новую row version и повторно проверяет `WHERE`; из-за этого modifying statement способен увидеть effects на целевой строке, которые обычный `SELECT` snapshot не видел. Для простого update известного key это полезно, для сложного predicate требует осторожности.

### Repeatable Read

Snapshot фиксируется в начале первого statement, который читает или меняет данные. Последующие reads видят собственные writes и тот же committed прошлый мир. Если transaction пытается изменить row, которую concurrent transaction изменила после snapshot, PostgreSQL aborts её с `could not serialize access due to concurrent update`; приложение повторяет transaction целиком.

Стабильный snapshot не гарантирует бизнес-инварианты по нескольким строкам. Snapshot Isolation предотвращает concurrent write/write commit одной logical row, но допускает цикл read/write dependencies на разных rows.

### Serializable

PostgreSQL выполняет MVCC как в Repeatable Read и отслеживает predicate reads через nonblocking `SIReadLock`. Если read/write dependencies образуют dangerous structure, одна transaction получает serialization failure. Блокировок сверх Repeatable Read обычно не добавляется, но tracking и retries стоят ресурсов.

Любой SQLSTATE `40001` требует retry всей transaction с новым snapshot. Нельзя повторять только последний statement: решения ранее в transaction были вычислены из уже недействительного snapshot. Для долгих read-only reports доступен `SERIALIZABLE READ ONLY DEFERRABLE`: начало может подождать safe snapshot, после чего чтение не завершится serialization failure.

### Выбор по инварианту

Read Committed подходит, когда каждый statement самодостаточен, conflicts сведены к одной строке/unique constraint, а изменение мира между statements допустимо. Repeatable Read нужен для согласованного multi-query snapshot, например отчёта, но writer должен переносить aborts. Serializable упрощает правила вида «прочитать predicate, затем изменить данные так, чтобы он оставался истинным».

Explicit `SELECT FOR UPDATE` решает локальный конфликт, если все contenders блокируют один и тот же materialized row. Он не заменяет predicate protection там, где «строки ещё нет» или правило охватывает неизвестный набор rows.

## Пример или трассировка

Подготовка:

```sql
CREATE TABLE counter (
    counter_id integer PRIMARY KEY,
    value integer NOT NULL
);
INSERT INTO counter VALUES (1, 10);
```

Read Committed history:

```text
Session A                                      Session B
BEGIN ISOLATION LEVEL READ COMMITTED;
SELECT value FROM counter WHERE counter_id=1;  -- 10
                                               BEGIN;
                                               UPDATE counter SET value=11 WHERE counter_id=1;
                                               COMMIT;
SELECT value FROM counter WHERE counter_id=1;  -- 11
COMMIT;
```

Два statements Session A получили разные snapshots, поэтому наблюдается non-repeatable read.

После сброса `value = 10` тот же сценарий на Repeatable Read даёт другой итог:

```text
Session A                                      Session B
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT value FROM counter WHERE counter_id=1;  -- 10
                                               BEGIN;
                                               UPDATE counter SET value=11 WHERE counter_id=1;
                                               COMMIT;
SELECT value FROM counter WHERE counter_id=1;  -- 10
UPDATE counter SET value=12 WHERE counter_id=1;
-- ERROR: could not serialize access due to concurrent update
ROLLBACK;
```

Повторный `SELECT` стабилен. Попытка писать поверх версии, появившейся после snapshot, завершает transaction ошибкой; корректный клиент запускает всю business operation заново и уже читает `11`.

## Эволюция и версии

| Версия или период | Было | Стало | Практический эффект | Источник |
| --- | --- | --- | --- | --- |
| До PostgreSQL 9.1 | `SERIALIZABLE` имел поведение нынешнего Snapshot Isolation | PostgreSQL 9.1 ввёл SSI для Serializable, а legacy behavior стало называться Repeatable Read | Код после upgrade должен retry `40001`; выбор Repeatable Read сохраняет старую SI-семантику, но не serializability | [Transaction Isolation](https://www.postgresql.org/docs/18/transaction-iso.html) |

## Trade-offs

- Read Committed снижает abort rate и удержание snapshot. Application должна переносить changing reads и строить atomic predicates в одном statement, constraint или lock protocol.
- Repeatable Read даёт стабильный snapshot для нескольких queries. Он удерживает старые versions дольше и допускает write skew; updating transactions получают serialization failures при concurrent row changes.
- Serializable позволяет рассуждать о committed result как о некотором serial order. Цена: SSI memory, false-positive aborts, retries и рост conflicts при долгих transactions/Seq Scans с coarse predicate locks.
- Explicit locks дают раннее ожидание и предсказуемый conflict point. Они уменьшают concurrency, создают deadlocks и не защищают нематериализованный predicate автоматически.
- `READ ONLY DEFERRABLE` переносит ожидание в начало и удобен для долгого consistent report. Он применим только к serializable read-only transaction и повышает start latency.

## Типичные ошибки

- Неверное предположение: одинаковое имя уровня означает одинаковые guarantees во всех СУБД. Симптом: миграция меняет допустимые anomalies. Причина: standard задаёт minima, vendors используют locking или MVCC по-разному. Исправление: описывать snapshot/conflict behavior конкретной версии; одного label недостаточно.
- Неверное предположение: Repeatable Read в PostgreSQL допускает phantom по таблице standard. Симптом: проектируют лишние range locks, но пропускают write skew. Причина: PostgreSQL SI предотвращает phantoms, а serialization anomaly остаётся. Исправление: тестировать history, которая нарушает invariant, а не запоминать одну таблицу phenomena.
- Неверное предположение: retry последнего statement исправляет `40001`. Симптом: transaction повторяет write, рассчитанный по старому snapshot. Причина: недействительна вся цепочка решений. Исправление: rollback и retry transaction function целиком с bounded backoff.
- Неверное предположение: Serializable убирает все ошибки приложения. Симптом: внешний side effect дублируется или sequence идёт с gaps. Причина: guarantee охватывает committed database transactions, не сеть, clock и nontransactional objects. Исправление: отдельно проектировать idempotency и external consistency.
- Неверное предположение: сменить default на Serializable достаточно. Симптом: клиенты отдают `40001` пользователю, throughput падает. Причина: не реализованы retries, transactions слишком длинные, pool перегружен. Исправление: общий retry layer, короткие boundaries и load test conflict rate.

## Когда применять

Выбирайте isolation по минимальной counterexample history. Если invariant нарушается двумя transactions на Read Committed, попробуйте выразить его одним atomic statement или constraint. Если это multi-row predicate, используйте Serializable либо materialize conflict в строке и согласованно блокируйте её.

Задокументируйте уровень рядом с transaction code, включая retry policy. Тест должен запускать две sessions в контролируемом порядке и проверять final state; последовательный unit test не доказывает concurrent guarantee.
