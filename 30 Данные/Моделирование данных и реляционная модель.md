# Моделирование данных и реляционная модель

## TL;DR

Реляционная модель хранит факты в отношениях, связывает их через значения и отделяет логическую структуру от физического размещения. Хорошая схема отвечает на три вопроса: какие факты существуют, чем они однозначно идентифицируются и какие состояния запрещены.

Проектировать нужно от инвариантов и операций, а не от будущих таблиц или JSON-ответов API. Таблицы, [[30 Данные/Первичные, внешние и составные ключи|ключи]] и ограничения переводят предметную модель в проверяемый контракт. Индексы и размещение появляются позже: они меняют цену доступа, но не смысл фактов.

## Область применимости

- Теоретическая основа: реляционная модель Эдгара Кодда, 1970 год. Отношение в математической модели представляет множество кортежей без порядка.
- Практическая реализация: PostgreSQL 18.4, выпущенный 2026-05-14; документация и поведение проверены 2026-07-18.
- SQL не совпадает с чистой реляционной алгеброй: запрос без `DISTINCT` сохраняет дубликаты, `NULL` вводит трёхзначную логику, а порядок строк определён только при `ORDER BY`.
- Вне scope: физические B-tree/LSM-структуры, распределённое размещение и dimensional modeling для OLAP.
- SQL-пример ниже статически сверен с PostgreSQL 18, но не исполнялся: локального сервера PostgreSQL нет.

## Ментальная модель

Схема похожа на язык фактов. Строка `order_line(order_id, line_no, product_id, quantity, unit_price)` утверждает: в заказе есть позиция с таким номером, товаром, количеством и зафиксированной ценой. Имя таблицы задаёт вид факта, столбцы задают его части, ключ отличает один факт от другого, ограничения отсекают бессмысленные утверждения.

В этой модели связь не хранится указателем на объект в памяти. Она выражается совпадающими значениями. Поэтому приложение может читать одни и те же данные разными путями, а СУБД может менять файлы, страницы и [[30 Данные/Индексы и цена чтения и записи|индексы]], не меняя логический результат запроса. Это и есть практический смысл независимости данных.

## Как устроено

### Три уровня решения

**Концептуальная модель** фиксирует сущности предметной области, их жизненный цикл, cardinality и обязательность связей. Здесь выясняют, может ли заказ жить без клиента, меняется ли цена уже созданной позиции, допускается ли несколько адресов доставки.

**Логическая модель** превращает ответы в отношения, атрибуты, candidate keys и ограничения. Связь one-to-many обычно представляется внешним ключом на стороне many. Для many-to-many создаётся отдельное отношение: оно хранит сам факт связи и его атрибуты.

**Физическая модель** выбирает типы, индексы, partitioning, compression и порядок обслуживания. Смешивать уровни опасно. Если `customer_email` копируют в каждую позицию заказа только ради одного быстрого отчёта, физическая оптимизация уже изменила логический источник истины.

### Инварианты раньше формы

Сначала выписывают правила, которые должны выдержать любую запись и любую гонку:

- у заказа ровно один владелец;
- номер позиции уникален внутри заказа;
- количество положительно;
- цена позиции фиксируется в момент покупки и не меняется вместе с каталогом;
- ссылка на товар не указывает в пустоту.

Часть правил выражается `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `NOT NULL` и `CHECK`. Межстрочные или межтабличные правила иногда требуют другой формы данных, `EXCLUDE`, триггера, блокировки либо уровня `SERIALIZABLE`. Ограничение должно находиться как можно ближе к данным: тогда его соблюдают все writers, а не один удачно написанный handler.

### От фактов к запросам

Модель проверяют рабочими вопросами и изменениями. Можно ли получить заказ с позициями без двусмысленного join? Что произойдёт при удалении клиента? Сколько строк надо согласованно изменить при переименовании товара? Ответы выявляют потерянные сущности, неверную cardinality и дублированные факты. [[30 Данные/Нормализация и денормализация|Нормализация]] формализует часть такой проверки через зависимости между атрибутами.

## Пример или трассировка

Минимальная модель заказа:

```sql
CREATE TABLE customer (
    customer_id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email text NOT NULL UNIQUE
);

CREATE TABLE product (
    product_id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    current_price numeric(12, 2) NOT NULL CHECK (current_price >= 0)
);

CREATE TABLE customer_order (
    order_id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id bigint NOT NULL REFERENCES customer,
    created_at timestamptz NOT NULL DEFAULT clock_timestamp()
);

CREATE TABLE order_line (
    order_id bigint NOT NULL REFERENCES customer_order ON DELETE CASCADE,
    line_no integer NOT NULL CHECK (line_no > 0),
    product_id bigint NOT NULL REFERENCES product,
    quantity integer NOT NULL CHECK (quantity > 0),
    unit_price numeric(12, 2) NOT NULL CHECK (unit_price >= 0),
    PRIMARY KEY (order_id, line_no)
);
```

`product.current_price` описывает текущий каталог, а `order_line.unit_price` описывает исторический факт продажи. Это два разных факта, поэтому совпадение значений не означает дублирование источника истины.

```sql
INSERT INTO customer (email) VALUES ('a@example.test');
INSERT INTO product (name, current_price) VALUES ('book', 20.00);
INSERT INTO customer_order (customer_id) VALUES (1);
INSERT INTO order_line (order_id, line_no, product_id, quantity, unit_price)
VALUES (1, 1, 1, 2, 18.00);

UPDATE product SET current_price = 22.00 WHERE product_id = 1;

SELECT ol.quantity * ol.unit_price AS charged,
       p.current_price AS catalog_price
FROM order_line AS ol
JOIN product AS p USING (product_id)
WHERE ol.order_id = 1;
```

Ожидаемый результат: `charged = 36.00`, `catalog_price = 22.00`. История заказа не переписалась вслед за каталогом. Попытка вставить вторую позицию `(order_id = 1, line_no = 1)` завершится нарушением primary key; отрицательное количество нарушит `CHECK`.

## Trade-offs

- Строгая схема ловит ошибки до чтения и делает данные самодокументируемыми. Цена: изменение модели требует миграции, а не произвольного нового поля в payload.
- Natural key выражает бизнес-идентичность, но бывает длинным и изменяемым. Surrogate key стабилен для ссылок, однако не заменяет `UNIQUE` на настоящем бизнес-ключе.
- Отдельное отношение для связи даёт явные cardinality и ограничения. Вложенный JSON удобнее получить одним чтением, но сложнее независимо проверять и обновлять его элементы.
- Исторический snapshot вроде `unit_price` намеренно повторяет значение из момента события. Копирование текущего `product.name` во все строки только ради скорости создаёт второй источник истины и требует протокола синхронизации.
- Универсальная таблица `entity(id, type, attributes jsonb)` ускоряет ранний прототип. Она переносит типы, обязательность и связи из СУБД в код, усложняя joins и контроль инвариантов.

## Типичные ошибки

- Неверное предположение: таблица равна объекту доменной модели. Симптом: широкая строка с десятками nullable-полей для разных состояний. Причина: несколько видов фактов склеены общей ORM-структурой. Исправление: выделить отношения по инвариантам и жизненному циклу, затем настроить отображение в коде.
- Неверное предположение: surrogate `id` делает строку корректной сущностью. Симптом: появляются два клиента с одним email или две одинаковые бизнес-записи. Причина: технический идентификатор не выражает candidate key предметной области. Исправление: сохранить surrogate key для ссылок и добавить `UNIQUE` на бизнес-ключ.
- Неверное предположение: внешний ключ описывает всю связь. Симптом: база принимает заказ с логически недопустимым владельцем или состоянием. Причина: FK проверяет существование referenced key, но не произвольный бизнес-предикат. Исправление: изменить границы ключа, добавить ограничения или выполнять конфликтующие операции в подходящей транзакции.
- Неверное предположение: порядок строк или физический адрес стабилен. Симптом: пагинация повторяет и пропускает записи после смены плана. Причина: отношение не имеет порядка, PostgreSQL без `ORDER BY` его не обещает. Исправление: задать полный детерминированный sort key и cursor по нему.
- Неверное предположение: JSON снимает необходимость моделирования. Симптом: один и тот же атрибут получает разные типы и названия, а ссылки указывают на удалённые данные. Причина: модель осталась неявной. Исправление: описать факты и инварианты даже для document-поля; часто стабильное ядро следует вынести в typed columns.

## Когда применять

Начинайте новую OLTP-схему с перечня команд, вопросов и инвариантов. Для каждого факта определите identity, владельца жизненного цикла, время действия и допустимую cardinality. Затем проверьте lossless joins на основных сценариях и только после этого подбирайте индексы.

Реляционная модель особенно полезна, когда несколько writers совместно меняют связанные данные и ошибки должны отсеиваться в общей точке. Для непрозрачных blobs, поискового индекса или событийного архива она может отвечать только за metadata и ссылки; это осознанная граница, а не отказ от модели.
