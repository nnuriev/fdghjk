# SQL или key-value

## TL;DR

SQL-хранилище выбирают, когда база должна сама поддерживать связи и инварианты между несколькими наборами данных, а запросы будут меняться после запуска системы. Key-value store выигрывает, когда доступ заранее сводится к `get/put/delete` по полному ключу, граница атомарности совпадает с одним ключом, а предсказуемая задержка и горизонтальное распределение важнее joins и ad hoc-запросов.

Это не выбор между «медленным» и «быстрым». Простая модель key-value убирает работу из СУБД, но обычно переносит её в схему ключей, код приложения, фоновые индексы и процедуры reconciliation. SQL делает обратный обмен: платит за более богатый planner, constraints и координацию транзакций, зато хранит часть корректности рядом с данными.

## Область применимости

Сравнение относится к модели доступа, а не ко всем продуктам с ярлыком SQL или NoSQL. Для конкретики используются PostgreSQL 18.4, Redis Open Source 8.8.0 и опубликованный дизайн Amazon Dynamo 2007 года. Современные системы смешивают свойства: SQL-СУБД умеют JSON и partitioning, а key-value продукты добавляют transactions, search и вторичные индексы. Поэтому решение принимают по реально нужным гарантиям конкретной версии.

Локальные PostgreSQL и Redis недоступны, поэтому команды в примере не исполнялись. Их синтаксис и заявленные границы сверены с документацией указанных версий; benchmark-результаты не приводятся.

## Ментальная модель

Реляционная база хранит утверждения о мире и проверяет отношения между ними. Запрос описывает требуемый результат, а optimizer выбирает путь к нему. Key-value store похож на распределённый словарь: приложение уже знает адрес данных и просит значение по ключу.

Отсюда главный вопрос: кто отвечает за связи и новые способы чтения? В SQL это schema, constraints, indexes и query planner. В key-value эту ответственность чаще несут key design и приложение.

## Как устроен выбор

### Форма доступа

У SQL естественная единица чтения — множество строк, заданное предикатом. Можно фильтровать по нескольким полям, соединять таблицы, группировать и позднее добавить новый индекс без изменения идентификаторов данных. Цена — планирование, поддержка статистики и непредсказуемо дорогие запросы, если API допускает произвольную форму.

В чистой key-value модели операция адресует полный key. `GET session:7f3a` предсказуем, потому что lookup не ищет условие по всему набору. Запрос «все активные сессии пользователей из Армении за неделю» не появляется автоматически: нужен отдельный индекс, иной key layout или экспорт в аналитическое хранилище.

### Граница корректности

В PostgreSQL одна транзакция может изменить несколько таблиц, проверить `UNIQUE`, `FOREIGN KEY` и другие ограничения, а затем опубликовать изменения атомарно. Механизм подробнее раскрыт в [[30 Данные/Транзакции и ACID|заметке о транзакциях и ACID]].

Key-value API часто делает сильной операцию над одним ключом. Это полезный контракт, если aggregate целиком помещается в значение. Как только инвариант охватывает `user:42`, `username:nina` и `quota:team:9`, нужны multi-key transaction конкретного продукта, conditional writes, idempotent workflow или последующая сверка. Нельзя переносить гарантию Redis-команды на Dynamo-подобный store и наоборот: одинаковый интерфейс не означает одинаковую consistency model.

### Распределение и нагрузка

Полный key удобно хешировать и направлять одному shard. Поэтому key-value хорошо ложится на [[30 Данные/Партиционирование и шардирование|шардирование]], но распределение не устраняет hot key: миллион запросов к одному key всё равно попадает в один owner или набор replicas. SQL тоже шардируется, только joins и transactions через shard boundary становятся дороже.

### Эволюция запросов

Если продукт ещё ищет свою модель, способность выполнить новый запрос и затем добавить подходящий индекс снижает стоимость неизвестности. Если контракт зрелый и все пути известны, ограниченный key-value API превращается в преимущество: проще оценить cost каждой операции и запретить случайный full scan.

## Сквозной пример

Сервис хранит пользовательскую сессию `session_id=7f3a` с payload и TTL.

В key-value store путь прямой:

```text
SET session:7f3a <payload> EX 1800
GET session:7f3a
```

Наблюдаемый результат: lookup знает полный ключ, одна запись содержит весь aggregate, а expiration входит в контракт хранилища. Join и cross-key invariant не нужны.

Теперь добавим требование: username уникален, пользователь принадлежит team, а создание пользователя и занятие username должны произойти вместе. В реляционной схеме `users(username UNIQUE, team_id REFERENCES teams)` фиксирует оба инварианта; одна транзакция вставляет строку и либо commit-ит все проверки, либо не оставляет пользователя.

В key-value варианте минимум появляются `user:<id>` и `username:<normalized-name> -> <id>`. Два конкурентных creator должны атомарно занять второй ключ и согласовать его с первым. Если продукт не даёт подходящей multi-key transaction на тех же shards, приложение строит conditional write и recovery protocol. Простота `GET` сохранилась, но инвариант никуда не исчез.

## Trade-offs

| Критерий | SQL | Key-value |
| --- | --- | --- |
| Доступ | Предикаты, joins, aggregates, ad hoc SQL | Полный key и небольшой фиксированный набор операций |
| Инварианты | Constraints и multi-row transactions | Обычно single-key atomicity; остальное зависит от продукта или приложения |
| Изменение запросов | Новый запрос часто добавляется без смены ключей | Новый access path требует нового ключа или производного индекса |
| Предсказуемость cost | Planner может выбрать разные планы | Point lookup проще ограничить и оценить |
| Масштабирование | Шардинг усложняет cross-shard joins/transactions | Key естественно маршрутизируется, но hot keys остаются |
| Операционная цена | Schema, migrations, vacuum/statistics/locks | Rebalancing, replication, conflict handling, derived indexes |

Ближайшая практическая альтернатива не всегда требует нового primary store. Сессии можно вынести в key-value cache, сохранив реляционную БД источником истины. И наоборот, PostgreSQL-таблица с primary key может обслужить умеренный key-value workload без второй технологии.

## Типичные ошибки

- **Неверное предположение:** key-value автоматически масштабируется линейно. **Симптом:** один shard насыщен, хотя cluster в среднем свободен. **Причина:** распределение ключей или трафика создало [[30 Данные/Hot partitions и hot keys|hot key/partition]]. **Исправление:** измерить распределение, изменить key design, cache/fan-out policy или границу aggregate.
- **Неверное предположение:** отсутствие schema в API означает отсутствие модели. **Симптом:** разные версии приложения записывают несовместимые payload, а вторичные ключи расходятся. **Причина:** schema и migration protocol спрятаны в коде. **Исправление:** версионировать value, зафиксировать ownership, invariants и процедуру backfill.
- **Неверное предположение:** SQL нужен для любого durable state. **Симптом:** каждая операция делает `SELECT` по primary key, но команда несёт стоимость сложной СУБД и её scaling model. **Причина:** возможности хранилища не используются. **Исправление:** сравнить реальный access contract с простым key-value store, включая его consistency и recovery.
- **Неверное предположение:** транзакция конкретного key-value продукта равна произвольной SQL-транзакции. **Симптом:** операция отклоняется на разных shards либо оставляет derived index позади. **Причина:** различаются scope, isolation и durability. **Исправление:** документировать гарантию на уровне команд, ключей, shards и replicas, затем построить failure trace.

## Когда применять

Начинайте с SQL, если домен содержит меняющиеся связи, уникальность и ссылочную целостность, несколько согласованных записей на одну бизнес-операцию или неизвестные будущие запросы. Это особенно разумно, пока объём и нагрузка помещаются в одну хорошо настроенную СУБД.

Выбирайте key-value как primary store, когда ключ известен до запроса, value образует самостоятельный aggregate, операции не требуют сканирования по содержимому, а команда готова принять точный consistency/durability contract продукта. Типичные случаи: sessions, idempotency records, feature state, carts или precomputed results. Выбор всё равно проверяют workload-тестом: labels «SQL» и «key-value» не заменяют измерения p99, recovery и стоимости эксплуатации.
