# Геопространственные индексы и поиск ближайших объектов

## TL;DR

Геопространственный индекс редко отвечает на бизнес-вопрос целиком. Он быстро строит небольшой набор candidates по bounding boxes или grid cells, после чего система вычисляет точное пространственное отношение или расстояние. Эта двухфазность — главный инвариант: approximate filter может возвращать лишних кандидатов, но не должен терять подходящие объекты.

В PostgreSQL/PostGIS R-tree-подобный поиск реализован через GiST operator classes: индекс хранит bounding boxes, а `ST_DWithin` и KNN-оператор `<->` используют его для сокращения чтения. H3 4.5.0 кодирует поверхность иерархическими шестиугольными cells; соседние cells удобны для распределённого candidate lookup, но grid distance не равен точному геодезическому расстоянию. Для движущихся объектов geometry и index недостаточно: нужны versioned location updates, удаление «призраков» из прежних cells, TTL и защита от hot cells.

## Область применимости

- SQL-механика относится к PostgreSQL 18.4 и PostGIS 3.6.4, актуальным на 2026-07-18.
- Cell-модель и имена C API относятся к H3 4.5.0, tag `v4.5.0` от 2026-05-21. Конкретная binding library и её API вне scope.
- Рассматриваются точки и обычный radius/nearest lookup. Polygon topology, map matching, routing graph и spatial analytics вынесены за границу заметки.
- SQL-пример статически сверен с документацией PostGIS 3.6.4; локального PostgreSQL/PostGIS runtime в workspace нет, поэтому execution plan не измерялся.

## Ментальная модель

Обычный B-tree использует одномерный порядок. Для точки на плоскости нет одного порядка, который одновременно хорошо сохраняет соседство во всех направлениях. Поэтому распространены два семейства индексов:

1. **Bounding hierarchy:** R-tree группирует пересекающиеся прямоугольники; PostGIS использует GiST как extensible access method для R-tree-подобного индекса bounding boxes.
2. **Spatial grid:** пространство заранее делится на адресуемые cells; объект относится к cell выбранного resolution, а поиск читает cell запроса и соседей.

Обе схемы сначала отвечают «где искать», а затем точная функция отвечает «действительно ли объект подходит».

```text
query point + radius
        ↓
index/grid candidate generation
        ↓
exact distance + business filters
        ↓
ranking / top N
```

Это частный случай общей цены индекса из [[30 Данные/Индексы и цена чтения и записи|заметки об индексах]]: чтение ускоряется сохранённой структурой, а writes оплачивают её поддержку.

## Как устроено

### R-tree и GiST в PostGIS

R-tree иерархически группирует minimum bounding rectangles. Internal node описывает область, покрывающую children. Query с bounding box отбрасывает ветви, которые точно не пересекаются с областью поиска. Перекрывающиеся boxes могут привести в несколько ветвей, поэтому cost зависит от распределения и формы объектов.

GiST (Generalized Search Tree) в PostgreSQL — framework access method: operator class задаёт, как объединять keys, проверять совместимость query с subtree и оценивать penalty. PostGIS GiST index для geometry/geography использует компактные bounding representations. Это не означает, что bounding-box match равен точному `ST_Intersects` или расстоянию: index condition отбирает candidates, spatial function выполняет точную проверку.

`ST_DWithin(a, b, radius)` — правильная форма radius predicate: PostGIS включает index-aware bounding-box фильтр и затем проверяет расстояние. Выражение `ST_Distance(a, b) < radius` может вычислять distance для слишком большого числа строк и не обязано дать такой же index path.

Для nearest-neighbor GiST поддерживает distance ordering через `<->`. У `geometry` значение зависит от units spatial reference system; у `geography` `<->` использует расстояние по сфере, тогда как `ST_Distance(geography, geography)` по умолчанию считает более точную spheroidal distance. Поэтому hard radius выражают через `ST_DWithin`, а требование точного final order — через `ST_Distance`; KNN `<->` можно использовать как быстрый shortlist только с явно доказанным overfetch/recall contract.

### H3 и cell indexes

H3 сопоставляет latitude/longitude с 64-bit cell index выбранного resolution. Более высокий resolution даёт меньшие cells. `gridDisk(origin, k)` перечисляет cells на расстоянии не более `k` grid steps. Затем сервис читает buckets этих cells и проверяет точное расстояние до каждого объекта.

Grid lookup особенно удобен вне одной SQL-базы: cell ID становится partition/index key в KV или stream. Иерархия позволяет агрегировать на coarse resolution, но containment между parent и children геометрически approximate; logical hierarchy H3 определена точно. Кроме того, 12 pentagons и связанные distortions создают edge cases для traversal, поэтому нельзя самостоятельно предполагать идеальную бесконечную hex-grid.

Cell boundary создаёт обязательное правило: поиск не может читать только cell пользователя. Ближайший объект может лежать в соседней cell через несколько метров. Радиус переводят в достаточное множество cells, затем exact filter убирает candidates за пределами окружности.

### Выбор precision

Resolution — не «чем выше, тем точнее», потому что итоговая точность даётся exact distance. Он управляет стоимостью candidate generation:

- coarse cells: меньше bucket reads, но больше лишних объектов и дорогих exact checks;
- fine cells: меньше объектов на cell, но radius пересекает больше cells, растут fanout и metadata;
- слишком coarse в центре города создаёт hot cell;
- слишком fine при частых движениях увеличивает число переходов между cells и churn.

Выбор делают по radius distribution, density histogram, update QPS, target candidate count и partition limits. Нужен load test на реальных пространственных распределениях: равномерная синтетика скрывает центр города, аэропорт или стадион.

### Движущиеся объекты

Перезапись одной точки водителя в GiST на каждое GPS-событие создаёт MVCC/index churn и vacuum pressure. Часто durable track и serving index разделяют: location events поступают в append-only pipeline, а latest-location projection хранит только последнюю принятую version и membership в текущей cell.

Update несёт `(object_id, seq, observed_at, received_at, coordinates, accuracy)`. Projection принимает только новую `seq`, добавляет объект в новую cell и удаляет из старой. Эти два действия не атомарны в распределённом KV, поэтому query повторно сверяет candidate с authoritative latest version/cell, stale membership получает TTL, а reconciliation удаляет остатки. Так at-least-once delivery не превращает одного водителя в несколько доступных машин.

Hot cell разбивают suffix-sharding по `hash(object_id)`, более высоким resolution либо дополнительному business partition. Query читает все suffixes, поэтому число shards ограничивают и меняют по метрикам. Если candidates слишком много, сервис применяет availability/status фильтр до дорогого ETA и ограничивает bounded top-N, но не обрезает произвольно так, чтобы потерять действительно ближайшего без документированной гарантии.

## Пример или трассировка

Таблица хранит текущие позиции свободных водителей как `geography(Point, 4326)`:

```sql
CREATE INDEX driver_position_gix
ON driver_position
USING GIST (position);

SELECT driver_id,
       ST_Distance(position, :passenger) AS distance_m
FROM driver_position
WHERE status = 'available'
  AND ST_DWithin(position, :passenger, 3000)
ORDER BY ST_Distance(position, :passenger), driver_id
LIMIT 20;
```

Трассировка:

1. GiST отбрасывает bounding regions, которые заведомо дальше radius predicate.
2. `ST_DWithin` проверяет реальное расстояние для candidates; строка в углу bounding box может быть исключена.
3. `ST_Distance` задаёт точный final order среди прошедших radius filter; `driver_id` стабилизирует ties.
4. Application затем проверяет freshness позиции, доступность и version. Для dispatch геометрическая близость — только первый этап; ETA по дорожному графу может изменить ranking.

Для unbounded KNN можно заменить radius plan на `ORDER BY position <-> :passenger`, получив index-assisted shortlist. Для `geography` его spherical order нельзя молча выдавать за spheroidal exact order: если business требует именно последний, shortlist пересортировывают по `ST_Distance`, а размер overfetch подтверждают данными.

Для H3 тот же flow читает `gridDisk` вокруг passenger cell, объединяет buckets, дедуплицирует `driver_id`, отбрасывает stale versions и считает точное расстояние. H3 cell ни на одном шаге не становится доказательством «в радиусе 3 км».

## Trade-offs

- PostGIS сохраняет exact spatial semantics, SQL filters и transaction boundary в одной системе. При очень частых moving-object updates индекс и MVCC становятся дорогими, а горизонтальное partitioning сложнее.
- H3/KV естественно partition-ится и удобно для streaming projections. Цена — явная работа с neighboring cells, duplicates, stale membership и exact geometry вне индекса.
- GiST KNN хорошо отвечает на top-N вокруг точки. Fixed-radius query с `ST_DWithin` лучше выражает hard boundary; часто нужны оба шага.
- Materialized latest-location ускоряет serving. Append-only event log надёжнее для replay/history, но не подходит для каждого online query; обычно хранят оба представления.
- Fine resolution снижает candidates в плотной зоне. Coarse resolution уменьшает fanout в редкой зоне. Adaptive scheme эффективнее, но усложняет routing и migration.

## Типичные ошибки

- **Неверное предположение:** совпадение bounding boxes доказывает spatial predicate. **Симптом:** выдача содержит объекты за радиусом или вне polygon. **Причина:** index filter approximate. **Исправление:** выполнять exact predicate после candidate generation.
- **Неверное предположение:** H3 grid distance равен метрам. **Симптом:** radius меняется с resolution и положением на Земле. **Причина:** hops измеряют topology grid, а не геодезическую длину. **Исправление:** cells использовать для candidates, итог — exact distance в согласованной CRS.
- **Неверное предположение:** соседние по координатам точки всегда в одной cell. **Симптом:** объект через границу cell пропадает. **Причина:** query читает только origin bucket. **Исправление:** покрывать радиус соседними cells с запасом и exact filter.
- **Неверное предположение:** последний пришедший GPS event самый свежий. **Симптом:** позиция откатывается после задержанного retry. **Причина:** arrival order перепутан с device sequence/event time. **Исправление:** monotonic sequence, временные sanity checks и idempotent projection.
- **Неверное предположение:** один cell равен одной partition. **Симптом:** центр города становится [[30 Данные/Hot partitions и hot keys|hot partition]]. **Причина:** spatial density неравномерна. **Исправление:** suffix-sharding/adaptive resolution, capacity limits и метрики candidates per query.

## Когда применять

GiST/PostGIS выбирают, когда нужны exact geometry predicates, SQL joins и умеренный update rate. H3-подобный cell index подходит для распределённой candidate generation, агрегации и moving objects. Гибрид часто практичнее: H3/KV обслуживает быстрый online shortlist, а PostGIS или geometry library выполняет exact validation и аналитические запросы.

Для истории координат latest projection не заменяет durable ingestion. Поток, replay и backpressure проектируются как [[50 Проектирование систем/Проектирование event ingestion pipeline|event ingestion pipeline]], а track архивируется отдельно от online neighbor index.
