# Пакеты, модули и направление зависимостей

## TL;DR

Package — единица компиляции, namespace и сокрытия; import graph обязан быть acyclic. Module задаёт versioned набор packages и dependency requirements через `go.mod`; workspace временно объединяет несколько local modules для разработки, не заменяя их release contracts. Устойчивое направление зависимостей идёт от orchestration к небольшим domain packages и от implementation к интерфейсам потребителя, а не через общий «utils» или import cycle.

## Область применимости

- Версия Go: language 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS/GOARCH: влияют на выбранные build files и cgo, но не на базовый запрет cycles.
- Область: package initialization, visibility/imports, modules, versions, toolchains и workspaces.

## Ментальная модель

Три уровня не нужно смешивать:

1. **Package**: каталог исходников с одним package name для выбранной build configuration; API — exported identifiers.
2. **Module**: subtree с `go.mod`, module path и versioned dependency graph.
3. **Workspace**: локальный `go.work`, который говорит toolchain использовать несколько module directories совместно.

Import — compile-time dependency. Если A импортирует B, A знает B и инициализируется после B. Cycle означает, что ни один package нельзя построить первым, поэтому toolchain отвергает граф.

## Как устроено

- Identifier считается exported, если начинается с Unicode uppercase letter; unexported доступен только внутри package.
- Package-level initialization идёт после imports: сначала dependency-ordered variables, затем `init` functions в порядке, в котором compiler получает файлы. Build systems по соглашению подают файлы лексически, но код не должен зависеть от порядка файлов.
- Program начинает `main` только после полной последовательной инициализации всех packages.
- `internal` ограничивает imports деревом, укоренённым выше каталога `internal`; это filesystem-enforced architectural boundary.
- `go.mod` фиксирует module path, `go` language version и minimum requirements. Minimal version selection выбирает для каждого module path максимальную из требуемых минимальных версий.
- С Go 1.21 `go` directive — строгая минимальная версия. `toolchain` предлагает более новую toolchain; автоматический выбор регулируется `GOTOOLCHAIN`.
- `replace` меняет источник module только в main module/workspace и само по себе не добавляет requirement. Публикуемый module не должен полагаться на чужой local replace.
- `go.work` полезен для атомарной разработки нескольких modules; consumers опубликованных versions автоматически не получают его настройки.

Чтобы разорвать cycle, обычно выделяют меньший package с данными/контрактом, инвертируют зависимость через [[60 Go/Интерфейсы и неявная реализация|interface у потребителя]] или переносят orchestration вверх. Слияние packages уместно, если boundary не отражает реального независимого инварианта.

## Код

```text
example.com/app/
├── go.mod
├── main.go
└── internal/greet/greet.go
```

```text
// go.mod
module example.com/app

go 1.26
```

```go
// internal/greet/greet.go
package greet

func Message(name string) string { return "hello, " + name }
```

```go
// main.go
package main

import (
	"fmt"
	"example.com/app/internal/greet"
)

func main() { fmt.Println(greet.Message("Go")) }
```

## Ожидаемый результат

Команда `go run .` из module root печатает:

```text
hello, Go
```

Package вне дерева `example.com/app` не сможет импортировать `example.com/app/internal/greet`. Если `greet` импортирует `main` или package, который уже зависит от `greet`, build завершится ошибкой import cycle.

## Trade-offs

Крупный package уменьшает imports и исключает cycles внутри себя, но размывает API и владение. Мелкие packages дают boundaries, если каждый владеет связным инвариантом; package-per-type создаёт ceremony и искусственные dependencies.

Один module упрощает atomic changes и versioning. Несколько modules нужны при независимых release cycles/consumers, но требуют compatibility discipline и усложняют cross-module changes. `go.work` облегчает разработку, но не устраняет необходимость выпустить согласованные module versions.

Interface inversion разрывает compile-time dependency, когда потребителю действительно нужна способность. [[60 Go/Embedding и composition|Embedding implementation]] через границу packages, напротив, может неявно расширить публичный method set и усилить coupling. Если есть только одна implementation, а cycle вызван произвольной раскладкой файлов, перенос типа или слияние packages может быть проще и честнее.

## Типичные ошибки

**Неверное предположение:** cycle можно исправить общим package `utils`. **Симптом:** dependency hub знает детали всех слоёв. **Причина:** boundary перенесён, а направление не исправлено. **Исправление:** выделить минимальный стабильный контракт/данные или поднять orchestration.

**Неверное предположение:** `go.work` описывает production dependency graph. **Симптом:** published module не собирается отдельно. **Причина:** consumer не использует ваш workspace. **Исправление:** проверять каждый module вне workspace и выпускать реальные versions.

**Неверное предположение:** local `replace` переносим. **Симптом:** CI/consumer не находит путь. **Причина:** replace действует только в main module/workspace и local path не опубликован. **Исправление:** требовать released version; replace оставлять локальной временной настройкой.

**Неверное предположение:** `init` — удобный dependency injection. **Симптом:** скрытый порядок, глобальное состояние, сложные tests. **Причина:** initialization не имеет явных parameters/errors. **Исправление:** explicit constructors и wiring в `main`; `init` оставлять для неизбежной регистрации без failure policy.

## Когда применять

- Package должен владеть одной связной ответственностью и скрывать её representation; предметные границы сверяйте с общей [[20 Бэкенд/Карта — Бэкенд|картой бэкенда]], а не только со структурой каталогов.
- Module boundary вводите из-за независимого version/release contract, не ради структуры каталогов.
- Workspace используйте как local development view и проверяйте modules независимо.
- Разрывайте cycles через ownership и dependency direction, а не через global registries.
