# Интерфейсы и неявная реализация

## TL;DR

Interface описывает требуемое поведение через method set; тип реализует его неявно, без декларации `implements`. Interface value хранит dynamic type и dynamic value. Это позволяет определять маленький контракт у потребителя и адаптировать существующие типы, но слишком широкий interface скрывает зависимости, а interface, возвращаемый «на всякий случай», ограничивает будущую эволюцию реализации.

## Область применимости

- Версия Go: 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS/GOARCH: не влияют.
- Область: basic interfaces как runtime values и generalized interfaces как constraints.

## Ментальная модель

Basic interface — множество значений, чей dynamic type предоставляет все требуемые методы согласно [[60 Go/Pointer и value receivers, method sets|method set конкретного типа]]. Когда concrete value присваивают переменной interface-типа, создаётся пара:

```text
(dynamic type, dynamic value)
```

Статический interface type определяет, какие операции разрешены вызывающему. Dynamic type определяет, какая реализация метода будет вызвана. Наличие обоих компонентов принципиально для [[60 Go/Nil, type assertions и type switches в интерфейсах|nil, type assertions и type switches]].

Неявная реализация меняет направление дизайна: package-потребитель может объявить минимальный interface рядом с местом использования. Producer не обязан заранее знать всех потребителей.

## Как устроено

- Method set interface объединяет явно объявленные и embedded methods; одинаковые имена должны иметь identical signatures. Type set такого interface, напротив, равен пересечению type sets всех его elements: значение должно удовлетворять каждому требованию одновременно.
- `any` — alias для `interface{}`: basic interface без методов, принимающий значение любого типа.
- Concrete value при присваивании interface копируется по обычной семантике. Если это pointer, копируется pointer; если большой struct — сам struct может быть скопирован.
- Вызов interface method выполняет dynamic dispatch к методу dynamic type.
- Type assertion восстанавливает concrete type или более узкий interface, но это runtime-проверка и часто признак потерянного статического контракта.
- Interface с type terms (`~int`, union `A | B`, `comparable`) может использоваться только как constraint, а не как тип обычной переменной; операции над такими множествами типов раскрыты в [[60 Go/Дженерики, constraints и type sets|дженериках, constraints и type sets]].

Interface value разрешено сравнивать, но операция может panic, если его dynamic value имеет несравнимый тип; это отделяет статическую допустимость от гарантий [[60 Go/Равенство и comparability типов|равенства и comparability]].

Interface не обязан жить в producer package. Возвращать concrete type обычно оставляет вызывающему больше возможностей; принимать минимальный interface позволяет тестовую замену и decoupling. Исключение — устойчивый API, который намеренно скрывает множество реализаций.

## Код

```go
package main

import (
	"fmt"
	"strings"
)

type Reader interface {
	Read([]byte) (int, error)
}

func first(r Reader) byte {
	b := make([]byte, 1)
	n, _ := r.Read(b)
	if n == 0 {
		return 0
	}
	return b[0]
}

func main() {
	r := strings.NewReader("Go")
	fmt.Printf("%c\n", first(r))
}
```

## Ожидаемый результат

```text
G
```

`*strings.Reader` реализует локальный `Reader` без импорта или упоминания этого interface в package `strings`.

## Trade-offs

Interface уменьшает coupling, если фиксирует маленькую стабильную способность. Concrete type проще читать, поддерживает полный API и не требует преждевременной абстракции. Абстракция выигрывает, когда реально существуют несколько реализаций или потребителю нужна узкая тестируемая граница.

Большой interface проще передать одним параметром, но изменение любого метода ломает все реализации и mocks. Несколько маленьких interfaces улучшают локальность, однако чрезмерное дробление создаёт ceremony. Граница должна совпадать с одним связным требованием потребителя.

Interface dispatch сам по себе редко становится главным bottleneck, но может ограничить inlining/devirtualization и скрыть allocations. Такие решения проверяются профилем, а не общей догмой.

## Типичные ошибки

**Неверное предположение:** producer должен объявить interface для каждого exported type. **Симптом:** огромный interface повторяет все методы реализации. **Причина:** абстракция создана до потребности. **Исправление:** возвращать concrete type и объявлять минимальный interface у потребителя.

**Неверное предположение:** interface value содержит только данные без типа. **Симптом:** неожиданный non-nil interface с nil pointer или panic сравнения. **Причина:** важна пара dynamic type/value. **Исправление:** рассуждать о двух компонентах и избегать typed nil при возврате error/interface.

**Неверное предположение:** type assertions — нормальный способ ветвить бизнес-логику по реализациям. **Симптом:** добавление реализации требует менять switch у потребителя. **Причина:** interface не выражает нужное поведение. **Исправление:** добавить маленький capability-interface или перенести polymorphism в метод.

## Когда применять

- Принимайте interface там, где функция использует узкое поведение и нужна подмена.
- Возвращайте concrete type, если сокрытие реализации не входит в контракт.
- Проверяйте реализацию compile-time присваиванием `var _ I = (*T)(nil)`.
- Не используйте `any`, если операции над значением известны статически.
