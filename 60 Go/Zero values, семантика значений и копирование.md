# Zero values, семантика значений и копирование

## TL;DR

Каждая переменная Go всегда содержит значение своего типа: до явной инициализации это **zero value**. Присваивание и передача аргумента копируют значение целиком, но копия не обязательно независима: slice, map, channel и pointer ссылаются на отдельное состояние; closure может разделять captured variables, а aliasing interface определяется его dynamic value. Поэтому правильный вопрос — не «копирует ли Go?», а «что именно входит в копируемое значение и чем оно владеет?».

## Область применимости

- Версия Go: 1.26; актуальная стабильная toolchain при проверке — Go 1.26.5.
- GOOS и GOARCH: не влияют на описанную семантику.
- Область: инициализация, присваивание, параметры и результаты функций.

## Ментальная модель

Переменная — отдельная ячейка со значением статического типа. Объявление без initializer записывает в неё zero value:

- числа — `0`, boolean — `false`, string — `""`;
- pointer, function, slice, map, channel и interface — `nil`;
- array и struct — рекурсивная композиция zero values элементов и полей.

Операция `b = a` создаёт **поверхностную копию (shallow copy)** значения `a`. Для `int`, array из чисел или struct только с такими полями копия автономна. Для struct со slice копируется descriptor, а backing array остаётся общим; эта граница подробнее разобрана в [[60 Go/Массивы и слайсы|модели массивов и слайсов]]. Для struct с pointer копируется адрес, а pointee остаётся общим, поэтому правила владения продолжаются в [[60 Go/Структуры, указатели и методы|структурах, указателях и методах]]. Язык не выполняет скрытый deep copy графа объектов.

## Как устроено

Value semantics действует единообразно:

1. Аргумент функции и receiver передаются присваиванием в новые переменные. Для методов последствия выбора receiver `T` или `*T` и доступность методов формализует [[60 Go/Pointer и value receivers, method sets|method set]].
2. Возвращаемое значение также присваивается вызывающей стороне.
3. Array входит в значение всеми элементами, поэтому копирование array копирует элементы.
4. Slice — небольшое значение, описывающее сегмент backing array; копируется описание, а не все элементы.
5. [[60 Go/Map|Map]] и channel — значения, ссылающиеся на runtime-объекты; их копии наблюдают то же состояние.
6. Interface копирует пару «динамический тип, динамическое значение». Если динамическое значение само содержит ссылку, общий объект сохраняется.

Zero value задуман как полезное начальное состояние, но не обещает одинаковую готовность всех типов. `bytes.Buffer`, `sync.Mutex` и `sync.WaitGroup` можно использовать без constructor. Из nil slice разрешены `len`, `range` и `append`. Nil map можно читать и обходить, но запись в неё вызывает panic. Nil function нельзя вызвать.

## Код

```go
package main

import "fmt"

type State struct {
	Fixed [2]int
	Items []int
}

func main() {
	a := State{Fixed: [2]int{1, 2}, Items: []int{10, 20}}
	b := a

	b.Fixed[0] = 9
	b.Items[0] = 99

	fmt.Println(a.Fixed, b.Fixed)
	fmt.Println(a.Items, b.Items)

	var s []int
	s = append(s, 7)
	fmt.Println(s)
}
```

## Ожидаемый результат

```text
[1 2] [9 2]
[99 20] [99 20]
[7]
```

Array `Fixed` скопирован независимо. Два slice descriptor указывают на общий backing array, поэтому изменение элемента видно через обе переменные. Nil slice остаётся допустимым аргументом `append`.

## Trade-offs

Value удобно передавать, когда оно мало, логически неделимо и копия должна быть независимой. Pointer уменьшает явное копирование большого значения и позволяет менять один объект, но добавляет aliasing, nil-состояние и требования к синхронизации.

Размер сам по себе не единственный критерий. Копирование struct со slice затрагивает мало байтов самого slice, но копия продолжает делить данные с оригиналом. Struct с `sync.Mutex` нельзя безопасно копировать после первого использования, даже если компилятор разрешает присваивание. Контракт владения важнее синтаксиса.

## Типичные ошибки

**Неверное предположение:** копия struct — глубокая копия. **Симптом:** изменение slice или map в копии меняет исходный объект. **Причина:** скопирован descriptor или ссылка на тот же runtime-объект. **Исправление:** явно клонировать изменяемые вложенные данные, например `slices.Clone`, `maps.Clone` или `copy`, и документировать владение.

**Неверное предположение:** любой zero value сразу поддерживает все операции. **Симптом:** `assignment to entry in nil map` или вызов nil function. **Причина:** nil — корректное состояние типа, но конкретная операция требует созданного объекта. **Исправление:** проверять контракт типа и вызывать `make`/constructor там, где это необходимо.

**Неверное предположение:** pointer receiver автоматически устраняет все копии. **Симптом:** сложный API с aliasing без измеримого выигрыша. **Причина:** оптимизация выбрана без профиля и без учёта escape analysis. **Исправление:** выбирать pointer по семантике изменения и владения, затем измерять производительность.

## Когда применять

- Проектируйте тип так, чтобы zero value был полезен, если это не разрушает инварианты.
- При code review раскладывайте составное значение на поля и отмечайте, какие из них несут aliasing.
- Для снимка состояния клонируйте только те части, которые должны изменяться независимо.
- Не копируйте значения с mutex, pool и другими типами, документация которых запрещает копирование после использования.
