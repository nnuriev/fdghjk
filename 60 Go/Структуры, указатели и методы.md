# Структуры, указатели и методы

## TL;DR

Struct — value, составленное из полей; присваивание копирует все поля по их собственной семантике. Pointer `*T` хранит адрес значения `T` и позволяет нескольким участникам наблюдать один объект. Методы объявляются у локально определённого типа, а receiver — обычный параметр со специальным синтаксисом. Компилятор автоматически вставляет `&` или `*` в некоторых selector/call выражениях, но эта синтаксическая помощь не меняет method set и владение.

## Область применимости

- Версия Go: 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS/GOARCH: layout, alignment и размер struct могут зависеть от архитектуры; языковая семантика полей и методов одинакова.
- Область: struct types, pointers, addressability, selectors и method declarations.

## Ментальная модель

Struct группирует значения и сам остаётся значением. Pointer не превращает Go в «reference semantics»: сам pointer тоже копируется как значение, просто копии содержат один адрес.

Метод

```go
func (c *Counter) Add(delta int) { ... }
```

можно мысленно читать как функцию с явным первым параметром `c *Counter`. Отличие в selector syntax, [[60 Go/Pointer и value receivers, method sets|правилах receiver и method sets]] и возможности удовлетворять interfaces.

Addressability отвечает на вопрос, существует ли у выражения устойчивая storage location, адрес которой разрешено взять. Переменная addressable; результат map index — нет; временный результат функции обычно не addressable.

## Как устроено

- Поля выбираются через `x.Field`. Для pointer to struct разрешено сокращение `p.Field` вместо `(*p).Field`.
- `&T{...}` создаёт значение и возвращает pointer на него. Где фактически окажется объект — на stack или heap — решает [[60 Go/Стеки и escape analysis|escape analysis]], а не оператор `new` или литерал сам по себе.
- `new(T)` возвращает `*T` на [[60 Go/Zero values, семантика значений и копирование|zero value]]. Начиная с Go 1.26 built-in `new` также принимает expression, вычисляет его и возвращает pointer на новое значение с этим результатом.
- Pointer arithmetic в языке отсутствует. `unsafe` образует отдельный контракт, зависящий от runtime и платформы.
- Метод можно объявить только в том же package, где определён receiver base type. Нельзя «дописать» метод к типу из чужого package.
- Вызов pointer-receiver метода через addressable `T` допускает сокращение `x.M()` до `(&x).M()`. Для non-addressable значения такого преобразования нет.

Поле с именем `_` участвует в layout, но к нему нельзя обратиться. Unexported поля доступны только внутри package. Равенство struct разрешено лишь тогда, когда все поля comparable; padding не сравнивается как произвольные bytes — сравнение задаётся значениями полей.

## Код

```go
package main

import "fmt"

type Counter struct {
	Name  string
	value int
}

func (c *Counter) Add(delta int) {
	c.value += delta
}

func (c Counter) Value() int {
	return c.value
}

func main() {
	c := Counter{Name: "jobs"}
	c.Add(3) // сокращение для (&c).Add(3)

	p := &c
	p.Add(2)
	fmt.Println(c.Name, c.Value(), p == &c)
}
```

## Ожидаемый результат

```text
jobs 5 true
```

Оба pointer-значения адресуют один `Counter`; value-receiver `Value` получает копию, но только читает её.

## Trade-offs

Struct value делает независимость и snapshot-семантику явными. Pointer нужен для изменения одного объекта, identity, optional состояния или когда копирование недопустимо. Цена pointer — aliasing, nil, потенциальный escape и необходимость синхронизации общего состояния.

Exported поля дают простой data container, но фиксируют representation в публичном API. Constructor и методы позволяют удерживать инварианты, но добавляют ceremony и могут сделать zero value бесполезным. [[60 Go/Embedding и composition|Embedding]] также способен продвинуть поля и методы во внешний API, поэтому его выбирают только как намеренную форму композиции. В Go обычно сначала ищут полезный zero value, а constructor вводят при обязательной валидации или скрытых зависимостях.

Layout-оптимизация перестановкой полей может уменьшить padding, но ухудшить смысловую группировку. Делайте это только при доказанной значимости размера множества объектов и измеряйте на целевой GOARCH.

## Типичные ошибки

**Неверное предположение:** `new(T)` обязательно выделяет heap. **Симптом:** преждевременные микрооптимизации. **Причина:** размещение определяет escape analysis. **Исправление:** описывать семантику владения отдельно, а allocations проверять compiler diagnostics и benchmark.

**Неверное предположение:** автоматическое взятие адреса работает в любом выражении. **Симптом:** pointer-receiver метод нельзя вызвать у элемента `map[K]T` или временного значения `T`. **Причина:** такое значение non-addressable. **Исправление:** сохранить value в переменную и при необходимости записать обратно либо хранить `*T` с ясным ownership; у `map[K]*T` метод из method set `*T` вызывается через сам pointer.

**Неверное предположение:** pointer делает concurrent mutation безопасной. **Симптом:** data race. **Причина:** общий адрес не создаёт happens-before. **Исправление:** установить синхронизацию или передавать ownership между goroutines.

## Когда применять

- Используйте struct для связного инварианта, а не только для сокращения числа параметров.
- Выбирайте pointer, когда API меняет тот же логический объект или nil имеет смысл.
- Не раскрывайте layout как часть API без необходимости.
- Не используйте `unsafe.Sizeof` одного запуска как переносимую гарантию для другого GOARCH.
