# Mutex, RWMutex и примитивы координации sync

## TL;DR

`sync.Mutex` защищает не отдельную переменную, а инвариант общего состояния: каждое чтение и изменение этого состояния выполняется под тем же lock. `RWMutex` допускает параллельных readers, но полезен только при измеренном выигрыше. `WaitGroup`, `Once` и `Cond` решают другие задачи — ожидание набора работ, однократную инициализацию и ожидание изменения условия — и не заменяют взаимное исключение.

## Область применимости

- Версия Go: 1.26; стабильная toolchain Go 1.26.5, проверено 2026-07-18.
- GOOS и GOARCH: гарантии API не зависят от платформы; производительность locks зависит от workload и scheduler.
- Пакеты или компоненты runtime: `sync.Mutex`, `sync.RWMutex`, `sync.WaitGroup`, `sync.Once`, `sync.Cond`.

## Ментальная модель

Lock — граница владения общим состоянием на время critical section. Успешный `Lock` даёт вызывающей goroutine эксклюзивное право работать с инвариантом; `Unlock` публикует результат следующему владельцу.

Один инвариант должен иметь один очевидный synchronization policy. Если поле иногда читается под lock, а иногда «потому что это всего лишь int» без lock, о поведении программы уже нельзя рассуждать как о последовательном.

## Как устроено

Для `Mutex` и `RWMutex` zero value сразу готово к использованию. Возврат из `Unlock` синхронизируется с последующим успешным `Lock` того же mutex, поэтому записи critical section видимы новому владельцу согласно [[60 Go/Модель памяти Go и happens-before|отношению happens-before]]. Mutex не привязан к goroutine и не является reentrant: повторный `Lock` тем же кодом заблокируется.

`RWMutex` разрешает несколько `RLock` либо один `Lock`. Когда writer уже ждёт `Lock`, новые `RLock` блокируются, чтобы writer смог продвинуться. Поэтому recursive read locking запрещён: внутренний `RLock` может ждать writer, который сам ждёт внешний `RUnlock`. Нельзя атомарно повысить read lock до write lock или понизить write lock; попытка строить upgrade-протокол обычно приводит к deadlock либо окну изменения состояния.

### Публичный контракт WaitGroup

В Go 1.26 основной безопасный способ запустить учитываемую задачу — `WaitGroup.Go`: он добавляет задачу, запускает goroutine и удаляет задачу после завершения функции без panic. `Wait` устанавливает completion barrier. Функция, переданная `Go`, не должна panic. Если счётчик равен нулю, вызов `Go` должен произойти до `Wait`; при ненулевом счётчике задача может сама вызвать `Go`. Для повторного использования новые `Go` вызывают только после возврата всех предыдущих `Wait`.

`Add`/`Done` остаются нужны для задач, запускаемых иным способом. Положительный `Add`, переводящий zero counter в ненулевое состояние, должен произойти до `Wait`; отрицательный `Add` и положительный `Add` при уже ненулевом counter допустимы во время работы. `Done` эквивалентен `Add(-1)`, отрицательный counter вызывает panic. Вызов `Done`, который разблокировал `Wait`, synchronizes before возврата этого `Wait`, поэтому после barrier видны результаты завершённых задач. Zero value готов к работе, но `WaitGroup` нельзя копировать после первого использования.

Это и есть переносимый API-контракт. Порядок `Add`/`Wait` нельзя заменять надеждой на scheduler: сначала задача должна быть зарегистрирована, затем она может стартовать, и только потом coordinator ждёт completion.

### Реализация WaitGroup в Go 1.26.5

Дальнейшее описание относится к исходнику standard library на tag `go1.26.5`. Это **implementation detail**, а не гарантия Go compatibility: packing полей, runtime semaphore и проверки misuse могут измениться без изменения публичного API.

Помимо marker `noCopy`, `WaitGroup` содержит `atomic.Uint64 state` и `uint32 sema`. `noCopy` помогает `go vet` замечать копирование после первого использования, но не ставит runtime-запрет. В старших `32` bits `state` хранится task counter, который операции читают как `int32`. В младшем слове старший bit зарезервирован для внутреннего `synctest`, а оставшиеся `31` bits содержат число waiters. Прикладной код не видит и не должен интерпретировать `synctest` flag — он обслуживает тестовую инфраструктуру runtime.

`Add(delta)` одним atomic add меняет старшие `32` bits. Если новый counter отрицателен, реализация паникует. Если после последнего `Done` counter стал нулём и waiters уже есть, goroutine сбрасывает `state` и вызывает `runtime_Semrelease` по одному разу на каждого waiter.

`Wait` атомарно читает `state`. При нулевом counter он возвращается сразу. Иначе CAS увеличивает младший waiter count, после чего goroutine засыпает через `runtime_SemacquireWaitGroup(&wg.sema, ...)`. Последний `Done` будит её через тот же semaphore. Упрощённый trace для двух задач выглядит так:

```text
Add(2):      counter=2, waiters=0
Wait():      counter=2, waiters=1 -> park on sema
Done():      counter=1, waiters=1
Done():      counter=0 -> state=0 -> Semrelease -> Wait returns
```

Исходник также пытается обнаружить `Add` одновременно с `Wait` при переходе из zero и reuse до возврата предыдущего `Wait`; для пойманных interleavings он вызывает panic. Эти проверки помогают диагностике, но не превращают запрещённый порядок в корректный protocol и не обязаны обнаруживать каждое misuse.

В `go1.26.5` метод `Go` внутри делает `Add(1)`, запускает goroutine и вызывает `Done`, когда deferred wrapper не обнаружил panic; `runtime.Goexit` тоже проходит по этой ветке. Если `f` паникует, wrapper повторно поднимает panic и намеренно не вызывает `Done`, чтобы `Wait` не успел разблокировать main goroutine раньше fatal panic. Переносимый вывод из этого внутреннего поведения один: соблюдать публичное предусловие «`f` must not panic», а не строить recovery protocol вокруг текущей реализации.

`Once.Do` публикует завершённую инициализацию всем вернувшимся callers. `Cond.Wait` атомарно отпускает lock и снова захватывает его перед возвратом; condition всё равно проверяют в цикле.

## Код

```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var (
		mu sync.Mutex
		wg sync.WaitGroup
		n  int
	)

	for i := 0; i < 4; i++ {
		wg.Go(func() {
			for j := 0; j < 1000; j++ {
				mu.Lock()
				n++
				mu.Unlock()
			}
		})
	}

	wg.Wait()
	fmt.Println(n)
}
```

## Ожидаемый результат

```text
4000
```

`Wait` возвращается только после четырёх задач, а `Mutex` не позволяет потерять read-modify-write increments. Обычный запуск выполнен в официальном Go Playground на Go 1.26.5 и дал `4000`; режим `-race` в Playground недоступен, проверено 2026-07-15. Контракт и описанные implementation details `WaitGroup` повторно сверены с документацией и исходником tag `go1.26.5` 2026-07-18.

## Trade-offs

- `Mutex` обычно проще и дешевле протокола owner goroutine для небольшого общего состояния; критерий выбора подробнее разобран в [[60 Go/Каналы или mutex|сравнении каналов и mutex]].
- `RWMutex` выигрывает лишь когда reads достаточно долгие и конкурентные, а writes редки. Для коротких sections bookkeeping и contention могут сделать обычный `Mutex` быстрее — измеряйте.
- `Cond` полезен для многократного изменения произвольного predicate. Одноразовый broadcast проще выразить закрытием channel.
- Чем шире critical section, тем проще сохранить инвариант, но тем выше contention. Сужайте section только если инвариант остаётся атомарным.

## Типичные ошибки

- Предположение: «lock относится к одному полю» → читатель наблюдает несовместимую пару полей → инвариант обновлён частично → защищайте одним lock всё связанное состояние.
- Предположение: «read безопасен без lock» → [[60 Go/Data races, deadlocks и livelocks|data race]] → read конфликтует с write → синхронизируйте все accesses.
- Предположение: «`RWMutex` всегда быстрее» → throughput падает → workload не окупает coordination overhead → сравните benchmark с `Mutex`.
- Предположение: «`RLock` можно рекурсивно брать для чтения» → goroutine зависает при ожидающем writer → новые readers блокируются до writer → не стройте recursive locking и передавайте ownership явно.
- Предположение: «`WaitGroup` можно копировать» → ожидание относится к другой копии state → coordination ломается → передавайте pointer и не копируйте после использования.
- Предположение: «`Add(1)` можно сделать внутри только что запущенной goroutine» → `Wait` иногда видит zero и возвращается до регистрации задачи → нарушен порядок положительного `Add` при zero counter → используйте `WaitGroup.Go` либо вызывайте `Add` до `go` statement.
- Предположение: «panic из `WaitGroup` — полноценная защита от misuse» → код проходит часть запусков и ломается на другом interleaving или версии → internal checks ловят лишь некоторые запрещённые состояния → доказывайте порядок по публичному контракту.
- Предположение: «`Cond.Wait` означает, что condition истинно» → код продолжает работу преждевременно → condition могла измениться до reacquire → проверяйте predicate в `for`.

## Когда применять

Используйте `Mutex`, когда несколько goroutines кратко читают и меняют одну структуру данных. Выбирайте `RWMutex` только после профиля, а [[60 Go/Пакет sync-atomic|atomic operations]] — когда состояние сводится к одной независимо интерпретируемой synchronization point. Применяйте `WaitGroup.Go`, когда нужны completion и память результатов, но error propagation и cancellation организованы отдельно.
