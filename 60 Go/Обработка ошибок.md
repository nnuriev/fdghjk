# Обработка ошибок

## TL;DR

Ошибка в Go — обычное значение, реализующее `error`; вызывающий явно проверяет её и решает policy. Контекст добавляют wrapping-ом `%w`, если underlying identity/type намеренно становится частью API. `errors.Is` проверяет semantic identity по всему error tree, `errors.As`/`errors.AsType` извлекают тип, `errors.Join` объединяет независимые причины. Текст ошибки не служит контрактом.

## Область применимости

- Версия Go: 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS/GOARCH: общая модель одинакова; конкретные системные error values зависят от ОС.
- Пакеты: `builtin`, `errors`, `fmt`; границы публичных package APIs.

## Ментальная модель

Error value несёт два независимых слоя:

1. Человекочитаемый контекст для диагностики.
2. Машиночитаемую семантику, которую API обещает через sentinel, concrete type, predicate или wrapping.

Wrapping строит дерево ошибок. Узел реализует `Unwrap() error` или `Unwrap() []error`; `Is`, `As` и `AsType` обходят сначала сам узел, затем children depth-first. `%w` — решение о совместимости: после него вызывающие вправе зависеть от underlying error.

## Как устроено

- `error` — interface с методом `Error() string`.
- Успех обычно обозначается nil interface. Typed nil error — non-nil и нарушает этот контракт; механизм этой ловушки раскрыт в [[60 Go/Nil, type assertions и type switches в интерфейсах|nil, type assertions и type switches]].
- `fmt.Errorf("read config: %w", err)` добавляет контекст и сохраняет underlying error для inspection.
- `%v` добавляет только текст и намеренно разрывает машинно-читаемую цепочку.
- `errors.Is(err, target)` проверяет equality или пользовательский метод `Is(error) bool` на каждом узле.
- `errors.As(err, &target)` находит assignable type/interface; target должен быть корректным non-nil pointer.
- С Go 1.26 `errors.AsType[E](err)` возвращает `(E, bool)` и типобезопасен, если `E` реализует error.
- `errors.Join` отбрасывает nil operands и возвращает nil для полностью nil набора. Любой non-nil result — wrapper с `Unwrap() []error`, в том числе при единственном non-nil operand; не полагайтесь на identity результата с исходной ошибкой.

Sentinel подходит для небольшого стабильного набора состояний. Typed error несёт structured fields. Predicate скрывает representation. В публичной документации нужно явно назвать, какой из этих контрактов гарантирован.

## Код

```go
package main

import (
	"errors"
	"fmt"
)

var ErrNotFound = errors.New("not found")

type QueryError struct {
	Key string
	Err error
}

func (e *QueryError) Error() string { return "query " + e.Key + ": " + e.Err.Error() }
func (e *QueryError) Unwrap() error { return e.Err }

func lookup(key string) error {
	return &QueryError{Key: key, Err: ErrNotFound}
}

func main() {
	err := fmt.Errorf("load user: %w", lookup("42"))
	fmt.Println(errors.Is(err, ErrNotFound))
	qe, ok := errors.AsType[*QueryError](err)
	fmt.Println(ok, qe.Key)
}
```

## Ожидаемый результат

```text
true
true 42
```

Два уровня wrapping не мешают найти sentinel и typed error. Код использует `errors.AsType`, поэтому требует Go 1.26 или новее.

## Trade-offs

Sentinel прост и дешёв, но расширяет публичную поверхность и не несёт контекстных полей. Typed error структурирован, однако связывает callers с representation. Predicate или capability-interface лучше сохраняет свободу реализации, если вызывающему нужен только один вопрос.

Wrapping underlying dependency удобно, но делает её error semantics частью вашего API. Если storage implementation может смениться, преобразуйте `sql.ErrNoRows` в собственный domain error вместо прямого `%w`, если не готовы поддерживать эту зависимость постоянно.

`errors.Join` сохраняет несколько причин, но порядок и presentation не должны становиться parsing API. Для workflow, где одна ошибка первична, а остальные — cleanup diagnostics, полезно явно различить их роль вместо плоского join.

## Типичные ошибки

**Неверное предположение:** достаточно сравнить `err == sentinel`. **Симптом:** wrapped error не распознаётся. **Причина:** equality не обходит tree. **Исправление:** использовать `errors.Is` для документированного target.

**Неверное предположение:** текст ошибки стабилен для программы. **Симптом:** ветвление ломается после добавления контекста/локализации. **Причина:** `Error()` предназначен для людей. **Исправление:** sentinel, typed error, predicate или `Is` contract.

**Неверное предположение:** `%w` — просто улучшенный `%v`. **Симптом:** callers начинают зависеть от ошибки внутренней БД/файловой системы. **Причина:** wrapping раскрывает underlying error. **Исправление:** wrap только намеренно экспортируемую семантику.

**Неверное предположение:** можно вернуть typed nil как `error`. **Симптом:** success выглядит как `err != nil`. **Причина:** interface содержит dynamic type. **Исправление:** возвращать literal `nil` по успешной ветви.

## Когда применять

- Добавляйте контекст один раз на смысловой границе операции.
- Документируйте проверяемую семантику ошибок exported функций.
- Используйте `errors.Is` для категорий/identity, `AsType` или `As` для structured data.
- Не логируйте и не возвращайте одну и ту же ошибку на каждом слое без добавления решения или контекста.
- Не превращайте ожидаемые ошибки в panic: области ответственности `error` и recovery разделены в [[60 Go/defer, panic и recover|модели defer, panic и recover]].
- Для отменённых операций сохраняйте отдельно error contract и lifecycle: `context.Cause` из [[60 Go/Context, deadlines и распространение отмены|модели context, deadlines и cancellation]] объясняет причину, но не отменяет предметную классификацию ошибки API.
