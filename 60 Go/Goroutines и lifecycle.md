# Goroutines и lifecycle

## TL;DR

Оператор `go` запускает функцию независимо от вызывающей goroutine, но не создаёт ни `join`, ни отмену, ни передачу ошибки. Поэтому у каждой goroutine должен быть владелец, условие завершения и способ дождаться остановки; иначе локальная асинхронность превращается в [[60 Go/Goroutine и channel leaks|утечку goroutine]] или незавершённую работу.

## Область применимости

- Версия Go: 1.26; стабильная toolchain Go 1.26.5, проверено 2026-07-15.
- GOOS и GOARCH: семантика примера не зависит от платформы.
- Пакеты или компоненты runtime: оператор `go`, каналы.

## Ментальная модель

Goroutine — не «фоновая функция», а задача внутри процесса. После запуска runtime планирует её независимо, но запустивший код по-прежнему отвечает за lifecycle:

1. кто запускает;
2. какое событие просит остановиться;
3. как наблюдается завершение;
4. куда возвращается ошибка.

Запуск без ответов на эти вопросы корректен только для задачи, результат и завершение которой действительно не важны до конца процесса.

## Как устроено

Перед созданием новой goroutine вызывающая goroutine вычисляет значение функции и её аргументы. Затем выполнение функции начинается независимо; спецификация не обещает, что она успеет выполниться до следующего действия вызывающего кода.

Возврат из функции завершает только эту goroutine. Возвращаемые значения отбрасываются. Возврат из `main.main` завершает программу целиком: runtime не ждёт остальные goroutines. Значит, `go f()` сам по себе не устанавливает отношения «работа завершена до продолжения».

Канал `done` в примере играет роль `join`. Закрытие канала после печати синхронизируется с приёмом, который наблюдает закрытие, поэтому после `<-done` видны и побочные эффекты, выполненные перед `close(done)`. Это конкретный synchronization edge из [[60 Go/Модель памяти Go и happens-before|модели памяти Go]], а не следствие того, что scheduler «успел» выполнить goroutine.

## Код

```go
package main

import "fmt"

func main() {
	done := make(chan struct{})

	go func() {
		defer close(done)
		fmt.Println("work")
	}()

	<-done
	fmt.Println("joined")
}
```

## Ожидаемый результат

```text
work
joined
```

Вторая строка не может появиться раньше первой: приём блокируется до `close(done)`. Пример выполнен в официальном Go Playground на Go 1.26.5; вывод совпал с ожидаемым, проверено 2026-07-15.

## Trade-offs

- Goroutine скрывает latency независимой операции и позволяет композицию блокирующего кода, но добавляет отдельный lifecycle, стек и точки блокировки.
- Синхронный вызов проще: результат, ошибка и завершение уже принадлежат стеку вызовов. Выбирайте goroutine только при реальной независимости или конкурентном ожидании.
- Канал завершения передаёт одно событие многим наблюдателям. Для группы однородных задач удобнее [[60 Go/Mutex, RWMutex и примитивы координации sync|`sync.WaitGroup`]]; для остановки дерева вызовов — `context.Context`.

## Типичные ошибки

- Предположение: «`go` автоматически дождётся функции» → иногда пропадает вывод или запись → `main` возвращается раньше → введите явный `join`.
- Предположение: «goroutine завершится, когда потеряны ссылки на неё» → число goroutines и удерживаемая память растут → goroutines не собираются GC как недостижимые объекты → обеспечьте достижимый путь к `return`.
- Предположение: «ошибка вернётся вызывающему коду» → сбой теряется → возвращаемое значение асинхронной функции отбрасывается → передавайте результат по каналу либо управляйте группой задач.
- Предположение: «Go 1.22 устранил любой capture/race в loop» → goroutines делят outer variable → новые per-iteration variables относятся только к объявленным самим loop → передавайте argument явно и синхронизируйте всё внешнее shared state.

## Когда применять

Запускайте goroutine на границе явно владеющего ею компонента. До запуска фиксируйте протокол остановки и ожидания. Для request-scoped работы обычно используйте [[60 Go/Context, deadlines и распространение отмены|отменяемый context]] и дождитесь всех дочерних задач перед возвратом владельца.
