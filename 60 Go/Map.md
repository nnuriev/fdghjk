# Map

## TL;DR

`map[K]V` — неупорядоченная таблица с ключами comparable-типа. Копирование map-переменной не копирует записи: несколько значений ссылаются на одну таблицу. Отсутствующий ключ возвращает zero value, а форма `v, ok := m[k]` отделяет отсутствие от записанного zero value. Одновременная запись без синхронизации — data race; наблюдаемый runtime panic не заменяет контракт синхронизации.

## Область применимости

- Версия Go: 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS и GOARCH: семантика одинакова; layout и performance runtime map зависят от реализации.
- Здесь рассматривается языковой контракт. Внутренняя реализация разобрана в [[60 Go/Runtime-устройство map|Runtime-устройство map]].

## Ментальная модель

Map — ссылка на изменяемый runtime-объект: при присваивании не происходит value-copy записей. Операции поиска, вставки и удаления определены языком, но порядок и внутренняя раскладка — нет.

Ключ должен поддерживать `==` и `!=`: допустимы boolean, числа, strings, pointers, channels, interfaces и arrays/structs, если все их компоненты comparable. Slice, map и function ключами быть не могут. Interface как статический тип ключа допустим, но вставка dynamic value несравнимого типа вызовет panic; рекурсивные правила и runtime-исключение разобраны в [[60 Go/Равенство и comparability типов|равенстве и comparability типов]].

## Как устроено

- `make(map[K]V, hint)` создаёт таблицу; `hint` — подсказка capacity, а не гарантия точного размера.
- Чтение nil map безопасно и ведёт себя как чтение пустой; запись в nil map вызывает panic.
- `delete` безопасен и для отсутствующего ключа, и для nil map.
- `clear(m)` удаляет все записи; для nil map это no-op.
- Значение элемента map не addressable: рост и реорганизация таблицы могут перемещать хранение. Чтобы изменить поле struct-value, прочитайте значение, измените копию и присвойте обратно либо храните pointer.
- Порядок `range` не определён и может различаться между проходами. Удалённая до посещения запись не попадёт в результат обхода; добавленная во время обхода может попасть в него или быть пропущена.

Спецификация языка не обещает thread safety. Если хотя бы одна goroutine пишет, все пересекающиеся доступы должны быть упорядочены mutex, channel ownership или другой синхронизацией; достаточность такого порядка доказывают через [[60 Go/Модель памяти Go и happens-before|модель памяти и happens-before]]. Две goroutine могут одновременно читать map, если никто её не изменяет.

## Код

```go
package main

import (
	"fmt"
	"sort"
)

func main() {
	m := map[string]int{"b": 2, "a": 0}

	v1, ok1 := m["a"]
	v2, ok2 := m["missing"]
	fmt.Println(v1, ok1, v2, ok2)

	keys := make([]string, 0, len(m))
	for k := range m {
		keys = append(keys, k)
	}
	sort.Strings(keys)
	fmt.Println(keys)
}
```

## Ожидаемый результат

```text
0 true 0 false
[a b]
```

Первый `0` прочитан из существующей записи, поэтому следом выведено `true`; второй — zero value отсутствующего элемента. Детерминированный вывод получен отдельной сортировкой ключей, а не свойством map.

## Trade-offs

Map подходит для динамического ассоциативного доступа. Slice или array обычно лучше для плотных целочисленных индексов: меньше overhead и предсказуемее locality. Отсортированный slice выигрывает, когда набор редко меняется, важен порядок и допустим бинарный поиск `O(log n)`.

`map[K]V` хранит value inline с точки зрения API и избегает отдельного nil у каждого элемента, но обновление большого struct требует read-modify-write. `map[K]*V` упрощает изменение общего объекта, однако добавляет allocations, aliasing, nil и отдельную синхронизацию полей.

Mutex вокруг обычной map даёт явные составные инварианты. `sync.Map` полезна для документированных профилей нагрузки, но теряет статическую типизацию ключей и значений и не гарантирует универсального выигрыша в скорости.

## Типичные ошибки

**Неверное предположение:** `m[k] == zero` означает отсутствие. **Симптом:** существующая запись с zero value считается отсутствующей. **Причина:** одно-value lookup объединяет два состояния. **Исправление:** использовать comma-ok.

**Неверное предположение:** стабильный порядок в локальном запуске — контракт. **Симптом:** flaky test, нестабильный вручную собранный JSON или SQL. **Причина:** порядок range не определён; отдельный encoder может вводить собственную сортировку, как `encoding/json` v1 в Go 1.26.5. **Исправление:** собрать и отсортировать ключи либо выбрать API/структуру с документированным порядком.

**Неверное предположение:** runtime всегда поймает concurrent access. **Симптом:** редкая порча логики, race report или fatal error. **Причина:** data race уже делает выполнение некорректным; runtime-проверка покрывает не все interleavings. **Исправление:** установить ownership или синхронизацию и запускать [[60 Go/Race detector|race detector]].

**Неверное предположение:** можно изменить `m[k].Field`. **Симптом:** compile-time error. **Причина:** map element не addressable. **Исправление:** read-modify-write либо pointer value с осознанным владением.

## Когда применять

- Используйте comma-ok, если zero value имеет предметный смысл.
- Сортируйте ключи на границе, где нужен воспроизводимый результат.
- Не публикуйте map для чтения, пока строящая её goroutine ещё может писать.
- Для составной операции «проверить затем изменить» защищайте весь инвариант одной синхронизацией.
