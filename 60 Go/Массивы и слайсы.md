# Массивы и слайсы

## TL;DR

Array `[N]T` — значение фиксированной длины; длина входит в тип и копируется вместе с элементами. Slice `[]T` — descriptor окна в backing array с `len` и `cap`. Reslicing и обычное копирование slice сохраняют aliasing; `append` либо пишет в существующий массив, либо выделяет новый. Код не должен зависеть от стратегии роста capacity: язык её не гарантирует.

## Область применимости

- Версия Go: 1.26; стабильная toolchain при проверке — Go 1.26.5.
- GOOS и GOARCH: конкретные размеры и стратегия allocations могут различаться, семантика одинакова.
- Область: built-in операции `len`, `cap`, `append`, `copy`, slicing expressions.

## Ментальная модель

Удобно представлять slice как тройку:

```text
(указатель на первый доступный элемент, len, cap)
```

`len` ограничивает допустимые индексы. `cap` показывает, насколько окно можно расширить вправо без смены backing array. Slice не владеет памятью автоматически и не кодирует, кто ещё видит тот же массив.

Array, напротив, содержит элементы непосредственно. `[3]int` и `[4]int` — разные типы. Pointer to array и slice могут давать доступ к тем же элементам, но несут разные контракты: array фиксирует размер в типе, slice — динамическое окно.

## Как устроено

- `a[low:high]` создаёт новый descriptor: `len = high-low`, capacity продолжается до capacity исходного окна.
- Полное выражение `a[low:high:max]` ограничивает capacity до `max-low`. Это позволяет запретить вызываемому коду перезаписать скрытый хвост через `append`.
- `append(s, x...)` возвращает slice. Результат нужно присвоить: descriptor мог измениться даже без allocation.
- Если новой длины хватает в capacity, `append` использует прежний backing array; иначе выделяет другой и копирует существующие элементы.
- `copy(dst, src)` копирует `min(len(dst), len(src))` элементов и корректен при overlap.
- Nil slice имеет `len == 0` и `cap == 0`; это один из практически полезных случаев [[60 Go/Zero values, семантика значений и копирование|zero value]]. Empty non-nil slice может вести себя так же в цикле, но различаться при сравнении с nil и при сериализации конкретной библиотекой.

Retention — важное следствие aliasing. Маленький slice на фрагмент большого массива удерживает доступным для GC весь backing array, пока runtime не может доказать обратное. Если нужен небольшой самостоятельный результат, его стоит скопировать.

## Код

```go
package main

import "fmt"

func main() {
	base := []int{1, 2, 3, 4}
	a := base[:2]
	b := base[:2:2]

	a = append(a, 9)
	fmt.Println(base, a)

	b = append(b, 8)
	b[0] = 7
	fmt.Println(base, b)
}
```

## Ожидаемый результат

```text
[1 2 9 4] [1 2 9]
[1 2 9 4] [7 2 8]
```

У `a` была capacity до конца `base`, поэтому `append` перезаписал `base[2]`. Полное slicing expression ограничило capacity `b`; его `append` создал другой backing array, и последующее `b[0] = 7` не изменило `base`.

## Trade-offs

Slice — обычный выбор для последовательностей: он дешёво передаётся и поддерживает динамический размер. Array полезен, когда длина — часть инварианта (`[32]byte` для digest), когда нужна comparability или когда значение должно копироваться целиком.

Предварительное выделение `make([]T, 0, n)` уменьшает reallocations, если `n` обоснован. Слишком большая capacity удерживает память и может быть хуже постепенного роста. Выбор подтверждают [[60 Go/Бенчмарки|бенчмарком]] и профилем allocations; если рассматривается повторное использование буферов, нужно отдельно учитывать trade-offs из [[60 Go/Снижение аллокаций и sync.Pool|снижения аллокаций и sync.Pool]], а не полагаться на текущую формулу роста runtime.

Клонирование устраняет aliasing, но стоит `O(n)` времени и памяти. Ограничение capacity через `s[:len(s):len(s)]` дешевле и защищает только от записи через append в хвост; существующие элементы всё ещё общие.

## Типичные ошибки

**Неверное предположение:** `append` меняет slice-переменную у вызывающего. **Симптом:** новая длина потеряна. **Причина:** функция вернула новый descriptor, но результат не присвоили. **Исправление:** всегда использовать `s = append(s, ...)` или возвращать новый slice из функции.

**Неверное предположение:** subslice — независимая коллекция. **Симптом:** изменения приходят из другой части программы или большой buffer не освобождается. **Причина:** общий backing array. **Исправление:** назначить явного владельца либо клонировать нужный диапазон.

**Неверное предположение:** capacity растёт по фиксированному коэффициенту. **Симптом:** расчёт указателей, allocations или latency ломается после обновления toolchain. **Причина:** стратегия роста — implementation detail. **Исправление:** полагаться только на контракт `append`, измерять конкретную версию и платформу.

**Неверное предположение:** `&s[0]` допустим для пустого slice. **Симптом:** panic index out of range. **Причина:** индексирование требует `len > 0`, даже если capacity положительна. **Исправление:** проверять длину или использовать API, принимающее slice.

## Когда применять

- Передавайте slice как окно для чтения; отдельно документируйте, может ли callee менять элементы или удерживать slice после возврата.
- Ограничивайте capacity на границе API, если append не должен затронуть скрытый хвост.
- Клонируйте вход, когда компонент должен владеть стабильным снимком.
- Используйте array, если фиксированная длина делает невозможным целый класс ошибок.
- Для `[]byte` применяйте те же правила aliasing и capacity, а различие между байтами и текстом проводите по [[60 Go/Строки, байты, руны и UTF-8|модели строк, байтов, рун и UTF-8]].
