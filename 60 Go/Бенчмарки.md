# Бенчмарки

## TL;DR

Go benchmark отвечает на узкий вопрос при контролируемых inputs и environment. В Go 1.26 предпочтительный цикл — `for b.Loop() { ... }`: setup до первого вызова и cleanup после последнего не измеряются, а compiler сохраняет arguments/results внутри тела от полного удаления. Один `ns/op` ничего не доказывает: нужны allocations, несколько samples, одинаковые toolchain/CPU/settings и объяснение практического эффекта. Microbenchmark формирует гипотезу; [[60 Go/Профилирование с pprof|production profile]] проверяет, важен ли этот участок для системы.

## Область применимости

- Версия Go: package `testing` и compiler Go 1.26.5.
- GOOS/GOARCH: пример предназначен для linux/amd64; абсолютные timings непереносимы между CPU, OS, power modes и virtualized environments.
- Компоненты: `go test`, `testing.B`, compiler optimization, allocator.
- Вне scope: статистическая методология сложных A/B experiments.

## Ментальная модель

Benchmark состоит из трёх слоёв:

1. **Experiment design**: один вопрос, репрезентативный input, контролируемые side effects.
2. **Measurement harness**: повторяет operation достаточно долго, исключает setup и сообщает metrics.
3. **Comparison**: сопоставляет распределения samples при одинаковом environment.

Измеряется не строка кода, а весь transitive work внутри timed region: [[60 Go/Аллокации, GC и GC pressure|allocations и GC assists]], locks, syscalls и scheduler effects. Если production bottleneck — database latency, ускорение локальной функции на 20% может не изменить end-to-end result.

## Как устроено

`testing.B` калибрует число iterations. Первый `b.Loop()` сбрасывает timer и allocation counters; `false` останавливает timer перед cleanup. Benchmark function с `B.Loop` выполняется один раз на measurement, в отличие от legacy `b.N` calibration, которая повторно вызывает function.

Compiler Go 1.26 распознаёт точную синтаксическую форму `for b.Loop() { ... }` и удерживает function arguments, results и assigned variables внутри braces через эквивалент `runtime.KeepAlive`. Это не распространяется на произвольный wrapper вокруг `b.Loop`.

`-benchmem` или `b.ReportAllocs()` добавляет `B/op` и `allocs/op`; эти значения отражают в том числе решения [[60 Go/Стеки и escape analysis|escape analysis]]. `b.SetBytes` позволяет вывести throughput. `-count` создаёт несколько samples; `-benchtime` меняет длительность/число iterations. Parallel benchmark через `RunParallel` измеряет contention и scheduler вместе с operation; если этот эффект нужно разложить по времени ожидания, нужен [[60 Go/Execution trace|execution trace]], а не только `ns/op`.

## Код

Файл `copy_test.go`:

~~~go
package copybench

import "testing"

func BenchmarkCopy1KiB(b *testing.B) {
	src := make([]byte, 1024)
	dst := make([]byte, 1024)

	b.SetBytes(int64(len(src)))
	b.ReportAllocs()

	for b.Loop() {
		copy(dst, src)
	}
}
~~~

Команда:

~~~text
go test -run="^$" -bench="^BenchmarkCopy1KiB$" -benchmem -count=5 copy_test.go
~~~

## Ожидаемый результат

Будет пять строк вида:

~~~text
BenchmarkCopy1KiB-<P>    <N>    <time> ns/op    <throughput> MB/s    0 B/op    0 allocs/op
~~~

Точный ожидаемый invariant — `0 B/op` и `0 allocs/op`: allocations setup исключены первым `b.Loop`, а `copy` внутри loop не выделяет heap memory. `P`, `N`, time и throughput зависят от машины и не должны фиксироваться как ожидаемые константы. В доступном официальном Go Playground на Go 1.26.5 отдельный `testing.AllocsPerRun` для той же операции подтвердил `0` allocations; benchmark harness и точный test-file package там не запускаются, поэтому саму команду `go test -bench` локально не проверяли, проверено 2026-07-15.

| до 1.24 | Основной шаблон — loop по `b.N`, ручной timer lifecycle и защита от dead-code elimination | — | Setup мог повторяться при calibration; benchmark code был легче написать неверно | [Go 1.24 Release Notes](https://go.dev/doc/go1.24) |
| 1.24 | — | Добавлен `B.Loop` | Setup/cleanup выполняются один раз на measurement, compiler удерживает loop body | [Go 1.24 Release Notes](https://go.dev/doc/go1.24) |

## Trade-offs

Microbenchmark даёт точный локальный сигнал и быстрый feedback, но исключает production concurrency, data distribution и I/O. End-to-end benchmark ближе к системе, однако шумнее и хуже локализует причину. Обычно нужны оба: профиль выбирает hot spot, microbenchmark проверяет изменение, integration load test подтверждает эффект.

Большой `-benchtime` уменьшает часть timer noise, но удлиняет suite и не исправляет thermal throttling или background load. Больше `-count` показывает variance, но samples на одной нестабильной машине не становятся автоматически репрезентативными.

Parallel benchmark полезен для shared structure, если задана реалистичная contention model. Для CPU algorithm он может измерить scheduler и memory bandwidth вместо самого algorithm.

## Типичные ошибки

**Неверное предположение:** setup автоматически не измеряется в любом benchmark. **Симптом:** результат отражает input generation. **Причина:** setup находится внутри timed loop либо legacy timer не сброшен. **Исправление:** располагать setup до точного `for b.Loop()`.

**Неверное предположение:** compiler обязательно выполнит результат, который никто не наблюдает. **Симптом:** невероятно малый `ns/op`. **Причина:** dead-code elimination или constant folding. **Исправление:** использовать прямой `B.Loop`, реалистичные inputs и проверять generated behavior/profile.

**Неверное предположение:** одна цифра быстрее означает regression/improvement. **Симптом:** вывод меняется между запусками. **Причина:** variance CPU frequency, scheduler и GC. **Исправление:** собирать несколько samples в одинаковой среде и сравнивать распределение.

**Неверное предположение:** меньше allocations всегда означает меньшую latency. **Симптом:** вариант с [[60 Go/Снижение аллокаций и sync.Pool|`sync.Pool`]] медленнее. **Причина:** synchronization, reset и cache effects дороже allocation. **Исправление:** измерять одновременно time, allocations и end-to-end effect.

## Когда применять

- После формулировки конкретной performance hypothesis.
- С inputs, близкими к production size и distribution.
- С зафиксированными Go version, GOOS/GOARCH, CPU, `GOMAXPROCS` и важными env settings.
- Вместе с CPU/heap profile, если цель — system-level ускорение.
