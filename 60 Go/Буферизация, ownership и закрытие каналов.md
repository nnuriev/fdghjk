# Буферизация, ownership и закрытие каналов

## TL;DR

Канал — очередь с фиксированной ёмкостью и протоколом завершения. Буфер определяет, сколько отправок может опередить приёмник; `close` сообщает «новых значений больше не будет», но не отменяет уже отправленные значения. Закрывает канал сторона, владеющая правом отправки и знающая, что отправителей больше нет.

## Область применимости

- Версия Go: 1.26; стабильная toolchain Go 1.26.5, проверено 2026-07-15.
- GOOS и GOARCH: семантика не зависит от платформы.
- Пакеты или компоненты runtime: каналы, `close`, `range`.

## Ментальная модель

Unbuffered channel — rendezvous: отправитель и получатель должны встретиться. Buffered channel — ограниченная очередь: отправка завершается без готового получателя, пока есть свободная ячейка.

Закрытие — не значение и не удаление очереди. Это необратимое изменение состояния канала:

- buffered values по-прежнему выдаются в FIFO-порядке;
- после опустошения приём немедленно возвращает zero value и `ok == false`;
- отправка или повторное закрытие паникуют.

## Как устроено

Для `make(chan T, n)` ёмкость равна `n`. Отправка блокируется, когда буфер полон; приём — когда буфер пуст и канал открыт. Для unbuffered channel ёмкость нулевая, поэтому завершение отправки требует соответствующего приёма.

`close(ch)` не нужен для освобождения памяти: канал и буфер собираются GC, когда недостижимы. Закрытие нужно получателям, которым важно отличить временное отсутствие значения от окончательного завершения потока.

Причина правила ownership проста: только отправляющая сторона может знать, что будущих отправок не будет. При нескольких producers нужен координатор, который сначала дожидается всех producers и лишь затем закрывает общий outbound channel.

## Код

```go
package main

import "fmt"

func main() {
	values := make(chan int, 2)
	values <- 10
	values <- 20
	close(values)

	for v := range values {
		fmt.Println(v)
	}

	v, ok := <-values
	fmt.Println(v, ok)
}
```

## Ожидаемый результат

```text
10
20
0 false
```

`range` получает оба buffered values и останавливается после опустошения закрытого канала. Последний приём возвращает zero value `int` и `false`. Пример выполнен в официальном Go Playground на Go 1.26.5; вывод совпал с ожидаемым, проверено 2026-07-15.

## Trade-offs

- Unbuffered channel даёт сильную обратную связь: успешная отправка означает, что получатель принял значение. Цена — непосредственная зависимость latency сторон.
- Буфер сглаживает краткий burst и уменьшает число rendezvous, но скрывает отставание consumer и добавляет память. Размер буфера должен следовать из допустимого числа in-flight элементов и выбранной политики [[60 Go/Backpressure|backpressure]], а не подбираться «пока тест проходит».
- Закрытие упрощает конечные потоки. Для request/response, где получатель заранее знает число ответов, канал можно не закрывать.

## Типичные ошибки

- Предположение: «получатель должен закрыть канал» → panic `send on closed channel` → получатель не знает о будущих sends → закрывайте со стороны owner отправки.
- Предположение: «закрытие отбрасывает очередь» → обработка неожиданно продолжается → buffered values сохраняются → для отмены используйте [[60 Go/select, cancellation и timeout|отдельный сигнал, наблюдаемый через `select`]] и явно решите судьбу очереди.
- Предположение: «zero value означает конец» → корректный ноль теряется → конец различим только через `ok` или `range` → проверяйте двухзначный receive.
- Предположение: «большой буфер устраняет блокировки» → растут память, latency и число заблокированных goroutines → буфер лишь откладывает насыщение, после которого возможен [[60 Go/Goroutine и channel leaks|leak на send]] → проектируйте backpressure.

## Когда применять

Используйте канал для передачи владения значением или события между goroutines. Как и при проектировании [[60 Go/Goroutines и lifecycle|lifecycle goroutine]], документируйте, кто отправляет, кто закрывает, что означает закрытие и допускается ли отказ получателя раньше producer.
