# Проектирование файлового хранилища

## Вопрос

Как раскрыть на System Design интервью тему «Проектирование файлового хранилища»: какие требования, инварианты и trade-offs определяют решение?

## Короткий ориентир

File service разделяет control plane и byte data plane. API хранит metadata, ownership, versions и upload state; clients передают bytes напрямую в object storage по scoped signed URLs. Publish/finalize делает object видимым только после проверки размера и checksum. Download идёт через CDN/object storage, а API выдаёт authorization decision и immutable version URL.

Multipart upload, metadata transaction и malware scan не образуют одну атомарную транзакцию. Поэтому нужен явный lifecycle `initiated → uploading → verifying → available | rejected | expired`, reconciliation и сборка abandoned parts. Immutable versions упрощают cache, rollback и concurrency; mutable filename остаётся pointer на version.

Полный разбор: [[50 Проектирование систем/Проектирование файлового хранилища|Проектирование файлового хранилища]].
