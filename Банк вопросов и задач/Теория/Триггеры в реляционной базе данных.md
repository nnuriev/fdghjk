# Триггеры в реляционной базе данных

## Вопрос

Объясните тему «Триггеры в реляционной базе данных»: какие гарантии даёт механизм и какой ценой для чтения, записи и эксплуатации?

## Короткий ориентир

Trigger связывает изменение relation с автоматически вызываемой database function. В PostgreSQL 18.4 выбор задают три оси: момент `BEFORE`/`AFTER`/`INSTEAD OF`, granularity `ROW`/`STATEMENT` и событие `INSERT`/`UPDATE`/`DELETE`/`TRUNCATE`. Row trigger видит `OLD`/`NEW`; transition tables дают `AFTER` trigger набор строк, изменённых statement целиком.

Trigger выполняется внутри транзакции statement: его database effects коммитятся и откатываются вместе с вызывающей операцией. Это делает trigger сильным инструментом для локального инварианта или transactional audit/outbox, но не превращает внешний HTTP-вызов в часть SQL-транзакции. Главная цена — скрытый control flow: write latency, блокировки, рекурсия и дополнительные записи появляются вне application code. Поэтому constraint, generated column, явная stored procedure или application transaction часто понятнее.
