# Синхронная и асинхронная обработка

## Вопрос

Как работает «Синхронная и асинхронная обработка» и какие ограничения, failure modes и trade-offs нужно учитывать в backend-системе?

## Короткий ориентир

Синхронный API удерживает request-response до терминального бизнес-результата. Асинхронный принимает команду долговечно, возвращает operation handle и завершает работу позже. Различие задаёт клиентский контракт, а не наличие goroutine, event loop или broker внутри реализации.

`202 Accepted` не означает успех операции: RFC 9110 прямо допускает, что принятая работа ещё не началась и позже может быть отклонена. Поэтому async API требует durable acceptance, state machine, terminal result/error, idempotent submission, правила cancellation, retention и способ наблюдать прогресс. Без них он лишь переносит timeout в непрозрачное «когда-нибудь».
