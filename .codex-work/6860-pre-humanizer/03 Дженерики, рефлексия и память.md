# Дженерики, рефлексия и память

## TL;DR

Generics переносят часть проверки универсального алгоритма в compile time; reflection работает с runtime type descriptors и требует защитных проверок. Allocator управляет размещением, GC — достижимостью, а не «освобождением всего ненужного по таймеру». Три задачи намеренно моделируют эти слои в упрощённом виде: serializer по tags, compaction памяти и graph tracing.

## Задание 9. Serializer по struct tags

![[90 Вложения/CurseHunter/6860/Кадры/09-homework-properties-serializer.jpg|720]]

### Контракт курса

```go
type Person struct {
    Name    string `properties:"name"`
    Address string `properties:"address,omitempty"`
    Age     int    `properties:"age"`
    Married bool   `properties:"married"`
}

func Serialize(person Person) string
```

Output идёт в declaration order, одна пара `key=value` на строку. `Address` пропускается только при empty value из-за `omitempty`; остальные zero values сериализуются. Задание со звёздочкой — generic serializer для произвольной структуры. Источник: [`homework/generics_and_reflection`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/generics_and_reflection/homework_test.go).

### Pipeline решения

1. Получить `reflect.Value` и `reflect.Type`.
2. Если принят pointer, разыменовать после проверки `nil`; убедиться, что kind — `Struct`.
3. Для каждого field по index прочитать `StructField.Tag.Get("properties")`.
4. Разобрать tag на name и options; пустой name требует явного policy.
5. При `omitempty` определить zero value через `Value.IsZero`.
6. Проверить export/accessibility до `Interface`.
7. Отформатировать поддерживаемый kind и добавить line.
8. Вернуть error для unsupported type вместо panic.

Тестовая signature не возвращает error, поэтому failure semantics потеряна. Для general serializer лучше `func Serialize[T any](v T) (string, error)`; generic parameter здесь не устраняет reflection, а только делает call site typed.

### Неочевидные решения

- Embedded fields: flatten, отдельная секция или reject?
- Duplicate tag names: last wins, error или обе строки?
- Newline и `=` внутри values: escaping обязателен для настоящего format.
- Pointers/interfaces: сериализовать element, identity или `nil`?
- Field order: declaration order стабилен для struct, но canonical format может требовать sorting.
- `omitempty`: zero semantics для arrays, structs и custom types нужно определить.

## Задание 10. In-place defragmentation

![[90 Вложения/CurseHunter/6860/Кадры/10-homework-defragmentation.jpg|720]]

### Контракт курса

```go
func Defragment(memory []byte, pointers []unsafe.Pointer)
```

В `memory` заняты четыре однобайтовых objects по offsets `0,5,10,15`; `pointers` указывает на них. После операции bytes должны лежать в `0..3`, остальные bytes — zero, а каждый pointer должен быть обновлён на новый address. Нельзя выделять новый memory buffer. Задание со звёздочкой — objects размером 2/4/8 bytes. Источник: [`homework/allocator`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/allocator/homework_test.go).

### Правильная модель

`pointers`, а не byte values, задают occupancy. Byte `0x00` может быть легитимным object value; `0xFF` не является универсальным allocation marker. Курс прямо требует различать free/used по pointers.

Надёжный алгоритм для unit-size objects:

1. Для каждого pointer вычислить offset относительно base `&memory[0]` и проверить bounds/alignment.
2. Отсортировать object descriptors по старому offset, если input pointers не обещаны ordered.
3. Идти write cursor слева направо.
4. Перемещать byte в `memory[write]`, очищать старую позицию только после безопасного чтения.
5. Обновлять соответствующий element `pointers[i]` на `&memory[write]`.
6. Очистить остаток после последнего live object.

Если source и destination ranges overlap, использовать semantics `copy`, похожую на `memmove`, а не наивный forward loop для произвольного object size.

### Invariants

- multiset object values сохранён;
- relative order live objects либо сохранён, либо contract явно разрешает permutation;
- каждый output pointer попадает внутрь `memory` и указывает на свой object;
- objects образуют плотный prefix;
- free suffix zeroed;
- нет временного чтения по pointer после перемещения, если он уже invalidated.

### Почему это не production Go allocator

В обычном Go application нельзя безопасно перемещать arbitrary objects и вручную переписывать все pointers: aliases могут находиться в stacks, heap, registers, interfaces и runtime metadata. Runtime collector знает pointer maps и обновляет/отслеживает graph по своим правилам; application-level `unsafe.Pointer` не даёт полного списка references.

## Задание 11. Трассировка достижимых объектов

![[90 Вложения/CurseHunter/6860/Кадры/11-homework-gc-trace.jpg|720]]

### Контракт курса

```go
func Trace(stacks [][]uintptr) []uintptr
```

Каждый simulated stack содержит machine words. Ненулевое значение трактуется как pointer; по нему рекурсивно читается следующий word. Нужно вернуть pointers в порядке первого обнаружения и не обходить одно значение повторно. Задание со звёздочкой — корректно обрабатывать cycles через visited set или coloring. Источник: [`homework/garbage_collector`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/garbage_collector/homework_test.go).

### Graph model

- vertices — addresses;
- root edges — nonzero words в `stacks`;
- object edge — word, прочитанный по address;
- reachable set — least fixed point от roots.

DFS и BFS оба дают `O(V+E)` time и `O(V)` visited memory. Конкретный output order зависит от worklist и порядка roots; тест ожидает depth-first-like discovery с сохранением order stack words.

### Безопасный псевдокод модели

```text
for each word in stacks, in order:
    visit(word)

visit(addr):
    if addr == 0 or visited(addr): return
    mark visited
    append addr to output
    visit(loadWord(addr))
```

В тесте адреса действительно валидны внутри текущего process, но произвольный `uintptr` разыменовывать опасно. Production-like simulator должен иметь явную heap map `address → object fields` и проверять membership до чтения.

### Коррекция ментальной модели GC

Настоящий Go GC — precise относительно известных pointer-bearing locations: compiler/runtime поддерживают stack maps и type metadata. Он не считает любой nonzero integer pointer, иначе integer, случайно равный address, удерживал бы object и мог бы привести к invalid traversal. `uintptr` сам по себе не поддерживает reachability объекта для GC.

Tri-color abstraction:

- white — ещё не обнаружен;
- gray — обнаружен, но outgoing pointers не просканированы;
- black — просканирован.

Concurrent marking должен не допустить скрытой black→white edge без barrier. Write barrier сохраняет invariant, пока mutator меняет graph.

### Эволюция и версии

Лекция записана на Go `1.23/1.24` и объясняет классический concurrent tracing GC. Начиная с Go `1.26` от февраля 2026 года Green Tea GC включён по умолчанию. Он оптимизирует marking/scanning small objects через locality и CPU scalability; release notes ожидают 10–40% reduction GC overhead в GC-heavy workloads, но конкретный эффект зависит от workload. Это меняет implementation/performance, а не отменяет reachability model.

`GOMEMLIMIT` — soft runtime memory limit, введённый в Go `1.19`. Если live heap и runtime overhead близки к limit, слишком частые cycles могут вызвать thrashing; runtime ограничивает GC CPU примерно 50% на окне, поэтому memory может превысить limit ради progress. Нельзя трактовать `GOMEMLIMIT` как hard RSS cap.

## Trade-offs

| Выбор | Выигрыш | Цена |
| --- | --- | --- |
| generics | compile-time reuse | сложнее constraints, возможен code-size/perf trade-off |
| reflection | runtime universality | panic boundary, slower path, weaker static checks |
| code generation | быстрый typed code | build tooling и generated surface |
| compaction | меньше fragmentation | relocation cost и pointer updates |
| non-moving GC | stable addresses проще | fragmentation/locality limitations |
| moving GC | compaction/locality | нужны точные roots и relocation protocol |

## Failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| generic значит без reflection | runtime panic остаётся | `T any` не раскрывает fields | constraint/codegen/validation |
| `omitempty` = пустая строка | structs/arrays ведут себя неверно | zero semantics шире | `IsZero` + contract |
| nonzero byte = live object | данные теряются | value смешан с metadata | allocation descriptors/pointers |
| любой word похож на pointer | invalid dereference | conservative toy model | pointer maps/type metadata |
| `GOMEMLIMIT` hard | process превышает cap | limit soft ради progress | container headroom + measurement |
