## Задание 1. Переставить байты числа

![[90 Вложения/CurseHunter/6860/Кадры/01-homework-byte-order.jpg|720]]

### Контракт курса

Реализовать `ToLittleEndian(number uint32) uint32`, для которой:

- `0x00000000 → 0x00000000`;
- `0xFFFFFFFF → 0xFFFFFFFF`;
- `0x00FF00FF → 0xFF00FF00`;
- `0x0000FFFF → 0xFFFF0000`;
- `0x01020304 → 0x04030201`.

Задание со звёздочкой — обобщить операцию на `uint16`, `uint32`, `uint64`. Точный тест находится в [`homework/data_types`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/data_types/homework_test.go).

### Ментальная модель

Функция выполняет **byte swap значения**, а не исследует byte order текущей машины. Для `uint32` нужно взять четыре octets и собрать их в обратном порядке:

```text
input:   [b3 b2 b1 b0]
output:  [b0 b1 b2 b3]
```

Endian становится существенным, когда число сериализуют в bytes или читают из bytes. Само целое число в выражении не имеет «порядка байтов» на уровне language semantics.

### Сильное решение

Для `uint32` достаточно shifts и masks либо `bits.ReverseBytes32`. Ручной вариант должен:

1. выделить каждый byte маской `0xFF`;
2. сдвинуть его в зеркальную позицию;
3. объединить части через bitwise OR;
4. работать без `unsafe` и без зависимости от GOARCH.

В generic-варианте нельзя просто написать constraint и вызвать один built-in для всех widths: `bits.ReverseBytes16/32/64` имеют разные signatures. Варианты — type switch, отдельные wrappers либо generic побайтовый алгоритм с аккуратным определением width. Если constraint включает named types с underlying unsigned integer, надо сохранить их type при возврате.

## Задание 2. Кольцевая очередь

![[90 Вложения/CurseHunter/6860/Кадры/02-homework-circular-queue.jpg|720]]

### Контракт курса

Нужна fixed-capacity FIFO queue положительных integers:

```go
type CircularQueue struct {
    values []int
    // добавить состояние
}

func NewCircularQueue(size int) CircularQueue
func (q *CircularQueue) Push(value int) bool
func (q *CircularQueue) Pop() bool
func (q *CircularQueue) Front() int
func (q *CircularQueue) Back() int
func (q *CircularQueue) Empty() bool
func (q *CircularQueue) Full() bool
```

`Push` и `Pop` возвращают `false`, если операция невозможна; `Front/Back` возвращают `-1` для empty queue. После `Push(1,2,3)`, `Pop`, `Push(4)` внутренний buffer по тесту равен `[4,2,3]`. Источник: [`homework/slices_and_arrays`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/slices_and_arrays/homework_test.go).

### Инварианты

Практичный state:

- `head` — index первого элемента;
- `tail` — index следующей позиции для записи;
- `count` — число элементов;
- `capacity == len(values)`.

Тогда:

```text
empty ⇔ count == 0
full  ⇔ count == len(values)
head  = (head + 1) mod capacity после Pop
tail  = (tail + 1) mod capacity после Push
back  = (tail - 1 + capacity) mod capacity
```

Одних `head` и `tail` недостаточно без дополнительного правила: `head == tail` может означать и empty, и full. Альтернативы — `count`, отдельный flag или один всегда пустой slot; последняя уменьшает usable capacity.

### Complexity и failure modes

- все операции — `O(1)` time;
- память — `O(capacity)`;
- modulo допустимо, а при power-of-two capacity может быть заменено mask, если это измеримо полезно;
- `size <= 0` в тестах не определён: production API должен вернуть error либо panic с документированным контрактом;
- sentinel `-1` работает только потому, что вход ограничен positive values; общий API лучше возвращает `(int, bool)`;
- тест привязан к private layout `values`, хотя внешняя FIFO semantics важнее физического порядка.

## Задание 3. Copy-on-write buffer

![[90 Вложения/CurseHunter/6860/Кадры/03-homework-cow-buffer.jpg|720]]

### Контракт курса

`NewCOWBuffer` сохраняет исходный `[]byte` без копирования. `Clone` увеличивает shared reference count. Пока копий несколько, `Update` создаёт отдельный backing array для изменяемой копии; если владелец один, запись идёт in place. `String` должен возвращать string без копирования. `Close` освобождает логическую ссылку.

```go
type COWBuffer struct {
    data []byte
    refs *int
}
```

Тест сравнивает addresses через `unsafe.SliceData` и `unsafe.StringData`; invalid index должен вернуть `false`. Источник: [`homework/strings`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/strings/homework_test.go).

### State machine

1. `New`: `refs=1`, buffer владеет переданным backing array по договорённости.
2. `Clone`: новая handle указывает на те же `data` и `refs`; count увеличивается.
3. `Update` при `refs>1`: count старой группы уменьшается, bytes копируются, новая группа получает `refs=1`, затем выполняется запись.
4. `Update` при `refs==1`: bytes изменяются на месте.
5. `Close`: count уменьшается ровно один раз, handle становится закрытым.

### Главная проблема: lifetime zero-copy string

Unsafe conversion `[]byte → string` без копирования создаёт string, который продолжает смотреть на mutable memory. Если после `String()` единственный owner выполнит in-place `Update`, ранее возвращённая строка изменится, хотя callers ожидают immutable string.

Значит, задача недоопределена без одного из контрактов:

- returned string живёт только до следующей mutation/Close;
- после публикации string buffer больше не мутирует in place;
- `String` всё-таки копирует;
- backing storage immutable, а каждая запись делает copy независимо от refs.

Для production безопаснее копия или явный borrowed-view API с ограниченным lifetime. `unsafe.String` требует отдельно доказать, что bytes не будут изменены и останутся alive.

### Concurrency и cleanup

Поле `*int` делает task не thread-safe: concurrent `Clone/Close/Update` дают data race. Atomic counter решает только count, но не весь transition; нужны lock или другой ownership protocol. Повторный `Close` должен быть idempotent либо явно запрещён. Finalizer из задания со звёздочкой не даёт своевременного освобождения и усложняет resurrection/lifetime; начиная с Go `1.24` для некоторых cleanup-сценариев появился `runtime.AddCleanup`, но explicit `Close` всё равно остаётся основным контрактом.

## Задание 4. Ordered map на BST

![[90 Вложения/CurseHunter/6860/Кадры/04-homework-ordered-map.jpg|720]]

### Контракт курса

Реализовать map с unique integer keys на binary search tree:

```go
type OrderedMap struct { /* ... */ }

func NewOrderedMap() OrderedMap
func (m *OrderedMap) Insert(key, value int)
func (m *OrderedMap) Erase(key int)
func (m *OrderedMap) Contains(key int) bool
func (m *OrderedMap) Size() int
func (m *OrderedMap) ForEach(action func(int, int))
```

`ForEach` обязан выдавать keys по возрастанию. Тест строит дерево с keys `10,5,15,2,4,12,14`, затем удаляет leaf и nodes с children. Источник: [`homework/maps`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/maps/homewrok_test.go).

### Ментальная модель

Для каждого node:

```text
все keys(left) < node.key < все keys(right)
```

In-order traversal `left → node → right` даёт sorted order. `Insert` должен заранее определить semantics duplicate key: курс говорит, что одинаковые keys хранить нельзя, но тест не уточняет, игнорировать новый value или обновлять существующий. Для map-совместимого API естественно обновить value без изменения `Size`.

### Удаление

Три случая:

1. leaf — parent link становится `nil`;
2. один child — parent link заменяется child;
3. два children — node заменяется successor из правого subtree или predecessor из левого, затем удаляется перемещённый node.

Root — такой же link, поэтому удобно писать helper, возвращающий новый subtree root. `Size` уменьшается только если key существовал.

### Complexity и выбор структуры

Обычный BST даёт `O(h)`, где `h` — height. На sorted input `h=n`, поэтому lookup/insert/delete деградируют до `O(n)`. Гарантированные `O(log n)` требуют AVL, red-black tree, B-tree/treap или другой balancing strategy. Для одного ordered traversal поверх редко меняющейся built-in map дешевле собрать keys и отсортировать: `O(n log n)` на traversal, без постоянной tree complexity.

Задание со звёздочкой предлагает generic `OrderedMap[K comparable, V any]`, но `comparable` недостаточно для BST ordering: оно разрешает equality, а не `<`. Нужен ordered constraint либо переданный comparator. Это хороший уточняющий вопрос к условию.

## Общие failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| integer сам «big-endian» | код зависит от хоста | смешаны value и serialization | описать bytes boundary |
| `head==tail` однозначно | queue путает full/empty | не хватает state | count/flag/reserved slot |
| zero-copy string безопасен | строка меняется после возврата | alias на mutable bytes | copy или строгий lifetime |
| BST всегда `O(log n)` | latency `O(n)` | дерево не сбалансировано | balanced tree/comparator |
| `comparable` разрешает `<` | generic code не компилируется | constraint даёт только equality | ordered constraint/comparator |
