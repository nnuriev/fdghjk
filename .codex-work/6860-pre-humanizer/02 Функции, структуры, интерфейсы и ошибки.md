# Функции, структуры, интерфейсы и ошибки

## TL;DR

В этих уроках важен не синтаксис, а дизайн контракта. Higher-order functions задают policy через callbacks; struct layout заставляет считать bits и alignment; interface переносит зависимость на поведение; error становится частью public API. Четыре задания проверяют эти идеи на `Map/Filter/Reduce`, compact layout, DI container и error aggregation.

## Коррекция: хвостовая рекурсия

![[90 Вложения/CurseHunter/6860/Кадры/18-tail-recursion-claim.jpg|720]]

Курс утверждает, что при tail recursion компилятор превращает вызовы в loop с одним stack frame. Для Go это нельзя использовать как гарантию. Спецификация не обещает tail-call optimization, а recursive call следует проектировать как обычный вызов, способный увеличивать stack. Если input depth не ограничен, безопасное решение — явный loop или собственный stack.

Интервью-формулировка: «Tail position делает оптимизацию теоретически возможной, но Go contract её не гарантирует. Поэтому correctness и memory bound не должны от неё зависеть».

## Задание 5. Map, Filter, Reduce

![[90 Вложения/CurseHunter/6860/Кадры/05-homework-map-filter-reduce.jpg|720]]

### Контракт курса

```go
func Map(data []int, action func(int) int) []int
func Filter(data []int, action func(int) bool) []int
func Reduce(data []int, initial int, action func(int, int) int) int
```

Тесты требуют:

- `nil` input для `Map/Filter` возвращает `nil`;
- non-nil empty input возвращает non-nil empty slice;
- `Map` сохраняет order и length;
- `Filter` сохраняет relative order selected elements;
- `Reduce` начинает с `initial` и обходит слева направо.

Источник: [`homework/functions`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/functions/homework_test.go).

### Инварианты решения

`Map` заранее знает output length, поэтому выделяет `make([]int, len(data))`. `Filter` знает только верхнюю границу: разумно создать `make([]int, 0, len(data))`, если ожидается высокая доля совпадений; иначе preallocation может зря удерживать память. `Reduce` не требует allocation.

Вызовы callbacks выполняются ровно один раз на элемент и строго в source order. Это важно, даже если математическое название наводит на мысль о parallelism: callbacks могут иметь side effects, а reducer может быть non-associative.

### Неопределённые границы

Тесты не задают поведение при `action == nil`: естественный результат — panic при вызове, но public API может валидировать input. Также не определено, разрешено ли reuse input storage. Возврат нового slice безопаснее и соответствует ожидаемым тестам, но in-place `Map` может быть отдельным API.

### Уточнения интервьюера

- Почему `Reduce` с subtraction зависит от direction?
- Какие условия нужны для parallel reduce?
- Как обобщить API с generics: `Map[S,T]`, `Filter[T]`, `Reduce[T,A]`?
- Нужно ли сохранять `nil` semantics или достаточно empty collection?

## Задание 6. Упаковать игровой профиль в 64 байта

![[90 Вложения/CurseHunter/6860/Кадры/06-homework-compact-struct.jpg|720]]

### Контракт курса

`GamePerson` должен занимать не более 64 bytes по `unsafe.Sizeof` и хранить:

- inline Latin name длиной 0–42 characters, без pointer на отдельную string data;
- `x`, `y`, `z` в диапазоне `[-2_000_000_000, 2_000_000_000]`;
- gold `0..2_000_000_000`;
- mana и health `0..1000`;
- respect, strength, experience, level `0..10`;
- три booleans;
- person type из трёх значений.

Конструктор использует functional options, getters должны восстановить исходные значения. Источник: [`homework/structs`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/structs/homework_test.go).

### Расчёт budget

Очевидная база:

- name data — `[42]byte`;
- четыре `int32` для coordinates и gold — 16 bytes;
- итого 58 bytes до length и мелких fields.

Остаётся 6 bytes, если layout не добавит padding. Mana и health требуют по 10 bits; четыре значения `0..10` — по 4 bits; три flags — 3 bits; type — 2 bits. Всего `20 + 16 + 3 + 2 = 41 bit`, то есть эти поля помещаются в `[6]byte` с семью spare bits.

Отдельную длину name хранить необязательно, если contract запрещает `\x00`: `[42]byte` zero-filled, а getter ищет первый zero. Тогда layout может быть ровно `16 + 42 + 6 = 64` bytes:

1. сначала четыре `int32`, чтобы не получить padding перед ними;
2. затем `[42]byte` name;
3. затем `[6]byte` bit field.

Использовать `uint64` для bit field проще, но он добавит 8 bytes и alignment, поэтому constraint будет нарушен. Альтернатива — плотнее кодировать ограниченный Latin alphabet, но для тестового контракта это не требуется. Главная часть задачи — аккуратно распределить 41 bit по шести bytes и доказать round trip на min/max значениях.

### Что обязан проговорить кандидат

1. Зафиксировать GOARCH: layout и `int` size платформозависимы; хранимые поля должны иметь fixed-width types.
2. Проверять range в options: truncate без ошибки разрушает invariant.
3. Не полагаться на случайный field order; подтвердить `unsafe.Sizeof` и offsets для целевых architectures.
4. Изолировать bit packing helpers и протестировать min/max каждого field.
5. Не интерпретировать «cache line 64 bytes» как универсальный hardware факт.

Functional options здесь решают API ergonomics, но не compression. Они применяются последовательно, и repeated option должен иметь понятный last-write-wins contract.

### Уточнения интервьюера

- Нужно ли поддерживать Unicode name? Тогда «42 characters» не равно 42 bytes.
- Как вернуть validation error, если `Option` имеет type `func(*GamePerson)` без error?
- Что важнее: компактность, простота или update cost?
- Как сериализовать bit-packed layout без привязки к machine endianness?

## Задание 7. DI container

![[90 Вложения/CurseHunter/6860/Кадры/07-homework-di-container.jpg|720]]

### Контракт курса

Container регистрирует constructor по string name и создаёт новый instance при каждом `Resolve`. Неизвестное имя возвращает `(nil, error)`. Тест проверяет, что два `UserService` имеют разные addresses. Источник: [`homework/interfaces`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/interfaces/homework_test.go).

Задание со звёздочкой — singleton registration; reflection и generics предлагаются как дополнительные инструменты.

### Минимальная модель

```text
registry[name] = provider
Resolve(name)  = provider()
```

Хранить provider как `func() any` лучше, чем arbitrary `any`: тогда invalid constructor отсекается при registration, а не panic во время resolve. String keys удобны для упражнения, но допускают typo и collision; typed keys или generic helper улучшают safety.

### Lifetimes

- **Transient**: provider вызывается при каждом resolve.
- **Singleton**: instance создаётся один раз; concurrent resolve требует `sync.Once` или эквивалентный state machine.
- **Scoped**: instance один на request/job scope; container должен знать boundary.

Singleton initialization error нельзя просто потерять: хранится пара `(value,error)`. Если retry after failure разрешён, `sync.Once` уже не подходит без дополнительного protocol.

### Production failure modes

- overwrite registration без явного policy;
- concurrent map read/write;
- constructor panic;
- dependency cycle `A → B → A`;
- partial singleton publication;
- service locator скрывает dependencies и усложняет tests.

В обычном Go явная constructor injection часто проще container: dependencies видны в signatures, compile-time type checking сильнее, lifecycle прозрачен. Container выигрывает, когда graph строится динамически, есть plugins или generated wiring.

### Уточнения интервьюера

- Как обнаружить cycle и вывести dependency path?
- Где хранить scope и cleanup order?
- Что произойдёт при concurrent Register и Resolve?
- Как зарегистрировать interface, не теряя concrete type safety?

## Задание 8. MultiError

![[90 Вложения/CurseHunter/6860/Кадры/08-homework-multierror.jpg|720]]

### Контракт курса

```go
type MultiError struct { /* ... */ }
func (e *MultiError) Error() string
func Append(err error, errs ...error) *MultiError
```

Два последовательных `Append` должны дать message `2 errors occured:\n\t* error 1\t* error 2\n`. Задание со звёздочкой — совместимость с `errors.Is`, `errors.As`, `errors.Unwrap`. Источник: [`homework/errors`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/errors/homework_test.go).

### Semantics до форматирования

Сначала нужно определить data contract:

- игнорируется ли `nil`;
- flatten ли вложенный `MultiError`;
- сохраняется ли order;
- возвращает ли пустой aggregate `nil` или non-nil error;
- можно ли вызывать `Append` с обычной исходной error;
- immutable ли aggregate после публикации.

Формат message вторичен. Machine logic должна обходить error tree через `Is/As`, а не парсить text.

### Интеграция со стандартной библиотекой

Для множества children нужен:

```go
func (e *MultiError) Unwrap() []error
```

Тогда `errors.Is` и `errors.As` обходят дерево. Начиная с Go `1.20`, `errors.Join` уже создаёт error, оборачивающую несколько non-nil errors. Собственный type остаётся полезен, если нужен стабильный formatting, metadata, categories или domain-specific API.

### Error from defer

Практический сценарий aggregation — основная operation плюс несколько cleanup errors. Named result позволяет присоединить close error, но правило должно сохранять primary cause и не маскировать panic. Если cleanup error не влияет на outcome, её всё равно надо наблюдать через log/metrics с ownership на одном уровне.

### Уточнения интервьюера

- Как избежать typed-nil `*MultiError` внутри `error`?
- Должен ли `Append(nil)` вернуть `nil`?
- Чем flat list отличается от error tree?
- Потокобезопасен ли aggregate и нужен ли он mutable?

## Общие failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| tail recursion bounded | stack overflow | нет гарантии TCO | loop/explicit stack |
| compact struct = переставить fields | размер всё ещё >64 B | information budget не сходится | считать bits и ranges |
| reflection делает DI typed | panic при resolve | type error отложена до runtime | typed provider API |
| error text — стабильный protocol | logic ломается после wrapping | string для человека | `Is/As/Unwrap` |
| log и return на каждом уровне | duplicate noisy logs | error обработана много раз | один owner handling |
