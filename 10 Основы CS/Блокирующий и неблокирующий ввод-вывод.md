# Блокирующий и неблокирующий ввод-вывод

## TL;DR

Blocking I/O разрешает системному вызову усыпить вызывающий thread, пока операция не сможет сделать progress. Non-blocking I/O запрещает именно это ожидание: если немедленный progress невозможен, вызов быстро возвращает `EAGAIN`/`EWOULDBLOCK`; для non-blocking `connect()` нормальным промежуточным результатом служит `EINPROGRESS`.

Non-blocking не делает устройство быстрее и не превращает readiness в completion. Приложение всё равно выполняет `read`, `write`, `accept` или проверку `SO_ERROR`, обрабатывает partial result и хранит незавершённое состояние. [[10 Основы CS/select, poll, epoll и kqueue|select/poll/epoll/kqueue]] лишь помогают спать до вероятной готовности сразу многих FD.

`O_NONBLOCK` хранится в open file description. Дубликаты после `dup()` и `fork()` видят один флаг. Для `recv()` Linux также даёт `MSG_DONTWAIT`, который действует только на один вызов и не меняет режим остальных владельцев descriptor.

## Область применимости

Поведение сверено для Linux на tag `v7.1`, Linux man-pages 6.18 и POSIX.1-2024. Главный scope — sockets, pipes, FIFOs и устройства, где операция естественно ждёт события. Для regular files и block devices Linux в этой реализации обычно игнорирует `O_NONBLOCK`: page-cache miss или device I/O всё равно способны кратко заблокировать thread. Non-blocking readiness API не заменяет asynchronous file I/O.

## Ментальная модель

Режим отвечает на один вопрос: что делать, если операция не может **немедленно** передать ни одного байта или принять connection?

```text
готово немедленно? ── да ─> выполнить доступную часть, вернуть count / result
      │
      нет
      ├─ blocking fd     -> поставить thread в wait queue, разбудить позже
      └─ non-blocking fd -> вернуть EAGAIN/EWOULDBLOCK
```

Thread, уснувший в ядре, не крутит busy loop и обычно не потребляет CPU. Цена проявляется в другом: его stack и runtime state остаются, wakeup требует scheduling, а отмена и deadline должны прервать ожидание корректным способом.

Non-blocking переносит ожидание из одного вызова в state machine приложения. Это обмен: меньше blocked threads и единая точка multiplexing, но больше состояний, buffers и веток ошибок.

## Как устроено

### Где хранится `O_NONBLOCK`

Флаг читают и меняют через `fcntl(F_GETFL/F_SETFL)` либо задают атомарно при создании: `socket(..., SOCK_NONBLOCK)`, `accept4(..., SOCK_NONBLOCK)`, `pipe2(O_NONBLOCK)`. Он принадлежит [[10 Основы CS/Файловые дескрипторы|open file description]], поэтому `dup(fd)` не создаёт отдельный blocking mode.

Это важно для библиотек. Если библиотека получила borrowed FD и включила `O_NONBLOCK`, она меняет поведение владельца и других дубликатов. Per-call `MSG_DONTWAIT` для `send`/`recv` избегает такого side effect, но относится только к socket API.

### Контракты `read` и `recv`

У blocking stream socket `read` ждёт, пока появится хотя бы часть данных, EOF или error; без специального протокольного условия он не обязан заполнить весь buffer. У non-blocking socket отсутствие данных даёт `EAGAIN`/`EWOULDBLOCK`.

Положительный short count — успех. Он означает «за этот вызов передано столько», а не EOF и не ошибку. При ненулевом запрошенном размере ноль у stream `read`/`recv` означает EOF после исчерпания buffered data; запрос нулевого размера тоже может вернуть `0` как no-op, не сообщая об EOF. Для ненулевого запроса код различает три исхода:

```text
n > 0       обработать n байт; возможно, продолжить
n == 0      peer закрыл write-half, входной поток завершён
n == -1
  EAGAIN    немедленный progress невозможен, ждать readiness
  EINTR     вызов прерван signal, решение о retry зависит от deadline
  другое    terminal или protocol-specific error
```

`MSG_WAITALL` не отменяет необходимость обрабатывать signals, disconnects и errors. Application framing всё равно нельзя строить на предположении «один вызов заполнит buffer».

### Контракт `write` и `send`

Blocking write может ждать свободного места, но также вправе вернуть partial count, например после signal. Non-blocking write принимает доступный prefix; если не помещается ни байта, возвращает `EAGAIN`.

После partial write приложение сохраняет offset в output buffer. Повтор с начала продублирует уже принятые bytes. Бесконечно регистрировать writable events тоже нельзя: socket почти всегда writable при малой очереди, поэтому event loop будет просыпаться впустую. Интерес к `EPOLLOUT` обычно включают, когда появился непринятый output. Flush продолжают до empty queue или `EAGAIN`: в первом случае interest выключают, во втором сохраняют, пока pending suffix не будет дописан.

### `accept()` и `connect()`

Blocking `accept()` ждёт connection в очереди listener. Non-blocking возвращает `EAGAIN`, когда очередь пуста. Даже после readable notification нужен non-blocking mode: connection мог забрать другой worker или удалить network error.

Blocking `connect()` ждёт outcome установки connection. Non-blocking обычно возвращает `EINPROGRESS`. Затем socket ждут как writable и читают `SO_ERROR`: `0` означает success, другое значение — failure. Один writable bit без `SO_ERROR` недостаточен.

После failed `connect()` portable программа закрывает socket и создаёт новый для следующей попытки: стандарт не обещает удобное повторное использование неопределённого состояния.

### Readiness не даёт reservation

Ready означает, что соответствующая операция **не блокировалась бы в момент проверки**. Между проверкой и I/O другой thread способен потребить data или место в buffer; Linux также документирует редкие spurious readiness для sockets. Поэтому socket в event loop оставляют non-blocking даже с level-triggered API.

В edge-triggered режиме правило строже: после события нужно повторять операцию, пока она не вернёт `EAGAIN`. Если прочитать лишь часть и снова уснуть, новое edge может не появиться, хотя старые данные остались в buffer.

### Deadlines, signals и cancellation

`EINTR` не означает автоматически «повтори тот же вызов». `SA_RESTART` перезапускает только часть interfaces, а повтор с исходным relative timeout способен растянуть общий deadline. Надёжный цикл хранит absolute monotonic deadline, после каждого wakeup пересчитывает остаток и проверяет cancellation.

Blocking design требует способа разбудить thread: timeout socket option, закрытие protocol state, cancellation FD, signal или runtime-specific механизм. Закрытие того же FD из другого thread не универсальная отмена на Linux: blocked syscall удерживает open file description и способен завершиться успешно.

### Blocking, non-blocking и asynchronous completion

Blocking и non-blocking описывают поведение попытки I/O. Readiness multiplexing сообщает, когда попытку стоит повторить. Asynchronous completion API принимает operation и позже сообщает, сколько реально передано. Эти оси связаны, но не тождественны. `epoll` — readiness API; `io_uring` способен работать как submission/completion API. Для regular-file latency одно лишь `O_NONBLOCK` не решает задачу.

## Пример или трассировка

У `socketpair()` обе стороны переведены в non-blocking mode, read side зарегистрирован в edge-triggered `epoll`:

```text
recv(a, 4096)                     -> -1 EAGAIN
peer send(b, "abcdef", 6)         -> 6
epoll_wait()                      -> a readable
recv(a, 4)                        -> "abcd"
recv(a, 4096)                     -> "ef"
recv(a, 4096)                     -> -1 EAGAIN
epoll_wait()                      -> можно ждать новое изменение

peer shutdown(b, SHUT_WR)
epoll_wait()                      -> a readable / hangup indication
recv(a, 4096)                     -> 0, EOF
```

Первый `EAGAIN` не означал разрыв connection. После event два успешных `recv()` вычерпали buffer, а третий зафиксировал границу, после которой безопасно спать. Если остановиться после `"abcd"`, оставшиеся `"ef"` могут не породить новое edge и state machine зависнет.

## Trade-offs

Thread-per-connection с blocking I/O проще: call stack естественно хранит protocol state, а локальный код выглядит последовательно. Он хорошо работает при умеренном concurrency и runtime, который дешёво multiplexes lightweight tasks. Цена растёт с числом OS threads, stacks, scheduler wakeups и операций, которые нельзя корректно отменить.

Non-blocking event loop держит тысячи mostly-idle connections на малом числе threads и централизует timers. Цена — явные states, buffers, fairness и сложная обработка lifecycle. CPU-heavy handler всё равно блокирует loop, поэтому вычисления выносят или режут budget на iteration.

Blocking с pool ограничивает число одновременно ожидающих операций и даёт естественный admission limit. Но при медленных dependencies все workers могут застрять. Non-blocking сохраняет способность принимать события, однако без bounded concurrency просто накапливает больше in-flight state.

Per-FD `O_NONBLOCK` удобен для event loop. Per-call `MSG_DONTWAIT` лучше, когда один владелец иногда хочет пробную операцию, а shared description должно остаться blocking.

## Типичные ошибки

### Крутить retry после `EAGAIN`

- **Неверное предположение:** non-blocking означает «повторять, пока получится».
- **Симптом:** core загружен на 100%, progress нет.
- **Причина:** busy loop проверяет неизменившееся состояние.
- **Исправление:** зарегистрировать interest и спать в readiness API; добавить deadline.

### Терять suffix после short write

- **Неверное предположение:** успешный `send()` принял весь buffer.
- **Симптом:** peer получает обрезанный frame или повтор prefix.
- **Причина:** return count проигнорирован.
- **Исправление:** хранить offset непринятых bytes и продолжать с него после writable readiness.

### Полагаться на readiness как на lock

- **Неверное предположение:** раз FD ready, следующий call точно не заблокируется.
- **Симптом:** event-loop thread зависает после гонки с другим consumer.
- **Причина:** readiness не резервирует data; состояние меняется между calls.
- **Исправление:** `O_NONBLOCK`, один I/O owner на connection и обработка `EAGAIN` как нормального исхода.

### Применять `O_NONBLOCK` к regular file как asynchronous disk I/O

- **Неверное предположение:** page-cache miss вернёт `EAGAIN`.
- **Симптом:** latency spike всё равно останавливает event loop.
- **Причина:** Linux на tag `v7.1` игнорирует `O_NONBLOCK` для regular files и block devices.
- **Исправление:** отдельные workers или подходящий asynchronous completion API; измерять конкретный storage path.

## Когда применять

Blocking I/O подходит, когда concurrency ограничен, последовательный код ценнее и каждое ожидание имеет deadline/cancellation. Non-blocking нужен event loop и системам с большим числом mostly-idle sockets. Выбор нельзя делать отдельно от admission: обе модели ломаются при безразмерном числе in-flight операций, только накапливают его в разных местах.
