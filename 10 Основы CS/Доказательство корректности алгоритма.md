# Доказательство корректности алгоритма

## TL;DR

Корректность — это утверждение вида: при выполнении предусловий алгоритм завершается и возвращает результат, удовлетворяющий postcondition. Тест показывает поведение на выбранных входах; proof объясняет, почему ошибка невозможна во всём заявленном классе.

На coding interview обычно хватает короткого proof sketch. Для цикла назовите инвариант, покажите initialization, maintenance и termination. Для recursion используйте индукцию по размеру подзадачи; для greedy — exchange или stays-ahead argument; для graph algorithms — свойство frontier, cut или relaxation. Доказательство должно опираться ровно на тот механизм, который реализован в коде.

## Ментальная модель

У программы есть контракт на трёх уровнях:

1. **Precondition:** какие входы допустимы и что о них известно.
2. **Invariant:** что остаётся истинным после каждого шага.
3. **Postcondition:** что обязано быть истинно при возврате.

Инвариант служит мостом. Он достаточно силён, чтобы вместе с условием остановки вывести postcondition, но остаётся простым для проверки после одного шага.

Для total correctness нужно ещё доказать termination. В цикле обычно убывает неотрицательная мера: длина интервала, число необработанных вершин, capacity остатка. В recursion уменьшается размер подзадачи и достигается base case.

## Как устроено

### Цикл

Proof loop invariant состоит из трёх частей:

- **Initialization:** инвариант истинен до первой итерации.
- **Maintenance:** если он истинен перед итерацией, тело сохраняет его.
- **Termination:** условие выхода вместе с инвариантом даёт требуемый результат.

Отдельно укажите variant, который строго уменьшается и ограничен снизу. Без этого доказана лишь partial correctness: если программа завершится, результат будет правильным.

### Рекурсия и dynamic programming

Base cases должны удовлетворять контракту непосредственно. Индукционный шаг предполагает корректность меньших подзадач и доказывает, что текущая комбинация даёт корректный ответ. Для DP та же зависимость должна совпадать с порядком вычисления states; чтение ещё не вычисленного state ломает механизм, даже если recurrence записана верно.

### Greedy

Локально лучший выбор сам по себе ничего не доказывает. Exchange argument берёт оптимальное решение, не совпадающее с greedy choice, и заменяет его первый фрагмент без ухудшения. Это доказывает безопасность выбора, но не весь алгоритм. Отдельно нужно показать optimal substructure: после фиксации выбора допустимые продолжения образуют подзадачу того же вида, её ограничения сохранены, а оптимальное продолжение вместе с greedy prefix даёт оптимум исходной задачи.

### Графы

Proof обычно привязан к frontier. BFS доказывает порядок слоёв, DFS — состояние открытых и завершённых вершин, Dijkstra — окончательность минимальной tentative distance при неотрицательных весах. Как только предусловие о весах снято, proof Dijkstra распадается в точке greedy choice.

## Пример или трассировка

Нужно вернуть первую позицию, где в отсортированном `a` можно поставить `x`, не нарушив порядок. Используем полуинтервал `[lo, hi)`:

```text
lo = 0, hi = len(a)
while lo < hi:
    mid = lo + (hi - lo) / 2
    if a[mid] < x:
        lo = mid + 1
    else:
        hi = mid
return lo
```

Инвариант:

- `0 <= lo <= hi <= len(a)`;
- для каждого индекса `i` из `0 <= i < lo` выполнено `a[i] < x`;
- для каждого индекса `i` из `hi <= i < len(a)` выполнено `a[i] >= x`.

Изначально обе доказанные области пусты, а границы равны `lo=0`, `hi=len(a)`. При `lo < hi` формула даёт `lo <= mid < hi`, поэтому `a[mid]` индексируется безопасно. Если `a[mid] < x`, отсортированность позволяет включить весь диапазон до `mid` в левую область. Иначе `mid` и всё справа подходят правой области. Оба обновления сохраняют порядок границ. На каждом шаге `hi-lo` уменьшается, значит цикл завершается.

Для `a = [1, 2, 2, 5]`, `x = 2`:

| `[lo, hi)` | `mid` | `a[mid]` | Новый диапазон |
| --- | ---: | ---: | --- |
| `[0,4)` | 2 | 2 | `[0,2)` |
| `[0,2)` | 1 | 2 | `[0,1)` |
| `[0,1)` | 0 | 1 | `[1,1)` |

При выходе `lo == hi == 1`. Инвариант утверждает, что всё левее меньше `2`, а позиция `1` и всё правее не меньше `2`; значит `1` — первая допустимая позиция. Если `x` отсутствует, тот же proof даёт insertion point.

## Trade-offs

Формальный proof может быть длиннее кода, но на интервью нужен минимальный аргумент, закрывающий риск алгоритма. Слишком слабая фраза «мы просмотрели всё» не объясняет pruning; слишком подробная символика расходует время и скрывает главный invariant.

Тесты и proof дополняют друг друга. Proof работает внутри модели и легко пропускает неверное предусловие, integer overflow или ошибку реализации. Тесты ловят конкретные расхождения, но не покрывают бесконечное пространство входов. Хороший процесс выводит тестовые классы из preconditions, invariants и postcondition.

## Типичные ошибки

**Неверное предположение:** пример доказывает алгоритм. **Симптом:** решение работает на показанном массиве, но ломается на дубликатах. **Причина:** trace проверяет один путь исполнения. **Исправление:** сформулировать утверждение для произвольного допустимого входа и доказать сохранение инварианта.

**Неверное предположение:** достаточно доказать правильный результат при выходе. **Симптом:** recursion не достигает base case или цикл зависает. **Причина:** termination не входит в partial correctness. **Исправление:** назвать строго убывающую, ограниченную снизу меру.

**Неверное предположение:** proof одного алгоритма переносится на похожий код. **Симптом:** Dijkstra применяют к отрицательным рёбрам или binary search смешивает закрытую и полуоткрытую границы. **Причина:** изменилось предусловие либо invariant. **Исправление:** привязать каждую ветвь proof к фактическому обновлению state.

**Неверное предположение:** greedy choice очевиден. **Симптом:** найден контрпример, где локальный optimum блокирует глобальный. **Причина:** нет safe-exchange property. **Исправление:** построить exchange/cut proof либо перейти к DP или полному поиску.

## Когда применять

- Перед кодом сформулируйте postcondition одним предложением.
- Во время кода держите invariant рядом с переменными, которые его реализуют.
- После кода отдельно проверьте termination, границы и соответствие типов математической модели.
- Если proof не складывается, не маскируйте это тестами: вероятно, выбран неверный state или отсутствует предусловие.
