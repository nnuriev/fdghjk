# Объяснение решения во время написания кода

## TL;DR

Полезное объяснение делает проверяемыми решения, а не озвучивает каждую набранную строку. Интервьюер должен слышать контракт, baseline, найденный bottleneck, выбранный invariant, сложность, риск и результат теста. Короткая пауза на размышление нормальна; после неё нужно сообщить, какие варианты рассматривались и почему выбран текущий.

Ритм работы: уточнить условия, назвать простое корректное решение, вывести оптимизацию, зафиксировать invariant, писать код законченными блоками, затем вручную исполнить его на normal и edge cases. Если обнаружена ошибка, проговорите, какое предположение было неверным и какое локальное изменение восстанавливает invariant.

## Ментальная модель

Интервью — совместная проверка гипотезы:

```text
утверждение -> основание -> кодирующий его state -> наблюдаемая проверка
```

Например: «Храню только уже просмотренные значения, поэтому complement в map всегда относится к другому индексу». Здесь есть утверждение, причина, representation и проверяемое следствие. Фраза «теперь создаю map» сообщает только синтаксис.

Объяснение должно идти на уровень выше текущей строки. Во время цикла говорите о том, что означает state до следующей итерации; во время теста — какой дефект выбранный case способен выявить.

## Как устроено

### 1. Уточнить контракт

Спросите только о свойствах, меняющих решение: размер и диапазон входа, duplicates, sortedness, mutability, отсутствие/множественность ответа, invalid inputs. Затем коротко повторите принятое предположение. Это предотвращает ситуацию, где хороший код решает не ту задачу.

### 2. Дать baseline

Назовите полный простой алгоритм и его стоимость. Baseline показывает понимание пространства поиска и даёт fallback. Если сразу видна оптимизация, baseline можно описать двумя предложениями без кода.

### 3. Вывести оптимизацию

Свяжите bottleneck и структуру: «Внутренний проход повторно ищет complement. Map превращает этот поиск в expected `O(1)`, итоговое время ожидаемо `O(n)`, память `O(n)`». Если гарантия expected, так и скажите; не подменяйте её worst case.

### 4. Зафиксировать invariant до кода

Одно предложение перед циклом экономит последующее объяснение. Примеры:

- `[lo, hi)` всегда содержит insertion point;
- queue BFS содержит discovered, но ещё не обработанные вершины в неубывающем порядке расстояния; visited ставится при enqueue;
- `dp[i]` — optimum для prefix длины `i`;
- stack backtracking содержит choices текущего partial solution.

### 5. Кодировать блоками

После initialization, основного цикла и return делайте короткую сверку. Не комментируйте `i++`; объясняйте границу, mutation или связь с invariant. Если нужна минута тишины, обозначьте её и потом резюмируйте решение.

### 6. Закрыть тестом и proof sketch

Пройдите один representative input с изменением state, затем edge case, который атакует границу. Завершите точной time/space complexity и условиями, при которых она верна.

## Пример или трассировка

Задача Two Sum, один ответ гарантирован.

**Уточнение:** «Можно ли использовать один элемент дважды? Нужны original indices? Значения и target помещаются в `int`?»

**Baseline:** «Переберу все пары `i < j`; это `O(n²)` времени и `O(1)` дополнительной памяти».

**Переход:** «Повторяется поиск `target-a[i]`. Пройду слева направо и сохраню ранее увиденные значения. Перед шагом `i` map содержит только индексы `< i`, поэтому найденный complement не совпадёт с текущим элементом».

**Во время кода:** «Сначала ищу complement, затем записываю текущее значение. Если сделать наоборот, случай `target == 2*a[i]` сможет использовать один индекс дважды».

**Трассировка:** для `[3,3]`, target `6` на шаге `0` map пуст; сохраняется `3 -> 0`. На шаге `1` complement `3` найден, возвращаются `(0,1)`.

**Завершение:** «Один проход, ожидаемое `O(n)` из-за map operations, `O(n)` памяти. Проверю ещё empty input только если контракт не гарантирует решение; при гарантии эта ветвь недостижима».

Такое объяснение короче построчного пересказа и одновременно раскрывает correctness, complexity и edge case с duplicates.

## Trade-offs

Непрерывный монолог мешает думать и перегружает канал. Полная тишина скрывает reasoning и не даёт интервьюеру скорректировать неверное предположение. Полезный компромисс — сообщать решения на границах фаз, а внутри сложного фрагмента брать короткую паузу.

Термины экономят время, только если сразу привязаны к механизму. «Использую sliding window» слабее, чем «при неотрицательных значениях удаление слева не увеличивает сумму, поэтому каждый указатель движется только вперёд».

## Типичные ошибки

**Неверное предположение:** нужно озвучивать каждую строку. **Симптом:** рассуждение тонет в «создаю переменную, запускаю цикл». **Причина:** смешаны explanation и диктовка. **Исправление:** говорить о contract, invariant, decision и test intent.

**Неверное предположение:** молчание выглядит как концентрация. **Симптом:** интервьюер не может отличить рабочую паузу от тупика. **Причина:** reasoning остаётся ненаблюдаемым. **Исправление:** обозначить паузу, затем резюмировать варианты и выбор.

**Неверное предположение:** название паттерна заменяет proof. **Симптом:** sliding window применён к отрицательным значениям и пропускает ответ. **Причина:** не названо свойство монотонности. **Исправление:** проговорить предусловие, которое делает movement безопасным.

**Неверное предположение:** найденную ошибку нужно скрыть. **Симптом:** patch накладывается без объяснения и создаёт второй дефект. **Причина:** потерян invariant. **Исправление:** назвать counterexample, восстановить invariant и повторить trace с точки изменения.

## Когда применять

- Тренируйте задачи в паре: партнёр должен после объяснения воспроизвести invariant и trade-off.
- Записывайте себя и отмечайте места, где звучит синтаксис вместо причинного решения.
- Используйте один и тот же каркас: contract, baseline, bottleneck, invariant, code, trace, complexity.
- При подсказке сначала переформулируйте её своими словами и только затем меняйте подход.
