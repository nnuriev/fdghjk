# Тайм-ауты на сетевых уровнях

## TL;DR

Один сетевой запрос проходит DNS, очередь connection pool, connect, TLS handshake, отправку запроса, ожидание headers и чтение body. Таймер каждой фазы защищает свой ресурс, но только единый end-to-end deadline отвечает на вопрос «когда весь результат перестаёт быть полезным». Phase timeout не должен расширять остаток этого бюджета.

Timeout — длительность от локального старта таймера; deadline — последний допустимый момент. Между сервисами безопаснее передавать оставшуюся длительность и на каждом hop строить локальный deadline: так gRPC избегает ошибки из-за clock skew. Истечение таймера запускает cancellation, но не доказывает, что удалённая операция не выполнилась, и не гарантирует мгновенную остановку работы.

## Область применимости

- Заметка описывает границы таймеров в синхронном сетевом request-response и streaming независимо от языка и конкретной библиотеки.
- Распространение общего бюджета и кооперативной отмены по дереву работ подробно разобрано в [[20 Бэкенд/Дедлайны запросов и распространение отмены|дедлайнах запросов]]. Здесь фокус — на фазах сетевого пути и несовпадении их границ.
- Конкретные поля `net/http` для Go 1.26.5 и их ограничения находятся в [[60 Go/Тайм-ауты HTTP-сервера и клиента|заметке о HTTP-тайм-аутах Go]].
- Поведение TCP retransmission timer соответствует RFC 6298; примеры proxy timeout semantics — Envoy 1.38.3; руководства gRPC проверены 2026-07-18.

## Ментальная модель

Deadline — это уменьшающийся бюджет, а phase timeouts — предохранители внутри него:

```text
end-to-end budget
  ├─ DNS
  ├─ pool/admission queue
  ├─ TCP connect
  ├─ TLS handshake
  ├─ request headers/body
  ├─ response headers
  ├─ response body or stream idle
  └─ retry backoff + next attempts
```

Перед каждой фазой компонент вычисляет:

```text
phase_deadline = min(parent_deadline, now + phase_cap)
remaining      = parent_deadline - now
```

Внутри одного процесса elapsed time желательно считать по monotonic clock. Через сеть нельзя просто доверять wall-clock timestamp двух машин: gRPC преобразует deadline в timeout, вычитает уже прошедшее время и на принимающей стороне восстанавливает локальный deadline.

Главные инварианты:

1. Очередь и backoff расходуют тот же бюджет, что и полезный I/O.
2. Более внутренний timeout может закончиться раньше родительского, но не выдаёт downstream новый полный бюджет.
3. Timeout ограничивает ожидание наблюдателя. Удалённый эффект после отправки bytes может иметь ambiguous outcome.
4. Idle timeout ограничивает период **без прогресса**, а не полную длительность stream.
5. Cancellation освобождает ресурсы только там, где transport и код действительно её обрабатывают.

## Как устроено

### Какую фазу ограничивает каждый таймер

Названия полей различаются между библиотеками, поэтому контракт задают через события start/end, а не через слово `timeout`.

| Таймер | От какого события до какого | Что ограничивает | Частая слепая зона |
|---|---|---|---|
| End-to-end deadline | Приём логической операции → окончательный ответ/отказ | Полную ценность работы, включая retries и локальную обработку | Сам по себе не объясняет, где ушло время |
| DNS lookup | Потребовался адрес → получен usable result/error | Resolver, IPC и DNS queries | Application/runtime cache может вообще пропустить lookup |
| Pool wait | Нужен connection/stream → ресурс выделен или очередь отказала | Локальную очередь перед сетью | Никакого packet ещё не отправлено, но budget уже расходуется |
| TCP connect | Начат connect → соединение установлено/error | SYN exchange и transport retries до connect | Не включает DNS; иногда proxy CONNECT и Happy Eyeballs имеют отдельные границы |
| TLS handshake | Есть transport → handshake и certificate validation завершены | Криптографическое согласование и проверку peer | Reused connection пропускает фазу; новый — нет |
| Request-header | Server начал читать → получил полные request headers | Slow clients и занятые server connections | Не ограничивает медленное request body |
| Request-body | После headers → body прочитано/прервано | Upload и server resources | Total timeout ломает легальный streaming; idle timeout не ограничивает общую длину |
| Response-header | Request отправлен → получены response headers | Queueing и обработку upstream до первого metadata | Не ограничивает медленный response body |
| Body/stream idle | Последний прогресс → следующий byte/frame | Зависший поток без событий | Регулярный маленький heartbeat способен держать stream бесконечно |
| Per-try timeout | Старт конкретного attempt → его outcome | Одну физическую попытку | Разные proxies включают или исключают pool wait и upload |
| TCP RTO | Сегмент требует подтверждения → retransmit/transport failure | Надёжность доставки bytes | Это transport recovery, а не deadline приложения |

RFC 6298 вычисляет TCP retransmission timeout из сглаженных RTT и их вариации; после retransmission timeout RTO экспоненциально увеличивается. Transport может продолжать восстанавливать доставку, когда application deadline уже истёк. Поэтому socket без cancellation способен удерживать connection и буферы дольше полезного запроса.

### Один бюджет и локальные пределы

End-to-end deadline выбирают из SLO и ценности результата, а phase caps — из роли ресурса и наблюдаемого распределения задержек. DNS cap не должен «забирать зарезервированные 100 ms» автоматически: если cache hit занял 1 ms, остаток остаётся запросу. Cap — верхняя граница, не квота, которую обязательно потратить.

Перед стартом дорогой фазы проверяют не только `remaining > 0`, но и то, хватит ли времени для минимально полезной работы и возврата ответа. Новый attempt за 5 ms до deadline обычно лишь добавляет нагрузку. Политика retries и jitter раскрыта в [[40 Распределённые системы/Retry, exponential backoff и jitter|отдельной заметке]], а семантическая допустимость повтора — в [[10 Основы CS/Retry safety|Retry safety]].

Очереди должны получать deadline при enqueue. Иначе request может истечь, пока ждёт pool/worker, а затем всё равно занять освободившийся ресурс. При dequeue проверяют остаток ещё раз. Сигнал cancellation распространяют в DNS API, connect, TLS, запись/чтение body и downstream handler; необрабатываемая отмена оставляет orphan work.

### Где границы расходятся

У клиента «response-header timeout» начинается после отправки запроса, а у server «request-header timeout» защищает чтение входных headers. Одинаковое слово `header` описывает противоположные стороны соединения. Аналогично per-try timeout одного proxy может начаться после получения всего downstream body и получения upstream connection; другая библиотека включит upload и pool wait. Без точных start/end две настройки с числом `500 ms` несопоставимы.

Connection reuse меняет путь. На cache hit и тёплом pooled connection нет DNS, connect и полного TLS handshake; холодный request проходит все фазы. Если timeout подобран только по steady state, deploy, scale-out или rotation certificates создают волну ложных отказов. Жизненный цикл и пределы pool разобраны в [[20 Бэкенд/Пулы соединений и keep-alive|пулах соединений]].

Внешний слой может вернуть timeout раньше внутреннего. Если proxy ждёт 800 ms, а backend продолжает работу 5 s без propagated cancellation, клиент видит быстрый отказ, но capacity не освобождается. Обратная ошибка — сумма независимых полных timeouts: DNS 1 s + connect 1 s + TLS 1 s + response 2 s способна превратить «client timeout 2 s» в неожиданный путь дольше двух секунд, если библиотека не трактует его как общий предел.

### Streaming и idle

Для потока неизвестной длины общий deadline может быть частью бизнес-контракта, но часто отдельно нужны:

- maximum stream duration — предельная жизнь stream;
- idle timeout — максимальный интервал без meaningful progress;
- per-read/per-write deadline — защита конкретной операции I/O;
- cancellation при потере интереса downstream.

Idle timer сбрасывают только событием, которое действительно означает прогресс. Пустой control frame или heartbeat не всегда должен разрешать бесконечно удерживать дорогой upstream. С другой стороны, total body timeout, рассчитанный для JSON response, оборвёт нормальный watch или большой download несмотря на устойчивый прогресс.

### Наблюдаемость

Одной метрики `timeout_total` недостаточно. Для каждого логического request записывают parent deadline, remaining budget при входе/выходе hop, phase, queue/pool wait, cold/reused connection, attempt number, bytes sent/received, last-progress age, инициатора cancellation и работу, законченную после отмены.

Нужно различать local timeout proxy, upstream reset, downstream disconnect и backend deadline. Иначе рост `504` можно ошибочно лечить увеличением backend timeout, хотя время съедает локальная pool queue или DNS failure.

## Пример или трассировка

У запроса end-to-end deadline `t=1200 ms`. В этой системе per-try timer стартует, когда upstream connection получен и request передан transport; stream idle равен 100 ms.

1. DNS cache miss занимает 70 ms. Остаток — 1130 ms.
2. Запрос ждёт свободный слот connection pool 140 ms, затем TCP connect занимает 90 ms, TLS handshake — 80 ms. Первый attempt начинается в `t=380 ms`.
3. Per-try limit 350 ms срабатывает в `t=730 ms`. Outcome неоднозначен: request bytes уже ушли, поэтому сам timeout не разрешает повтор.
4. Контракт операции допускает retry; jittered backoff занимает 40 ms. Второй attempt стартует в `t=770 ms`, а не получает новые 1200 ms.
5. Response headers приходят через 180 ms, в `t=950 ms`. Body даёт chunk каждые 60 ms: `1010`, `1070`, `1130`, `1190`. Idle timeout ни разу не срабатывает.
6. В `t=1200 ms` общий deadline отменяет stream до terminal response. Backend обязан заметить cancellation; уже зафиксированный side effect она не откатывает.

Наблюдаемый результат: все локальные фазы укладывались в свои caps, но логический request всё равно не успел. Без end-to-end deadline регулярные chunks могли бы удерживать stream неограниченно; без метрик по фазам итог выглядел бы просто как «timeout на 1,2 s».

## Trade-offs

Короткие phase limits быстрее освобождают scarce connections и threads, но режут редкий здоровый tail, особенно на cold path. Длинные limits уменьшают false timeouts, зато позволяют queue buildup и удерживают ресурсы во время outage. Настройки выбирают по latency distributions каждой фазы и проверяют под cold start, packet loss и saturation, а не только по среднему времени ответа.

Один общий deadline проще сохраняет бизнес-бюджет, но не локализует зависание. Набор phase timers лучше защищает отдельные ресурсы, однако повышает сложность и риск несовместимых границ. Практический вариант — обязательный общий deadline плюс минимальный набор измеряемых phase caps, каждый вложен в остаток.

Absolute deadline выражает единый момент окончания, но прямой timestamp между машинами чувствителен к clock skew. Remaining duration переносится устойчивее, хотя теряет часть времени на сериализацию и transit; корректный proxy вычитает elapsed time, а не пересылает исходную длительность без изменений.

## Типичные ошибки

### Каждый слой начинает полный timeout заново

- **Неверное предположение:** локальные `2 s` на gateway и backend вместе всё равно равны двум секундам.
- **Симптом:** клиент давно ушёл, а нижние сервисы продолжают работу и создают очередь.
- **Причина:** timeout интерпретирован как новый budget, а не cap внутри parent deadline.
- **Исправление:** переносить остаток, строить локальный deadline и проверять его до queue, attempt и backoff.

### Таймер начинается после «настоящей сети»

- **Неверное предположение:** request timeout покрывает DNS, pool wait, connect и TLS по умолчанию.
- **Симптом:** наблюдаемая latency заметно превышает настройку либо timeout не указывает реальную фазу.
- **Причина:** библиотечный timer стартует только перед ожиданием headers или после получения connection.
- **Исправление:** документировать start/end каждого поля и добавить общий deadline, включающий весь путь.

### Timeout считают доказательством отсутствия эффекта

- **Неверное предположение:** если response не пришёл вовремя, server не применил request.
- **Симптом:** retry создаёт дубликат платежа или заказа.
- **Причина:** timeout произошёл после отправки bytes или durable commit, а ответ потерялся.
- **Исправление:** считать outcome ambiguous и применять правила [[20 Бэкенд/Идемпотентные и неидемпотентные операции|идемпотентности]] и [[20 Бэкенд/Ключи идемпотентности и дедупликация запросов|дедупликации]].

### Idle timeout используют как total timeout

- **Неверное предположение:** если поток постоянно активен, он обязательно скоро завершится.
- **Симптом:** heartbeat удерживает stream и upstream resource бесконечно.
- **Причина:** idle timer сбрасывается прогрессом и не ограничивает общую длительность.
- **Исправление:** отдельно задать maximum stream duration или бизнес-deadline; определить, какие frames считаются meaningful progress.

### Все timeouts агрегируют в один код

- **Неверное предположение:** одинаковый status означает одинаковую причину.
- **Симптом:** увеличение лимита не помогает, а tail latency и capacity ухудшаются.
- **Причина:** DNS, pool exhaustion, connect, TLS, upstream processing и downstream disconnect потеряли provenance.
- **Исправление:** маркировать фазу, инициатора, остаток бюджета и cold/reused path; отделять local reply proxy от upstream outcome.

## Когда применять

Модель нужна при проектировании HTTP/RPC clients, reverse proxies, service mesh и server limits, а также при разборе tail latency и 504/deadline exceeded. Начинают с одного end-to-end budget, затем добавляют phase caps только для ресурсов, где зависание измеримо и требует отдельной защиты.

Перед rollout таблицу таймеров проверяют как контракт: кто запускает timer, что его сбрасывает, входит ли queue/body/retry, что отменяется при expiry и какой error получает соседний слой. Если на эти вопросы нет ответа, числовое значение само по себе почти ничего не гарантирует.
