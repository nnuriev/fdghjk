# Edge cases, невалидный ввод и overflow

## TL;DR

Edge case — допустимый вход на границе поведения; invalid input нарушает предусловие. Их нельзя смешивать: пустой список может быть корректным значением, а индекс вне диапазона — ошибкой контракта. До кода нужно решить, какие входы гарантирует задача и как функция сообщает отсутствие ответа или нарушение предусловия.

Overflow тоже часть контракта. В Go целочисленные операции над runtime values не обязаны выдавать ошибку: unsigned arithmetic вычисляется по модулю `2^n`, signed overflow детерминирован представлением и операцией. Поэтому проверки размера, суммы, произведения и midpoint должны предшествовать опасной операции либо использовать более широкий тип или `math/bits`.

## Ментальная модель

Разделите пространство входов на классы эквивалентности, внутри которых алгоритм идёт по одному причинному пути. Затем проверьте точки разрыва между классами:

- размер: `0`, `1`, минимальный рабочий, максимальный;
- значения: minimum/maximum типа, zero, negative, duplicates, all equal;
- форма: sorted, reverse-sorted, disconnected, cycle, no solution, multiple solutions;
- представление: bytes, Unicode code points или grapheme clusters;
- контракт: valid edge, invalid value, отсутствующий результат.

Для каждой арифметической операции задайте вопрос: помещается ли математический результат в выбранный тип до того, как вычисление произойдёт?

## Как устроено

### Допустимые границы

Пустой вход не должен автоматически считаться ошибкой. Например, сумма пустого диапазона равна нулю, BFS пустого graph не имеет стартовой вершины, а median пустой последовательности может быть неопределён по контракту. Ответ следует из задачи, а не из общего правила.

Дубликаты часто меняют proof: binary search должен вернуть любой match, first/last occurrence или insertion point; set удаляет multiplicity, а multiset сохраняет. Отрицательные значения ломают sliding-window аргумент, если он опирается на монотонный рост суммы.

### Invalid input

На интервью сначала уточните, гарантированы ли constraints. Если гарантированы, достаточно назвать предположение и не раздувать код defensive checks. Если функция проектируется как библиотечная, выберите наблюдаемый outcome: `(value, bool)` для нормального отсутствия, `error` для проверяемого нарушения, panic только для programmer error по явно заявленному контракту.

Production-граница рассматривается подробнее в [[20 Бэкенд/Защитная обработка невалидного ввода|защитной обработке невалидного ввода]], а проверки вокруг порогов и failure paths — в [[20 Бэкенд/Тестирование границ и failure paths|тестировании границ]]. Interview-задача уже, но различие между edge и invalid остаётся тем же.

### Overflow в Go

- `int` и `uint` имеют 32 или 64 бита; код с индексами обычно использует `int`, но агрегаты могут требовать `int64`.
- Untyped constants вычисляются с высокой точностью на этапе компиляции, однако присваивание требует representability.
- Проверка после `a+b` опаздывает, если результат уже overflowed. Для положительных `int64` сначала проверяют `a > math.MaxInt64-b`.
- Midpoint безопаснее писать `lo + (hi-lo)/2`, если invariant гарантирует `0 <= lo <= hi` и разность представима.
- `-MinInt` и `abs(MinInt)` не представимы в том же signed type.
- `math/bits.Add64`, `Mul64` и родственные функции возвращают carry или полную ширину результата, когда это входит в контракт.

## Пример или трассировка

Пусть binary search работает не с slice index, а с произвольным закрытым диапазоном `int64`:

```text
lo = MaxInt64 - 4
hi = MaxInt64
```

Математическая сумма `lo+hi` больше `MaxInt64`, поэтому `(lo+hi)/2` сначала overflowed и уже затем делится. Формула `lo+(hi-lo)/2` вычисляет разность `4`, половину `2` и результат `MaxInt64-2`, то есть остаётся в диапазоне.

Но формула безопасна не магически. Если `lo > hi`, разность отрицательна и invariant поиска уже нарушен; если диапазон допускает полный переход от `MinInt64` к `MaxInt64`, сама разность не помещается. Исправление начинается с модели диапазона, а не с механической замены выражения.

Минимальная матрица для lower bound:

| Класс | Вход | Что проверяет |
| --- | --- | --- |
| empty | `[]`, `x=3` | возврат insertion point `0` без индексирования |
| singleton left | `[5]`, `x=3` | позиция до первого элемента |
| singleton right | `[5]`, `x=7` | позиция после последнего элемента |
| duplicates | `[2,2,2]`, `x=2` | именно first position |
| absent gap | `[1,4]`, `x=3` | внутренний insertion point |

## Trade-offs

Defensive validation повышает ясность публичного API, но в ограниченной interview-функции может скрыть основной алгоритм и создать неоговорённую семантику. Правильный компромисс — вслух зафиксировать constraints, а в код добавлять только проверки, меняющие outcome.

Переход с `int` на `int64` расширяет диапазон, но не доказывает отсутствие overflow. `big.Int` снимает фиксированный предел ценой allocations, более сложного API и другой модели производительности. Saturating arithmetic, wraparound и error — три разные семантики; выбирать их молча нельзя.

## Типичные ошибки

**Неверное предположение:** edge case означает invalid input. **Симптом:** корректный пустой результат превращается в error. **Причина:** не сформулирован domain функции. **Исправление:** выписать precondition и нормальный outcome для каждого граничного класса.

**Неверное предположение:** `int` всегда 64-битный. **Симптом:** код проходит на amd64 и ломается либо не компилируется для 32-bit target. **Причина:** размер `int` зависит от реализации/архитектуры. **Исправление:** выбирать тип по контракту и не смешивать index type с типом агрегата без проверки conversion.

**Неверное предположение:** signed overflow вызовет panic. **Симптом:** сравнение, счётчик или distance становится отрицательным и ведёт алгоритм по неверной ветви. **Причина:** спецификация Go определяет значение overflowed operation, а не runtime exception. **Исправление:** проверять до операции или использовать API с carry/full-width result.

**Неверное предположение:** набор случайных тестов покрывает границы. **Симптом:** ошибка `<=` вместо `<` проявляется только на последнем элементе. **Причина:** случайная выборка не нацелена на discontinuities. **Исправление:** сначала составить boundary matrix, затем дополнять fuzzing.

## Когда применять

- Перед реализацией перечислите минимум: empty, singleton, duplicates, no solution и numeric extremes, если они допустимы.
- Для каждого `+`, `-`, `*`, shift и conversion оцените математический диапазон результата.
- Согласуйте invalid-input policy до того, как писать branches.
- В конце проведите ручную трассировку на одном нормальном и двух граничных входах.
