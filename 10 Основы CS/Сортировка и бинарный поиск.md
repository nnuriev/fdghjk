# Сортировка и бинарный поиск

## TL;DR

Вопрос заметки: когда выгодно сначала создать порядок, а затем использовать binary search, и какой invariant делает поиск корректным?

Sorting платит preprocessing, чтобы превратить многие будущие решения в monotonic scan/search. Comparison sort требует `Ω(n log n)` comparisons в worst-case; linear-time sorting возможна при дополнительных ограничениях на keys. Binary search занимает `Θ(log n)`, если predicate монотонен на search space. Sorted array — частный случай: predicate `a[i] ≥ target` сначала false, затем true.

Надёжный шаблон держит half-open interval `[lo, hi)` и invariant «первый true находится внутри, либо равен `hi`». Каждая итерация обязана строго уменьшать interval. Comparator sorting и predicate search должны задавать один и тот же order; иначе корректный search code работает поверх некорректного контракта.

## Ментальная модель

Sorting покупает информацию о порядке один раз. Binary search тратит её по одному bit на comparison: каждый шаг отбрасывает примерно половину кандидатов.

```text
неупорядоченные данные: проверить кандидатов по одному
упорядоченные данные:   сравнение исключает целый interval
```

Search не обязан искать готовое значение в array. Он ищет boundary монотонного predicate:

```text
false false false | true true true
                  ^ первый true
```

Вся доказательность сидит в двух местах: почему predicate действительно монотонен и почему update сохраняет boundary в `[lo,hi)`.

## Как устроено

### Sorting algorithms меняют разные ресурсы

| Алгоритм | Time | Auxiliary space | Stable | Практический смысл |
| --- | --- | --- | --- | --- |
| Insertion sort | worst `Θ(n²)` | `Θ(1)` | да | малые или почти sorted inputs |
| Merge sort | `Θ(n log n)` | обычно `Θ(n)` для arrays | да | predictable bound и stability |
| Heapsort | `Θ(n log n)` | `Θ(1)` | нет | worst-case bound при малой памяти |
| Quicksort | expected `Θ(n log n)` при randomized pivot или random permutation, worst `Θ(n²)` | recursion stack зависит от partitions | нет в обычной in-place форме | locality и малые constants при хорошем pivot policy |
| Counting sort | `Θ(n+U)` | `Θ(n+U)` либо `Θ(U)` по варианту | может быть stable | integer keys из ограниченного universe `U` |

Stable sort сохраняет исходный порядок equal keys. Это нужно при последовательной сортировке по нескольким полям: сначала по менее значимому, затем stable sort по более значимому. «In-place» и «stable» — независимые свойства.

Lower bound `Ω(n log n)` относится к comparison decision tree. Counting/radix sort не нарушают его: они используют структуру keys, а не только pairwise comparisons.

### Sort-or-not — вопрос числа запросов и побочных эффектов

Один lookup в unsorted array стоит `Θ(n)`. Sorting ради одного lookup стоит `Θ(n log n)` и проигрывает. Для `q` запросов варианты выглядят так:

```text
linear scan:          Θ(qn)
sort + binary search: Θ(n log n + q log n)
hash index:           expected Θ(n + q), но без order/range
```

Sorting меняет порядок либо требует copy. Если исходный order семантически значим, эта цена входит в решение. При online updates sorted array платит `Θ(n)` за insertion; тогда balanced tree или hash index могут быть ближе к контракту.

### Half-open binary search

Ищем минимальный index `i` из `[0,n)`, для которого `P(i)=true`; если такого нет, возвращаем `n`.

```text
lo = 0
hi = n
while lo < hi:
    mid = lo + (hi - lo) / 2
    if P(mid):
        hi = mid
    else:
        lo = mid + 1
return lo
```

Loop invariant: все indexes `< lo` точно false; все indexes `≥ hi` точно true либо лежат за исходным range; искомая boundary остаётся в `[lo,hi]`. При `P(mid)=true` boundary не правее `mid`, поэтому `hi=mid`. Иначе `mid` точно false, значит `lo=mid+1`. Interval строго сокращается, termination следует из non-negative integer length.

Формула `lo + (hi-lo)/2` избегает overflow суммы `lo+hi` при representable non-negative bounds. Сама по себе формула не исправляет invalid bounds и не доказывает monotonicity.

### Binary search по ответу

Search space может состоять из capacities, deadlines или distances. Predicate «можно ли выполнить задачу при budget `x`» должен менять значение один раз. Если feasible при `x`, но infeasible при большем `x`, направление выбрано неверно либо predicate не монотонен.

Стоимость равна `O(C(n) log R)`, где `C(n)` — проверка predicate, `R` — размер discrete search range. Для real values число iterations задаёт требуемая precision; floating-point boundary и error tolerance становятся частью контракта.

## Пример или трассировка

Найдём lower bound для `target=6` в sorted array `[1,3,3,6,8,10]`: первый index, где `a[i] ≥ 6`.

| Шаг | `[lo,hi)` | `mid` | `a[mid] ≥ 6` | Новый interval |
| ---: | --- | ---: | --- | --- |
| 1 | `[0,6)` | 3 | true (`6`) | `[0,3)` |
| 2 | `[0,3)` | 1 | false (`3`) | `[2,3)` |
| 3 | `[2,3)` | 2 | false (`3`) | `[3,3)` |

Возвращается index `3`. Ручная проверка invariant: indexes `0..2` содержат values `<6`, index `3` — первый подходящий. Для `target=11` все проверки false и результат был бы `n=6`; это корректная insertion position, а не ошибка off-by-one.

Если array не sorted по тому же comparator, первая проверка ещё может случайно дать правильный ответ, но predicate `a[i] ≥ target` уже не монотонен. Guarantee исчезает.

## Trade-offs

Merge sort выбирают ради stability и predictable time, heapsort — ради worst-case `n log n` с constant auxiliary array space, quicksort — ради хорошей locality/average behavior при управляемом pivot. Реальные library algorithms могут быть hybrid и меняться между версиями; опирайтесь на documented API contract, а не на угаданную внутреннюю реализацию.

Binary search выигрывает на static ordered data и repeated queries. Hash lookup быстрее по expected asymptotics для exact key, зато не даёт predecessor/range и тратит больше metadata. Balanced tree поддерживает dynamic order с `O(log n)` updates, но сложнее и хуже использует contiguous memory.

## Типичные ошибки

- **«Binary search работает для любого predicate» → нужный ответ отбрасывается на раннем шаге → truth values меняются больше одного раза → сначала доказать monotonicity либо выбрать другой search.**
- **«Используем closed interval, но updates возьмём из half-open шаблона» → loop зависает на двух элементах или пропускает boundary → смешаны два контракта bounds → выбрать один interval convention и записать invariant.**
- **«После sorting indexes сохраняют смысл» → возвращается позиция в reordered array вместо исходного объекта → sort разрушил identity/order → сортировать пары `(value, originalIndex)` либо строить отдельный index.**
- **«Comparison sort бывает линейным в общем случае» → lower bound будто нарушен → алгоритм использует ограниченный integer universe или другую операцию → назвать key assumptions и `U`.**
- **«Comparator допускает циклы или меняется во время sort» → результат не образует order, search непредсказуем → нарушены transitivity/consistency → зафиксировать total/strict weak ordering и не мутировать keys.**
- **«Quicksort всегда `O(n log n)`» → adversarial partitions дают quadratic time и linear stack → expected/worst case смешаны → назвать pivot policy, depth guard или выбрать algorithm с worst-case guarantee.**

## Когда применять

Sorting оправдана, если order сам нужен, запросов много или последующий linear scan использует monotonicity. Для единственного exact lookup сначала сравните scan и hash index. Binary search применяйте после явного доказательства вида `false* true*` и с одним interval convention.

В Go `slices.BinarySearch` возвращает earliest matching/insertion position для increasing slice, а `sort.Search` принимает monotone predicate. Это library contracts Go 1.26.5; конкретный algorithm сортировки стандартной библиотеки не стоит превращать в собственную гарантию.
