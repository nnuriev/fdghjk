# Префиксные суммы и difference arrays

## TL;DR

Префиксная сумма (prefix sum) заменяет повторный просмотр диапазона двумя обращениями к заранее посчитанным границам. Для массива `a` длины `n` строят `p` длины `n+1`, где `p[i]` хранит сумму `a[0:i)`. Тогда сумма полуинтервала `[l,r)` равна `p[r]-p[l]`: построение занимает `O(n)`, каждый статический запрос — `O(1)`.

Difference array решает обратную пакетную задачу. Прибавление `x` ко всему `[l,r)` записывают двумя событиями: `delta[l] += x` и `delta[r] -= x`. Один prefix scan после всех обновлений восстанавливает добавку в каждой позиции. Поэтому `q` обновлений и финальная материализация стоят `O(q+n)`, но между обновлениями нельзя получать актуальные range sums за `O(1)`.

Практическое правило: prefix sums берите для многих запросов к почти неизменным данным, difference array — для многих отложенных range updates. Если запросы и обновления чередуются online, нужна динамическая структура, например Fenwick tree или segment tree.

## Ментальная модель

Обычный range-запрос снова складывает элементы, которые уже участвовали в соседних запросах. Prefix array сохраняет контрольные суммы на каждой границе [[10 Основы CS/Массивы, строки, хеш-таблицы и множества|массива]]. Чтобы узнать вклад между двумя границами, достаточно вычесть накопленное слева из накопленного справа.

Difference array хранит не значения, а изменения на границах. Событие `+x` включает добавку в `l`, событие `-x` выключает её в `r`. Prefix scan проводит эти события слева направо и в каждой позиции показывает активную добавку.

Для непустого массива эти представления обратны для сложения:

```text
d[0] = a[0]
d[i] = a[i] - a[i-1],  i > 0

a[i] = d[0] + d[1] + ... + d[i]
```

Последнее равенство следует из телескопической суммы: соседние `+a[j]` и `-a[j]` сокращаются, остаётся `a[i]`. Тот же механизм объясняет range update: внутри диапазона изменения взаимно сокращаются, заметны только две границы.

## Как устроено

### Сначала зафиксировать границы

Удобный контракт для массива длины `n` — полуинтервалы `[l,r)` с предусловием `0 <= l <= r <= n`. Левая граница входит в диапазон, правая не входит. Тогда длина равна `r-l`, пустой диапазон `[i,i)` имеет сумму `0`, а соседние диапазоны `[l,m)` и `[m,r)` стыкуются без особых случаев.

### Prefix sums для статических запросов

Строим массив `p` длины `n+1`:

```text
p[0] = 0
p[i+1] = p[i] + a[i]
```

По определению `p[k] = sum(a[0:k))`. Поэтому

```text
sum(a[l:r)) = p[r] - p[l].
```

Это не мнемоника, а доказательство: из суммы первых `r` элементов вычитаются ровно первые `l`, остаются индексы `l, ..., r-1`. Построение читает каждый элемент один раз, значит стоит `O(n)` времени и `O(n)` дополнительной памяти. `q` запросов после построения стоят `O(n+q)` вместо `O(qn)` в худшем случае.

Scan у Blelloch определён для ассоциативного оператора с identity. Но формула произвольного range-запроса требует ещё обратной операции. Для сложения это вычитание, для XOR — сам XOR; из двух prefix minima восстановить minimum произвольного подмассива нельзя.

### Difference array для пакетных обновлений

Для `q` прибавлений к диапазонам выделяем `delta` длины `n+1`, чтобы индекс `r=n` оставался допустимым. Для каждого обновления `(l,r,x)` выполняем:

```text
delta[l] += x
delta[r] -= x
```

Затем идём слева направо:

```text
active += delta[i]
a[i] += active
```

До `l` обе границы ещё не пройдены, поэтому вклад равен нулю. На `l` включается `+x`; на индексах `l, ..., r-1` он остаётся активен; в `r` событие `-x` его отменяет. Это и есть [[10 Основы CS/Доказательство корректности алгоритма|инвариант цикла]]: перед обработкой `a[i]` переменная `active` равна сумме всех обновлений, чьи полуинтервалы содержат `i`.

Каждое обновление меняет две ячейки за `O(1)`, финальный scan стоит `O(n)`. Цена этой экономии — отложенный результат: до scan отдельные `delta[i]` не равны значениям массива.

### Два измерения

Для матрицы `A` размера `H x W` двумерный prefix `P` имеет размер `(H+1) x (W+1)`:

```text
P[y+1][x+1] = A[y][x]
              + P[y][x+1]
              + P[y+1][x]
              - P[y][x]
```

Последнее вычитание убирает область, которую два предыдущих слагаемых посчитали дважды. Сумма прямоугольника `[y1,y2) x [x1,x2)` равна

```text
P[y2][x2] - P[y1][x2] - P[y2][x1] + P[y1][x1].
```

Для пакетного прибавления `v` к тому же прямоугольнику двумерный difference array размера `(H+1) x (W+1)` получает четыре угловых события:

```text
D[y1][x1] += v
D[y2][x1] -= v
D[y1][x2] -= v
D[y2][x2] += v
```

Двумерный prefix scan восстанавливает вклад во всех ячейках. Время построения и память — `O(HW)`, запрос или запись события — `O(1)`.

## Пример или трассировка

Пусть `a = [3, 1, 4, 1, 5]`. Строим:

```text
p = [0, 3, 4, 8, 9, 14]
```

Запрос `[1,4)` должен сложить `1+4+1`. По границам получаем `p[4]-p[1] = 9-3 = 6`.

Теперь отложенно прибавим `2` к `[1,4)` и `-1` к `[0,2)`. Массив событий длины `6` меняется так:

| Шаг | `delta` |
| --- | --- |
| Начало | `[0, 0, 0, 0, 0, 0]` |
| `+2` на `[1,4)` | `[0, 2, 0, 0, -2, 0]` |
| `-1` на `[0,2)` | `[-1, 2, 1, 0, -2, 0]` |

Ручной scan первых пяти позиций даёт активные добавки `[-1, 1, 2, 2, 0]`. После прибавления к исходному массиву получаем `[2, 2, 6, 3, 5]`: индекс `1` попал в оба обновления, индексы `2` и `3` только в первое, индекс `4` ни в одно. Границы и наблюдаемый результат проверены вручную.

## Trade-offs

| Условия | Подход | Стоимость | Ограничение |
| --- | --- | --- | --- |
| Запросов мало | Прямой проход по `[l,r)` | `O(r-l)` на запрос, `O(1)` памяти | Повторяет работу на пересекающихся диапазонах |
| Много статических range sums | Prefix sums | `O(n)` build, `O(1)` query, `O(n)` памяти | Изменение `a[i]` требует поправить suffix `p`, то есть `O(n)` |
| Много range additions, ответ нужен после всех | Difference array | `O(1)` на update, `O(n)` на материализацию | Не поддерживает online range sum само по себе |
| Updates и queries чередуются | Fenwick/segment tree | Обычно `O(log n)` на операцию, `O(n)` памяти | Сложнее реализация и proof, выше константы |

Двумерный prefix выигрывает при плотной матрице и произвольных прямоугольниках, но платит `O(HW)` памятью и построением. Если запросы лежат только внутри отдельных строк или столбцов, достаточно 1D prefixes по нужным линиям. Для огромного разреженного пространства плотная таблица обычно проигрывает coordinate compression и sweep line.

Наконец, тип аккумулятора выбирают по максимальной промежуточной, а не средней сумме. Prefix может достигать `n * max(abs(a[i]))`, а активная difference-сумма — суммы всех `q` обновлений. Даже если финальный ответ помещается в тип, промежуточные сложения и inclusion-exclusion могут вызвать [[10 Основы CS/Edge cases, невалидный ввод и overflow|overflow]].

## Типичные ошибки

**Неверное предположение:** prefix хранит сумму до индекса включительно, но запрос написан для `[l,r)`. **Симптом:** диапазон теряет первый элемент или захватывает соседний. **Причина:** смешаны координаты элементов и границ. **Исправление:** хранить `p` длины `n+1` с контрактом `p[i] = sum(a[0:i))`.

**Неверное предположение:** после записи двух событий difference array уже содержит обновлённые значения. **Симптом:** point query между обновлениями возвращает бессмысленную границу вместо значения. **Причина:** события ещё не интегрированы prefix scan. **Исправление:** материализовать результат после batch либо выбрать динамическую структуру.

**Неверное предположение:** ассоциативности достаточно, чтобы вычесть любой prefix. **Симптом:** пытаются получить range minimum как разность двух prefix minima. **Причина:** у `min` нет обратной операции, удаляющей вклад левой части. **Исправление:** проверить алгебраические свойства операции и для RMQ использовать подходящую структуру данных.

**Неверное предположение:** четыре знака в 2D difference array можно расставить по памяти. **Симптом:** обновление растекается вправо или вниз за пределы прямоугольника. **Причина:** одна из двух компенсаций либо их пересечение потеряны. **Исправление:** вывести формулу как произведение двух 1D границ: `+` в начале обеих осей, два `-` на противоположных границах, `+` в диагональном углу.

**Неверное предположение:** `int` безопасен, потому что каждый элемент мал. **Симптом:** сумма меняет знак или становится меньше после положительного обновления. **Причина:** accumulator хранит сумму многих элементов или updates. **Исправление:** оценить верхнюю границу промежуточных значений и выбрать тип либо checked arithmetic под этот контракт.

## Когда применять

- Сначала оцените число queries, число updates и будут ли они чередоваться.
- Зафиксируйте `[l,r)` и проверьте `0 <= l <= r <= n` до вывода формул.
- Для prefix query докажите телескопическое сокращение; для difference update назовите момент включения и выключения события.
- В 2D заранее оцените `H*W` по памяти и определите порядок координат `[y][x]`.
- Перед объявлением решения завершённым проверьте пустой диапазон, весь массив, `r=n`, одноэлементный диапазон, пересекающиеся updates и верхнюю границу accumulator.
