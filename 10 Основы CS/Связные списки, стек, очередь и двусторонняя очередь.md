# Связные списки, стек, очередь и двусторонняя очередь

## TL;DR

Вопрос заметки: как выбрать линейную структуру по разрешённым точкам доступа и доказать, что операции не ломают её границы?

Linked list задаёт физическое представление через nodes и links. Stack, queue и deque задают интерфейсы: LIFO, FIFO и операции с обоих концов. Один интерфейс можно реализовать linked nodes либо dynamic/ring array, поэтому название ADT само по себе не определяет locality, allocations и amortized behavior.

У linked list вставка или удаление занимает `Θ(1)` только когда уже известны нужный node и, для singly linked list, predecessor. Поиск позиции остаётся `Θ(n)`. Для stack достаточно одного конца. Queue требует `head` и `tail`, deque на linked representation обычно требует двусторонние links; ring buffer сохраняет элементы contiguous и часто выигрывает на практике.

## Ментальная модель

Смотрите на структуру как на набор разрешённых дверей:

```text
stack:  push/pop <-> один конец
queue:  enqueue -> tail, dequeue <- head
deque:  push/pop <-> head и tail
list:   переходы между соседними nodes
```

Инвариант хранит смысл дверей. Для непустой linked queue `head` указывает на следующий выдаваемый элемент, `tail` — на последний, а `tail.next = nil`. Для пустой очереди оба указателя равны `nil`. Каждая операция обязана сохранить обе ветки; большая часть pointer bugs рождается на переходах `0 ↔ 1` элемент.

Array даёт быстрый индекс, list — дешёвый relink уже найденного места. Это обмен, а не иерархия «простая/продвинутая структура».

## Как устроено

### Singly и doubly linked list

Singly linked node хранит value и `next`. Traversal вперёд стоит `Θ(n)`, вставка после известного node — `Θ(1)`. Удаление известного node всё равно требует predecessor, если нельзя применить специальный трюк с копированием successor; этот трюк не работает для tail и меняет identity node.

Doubly linked node добавляет `prev`. Удаление известного node и движение в обе стороны занимают `Θ(1)`, но каждый node становится тяжелее, а число обновляемых links растёт. Sentinel nodes позволяют убрать часть special cases: пустой список представлен связью sentinel с самим собой либо парой граничных sentinels.

Сложность `Θ(1)` не включает поиск node по индексу или значению. Фраза «удаление из списка constant» без precondition неполна.

### Stack

Stack поддерживает `push`, `pop` и `peek` на вершине. Array-backed stack использует конец dynamic array и получает amortized `Θ(1)` push, worst-case `Θ(1)` pop без shrink. Linked stack вставляет и удаляет head за worst-case `Θ(1)`, но платит allocation и pointer на элемент.

Stack хранит отложенную работу: return addresses, незавершённые обходы, открывающие скобки. Его LIFO-порядок должен вытекать из задачи; использовать stack только потому, что «нужен container», недостаточно.

### Queue

Queue выдаёт элементы в порядке поступления. Linked queue с `head/tail` даёт worst-case `Θ(1)` enqueue/dequeue. Array queue нельзя реализовывать удалением `a[0]` со сдвигом `Θ(n)`. Нужен head index либо circular buffer.

Circular buffer хранит индексы по модулю capacity. Полезный инвариант: `0 ≤ size ≤ capacity`, `head` указывает на первый элемент, позиция новой записи равна `(head + size) mod capacity`. Чтобы различить full и empty, хранят `size`, отдельный flag либо оставляют один slot пустым.

### Deque

Deque поддерживает вставку и удаление с обоих концов. Doubly linked list даёт worst-case `Θ(1)` на эти операции. Ring deque делает то же с индексами и обычно лучше использует cache, но resize одной операции стоит `Θ(n)`, а guarantee становится amortized.

Deque нужен, когда порядок удаления зависит от обоих концов: 0-1 BFS, monotonic queue, work stealing. Если используется только один конец или FIFO-пара, stack/queue выражают контракт точнее.

## Пример или трассировка

Реализуем FIFO queue через singly linked nodes. Начальное состояние: `head=nil`, `tail=nil`, `size=0`.

| Операция | Изменение links | Head | Tail | Выдано |
| --- | --- | --- | --- | --- |
| `enqueue(A)` | новый `A`; оба указателя на него | A | A | — |
| `enqueue(B)` | `A.next=B`; `tail=B` | A | B | — |
| `enqueue(C)` | `B.next=C`; `tail=C` | A | C | — |
| `dequeue()` | сохранить A; `head=A.next` | B | C | A |
| `dequeue()` | сохранить B; `head=B.next` | C | C | B |
| `dequeue()` | `head=nil`; после опустошения `tail=nil` | nil | nil | C |

Ручная проверка: выдача равна `A, B, C`, то есть FIFO. Последний шаг обновляет оба края. Если оставить `tail=C`, следующий `enqueue(D)` может связать новый node со старым detached tail либо получить состояние `head=D, tail=C`. Симптом проявится позже, а причина находится в переходе `1 → 0`.

Те же операции в ring buffer capacity `4` меняли бы только два integer fields и slots массива. Смысл queue сохранился бы, физическое представление и profile allocations изменились.

## Trade-offs

| Реализация | Bounds на концах | Сильная сторона | Цена |
| --- | --- | --- | --- |
| Singly linked | head `Θ(1)`; tail insert с tail pointer `Θ(1)` | простой relink, стабильные node references | нет обратного хода, allocation на node |
| Doubly linked | оба конца и delete known node `Θ(1)` | двустороннее движение | два links и больше invariant updates |
| Dynamic array stack | amortized `Θ(1)` push/pop at end | locality, мало metadata | resize spike |
| Ring queue/deque | amortized `Θ(1)` на концах | locality и bounded variant без allocations | modulo/index invariants, resize либо fixed capacity |

Linked representation выигрывает, если nodes уже существуют и часто перемещаются между списками без копирования. Array/ring buffer обычно лучше для компактных values и последовательного доступа. Нужна жёсткая capacity и predictable latency — fixed ring buffer убирает resize, но вводит явный full outcome.

Прикладная комбинация `map + doubly linked list` для LRU разобрана в [[50 Проектирование систем/Проектирование и реализация in-memory cache в Go|заметке об in-memory cache]]. Она не заменяет базовые инварианты списка: map находит node, list обновляет порядок за `Θ(1)`.

## Типичные ошибки

- **«Удаление из linked list всегда `O(1)`» → решение неожиданно сканирует список → predecessor или node ещё не известен → включить поиск в bound либо изменить API так, чтобы передавать handle.**
- **«Queue на slice можно dequeue физическим сдвигом либо одним reslice» → сдвиги дают суммарное `Θ(n²)`, а `a=a[1:]` без очистки может удерживать backing array и references → обычный array не кодирует ring lifecycle → использовать head index/ring buffer и очищать освобождённый slot.**
- **«После удаления последнего node достаточно сменить head» → empty queue хранит stale tail → следующий enqueue ломает links → на переходе `size=1 → 0` обнулить оба края.**
- **«Doubly linked delete обновляет две ссылки» → traversal назад попадает в удалённый node → забыта одна из четырёх границ `prev.next`/`next.prev` и edge cases → записать invariant и использовать sentinels.**
- **«Deque нужен на всякий случай» → API разрешает операции, которых алгоритм не ожидает → порядок обработки меняется без ошибки типов → выбрать самый узкий ADT.**
- **«Linked list экономит копирование» → latency и memory растут → каждый node добавляет pointers, allocation и плохую locality → сравнить с ring/dynamic array на реальном размере элемента.**

## Когда применять

Stack берите для вложенности и LIFO-worklist, queue — для FIFO-frontier, deque — когда алгоритм доказывает операции с обоих концов. Linked list оправдан, если важны стабильные handles и `Θ(1)` relink известного node. Для index-heavy workload и compact values начните с array/ring buffer.

На интервью сначала назовите ADT и порядок выдачи, потом реализацию. После каждой операции проверяйте empty, one-element, full/wrap-around и stale-reference cases. Такой порядок отделяет доказательство алгоритма от pointer bookkeeping.
