# Массивы, строки, хеш-таблицы и множества

## TL;DR

Вопрос заметки: как выбрать представление для последовательности, текста и membership-запросов так, чтобы нужная операция была дешёвой, а скрытые предпосылки оставались явными?

Массив хранит элементы в индексируемой последовательности и даёт `Θ(1)` random access. За вставку в середину приходится сдвигать хвост, то есть платить `Θ(n)`. Строка добавляет кодировку и семантику символов: индекс по byte не обязан совпадать с пользовательским символом. Hash map реализует соответствие `key → value`, set оставляет только принадлежность; при контролируемом load factor операции обычно имеют expected `Θ(1)`, но worst-case остаётся `Θ(n)`.

Практическое правило: сначала выпишите операции и допустимый key universe. Плотные небольшие integer keys часто лучше хранить в массиве, порядок и range queries требуют упорядоченной структуры, а hash table выигрывает на точечных lookup без требования порядка.

## Ментальная модель

Эти структуры отвечают на разные вопросы:

```text
array:    что лежит в позиции i?
string:   какая последовательность encoded symbols дана на входе?
hash map: какое value связано с key?
set:      встречался ли key?
```

У массива адрес элемента вычисляется из base address и индекса, поэтому доступ не зависит от `n`. Hash table сначала превращает key в индекс группы или bucket, затем разрешает collision. Значит, `O(1)` lookup опирается на hash distribution, сравнение ключей и запас capacity. Это не свойство «словаря» само по себе.

Строка выглядит как массив символов, но это опасное упрощение. В UTF-8 один code point занимает переменное число bytes, а один видимый grapheme может состоять из нескольких code points. Перед алгоритмом выберите единицу: bytes для protocol/token parsing, runes для Unicode code points либо отдельную grapheme segmentation. Детали Go вынесены в [[60 Go/Строки, байты, руны и UTF-8|заметку о строках, байтах и рунах]].

## Как устроено

### Массив и dynamic array

Contiguous layout даёт индексный доступ `Θ(1)` и обычно хорошую spatial locality. Traversal занимает `Θ(n)`. Поиск значения без дополнительного порядка тоже `Θ(n)`. Вставка или удаление в позиции `i` сдвигает `n-i` элементов.

Dynamic array держит `length ≤ capacity`. Append без resize стоит `Θ(1)`, resize копирует текущие элементы. При geometric growth последовательность append имеет `Θ(1)` amortized cost, но отдельный append может занять `Θ(n)`. В Go array и slice имеют разные value/aliasing contracts; их не нужно заново выводить из алгоритмической таблицы, они разобраны в [[60 Go/Массивы и слайсы|заметке о массивах и слайсах]].

### Hash map и set

Map поддерживает `insert`, `lookup` и `delete` по key. Set использует тот же механизм, но value не несёт предметной информации. Типичные реализации решают collisions chaining либо probing. При росте load factor увеличивается число проверяемых кандидатов, поэтому table периодически resize/rehash.

Expected constant time требует предпосылки о распределении hashes или randomization. Все keys могут столкнуться, и тогда поиск просматривает `Θ(n)` кандидатов. Дополнительно цена hash/equality зависит от длины key: hashing строки длины `L` не бесплатен. Если алгоритм многократно ищет один и тот же длинный ключ, запись `O(1)` без `L` скрывает работу.

В Go key должен быть comparable, порядок iteration по map не гарантирован, а concurrent mutation требует отдельной синхронизации. Эти языковые ограничения уже описаны в [[60 Go/Map|заметке о map]], [[60 Go/Равенство и comparability типов|comparability типов]] и [[60 Go/Runtime-устройство map|runtime-устройстве map]]. Здесь достаточно алгоритмического контракта.

### Плотный universe иногда отменяет hashing

Если keys лежат в `[0, U)` и `U` умеренно, direct-address array даёт worst-case `Θ(1)` без collisions. Цена — `Θ(U)` памяти и очистка/инициализация. Hash table расходует `Θ(n)` slots с capacity slack и подходит для разреженного либо неизвестного universe. Sorted array хранит `Θ(n)` элементов и даёт `Θ(log n)` lookup, зато сохраняет порядок и поддерживает range traversal.

### Состояние должно соответствовать вопросу

Для membership нужен set. Для частоты — map `key → count`. Для последней позиции — map `key → index`. Попытка хранить больше «на всякий случай» усложняет invariant и увеличивает память. Попытка хранить меньше приводит к повторному сканированию.

## Пример или трассировка

Задача: найти первый повтор в потоке `[7, 1, 4, 7, 1]` и вернуть индекс второго появления. Нужен только вопрос «видели ли значение раньше», поэтому достаточно set.

| Индекс | Значение | Set до шага | Результат lookup | Действие |
| ---: | ---: | --- | --- | --- |
| 0 | 7 | `{}` | нет | добавить `7` |
| 1 | 1 | `{7}` | нет | добавить `1` |
| 2 | 4 | `{7, 1}` | нет | добавить `4` |
| 3 | 7 | `{7, 1, 4}` | да | вернуть `3` |

Ручная трассировка делает четыре lookup и три insert. При expected `Θ(1)` на операцию время равно expected `Θ(n)`, дополнительная память — `O(n)` в случае без повторов. Worst-case hash behavior остаётся `Θ(n²)` для всей процедуры, если каждая операция деградирует до линейной.

Если значения гарантированно лежат в `[0, 9]`, boolean array длины `10` даёт тот же алгоритм с worst-case `Θ(1)` lookup и `Θ(1)` space относительно `n`. Если нужен индекс первого появления, set уже недостаточен: map должен хранить `value → firstIndex`. Выбор структуры следует из требуемого ответа.

## Trade-offs

| Представление | Сильная сторона | Цена | Ближайшая альтернатива |
| --- | --- | --- | --- |
| Array/dynamic array | индекс, compact traversal, locality | сдвиги при вставке; resize spike | linked structure для частых локальных relink |
| Hash map/set | точечные membership и association | нет порядка, collisions, capacity slack | sorted array/tree для order/range |
| Direct-address array | worst-case constant lookup | память `Θ(U)` | hash table при разреженном universe |
| Sorted array | binary search и ordered scan | вставка `Θ(n)` | balanced BST для динамических updates |

Hash table часто сокращает время ценой памяти. Sorting иногда делает обратный обмен: `O(n log n)` preprocessing позволяет пройти массив двумя pointers с `O(1)` auxiliary space. Выбор зависит от того, можно ли менять порядок и сколько запросов последует.

## Типичные ошибки

- **«Индекс строки указывает на символ» → алгоритм режет UTF-8 внутри encoding → индекс относится к byte offset → заранее выбрать bytes, code points или grapheme clusters и использовать подходящий traversal.**
- **«Map всегда `O(1)`» → adversarial либо неудачные collisions дают длинные probes/chains → потеряны assumptions hashing и load factor → назвать expected и worst-case отдельно.**
- **«Set и map взаимозаменяемы» → после нахождения повтора нельзя восстановить первую позицию или count → структура хранит меньше требуемого состояния → вывести value из формулировки результата.**
- **«Порядок map можно использовать» → тест проходит случайно и ломается на другом запуске → iteration order не входит в контракт → сортировать keys либо выбрать ordered structure.**
- **«Append всегда constant» → редкий resize нарушает per-operation latency budget → constant bound только amortized → reserve capacity или учитывать spike.**
- **«Hash key бесплатен» → длинные строки превращают “constant lookup” в заметный scan → стоимость зависит от длины key и equality checks → сохранить параметр `L` или использовать canonical compact key.**

## Когда применять

Array подходит для плотной индексной обработки, scanning и dynamic programming. Map/set берите для membership, counts, deduplication и связи объектов по ключу, когда порядок не нужен. Direct-address array стоит проверить до map, если key universe мал и известен.

Перед объявлением решения сформулируйте: единицу строки, допустимые keys, требования к order, expected/worst-case lookup, max memory и возможность менять исходную последовательность. Этого набора обычно хватает, чтобы выбор перестал быть вкусовым.
