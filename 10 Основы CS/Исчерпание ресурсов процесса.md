# Исчерпание ресурсов процесса

## TL;DR

Memory, CPU, [[10 Основы CS/Файловые дескрипторы|file descriptors]] и [[10 Основы CS/Сокеты|connections]] исчерпываются по-разному. Memory allocation может вернуть `ENOMEM` или закончиться OOM kill; CPU saturation обычно проявляется queueing latency и cgroup throttling, а не ошибкой syscall; FD дают `EMFILE` на process limit и `ENFILE` на system-wide limit; «лимит connections» распадается на SYN queue, accept queue, FD, socket memory, ephemeral ports и иногда conntrack.

Общий механизм один: конечный budget, растущий demand, очередь или reclaim, затем явный либо скрытый отказ. Поэтому одного usage gauge мало. Нужно видеть pressure, скорость роста, длину очереди и отказы на каждой границе.

Повышение лимита покупает headroom или переносит отказ дальше. Оно не добавляет CPU, downstream throughput и память на каждое соединение. Надёжный service ограничивает admission и concurrency раньше hard limit, держит bounded queues, распространяет deadlines и умеет быстро сбрасывать лишнюю нагрузку через [[40 Распределённые системы/Load shedding|load shedding]].

## Область применимости

Implementation-dependent механизмы и interfaces сверены по Linux tag `v7.1`; syscall contracts — по Linux man-pages 6.18. Cgroup рассматривается в unified hierarchy v2. Значения limits зависят от host, container runtime, system manager, network namespace и deployment policy; заметка описывает причинную модель, а не рекомендуемые числа.

## Ментальная модель

Для любого ресурса задайте пять вопросов:

```text
budget       кто и на каком уровне поставил предел?
accounting   что именно считается и кому принадлежит charge?
pressure     что происходит до hard failure: queue, reclaim, throttle?
failure      какой signal видит приложение: errno, latency, kill, timeout?
recovery     что освобождает ресурс и может ли control plane ещё работать?
```

Лимиты образуют иерархию:

```text
application budget
  <= process rlimit / connection pool
  <= cgroup budget
  <= namespace / host kernel budget
  <= physical CPU, memory, storage и network capacity
```

Эффективен самый узкий предел, но его симптом часто виден на другом слое. Например, downstream slowdown удерживает connections, те занимают FD и socket buffers, memory reclaim замедляет workers, accept queue переполняется, а клиент видит только connect timeout.

## Как устроено

### Memory: reservation, residency, reclaim и OOM

Virtual address space, resident memory и accounted cgroup memory — разные величины. `RLIMIT_AS` ограничивает размер address space: `mmap`, `brk` и `mremap` при превышении возвращают `ENOMEM`, а отказ автоматического расширения stack приводит к `SIGSEGV`. Обработать такой [[10 Основы CS/Сигналы и graceful termination|signal]] можно только на заранее настроенном alternate signal stack через `sigaltstack()`. Большой VIRT сам по себе не доказывает pressure: часть mappings не resident, часть shared или file-backed.

Linux допускает memory overcommit по policy: успешное reservation не всегда означает, что все будущие page faults обеспечены physical memory и swap. При фактическом pressure kernel reclaim clean page cache, выполняет writeback dirty pages и способен swap anonymous memory. До OOM процесс часто уже страдает от reclaim stalls и thrashing.

Cgroup v2 учитывает user memory, page cache, kernel structures и TCP socket buffers. `memory.high` создаёт throttling и heavy reclaim pressure, но сам не вызывает OOM killer; `memory.max` — hard containment boundary. Если usage нельзя уменьшить, OOM killer действует внутри cgroup. Некоторые allocations вместо OOM возвращают `ENOMEM` или отказываются иначе, поэтому отсутствие `oom_kill` не доказывает отсутствие allocation failures.

Наблюдают не один RSS. Полезны `memory.current`, breakdown в `memory.stat`, counters `memory.events` (`high`, `max`, `oom`, `oom_kill`), swap и `memory.pressure`. PSI показывает долю времени, когда tasks реально stalled на memory, и часто предупреждает о деградации раньше kill.

Меры защиты: ограничивать размер входных сообщений и caches, связывать buffers с concurrency budget, освобождать references, задавать `memory.high` как раннюю pressure boundary и оставлять reserve для control/telemetry. Без bound один медленный peer способен удержать output buffers всех connections.

### CPU: saturation и quota throttling

CPU не «заканчивается» как slot в таблице. Runnable tasks ждут scheduler, throughput упирается в доступные cycles, а latency растёт вместе с queue. Процесс может продолжать отвечать без syscall errors, но deadlines уже срываются.

Cgroup `cpu.max` задаёт quota на period для fair-class workload. После расходования quota cgroup throttled до следующего period, даже если request latency чувствительна к burst. `cpu.stat` показывает `nr_throttled` и `throttled_usec`; CPU PSI показывает время, когда runnable tasks ждали CPU. High utilization без queueing может быть здоровой работой, а заметный pressure/throttling при растущей latency — saturation.

`RLIMIT_CPU` ограничивает накопленное CPU time процесса: soft limit приводит к `SIGXCPU`, hard boundary — к `SIGKILL`. Это guardrail runaway computation, а не средство распределить service capacity по requests.

CPU overload лечат admission/concurrency limit, устранением busy loops и lock contention, профилированием hot path, fair scheduling тяжёлой работы и load shedding. Добавить unbounded worker queue значит превратить CPU shortage в memory shortage и устаревшие requests.

### File descriptors: per-process и system-wide limits

Каждый accepted socket, pipe end, открытый file, epoll/signalfd/timerfd занимает descriptor. `RLIMIT_NOFILE` задаёт значение на единицу больше максимального FD number процесса. `open`, `socket`, `pipe`, `dup` и `accept` при исчерпании возвращают `EMFILE`.

System-wide `/proc/sys/fs/file-max` ограничивает open file descriptions; достижение даёт `ENFILE`. `/proc/sys/fs/nr_open` ограничивает, насколько высоко можно поднять `RLIMIT_NOFILE`. Важно различать FD entries процесса и shared open file descriptions: `dup()` расходует ещё один FD, но ссылается на то же description.

Простой count смотрят в `/proc/<pid>/fd`, limit — в `/proc/<pid>/limits`, system state — в `/proc/sys/fs/file-nr`. Для leak нужны разбивка по type/owner и trend. Высокий стабильный count может быть нормальным pool, а медленный монотонный рост при постоянном traffic — подозрительным.

На `EMFILE` listener часто остаётся readable. Naive loop немедленно повторяет `accept()`, снова получает `EMFILE` и сжигает CPU. Один operational escape pattern держит reserve FD: закрывает его, принимает и сразу закрывает одно connection, снова открывает reserve, затем включает backoff/load shedding. Это освобождает event loop из hot spin, но не чинит leak и не создаёт capacity.

### Connections: несколько бюджетов вместо одного

Connection проходит ряд состояний, и каждое расходует свой ресурс:

| Слой | Что ограничено | Типичный симптом |
| --- | --- | --- |
| Незавершённый TCP handshake | per-listener SYN backlog, memory | retransmits, connect timeout, SYN drops |
| Установлен, но не принят | accept queue, `backlog` и `somaxconn` | timeout или refusal в зависимости от protocol/settings |
| Принят process | FD, application state, socket buffers, cgroup memory | `EMFILE`, `ENFILE`, `ENOBUFS`/`ENOMEM`, растущий RSS |
| Outbound connect | local address/ephemeral port и 4-tuple availability | `EADDRNOTAVAIL` при исчерпании automatic local ports |
| Stateful firewall/NAT, если включён | conntrack table | packet drops и failures вне process counters |

Начиная с Linux 2.2 `listen(backlog)` относится к полностью established sockets, ожидающим `accept()`. Незавершённые requests обычно ограничены `tcp_max_syn_backlog`; при включённых TCP syncookies Linux игнорирует этот параметр и не задаёт очереди logical maximum. `backlog` сверху silently capped значением `somaxconn`. Увеличить очереди полезно для краткого burst, но если process стабильно принимает медленнее arrival rate, очередь лишь увеличит время до отказа.

Accepted connection расходует минимум один FD и обычно ещё память на socket buffers, parser state, timers и TLS state. Обработка request может дополнительно удерживать downstream connection с отдельным FD. Поэтому maximum connections выводят из memory per connection, FD budget, CPU per active request и допустимого concurrency downstream, а не только из `ulimit -n`.

Outbound sockets без explicit bind получают local port из `ip_local_port_range` с учётом reserved ports и доступных tuples. `TIME_WAIT` защищает TCP от старых segments и не равен FD leak: socket state может существовать без userspace FD. Расширять port range, добавлять source IP или менять reuse policy стоит после подтверждения именно tuple/port exhaustion.

Если host использует netfilter connection tracking, `nf_conntrack_max` создаёт ещё один independent ceiling. Process может иметь свободные FD и memory, но packets уже теряются на host/network namespace layer.

### Связанные отказы и backpressure

Ресурсные аварии редко остаются в одной колонке. CPU throttle увеличивает request duration; in-flight растёт; buffers и connections живут дольше; memory pressure вызывает reclaim; latency ещё выше. Retry клиентов повышает arrival rate и замыкает feedback loop.

Защита должна стоять перед дорогой аллокацией:

1. Admission limit отклоняет новую работу до создания большого state.
2. Bounded concurrency ограничивает одновременно активные операции.
3. Bounded queue поглощает только рассчитанный burst и имеет deadline/overflow policy.
4. Backpressure сообщает upstream, что capacity занята; load shedding быстро завершает работу, которую уже нельзя выполнить в SLO.
5. Headroom оставляет возможность health checks, shutdown и telemetry даже при data-plane saturation.

[[40 Распределённые системы/Backpressure и queue buildup|Backpressure]] без bounded queue лишь перемещает bytes. Limit без overload policy превращает предсказуемый reject в случайный kernel failure.

## Пример или трассировка

Service имеет `RLIMIT_NOFILE = 8192`, принимает TCP и вызывает медленный downstream:

```text
t0  downstream latency растёт
    -> requests дольше держат inbound и outbound sockets

t1  active connections и open_fds растут
    -> socket buffers повышают memory.current
    -> workers чаще ждут, очередь requests растёт

t2  open_fds приближается к 8192
    -> accept4() возвращает EMFILE
    -> listener остаётся readable

t3  naive event loop повторяет accept4() без backoff
    -> CPU usage/pressure растёт на error loop
    -> полезный throughput падает

t4  established accept queue заполняется
    -> новые clients получают timeout/refusal
    -> retries увеличивают arrival rate
```

Простое повышение `RLIMIT_NOFILE` до 65536 отложит `t2`, но позволит удержать больше socket buffers и downstream work; failure сместится в memory, CPU или remote service.

Исправленный control path ставит concurrency cap ниже FD/memory hard limits, прекращает admission при downstream saturation, отдаёт bounded overload response, закрывает idle connections и сохраняет reserve для health/shutdown. После этого FD limit снова становится последней защитой, а не механизмом traffic management.

## Trade-offs

Жёсткий limit локализует failure и защищает соседей, но слишком низкое значение режет полезный burst. Высокий limit повышает utilization и headroom, зато увеличивает blast radius leak и объём state перед OOM. Между ними нужен ранний application threshold с наблюдаемым reject.

Queue сглаживает burst, если service rate затем догонит arrival. При устойчивой перегрузке она только добавляет latency и memory. Reject дешевле и честнее, когда deadline уже нельзя выполнить; выбор выражается SLO и retry contract клиентов.

Большой connection pool уменьшает setup overhead и даёт parallelism. Он одновременно расходует FD/ports/memory и способен перегрузить dependency. Практический размер ограничивает [[20 Бэкенд/Пулы соединений и keep-alive|пул]], а OS limits оставляют аварийным ceiling.

Swap и reclaim способны отложить OOM, сохранив throughput batch workload. Для latency-sensitive service длительный memory pressure иногда хуже быстрого controlled restart. Решение зависит от recovery model; OOM нельзя считать штатным admission controller.

## Типичные ошибки

### Смотреть только utilization

- **Неверное предположение:** пока memory ниже limit или CPU ниже 100%, overload нет.
- **Симптом:** tail latency рушится до срабатывания alert.
- **Причина:** queueing, reclaim и cgroup throttling начались раньше hard failure.
- **Исправление:** наблюдать PSI, throttle/reclaim counters, queue wait и error rate вместе с usage.

### Поднимать limit без расчёта следующего ресурса

- **Неверное предположение:** `EMFILE` лечится только большим `RLIMIT_NOFILE`.
- **Симптом:** `EMFILE` исчез, затем process OOM-killed или downstream падает сильнее.
- **Причина:** каждый новый FD удерживает memory и работу; bottleneck перенесён.
- **Исправление:** посчитать per-connection state, concurrency и downstream budget; устранить leak, оставить headroom.

### Считать connections одним счётчиком

- **Неверное предположение:** число ESTABLISHED объясняет любой connect failure.
- **Симптом:** tuning accept backlog не влияет на outbound `EADDRNOTAVAIL` или conntrack drops.
- **Причина:** исчерпан другой слой: ports/tuples, SYN queue, FD, memory или conntrack.
- **Исправление:** локализовать phase failure и сверить errno/packet state с соответствующим budget.

### Использовать unbounded queue как backpressure

- **Неверное предположение:** если request принят в память, overload обработан.
- **Симптом:** растут RSS и timeout, клиенты получают устаревшие ответы, затем OOM.
- **Причина:** очередь скрыла превышение service rate.
- **Исправление:** bounded queue, absolute deadline, admission control и явная overflow policy.

## Когда применять

Ресурсный budget нужен каждому long-running process, особенно server, worker и batch runner в cgroup/container. До production зафиксируйте soft operating threshold, hard containment limit, метрики pressure, expected errno/signal, overload response и recovery для memory, CPU, FD и каждого слоя connections. Chaos/load test должен довести один ресурс до границы и подтвердить, что control plane остаётся жив, а failure остаётся локальным.
