# Системные вызовы

## TL;DR

Системный вызов (system call, syscall) — стабильная user-space граница, через которую thread просит kernel выполнить привилегированную операцию: открыть файл, читать socket, создать process или изменить memory mapping. Library function и syscall не одно и то же. libc wrapper может подготовить arguments, выбрать один из нескольких syscalls, обработать error или вообще выполнить операцию через vDSO без входа в kernel.

Syscall сам по себе не означает context switch. Он всегда проходит границу [[10 Основы CS/User mode и kernel mode|user mode/kernel mode]], но scheduler нужен лишь при блокировке, вытеснении или другой причине reschedule.

## Область применимости

- Linux user-space ABI из Linux man-pages 6.18; kernel implementation upstream Linux 7.1.
- Для register-level примеров подразумевается Linux x86-64. Номера, registers и entry instruction различаются между architectures и ABI.
- libc: GNU C Library 2.43 там, где обсуждается поведение wrapper.
- Вне scope: написание нового syscall, compat ABI, seccomp policy language и подробности `io_uring`.

## Ментальная модель

Syscall похож на удалённый вызов привилегированного компонента, хотя физически kernel исполняется на том же CPU. ABI определяет номер операции, размещение arguments, result и error convention. Kernel считает все входные данные недоверенными, проверяет их в контексте current task и может вернуть partial progress.

Есть три слоя:

1. Application вызывает функцию языка или libc, например `read()`.
2. Wrapper реализует удобный API: приводит types к ABI, выбирает syscall, при необходимости повторяет или преобразует результат.
3. Kernel syscall handler проверяет descriptor, permissions, user memory и состояние объекта, затем выполняет работу либо ставит task в ожидание.

Не любая функция стандартной библиотеки доходит до слоя 3. `strlen()` полностью работает в user space. Некоторые time functions libc вызывает через vDSO: kernel заранее отображает небольшой ELF object и данные времени в address space process, поэтому обычный function call не виден `strace` и seccomp.

## Как устроено

### Entry ABI и dispatch

На x86-64 wrapper кладёт syscall number в `rax`, до шести arguments — в определённые ABI registers, затем выполняет `syscall`. Architecture entry code сохраняет user state и приходит в общий Linux entry path. До handler могут сработать ptrace, seccomp, audit и tracing hooks. После проверки номера kernel dispatch вызывает реализацию и записывает result в return register.

Raw Linux ABI обычно кодирует ошибку отрицательным `errno`. libc wrapper превращает её в `-1` и записывает положительный номер в thread-local `errno`. Поэтому `syscall()` из libc и произвольная assembler sequence требуют знания architecture ABI; переносимый код предпочитает обычный wrapper.

### User pointers и descriptors

Числовой file descriptor — индекс в descriptor table current process. Kernel находит ссылку на open file description и вызывает операции конкретного object: regular file, pipe, socket, device или `eventfd`. Одинаковый `read()` поэтому имеет разное blocking и offset behavior.

Buffer остаётся в user address space. Kernel копирует bytes через uaccess API либо, для специальных interfaces, временно pin'ит pages и организует shared memory. Он обязан учитывать page faults, permissions и конкурентное изменение mapping. Zero-copy API сокращает копирование payload, но не отменяет validation, ownership и completion protocol.

### Blocking, partial result и signals

Syscall вправе завершить меньше работы, чем запросил caller. `read(fd, buf, 4096)` возвращает от 1 до 4096 bytes, zero на EOF или `-1` через wrapper при error. Для stream socket граница одного `read` не совпадает с application message.

Если данных нет и descriptor blocking, handler помещает task в wait queue и вызывает scheduler. Non-blocking descriptor вместо sleep возвращает `EAGAIN`/`EWOULDBLOCK`. Signal способен прервать ожидание: когда bytes ещё не прочитаны, `read()` возвращает `EINTR`; часть syscalls libc/kernel может автоматически restart в зависимости от signal disposition и `SA_RESTART`. Application всё равно должна следовать контракту конкретной операции, а не безусловно повторять любой failed syscall: повтор non-idempotent операции после неясного partial progress опасен.

## Пример или трассировка

Пусть thread вызывает `read(pipefd, buf, 4096)`, а pipe пуст:

1. libc wrapper готовит syscall number и arguments, затем выполняет architecture entry instruction.
2. Kernel entry path устанавливает kernel execution state и запускает hooks.
3. Handler проверяет `pipefd`, право чтения и базовую допустимость user range `buf`; это ещё не гарантирует, что все его pages mapped и writable.
4. В pipe нет bytes. Для blocking descriptor task добавляется в wait queue и перестаёт быть runnable.
5. [[10 Основы CS/Планирование CPU и переключение контекста|Scheduler]] переключает CPU на другой thread.
6. Writer кладёт 100 bytes и будит waiters. Читающий task снова runnable, но ждёт решения scheduler.
7. После запуска kernel обращается к user memory и копирует 100 bytes в `buf`. На этом шаге page fault может быть штатно разрешён; недопустимый buffer даст `EFAULT`. Для корректного `buf` kernel возвращает `100`, entry path восстанавливает user context, а wrapper отдаёт caller то же значение.

Запрошено 4096 bytes, получено 100. Это успешный partial result. Если signal пришёл до записи и restart не применяется, wrapper вернёт `-1`, а `errno` будет `EINTR`.

## Trade-offs

Много мелких syscalls сокращает latency отдельной операции и упрощает streaming, но увеличивает entry/exit, validation и cache pressure. Batching (`readv`/`writev`, batched message operations, `io_uring`) амортизирует crossings, зато требует очередей, completion accounting и корректной обработки partial batch.

`mmap()` меняет явные `read`/`write` на memory accesses и page faults. Это удобно для random access и sharing, но ошибки и I/O latency теперь возникают в fault path, а durability по-прежнему требует отдельного protocol. Выбор между syscall I/O и mapping разобран дальше в заметках про [[10 Основы CS/Стек, heap и виртуальная память|virtual memory]] и [[10 Основы CS/Paging и page faults|page faults]].

Синхронный syscall API сохраняет простой control flow. Non-blocking readiness и async completion повышают concurrency одного thread, но переносят state machine и lifecycle в application или runtime.

## Типичные ошибки

**Неверное предположение:** каждая libc function — syscall с тем же именем. **Симптом:** `strace` не показывает `clock_gettime()` либо wrapper вызывает другой syscall. **Причина:** часть функций работает в user space, vDSO или адаптирует несколько ABI. **Исправление:** проверять контракт libc и actual trace на нужной architecture.

**Неверное предположение:** успешный syscall выполнил весь request. **Симптом:** усечённый frame или потеря bytes. **Причина:** `read`/`write` допускают partial result. **Исправление:** строить loop по returned count и отдельно обрабатывать EOF/error.

**Неверное предположение:** `EINTR` всегда означает «ничего не произошло, безопасно повторить». **Симптом:** duplicate effect для операции с partial progress. **Причина:** error contract различается по syscall; некоторые операции возвращают выполненную часть, другие могут скрывать сложный state. **Исправление:** читать manual page конкретного API и повторять только доказанно безопасный шаг.

**Неверное предположение:** zero-copy устраняет kernel work. **Симптом:** ожидалась нулевая CPU cost, но остаются system time и pins. **Причина:** descriptor lookup, permission checks, page ownership и completion никуда не исчезли. **Исправление:** измерять весь path, включая memory pressure и release.

## Когда применять

- По trace различайте library API, actual syscall и vDSO fast path.
- В I/O loops считайте returned bytes, а не requested bytes.
- При retry фиксируйте, известен ли partial progress и идемпотентна ли операция.
- При сравнении syscall overhead указывайте architecture, kernel, libc, security hooks и workload.
