# Файловая система и буферизация

## TL;DR

Файловая система связывает namespace имён с objects и отвечает за размещение данных и metadata. Linux VFS даёт общий интерфейс разным implementations: pathname разрешается через mount tree и dentries до inode, `open()` создаёт file object, а дальнейшие операции идут через [[10 Основы CS/Файловые дескрипторы|FD]]. Имя, открытый file object и данные на storage имеют разные lifetime.

Обычный file I/O проходит минимум через два независимых слоя buffering. `stdio` (`FILE *`) способен держать bytes в user space; после `fflush()` или raw `write()` bytes обычно попадают в kernel page cache и становятся dirty. Writeback отправляет их на backing storage позже. Поэтому успешные `fprintf`, `fflush`, `write`, `close` и `rename` дают разные гарантии.

`write()` подтверждает, сколько bytes приняло ядро, но не обещает durability. `fsync(file_fd)` синхронизирует file data и нужную metadata; создание или rename имени требует также `fsync()` directory FD. Atomic visibility через `rename()` не равна crash durability. Протокол записи обязан назвать обе гарантии отдельно.

## Область применимости

Implementation details сверены по Linux tag `v7.1`; userspace contracts — по Linux man-pages 6.18 и POSIX.1-2024. Основной scope: local POSIX-like filesystems и buffered I/O. Network filesystems, FUSE, copy-on-write filesystems, device caches и mount options способны менять latency и failure modes; для них durability protocol проверяют отдельно по документации конкретной filesystem и storage stack.

## Ментальная модель

Read path можно представить как разрешение имени, затем доступ к данным:

```text
pathname
  -> mount namespace
  -> dentry cache / directory lookup
  -> inode
  -> open file description
  -> page cache
  -> filesystem mapping
  -> block/device/storage
```

Write path добавляет несколько границ подтверждения:

```text
application objects
  -> libc stdio buffer
  -> write() syscall
  -> dirty page-cache folios
  -> filesystem writeback
  -> block layer / device cache
  -> stable storage
```

Когда вызов вернул success, нужно спросить: до какой стрелки дошли данные? Без этого слово «записано» двусмысленно.

Ещё четыре инварианта:

1. Dentry связывает имя с inode; inode хранит identity и metadata объекта, но имя находится в directory entry.
2. Несколько hard links могут указывать на один inode. Открытый file object переживает `unlink()` последнего имени до закрытия последней ссылки.
3. Page cache ускоряет reads и объединяет writes, но dirty memory требует writeback до reclaim и способна создать latency под pressure.
4. Atomicity, visibility, ordering и durability — разные свойства. Одно не следует автоматически из другого.

## Как устроено

### VFS, dentry, inode и file object

Virtual File System (VFS) выбирает mounted filesystem и вызывает её operations для `open`, `stat`, `read`, `write`, `rename` и других syscalls. Dentry cache хранит результаты pathname lookup в RAM, включая negative dentries для отсутствующих имён. Dentry указывает на inode; hard links создают несколько dentries к одному inode.

Inode описывает object: type, owner, permissions, timestamps, size и filesystem-specific mapping данных. Имя не принадлежит inode как единственное поле. Поэтому rename меняет directory entries, не меняя уже открытые FD, а unlink одного hard link не удаляет данные, пока остаётся link count или open reference.

`open()` создаёт kernel `struct file`, userspace-модель которого называется open file description. Он хранит offset и status flags и ссылается на dentry/inode. VFS получает `struct file` по FD и вызывает операции конкретной filesystem. Это и есть слой, который делает единый `read(fd, ...)` возможным для разных filesystems и типов objects.

### User-space buffering

`fopen()`/`fdopen()` создают `FILE *` с buffer и дополнительным состоянием. Output stream обычно копит bytes, чтобы реже вызывать `write()`. Terminal output часто line-buffered, обычный файл — fully buffered; режим можно изменить через `setvbuf()`.

`fflush(stream)` передаёт накопленный output underlying write function. После success bytes больше не лежат только в `stdio`, но обычно остаются в kernel buffers. Для физической durability нужен `fsync(fileno(stream))`, причём сначала `fflush()`: иначе `fsync()` не видит bytes, которые ещё не вышли из libc.

`fclose()` пытается flush и закрыть stream, поэтому его return value тоже проверяют. Смешивать raw `read`/`write` и `stdio` на одном open file description без предусмотренных POSIX synchronization points опасно: library buffer и kernel offset расходятся в представлении о позиции.

### Page cache, read и writeback

Buffered read сначала ищет file folios в page cache. Hit позволяет скопировать data без device I/O. При miss filesystem загружает folio; readahead способен заранее запросить следующие. Cached data остаются reclaimable, пока memory manager не решит освободить их.

Buffered `write()` копирует data в page-cache folios, помечает их dirty и обычно возвращается до storage write. Позже writeback сгруппирует dirty folios и вызовет filesystem/block operations. Это повышает throughput, объединяет мелкие writes и скрывает device latency. Цена — разрыв между успешным syscall и durable state, фоновые stalls при накоплении dirty memory и поздние errors.

Начиная с Linux 4.13 writeback error обещано сообщить последующему `fsync()` соответствующего file description, даже если error уже проявился в `write()`. Поэтому `fsync()` нужно проверять. Успех предыдущего `write()` и `close()` без `fsync()` не заменяет проверку.

### `fsync()`, `fdatasync()` и directory metadata

`fsync(fd)` ждёт переноса modified file data и связанной metadata к permanent storage, включая flush device cache, если stack корректно это поддерживает. `fdatasync()` может не синхронизировать metadata, которая не нужна для последующего чтения данных; изменение size, напротив, нужно сохранить.

`fsync(file_fd)` не гарантирует, что directory entry с именем файла уже durable. После создания, unlink или rename синхронизируют также directory FD. Если filesystem не поддерживает directory `fsync()` или storage лжёт о flush, верхний слой не способен вывести гарантию только из portable syscall names.

`sync()` охватывает более широкий system state и создаёт лишнюю работу; он не заменяет точечный application commit protocol. `O_SYNC`/`O_DSYNC` переносят часть ожидания в каждую write, уменьшая окно dirty data ценой latency и throughput.

### Atomic rename и crash consistency

В пределах одного mounted filesystem `rename(old, new)` атомарно заменяет существующий `new`: concurrent lookup не видит момента, когда `new` отсутствует. Это гарантия namespace visibility во время работы системы. Между mount points вызов возвращает `EXDEV`, и копирование с последующим удалением уже не атомарно.

После power loss без directory `fsync()` rename может не пережить crash, даже если file contents были синхронизированы. Обратная ошибка тоже возможна: durable имя указывает на file, data которого не были синхронизированы. Поэтому распространённый local-filesystem protocol замены состоит из записи temp file, `fsync(temp)`, `rename(temp, target)` и `fsync(parent_dir)`.

Такой protocol не обещает application transaction между несколькими files. Journal filesystem защищает свои on-disk invariants, но не угадывает требуемую бизнес-атомарность группы updates.

### Direct I/O как ближайшая альтернатива

`O_DIRECT` пытается уменьшить cache effects и копирование через page cache. Он накладывает filesystem/device-specific требования к alignment и размеру I/O, способен сделать partial inconsistent write при error и сам по себе не гарантирует durability. Смешивание direct I/O, buffered I/O и `mmap()` над overlapping region требует осторожности и часто даёт хуже производительность из-за coherence work.

Buffered I/O — разумный default: kernel умеет readahead, write coalescing и reclaim. Direct I/O оправдан, когда application владеет собственным cache/scheduler и измерение показывает двойное caching или непредсказуемый writeback как реальную проблему.

## Пример или трассировка

Нужно заменить `config` так, чтобы читатель видел старую либо целую новую версию, а подтверждённый success переживал crash на поддерживаемой local filesystem:

```text
dirfd = open(parent, O_RDONLY | O_DIRECTORY | O_CLOEXEC)
tmpfd = openat(dirfd, ".config.tmp",
               O_WRONLY | O_CREAT | O_EXCL | O_CLOEXEC, 0600)

write loop(tmpfd, complete_new_bytes)  -> обработать partial writes и errors
fsync(tmpfd)                           -> data + нужная file metadata durable
close(tmpfd)                           -> проверить diagnostic error

renameat(dirfd, ".config.tmp", dirfd, "config")
                                       -> atomic namespace replacement
fsync(dirfd)                           -> directory change durable
close(dirfd)
```

Crash boundaries дают разные outcomes:

- до `fsync(tmpfd)` temp file может быть пустым или неполным; target ещё старый;
- после `fsync(tmpfd)`, но до `renameat()` содержимое открытого temp file durable, но само temp name без `fsync` directory после crash не гарантировано;
- после `renameat()`, но до `fsync(dirfd)` текущие readers видят новую целую версию, однако crash не обязан сохранить новое directory state;
- после успешного `fsync(dirfd)` protocol считает replace committed при условии корректной filesystem и storage flush semantics.

При recovery приложение также решает судьбу orphan temp files. Если нужно сохранить owner, mode, xattrs или ACL старого target, их изменение входит в protocol до file `fsync()`; rename сам их не копирует.

## Trade-offs

User-space buffering сокращает syscalls и позволяет форматировать output, но создаёт ещё одну failure boundary и задерживает visibility. Raw I/O проще связывает return с kernel state, зато caller сам объединяет мелкие operations и ведёт partial-write loop.

Page cache даёт хорошие defaults для mixed workloads и использует свободную RAM. Application cache поверх него способен дублировать data; direct I/O убирает часть дублирования, но переносит scheduling, alignment и eviction policy в приложение.

`fsync()` на каждый record минимизирует loss window, но сериализует latency storage path. Batch/group commit амортизирует flush, ценой явно измеряемого окна возможной потери. Правильный выбор задаёт durability SLO, а не абстрактное желание «писать надёжно».

Atomic rename удобен для одного immutable replacement. In-place overwrite экономит дополнительное место, но crash способен оставить смешанное состояние. Для нескольких objects нужен journal/WAL или transactional storage protocol, а не цепочка независимых renames с надеждой на общий порядок.

## Типичные ошибки

### Приравнивать `write()` к durability

- **Неверное предположение:** syscall вернул полный count, значит data на диске.
- **Симптом:** после power loss file пуст, старый или частично восстановлен.
- **Причина:** success подтвердил копирование в kernel-managed state; writeback ещё не завершился.
- **Исправление:** определить commit point и проверять `fsync`/`fdatasync`; для имени синхронизировать directory.

### Вызывать `fsync()` до `fflush()`

- **Неверное предположение:** file descriptor знает всё, что записано через `FILE *`.
- **Симптом:** `fsync()` успешен, но последние formatted bytes теряются.
- **Причина:** они оставались в libc buffer и не дошли до kernel.
- **Исправление:** `fflush(stream)`, проверить result, затем `fsync(fileno(stream))`, затем проверить `fclose()`.

### Считать rename durable transaction

- **Неверное предположение:** atomic replacement переживает crash автоматически.
- **Симптом:** runtime readers видели новую версию, после reboot вернулась старая или исчезло имя.
- **Причина:** visibility directory update и его persistence — разные границы.
- **Исправление:** `fsync` new file до rename и parent directory после; проверить filesystem/storage contract.

### Отключать page cache без измерения

- **Неверное предположение:** cache всегда «лишняя копия».
- **Симптом:** throughput падает, alignment усложняет code, tail latency не улучшается.
- **Причина:** потеряны readahead, coalescing и общая reclaim policy.
- **Исправление:** начинать с buffered I/O; direct path вводить по профилю workload и benchmark на целевой filesystem.

## Когда применять

Эта модель нужна при проектировании config/state files, WAL, uploads, caches и любых crash-sensitive updates. Для каждой операции назовите отдельно: когда data видна другим readers, какой объём update атомарен, сколько подтверждённых данных допустимо потерять, какой syscall образует commit point и на какой filesystem/storage это проверено.
