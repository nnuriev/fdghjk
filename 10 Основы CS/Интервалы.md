# Интервалы

## TL;DR

В interval-задаче сначала определяют смысл концов, а уже потом пишут сравнения. Для полуинтервалов `[l,r)` точка `r` не занята, поэтому `[1,3)` и `[3,5)` не пересекаются; для закрытых `[l,r]` они пересекаются в `3`. От этой договорённости зависят merge, совместимость расписания и порядок событий sweep line.

После сортировки по левой границе merge сводится к одному scan с инвариантом: выведенные группы уже нельзя расширить будущими интервалами, а `current` равен объединению последней ещё не завершённой группы. Sweep line использует другую проекцию той же идеи: превращает границы в упорядоченные события и поддерживает состояние активных интервалов.

Для максимального числа непересекающихся интервалов без весов оптимален greedy по раннему времени окончания. Веса меняют задачу: earliest-finish больше не гарантирует optimum, а weighted interval scheduling решается dynamic programming по последнему совместимому интервалу.

## Ментальная модель

Интервал — множество точек плюс договорённость о границах. Пара чисел сама по себе недостаточна. Сравнение `next.start <= current.end` может быть правильным для закрытых интервалов, неправильным для overlap полуинтервалов и снова правильным для полуинтервалов, если задача просит склеивать соседние участки без зазора.

Сортировка превращает глобальную геометрию в локальный переход. Когда интервалы идут по неубывающему `start`, новый интервал уже не сможет вернуться и пройти выбранный merge predicate с группой, которую scan закрыл раньше. Sweep line делает то же с границами: между соседними координатами множество активных интервалов не меняется.

Отсюда рабочая цепочка: выбрать семантику → нормализовать вход → определить invariant или event order → только затем выбрать merge, sweep, greedy либо DP.

## Как устроено

### Семантика концов

Для полуинтервала `[l,r)`:

- допустимый контракт обычно `l <= r`;
- длина на числовой оси равна `r-l`;
- `[i,i)` пуст;
- `[a,b)` и `[b,c)` не пересекаются;
- точка `x` принадлежит интервалу при `l <= x < r`.

Для закрытого `[l,r]` обе границы входят в множество, поэтому `[a,b]` и `[b,c]` имеют общую точку. В дискретной целочисленной области длина равна `r-l+1`. Закрытый `[l,r]` иногда переводят в `[l,r+1)`, но такой переход требует существования следующего значения и ломается на `r`, равном максимуму integer type. Безопаснее сохранить исходную семантику, если верхняя граница не допускает корректного successor.

Нужно отдельно решить судьбу `l > r`, пустых интервалов, дубликатов и интервалов с бесконечной границей. Пустой `[i,i)` не влияет на покрытие; его обычно отбрасывают до merge и sweep. Это часть контракта, а не cleanup после алгоритма.

### Merge как scan с инвариантом

Сортируем интервалы по `(start,end)`. Первый становится `current`. Для каждого `next`:

1. если он пересекает `current` по выбранной семантике, заменяем конец на `max(current.end, next.end)`;
2. иначе добавляем `current` в ответ и начинаем новую группу с `next`;
3. после scan добавляем последний `current`.

Для overlap полуинтервалов условие равно `next.start < current.end`. Для закрытых интервалов — `next.start <= current.end`. Если нужен канонический union без разрыва, соседние полуинтервалы разрешено склеивать по `<=`, но это уже отдельная policy: они соприкасаются границами, хотя их пересечение пусто.

Инвариант после обработки первых `k` интервалов:

- `output` вместе с `current` покрывает ровно их union;
- `output` содержит все завершённые группы по выбранному merge predicate в отсортированном порядке;
- `current` покрывает последнюю, ещё открытую группу;
- ни один будущий интервал не пройдёт merge predicate с завершённой группой.

Последний пункт следует из сортировки и монотонности predicate по `start`. Если `next` уже нельзя присоединить к `current`, start любого будущего интервала не меньше `next.start`, поэтому он тоже не присоединится. Это связывает реализацию с [[10 Основы CS/Доказательство корректности алгоритма|доказательством корректности]], а не оставляет фразу «сортируем и сливаем» без основания.

Сортировка стоит `O(n log n)`, scan — `O(n)`, ответ — до `O(n)` памяти. Если вход уже отсортирован по start, остаётся линейная часть.

### Sweep line и события на одной координате

Sweep line кодирует начало и конец как events, сортирует их по координате и поддерживает active set, counter или другую структуру. Между двумя соседними координатами событий состояние постоянно.

Для полуинтервалов `[l,r)` событие конца на координате `x` нужно обработать раньше события начала на том же `x`, если maximum active count измеряется после каждого event. Иначе `[1,3)` и `[3,5)` на мгновение будут ошибочно считаться одновременно активными. Для закрытых интервалов порядок обратный: start раньше end, потому что оба интервала содержат общую точку.

Пустые полуинтервалы нужно заранее удалить или агрегировать их `+1` и `-1` на одной координате. Если отдельно обработать end пустого `[x,x)` раньше start, промежуточный counter станет отрицательным, хотя интервал не покрывает ни одной точки.

У полуинтервалов есть ещё более устойчивый вариант: агрегировать на каждой координате все `+1` для starts и `-1` для ends, применить итоговую дельту, а получившийся counter считать активным на сегменте до следующей координаты. Это та же граница-событие, что и в [[10 Основы CS/Префиксные суммы и difference arrays|difference array]], только координаты часто приходится сортировать или сжимать.

Если задача требует перечислять пересечения, одного counter мало: active set должен хранить сами объекты и поддерживать нужный range search. В этом месте sweep line перестаёт быть «одним циклом» и наследует стоимость выбранной структуры.

### Scheduling: unweighted greedy и weighted DP

В unweighted interval scheduling нужно выбрать максимальное число попарно совместимых интервалов. Сортируем по `end` и каждый раз берём первый интервал, совместимый с уже выбранным. Для `[l,r)` совместимость с последним выбранным означает `next.start >= last.end`.

Exchange argument короткий. Пусть greedy первым выбрал интервал `g` с минимальным end, а некоторое оптимальное расписание начинается с `o`. Замена `o` на `g` не уменьшает число оставшихся возможностей, потому что `g.end <= o.end`. После замены остаётся подзадача того же вида справа от `g`; индукция повторяет аргумент.

С весами этот обмен не сохраняет ценность. Интервалы `[0,2)` веса `2` и `[2,4)` веса `2` дают greedy-сумму `4`, но один `[0,4)` веса `5` лучше. После сортировки по end определим `p(i)` как последний интервал, совместимый с `i`. Тогда

```text
OPT(i) = max(OPT(i-1), weight(i) + OPT(p(i))),  OPT(-1) = 0.
```

Первый вариант пропускает `i`, второй берёт его и оставляет только совместимый prefix. `p(i)` находят binary search, поэтому сортировка, предвычисление переходов и DP занимают `O(n log n)` времени; таблица и восстановление ответа — `O(n)` памяти.

## Пример или трассировка

Даны полуинтервалы в произвольном порядке:

```text
[5,7), [1,4), [3,6), [8,10), [10,12)
```

После сортировки: `[1,4), [3,6), [5,7), [8,10), [10,12)`.

| `current` до шага | `next` | Проверка overlap `<` | Результат шага |
| --- | --- | --- | --- |
| `[1,4)` | `[3,6)` | `3 < 4` | расширить до `[1,6)` |
| `[1,6)` | `[5,7)` | `5 < 6` | расширить до `[1,7)` |
| `[1,7)` | `[8,10)` | `8 < 7` ложно | вывести `[1,7)`, открыть `[8,10)` |
| `[8,10)` | `[10,12)` | `10 < 10` ложно | вывести `[8,10)`, открыть `[10,12)` |

После scan выводим последний интервал. Для merge только пересечений ответ равен `[1,7), [8,10), [10,12)`. Если контракт требует склеивать и соседнее покрытие, последняя проверка использует `<=`, ответ становится `[1,7), [8,12)`. Ручная проверка по точкам подтверждает оба результата: в координате `10` первый полуинтервал уже закончился, второй только начался.

## Trade-offs

Merge нужен, когда результатом служит union или нормализованный список покрытых участков. Он прост и после сортировки линеен, но теряет информацию о том, сколько исходных интервалов покрывало точку.

Sweep line сохраняет динамику покрытия и решает maximum overlap, intersections, room allocation и задачи с геометрическими events. Цена — строгий event ordering и active structure. Для одного счётчика достаточно сортировки `2n` событий; для поиска пересекающихся объектов могут понадобиться balanced tree, heap или range-search structure.

Earliest-finish greedy оптимизирует число интервалов только в unweighted модели. Weighted scheduling требует DP; выбор максимального веса, минимальной длины или самого раннего start не даёт общего proof. Это хороший пример границы greedy: локальный выбор безопасен ровно пока exchange не ухудшает objective.

Полуинтервалы обычно упрощают длины, пустые диапазоны и стыковку соседних сегментов. Закрытые интервалы естественнее, когда endpoint сам представляет занятый момент или допустимое значение. Выбирать нужно по предметной семантике, а не по привычному знаку сравнения.

Сортировка in place экономит `O(n)` памяти, но меняет порядок входа. Если исходные indices входят в ответ либо вход менять нельзя, храните indices в интервалах или сортируйте копию.

## Типичные ошибки

**Неверное предположение:** слово «пересекаются» однозначно задаёт сравнение. **Симптом:** `[1,3)` и `[3,5)` ошибочно сливаются или закрытые `[1,3]` и `[3,5]` ошибочно разделяются. **Причина:** не зафиксирована семантика концов и policy для touching intervals. **Исправление:** записать membership predicate и один граничный пример до кода.

**Неверное предположение:** порядок событий с одинаковой координатой не влияет на ответ. **Симптом:** maximum concurrency завышен на стыках либо теряется пересечение закрытых интервалов. **Причина:** промежуточное active state соответствует другой семантике. **Исправление:** для `[l,r)` обрабатывать end раньше start или агрегировать дельты; для closed intervals, где общая точка считается занятой, start раньше end.

**Неверное предположение:** сортировки по start достаточно для любого scheduling greedy. **Симптом:** длинный ранний интервал блокирует несколько коротких и ответ содержит меньше работ. **Причина:** early start ничего не говорит о числе оставшихся возможностей. **Исправление:** в unweighted задаче сортировать по end и дать exchange proof.

**Неверное предположение:** earliest-finish остаётся оптимальным после добавления weights. **Симптом:** выбранных интервалов больше, но их суммарный weight меньше optimum. **Причина:** exchange сохраняет совместимость, но не сохраняет objective. **Исправление:** перейти к recurrence по `p(i)` и восстановить выбранный набор из DP.

**Неверное предположение:** closed integer interval всегда безопасно преобразовать в `[l,r+1)`. **Симптом:** правая граница переполняется и становится минимальным числом типа. **Причина:** у максимального integer нет представимого successor. **Исправление:** проверить границу до преобразования, использовать более широкий тип либо оставить closed representation и явные сравнения; общий разбор см. в [[10 Основы CS/Edge cases, невалидный ввод и overflow|заметке про overflow]].

## Когда применять

- Для union и merge сортируйте по start и сформулируйте overlap/coalescing predicate.
- Для maximum overlap или обработки точечных событий используйте sweep line и заранее задайте tie-break.
- Для максимального количества совместимых интервалов без весов берите earliest finish и докажите exchange.
- Для weighted scheduling сортируйте по end, найдите `p(i)` и используйте DP.
- Перед завершением проверьте пустой вход, один интервал, одинаковые границы, вложенные и дублирующиеся интервалы, touching endpoints, полностью раздельные интервалы, invalid `l > r` и максимальное значение координаты.
