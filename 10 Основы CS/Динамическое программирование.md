# Динамическое программирование

## TL;DR

Вопрос заметки: как вывести dynamic programming из пространства решений так, чтобы 1D, 2D, subsequence и knapsack-задачи оставались одной моделью, а не набором формул?

DP вычисляет directed acyclic graph подзадач один раз на state. Надёжный вывод идёт в пяти шагах: определить смысл state, записать transitions, задать base cases, выбрать topological evaluation order и извлечь ответ. Complexity равна числу достижимых states, умноженному на число transitions из state; размер таблицы сам по себе ещё не даёт полной оценки.

1D и 2D описывают число координат state, а не разные алгоритмы. Subsequence DP хранит позицию или пару префиксов. Knapsack добавляет capacity. Для 0/1 knapsack одномерная таблица обновляется по capacity вниз, чтобы item использовался один раз; проход вверх читает уже обновлённое значение той же строки и тем самым реализует unbounded knapsack.

`O(nW)` для knapsack pseudo-polynomial: `W` — числовое значение capacity, тогда как длина его binary encoding равна `Θ(log W)`.

## Ментальная модель

Начните с дерева всех решений, как в [[10 Основы CS/Рекурсия и backtracking|backtracking]]. Если разные paths приходят к одинаковому вопросу о будущем, эти nodes можно склеить в один state. Получается DAG подзадач.

State — минимальная достаточная summary прошлого. Два partial paths можно склеить, только если множество допустимых продолжений и их вклад в objective совпадают. Например, для 0/1 knapsack после первых `i` items будущее определяется `i` и оставшейся capacity; конкретный порядок прошлых выборов уже не нужен.

Transition перечисляет последний либо следующий выбор. Его корректность требует двух частей:

1. полнота: любой допустимый ответ попадает хотя бы в одну ветвь recurrence;
2. безопасность: каждая ветвь строит допустимый answer из корректных smaller states.

Evaluation order — topological order state DAG. Top-down recursion получает его по demand и memoizes visited states. Bottom-up явно располагает loops так, чтобы dependencies были готовы до чтения.

## Как устроено

### Пять решений до кода

1. **State.** Одно предложение: что означает `dp[...]`.
2. **Transition.** Какие взаимоисключающие первые/последние решения покрывают все случаи.
3. **Base.** Самые маленькие prefixes, пустая последовательность, нулевая capacity.
4. **Order.** Почему каждое dependency уже вычислено.
5. **Answer.** В какой cell лежит objective; нужны ли parent pointers для witness.

Эта дисциплина продолжает [[10 Основы CS/От brute force к оптимальному решению|переход от brute force к оптимальному решению]]: recurrence фиксирует корректный exhaustive baseline, memoization удаляет повторную работу, а bottom-up меняет порядок исполнения без изменения смысла state.

### 1D DP

Одна координата обычно обозначает prefix length, position, amount или capacity. Пример: `dp[i]` — лучший score на первых `i` элементах. Transition читает ограниченное число более ранних states, например `dp[i-1]` и `dp[i-2]`.

Rolling storage допустим, если future использует лишь фиксированный dependency frontier. Это экономия памяти, а не времени. Если нужно восстановить сами решения или transitions читают далёкую историю, одной пары переменных недостаточно.

### 2D и subsequences

Для двух sequences естественный state — пара prefixes. Longest common subsequence использует `dp[i][j]`, длину LCS для первых `i` и `j` элементов:

```text
если x[i-1] == y[j-1]: 1 + dp[i-1][j-1]
иначе: max(dp[i-1][j], dp[i][j-1])
```

Это 2D не из-за таблицы на бумаге, а потому что future зависит от двух независимо движущихся границ.

Longest increasing subsequence можно описать 1D state: `dp[i]` — длина лучшей increasing subsequence, заканчивающейся ровно в `i`. Тогда просматриваются все `j<i` с `a[j]<a[i]`, время `Θ(n²)`. Алгоритм с tails и binary search достигает `O(n log n)`, но использует другой invariant; это не механическая «оптимизация DP-таблицы».

### Knapsack и направление обновления

2D recurrence 0/1 knapsack:

```text
dp[i][c] = max(
    dp[i-1][c],
    dp[i-1][c-weight[i]] + value[i]
)
```

Обе ветви читают предыдущую строку, поэтому item `i` нельзя использовать повторно. При сжатии в 1D capacity перебирают вниз: `dp[c-weight]` ещё относится к предыдущей строке. Для unbounded knapsack capacity перебирают вверх, и текущее item разрешено читать повторно из уже обновлённого state.

`O(nW)` states и `O(1)` transition дают `O(nW)` времени. Это pseudo-polynomial относительно bit-length input. При `W=2^k` таблица экспоненциальна по `k`, хотя запись `W` занимает `k+1` bits.

### Memoization, tabulation и reconstruction

Top-down вычисляет только reachable states и часто проще следует recurrence. Цена — recursion overhead, memo lookup и stack depth. Bottom-up предсказуемо проходит таблицу, лучше использует locality и упрощает rolling memory, но может вычислить unreachable cells.

Если нужен witness, одного objective value мало. Хранят choice/parent либо повторно проходят полную таблицу и проверяют, какой transition дал value. Aggressive space compression может уничтожить информацию для reconstruction; это часть trade-off, а не бесплатная оптимизация.

## Пример или трассировка

0/1 knapsack: items `(weight,value)` равны `(2,3)`, `(3,4)`, `(4,5)`, capacity `5`. `dp[i][c]` — максимальная value из первых `i` items при capacity не больше `c`.

| Рассмотренные items | `c=0` | `1` | `2` | `3` | `4` | `5` |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| ни одного | 0 | 0 | 0 | 0 | 0 | 0 |
| `(2,3)` | 0 | 0 | 3 | 3 | 3 | 3 |
| `(2,3),(3,4)` | 0 | 0 | 3 | 4 | 4 | 7 |
| все три | 0 | 0 | 3 | 4 | 5 | 7 |

Итог `7`: items весов `2+3`. Для последнего item веса `4` cell `dp[3][5]` сравнивает skip=`7` и take=`dp[2][1]+5=5`, поэтому сохраняет `7`.

Сжатая 1D строка после первого item равна `[0,0,3,3,3,3]`. Если ошибочно идти по capacity вверх, при `c=4` transition прочитает уже обновлённое `dp[2]=3` и запишет `6`, то есть использует item веса `2` дважды. Проход вниз читает старую строку и сохраняет 0/1 semantics. Ручная трассировка показывает причинный механизм loop direction, а не правило для запоминания.

## Trade-offs

DP выигрывает у plain backtracking, когда paths сильно сходятся в одинаковые states. Если почти каждый path имеет уникальный state, memo table повторяет экспоненциальное дерево и добавляет overhead. Safe pruning может быть практичнее на разреженном search space.

[[10 Основы CS/Жадные алгоритмы|Greedy algorithm]] хранит один partial choice и обычно быстрее, но требует safe-choice proof. DP сохраняет несколько конкурирующих summaries до тех пор, пока state не позволяет их сравнить. Это дороже, зато выдерживает зависимости, для которых exchange не работает.

2D table проще проверять и восстанавливать. 1D compression уменьшает auxiliary memory с `O(nW)` до `O(W)`, но делает evaluation order частью correctness и усложняет reconstruction. Top-down лучше пропускает unreachable states; bottom-up часто выигрывает locality и отсутствие глубокой рекурсии.

## Типичные ошибки

- **«State — это набор параметров исходной функции» → memo table остаётся экспоненциальной или склеивает разные futures → не доказана достаточность и минимальность summary → определить эквивалентность подзадач через допустимые продолжения.**
- **«Recurrence выглядит знакомо» → пропущен класс решений или один item учитывается дважды → transitions не образуют полное разбиение → вывести их из first/last choice и доказать completeness/safety.**
- **«После сжатия loops можно оставить как есть» → 0/1 knapsack превращается в unbounded → transition читает current row вместо previous → для 0/1 идти по capacity вниз, для unbounded вверх.**
- **«`O(nW)` polynomial по input size» → algorithm не укладывается при большом binary-encoded `W` → числовое значение спутано с длиной encoding → назвать bound pseudo-polynomial и проверить constraints.**
- **«Rolling array всегда лучше» → невозможно восстановить subsequence/items → удалены parent/previous rows, нужные witness → хранить choices, полную таблицу или планировать recomputation.**
- **«Memoization автоматически завершит recursion» → цикл состояний вызывает незавершённый повторный вход → state graph не DAG либо отсутствует progress → доказать acyclicity/evaluation order.**

## Когда применять

- Есть choices, одинаковые подзадачи достигаются разными путями, а answer агрегирует optimum/count/feasibility.
- State имеет умеренный product dimensions и transition можно вычислить в пределах constraints.
- Для sequences начинайте с prefixes/end-position; для capacities — с `item index + capacity`; затем доказывайте, можно ли удалить координату или сжать слой.
- До объявления complexity посчитайте reachable states, transitions на state, output/reconstruction memory и numeric magnitude dimensions.
