# Оценка временной и пространственной сложности

## TL;DR

Вопрос заметки: как сформулировать проверяемую оценку ресурсов алгоритма и не смешать worst-case, average-case и amortized analysis?

Сначала фиксируют размер входа и модель допустимых элементарных операций. Затем считают время и пиковую дополнительную память как функции этого размера. Worst-case берёт максимум по входам одного размера, average-case требует явно заданного распределения входов, а amortized analysis распределяет полную стоимость последовательности операций между её элементами. Вероятность для amortized-оценки не нужна.

`O`, `Ω` и `Θ` отвечают на другой вопрос: это асимптотическая верхняя, нижняя и точная границы уже выбранной функции стоимости. Поэтому `O(n)` не означает «worst-case», а `Ω(n)` не означает «best-case».

## Ментальная модель

Оценка сложности похожа на контракт с четырьмя полями:

```text
ресурс + размер входа + модель вычислений + квантор по сценариям
```

Например: «для массива из `n` элементов binary search делает `Θ(log n)` сравнений в worst-case в модели, где индексный доступ и сравнение занимают `Θ(1)`». Если убрать хотя бы одно поле, утверждение становится двусмысленным. Для графа одного `n` обычно мало: естественный размер равен `|V| + |E|`. Для строки нужно уточнить, считаются bytes, code points или пользовательские grapheme clusters; семантика Go разобрана в [[60 Go/Строки, байты, руны и UTF-8|заметке о строках, байтах и рунах]].

Асимптотика отбрасывает постоянные множители и младшие члены, но не отменяет реальную стоимость. Два алгоритма с `Θ(n)` могут сильно различаться по locality, allocations и числу проходов. Сначала доказывают класс роста, затем измеряют константы на репрезентативном входе.

## Как устроено

### Сначала выбрать функцию стоимости

Пусть `T(x)` — число элементарных шагов на конкретном входе `x`, а `|x| = n`. Тогда:

- worst-case: `T_worst(n) = max(T(x) : |x| = n)`;
- average-case: `T_avg(n) = E[T(X) \mid |X|=n]`, где распределение случайного входа `X` задано явно;
- best-case: минимум по входам размера `n`; обычно он мало говорит о гарантии для вызывающего кода.

Ожидаемое время randomized algorithm тоже записывают через математическое ожидание, но случайность принадлежит алгоритму, входу либо обоим. Фраза «в среднем `O(n)`» без модели вероятности ничего не доказывает.

Amortized analysis начинает с последовательности `op₁, …, opₖ`. Если её фактическая стоимость не превосходит `F(k)`, amortized cost одной операции равна `F(k)/k`. Отдельная операция всё ещё может быть дорогой. Распространённые способы доказательства: aggregate method считает всю последовательность, accounting method выдаёт дешёвым операциям credit, potential method хранит накопленную «предоплату» в функции состояния.

### `O`, `Ω` и `Θ` не кодируют сценарий

`f(n) ∈ O(g(n))`, если после некоторого `n₀` существует константа `c`, для которой `f(n) ≤ c·g(n)`. `Ω` задаёт симметричную нижнюю границу, `Θ` требует обе границы сразу. Это множества функций, а не синонимы «плохого», «хорошего» и «обычного» случая.

Точная формулировка выглядит так: «worst-case time равно `Θ(n log n)`». Более слабое `O(n²)` для той же функции может быть истинно, но почти бесполезно. На собеседовании лучше называть tight bound и отдельно пояснять, какой случай анализируется.

### Пространство считают по пиковому живому состоянию

Нужно различать:

- total space: вход, output и рабочая память;
- auxiliary space: память сверх входа и требуемого результата;
- stack space рекурсии: глубина одновременно активных вызовов, а не их общее число;
- пиковую память: максимум живых объектов в один момент, а не сумму всех allocations за выполнение.

Алгоритм, который возвращает массив из `n` элементов, обязан потратить `Ω(n)` output space. При этом его auxiliary space может быть `O(1)`. Маркировка «in-place» тоже требует оговорки: рекурсивный partition может менять массив на месте, но удерживать `Θ(log n)` stack frames на сбалансированном входе и `Θ(n)` в худшем случае.

### Несколько параметров сохраняют причину стоимости

Запись `O(n²)` иногда прячет полезную структуру. Traversal adjacency list занимает `Θ(|V| + |E|)`, а не абстрактное `O(n²)`. Для batch из `q` запросов после preprocessing полезнее `O(n log n + q log n)`, чем попытка свести всё к одному `n`. Сохраняйте параметры, пока они влияют на выбор решения.

## Пример или трассировка

Динамический массив начинает с capacity `1` и удваивает её при переполнении. Считаем одну запись нового элемента за `1`, а перенос одного старого элемента при resize ещё за `1`.

| Append | Capacity после операции | Запись нового | Скопировано старых | Стоимость операции |
| ---: | ---: | ---: | ---: | ---: |
| 1 | 1 | 1 | 0 | 1 |
| 2 | 2 | 1 | 1 | 2 |
| 3 | 4 | 1 | 2 | 3 |
| 4 | 4 | 1 | 0 | 1 |
| 5 | 8 | 1 | 4 | 5 |
| 6 | 8 | 1 | 0 | 1 |
| 7 | 8 | 1 | 0 | 1 |
| 8 | 8 | 1 | 0 | 1 |

Ручная сумма равна `15`: восемь записей новых элементов и `1 + 2 + 4 = 7` копирований. Один пятый append стоит `Θ(n)` относительно текущего размера, но восемь append вместе стоят меньше `2n`. В общем случае geometric series `1 + 2 + 4 + … < n`, поэтому `n` append занимают `Θ(n)` времени, а один append имеет `Θ(1)` amortized time.

Это не average-case: входы и вероятности не выбирались. Гарантия относится к любой последовательности append при описанной политике роста. В Go конкретная стратегия роста capacity не входит в language contract, поэтому алгоритмический вывод нельзя превращать в обещание точных capacity; детали aliasing и `append` находятся в [[60 Go/Массивы и слайсы|заметке о массивах и слайсах]].

## Trade-offs

Worst-case bound нужен для latency budget, hard deadline и adversarial input. Он часто консервативен. Expected/average bound точнее описывает randomized algorithm или известный workload, зато целиком зависит от модели вероятности. Amortized bound силён для stateful data structure: он не предполагает «типичный» вход, но разрешает редкие дорогие операции и потому не гарантирует малую latency каждой из них.

Асимптотика хорошо отсекает решения, которые перестают масштабироваться. Benchmark отвечает на соседний вопрос: какая реализация быстрее на выбранной машине и распределении данных. Одно другим не заменяют.

## Типичные ошибки

- **«`O` означает worst-case» → оценка звучит точнее, чем доказано → квантор по входам смешан с видом асимптотической границы → отдельно назвать case и затем `O/Ω/Θ`.**
- **«Amortized значит average» → в доказательство незаметно попадает “типичный” workload → amortized bound относится к полной последовательности без распределения → посчитать aggregate cost либо явно задать potential.**
- **«Два вложенных цикла всегда дают `O(n²)`» → monotonic pointer algorithm переоценивается → внутренний индекс за всё выполнение движется не более `n` раз → считать общее число движений, а не форму отступов.**
- **«Память равна числу allocations» → последовательные временные buffers складываются → space complexity измеряет peak live state → проследить lifetime и максимум одновременно живых объектов.**
- **«`n` очевидно» → оценка графа или Unicode-строки меняет смысл между участниками → размер входа не определён → назвать параметры и единицы до анализа.**
- **«У hash lookup `O(1)` без оговорок» → adversarial collisions превращают операцию в линейную → expected/amortized предпосылки потеряны → назвать модель hashing, load factor и worst-case отдельно.**

## Когда применять

Оценку фиксируют до выбора оптимизации и повторяют после изменения представления данных. На интервью проговорите четыре вещи: размер входа, основной инвариант, время по требуемому case и auxiliary space. Если решение stateful либо randomized, сразу уточните amortized или expected характер гарантии.

В production добавьте пределы входа, реальные allocations, cache behavior и latency distribution. Асимптотика говорит, как растёт цена; capacity planning требует ещё констант и профиля нагрузки.
