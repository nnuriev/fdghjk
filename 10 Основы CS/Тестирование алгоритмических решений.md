# Тестирование алгоритмических решений

## TL;DR

Решение не завершено после последней строки алгоритма. Сначала из контракта выводят test oracle и классы входов, затем проверяют normal path, границы, отсутствие ответа, дубликаты и numeric limits. Один тест должен различать корректный алгоритм и наиболее вероятную ошибочную версию, а не повторять пример из условия.

Сильная схема для interview-задачи: точные example tests, boundary table, property checks и differential comparison с brute force на малых входах. Fuzzing расширяет выборку, но не заменяет oracle; benchmark не доказывает корректность.

## Ментальная модель

Тест — это тройка:

```text
вход + ожидаемое свойство + причина, почему этот вход отличает ошибку
```

Oracle может быть точным значением, множеством допустимых результатов, invariant или медленной эталонной реализацией. Если задача разрешает любой из нескольких ответов, сравнивать с одной заранее выбранной парой неверно: нужно проверить допустимость индексов и postcondition.

Тестовые классы строятся вокруг мест, где меняется управление: пустой/непустой, найдено/не найдено, `<`/`==`/`>`, новая вершина/visited, capacity хватает/не хватает.

## Как устроено

### 1. Зафиксировать oracle

Для lower bound oracle — первая позиция `i`, где `a[i] >= x`, либо `len(a)`. Для shortest path — валидный path и его минимальная стоимость, а не конкретная последовательность, если optimum не единственен. Для in-place алгоритма проверяются и return value, и mutation входа.

### 2. Разбить входы на классы

Минимальный набор обычно включает:

- smallest valid input и первый размер, на котором появляется основная ветвь;
- ответ в начале, середине и конце;
- отсутствие ответа;
- duplicates/all equal;
- уже упорядоченный и обратный порядок, если алгоритм зависит от порядка;
- maximum/minimum values, если есть арифметика;
- malformed input, только если он входит в контракт.

### 3. Добавить свойства

Property полезна, когда exact output дорог или не единственен. После сортировки последовательность не убывает и multiset элементов сохранён. Для reverse выполняется `reverse(reverse(x)) == x`. Для union-find connectivity симметрична и транзитивна. Property должна быть независимее реализации, иначе тест повторит ту же ошибку.

Практика table-driven tests раскрыта в [[60 Go/Table-driven tests в Go|отдельной Go-заметке]], а генерация входов и сохранение найденного regression case — в [[60 Go/Fuzzing|fuzzing]]. Property-based подход и его ограничения отдельно разобраны в [[20 Бэкенд/Property-based testing|property-based testing]].

### 4. Сравнить с oracle

Для `n <= 8` часто можно перебрать все либо много случайных входов и сравнить оптимизированную функцию с очевидным `O(2^n)`, `O(n²)` или `O(n³)` baseline. Это ловит ошибки optimization-specific invariants, не заставляя вручную вычислять тысячи ответов.

### 5. Проверить сложность отдельно

Unit test не подтверждает `O(n log n)`. Для этого нужны анализ и, при необходимости, измерение роста на контролируемых размерах. Timeout-тест хрупок: нагрузка машины меняет wall time. Лучше считать ключевые операции в instrumented версии или проводить benchmark после доказательства асимптотики.

## Пример или трассировка

Ниже self-contained harness для lower bound. Он использует те же cases, которые затем переносятся в `testing` package. Код рассчитан на Go 1.26.

```go
package main

import "fmt"

func lowerBound(a []int, x int) int {
	lo, hi := 0, len(a)
	for lo < hi {
		mid := lo + (hi-lo)/2
		if a[mid] < x {
			lo = mid + 1
		} else {
			hi = mid
		}
	}
	return lo
}

func main() {
	tests := []struct {
		a    []int
		x    int
		want int
	}{
		{nil, 3, 0},
		{[]int{5}, 3, 0},
		{[]int{5}, 5, 0},
		{[]int{5}, 7, 1},
		{[]int{2, 2, 2}, 2, 0},
		{[]int{1, 4}, 3, 1},
		{[]int{1, 2, 2, 5}, 6, 4},
	}

	for i, tc := range tests {
		if got := lowerBound(tc.a, tc.x); got != tc.want {
			panic(fmt.Sprintf("case %d: got %d, want %d", i, got, tc.want))
		}
	}
	fmt.Printf("%d tests passed\n", len(tests))
}
```

Ожидаемый результат:

```text
7 tests passed
```

Локальная Go toolchain недоступна; пример проверен статически, но компиляция и наблюдаемый вывод пока не подтверждены. Поэтому статус заметки остаётся `черновик`.

Набор различает как минимум четыре дефекта: индексирование empty slice, неправильный insertion point справа, возврат последнего duplicate и off-by-one на `hi`.

## Trade-offs

Example tests легко читать и точно локализуют regression, но охватывают только выбранные точки. Fuzzing исследует больше входов, зато требует дешёвого oracle/property и воспроизводимого target. Differential test даёт сильный oracle на малых размерах, но разделяемая ошибка в двух реализациях всё ещё возможна.

На интервью время ограничено. Лучше прогнать четыре выбранных класса вслух и полностью протрассировать один, чем назвать двадцать tests и не проверить ни одного.

## Типичные ошибки

**Неверное предположение:** happy path достаточен. **Симптом:** пример из условия проходит, empty input вызывает panic. **Причина:** test set не пересекает branch boundaries. **Исправление:** строить cases из условий `if`, границ циклов и preconditions.

**Неверное предположение:** expected output обязан быть единственным. **Симптом:** корректный shortest path или пара индексов отклоняется. **Причина:** тест сравнивает representation, а не postcondition. **Исправление:** валидировать результат и objective value.

**Неверное предположение:** property сама по себе верна. **Симптом:** faulty sort проходит проверку «длина сохранилась». **Причина:** свойство слишком слабое. **Исправление:** комбинировать order, multiset preservation и exact examples.

**Неверное предположение:** fuzzing докажет отсутствие ошибок. **Симптом:** target долго работает без failure, но систематически пропускает класс входов. **Причина:** генератор и oracle не покрывают нужную семантику. **Исправление:** добавить seed corpus из boundary matrix и сохранить каждый найденный counterexample как regression.

## Когда применять

- До кода выпишите хотя бы normal, empty/minimal, boundary и no-solution cases.
- После baseline используйте его как oracle для оптимизированной версии на малых входах.
- Перед фразой «готово» вручную обновляйте state теми же шагами, что выполняет код.
- Для Go переносите матрицу в table-driven test; fuzzing добавляйте к свойствам с дешёвой проверкой.
