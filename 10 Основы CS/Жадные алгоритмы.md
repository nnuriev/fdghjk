# Жадные алгоритмы

## TL;DR

Вопрос заметки: когда локально лучший выбор можно зафиксировать навсегда и каким доказательством отделить алгоритм от правдоподобной эвристики?

Greedy algorithm строит решение по одному необратимому выбору и не пересматривает прошлое. Быстрота следует из отказа исследовать альтернативные ветви. Цена этого отказа — proof obligation: нужно показать, что существует optimal solution, совместимое с greedy choice, а остаток задачи имеет тот же контракт.

Для interval scheduling earliest-finish rule доказывается exchange argument. Для spanning-tree algorithms аналогичную безопасность даёт cut property. Если exchange, stays-ahead, cut или другая структурная лемма не получается, локальная интуиция не гарантирует global optimum; тогда проверяют dynamic programming, graph algorithm или backtracking.

## Ментальная модель

Greedy choice должен оставить будущему не меньше возможностей, чем любой конкурентный первый выбор. Формально алгоритм поддерживает partial solution `G`. На шаге выбирается элемент `g`, после чего надо доказать safe-choice lemma:

```text
существует optimal solution O, которое содержит G ∪ {g}
```

Exchange proof берёт произвольное optimal `O`, несовместимое с `g`, заменяет в нём один элемент на `g` и показывает две вещи: feasibility сохранилась, objective не ухудшился. После обмена greedy prefix входит в некоторое optimal solution. Индукция повторяет аргумент для остатка.

Это важнее названия алгоритма. «Берём минимальное» не образует greedy proof, пока не названы invariant, допустимый обмен и metric, который не ухудшается.

## Как устроено

### Вывести rule из того, что ограничивает будущее

В interval scheduling задача — выбрать максимум непересекающихся полуинтервалов `[start, finish)`. Чем раньше заканчивается первый выбранный интервал, тем больше временной suffix остаётся. Поэтому кандидатов сортируют по `finish` и берут следующий с `start ≥ finish(last)`.

Выбор «раньше начинается» выглядит естественно, но длинный ранний интервал способен закрыть много коротких. «Минимальная длительность» тоже не сравнивает положение интервала относительно уже выбранного suffix. Greedy criterion должен быть связан с доказанным dominance, а не с локально приятной цифрой.

### Exchange argument для intervals

Пусть `g` заканчивается раньше всех доступных intervals, а `o` — первый interval в optimal schedule `O`. Тогда `finish(g) ≤ finish(o)`. Заменим `o` на `g`. Все остальные intervals в `O` начинались не раньше `finish(o)`, значит начинаются не раньше `finish(g)`. Feasibility и число intervals сохраняются.

После замены существует optimal schedule, начинающийся с `g`. Удаляем intervals, конфликтующие с `g`, и повторяем доказательство на оставшемся suffix. Это greedy-choice property плюс optimal substructure, выведенные из exchange, а не принятые на веру.

### Cut proof как соседняя схема

В minimum spanning tree partial forest задаёт cut между компонентой и остальными vertices. Lightest edge, пересекающий подходящий cut, safe: если optimal tree содержит другое crossing edge, добавление greedy edge создаёт cycle, а удаление не более лёгкого crossing edge сохраняет spanning tree и не увеличивает weight. Это снова exchange, но структура допустимого обмена задаётся cut/cycle.

Одинаковый шаблон не означает, что каждый local minimum безопасен. Нужна конкретная лемма для семейства feasible solutions. Matroid structure формализует широкий класс задач, где greedy по весу корректен; произвольный constraint таким свойством не обладает.

### Стоимость

Greedy loop часто линейный после preprocessing. Interval scheduling тратит `O(n log n)` на sorting и `O(n)` на scan. Если input уже отсортирован по finish, время `Θ(n)`. Auxiliary memory зависит от sorting contract и необходимости вернуть сами intervals.

Сложность не компенсирует отсутствие proof. Линейная эвристика, которая иногда ошибается, решает другую задачу.

## Пример или трассировка

Intervals обозначены как `[start, finish)`:

```text
A=[1,4), B=[3,5), C=[0,6), D=[5,7), E=[5,9), F=[8,9)
```

После sorting по finish порядок: `A, B, C, D, E, F`.

| Кандидат | Последний finish до шага | Решение | Выбранные |
| --- | ---: | --- | --- |
| A `[1,4)` | — | выбрать | A |
| B `[3,5)` | 4 | пропустить: `3 < 4` | A |
| C `[0,6)` | 4 | пропустить: `0 < 4` | A |
| D `[5,7)` | 4 | выбрать | A, D |
| E `[5,9)` | 7 | пропустить: `5 < 7` | A, D |
| F `[8,9)` | 7 | выбрать | A, D, F |

Получено три совместимых interval. Exchange proof подтверждает optimality: любой optimal first interval можно заменить на `A`, потому что `A` заканчивается не позже. После `A` тот же аргумент выбирает `D`, затем `F`.

Контрпример к earliest-start: intervals `[0,10)`, `[1,2)`, `[2,3)`, `[3,4)`. Правило earliest-start берёт один длинный interval; optimal schedule содержит три коротких. Локальная характеристика не сохраняет будущую свободу.

## Trade-offs

Greedy обычно хранит мало state, быстро исполняется и легко восстанавливает решение. Dynamic programming хранит несколько альтернативных summaries и справляется с глобальными зависимостями, но платит числом состояний. Связь разобрана в [[10 Основы CS/Динамическое программирование|заметке о динамическом программировании]].

Показательная граница — knapsack. Для fractional knapsack можно брать доли items, и сортировка по value/weight имеет exchange proof. В 0/1 knapsack выбор необратим: при capacity `50` items `(weight,value)=(10,60),(20,100),(30,120)` ratio-greedy берёт первые два и получает `160`, а items весов `20+30` дают `220`. Нужен DP или иной exact method.

Greedy иногда выбирают как approximation, когда exact optimum слишком дорог. Тогда контракт должен назвать approximation ratio или хотя бы отсутствие exact guarantee; нельзя молча выдавать эвристику за решение исходной задачи.

## Типичные ошибки

- **«Локально лучший выглядит очевидно» → найден counterexample с худшим global objective → отсутствует safe-choice lemma → построить exchange/cut proof либо сменить подход.**
- **«Earliest start оставляет больше времени» → длинный interval блокирует множество коротких → start ничего не говорит о свободном suffix после завершения → выбирать earliest finish и доказать заменой.**
- **«Greedy для fractional knapsack переносится на 0/1» → ratio order даёт `160` вместо `220` → нельзя обменять дробную часть item, feasibility family изменилась → использовать 0/1 DP.**
- **«Sorting — техническая деталь» → scan делает выбор не по доказанному criterion → proof предполагает глобально earliest finish/lightest safe edge → включить порядок в algorithm contract.**
- **«Полученный ответ feasible, значит optimal» → tests проверяют ограничения, но не максимум → feasibility и optimality — разные обязательства → отдельно сформулировать exchange/stays-ahead/cut argument.**

## Когда применять

- Ищите dominance: можно ли заменить первый выбор optimal solution на greedy choice без ухудшения?
- Для graph problems ищите cut/cycle property; для sequence scheduling — exchange или stays-ahead invariant.
- Перед кодом попробуйте сломать rule минимальным counterexample. Это быстрее, чем доказывать неверную идею.
- Если локальный выбор меняет ценность будущих вариантов и безопасного обмена нет, переходите к DP, shortest paths или search.
