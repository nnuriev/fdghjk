# Балансировка сетевой нагрузки

## TL;DR

Load balancer распределяет соединения или запросы между несколькими backends. [[10 Основы CS/Reverse proxy|Reverse proxy]] — частный L7-посредник: для клиента он выглядит как origin server, сам выбирает upstream и пересылает запрос дальше. Балансировщик не создаёт capacity и не знает бизнес-инварианты. Он лишь выбирает один endpoint из доступного ему набора, а этот набор уже зависит от discovery, health checks, readiness и draining.

L4-балансировка работает с адресами и transport connections, почти не разбирая прикладной протокол. L7-балансировка понимает HTTP или RPC и может маршрутизировать по host, path, headers и method. За эту гибкость платят TLS termination, парсингом протокола, connection pools, дополнительной latency и риском изменить HTTP-семантику. Выбор алгоритма вторичен: сначала нужно определить eligible endpoints, пределы очередей и поведение при отказе.

## Область применимости

Термины gateway и reverse proxy используются по RFC 9110 от июня 2022 года. Конкретные алгоритмы, active/passive health checking и draining сверены с Envoy 1.38.3; состояния EndpointSlice — с Kubernetes 1.36. Глобальная маршрутизация между регионами разобрана отдельно в [[40 Распределённые системы/Multi-region architecture|multi-region architecture]].

## Ментальная модель

Балансировка — конвейер из пяти решений:

```text
connection/request
  -> определить route
  -> построить eligible endpoint set
  -> выбрать endpoint
  -> получить connection из pool или открыть новую
  -> переслать attempt и учесть outcome
```

Алгоритм `round robin` отвечает только на третий шаг. Он ничего не исправит, если discovery устарел, readiness проверяет не тот путь, pool исчерпан или все backends уже перегружены.

Полезны четыре инварианта:

1. Новый attempt отправляется только endpoint, который подходит route, locality и текущей health policy.
2. Один пользовательский request может породить несколько upstream attempts из-за retry или hedging. Бизнес-операция от этого не становится идемпотентной.
3. Балансировщик принимает решение по локальному snapshot. Абсолютно свежего общего health state в распределённой системе нет.
4. Очереди, число соединений и retries ограничены. Иначе proxy переносит перегрузку дальше и увеличивает её.

## Как устроено

### L4, L7 и reverse proxy

L4-балансировщик выбирает backend по данным transport layer: destination address, port, protocol, иногда source address. После выбора он пересылает TCP или UDP traffic, не зная HTTP path и status code. TLS можно завершить раньше, пропустить насквозь или завершить на backend. Connection обычно закреплена за одним endpoint на весь срок жизни, поэтому несколько длинных connections легко дают неравномерную фактическую нагрузку при ровном числе соединений.

L7-балансировщик завершает прикладное соединение и создаёт отдельное upstream-соединение. Для HTTP он видит host, path, method и headers, умеет выбирать route, переиспользовать upstream pools и балансировать отдельные requests или streams. RFC 9110 называет gateway, или reverse proxy, intermediary, который выглядит origin server для внешнего соединения и пересылает запрос одному или нескольким inbound servers.

L7-посредник обязан сохранять семантику протокола: корректно обрабатывать hop-by-hop fields, размер тела, streaming, cancellation и trusted forwarding headers. Чем больше преобразований делает proxy, тем шире его compatibility и security surface. API-aware policy поверх этого механизма относится уже к [[20 Бэкенд/API gateway|API gateway]].

### От topology к eligible set

Полный список instances приходит из [[40 Распределённые системы/Service discovery|service discovery]], но discovered не означает готовый к traffic. Из topology исключают endpoint, если он:

- не относится к нужной версии, zone или route;
- ещё не прошёл readiness или initial health check;
- начал termination и больше не принимает новую работу;
- временно ejected по passive outlier detection;
- исчерпал локальный concurrency или connection budget, если policy это учитывает.

Health — наблюдение, а не доказательство. Active check периодически посылает специальный request. Passive check судит по реальным errors, timeouts и resets. Первый обнаруживает отказ без пользовательского traffic, второй видит настоящий path, но может принять перегрузку или ошибку одного route за смерть всего host. Порог последовательных failures и recovery hysteresis уменьшают flapping.

Readiness отвечает «готов ли process принять новую работу», liveness — «нужно ли перезапустить process». Смешивание этих сигналов создаёт restart loop: временно недоступная dependency снимает readiness, оркестратор считает процесс мёртвым, перезапускает его и усиливает outage.

### Выбор endpoint

Weighted round robin подходит для близких по стоимости requests и известных различий capacity. Least-request учитывает текущий in-flight и лучше реагирует на разную длительность, но его локальное число не отражает CPU, lock contention или скрытую очередь внутри backend. Random прост и при большом пуле избегает общей координации.

Ring hash и Maglev дают affinity: один hash key с высокой вероятностью попадает к тому же endpoint после небольших изменений topology. Это полезно для cache locality и connection stickiness, но не даёт session consistency. После failover запрос уйдёт другому host, а один hot key всё равно создаст hotspot. Состояние, нужное для корректности, хранят вне process либо реплицируют по отдельному протоколу.

### Connection pools, retries и draining

Выбор host не равен открытию новой connection. Reverse proxy обычно держит upstream pool, а HTTP/2 multiplexes несколько streams в одном соединении. Малый pool создаёт ожидание, большой способен одновременно перегрузить backend. Эти эффекты и метрики ожидания подробно разобраны в [[20 Бэкенд/Пулы соединений и keep-alive|пулах соединений и keep-alive]].

Retry создаёт новый attempt и должен укладываться в исходный [[20 Бэкенд/Дедлайны запросов и распространение отмены|deadline]]. Его разрешают только для явно повторяемого исхода и ограничивают retry budget. Timeout не доказывает, что первый backend не совершил эффект; для небезопасной операции нужны [[20 Бэкенд/Ключи идемпотентности и дедупликация запросов|idempotency key и дедупликация]].

При rollout endpoint сначала снимает readiness. После распространения topology proxy прекращает новые назначения, но уже открытые connections и streams остаются. Draining закрывает их протокольно: HTTP/1.1 может получить `Connection: close`, HTTP/2 — `GOAWAY`; затем действует общий grace deadline. Почему порядок readiness → drain → force важен, показано в [[20 Бэкенд/Graceful shutdown backend-сервиса|graceful shutdown backend-сервиса]].

### Наблюдаемость и overload

Нужны метрики по обеим сторонам proxy: accepted requests, upstream attempts, retries, local rejects, pool wait, active connections/streams, per-endpoint latency/errors, health transitions, ejections и время draining. Пользовательская latency и upstream latency различаются на queueing, TLS, routing filters и retries.

Если все eligible endpoints насыщены, новый алгоритм выбора capacity не добавит. Балансировщик должен ограничить admission или быстро отказать согласно [[40 Распределённые системы/Load shedding|load shedding]], а не складывать requests в безразмерную очередь. Panic-mode, при котором traffic снова идёт в unhealthy set, — осознанный выбор между вероятным успехом и контролируемым отказом, а не безопасный default.

## Пример или трассировка

У HTTP-сервиса три backends: `A`, `B`, `C`. Reverse proxy балансирует GET-запросы по least-request.

1. Во время rollout `B` снимает readiness. Новый snapshot исключает `B`, но два его текущих HTTP/2 streams продолжают работу в пределах drain deadline.
2. `GET /catalog/42` приходит в proxy. В eligible set остались `A` и `C`; у `A` пять in-flight requests, у `C` два, поэтому выбран `C`.
3. Connection из pool к `C` уже существует. Proxy открывает новый stream и передаёт оставшийся deadline.
4. `C` сбрасывает stream до получения response headers. GET допускает повтор по контракту, retry budget не исчерпан, а retry policy исключает предыдущий host, поэтому proxy делает второй attempt к `A`.
5. Клиент получает ответ `200`. Метрики фиксируют один downstream request, два upstream attempts и один retry. Если бы первым был `POST` с неизвестным outcome, автоматический retry без idempotency contract мог бы удвоить эффект.

Наблюдаемый результат: draining не оборвал работу на `B`, least-request выбрал менее занятый endpoint, а retry остался видимым отдельным attempt. Считать только внешние `200` было бы опасно: деградация `C` исчезла бы из telemetry.

## Trade-offs

L4 proxy дешевле по обработке и сохраняет end-to-end TLS, но не умеет route по HTTP-семантике и обычно балансирует connection, а не request. L7 proxy даёт маршрутизацию, retries, observability и protocol mediation. Цена — termination trust boundary, parsing cost и более сложное поведение при streaming.

Центральный reverse proxy упрощает policy и скрывает topology от clients. Client-side balancing убирает один data-plane hop и видит каждый endpoint, зато каждая client library должна корректно реализовать discovery, health, retry и rollout. Server-side proxy особенно полезен для внешних и разнородных clients; внутри однородного RPC-стека выбор зависит от стоимости общего hop и зрелости libraries.

Round robin предсказуем, least-request лучше видит неодинаковый in-flight, hash affinity сохраняет locality. Ни один алгоритм не лечит hot key, общий downstream bottleneck или недостаток capacity. Выбирать нужно по единице работы и доступным сигналам, а не по названию алгоритма.

## Типичные ошибки

### Health endpoint проверяет только process

- **Неверное предположение:** ответ `200 /healthz` означает способность выполнить любой request.
- **Симптом:** proxy продолжает слать writes instance с недоступной database.
- **Причина:** проверена liveness процесса, а не readiness критического path.
- **Исправление:** разделить liveness/readiness, проверять минимально необходимую способность и сохранять отдельные degraded routes, если это безопасно.

### Proxy повторяет каждый timeout

- **Неверное предположение:** timeout означает, что backend ничего не сделал.
- **Симптом:** дублируются платежи или команды, а outage усиливается retry traffic.
- **Причина:** transport outcome принят за business outcome.
- **Исправление:** retry только по contract, общий deadline и budget; небезопасные операции связывать с idempotency key.

### Sticky routing считают хранилищем session

- **Неверное предположение:** consistent hash всегда вернёт пользователя тому же process.
- **Симптом:** после rollout или failover session пропадает.
- **Причина:** affinity вероятностна и меняется вместе с topology.
- **Исправление:** вынести обязательное состояние из instance; affinity оставлять оптимизацией locality.

### Instance удаляют без draining

- **Неверное предположение:** удаление endpoint мгновенно прекращает все обращения.
- **Симптом:** rollout обрывает long requests и streams.
- **Причина:** topology update не закрывает уже установленные connections.
- **Исправление:** readiness, propagation window, protocol-aware draining и bounded force phase.

## Когда применять

Load balancer нужен, когда один logical endpoint обслуживают несколько interchangeable instances или когда traffic надо разделить по route, version, locality либо capacity class. Reverse proxy оправдан, если clients не должны знать topology, нужен TLS termination, единая HTTP-routing boundary или управляемый upstream pool.

До выбора продукта зафиксируйте единицу балансировки, источник endpoints, health semantics, retry contract, queue и connection limits, draining sequence и поведение при пустом healthy set. Если backend state нельзя безопасно перенести между instances, сначала разберите его consistency model: балансировщик эту проблему не решит.
