# Примитивы синхронизации

## TL;DR

Примитив синхронизации решает две разные задачи: координирует выполнение и создаёт порядок видимости памяти. POSIX mutex временно передаёт одному thread владение инвариантом. Condition variable позволяет ждать, пока изменится защищённый mutex predicate. Semaphore хранит число доступных разрешений и не имеет владельца. Linux futex находится уровнем ниже: связывает atomic state в user space с парковкой и пробуждением thread в kernel.

Главное практическое правило: ждать нужно состояние, а не уведомление. Predicate проверяют под тем же mutex и всегда в цикле. Futex, signal или wake сами по себе не заменяют [[10 Основы CS/Атомики и memory ordering|протокол атомарности и порядка памяти]]. В прикладном коде почти всегда выбирают pthread primitive или runtime языка, а не прямой `futex()`.

## Область применимости

- Пользовательские threads и процессы с общей памятью по POSIX.1-2024, The Open Group Base Specifications Issue 8.
- Linux-specific часть сверена с Linux man-pages 6.18. Futex — Linux API, а не POSIX.
- Детали API Go вынесены в [[60 Go/Mutex, RWMutex и примитивы координации sync|заметку о `sync`]]. Здесь Go нужен только как граница переноса модели: POSIX mutex имеет thread-owner, а `sync.Mutex` не привязан к goroutine и может быть разблокирован другой goroutine по контракту Go.
- Вне scope: distributed locks, транзакционная изоляция, lock-free memory reclamation и внутренние locks самого Linux kernel.

## Ментальная модель

У любого blocking primitive есть три логических слоя:

1. **Shared state** отвечает, можно ли продолжать: lock bit, predicate, permit counter или futex word.
2. **Wait protocol** закрывает гонку между проверкой state и засыпанием. Иначе producer способен изменить state и послать wake как раз в этом окне, после чего consumer уснёт навсегда.
3. **Memory-order edge** публикует записи одной стороне и разрешает другой стороне их наблюдать. Сам факт, что scheduler позже запустил waiter, такого edge не создаёт.

| Primitive | Какое состояние представляет | Есть owner | Сохраняет событие для будущего waiter | Для чего подходит |
| --- | --- | --- | --- | --- |
| Mutex | `locked/unlocked` и защищаемый инвариант | Да, в POSIX | Не применимо | Короткая exclusive critical section |
| Condition variable | Собственного business state нет; он лежит в predicate под mutex | Нет | Нет: signal без waiter не накапливается | Ожидание произвольного изменения predicate |
| Semaphore | Неотрицательный permit counter | Нет | Да, как значение counter | Ограничение числа одновременных пользователей ресурса |
| Futex | 32-bit user-space word плюс kernel wait queue только на время ожидания | Политику задаёт библиотека | Только согласно протоколу над word | Реализация mutex, condvar, semaphore и других waits |

Mutex, semaphore и condition variable задают прикладную семантику. Futex даёт более узкую операцию: «заблокируй меня, только если word всё ещё равен ожидаемому значению». Это строительный блок, а не готовый lock.

## Как устроено

### Mutex: владение инвариантом

Успешный `pthread_mutex_lock()` возвращает mutex в locked state, а вызывающий thread становится owner. Если mutex уже принадлежит другому thread, caller блокируется до освобождения. `pthread_mutex_unlock()` release-операцией публикует изменения critical section; последующее успешное acquire того же mutex позволяет новому владельцу видеть эти изменения по правилам POSIX memory synchronization.

Lock защищает не строку кода и не отдельное поле, а инвариант. Если `balance` и `version` должны меняться согласованно, оба поля читают и пишут по одному lock policy. Atomic read одного из полей не исправит snapshot, составленный из разных логических моментов.

POSIX различает `NORMAL`, `ERRORCHECK`, `RECURSIVE` и `DEFAULT` mutex. Повторный захват `NORMAL` тем же thread приводит к deadlock; `ERRORCHECK` возвращает ошибку; `RECURSIVE` ведёт lock count. Универсально полагаться на поведение `DEFAULT` при ошибочном relock нельзя. Robust mutex решает ещё одну задачу: если owner умер, следующий `lock` может вернуть `EOWNERDEAD`, уже передав mutex caller. Защищённое состояние при этом помечено inconsistent; его надо восстановить и подтвердить через `pthread_mutex_consistent()`, а не молча продолжать.

Стандарт не требует конкретной реализации fast path. На Linux futex позволяет библиотеке выполнить uncontended acquisition атомарно в user mode и войти в kernel только при вероятном длительном ожидании. Это объясняет, почему каждый `lock` не обязан быть system call, но не обещает, что конкретная версия libc использует один и тот же алгоритм.

### Condition variable: ждать predicate, а не signal

Condition variable связывают с mutex и predicate, например `queue_not_empty`. `pthread_cond_wait()` вызывают с уже захваченным mutex. Операция атомарно относительно конкурирующего доступа освобождает mutex и блокирует caller; перед возвратом она снова захватывает тот же mutex.

Канонический protocol выглядит так:

```text
lock(m)
while !predicate:
    cond_wait(cv, m)   # атомарно unlock + block; перед return снова lock
use_state()
unlock(m)
```

Цикл обязателен по двум причинам. Wait допускает spurious wakeup, а predicate мог снова стать false, пока разбуженный thread конкурировал за mutex. `signal` разблокирует как минимум одного текущего waiter, `broadcast` — всех; после этого они ещё должны получить mutex. Если waiter пока нет, notification не сохраняется. Поэтому durable truth лежит в predicate, а condition variable лишь сообщает, что его стоит проверить снова.

Один `pthread_cond_t` нельзя одновременно ждать с разными mutex, пока существует хотя бы один waiter: POSIX создаёт на этот период dynamic binding между condition variable и mutex. Нарушение даёт undefined behavior.

### Semaphore: учёт разрешений

POSIX semaphore хранит неотрицательное число permits. Wait-operation при положительном значении атомарно уменьшает его и продолжает; при нуле добавляет caller в множество ожидающих. `sem_post()` возвращает permit: увеличивает counter либо позволяет одному waiter завершить `sem_wait()`.

Владельца нет. Один thread может выполнить `sem_wait()`, другой — `sem_post()`. Поэтому binary semaphore со значениями 0/1 всё равно не равен mutex: он не доказывает ownership, допускает лишний `post` и выражает наличие permit, а не exclusive critical section. Counting semaphore естественно ограничивает connections, workers или slots, но корректность требует вернуть ровно столько permits, сколько было получено, включая error и cancellation paths.

Unnamed semaphore с `pshared == 0` доступен threads одного процесса; при ненулевом `pshared` его можно разместить в памяти, общей для процессов. POSIX допускает, что unnamed semaphore внутри реализации использует file descriptor, поэтому `sem_init()` может завершиться ошибкой `EMFILE` или `ENFILE`. На конкретную внутреннюю форму `sem_t` полагаться нельзя.

### Futex: atomic compare-and-block

Linux futex word имеет размер 32 bits и должен быть выровнен на четыре bytes. В uncontended path библиотека меняет word atomic instruction в user space. Когда продолжить нельзя, она вызывает `FUTEX_WAIT(uaddr, expected)`. Kernel загружает word, сравнивает его с `expected` и блокирует caller как одну атомарно упорядоченную compare-and-block операцию относительно конкурирующих futex operations на том же word.

Это закрывает lost-wakeup window. Если unlock успел изменить word до входа waiter в kernel, сравнение не совпадёт и wait вернётся с `EAGAIN` вместо сна. Если waiter уже заблокирован, unlock path меняет state и вызывает `FUTEX_WAKE`, чтобы разбудить ограниченное число waiters.

Futex не определяет смысл значений word, ownership, fairness или защищаемый invariant. Он также не даёт отдельного контракта публикации произвольного payload: acquire/release ordering должен быть частью atomic protocol над word. Прямой futex-код обязан обрабатывать повторную проверку state, `EAGAIN`, timeout, interruption, spurious wakeups, lifetime общей памяти, process-shared keys, robust owner death и при необходимости priority inheritance. Именно поэтому man-pages рекомендует большинству программ использовать NPTL и другие библиотеки поверх futex.

### Happens-before через primitive

Рассмотрим mutex publication:

```text
Thread P                         Thread C
lock(m)
payload = 42
ready = true
unlock(m)  --release-->         lock(m)  --acquire-->
                                 if ready: read payload  # обязано увидеть 42
                                 unlock(m)
```

Запись `payload` sequenced-before release. Последующий acquire того же mutex идёт после release в synchronization order. Транзитивно запись happens-before чтением. Реальное время, cache coincidence и порядок запуска CPU в доказательстве не участвуют.

Condition wait получает тот же эффект при повторном захвате mutex. Успешные `sem_post()`, `sem_wait()` и timed/try variants входят в перечень POSIX memory-synchronizing operations. Для точного значения acquire, release, modification order и sequential consistency нужна соседняя заметка [[10 Основы CS/Атомики и memory ordering|«Атомики и memory ordering»]]. В Go похожие доказательства формулируются через собственную [[60 Go/Модель памяти Go и happens-before|модель памяти Go]], а не через POSIX API.

## Пример или трассировка

Есть один producer, один consumer, `ready == false`, `payload` пока не задан. Оба используют `m` и `cv`.

1. Consumer захватывает `m`, видит `ready == false` и входит в `pthread_cond_wait(cv, m)`.
2. Wait атомарно освобождает `m` и регистрирует ожидание. Producer не может вклиниться между отдельными «unlock» и «начать ждать», потому что такого открытого окна в операции нет.
3. Producer захватывает `m`, записывает `payload = 42`, затем `ready = true`, вызывает `pthread_cond_signal(cv)` и освобождает `m`.
4. Consumer пробуждается, но продолжить может только после повторного acquire `m`. Цикл снова проверяет `ready`, затем читает `payload == 42`.

Другой interleaving тоже безопасен: producer может закончить до шага consumer. Тогда consumer позже захватит `m`, сразу увидит `ready == true` и вообще не вызовет `wait`. Signal уже исчез, но это не потеря события: состояние хранится в `ready`.

Наблюдаемый инвариант: consumer читает payload только при истинном predicate под mutex. Замена цикла на «проверить без lock, затем ждать» создаёт окно lost wakeup; замена `while` на `if` ломается при spurious wakeup или конкуренции нескольких consumers.

## Trade-offs

- **Mutex или atomic.** Mutex проще защищает несколько полей и check-then-act transition. Atomic подходит для одной synchronization point или тщательно доказанного protocol; составные инварианты разобраны отдельно в [[60 Go/Пакет sync-atomic|заметке о Go atomics]].
- **Condition variable или semaphore.** Condvar ждёт произвольный predicate и не запоминает signal. Semaphore хранит число permits, зато сам predicate выразить не умеет. Очередь задач обычно нуждается в mutex плюс condvar; лимит одновременно занятых slots — в counting semaphore.
- **Blocking или spinning.** Spin избегает sleep/wakeup только при очень коротком и ограниченном ожидании. Если owner вытеснен, critical section длинная или runnable threads больше CPU, waiter расходует CPU без progress. Для обычного user-space кода blocking mutex безопаснее как default.
- **`signal` или `broadcast`.** Signal уменьшает число разбуженных contenders, но годится лишь когда одного progress достаточно. Broadcast нужен, если новое состояние разрешает работу нескольким waiters или разные waiters проверяют разные predicates; цена — herd и повторная конкуренция за mutex.
- **Pthread primitive или raw futex.** Pthread API переносимее и закрывает lifecycle, clocks, robust/PI modes и edge cases. Raw futex оправдан внутри runtime или synchronization library, где эти контракты реализуют и тестируют явно.
- **Обычный или PI mutex.** Priority inheritance ограничивает priority inversion в realtime scheduling, но требует поддерживаемого protocol и добавляет bookkeeping. Для throughput-oriented сервиса сначала сокращают critical section и проверяют contention; PI решает другую задачу — deadline predictability.

Выбор channel ownership против shared-memory lock в Go имеет другие API trade-offs; их не следует выводить напрямую из POSIX. Они разобраны в [[60 Go/Каналы или mutex|«Каналы или mutex»]].

## Типичные ошибки

### Проверка predicate вне mutex

- **Неверное предположение:** между проверкой и `wait` producer не успеет выполнить signal.
- **Симптом:** consumer редко зависает навсегда при уже готовой работе.
- **Причина:** signal попал в окно до регистрации waiter и не сохранился.
- **Исправление:** менять и проверять predicate под одним mutex; ждать через atomic unlock-and-block operation.

### `if` вместо `while` вокруг condition wait

- **Неверное предположение:** возврат из `wait` доказывает истинность condition.
- **Симптом:** consumer читает пустую queue или нарушает bounds.
- **Причина:** spurious wakeup либо другой consumer успел изменить state до reacquire mutex.
- **Исправление:** после каждого возврата заново проверять predicate в цикле.

### Несогласованный порядок locks

- **Неверное предположение:** каждая critical section локально корректна, значит весь protocol безопасен.
- **Симптом:** T1 держит `A` и ждёт `B`, T2 держит `B` и ждёт `A`.
- **Причина:** wait-for graph получил cycle.
- **Исправление:** ввести глобальный lock order, объединить инвариант под одним mutex либо убрать вложенное владение. Отсутствие data race не доказывает liveness; различие раскрыто в [[60 Go/Data races, deadlocks и livelocks|заметке о races и deadlocks]].

### Долгая работа под mutex

- **Неверное предположение:** lock cost сводится к одной atomic instruction.
- **Симптом:** растут tail latency, convoy и context switches.
- **Причина:** waiter платит временем всей critical section; blocking I/O или callback внутри неё делает hold time неограниченным.
- **Исправление:** под lock менять только общий state, а долгую работу выполнять вне section, сохраняя инвариант через snapshot или ownership transfer.

### Semaphore используют как mutex

- **Неверное предположение:** counter 0/1 автоматически даёт mutex semantics.
- **Симптом:** чужой `post`, двойной release или забытый permit пропускает лишнего участника либо блокирует всех.
- **Причина:** semaphore не хранит owner и считает permits.
- **Исправление:** для exclusive ownership использовать mutex; semaphore оставлять для capacity protocol.

### Wake считают memory barrier для payload

- **Неверное предположение:** `FUTEX_WAKE` или случайное планирование само публикует соседние данные.
- **Симптом:** waiter проснулся, но protocol допускает stale или racing read.
- **Причина:** compare-and-block относится к futex word; порядок payload должен следовать из acquire/release operations или готового lock contract.
- **Исправление:** применять корректные atomics либо библиотечный primitive и доказывать happens-before.

### Fairness принимают за гарантию

- **Неверное предположение:** самый старый waiter обязательно получит mutex следующим.
- **Симптом:** один thread долго не продвигается под постоянным contention.
- **Причина:** общий POSIX contract отдаёт выбор scheduling policy и не обещает универсальную FIFO fairness.
- **Исправление:** не строить correctness на порядке пробуждения; если bounded waiting обязательна, выбрать primitive и scheduler с явной гарантией и проверить платформу.

## Когда применять

Начните с формулировки состояния и владельца:

- общий составной инвариант с короткими accesses — mutex;
- ожидание изменения этого инварианта — condition variable с тем же mutex и predicate-loop;
- счётное число interchangeable permits — semaphore;
- реализация собственного низкоуровневого wait primitive на Linux — futex плюс отдельный atomic protocol.

Затем проверьте safety и liveness раздельно. Для safety нарисуйте happens-before path от каждой публикации к чтению. Для liveness постройте wait-for graph, задайте timeout/cancellation policy, оцените priority inversion и запретите неограниченную работу под lock. Если объяснение требует предположить «scheduler обычно запускает нужный thread», synchronization protocol ещё не закончен.
