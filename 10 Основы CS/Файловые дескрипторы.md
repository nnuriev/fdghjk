# Файловые дескрипторы

## TL;DR

Файловый дескриптор (file descriptor, FD) — небольшое неотрицательное число, индекс в таблице конкретного процесса. Само число не хранит файл или сокет. Запись таблицы ссылается на открытое файловое описание (open file description), а уже оно — на объект ядра и его операции.

Это различие объясняет почти все неочевидные случаи. `dup()` и `fork()` создают новые дескрипторы, которые ссылаются на то же open file description, поэтому делят текущую позицию и status flags, включая `O_NONBLOCK`. Отдельный `open()` создаёт новое описание даже для того же inode. `FD_CLOEXEC`, напротив, принадлежит записи таблицы дескрипторов и не разделяется так же.

FD нужно воспринимать как короткоживущий capability: владение им обязано быть явным, закрытие — ровно один раз, а передача между потоками требует протокола lifetime. После `close()` ядро быстро переиспользует число, поэтому старый `fd = 7` может начать обозначать совершенно другой объект.

## Область применимости

Ниже описан Linux на фиксированном tag `v7.1` с интерфейсами Linux man-pages 6.18 и общая модель POSIX.1-2024. Файлы, каталоги, pipes, [[10 Основы CS/Сокеты|сокеты]], `eventfd`, `signalfd` и экземпляр `epoll` представлены дескрипторами, но поддерживают разные операции. Унифицировано именование объекта в системном вызове, а не его семантика.

## Ментальная модель

Полезно держать в голове три уровня:

```text
процесс
  fd 3 ─┐                         ┌─ inode / pipe / socket / epoll object
  fd 8 ─┴─> open file description ┤
              offset              └─ набор операций ядра
              status flags
              reference count
```

На уровне процесса находятся число FD и descriptor flags, прежде всего `FD_CLOEXEC`. В системной таблице открытых файлов находится open file description: текущая позиция, access mode, `O_APPEND`, `O_NONBLOCK` и другие status flags. Нижний уровень зависит от типа объекта: для обычного файла это dentry и inode, для сокета — socket state и buffers.

Отсюда следуют четыре инварианта:

1. Число FD имеет смысл только внутри таблицы конкретного процесса и пока запись не закрыта.
2. Несколько FD могут владеть одним open file description; последнее живёт до исчезновения последней ссылки, а не до первого `close()`.
3. Имя в файловой системе и открытый объект имеют разные lifetime. `unlink()` удаляет directory entry, но открытый FD продолжает работать.
4. Успешный `close()` завершает владение дескриптором, но не служит контрактом долговечности данных. Для этого нужен `fsync()` или другой явно выбранный механизм из [[10 Основы CS/Файловая система и буферизация|модели буферизации и durability]].

## Как устроено

### `open()`, `dup()` и общий offset

Успешный `open()` выбирает самый маленький свободный номер FD и создаёт новое open file description. Два независимых вызова `open()` одного pathname получают разные descriptions и разные offsets. `dup()`, `dup2()`, `dup3()` и унаследованный после `fork()` дескриптор ссылаются на прежнее description. Чтение через один FD сдвигает позицию, которую увидит другой.

Разделяются file status flags: `O_APPEND`, `O_NONBLOCK`, `O_ASYNC` и часть остальных флагов, доступных через `F_GETFL`/`F_SETFL`. Не разделяется `FD_CLOEXEC`: это file descriptor flag отдельной записи. Такое расхождение практично, но опасно. Поток, включивший `O_NONBLOCK` через один дубликат, меняет поведение операций через все дескрипторы того же description.

### Наследование и `CLOEXEC`

После `fork()` ребёнок получает копии таблицы FD, которые продолжают ссылаться на те же open file descriptions. После `execve()` открытые дескрипторы сохраняются, кроме помеченных `FD_CLOEXEC`.

В многопоточной программе нельзя надёжно сделать `open(); fcntl(F_SETFD, FD_CLOEXEC)` двумя шагами: между ними другой поток успеет выполнить `fork()` и `execve()`, а новый процесс унаследует FD. Флаги `O_CLOEXEC`, `SOCK_CLOEXEC`, `accept4(..., SOCK_CLOEXEC)` и аналогичные варианты устанавливают close-on-exec атомарно вместе с созданием дескриптора.

Наследование без явного контракта даёт утечки, зависающие pipes и лишние полномочия у дочернего процесса. Безопасный default для инфраструктурного кода — создавать FD с `CLOEXEC`, а передавать через `exec` только намеренно.

### Lifetime, `unlink()` и `close()`

Directory entry связывает имя с inode. Если удалить последнее имя через `unlink()`, inode и его данные освобождаются лишь после закрытия последнего FD и исчезновения прочих ссылок. Поэтому процесс может читать удалённый файл, а место на диске остаётся занятым. Это нормальный механизм для временных файлов и частая причина расхождения между `du` и фактическим свободным местом.

Linux разрешает другой тонкий сценарий: один поток блокируется в I/O, а второй закрывает тот же FD. Выполняющийся системный вызов удерживает ссылку на open file description и может успешно закончить уже после `close()` во втором потоке. Закрытие по номеру не служит универсальным механизмом отмены I/O.

Повторять `close(fd)` после ошибки тоже нельзя. Linux освобождает номер рано, а поздняя фаза flush может вернуть ошибку. К моменту повтора тот же номер уже способен принадлежать чужому объекту. Ошибку `close()` используют для диагностики возможной потери данных; если нужна durability, перед закрытием проверяют `fsync()`.

### FD как capability и проблема переиспользования

Доступ к FD даёт право выполнять разрешённые операции над уже открытым объектом без нового pathname lookup. Directory FD вместе с `openat()`/`renameat()` закрепляет точку разрешения пути и убирает часть TOCTOU-гонок: каталог можно переименовать, но открытая ссылка останется на том же объекте.

Цена такой компактности — reuse. Если компонент закрыл `fd = 12`, другой поток открыл файл и снова получил 12, запоздалый callback не отличит старый объект от нового по одному числу. Event loop обычно связывает событие с объектом приложения, поколением или token и централизует close. В [[10 Основы CS/select, poll, epoll и kqueue|epoll]] особенно важно учитывать уже выданные batches событий.

### Лимиты и наблюдаемость

`RLIMIT_NOFILE` задаёт значение на единицу больше максимального номера FD, доступного процессу. Исчерпание даёт `EMFILE`; системный лимит open file descriptions даёт `ENFILE`. Эти состояния и их связь с соединениями разобраны в [[10 Основы CS/Исчерпание ресурсов процесса|исчерпании ресурсов процесса]].

Снимок процесса виден в `/proc/<pid>/fd`, подробности отдельных entries — в `/proc/<pid>/fdinfo`, лимиты — в `/proc/<pid>/limits`. Считать открытые FD полезно, но для поиска leak важнее группировать их по типу и возрасту: тысяча ожидаемых sockets и тысяча забытых файлов выглядят одинаково как одно число.

### Диагностика сетевых соединений через `ss` и `lsof`

`ss` начинает с сетевой стороны: читает состояние сокетов ядра и умеет показать TCP state, локальный и удалённый endpoint, очереди и, при достаточных правах, процесс-владелец. `lsof` начинает с process FD table и связывает конкретный номер FD с socket. Поэтому команды дополняют друг друга:

```text
ss -s
ss -tanp
lsof -nP -a -p 4242 -iTCP
lsof -nP -iTCP -sTCP:ESTABLISHED
```

`ss -s` даёт сводку, а `ss -tanp` показывает все TCP-сокеты без DNS/service-name resolution; `-p` добавляет процесс, если это разрешают права. Первый вызов `lsof` ограничивает снимок PID 4242 и TCP-FD, второй ищет установленные TCP-соединения во всей доступной process table. `-nP` отключает разрешение имён и не превращает диагностику в серию DNS-запросов.

Это снимок, а не трасса событий. Короткоживущий socket может исчезнуть между двумя командами, PID и FD могут быть переиспользованы, а чужой process скрыт политикой доступа. Для leak ищут тренд по числу FD и состояниям соединений, затем сопоставляют его с pool metrics и `/proc/<pid>/fd`; один большой снимок ещё не доказывает утечку.

## Пример или трассировка

Пусть файл содержит `abcdef`:

```text
fd3 = open("data", O_RDONLY | O_CLOEXEC)  -> 3, description A, offset 0
fd4 = dup(fd3)                            -> 4, description A, offset 0
read(fd3, 2)                              -> "ab", A.offset = 2
read(fd4, 2)                              -> "cd", A.offset = 4

fd5 = open("data", O_RDONLY | O_CLOEXEC)  -> 5, description B, offset 0
read(fd5, 2)                              -> "ab", B.offset = 2

unlink("data")                           -> имя исчезло
read(fd4, 2)                              -> "ef": description A ещё держит inode
close(fd3); close(fd4); close(fd5)         -> последняя ссылка исчезла, данные можно освободить
```

Наблюдаемый результат показывает границу shared state. Дубликаты 3 и 4 делят offset; новый `open()` на FD 5 не делит его. `unlink()` меняет namespace, но не разрывает уже открытые ссылки.

## Trade-offs

Повторный `open()` даёт независимые offsets и status flags, но снова разрешает pathname, проверяет права и создаёт новое open file description. `dup()` дешёво передаёт то же открытое состояние, зато связывает участников общим offset и `O_NONBLOCK`. Выбор зависит от того, нужно ли разделять позицию и режим, а не от желания получить «ещё один номер».

Path API удобен, когда операция действительно относится к текущему имени. FD-relative API устойчивее к rename и уменьшает TOCTOU surface, но требует строгого lifetime и закрытия directory FD.

Наследование FD через `exec` подходит для supervision и socket activation. В обычном библиотечном коде оно чаще означает leak; поэтому атомарный `CLOEXEC` лучше считать исходным режимом.

## Типичные ошибки

### Считать FD самим объектом

- **Неверное предположение:** два разных номера всегда имеют независимое состояние.
- **Симптом:** чтение через один дубликат неожиданно сдвигает позицию другого или включает ему non-blocking mode.
- **Причина:** оба FD ссылаются на одно open file description.
- **Исправление:** явно различать descriptor flags и status flags; для независимого состояния делать отдельный `open()`.

### Закрывать один FD из нескольких владельцев

- **Неверное предположение:** «кто закончил первым, тот и закрывает», а остальные заметят отмену.
- **Симптом:** use-after-close по номеру, reuse чужого FD или I/O, которое всё равно завершилось.
- **Причина:** ownership не определён, а номер переиспользуется независимо от логики приложения.
- **Исправление:** один владелец закрывает FD; передача владения и отмена задаются отдельным протоколом.

### Ставить `FD_CLOEXEC` вторым вызовом

- **Неверное предположение:** между `open()` и `fcntl()` ничего существенного не произойдёт.
- **Симптом:** редкий leak FD в запущенный дочерний процесс; pipe не получает EOF.
- **Причина:** `fork()`/`execve()` в другом потоке попадает в окно гонки.
- **Исправление:** использовать `O_CLOEXEC`, `dup3()`, `accept4()` и другие атомарные варианты.

### Повторять `close()` после ошибки

- **Неверное предположение:** ошибка означает, что FD остался открыт.
- **Симптом:** закрывается уже другой объект с переиспользованным номером.
- **Причина:** Linux освобождает descriptor до поздней операции, которая сообщила ошибку.
- **Исправление:** не retry `close()`; проверять ошибку для диагностики, а durability подтверждать до него через `fsync()`.

## Когда применять

Модель FD нужна при любом системном I/O, но особенно при проектировании event loop, subprocess management, временных файлов, socket pools и безопасных операций с путями. Перед передачей FD ответьте на четыре вопроса: кто владеет close, разделяется ли open file description, должен ли FD пережить `exec`, как запоздалый callback отличит старое поколение от нового.
