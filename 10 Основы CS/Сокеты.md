# Сокеты

## TL;DR

Сокет (socket) — объект ядра, который представляет endpoint обмена данными. Процесс работает с ним через [[10 Основы CS/Файловые дескрипторы|файловый дескриптор]], поэтому к сокетам применимы `read`/`write`, `close`, `O_NONBLOCK`, readiness API и лимиты FD. При этом сокет сохраняет собственное protocol state, адреса и очереди данных.

Для TCP server lifecycle выглядит как `socket → bind → listen → accept`. Listening socket принимает connection requests, а `accept()` создаёт новый connected socket; listener и каждое соединение имеют разные FD. Для клиента `connect()` запускает установление связи. Успешный `send()` означает, что ядро приняло байты в локальный networking path, а не что peer application их прочитало.

`SOCK_STREAM` даёт упорядоченный поток байтов без границ сообщений. Один `send()` не обязан соответствовать одному `recv()`. `SOCK_DGRAM` сохраняет границы datagrams, но одна datagram может потеряться, продублироваться или прийти не по порядку в зависимости от протокола. Протокол приложения обязан сам определить framing, deadlines и пределы buffers.

## Область применимости

Заметка описывает Linux на tag `v7.1`, Linux man-pages 6.18 и POSIX.1-2024 на уровне Socket API. TCP congestion control, retransmission, UDP и IP routing остаются вне scope: здесь важен контракт между приложением и ядром. Non-blocking режим раскрыт в [[10 Основы CS/Блокирующий и неблокирующий ввод-вывод|отдельной заметке]], а ожидание многих сокетов — в [[10 Основы CS/select, poll, epoll и kqueue|readiness API]].

## Ментальная модель

Сокет — двусторонний канал с локальным состоянием по обе стороны системного вызова:

```text
application
  send(bytes) -> [local send buffer] -> protocol/network -> [peer receive buffer] -> recv(bytes)
  recv(bytes) <- [local receive buffer] <- protocol/network <- [peer send buffer]
```

Между `send()` и `recv()` нет общей транзакции. Успех на одной границе не подтверждает следующую. Буфер отправителя, сеть, буфер получателя и код peer application могут отказать в разные моменты.

Для server полезна вторая схема:

```text
SYN / connection request
  -> protocol handshake state
  -> очередь установленных соединений listener
  -> accept()
  -> отдельный connected socket
```

Listening FD описывает точку приёма, а accepted FD — одно соединение. `accept()` не «превращает» listener в connected socket.

## Как устроено

### Domain, type и protocol

`socket(domain, type, protocol)` выбирает семейство адресов, семантику обмена и конкретный протокол. Частые domains: `AF_INET`/`AF_INET6` для IP и `AF_UNIX` для локального IPC. `SOCK_STREAM` задаёт connection-oriented byte stream, `SOCK_DGRAM` — message-oriented datagrams, `SOCK_SEQPACKET` — connection-oriented сообщения с сохранением границ.

Флаги `SOCK_CLOEXEC` и `SOCK_NONBLOCK` можно добавить к `type`, чтобы атомарно настроить новый FD. Это устраняет окна гонки между `socket()` и последующим `fcntl()`.

Адрес не равен соединению. `bind()` назначает локальный address; `listen()` переводит подходящий stream socket в passive state; `connect()` задаёт peer и запускает protocol-specific установление связи. Для UDP вызов `connect()` не создаёт TCP-подобный handshake: он фиксирует default peer и фильтрует входящие datagrams по адресу.

### Server lifecycle и две входные очереди TCP

После `listen(fd, backlog)` сокет становится passive. В Linux начиная с 2.2 `backlog` ограничивает очередь полностью установленных TCP connections, которые ещё не забрал `accept()`. Незавершённые handshakes обычно учитываются отдельно через `tcp_max_syn_backlog`; при включённых TCP syncookies Linux игнорирует этот параметр и не задаёт очереди logical maximum. Пользовательский `backlog` сверху ограничен `somaxconn`.

`accept()` извлекает одно pending connection, создаёт новый socket object и возвращает новый FD. Listening socket остаётся доступен для следующих connections. На Linux новый FD от обычного `accept()` не наследует `O_NONBLOCK` и `O_ASYNC`; `accept4(..., SOCK_NONBLOCK | SOCK_CLOEXEC)` задаёт нужные флаги атомарно и явно.

Readiness listener означает, что `accept()` вероятно сможет забрать connection без ожидания. Это не гарантия: другой поток может принять его первым, а asynchronous network error — удалить request. Поэтому listener event loop держат non-blocking и вызывают `accept4()` до `EAGAIN`.

### Поток байтов, partial I/O и EOF

TCP socket типа `SOCK_STREAM` не хранит границы `send()`. Если peer отправил header и body тремя вызовами, локальный `recv()` способен вернуть их одним куском, частью вызова или несколькими фрагментами. Короткий `recv()` не означает конец сообщения. Приложение использует length prefix, delimiter, фиксированный размер или framing протокола.

`send()` тоже может принять меньше запрошенного. Причины включают заполненный buffer, signal после частичной передачи и ограничения конкретного socket type. Код хранит непринятый suffix и продолжает после новой writable readiness. Для datagram short send не трактуют как нормальное продолжение datagram: сообщение передаётся целиком либо вызов завершается ошибкой согласно protocol semantics.

Для stream `recv() == 0` при ненулевом запрошенном размере означает orderly shutdown write-half со стороны peer после чтения всех уже buffered bytes. Это EOF потока, а не «пустое сообщение». Вызов с нулевым размером сам является no-op и тоже может вернуть `0`, не сообщая об EOF. `EAGAIN` означает, что в момент вызова данных нет; connection при этом остаётся открытым.

### Readiness, ошибки и завершение `connect()`

Readable socket способен означать данные, EOF, pending error или connection для listener. Writable socket означает, что хотя бы небольшая запись не должна блокироваться; большой `send()` всё равно может заполнить buffer. Writable также сигнализирует завершение non-blocking `connect()`, но не его успех. После события нужно прочитать `SO_ERROR` через `getsockopt()`: ноль означает успех, ненулевое значение — сохранённую ошибку подключения.

Readiness — подсказка к попытке I/O. Между notification и системным вызовом состояние меняется, поэтому корректный event loop всё равно обрабатывает `EAGAIN`, EOF, `POLLERR`/`EPOLLERR` и partial result.

### Buffers и backpressure

Send и receive buffers принадлежат ядру и ограничены. Если приложение пишет быстрее, чем peer и сеть освобождают send buffer, blocking `send()` ждёт, а non-blocking возвращает `EAGAIN`. Это локальное проявление [[40 Распределённые системы/Backpressure и queue buildup|backpressure]]: предел нужно перенести в admission/concurrency приложения, иначе данные просто переедут в ещё одну очередь в памяти.

Увеличение `SO_SNDBUF` или `SO_RCVBUF` поглощает более длинный burst и способно поднять throughput на path с большим bandwidth-delay product. Цена — память на connection и более длинная скрытая очередь. Buffer size не лечит peer, который перестал читать.

### `shutdown()` и `close()`

`shutdown(fd, SHUT_WR)` запрещает дальнейшую отправку и для connection-oriented protocol инициирует orderly завершение направления записи. Вторая половина остаётся доступна: процесс может продолжать `recv()` ответ до EOF. `SHUT_RD` и `SHUT_RDWR` закрывают другие направления по контракту API.

`close(fd)` убирает одну descriptor reference. Если есть `dup()` или FD у другого процесса, socket может остаться открытым. При последней ссылке ядро начинает protocol-specific teardown. `SO_LINGER` меняет поведение последнего close и легко превращает его в блокирующую или abortive операцию; использовать option нужно только с явно проверенным протокольным требованием.

Запись в socket после закрытия read side у peer может вернуть `EPIPE` и сгенерировать `SIGPIPE`. Серверы обычно блокируют или игнорируют `SIGPIPE`, либо используют `MSG_NOSIGNAL`, а затем обрабатывают `EPIPE` как обычный I/O outcome.

## Пример или трассировка

Минимальный non-blocking TCP server проходит такие состояния:

```text
lfd = socket(AF_INET6, SOCK_STREAM | SOCK_NONBLOCK | SOCK_CLOEXEC, 0)
bind(lfd, local_address)
listen(lfd, backlog)
register lfd for readable events

lfd readable
  accept4(lfd, ..., SOCK_NONBLOCK | SOCK_CLOEXEC) -> cfd
  accept4(lfd, ...)                               -> EAGAIN, очередь опустела

cfd readable
  recv(cfd, 4 KiB) -> 900 bytes
  parser          -> header complete, body incomplete
  recv(cfd, 4 KiB) -> EAGAIN

позже cfd readable
  recv(cfd, 4 KiB) -> remaining body
  application     -> готовит response
  send(cfd, response) -> short count
  сохранить suffix, ждать writable

cfd writable
  send(cfd, suffix) -> весь остаток
  shutdown(cfd, SHUT_WR)
  recv(cfd, ...) -> 0 после orderly peer shutdown
  close(cfd)
```

Граница request не совпала с первым `recv()`, writable event не отменил short write, а half-close позволил дочитать входное направление. Если бы `accept()` и `send()` считались «операциями целиком», server терял бы connections и обрезал responses под нагрузкой.

## Trade-offs

Stream удобен для надёжной упорядоченной передачи произвольного объёма, но требует application framing и не сохраняет сообщения. Datagram сохраняет одну границу сообщения и не требует connection state в том же виде, зато приложение учитывает размер datagram, loss, duplication и ordering по контракту выбранного протокола.

Blocking socket с отдельным thread проще рассуждать локально: call возвращается, когда есть progress или error. Non-blocking event loop обслуживает много mostly-idle connections меньшим числом threads, но требует явной state machine, partial I/O и справедливого scheduling.

Большие kernel buffers сглаживают bursts. Малые быстрее проявляют backpressure и ограничивают память. Правильный размер зависит от throughput, RTT, числа connections и допустимой queueing latency; «чем больше, тем лучше» неверно.

`shutdown(SHUT_WR)` выражает protocol half-close, `close()` завершает локальное владение FD. Первый нужен, когда peer должен увидеть EOF, но ответ ещё надо дочитать. Второй достаточен, когда application protocol уже завершён или outcome больше не нужен.

## Типичные ошибки

### Считать TCP сообщениями

- **Неверное предположение:** один `send()` на peer даст один `recv()`.
- **Симптом:** parser принимает обрезанный request или склеивает два requests.
- **Причина:** `SOCK_STREAM` сохраняет порядок байтов, но не границы вызовов.
- **Исправление:** ввести framing и incremental parser с bounded message size.

### Считать writable успешным подключением

- **Неверное предположение:** `EPOLLOUT` после `EINPROGRESS` означает успешный `connect()`.
- **Симптом:** приложение начинает protocol handshake на failed socket.
- **Причина:** writable сообщает о завершении attempt, включая failure.
- **Исправление:** прочитать `SO_ERROR`; при ошибке закрыть socket и создать новый для следующей portable попытки.

### Забывать flags accepted socket

- **Неверное предположение:** accepted socket наследует `O_NONBLOCK` listener на Linux.
- **Симптом:** один редкий `recv()` блокирует весь event-loop thread.
- **Причина:** Linux `accept()` не наследует эти file status flags.
- **Исправление:** использовать `accept4(..., SOCK_NONBLOCK | SOCK_CLOEXEC)`.

### Лечить медленного peer увеличением buffer

- **Неверное предположение:** больший `SO_SNDBUF` устраняет backpressure.
- **Симптом:** RSS растёт, responses устаревают в очереди, tail latency увеличивается.
- **Причина:** buffer лишь откладывает момент насыщения.
- **Исправление:** ограничить per-connection output, concurrency и deadline; медленных consumers закрывать по contract.

## Когда применять

Socket API нужен для network I/O и локального IPC через `AF_UNIX`. До выбора модели сервера зафиксируйте socket type, framing, maximum message, connection и idle deadlines, поведение half-close, предел output queue и владельца `close()`. Настройка transport options идёт после этих решений: options не восполняют отсутствующий protocol contract.
