# HTTP/1.1

## TL;DR

HTTP/1.1 — текстовый протокол обмена сообщениями поверх надёжного упорядоченного byte stream, обычно TCP. Его главный эксплуатационный инвариант: обе стороны и каждый intermediary должны одинаково определить границу текущего сообщения. Пока получатель не понял, где заканчивается body, он не знает, с какого байта начинается следующий response или request в persistent connection.

Протокол повторно использует соединения по умолчанию, но не умеет независимо доставлять несколько ответов внутри одного соединения: при pipelining ответы обязаны идти в порядке запросов. Поэтому медленный первый ответ задерживает готовый второй. HTTP/1.1 задаёт framing и транспорт сообщения; смысл методов, статус-кодов, полей и кэшей разобран отдельно в [[20 Бэкенд/HTTP-методы, статус-коды, заголовки и семантика кэширования|заметке о семантике HTTP]].

## Область применимости

- Базовая версия — HTTP/1.1 по RFC 9112, опубликованному в июне 2022 года, совместно с HTTP Semantics по RFC 9110.
- Рассматриваются message framing, persistence, pipelining, hop-by-hop state, upgrade и границы доверия между intermediaries.
- Вне scope: правила HTTP-кэширования, бизнес-безопасность повторов, настройка connection pool и конкретные серверные фреймворки. Эти решения используют HTTP/1.1, но не определяются его синтаксисом.

## Ментальная модель

Представьте соединение как одну непрерывную ленту байтов. Request line или status line и поля заголовка описывают очередное сообщение, а framing-правила отрезают от ленты ровно его body. После этого следующий байт можно интерпретировать как начало нового сообщения.

Отсюда следуют два инварианта:

1. одно HTTP/1.1-соединение имеет один общий порядок байтов и одну текущую позицию parser;
2. перед повторным использованием соединения получатель обязан полностью обработать текущее сообщение либо закрыть соединение.

Если два участника цепочки по-разному вычисляют границу, один и тот же хвост байтов для первого будет body, а для второго — новым request. Это не просто parse error, а основа класса request smuggling.

## Как устроено

### Сообщение и его framing

HTTP/1.1-сообщение состоит из start-line, полей, пустой строки и необязательного body. TCP не сохраняет границы записей приложения: один `write` может быть прочитан частями, а несколько `write` — одним `read`. Поэтому длина body выводится только по правилам HTTP, а не по числу прочитанных TCP-сегментов или по временному отсутствию данных.

Для response RFC 9112 задаёт строгий приоритет признаков длины:

1. ответы на `HEAD`, а также `1xx`, `204` и `304` не содержат message body независимо от полей;
2. успешный `2xx` на `CONNECT` превращает соединение в tunnel сразу после header section;
3. если присутствует `Transfer-Encoding`, он имеет приоритет над `Content-Length`; одновременное присутствие полей считается подозрительным и требует специальной обработки;
4. финальный coding `chunked` задаёт конец последовательностью chunks и нулевым chunk;
5. `Transfer-Encoding` с финальным coding, отличным от `chunked`, делает request некорректным, а response ограничивает закрытием соединения;
6. корректный `Content-Length` задаёт точное число октетов;
7. request без этих признаков имеет body длины ноль, а обычный response без них ограничен закрытием соединения.

Close-delimited response принципиально неоднозначен при обрыве: клиент не может отличить корректный конец body от потерянного хвоста. Поэтому для устойчивого reuse нужен self-defined message length — `Content-Length`, chunked framing либо статус, который исключает body.

Sender не должен отправлять `Content-Length` вместе с `Transfer-Encoding`. Получатель запроса с обоими полями может отвергнуть его; если сервер всё же обрабатывает сообщение по `Transfer-Encoding`, после ответа он обязан закрыть соединение. Некорректный или противоречивый `Content-Length` — unrecoverable framing error, кроме строго описанного случая списка одинаковых допустимых значений.

### Chunked transfer coding

`Transfer-Encoding: chunked` позволяет передавать body, когда итоговая длина заранее неизвестна. Каждый chunk несёт размер в hexadecimal, затем данные; нулевой chunk завершает body и может предварять trailers. Chunk boundaries — деталь передачи на этом hop, а не семантические части ресурса.

`Transfer-Encoding` является hop-by-hop: intermediary может декодировать или перекодировать transfer codings, но обязан однозначно обработать входное сообщение и корректно сформировать framing для следующего соединения. Передавать сырой неоднозначный поток дальше опасно, потому что следующий parser может выбрать другую границу.

### Persistent connections и pipelining

HTTP/1.1 использует persistent connections по умолчанию. `Connection: close` сообщает, что после текущего сообщения соединение не должно использоваться повторно. Для reuse сервер должен прочитать весь request либо закрыть соединение, а клиент — прочитать весь response либо отказаться от reuse. Жизненный цикл и ограничения такого reuse раскрыты в [[20 Бэкенд/Пулы соединений и keep-alive|заметке о пулах соединений и keep-alive]].

Pipelining разрешает клиенту отправить несколько requests, не дожидаясь responses. Сервер может параллельно обрабатывать последовательность idempotent requests, но responses обязан отправить в порядке соответствующих requests. Это создаёт application-layer head-of-line blocking: готовый быстрый response не может обогнать медленный первый. Отдельные TCP connections снимают этот порядок ценой дополнительных handshakes и ресурсов; [[10 Основы CS/HTTP-2 и multiplexing|HTTP/2]] решает его streams внутри одного connection.

### Hop-by-hop state и intermediaries

Поле `Connection` перечисляет параметры, действующие только для текущего соединения. Intermediary обязан удалить само поле и все поля, названные в нём, прежде чем переслать сообщение. Кроме того, он должен корректно обработать известные hop-by-hop поля, включая `Keep-Alive`, `TE`, `Transfer-Encoding` и `Upgrade`; неформальное `Proxy-Connection` тоже нельзя слепо пересылать.

Это важно для [[10 Основы CS/Reverse proxy|L7 reverse proxy]]: proxy завершает одно HTTP-соединение и создаёт другое, поэтому обязан отдельно проверить framing на входе и сформировать однозначное сообщение на выходе. На L4-балансировщике HTTP parser отсутствует, и такая нормализация невозможна.

### `Expect: 100-continue` и protocol upgrade

С `Expect: 100-continue` клиент может сначала отправить headers большого request и подождать, примет ли сервер body. Сервер отвечает итоговым status или промежуточным `100 Continue`; клиент не обязан ждать бесконечно и может начать body после собственного timeout. Сервер, увидев expectation и намереваясь читать body, не должен ждать его до отправки `100`, иначе стороны взаимно заблокируются. Поэтому ожидание `100 Continue` — отдельная фаза внутри [[10 Основы CS/Тайм-ауты на сетевых уровнях|сетевого timeout budget]] и общего [[20 Бэкенд/Дедлайны запросов и распространение отмены|дедлайна запроса]].

`Upgrade` вместе с `Connection: upgrade` предлагает перейти на другой протокол. Если сервер отвечает `101 Switching Protocols`, после header section HTTP/1.1 framing прекращается и те же байты соединения разбираются уже новым протоколом. До переключения клиент обязан закончить текущий request; upgrade не создаёт второй независимый канал.

## Пример или трассировка

Клиент отправляет два request по одному persistent connection, не ожидая первый response:

```text
C -> S  GET /slow HTTP/1.1\r\nHost: api.example\r\n\r\n
C -> S  GET /fast HTTP/1.1\r\nHost: api.example\r\n\r\n
```

Сервер завершает `/fast` через 10 ms, а `/slow` — через 500 ms. Несмотря на это, wire order обязан остаться таким:

```text
S -> C  HTTP/1.1 200 OK\r\nContent-Length: 4\r\n\r\nslow
S -> C  HTTP/1.1 200 OK\r\nContent-Length: 4\r\n\r\nfast
```

Наблюдаемый результат: latency второго request не меньше примерно 500 ms, хотя его обработка заняла 10 ms. `Content-Length: 4` позволяет parser точно отделить первый response и затем прочитать второй; без self-defined length первый response пришлось бы завершить закрытием соединения, и второй response по нему передать было бы нельзя.

## Trade-offs

| Выбор | Выигрыш | Цена и граница применимости |
| --- | --- | --- |
| Persistent connection вместо нового connection на request | Меньше TCP/TLS handshakes и latency | Нужны корректный framing, полный drain body, idle lifecycle и защита от stale connection |
| Несколько HTTP/1.1 connections вместо pipelining | Реальный параллелизм без порядка responses одного connection | Больше sockets, handshakes, state в NAT и proxy; нужна граница пула |
| `Content-Length` | Простой и проверяемый framing, возможен reuse | Длина должна быть известна и точно совпадать с body; ошибка ломает синхронизацию parser |
| `chunked` | Streaming без знания полной длины | Дополнительный framing; intermediary должен однозначно обработать transfer coding и заново сформировать framing следующего hop |
| Close-delimited response | Работает без вычисленной длины | Нельзя надёжно отличить завершение от truncation и нельзя продолжить reuse |
| HTTP/2 вместо нескольких HTTP/1.1 connections | Multiplexing и независимый порядок responses | Бинарный framing, HPACK, flow control и общая судьба TCP connection усложняют систему |

## Типичные ошибки

### Считать конец TCP-read концом HTTP body

Неверное предположение: если `read` вернул меньше buffer или временно нет новых байтов, body закончен. Симптом: усечённые bodies, следующий response попадает в предыдущий либо parser зависает. Причина: TCP — byte stream без message boundaries. Исправление: следовать приоритету framing из RFC 9112 и считать EOF границей только для close-delimited response.

### Принимать `Content-Length` и `Transfer-Encoding` независимо

Неверное предположение: оба поля можно сохранить, а downstream сам выберет удобное. Симптом: front proxy и backend видят разное число requests, возможен request smuggling. Причина: расхождение parsers по границе сообщения. Исправление: отвергать неоднозначный request либо строго нормализовать его на каждом HTTP hop; не пересылать одновременно оба framing-сигнала.

### Ожидать параллелизма от pipelining

Неверное предположение: если requests отправлены одновременно, готовые responses вернутся независимо. Симптом: быстрые операции стоят за медленной. Причина: RFC 9112 требует response order, соответствующий request order. Исправление: использовать ограниченный пул HTTP/1.1 connections либо HTTP/2/HTTP/3, если обе стороны их поддерживают.

### Переиспользовать соединение после непрочитанного body

Неверное предположение: достаточно закрыть объект response, не доводя parser до следующей message boundary. Симптом: connection не возвращается в pool, следующий request получает мусор или соединение закрывается. Причина: оставшиеся байты принадлежат предыдущему response. Исправление: полностью прочитать/корректно discard body по правилам библиотеки либо закрыть transport connection.

### Пересылать hop-by-hop поля end-to-end

Неверное предположение: proxy может без разбора скопировать все headers. Симптом: upstream применяет connection options предыдущего hop или получает конфликтующий framing. Причина: `Connection` и названные им поля относятся к конкретной паре peers. Исправление: удалить их, проверить входное сообщение и заново сформировать framing для исходящего hop.

## Когда применять

HTTP/1.1 остаётся подходящим, когда endpoint или intermediary не поддерживает более новый HTTP, важны простота wire-debugging и небольшой concurrency на connection. Для высокой параллельности один HTTP/1.1 connection неудобен: выбирают ограниченный пул либо согласованный HTTP/2/HTTP/3.

При диагностике сначала проверяйте не «пакеты HTTP», а границы сообщений: какой именно пункт RFC определил длину body, все ли hops пришли к одному результату, дочитан ли body перед reuse и не удерживает ли первый response очередь. Это обычно быстрее отделяет framing bug от проблемы TCP, приложения или [[20 Бэкенд/Дедлайны запросов и распространение отмены|истёкшего deadline]].
