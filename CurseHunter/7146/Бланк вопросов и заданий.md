## 1. Базовая семантика channel

Channel value — ссылка на runtime object. Send передаёт значение элемента; последующая мутация исходной переменной не меняет уже отправленный scalar value. Но если элемент содержит pointer, slice, map или другой reference-like descriptor, копируется descriptor, а underlying data остаётся общей. Фраза «каналы убирают shared memory» верна только при соблюдении ownership protocol.

Unbuffered channel — rendezvous: send завершается, когда receiver готов принять value. Buffered channel позволяет send завершиться после помещения value в свободный slot; это decoupling до размера buffer, а не гарантия «асинхронности навсегда».

`len(ch)` и `cap(ch)` — моментальные наблюдения. Они не резервируют slot и не доказывают, что следующая операция не заблокируется.

### Матрица опасных операций

![[90 Вложения/CurseHunter/7146/Кадры/002-channel-state-matrix.jpg|720]]

| Состояние | Receive | Send | Close |
| --- | --- | --- | --- |
| `nil` | блокируется навсегда | блокируется навсегда | panic |
| open, unbuffered | ждёт sender | ждёт receiver | закрывает и будит waiters |
| open, buffered | читает, если buffer не пуст | пишет, если есть slot | закрывает; buffer сохраняется |
| closed | сначала buffered values с `ok=true`, затем zero value и `ok=false` | panic | panic |

Поправка к слайду: receive из closed buffered channel не сразу возвращает zero value. Сначала читаются все ранее отправленные элементы.

### Кто закрывает channel?

Закрытие означает «новых values больше не будет», а не «receiver больше не хочет читать». Поэтому data channel обычно закрывает единственный producer либо coordinator, который дождался всех producers. Receiver не должен закрывать чужой input: это создаёт send-after-close.

Close пустого `chan struct{}` удобно использовать как broadcast: все текущие и будущие receivers получают ready signal. Но повторный close паникует, поэтому lifecycle должен иметь одного owner или `sync.Once`.

## 2. `select`: readiness, fairness и control flow

Если готов ровно один communication case, выбирается он. Если готовы несколько, Go выбирает один uniform pseudo-random case. Порядок cases в исходнике не задаёт priority и не является SLA fairness.

`default` выполняется, только если ни одна communication не может продолжиться немедленно. Это правильный primitive для non-blocking operation:

```go
select {
case v, ok := <-ch:
	return v, ok, true // операция состоялась
default:
	return zero, false, false
}
```

Третий boolean отделяет «операция не готова» от «channel closed». Проверка `len(ch) > 0` перед receive — TOCTOU и не работает для unbuffered channel, где partner может быть готов при `len==0`.

![[90 Вложения/CurseHunter/7146/Кадры/002-nonblocking-toctou.jpg|720]]

Обычный `break` внутри `select` выходит только из `select`, не из surrounding `for`. `continue` относится к ближайшему loop. Для выхода используют `return` или labeled `break`.

### Можно ли задать priority?

Дублирование одного case повышает его статистический вес, но не создаёт строгого priority и усложняет reasoning. Staged select — сначала non-blocking проверка high-priority channel, затем общий blocking select — даёт предпочтение только в момент первой проверки. Для строгого priority нужен явный dispatcher/queue; в обоих вариантах надо отдельно решить starvation low-priority work.

### Когда вычисляется RHS send в `select`?

Channel operands и right-hand-side send expressions вычисляются при входе в `select`, до выбора ready case. Поэтому конструкция

```go
select {
case result <- slowQuery():
default:
}
```

не предотвращает `slowQuery()`: вызов уже выполнен. `default` защищает только сам send после вычисления value.

## 3. Closed channel, `IsClosed` и nil как переключатель

Универсального безопасного `IsClosed(ch)` нет. Non-blocking receive изменяет состояние: он может потребить buffered value. Даже форма `_, ok := <-ch` отвечает на вопрос о closure только ценой receive operation. Sender не может сделать атомарное «проверить, затем отправить»; send/close координируются ownership protocol.

После close receive case всегда ready. Если оставить такой channel в `select`, loop может spin и мешать другим cases. Присваивание локальной channel variable значения `nil` отключает case, потому что операция над nil channel никогда не ready.

![[90 Вложения/CurseHunter/7146/Кадры/002-nil-disables-case.jpg|720]]

Boolean `lhsClosed=true` сам по себе не отключает `case <-lhs`. Нужна именно операция `lhs=nil` либо пересборка control flow.

## 4. Goroutine leaks и deadlocks

### Worker читает `range`, но channel не закрывается

`for v := range in` завершится только после close и drain. Если owner больше не отправляет, но не закрывает channel и не предоставляет cancellation, worker остаётся blocked. Решение — явный owner close либо `context`/done case.

### First-response-wins

Пять goroutines отправляют результат, caller читает один. На unbuffered channel четыре losers остаются blocked. Buffer capacity `4` в показанном исправлении достаточен: один result забирает caller, остальные четыре помещаются в buffer и producers завершаются.

![[90 Вложения/CurseHunter/7146/Кадры/002-first-response-leak.jpg|720]]

У этого исправления узкий invariant: capacity должна покрывать все непрочитанные результаты. Более общий вариант отменяет losers через context либо делает loser send non-blocking. Сам backend call тоже должен принимать context, иначе goroutine перестанет блокироваться на send, но работа продолжит расходовать resources.

### Mutex плюс channel

Если consumer удерживает mutex и ждёт `<-ch`, а producer должен взять тот же mutex до `ch <- value`, возникает cycle: consumer ждёт producer, producer ждёт mutex. Нельзя держать lock через potentially blocking communication без доказанного protocol.

### Последовательное вычисление аргументов

`Process(<-Fetch1(), <-Fetch2())` может занять около четырёх секунд, хотя каждая operation длится две: первый receive должен закончиться до вычисления второго argument. Сначала вызывают обе функции, сохраняют channels, затем читают результаты — тогда operations перекрываются.

### «Channel thread-safe» не относится к переменной

Concurrent sends на одном channel синхронизируются runtime. Но `shared = make(chan int)` — обычная concurrent запись в variable, содержащую channel value. Сто goroutines, переназначающие эту variable, создают data race.

## 5. Когда channel, а когда mutex

Channel удобен для передачи ownership, распределения work, asynchronous results и lifecycle signals. Mutex проще для краткой защиты in-memory state: cache, counters, compound invariants.

Критерий не «что быстрее вообще», а форма ownership:

- state принадлежит одному worker и остальные посылают commands — channel естественен;
- несколько goroutines читают/меняют одну структуру на месте — mutex часто прямее;
- операция должна быть атомарна вместе с несколькими fields — channel protocol или critical section должны охватывать весь invariant.

## 6. Barrier и semaphore

### Reusable barrier курса

Barrier имеет `size`, counter и два buffered turnstiles. `Before` увеличивает count; последний участник кладёт `size` tokens в `beforeCh`. `After` уменьшает count; последний выходящий кладёт tokens в `afterCh`. Второй turnstile не даёт быстрому worker начать следующую фазу, пока предыдущую не закончили все.

![[90 Вложения/CurseHunter/7146/Кадры/003-barrier.jpg|720]]

Failure modes: `size<=0`, пропавший participant, panic между фазами и cancellation приводят к вечному ожиданию. Production barrier должен определить broken-barrier semantics, timeout/cancel и возможность динамического числа участников.

### Semaphore на buffered channel

`Acquire` отправляет empty token в channel capacity `N`, `Release` читает token. Когда buffer заполнен, следующий `Acquire` блокируется — одновременно работают не более `N` holders.

В wrapper обязательно `defer Release()` после успешного Acquire: panic/early return иначе навсегда уменьшит capacity. `N==0` блокирует все calls; лишний `Release` блокируется на чтении пустого channel. Для cancellation Acquire делают через `select` с `ctx.Done()`.

## 7. Graceful shutdown и lifecycle

Курс получает `SIGINT/SIGTERM`, worker select-ит signal и ticker, main ждёт `WaitGroup`. В прикладном коде удобнее `signal.NotifyContext`: root context распространяется в workers, после cancel main прекращает admission, ждёт drain с deadline и освобождает resources.

Важно разделить:

1. stop accepting new work;
2. cancel или drain current work;
3. close owned outputs только после producers;
4. дождаться workers;
5. ограничить shutdown deadline.

Worker с методом `Shutdown`, который делает `close(stopCh); <-doneCh`, ждёт подтверждение завершения. Но показанный метод не idempotent: второй вызов panic на повторном close. Нужны `sync.Once`, explicit state machine или documented single-owner call.

## 8. Первый ответ, fan-in, fan-out и tee

### Первый ответ от replicas

Курс запускает запрос к каждому адресу и принимает один result через buffer capacity 1.

![[90 Вложения/CurseHunter/7146/Кадры/003-first-response.jpg|720]]

Все `Query` всё равно выполняются: RHS send вычисляется до выбора select case. Loser send попадает в `default`, но backend work уже завершён. Empty `addresses` блокирует caller навсегда; errors не моделируются; losers нельзя отменить. Рабочий hedged request принимает context, отменяет лишние calls после приемлемого ответа и обычно запускает hedge после delay, чтобы не умножать load постоянно.

### Fan-in

По goroutine на input пересылают values в общий output; отдельная closer goroutine ждёт `WaitGroup` и закрывает output. Worker не должен закрывать общий channel: он не знает, закончили ли peers. Nil input или ранний выход downstream без cancellation оставляет forwarding goroutine blocked.

### Fan-out

`SplitChannel` курса — один round-robin dispatcher, а не competing-workers fan-out. Он даёт deterministic distribution, но slow consumer одного output блокирует dispatcher и все остальные outputs. Competing workers work-conserving, зато assignment nondeterministic.

### Tee

Один value последовательно отправляется в `out1`, затем `out2`. Медленный или забытый consumer блокирует оба направления. Если нужны независимые subscribers, определяют buffer, drop/backpressure policy и отдельный lifecycle на подписчика.

## 9. Filter, transformer и pipeline

Stage владеет созданным им output и закрывает его после drain input. `filter` пропускает подходящие values, `transformer` меняет каждый value, pipeline соединяет stages.

![[90 Вложения/CurseHunter/7146/Кадры/003-pipeline.jpg|720]]

Главная interview-ловушка — early downstream exit. Если consumer прочитал один результат и ушёл, upstream stages могут навсегда зависнуть на send. Каждый potentially blocking receive/send reusable stage select-ит против общего cancellation.

Parallel pipeline добавляет ещё три решения: bounded concurrency, сохранение/отказ от input order и policy первой ошибки. Fan-out повышает throughput только для подходящего bottleneck; fan-in должен закрыть output после всех workers.

## 10. Done, `OrDone` и bridge

`done` channel — broadcast cancellation. Закрытие отдельного `completed` channel — acknowledgment, что worker действительно закончил cleanup. Это разные события.

В `OrDone` курса receive отменяем, но последующий `out <- value` выполняется без select. Если downstream перестал читать, wrapper блокируется и уже не реагирует на `done`.

![[90 Вложения/CurseHunter/7146/Кадры/003-or-done.jpg|720]]

Надёжная отправка:

```go
select {
case out <- value:
case <-done:
	return
}
```

Кроме того, upstream producer в примере сам не слушает `done`; после остановки `OrDone` он может зависнуть на `in <- value`. Cancellation должна проходить через всю цепочку.

`Bridge(chan chan T)` курса последовательно drain-ит каждый inner channel. Это flattening с сохранением порядка channels, не concurrent fan-in: зависший первый inner не даст перейти ко второму.

## 11. Future и Promise

Future запускает task и возвращает handle с blocking `Get`. Если result channel unbuffered и `Get` никогда не вызывается, task goroutine блокируется на публикации результата. Buffer capacity 1 отвязывает завершение task от наличия waiter, но всё равно нужны cancellation и политика повторных `Get`.

Promise курса выполняет task, записывает `value/error`, затем закрывает `waitCh`; close синхронизирует публикацию с callbacks, ожидающими receive. Надо определить panic handling, число callbacks и lifetime callback goroutines.

В варианте «Future with Promise» комментарий прямо ограничивает `Set` одним goroutine. Поля `promised` и result channel без lock не поддерживают concurrent/repeated setters; двойная публикация может дать race или send/close failure. Production contract обычно разрешает ровно одного setter и делает это структурным invariant.

## 12. Самописный `errgroup`

Группа хранит `WaitGroup`, первую error через `sync.Once` и закрывает `doneCh`. Внешний вид похож на `errgroup`, но running task не получает context, поэтому первая error не останавливает уже начатую работу. Проверка `doneCh` до запуска только отбрасывает некоторые ещё не стартовавшие tasks.

![[90 Вложения/CurseHunter/7146/Кадры/003-errgroup.jpg|720]]

Другой contract risk — concurrent `Go` и `Wait`: положительный `WaitGroup.Add` при нулевом counter должен произойти до `Wait`. API должен запретить новые tasks после начала ожидания либо иметь отдельную state machine. На практике используют `errgroup.WithContext`, а task обязан слушать полученный context.

## 13. `singleflight`

Map `key → *call` объединяет только одновременно выполняющиеся calls с одинаковым key. Первый создаёт call и запускает action, остальные ждут закрытия `call.done`; после completion entry удаляется.

![[90 Вложения/CurseHunter/7146/Кадры/003-singleflight.jpg|720]]

Это suppression дубликатов, не cache: следующий call после удаления снова выполняет action. В реализации курса waiter не может отменить собственное ожидание, а action не принимает shared context. Нужно решить, чей context управляет общей работой, можно ли одному caller отменить её для всех и нужен ли `DoChan`-подобный API.

Курс конвертирует panic action в generic error. Это отдельная semantic, которую нельзя вводить молча: стандартная библиотека/готовый package может иначе обрабатывать panic и `runtime.Goexit`.

## 14. Leaky-bucket rate limiter

`Allow` non-blocking отправляет token в buffer capacity `limit`; full buffer означает reject. Ticker раз в `period/limit` удаляет один token. Получается burst до `limit` и средняя скорость около `limit/period`.

![[90 Вложения/CurseHunter/7146/Кадры/003-rate-limiter.jpg|720]]

Failure modes показанного кода:

- `limit<=0` даёт division by zero либо zero-capacity semantics;
- слишком малый interval может округлиться до `0`, а `time.NewTicker` с non-positive duration паникует;
- background goroutine и ticker никогда не останавливаются — нужен `Close`/context;
- distributed limit не получается: каждый process имеет собственный bucket;
- wall-clock refill, burst и reject policy должны быть частью API contract.

## 15. Внутреннее устройство `hchan`

Урок показывает runtime Go `1.24.2`. Это implementation detail конкретной версии, а не часть language specification.

![[90 Вложения/CurseHunter/7146/Кадры/004-hchan.jpg|720]]

Основные поля `hchan`:

- `qcount`, `dataqsiz`, `buf` — заполнение, capacity и circular buffer;
- `sendx`, `recvx` — позиции записи/чтения;
- `sendq`, `recvq` — очереди blocked goroutines (`sudog`);
- `closed`, element metadata и internal lock.

Основные пути operation:

1. если opposite waiter уже есть, runtime может передать value напрямую;
2. buffered send/receive работает с circular slot и двигает index;
3. если продолжить нельзя, goroutine оформляется как waiter и паркуется;
4. close помечает state и пробуждает waiters: receivers завершаются по close semantics, blocked senders получают panic.

![[90 Вложения/CurseHunter/7146/Кадры/004-direct-handoff.jpg|720]]

Runtime действительно может копировать value прямо в stack waiting receiver. В полном buffered channel receive сначала забирает head, а value waiting sender переносится в освободившийся slot. Эти оптимизации объясняют отсутствие лишней промежуточной очереди, но прикладной код должен опираться на happens-before и documented channel semantics, а не на layout `hchan`.

## Сводка типичных ошибок

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| `len(ch)` резервирует operation | редкий block после «проверки» | TOCTOU | `select` с `default` |
| closed channel можно оставить в loop | CPU spin/starvation | receive всегда ready | после drain присвоить local channel `nil` |
| `IsClosed` ничего не меняет | потерян buffered value | probe сам является receive | ownership protocol, `v, ok` в месте обработки |
| buffer 1 чинит любое first-response-wins | часть producers течёт | непрочитанных results больше capacity | capacity `N-1`, non-blocking send или cancellation |
| done в одном stage отменяет pipeline | upstream всё равно висит | cancellation не распространена | select на каждом blocking boundary |
| `singleflight` кэширует result | повторный call снова дорогой | entry удаляется после flight | отдельный cache и freshness policy |
| channel защищает variable с channel | race при переназначении | runtime синхронизирует object operations, не обычную variable | lock/ownership для variable |
| source order задаёт priority `select` | low-priority case выигрывает | выбор между ready cases pseudo-random | staged selection/очередь |
