# Конкурентность и паттерны

## Как рассуждать на этих задачах

Почти все задачи блока 59–76 проверяют не знание конкретного паттерна, а четыре инварианта:

1. **Ownership:** кто создаёт, закрывает и перестаёт использовать канал или другой mutable resource.
2. **Liveness:** может ли send, receive, `Wait`, lock или внешний вызов остаться без парной операции.
3. **Cancellation:** кто сообщает об остановке и способны ли уже запущенные workers действительно её заметить.
4. **Boundedness:** ограничены ли число goroutines, размер очереди, время ожидания и объём удерживаемой истории.

Слово «потокобезопасно» означает только отсутствие запрещённых concurrent accesses. Оно само по себе не доказывает корректность результата, отсутствие deadlock и ограниченность ресурсов.

## Отмена, таймауты и каналы

### 59. Таймаут для потенциально зависшего запроса

В задаче операция запускается отдельно, а вызывающий код ждёт либо результат, либо `ctx.Done()`:

```go
func simulateRequest(ctx context.Context) (string, error) {
	result := make(chan string, 1)

	go func() {
		// потенциально долгий запрос
		result <- "ok"
	}()

	select {
	case <-ctx.Done():
		return "", ctx.Err()
	case v := <-result:
		return v, nil
	}
}
```

В `main` показан `context.WithTimeout(..., time.Second)`. Важный follow-up: такой wrapper ограничивает ожидание caller, но не обязательно останавливает саму операцию. Если blocking API не принимает `Context`, worker может продолжить работу после возврата.

`result` лучше делать buffered capacity `1`: если timeout выиграл, единственная отправка завершится без receiver и goroutine сможет выйти. Это ограниченное послабление, а не универсальное лечение утечек. Для stream results или нескольких send нужна полноценная cancellation-aware producer loop.

Для HTTP правильная граница проходит через сам запрос:

```go
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
resp, err := client.Do(req)
```

Тогда отмену видит transport. При этом connection, DNS, TLS и server behavior всё равно требуют осмысленных transport/client timeouts, а успешный `resp.Body` нужно закрыть.

### 60. Wrapper с caller deadline и default timeout

Исходная legacy-функция `getDiscount()` спит две секунды и не принимает `Context`. Wrapper запускает её в goroutine, а затем ждёт результат или deadline. Если caller уже передал deadline, его надо сохранить; если нет — добавить локальный default:

```go
func getDiscountWithTimeout(ctx context.Context) (int, error) {
	if _, ok := ctx.Deadline(); !ok {
		var cancel context.CancelFunc
		ctx, cancel = context.WithTimeout(ctx, time.Second)
		defer cancel()
	}

	result := make(chan int, 1)
	go func() { result <- getDiscount() }()

	select {
	case <-ctx.Done():
		return 0, ctx.Err()
	case v := <-result:
		return v, nil
	}
}
```

В показанной промежуточной версии `result` unbuffered. Если timeout срабатывает раньше, worker навсегда блокируется на send. Ещё одна граница решения: wrapper не может насильно отменить `getDiscount`; он лишь не удерживает caller и позволяет worker завершить единственный send.

Не следует закрывать `result` со стороны receiver. Только producer знает, что отправок больше не будет; для одного результата закрытие вообще не требуется.

### 61. Deadlock из-за отправки в канал из его же consumer loop

В исходной программе один goroutine ждёт события в `select`. Ticker case пытается выполнить `ch <- true`, но receiver этого же канала находится в том же `select` и не может одновременно принять значение. Первый tick блокирует loop. Позже worker также блокируется на `ch <- false`.

Неверное предположение → «раз send находится в `select`, он не заблокирует». На деле выбирается готовый receive от ticker, после чего тело case выполняется как обычный последовательный код.

Минимальные варианты исправления:

- обработать tick прямо в соответствующем case, не пересылая его в собственный input channel;
- разделить producer и consumer между разными goroutines;
- сделать result channel buffered, если producer должен иметь право завершить единственную отправку после ухода receiver.

`ticker.Stop()` освобождает ticker resource, но не закрывает `ticker.C`. `WaitGroup.Wait()` безопаснее держать вне receive loop либо запускать отдельного closer, который ждёт producers и только затем закрывает aggregate channel.

## Состояние под нагрузкой

### 62. Top documents: дорогой scorer, cache и locks

Контракт задачи:

- `AddDocument`/update меняет набор документов;
- `GetTopDocuments(user, limit)` вычисляет user-specific score для миллионов документов;
- вызовы конкурентны, scorer может быть дорогим;
- повторные запросы желательно кешировать.

Показанный вариант держит `RLock` во время scoring и сортировки всех документов. Data race нет, но latency записи остаётся высокой: любой update ждёт полный `O(N log N)` read-side critical section.

Практическая схема:

1. под коротким `RLock` снять immutable snapshot документов и generation;
2. отпустить lock;
3. вызвать scorer вне lock;
4. вместо полной сортировки поддерживать min-heap размера `limit`, получая `O(N log limit)`;
5. кешировать результат по `(user, limit, generation)` либо инвалидировать cache при update;
6. публиковать результат только если generation всё ещё допустима для выбранной consistency semantics.

Cache без версии возвращает логически устаревший top после изменения документа. Single global lock вокруг scorer ещё опаснее, если scorer вызывает внешний код: появляются unpredictable latency, reentrancy и возможность lock inversion.

На интервью стоит уточнить:

- нужен snapshot на момент начала или strict latest result;
- зависит ли score только от `(user, document)`;
- разрешены ли stale results и какой TTL;
- каков typical `limit` и можно ли precompute popular users;
- нужна ли стабильная tie-breaking order.

### 63. Парковка: atomic check-then-act всё ещё некорректен

![[90 Вложения/CurseHunter/7171/Кадры/7171-63-atomic-check-then-act.jpg]]

*Кадр урока 63: `Load() == 0` и последующий `Add(-1)` — две отдельные атомарные операции, а не одна транзакция.*

Задача моделирует ограниченное число мест. Наивный `slots--` имеет data race. В разборе предлагается заменить поле на `atomic.Int64`, но последовательность

```go
if p.slots.Load() == 0 {
	return false
}
p.slots.Add(-1)
return true
```

не атомарна как бизнес-операция. Два goroutines могут одновременно увидеть `1`, оба выполнить `Add(-1)` и получить `-1`.

Корректный lock-free acquire — CAS loop:

```go
func (p *Parking) TryEnter() bool {
	for {
		old := p.slots.Load()
		if old <= 0 {
			return false
		}
		if p.slots.CompareAndSwap(old, old-1) {
			return true
		}
	}
}
```

Exit делает `Add(1)`, но production-вариант также должен защищать верхнюю границу: двойной exit не может создать несуществующее место. Если нужна очередь ожидания, fairness и blocking acquire, channel-semaphore или mutex+condition обычно проще доказать, чем CAS.

### 64. Аналитика событий за последние пять минут

Интерфейс получает `HandleEvent(userName, currentTime)` и должен вернуть `GetCount(userName, currentTime)` за sliding window пять минут. Требуются concurrent access, удаление старых событий и быстрый read.

Минимальная точная модель — для каждого пользователя хранить возрастающий список timestamps и индекс/срез живого suffix. На запросе граница равна `currentTime-5m`; старый prefix удаляется, count равен длине suffix.

Нужно заранее определить инварианты:

- timestamps приходят монотонно для каждого пользователя или могут опаздывать;
- окно включает обе границы или полуинтервал `(start, now]`;
- разрешён ли `currentTime` из будущего;
- когда удаляется пустой user entry;
- вызывается ли cleanup только чтениями или background janitor.

Если события могут приходить не по времени, простой prefix trim неверен: потребуется ordered structure или time buckets. При очень высоком RPS хранение каждого timestamp дорого; ring of second/minute buckets уменьшает память ценой временной точности.

В промежуточном коде timestamps читаются после раннего `RUnlock`. Это безопасно лишь если дальнейшие writers не меняют доступный backing array. Копия slice header не даёт immutable snapshot. Надёжнее либо считать/trim под lock, либо копировать нужный диапазон, либо публиковать immutable per-user snapshots.

### 65. Параллельные HTTP-запросы по списку URL

Задача усложняется поэтапно. Сначала запросы идут последовательно и печатают `url — ok` только для `200 OK`. Затем каждый запрос запускается параллельно, а результаты печатает основной goroutine. На следующем этапе число URL заранее неизвестно: нельзя завершать consumer через `for i := 0; i < len(urls); i++`.

Устойчивый lifecycle:

```go
results := make(chan Result)
var wg sync.WaitGroup

for url := range urls {
	wg.Add(1)
	go func(url string) {
		defer wg.Done()
		results <- fetch(ctx, client, url)
	}(url)
}

go func() {
	wg.Wait()
	close(results)
}()

for result := range results {
	// единственный owner вывода
}
```

В реальном решении нельзя создавать новый `http.Client` без необходимости на каждую итерацию: transport предназначен для reuse и connection pooling. Для каждого успешного response нужно закрыть body; если важен reuse соединения, body обычно дочитывают до EOF. `Client.Timeout` ограничивает весь exchange, а request context позволяет общую отмену.

Запуск goroutine на каждый URL без верхней границы не масштабируется на unbounded input. Следующий follow-up превращает задачу в worker pool или semaphore с лимитом concurrency.

### 66. Web crawler: статистика загрузок за десять минут

`OnPageDownloaded(host, time)` получает события из нескольких goroutines, а `Count(host, now)` возвращает число загрузок host за последние десять минут. Это тот же sliding-window механизм, что в задаче 64, но с другой нагрузкой: в примере host может иметь сотни тысяч событий.

Выбор representation зависит от требуемой точности:

- точные timestamps + binary search дают точный count и `O(log n)` поиск границы, но память `O(events in window)`;
- 600 one-second buckets дают ограниченную память на активный host, но погрешность на границе до размера bucket;
- глобальные time buckets удобны для periodic expiration, но требуют аккуратного удаления host с нулевыми counters.

Сначала надо спросить об expected cardinality hosts и RPS: `600 × число всех когда-либо виденных hosts` тоже может стать утечкой. Cleanup — часть контракта, а не косметическая оптимизация.

### 67. Rate limiter с гарантией «в любой интервал длиной одна секунда»

![[90 Вложения/CurseHunter/7171/Кадры/7171-67-strict-rate-limit.jpg]]

*Кадр урока 67: формулировка требует ограничивать любой скользящий интервал, а не календарную секунду.*

Условие требует `Allow() bool`, RPS от `1` до `100000`, concurrent calls и не более RPS успешных запросов **в любом** секундном интервале.

Это важная формулировка. Fixed-window counter не выполняет её: RPS запросов могут пройти в конце одного календарного окна и ещё RPS — в начале следующего. Token bucket допускает burst по определению и тоже не равен строгому sliding-window limit, если capacity больше одного token.

Точные варианты:

- sliding log timestamps: удалить всё `<= now-1s`, разрешить запрос, если осталось меньше limit; просто, но память и работа зависят от RPS;
- ring of small buckets: bounded memory и более дешёвый учёт, но результат аппроксимирован размером bucket;
- distributed limiter: atomic script/transaction в общем store, потому что process-local mutex не ограничивает совокупный RPS replicas.

В видео также сравниваются fixed window, sliding window и token bucket. Для token bucket нужно определить `refillRate`, `capacity` и допустимый burst. Формула `capacity = refillRate × burstDuration` имеет смысл только после согласования burst semantics. Clock следует инъецировать для deterministic tests; wall-clock jumps не должны ломать refill, поэтому duration обычно считают по monotonic component `time.Time` внутри процесса.

Тесты должны проверять не только последовательные calls, но и одновременный старт значительно большего числа goroutines, границу окна, длительный idle и race detector.

### 68. Client-side round-robin balancer

Дан интерфейс `Backend`, десятки экземпляров и требование равномерно выбирать их через Round Robin при concurrent calls.

Базовый выбор индекса можно сделать под mutex или через atomic counter:

```go
i := next.Add(1) - 1
backend := immutableBackends[i%uint64(len(immutableBackends))]
```

Но это решает лишь concurrent rotation. Нужны ответы на follow-ups:

- что делать с пустым списком;
- retry разрешён только до передачи request или операция idempotent;
- пассивно исключать failed backend или нужен active health check;
- как backend возвращается в rotation;
- как обновлять список без race: immutable snapshot через `atomic.Pointer`/`atomic.Value` либо lock;
- нужен ли least-connections/latency-aware выбор вместо равномерного числа запросов.

Round Robin равномерно распределяет **число выборов**, а не стоимость или latency. Медленный instance накапливает больше in-flight work, поэтому при неодинаковой нагрузке least-loaded может быть ближе к цели.

### 69. Первый успешный ответ нескольких поисковых серверов

`Search(servers, query, searchFunc)` запускает поиск на всех серверах, возвращает первый успешный result, а если ошиблись все — error.

Главная ловушка — ранний return оставляет остальных workers на unbuffered send. Result channel capacity `len(servers)` позволяет каждому закончить ровно одну отправку независимо от receiver. Лучше также отменить child context после первого успеха, но это помогает только если `searchFunc` принимает и соблюдает context; исходный `SearchFunc(server, query)` отменить нельзя.

Пустой `servers` нельзя ждать бесконечно — это немедленная ошибка. Чтобы понять, что ошиблись все, consumer считает ровно число запущенных workers или читает канал до закрытия producer-owned closer.

«Первая» означает первую завершившуюся успешную операцию, а не первый server в input. Порядок nondeterministic, и тест не должен требовать конкретный backend при одновременном успехе.

### 70. MultiProcess: гонка асинхронных JobFunc

`MultiProcess(ctx, input, jobs)` параллельно запускает все `JobFunc`. Первый success возвращается; если все jobs завершились с error, условие просит вернуть последнюю error.

Это почти та же задача, что 69, но context входит в `JobFunc`. После успеха wrapper должен вызвать child `cancel`, а workers — передавать context дальше в I/O и проверять его в собственных loops.

Фраза «последняя ошибка» неоднозначна при concurrency:

- последняя по input order;
- последняя по completion time;
- произвольная из полученных;
- aggregate всех ошибок.

Нужно согласовать semantics. Последняя по completion time nondeterministic при simultaneous completion; для diagnostics обычно полезнее aggregate (`errors.Join`) с identity job, чем потеря всех предыдущих causes.

Result channel следует буферизовать по числу jobs или обеспечить cancellation-aware send:

```go
select {
case results <- r:
case <-ctx.Done():
}
```

### 71. Версионирование документов, пришедших не по порядку

Документы группируются по `URL`. Для каждой группы требуется:

- `Text` и `FetchTime` из версии с максимальным `FetchTime`;
- `PubDate` из версии с минимальным `FetchTime`;
- `FirstFetchTime` равен минимальному `FetchTime`;
- сообщения могут идти в любом порядке и дублироваться;
- обновление concurrent.

Удобно хранить два экстремума: `firstFetch` и `lastFetch`. Incoming document сравнивается с обоими под одним per-URL critical section. Это одна логическая операция: два отдельных atomic fields не сохранят согласованную пару `Text+FetchTime`.

Нужно определить tie semantics при одинаковом `FetchTime`, но разных payload: первый wins, последний wins, version ID или content hash. Без этого duplicate/reorder результат зависит от arrival order.

Глобальный mutex прост и корректен, но сериализует разные URL. Sharded map или per-key entry locks уменьшают contention. При per-key lock важно безопасно создать сам entry: lookup-or-create остаётся защищённым map-level lock или `sync.Map.LoadOrStore`.

## Паттерны координации

### 72. Fan-in: объединить N каналов

![[90 Вложения/CurseHunter/7171/Кадры/7171-72-fan-in.jpg]]

*Кадр урока 72: условие fan-in и заготовка функции.*

Нужно реализовать `merge(channels ...<-chan int64) <-chan int64`, а затем добавить `Context` для отмены.

Классическая схема запускает один copier на input и один closer:

```go
func merge(ctx context.Context, inputs ...<-chan int64) <-chan int64 {
	out := make(chan int64)
	var wg sync.WaitGroup
	wg.Add(len(inputs))

	for _, input := range inputs {
		input := input
		go func() {
			defer wg.Done()
			for {
				select {
				case <-ctx.Done():
					return
				case v, ok := <-input:
					if !ok {
						return
					}
					select {
					case out <- v:
					case <-ctx.Done():
						return
					}
				}
			}
		}()
	}

	go func() {
		wg.Wait()
		close(out)
	}()
	return out
}
```

Вторая cancellation check нужна потому, что goroutine может успешно получить input value и затем навсегда ждать consumer на `out`. Ни один copier не закрывает общий output: только отдельный closer знает, что закончились все producers.

### 73. Sharded in-memory cache

Начальный cache — `map[string]string` с одним `RWMutex`. Он корректен, но все keys конкурируют за одну lock domain. Sharded cache выбирает shard стабильным hash и даёт каждому shard собственные map и lock:

```go
type shard struct {
	mu   sync.RWMutex
	data map[string]string
}

func (c *Cache) shardFor(key string) *shard {
	return c.shards[hash(key)%uint64(len(c.shards))]
}
```

Число shards должно быть положительным; hash должен всегда выбирать тот же shard для key. Sharding уменьшает contention между разными shards, но ничего не делает с hot key и добавляет hash overhead. Слишком много shards увеличивает memory overhead и ухудшает locality.

Для resize, global size, clear и snapshot придётся координировать все shards. Locks надо брать в одном порядке, иначе две global operations могут получить deadlock. `RWMutex` не гарантирует, что read-heavy workload быстрее обычного `Mutex`; это проверяется benchmark на реальном access pattern.

### 74. Worker pool для обработки изображений

Есть дорогой `ProcessImage(Task)`, десять tasks и ограничение в три workers. Stable topology:

1. main создаёт `tasks` и `results`;
2. фиксированное число workers читает `tasks` до закрытия;
3. producer или main закрывает `tasks` после последней отправки;
4. отдельный goroutine ждёт `WaitGroup` workers и закрывает `results`;
5. consumer читает `results` до закрытия.

Нельзя ждать workers до начала чтения unbuffered `results`: workers заблокируются на send, а main — на `Wait`. Buffer capacity равный числу tasks случайно скрывает ошибку lifecycle, но не заменяет concurrent draining для unbounded stream.

Production follow-ups: backpressure, per-task error, panic recovery policy, ordering results, graceful shutdown и context-aware receive/send. Worker pool ограничивает concurrency, но очередь также должна быть bounded, иначе overload переедет из goroutines в memory.

### 75. Pipeline финансовых транзакций

Заданы стадии:

1. сгенерировать/read transactions;
2. убрать отрицательные amounts;
3. конвертировать сумму в доллары;
4. сохранить/напечатать результат.

Каждая intermediate stage возвращает receive-only channel, запускает producer goroutine и закрывает **свой output** после исчерпания input:

```go
func filter(in <-chan Transaction) <-chan Transaction {
	out := make(chan Transaction)
	go func() {
		defer close(out)
		for tr := range in {
			if tr.Amount >= 0 {
				out <- tr
			}
		}
	}()
	return out
}
```

Если downstream прекращает читать досрочно, upstream зависает на send. Поэтому reusable pipeline требует `Context`/done channel во всех receive и send points. Для currency conversion надо также спросить, откуда берётся rate, фиксирован ли snapshot rate для всей batch и как проходят ошибки; `float64` обычно неподходящ для денежных инвариантов без определённых rounding rules.

### 76. Semaphore для ограничения загрузок

![[90 Вложения/CurseHunter/7171/Кадры/7171-76-dynamic-semaphore.jpg]]

*Кадр урока 76: в условии отдельно зафиксирован динамически изменяемый лимит.*

Условие: каждый файл загружается отдельным соединением, remote server ограничивает число активных connections, а приложение не должно запускать просто фиксированный worker pool. Показан channel-semaphore:

```go
sem := make(chan struct{}, limit)

sem <- struct{}{} // acquire до запуска работы
go func(file string) {
	defer func() { <-sem }()
	defer wg.Done()
	downloadFile(file)
}(file)
```

Acquire до `go` ограничивает не только активную работу, но и число созданных goroutines. Если acquire сделать внутри goroutine, одновременно активных downloads всё равно будет `limit`, однако ожидающих goroutines может накопиться сколько угодно.

В условии говорится о **динамически изменяемом** лимите, но buffered channel с capacity `const limit` этого не реализует: capacity канала изменить нельзя. Для runtime resize нужен отдельный limiter/controller, который выдаёт permits по mutable target, либо замена limiter instance с чётким lifecycle старых permits. `semaphore.Weighted` поддерживает разные веса, но изменение уже созданного total capacity также не является простой операцией.

Надо определить fairness, поведение при отмене во время ожидания permit и порядок `release`. `defer release` ставится сразу после успешного acquire, иначе error/panic path потеряет token.

## Сравнение ближайших инструментов

| Задача | Ближайший инструмент | Что он гарантирует | Чего не гарантирует |
|---|---|---|---|
| Защитить короткое составное состояние | `sync.Mutex` | mutual exclusion и happens-before через unlock/lock | fairness и bounded wait |
| Много reads, редкие короткие writes | `sync.RWMutex` | параллельные readers при отсутствии writer | ускорение на любом workload |
| Один scalar transition | `sync/atomic` | atomicity конкретной операции | atomicity последовательности `Load → check → Add` |
| Ограничить одновременную работу | channel semaphore | не более `cap` полученных permits | динамический resize и fairness |
| Обработать поток фиксированным числом workers | worker pool | bounded active workers | bounded queue без отдельного лимита |
| Остановить дерево операций | `Context` | propagation cancellation/deadline | принудительную остановку кода, который context игнорирует |
