# Алгоритмические задачи

## Как проходить секцию

Рабочий протокол одинаков для всех 30 задач:

1. Пересказать условие и уточнить контракт: допустимые значения, дубликаты, Unicode, возможность менять вход, единственность ответа.
2. Назвать простой baseline и его сложность. Это фиксирует корректность до оптимизации.
3. Сформулировать инвариант оптимального решения: что уже доказано про обработанный префикс, окно, указатели, стек или deque.
4. Проговорить edge cases до кода: пустой ввод, один элемент, `k = 0`, `k > n`, отрицательные значения, overflow суммы.
5. Написать код, пройти его на минимальном и граничном примере, затем явно назвать time/space complexity.

Главная собеседовательная ошибка — узнать название паттерна, но не проверить его предусловия. Например, динамическое sliding window для минимальной длины с суммой не меньше `k` корректно на положительных числах, но ломается при отрицательных: после расширения сумма уже не обязана монотонно расти.

## Массивы и срезы

### 1. Слияние двух отсортированных массивов in-place

**Условие.** Первый массив имеет длину `m+n`: первые `m` позиций заняты, последние `n` зарезервированы. Второй массив содержит `n` элементов. Заполнить первый массив отсортированным объединением.

**Инвариант.** Позиции правее `write` уже содержат наибольшие элементы итогового массива в правильном порядке. Сравниваем хвосты двух ещё не слитых префиксов и записываем больший элемент справа. Движение слева потребовало бы сдвигов; движение справа не затирает непрочитанные значения.

Если первый префикс закончился, остаток второго надо скопировать. Если закончился второй, остаток первого уже стоит правильно. Полный разбор есть в [[CurseHunter/7253/Бланк вопросов и заданий#3. Merge Sorted Array — LeetCode 88|курсе 7253]].

### 2. Стабильное удаление нулей

**Уточнить:** надо вернуть новый срез или новую логическую длину? Разрешено ли менять вход? Что должно остаться в хвосте backing array?

`write` указывает на следующую позицию результата. Каждый ненулевой элемент переносим в `a[write]` и увеличиваем `write`. Инвариант: `a[:write]` — стабильная фильтрация уже просмотренного префикса. Если API требует физически очистить хвост, выполняем `clear(a[write:])`; если вернуть укороченный срез — `a[:write]`.

Нельзя удалять элементы через `append(a[:i], a[i+1:]...)` внутри прямого цикла без управления индексом: после сдвига следующий элемент перескакивает, а повторные сдвиги дают `O(n²)`.

### 3. Произведение элементов, кроме текущего

**Условие.** Для каждого `i` вернуть произведение всех `nums[j]`, где `j != i`, без деления.

Первый проход записывает в `ans[i]` произведение слева от `i`. Второй идёт справа, умножает `ans[i]` на накопленное произведение справа и обновляет accumulator. Нули обрабатываются автоматически: один ноль оставляет ненулевой результат только на его позиции, два нуля обнуляют всё.

Сложность `O(n)` и `O(1)` дополнительной памяти, если output не считать. На собеседовании стоит спросить про integer overflow: Go не сигнализирует переполнение обычных integer operations, поэтому тип результата является частью контракта.

### 4. Найти все отсутствующие числа из `[1,n]`

**Предусловия in-place решения:** длина массива равна `n`, каждое значение лежит в `[1,n]`, вход можно менять.

Для каждого значения `v` берём индекс `abs(v)-1` и делаем значение на этой позиции отрицательным. После прохода положительное `nums[i]` означает, что `i+1` не встретилось. Повтор обрабатывается идемпотентно: ставим `-abs(nums[idx])`, а не просто меняем знак.

Если вход immutable или диапазон не гарантирован, честная альтернатива — `map[int]struct{}` за `O(n)` памяти. Попытка применить marking без проверки диапазона приводит к panic по индексу.

### 5. Вращение массива

Для правого вращения длины `n` нормализуем `k %= n`, затем разворачиваем весь массив, первые `k` элементов и остаток. Доказательство удобно строить на блоках: `[A B] → reverse(B) reverse(A) → B A` после двух локальных разворотов.

Сначала надо обработать `n == 0`, иначе `k % n` panic. Для отрицательного `k` контракт либо запрещает значение, либо нормализует его как `((k % n) + n) % n`. Альтернатива — cycle replacement через `gcd(n,k)`: тоже `O(n)`/`O(1)`, но сложнее доказать и реализовать без пропусков.

## Связные списки

### 6. Удалить `n`-й узел с конца

Dummy node перед головой убирает отдельный случай удаления первого узла. Сдвигаем `fast` на `n+1` шагов от dummy, затем двигаем `fast` и `slow` вместе. Когда `fast == nil`, `slow.Next` — удаляемый узел.

**Уточнить:** что делать при `n <= 0`, пустом списке или `n > length`? В production API лучше вернуть ошибку или исходный список; молчаливый panic скрывает нарушение предусловия.

### 7. Развернуть связный список

На каждой итерации сохраняем `next := cur.Next`, направляем `cur.Next = prev`, затем сдвигаем `prev, cur = cur, next`. Инвариант: `prev` — уже развёрнутый префикс, `cur` — первый непрочитанный узел.

Рекурсивный вариант выразителен, но использует `O(n)` stack и может переполнить его на длинном списке; Go не гарантирует tail-call optimization. Для production-контракта iterative вариант надёжнее.

### 8. Найти цикл — Floyd's Cycle Detection

`slow` идёт на один узел, `fast` — на два. В конечном ациклическом списке `fast` достигнет `nil`; в цикле относительная скорость равна одному, поэтому указатели встретятся. Это отвечает только на вопрос о наличии цикла.

Follow-up: чтобы найти вход в цикл, после встречи переставить один указатель в head и двигать оба по одному шагу. Точка следующей встречи — вход. Чтобы найти длину цикла, после первой встречи сделать полный круг одним указателем.

### 9. Слить два отсортированных списка

Dummy node хранит начало результата, `tail` — последний присоединённый узел. Выбираем меньший из текущих узлов двух списков и переставляем ссылку; когда один список закончился, цепляем остаток второго целиком.

Уточнить tie policy: выбор левого узла при равенстве делает merge стабильным относительно источников. Если входные списки нельзя мутировать, надо создавать новые узлы и признать `O(m+n)` памяти.

### 10. Удалить дубликаты из отсортированного списка

Для варианта «оставить по одному» сравниваем `cur.Val` и `cur.Next.Val`: при равенстве перепрыгиваем `cur.Next`, иначе двигаем `cur`. Отсортированность важна: только она делает все копии значения смежными.

На собеседовании уточнить, не требуется ли вариант «удалить все значения, встречавшиеся более одного раза». Это другая задача: нужен dummy, pointer перед группой и пропуск всей группы дубликатов.

## Строки и стек

### 11. Проверить, являются ли строки анаграммами

Сначала зафиксировать алфавит и семантику: ASCII bytes, Unicode code points или пользовательские grapheme clusters; учитывать ли case, пробелы и Unicode normalization. Для lowercase Latin достаточно массива `[26]int`; для произвольных code points — `map[rune]int`.

Частотное решение `O(n)` не означает автоматически меньшую константу для любого Unicode. Сортировка `[]rune` даёт `O(n log n)`, зато проще при уже доступной сортировке. Базовый вариант и ограничения разобраны в [[CurseHunter/7252/Бланк вопросов и заданий#4. Valid Anagram — LeetCode 242|курсе 7252]].

### 12. Реверс слов с сохранением палиндромов

![[90 Вложения/CurseHunter/7171/Кадры/7171-101-reverse-words.jpg]]

*Кадр урока 101: полное условие и примеры для латиницы и Unicode.*

**Условие курса.** Строка содержит слова, разделённые одиночными пробелами. В каждом непалиндромном слове нужно развернуть буквы, а палиндром оставить без изменения. Порядок самих слов не меняется. Пример: `"привет мир ара" → "тевирп рим ара"`.

Показанное решение делает `strings.Split(s, " ")`, для каждого слова проверяет palindrome и разворачивает `[]rune`. Время `O(n)`, дополнительная память `O(n)` из-за массива слов, rune conversions и нового результата. Палиндром проверяется на тех же runes, иначе контракт внутренне противоречив.

Follow-up: если разделители могут быть несколькими пробелами или tabs/newlines, `strings.Fields` нормализует их, но не сохраняет исходное форматирование. Для точного сохранения separators нужен scanner по rune boundaries.

### 13. Валидировать скобочную последовательность

Opening brackets кладём в stack. Для closing bracket stack обязан быть непустым, а вершина — парной открывающей скобкой; после прохода stack обязан опустеть. Ранний reject возможен при нечётной длине, если алфавит состоит только из однобайтовых brackets.

Нужно спросить, допустимы ли прочие символы: игнорировать их или считать ошибкой. Ошибка `pop` из пустого stack — типичный симптом проверки пары до проверки длины.

### 14. Самая длинная подстрока без повторений

Храним `last[x]` как позицию после последнего вхождения символа. Для правой границы `right` обновляем `left = max(left, last[x])`, затем `last[x] = right+1`. Инвариант: окно `[left,right]` не содержит повторов, а `left` никогда не движется назад.

В Go `range` по строке возвращает byte offsets, а не последовательные rune indices. Если результат измеряется в characters, удобнее работать с `[]rune`; если в bytes — явно сказать об этом. Grapheme clusters требуют отдельной Unicode-библиотеки.

### 15. Найти все анаграммы шаблона в строке

Окно имеет фиксированную длину шаблона. Поддерживаем частоты добавляемого правого и удаляемого левого символа; при равенстве frequency vectors сохраняем левый индекс. Сравнивать всю таблицу на каждом шаге допустимо для фиксированного алфавита, а для большого алфавита лучше поддерживать число mismatched counters.

Индексы результата снова зависят от контракта: LeetCode возвращает byte-compatible indices для lowercase ASCII; Unicode-версия должна определить, нужны byte offsets или rune positions.

## Хеш-таблицы и агрегация

### 16. Two Sum

Перед добавлением `nums[i]` ищем `target-nums[i]` среди уже просмотренных значений. Такой порядок не позволяет использовать один индекс дважды. Инвариант: map содержит индекс каждого значения из строгого префикса `nums[:i]`.

Уточнить число ответов, поведение при дубликатах и overflow `target-nums[i]`. Полный разбор вариантов одного ответа и всех пар есть в [[CurseHunter/7252/Бланк вопросов и заданий#1. Two Sum — LeetCode 1|курсе 7252]] и [[Авито/Решения/Алгоритмы/Поиск пар в списке чисел|решении задачи Авито]].

### 17. Сгруппировать элементы по критерию

**Условие курса.** Для `[]ServerStat{Server int, Stability float64}` вернуть `map[string][]int`: string-представление стабильности → ID серверов с таким значением. В показанном решении key вычисляется через `strconv.FormatFloat(stat.Stability, 'f', -1, 64)`, затем `Server` добавляется в `groups[key]`.

Append сохраняет input order внутри каждой группы, но Go не гарантирует iteration order самой map. Для deterministic JSON/test keys надо сортировать или хранить порядок отдельно.

Главный modeling follow-up: `float64` как category key. `FormatFloat(..., -1, 64)` даёт кратчайшую строку, которая round-trip восстанавливает ровно тот же binary float, но математически ожидавшиеся одинаковыми метрики после разных вычислений могут отличаться на младших bits и попасть в разные группы. Если стабильность задана с фиксированной точностью, лучше хранить integer basis points/decimal value или явно нормализовать по согласованному rounding rule.

### 18. Отфильтровать элементы по множеству

**Условие курса.** `sellers map[int][]string` сопоставляет продавца всем городам работы, а `cities []string` задаёт интересующий набор. Вернуть для каждого продавца только города, входящие в `cities`. Продавец без совпадений в result не появляется.

Строим `map[string]struct{}` из `cities`, затем проходим списки продавцов. Время `O(len(cities)+общее число записей sellers)`, память `O(len(cities)+размер result)`. Порядок городов каждого продавца сохраняется, дубликаты исходного списка тоже сохраняются. Если нужны set semantics, result придётся отдельно дедуплицировать.

Map iteration делает порядок продавцов nondeterministic. Это не влияет на map result, но влияет на сериализацию/тесты без canonical sorting.

### 19. Определить чемпионов

![[90 Вложения/CurseHunter/7171/Кадры/7171-108-find-champions.jpg]]

*Кадр урока 108: входная структура и критерий участия во всех днях.*

**Условие курса.** `statistics [][]UserSteps` хранит отдельный список участников для каждого дня. Чемпионом может быть только пользователь, участвовавший **во всех** днях. Среди них вернуть все ID с максимальной суммой шагов и саму сумму в `Champions{UserIds, Steps}`. Пример из видео: два дня, user `1` набрал `2000+3500`, user `2` — `1500+4000`; оба возвращаются с `Steps=5500`.

Нужны две map: total steps и число **различных дней** участия. Затем рассматриваются только users с `days == len(statistics)`, находится maximum и все ties. Простое `days[user]++` на каждую запись неверно, если один user может встретиться дважды в одном daily slice: дубли надо агрегировать/дедуплицировать внутри дня.

Пустая статистика, отрицательные steps и deterministic order champions требуют отдельного контракта. Если events могут повторяться, нужен event ID или иной idempotency key, иначе повторная доставка удвоит счёт.

### 20. Первый уникальный символ

Первый проход считает частоты, второй в исходном порядке возвращает первый символ с count `1`. Один проход недостаточен без дополнительной структуры кандидатов, потому что поздний повтор может инвалидировать раннего лидера.

Для строки надо согласовать, возвращается byte offset или rune index. `range` даёт byte offset; это часто правильный API для slicing, но не порядковый номер Unicode code point.

## Два указателя

### 21. Найти `k` ближайших элементов

Для отсортированного массива ответ образует contiguous window длины `k`. Binary search ищет его левую границу `left` в `[0,n-k]`: если `x-a[mid] > a[mid+k]-x`, окно надо сдвинуть вправо, иначе оставить левее. Это даёт `O(log(n-k)+k)`.

Уточнить tie rule: стандартный контракт предпочитает меньший элемент, поэтому при равенстве не сдвигаемся вправо. Альтернатива — binary search insertion point и расширение двух указателей за `O(log n+k)`; она проще объясняется, но требует аккуратного выхода за границы.

### 22. Контейнер с максимальной водой

Площадь равна `(right-left) * min(h[left],h[right])`. Двигаем границу с меньшей высотой: сохранение её и уменьшение ширины не может дать большую площадь, какая бы высота ни стояла с другой стороны. При равенстве можно двигать любую или обе.

Именно этот dominance argument отличает доказанное greedy решение от эвристики. Полный разбор есть в [[CurseHunter/7253/Бланк вопросов и заданий#12. Container With Most Water — LeetCode 11|курсе 7253]].

### 23. Минимальная разница среди `k` элементов

Сортируем числа. Любое выбранное множество размера `k` имеет minimum и maximum; оптимальный набор между этими границами можно заменить на `k` смежных элементов sorted order без увеличения range. Поэтому сканируем окна и минимизируем `a[i+k-1]-a[i]`.

Проверить `k <= 0`, `k > n`, `k == 1` и overflow вычитания. Если порядок входа нельзя менять, сортируем копию и признаём `O(n)` памяти.

### 24. Минимальная разница между элементами двух массивов

Сортируем оба массива, ставим по указателю в начало. Обновляем `abs(a[i]-b[j])`, затем двигаем указатель меньшего значения: только его увеличение может уменьшить разницу; движение большего не поможет. При нулевой разнице можно завершить сразу.

Нельзя вычислять `abs` через переполняющееся вычитание для крайних signed integers. Безопасный вариант зависит от допустимого диапазона: wider type, checked subtraction или unsigned distance.

### 25. Удалить дубликаты из отсортированного массива in-place

`write` — длина уникального префикса. Для каждого `read` сравниваем элемент с `a[write-1]`; новое значение записываем в `a[write]` и увеличиваем `write`. Возвращаем `write` или `a[:write]` по контракту.

Отсортированность — обязательное предусловие. Для unsorted массива нужны set и `O(n)` память либо предварительная сортировка с изменением порядка.

## Скользящее окно

### 26. Максимум в каждом окне

Deque хранит indices в порядке убывания значений. Перед добавлением `i` удаляем с хвоста все не большие `a[i]`: они моложе и не хуже, поэтому больше никогда не станут максимумом. С головы удаляем indices вне окна. Каждый индекс входит и выходит не более одного раза, значит время `O(n)`.

Уточнить поведение при `k <= 0` и `k > n`. Хранить значения вместо indices недостаточно: без индекса нельзя надёжно определить, вышел ли максимум из окна.

### 27. Минимальное окно, покрывающее шаблон

`need` хранит multiplicity символов шаблона, `missing` — сколько обязательных вхождений ещё не покрыто. Расширяем right; когда `missing == 0`, сжимаем left, сохраняя лучший диапазон, пока удаление очередного символа снова не создаст deficit.

Нельзя свести задачу к set membership: повторяющиеся символы шаблона требуют counts. Пустой шаблон и Unicode semantics надо определить отдельно.

### 28. Выбрать отпуск с минимумом встреч

![[90 Вложения/CurseHunter/7171/Кадры/7171-117-optimal-vacation.jpg]]

*Кадр урока 117: горизонт `P`, длина отпуска `V` и ожидаемый результат `[5, 0]`.*

**Условие курса.** Уже завтра — день `1`. В ближайшие `periodLength=P` дней нужно полностью разместить отпуск длиной `vacationLength=V` и вернуть `[день начала X, число пропущенных встреч Y]`. `daysWithMeetings` отсортирован и содержит пары `(day, meetings)`. В примере `P=30`, `V=7`; ответ `[5,0]`.

Показанный подход строит dense array `meetingsByDay` длиной `P+1`, суммирует окно дней `1..V`, затем перебирает starts `2..P-V+1`: добавляет новый правый день и вычитает ушедший левый. Время `O(P+D)`, память `O(P)`, где `D=len(daysWithMeetings)`. Условие «весь отпуск в горизонте» важно: последний допустимый start равен `P-V+1`.

Нужно проверить `V<=0`, `V>P`, события с `day<1`/`day>P`, duplicate days и tie rule. В видео сохраняется первый minimum, если обновлять ответ только при строгом `<`.

### 29. Максимальная сумма подмассива размера `k`

Суммируем первое окно, затем двигаем его за `O(1)` на шаг. Инициализировать best нулём нельзя, если все числа отрицательные: best должен начинаться с суммы первого валидного окна.

Для больших значений нужен тип суммы шире типа элемента. При `k <= 0` или `k > n` API должен вернуть ошибку/признак отсутствия, а не случайное число.

### 30. Минимальная длина подмассива с суммой не меньше `k`

![[90 Вложения/CurseHunter/7171/Кадры/7171-119-min-subarray.jpg]]

*Кадр урока 119: условие требует сумму `>= k`, положительные числа и сложность `O(n)`.*

**Условие курса.** Дан массив **положительных** целых чисел. Найти длину кратчайшего непустого contiguous subarray с суммой `>= k`; если его нет — вернуть `0`. Пример `[2,3,1,2,4,3], k=7 → 2` за счёт `[4,3]`. Это LeetCode 209, а не поиск точного равенства сумме.

Расширяем `right`, добавляя `nums[right]`. Пока `sum >= k`, обновляем minimum и удаляем `nums[left]`, пытаясь сжать окно. Положительность обеспечивает монотонность: расширение не уменьшает сумму, удаление слева не увеличивает её. Каждый элемент добавляется и удаляется не более одного раза, поэтому время `O(n)`, память `O(1)`.

При нулях алгоритм остаётся корректным, хотя условие курса говорит о положительных числах. При отрицательных доказательство исчезает; prefix hash map из задачи «точная сумма» тоже не решает условие `>= k`. Общий вариант для integers использует prefix sums и monotonic deque за `O(n)`.

Для `k<=0` mathematically любой непустой positive subarray подходит, но API должен явно вернуть `1` или отклонить input; пустой subarray в условии не является кандидатом.
