## Урок 1. Микроархитектура процессора

### Что именно поставлено в курсе

#### 1. Какие недостатки у организации чипсета через общую системную шину?

![[90 Вложения/CurseHunter/6817/Кадры/01-chipset-tradeoffs-00-15-00.jpg]]

На слайде сопоставляются схемы с CPU, memory controller, northbridge/southbridge и общей шиной. Сильный ответ начинается не с названий исторических мостов, а с потока данных:

1. несколько CPU и устройств конкурируют за общий канал;
2. пропускная способность и арбитраж общей шины становятся общим пределом;
3. удалённость memory controller увеличивает latency доступа к RAM;
4. масштабирование числом процессоров усиливает contention, поэтому современные системы переносят контроллер памяти ближе к CPU и используют point-to-point interconnects.

**Интервью-применение:** объяснить, почему добавление CPU не даёт линейного ускорения memory-bound workload и почему topology нельзя сводить к числу ядер.

#### 2. Где применяются DRAM и SRAM и почему?

Причинная цепочка ответа:

- DRAM хранит бит через заряд в конденсаторе, нуждается в refresh, но даёт высокую плотность и меньшую цену на бит — поэтому подходит для основной памяти;
- SRAM хранит состояние в статической ячейке, не требует refresh при наличии питания, быстрее, но занимает больше площади — поэтому используется в небольших быстрых структурах, прежде всего в cache;
- практический вывод для разработчика: доступ к RAM на порядки дороже регистрового или cache hit, а последовательный доступ и locality обычно важнее «красивой» абстракции данных.

#### 3. Как процессор реализует сравнения и инструкции управления?

![[90 Вложения/CurseHunter/6817/Кадры/03-flags-00-57-00.jpg]]

Курс подводит к регистру флагов x86. Арифметическая или специальная compare-инструкция обновляет признаки результата, например zero, sign, carry и overflow. Условный переход проверяет нужную комбинацию флагов. В ответе важно не смешивать:

- carry — перенос в беззнаковой арифметике;
- overflow — выход знакового результата за представимый диапазон;
- sign и zero — свойства представленного результата, а не самостоятельное сравнение высокоуровневых значений.

#### 4. Как процессор понимает, какую инструкцию исполнять дальше?

Program counter / instruction pointer связывает текущий control flow с адресом инструкции. Обычное последовательное исполнение продвигает его к следующей инструкции, а branch, call, return, interrupt или exception меняют траекторию. Для Senior-ответа этого недостаточно: нужно добавить, что современный CPU заранее выбирает и исполняет инструкции спекулятивно, а архитектурно видимый порядок восстанавливается при commit/retirement.

#### 5. Какие hazards возникают в pipeline?

![[90 Вложения/CurseHunter/6817/Кадры/02-pipeline-00-33-00.jpg]]

Курс выделяет три класса:

- **data hazard** — инструкция зависит от результата, который ещё не готов; применяются forwarding, stall или переупорядочивание;
- **structural hazard** — нескольким стадиям одновременно нужен один аппаратный ресурс; помогают дублирование ресурса или stall;
- **control hazard** — следующий адрес зависит от ещё не разрешённого branch; применяются prediction и speculative execution, а ошибка предсказания требует отбросить неверный путь.

**Интервью-применение:** связать branch misprediction и cache miss с наблюдаемым ростом latency, но не приписывать каждой медленной функции одну из этих причин без профиля и измерения.

#### 6. Чем SIMD отличается от SMT / Hyper-Threading?

- SIMD выполняет одну инструкцию над несколькими элементами данных и требует data-level parallelism.
- SMT позволяет одному физическому core поддерживать несколько hardware threads, чтобы занимать простаивающие execution units независимой работой.
- SIMD ускоряет подходящую операцию внутри потока; SMT повышает utilization ядра, но потоки делят его ресурсы и могут мешать друг другу.

#### 7. Что не так с конкурентным Go-кодом, который ждёт `done` и затем читает `name` без синхронизации?

![[90 Вложения/CurseHunter/6817/Кадры/04-go-memory-model-00-42-00.jpg]]

В демонстрации одна goroutine записывает `name` и `done`, а другая циклически читает `done`, после чего использует `name`. Это data race: обычные чтения и записи не образуют synchronizes-before / happens-before. Нельзя доказывать корректность рассуждением «сначала в исходнике записали `name`, потом `done`». Исправление должно ввести синхронизацию — channel, mutex или атомарный протокол с доказанным порядком публикации данных. Простая замена только `done` на произвольный atomic-флаг требует отдельно объяснить публикацию остальных данных.

### Что проверяет хороший ответ по уроку 1

Кандидат должен уметь пройти цепочку `аппаратный механизм → наблюдаемая задержка или ordering → следствие для Go-кода → способ измерения`. Перечислить термины `cache`, `pipeline`, `SIMD` и `Hyper-Threading` недостаточно: важны различия механизмов, общий ограничивающий ресурс и условия, при которых оптимизация вообще применима.

## Урок 2. Внутреннее устройство оперативной памяти

Отдельного слайда со списком собеседовательных вопросов в уроке нет. Ниже вопросы реконструированы из механизмов, которые лектор разбирает на экране. Это **интервью-применение**, а не приписываемые автору дословные формулировки.

### 1. Как виртуальный адрес превращается в физический?

![[90 Вложения/CurseHunter/6817/Кадры/05-page-table-single-00-27-00.jpg]]

В минимальной модели виртуальный адрес делится на virtual page number и offset внутри страницы. Page table entry (PTE), найденная по номеру виртуальной страницы, содержит номер физического page frame и атрибуты доступа. Offset при переводе не меняется. Поэтому для страницы `4 KiB = 2^12` младшие 12 бит адреса — offset, а остальные биты участвуют в выборе записи таблицы страниц.

Полезно проговорить границы этой модели:

- размер и число уровней page table зависят от ISA и режима адресации;
- PTE хранит frame number вместе с признаками присутствия, прав доступа, обращения и модификации; точный набор битов архитектурно-зависим;
- virtual address принадлежит address space процесса, поэтому одинаковые виртуальные адреса разных процессов могут отображаться в разные физические страницы.

### 2. Зачем нужны многоуровневые таблицы страниц, если одноуровневая проще?

![[90 Вложения/CurseHunter/6817/Кадры/06-page-table-multilevel-00-33-00.jpg]]

Линейная таблица обязана иметь слот для каждой virtual page, даже если большая часть адресного пространства не используется. Иерархия позволяет не создавать нижние уровни для пустых диапазонов. Цена экономии — несколько зависимых чтений при page walk, дополнительная память под узлы и более сложное обслуживание отображений.

TLB и page-walk caches компенсируют эту цену на повторных обращениях. Huge page может завершить walk на более высоком уровне: один entry покрывает больший диапазон и нижние уровни для него не нужны.

### 3. Чем TLB miss отличается от page fault?

![[90 Вложения/CurseHunter/6817/Кадры/07-mmu-tlb-00-57-00.jpg]]

Это разные события:

1. CPU передаёт virtual address в MMU.
2. При TLB hit готовый перевод берётся из TLB.
3. При TLB miss аппаратный page walk может найти корректную present-запись в page tables, заполнить TLB и продолжить исполнение — перехода в ядро и чтения с диска здесь не требуется.
4. Page fault возникает, если перевод требует участия ОС: страница не отображена или не присутствует, нарушены права либо нужно реализовать lazy allocation / copy-on-write.
5. Диск участвует только в некоторых fault-сценариях, например когда содержимое действительно было вытеснено в swap или загружается file-backed page.

Типичная ошибка — объяснять любой TLB miss как «ОС загружает страницу с диска». Симптом такого ответа: смешаны cache miss перевода, exception и I/O. Исправление — отдельно пройти fast path, аппаратный page walk и fault path.

### 4. Когда huge pages ускоряют программу и какой ценой?

Одна TLB entry покрывает больше памяти, уменьшаются TLB pressure и объём нижних уровней page tables. Это особенно полезно для большого, плотно используемого working set. Выигрыш не бесплатный:

- крупнее внутренняя фрагментация, если отображённый диапазон используется частично;
- сложнее найти физически непрерывный большой блок, возможны allocation/compaction costs;
- fault, zeroing и copy-on-write затрагивают больший объём и могут дать заметный latency spike;
- фактический эффект зависит от access pattern, архитектуры и политики ОС, поэтому его подтверждают профилем и измерением TLB misses, а не включают по умолчанию.

Transparent Huge Pages (THP) автоматизируют promotion/demotion, но не отменяют этих trade-offs. Для критичного сервиса нужно проверить режим THP, фактическое покрытие отображений и хвосты задержки.

### 5. Почему последовательный доступ к DRAM обычно выгоднее случайного?

В уроке показаны banks, row/column addressing и чтение памяти cache lines. После активации строки данные оказываются в row buffer. Повторное обращение к той же открытой строке в этом банке дешевле, чем row conflict, при котором прежнюю строку нужно закрыть и активировать другую. Независимые банки, напротив, дают контроллеру возможность перекрывать часть работы. На уровне программы эта закономерность соединяется с cache locality: плотный обход чаще переиспользует уже загруженные cache lines и лучше подходит для prefetching.

Сильный ответ не обещает одинаковую цену каждого «соседнего» чтения: memory controller переупорядочивает запросы, банки работают параллельно, а итог зависит от cache hits, TLB, NUMA topology и конкуренции.

### Что проверяет хороший ответ по уроку 2

Кандидат не останавливается на определении virtual memory. Он умеет вычислить число offset bits, объяснить экономию многоуровневой таблицы, разделить TLB miss и page fault и назвать workload, на котором huge pages одновременно помогают производительности и создают риск для latency или памяти.

## Урок 3. Внутреннее устройство кэша процессора

### Что именно поставлено в курсе

#### 1. Каковы плюсы и минусы look-aside и look-through cache?

![[90 Вложения/CurseHunter/6817/Кадры/08-cache-topology-00-03-00.jpg]]

На слайде вопрос `Pros and cons?` поставлен прямо.

- При **look-aside** cache наблюдает тот же запрос, что и memory controller, и lookup можно начать параллельно с обращением вниз. Hit потенциально обнаруживается без дополнительного последовательного звена, но запрос уже занимает общий interconnect; cache должна вовремя подавить или завершить обращение к памяти. Растут traffic, сложность арбитража и требования к timing.
- При **look-through** cache стоит на пути запроса: hit отсекает обращение к memory controller, а miss пересылается дальше. Это уменьшает downstream traffic и локализует cache policy, но lookup становится последовательным звеном miss path, а сама cache может стать bottleneck.

Правильный выбор зависит не от абстрактного «быстрее», а от topology, hit rate, latency lookup, пропускной способности interconnect и числа участников.

#### 2. Чем отличаются direct-mapped, fully associative и set-associative cache?

- Direct-mapped даёт каждому memory block ровно одну возможную line: lookup дешёвый, но конфликтующие адреса могут постоянно вытеснять друг друга.
- Fully associative разрешает разместить block в любой line: минимум conflict misses, но дорогой параллельный поиск и replacement metadata.
- Set-associative ограничивает block одним set, но оставляет несколько ways внутри него. Это практический компромисс между lookup cost и conflict misses.

На интервью одних определений мало. Нужно по схеме `tag | set index | line offset` определить, какие адреса конфликтуют, а затем объяснить, почему увеличение associativity снижает один класс misses, но повышает цену lookup и replacement.

#### 3. Зачем нужна cache coherence и как она влияет на программу?

![[90 Вложения/CurseHunter/6817/Кадры/10-cache-coherence-00-57-00.jpg]]

Оба вопроса поставлены на слайде прямо. Private caches могут хранить копии одной cache line. Coherence protocol отслеживает ownership и допустимость копий; перед записью ядро обычно получает линию в состояние, разрешающее модификацию, а конкурирующие копии инвалидируются или обновляются по правилам конкретного протокола.

Практическое следствие — **false sharing**. Потоки изменяют разные логические поля, но если поля лежат в одной cache line, ownership линии передаётся между ядрами. Data race может не быть, а throughput всё равно падает из-за coherence traffic.

![[90 Вложения/CurseHunter/6817/Кадры/09-false-sharing-benchmark-00-48-00.jpg]]

В записанном запуске на `darwin/arm64`, Apple M3 Pro и Go `1.23.4` вариант с padding показал `10.41 ns/op`, без padding — `454.1 ns/op`; оба — `0 B/op`, `0 allocs/op`. Это доказательство конкретной демонстрации, а не переносимый коэффициент ускорения. Для своего кода нужны повторяемый benchmark, проверка layout и профиль contention / hardware counters.

#### 4. Для чего нужен DMA, каковы его плюсы и минусы и что происходит с coherence?

![[90 Вложения/CurseHunter/6817/Кадры/11-dma-01-21-00.jpg]]

Все три вопроса поставлены на слайде прямо.

DMA позволяет устройству переносить блоки между device и memory без копирования каждого слова инструкциями CPU. CPU и драйвер настраивают descriptor/mapping, устройство выполняет передачу, затем сообщает о завершении. Это освобождает execution resources CPU и обычно выгодно для больших или потоковых I/O.

Цена механизма:

- setup descriptors, mapping/unmapping и completion имеют фиксированную стоимость, поэтому маленькая передача может не окупиться;
- устройство имеет собственные ограничения адресации, а IOMMU translation и bus contention добавляют работу;
- buffer lifetime, alignment, pinning/scatter-gather и ownership должны быть согласованы с драйвером;
- «coherent DMA» не отменяет memory barriers: visibility данных и ordering операций — разные гарантии.

Coherence зависит от платформы и типа mapping. На DMA-incoherent системе dirty CPU cache line может скрыть данные от устройства, а запись устройства в RAM — остаться невидимой CPU либо быть перезаписана поздним write-back. Linux DMA API решает это через coherent mappings либо явные `dma_map_*` / `dma_sync_*` / `dma_unmap_*` для streaming mappings. Самодельная передача физического адреса устройству обходит эти гарантии.

### Что проверяет хороший ответ по уроку 3

Кандидат связывает layout и access pattern с cache line traffic, отличает correctness coherence от ordering memory model и не лечит false sharing padding во всех структурах подряд: padding расходует память и нужен только после подтверждённого совместного горячего доступа с разных ядер.

## Урок 4. Введение в операционные системы

### Что именно поставлено в курсе

#### 1. Чем ядро отличается от операционной системы?

![[90 Вложения/CurseHunter/6817/Кадры/12-kernel-vs-os-00-09-00.jpg]]

Вопрос поставлен на слайде прямо. Kernel работает в привилегированном режиме и управляет процессами, virtual memory, interrupts, devices, filesystems и networking через системные механизмы. Операционная система — более широкий комплект: kernel плюс userspace libraries, utilities, daemons, init/service manager, configuration и интерфейсы, которые вместе дают прикладную среду.

Граница не абсолютно одинакова во всех системах. Например, часть сервисов может находиться в kernel space у monolithic kernel и в изолированных userspace servers у microkernel. Поэтому хороший ответ сначала фиксирует рассматриваемую архитектуру.

#### 2. В чём trade-off monolithic kernel и microkernel?

- Monolithic kernel держит основные подсистемы в одном привилегированном address space. Прямые вызовы и общий доступ к структурам дают короткий fast path, но ошибка драйвера способна повредить всё ядро, а trusted computing base крупнее.
- Microkernel оставляет в ядре минимальные scheduling, IPC и memory-management primitives, а сервисы изолирует в userspace. Это улучшает fault isolation и уменьшает привилегированное ядро, но добавляет IPC, context/address-space transitions и более сложные end-to-end protocols.

Нельзя выбирать архитектуру по лозунгу «надёжность против скорости»: современные системы используют гибридные решения, а цена зависит от IPC design, hardware support и того, какие сервисы действительно вынесены.

#### 3. Чем процесс отличается от потока?

Процесс задаёт resource container: address space, descriptors, credentials и другие объекты ОС. Потоки одного процесса разделяют эти ресурсы, но имеют собственные registers, stack и scheduling state. Поэтому thread creation и коммуникация обычно дешевле межпроцессной изоляции, но повреждение общей памяти и data races становятся общей ответственностью потоков.

Go goroutine не равна OS thread. Scheduler мультиплексирует G на M через P; blocking syscall или cgo может потребовать дополнительные M. Поэтому число goroutines, threads и логических CPU — разные величины.

#### 4. Как Go понимает, где расположен текущий исполняемый файл?

![[90 Вложения/CurseHunter/6817/Кадры/13-executable-path-00-36-00.jpg]]

Формулировка поставлена на слайде рядом с `os.FindProcess` и `os.StartProcess`, но в ответе важно разделить два случая:

- `os.StartProcess(name, ...)` не «угадывает» файл: вызывающий передаёт `name`, а поиск по `PATH` делает более высокий уровень `os/exec`;
- для пути уже запущенного процесса служит `os.Executable()`, реализация которого платформо-зависима.

В показанной версии Go `1.23.4` на Linux `os.Executable()` вызывает `readlink("/proc/self/exe")` и удаляет суффикс ` (deleted)`, если исходный файл был удалён. На NetBSD используется `/proc/curproc/exe`; другие ОС имеют собственные реализации. Считать результат стабильной идентичностью файла нельзя: symlink semantics различаются, а файл после запуска можно переместить или заменить.

#### 5. Почему логических CPU может быть больше, чем физических cores, и что возвращает `runtime.NumCPU`?

![[90 Вложения/CurseHunter/6817/Кадры/14-logical-vs-physical-cpu-00-45-00.jpg]]

Вопрос поставлен на слайде прямо. Simultaneous multithreading (SMT, у Intel — Hyper-Threading) предоставляет одному physical core несколько hardware thread contexts. ОС видит их как logical CPUs. Они делят execution units, cache и bandwidth ядра, поэтому два logical CPUs не равны двум независимым physical cores и ускорение зависит от того, какие ресурсы простаивают или конкурируют.

В Go `1.23.4` `runtime.NumCPU()` возвращает число logical CPUs, доступных процессу на момент старта; изменения CPU allocation после старта не отражаются. `GOMAXPROCS` в этой версии по умолчанию равен `NumCPU`, но означает максимум одновременно исполняющих Go-код CPU/P, а не число goroutines и не гарантированное число OS threads. Для container limits это особенно важно: container-aware default появился позднее, в Go `1.25`, поэтому поведение нельзя переносить между версиями без оговорки.

### Что проверяет хороший ответ по уроку 4

Кандидат различает privilege boundary, isolation boundary и scheduling abstraction. Он не называет goroutine «лёгким потоком» без уточнения, не считает logical CPU отдельным физическим core и способен показать, где заканчивается переносимый API Go и начинается платформенная реализация ОС.

## Урок 5. Прерывания и системные вызовы

### 1. Что происходит при системном вызове и всегда ли это context switch?

![[90 Вложения/CurseHunter/6817/Кадры/15-syscall-boundary-00-09-00.jpg]]

Программа вызывает wrapper стандартной библиотеки или выполняет низкоуровневый syscall. Номер операции и аргументы передаются по ABI выбранной OS/architecture. Специальная инструкция, например `SYSCALL` на x86-64 или `SVC` на arm64, переводит CPU на заранее настроенную kernel entry point. Ядро сохраняет необходимое состояние, проверяет номер, аргументы и права, исполняет обработчик и возвращается в user mode.

Privilege transition и scheduler context switch — разные события. Быстрый неблокирующий syscall может вернуться в тот же thread без переключения на другую задачу. Context switch появляется, если вызов блокируется, истекает quantum, приходит более приоритетная работа или scheduler принимает другое решение.

Схема через interrupt descriptor table полезна как историческая модель, но не универсальна. Современный x86-64 fast path `SYSCALL` настраивается через model-specific registers, а не выбирает обычную IDT entry. На интервью сначала называют конкретные OS/ISA, затем описывают их путь.

### 2. Чем отличаются `os`, `syscall` и `golang.org/x/sys/unix`?

- `os`, `net`, `io` и соседние пакеты дают переносимые высокоуровневые контракты. Там, где операция может блокироваться, реализации `os` и `net` согласованы с runtime scheduler; поддерживаемые дескрипторы `net` и части `os` используют runtime poller.
- `syscall` отражает низкоуровневый интерфейс ОС, но заморожен; документация Go рекомендует для нового платформенного кода `golang.org/x/sys`.
- `golang.org/x/sys/unix` покрывает больше Unix-specific API, но остаётся платформенным. Его структуры, константы и semantics нужно проверять для выбранных `GOOS/GOARCH` и версии kernel/libc.

Прямой syscall оправдан, когда высокоуровневого API нет или его контракт мешает задаче. Цена — ручная работа с ownership file descriptors, `errno`, EINTR, partial I/O, alignment и platform build tags.

### 3. Почему буферизованное чтение может быть на порядки быстрее?

Каждый `read` имеет фиксированную цену перехода, проверки descriptor, копирования и bookkeeping. Буфер собирает много мелких логических чтений в меньшее число крупных syscalls. Выигрыш появляется, если приложение иначе вызывает kernel path для каждого короткого фрагмента.

![[90 Вложения/CurseHunter/6817/Кадры/17-buffered-syscall-benchmark-00-45-00.jpg]]

В демонстрации курса на `darwin/arm64`, Apple M3 Pro, buffered-вариант показал `514.9 ns/op`, not-buffered — `72833 ns/op`. Сравнение подтверждает конкретный пример, а не постоянную стоимость syscall: варианты используют разные reader paths, а итог зависит от размера данных, cache state, benchmark setup и ОС.

Интервью-проверка здесь простая: кандидат должен предложить уменьшить число переходов и измерить это, а не заявлять, что «syscall всегда медленный» или что один syscall обязательно означает context switch.

### 4. Чем hardware interrupt, exception и Unix signal отличаются друг от друга?

- Hardware interrupt асинхронно сообщает CPU о внешнем событии, например завершении I/O. Interrupt controller выбирает target CPU и priority; ядро выполняет interrupt handler.
- Exception синхронно связана с текущей инструкцией: page fault, division fault, invalid opcode или программный trap.
- Signal — абстракция ядра для доставки события процессу или thread. Его источником может быть exception, другой процесс, terminal, timer или само ядро, но signal не равен аппаратному interrupt.

Такое разделение не даёт смешать три уровня: событие CPU, обработку ядра и уведомление userspace.

### 5. Как зарегистрировать обработку сигналов в Go?

![[90 Вложения/CurseHunter/6817/Кадры/16-go-signal-handler-00-33-00.jpg]]

Слайд задаёт этот вопрос напрямую. Для shutdown-сценария удобен `signal.NotifyContext`:

```go
ctx, stop := signal.NotifyContext(
    context.Background(),
    os.Interrupt,
    syscall.SIGTERM,
)
defer stop()

<-ctx.Done()
```

Если используется `signal.Notify`, канал должен иметь достаточный buffer: пакет не блокируется при отправке. Для одного ожидаемого значения документация рекомендует buffer `1`. После завершения нужно вызвать `signal.Stop` либо `stop` от `NotifyContext`, чтобы освободить регистрацию и при необходимости вернуть default behavior.

`SIGKILL` и `SIGSTOP` перехватить нельзя. Synchronous `SIGSEGV`/`SIGBUS`/`SIGFPE`, возникшие из-за исполнения Go-кода, runtime обычно превращает в panic; это не замена штатному recovery от произвольного повреждения памяти.

### Что проверяет хороший ответ по уроку 5

Кандидат отделяет privilege transition от scheduling, объясняет эффект batching на числе syscalls и знает, что Go runtime сам участвует в обработке syscall и signal. Список номеров из syscall table без этого механизма почти ничего не показывает.

## Урок 6. Механизмы управления памятью в операционной системе

### 1. Что происходит при `os.Create(filepath.Join(wd, "file.txt"))`?

![[90 Вложения/CurseHunter/6817/Кадры/18-create-file-path-00-03-00.jpg]]

Слайд задаёт этот вопрос напрямую. `filepath.Join` только строит строку. `os.Create` в Go `1.23.4` вызывает `OpenFile` с `O_RDWR|O_CREATE|O_TRUNC` и mode `0666` до применения process umask. Дальше platform wrapper передаёт запрос ядру. На Linux VFS выполняет path lookup, проверяет mount, permissions и тип объекта, вызывает реализацию конкретной filesystem и создаёт file descriptor, который ссылается на open file description.

Если файл уже существует, `O_TRUNC` обнуляет его длину. Возвращённый `*os.File` нужно закрыть явно и проверить ошибку `Close`: полагаться на finalizer для освобождения descriptor нельзя. Сам факт успешного `Create` не гарантирует, что пользовательские данные уже устойчиво лежат на носителе: для durability нужны записи, обработка ошибок и подходящий `Sync`, а storage stack может иметь дополнительные caches.

### 2. Почему программа с почти двухтибибайтным слайсом может завершиться успешно?

![[90 Вложения/CurseHunter/6817/Кадры/19-lazy-allocation-00-36-00.jpg]]

На слайде создан `[]int64` длиной `268_435_456_000`, затем прочитаны только первые `1000` элементов. Вопрос «Почему программа завершается успешно?» поставлен прямо.

Сначала нужно проверить арифметику условия. Слайс требует `2_147_483_648_000` bytes, то есть примерно `1.95 TiB` (`2.15 TB`), поэтому комментарий `1000 GB` на слайде неточен. Если runtime и ОС разрешили зарезервировать такой virtual range, физические страницы выдаются по мере первого доступа. Цикл касается около `8 KiB`, а не всего диапазона; zero-filled pages могут материализоваться через page faults только для затронутой части.

Это не гарантия Go. Запуск может упасть ещё на `make`, если не хватает contiguous virtual address range, runtime отклоняет размер или политика commitment запрещает reservation. Позже процесс может получить OOM при фактическом заполнении. На Linux результат зависит в том числе от `vm.overcommit_memory`, CommitLimit, cgroup limits и memory pressure; запись курса выполнялась на Darwin, поэтому Linux-specific knobs нельзя выдавать за объяснение любого запуска.

### 3. Чем memory-mapped file отличается от обычного `read`/`write`?

`mmap` создаёт отображение file pages в virtual address space. Первый доступ может вызвать fault, после которого page cache-backed frame становится доступен как память. Это убирает явный userspace buffer и позволяет ОС подгружать нужные страницы, но не делает I/O бесплатным.

Trade-offs:

- random access к большой базе может стать проще, а копирований между application buffers — меньше;
- faults переносят latency в произвольную инструкцию load/store, что осложняет tail-latency control;
- `MAP_SHARED` публикует изменения в общее отображение, `MAP_PRIVATE` использует copy-on-write; durability всё равно требует понимания `msync`/`fsync` и filesystem semantics;
- truncate файла под активным mapping, address-space pressure и SIGBUS на недоступной странице требуют отдельной обработки.

### 4. Чем `fork` отличается от `clone` и почему обычный raw `fork` опасен в Go?

![[90 Вложения/CurseHunter/6817/Кадры/20-fork-vs-clone-01-03-00.jpg]]

Оба вопроса поставлены на слайде прямо.

`fork` создаёт child с отдельным virtual address space, первоначально основанным на copy-on-write pages. Parent получает child PID, child — `0`. Descriptors копируются, но указывают на те же open file descriptions, поэтому разделяют offsets и status flags.

Linux `clone`/`clone3` позволяют flags выбрать, какие ресурсы разделяются: address space (`CLONE_VM`), descriptor table (`CLONE_FILES`), signal handlers (`CLONE_SIGHAND`), thread group (`CLONE_THREAD`), namespaces и другие части execution context. Более того, glibc `fork` в NPTL давно реализован через `clone` с fork-equivalent flags; противопоставлять их как два полностью независимых kernel mechanisms неточно.

Go runtime многопоточен. После `fork` в child остаётся только вызвавший thread, но память содержит locks и runtime state, которые могли принадлежать исчезнувшим threads. До `exec` многопоточная POSIX-программа может безопасно вызывать только async-signal-safe functions. Поэтому прикладной Go-код запускает процесс через `os/exec` или `os.StartProcess`: runtime и `syscall.ForkExec` координируют fork/exec, file descriptors и минимальный child path. Продолжать обычное выполнение Go после raw fork нельзя считать корректным.

### 5. Как построить межпроцессное взаимодействие через pipe?

![[90 Вложения/CurseHunter/6817/Кадры/21-pipe-ipc-01-09-00.jpg]]

Вопрос поставлен на слайде прямо. Anonymous pipe — однонаправленный byte stream с read end и write end. До `fork` процесс создаёт pipe; после `fork` parent и child закрывают неиспользуемые ends. Для двустороннего протокола нужны два pipes или socket pair.

Критичные инварианты:

- `read` возвращает EOF только когда закрыты все write ends; забытый descriptor способен повесить consumer навсегда;
- заполненный pipe создаёт backpressure и блокирует writer либо возвращает `EAGAIN` в nonblocking mode;
- reads и writes могут быть partial; message boundaries pipe не сохраняет;
- Linux гарантирует атомарность записи не больше `PIPE_BUF` относительно других writers, но это не превращает поток в очередь структурированных сообщений;
- parent должен собирать child через `Wait`, иначе завершившийся процесс останется zombie до обработки статуса.

### Что проверяет хороший ответ по уроку 6

Кандидат умеет проследить resource lifetime от Go API до kernel object, не путает reservation виртуального адреса с потреблённой RAM и объясняет, почему `fork` в многопоточном runtime требует короткого, контролируемого пути к `exec`.

## Урок 7. Введение в ассемблер Go

### 1. Можно ли обойтись без frame pointer и как тогда раскручивать стек?

![[90 Вложения/CurseHunter/6817/Кадры/22-frame-pointer-unwind-00-27-00.jpg]]

Оба вопроса поставлены на слайде прямо. Frame pointer chain даёт простой способ пройти caller frames: каждый frame хранит прежний pointer и return address. Но это не единственный способ. Compiler может описать frame layout и unwind rules в metadata; unwinder восстанавливает caller state по program counter, stack pointer и таблицам. Цена omission — profiler/debugger зависят от полноты metadata и поддержки асинхронной выборки, зато hardware register можно использовать иначе и пролог/эпилог иногда короче.

В Go нельзя переносить эту схему напрямую на `FP` из `.s`: Go assembler `FP` — pseudo-register для arguments, а не обязательно аппаратный `RBP`/`X29`. Runtime stack walking учитывает собственный ABI, stack maps и metadata; цепочка hardware frame pointers — лишь один из возможных источников для внешних profilers.

### 2. Почему Go assembler нельзя читать как обычный Intel или ARM syntax?

Go assembler основан на Plan 9 style и описывает semi-abstract instruction set. Некоторые mnemonics почти напрямую соответствуют машине, другие toolchain переписывает или расширяет. Data flow в большинстве инструкций идёт слева направо: `MOVQ src, dst`, тогда как Intel syntax обычно записывает `MOV dst, src`.

![[90 Вложения/CurseHunter/6817/Кадры/25-go-asm-syntax-01-12-00.jpg]]

Четыре общих pseudo-register:

- `SB` адресует global symbols;
- `FP` адресует named arguments/results в caller frame;
- virtual `SP` адресует locals текущего frame;
- `PC` используется для branches.

Их нельзя механически отождествлять с hardware registers. Набор реальных registers, mnemonics и addressing modes зависит от `GOARCH`.

### 3. Как читать минимальную Go assembly function и что скрывает `TEXT`?

![[90 Вложения/CurseHunter/6817/Кадры/24-go-asm-function-00-57-00.jpg]]

Go prototype в `.go` задаёт типовой контракт и даёт `vet` возможность проверить argument offsets. В `.s` директива `TEXT ·IntSum(SB), NOSPLIT, $frameSize-argSize` объявляет symbol, flags, local frame size и stack argument size; минус здесь служит разделителем, а не вычитанием. `NOSPLIT` убирает обычную stack-growth check; это обязанность доказать, что функция и её call chain укладываются в допустимый nosplit budget, а не бесплатный флаг ускорения.

Начиная с Go `1.17` compiler использует register-based internal ABI. Обычная пользовательская assembly function остаётся ABI0, а toolchain при необходимости строит ABI wrapper. Поэтому пример с `a+0(FP)` и `ret+16(FP)` может быть корректен для ABI0, но не описывает фактическое размещение каждого внутреннего Go-вызова. Для Go `1.23.4` нужно сверяться с `abi-internal.md`, prototype и результатом `go vet`.

Если assembly function хранит pointers в local frame или вызывает другие функции, сборщику мусора и stack unwinder нужны корректные stack maps и `FUNCDATA`/`PCDATA`. Часть argument metadata toolchain выводит из Go prototype, но произвольные pointer locals в пользовательском assembly автоматически безопасными не становятся. Практичный дизайн — держать pointers и allocations в Go, а в assembly выносить короткий leaf hot path с Go prototype и architecture-specific tests.

### 4. Почему один `.s` не переносится между amd64 и arm64?

![[90 Вложения/CurseHunter/6817/Кадры/23-asm-platform-differences-00-54-00.jpg]]

У архитектур различаются registers, instruction set, calling convention details, vector widths/features и даже названия move/load operations. Практический layout пакета — отдельные `name_amd64.s` и `name_arm64.s` плюс Go fallback/build tags. API остаётся одним, реализации проверяются на каждой целевой архитектуре.

Проверить сгенерированный код можно через `go tool compile -S`, `go build -gcflags=-S` и `go tool objdump`. На собеседовании ждут не память на все mnemonics, а способность найти hot path, прочитать loads/branches/calls, сопоставить их с source и проверить гипотезу benchmark/profile.

### 5. Когда SIMD в ручном assembly оправдан?

SIMD обрабатывает несколько элементов одной инструкцией и выигрывает на плотном цикле без сложных dependencies и branches. Перед ручной реализацией нужно доказать, что:

1. цикл действительно горячий;
2. данные достаточно велики, чтобы окупить dispatch и tail handling;
3. algorithm и memory access уже оптимальны;
4. доступна feature detection и scalar fallback;
5. benchmark проверяет разные размеры, alignment и позиции совпадения, а tests — остаток меньше vector width.

Иначе assembly закрепляет ABI/ISA complexity, усложняет portability и может проиграть обычному Go на маленьком input.

### Что проверяет хороший ответ по уроку 7

Кандидат воспринимает assembly как инструмент проверки модели, а не как автоматический синоним скорости. Он различает hardware и pseudo-registers, знает границу ABI0/ABIInternal и помнит про GC metadata, stack growth и architecture fallback.

### Задание 1. Сумма элементов слайса

Тестовый контракт:

```go
func SumSlice(x []int32) int64
```

Нужно вернуть сумму для положительных, отрицательных и смешанных наборов. Проверки `MaxInt32 + 1` и `MinInt32 - 1` требуют накопителя типа `int64`: если сложить значения как `int32` и расширить только результат, overflow уже произошёл. Каждый операнд нужно до сложения расширить до `int64` с сохранением знака (sign extension).

**Инвариант:** после обработки первых `i` элементов накопитель равен их математической сумме в диапазоне `int64`.

**Сложность:** `O(n)` по времени, `O(1)` дополнительной памяти. Нужны отдельные тесты для пустого слайса и достаточно большого `n`; существующий тестовый файл пустой случай не фиксирует.

### Задание 2. Вариант бинарного поиска `LowerBound`

```go
func LowerBound(slice []int64, value int64) int64
```

По tests требуется индекс последнего элемента `<= value`; если такого элемента нет, вернуть `-1`. Это не стандартный C++-подобный `lower_bound`, который ищет первый элемент `>= value`. Например:

- `[1, 2, 3, 4]`, `3` → `2`;
- `[10, 20, 30]`, `5` → `-1`;
- `[5, 6, 7]`, `11` → `2`;
- `[1, 2, 6, 8]`, `7` → `2`.

![[90 Вложения/CurseHunter/6817/Кадры/27-lower-bound-homework-01-15-00.jpg]]

Удобный инвариант: `lo == -1` либо `slice[lo] <= value`; `hi == len(slice)` либо `slice[hi] > value`. На старте `lo=-1`, `hi=len(slice)`. Пока `hi-lo>1`, выбираем середину. Если `slice[mid] <= value`, переносим `lo=mid`, иначе `hi=mid`. Ответ — `lo`.

**Сложность:** `O(log n)` time, `O(1)` memory. Предусловие — slice отсортирован по неубыванию. Для полноты стоит добавить duplicates и крайние значения; tests курса их не задают.

### Задание 3. Числа Фибоначчи

```go
func Fibonacci(n uint64) uint64
```

Tests фиксируют `F(0)=0`, `F(1)=1`, `F(2)=1` и проверяют значения до `F(70)=190392490709135`. Итеративное решение держит два соседних числа и обновляет их `n` раз. Рекурсия создаёт лишние calls и при наивной реализации имеет exponential time.

**Инвариант:** перед итерацией `i` две рабочие переменные (в assembly — два регистра) содержат `F(i)` и `F(i+1)`. **Сложность:** `O(n)` по времени, `O(1)` памяти. Контракт не определяет overflow: `F(94)` уже не помещается в `uint64`, поэтому production API должен либо ограничить `n`, либо вернуть признак переполнения, либо выбрать big integers.

### Задание 4. Подсчёт слов с Unicode whitespace

Тестовый контракт:

```go

```

Однопроходный автомат хранит `insideWord`. Пробельная руна сбрасывает флаг. Непробельная увеличивает счётчик только при переходе `false → true`, затем устанавливает флаг. Получаем `O(n)` по времени и `O(1)` памяти.

Нельзя заменить Unicode predicate проверкой `r == ' '`: тесты это обнаружат. README обещает result `int64`, тестовый prototype — `int32`; снова приоритет у test contract. Assembly может вызвать Go helper `isSpace`, но call требует корректного ABI и stack metadata; альтернатива с собственной Unicode table должна совпадать с версией Unicode в выбранной Go toolchain.

## Урок 9. Устройство памяти Go и бенчмарки

Кадры урока одновременно показывают исходники Go `1.23.4` и SDK `1.24.0`. Поэтому устройство аллокатора (allocator) ниже зафиксировано для Go `1.23.4`, а эволюция benchmark API — отдельно, начиная с Go `1.24`. Актуальная стабильная версия на дату проверки — Go `1.26.5`; внутренние структуры runtime не являются публичным API и между этими версиями могли меняться.

### 1. Как Go выделяет малый объект и где участвуют `mcache`, `mcentral`, `mspan`, `mheap`?

![[90 Вложения/CurseHunter/6817/Кадры/29-go-allocator-hierarchy-00-16-00.jpg]]

В Go `1.23.4` heap организован в arenas, а runtime управляет им страницами по `8 KiB`. Это внутренняя page size allocator, её нельзя отождествлять с размером страницы ОС. `mspan` описывает непрерывный набор runtime pages. Для малых объектов span делится на slots одного size class.

Путь малого объекта:

1. Runtime округляет размер до size class и ищет свободный slot в span из per-P `mcache`. В обычном fast path глобальная блокировка не нужна.
2. Если span заполнен, `mcache` запрашивает другой span у `mcentral` соответствующего span class. Span class кодирует size class и признак scan/noscan.
3. Если у `mcentral` нет подходящего span, страницы берутся у глобального `mheap`.
4. Если свободного диапазона нет и там, runtime резервирует и отображает новые heap arenas через ОС.

Объекты крупнее `32 KiB` в этой версии обходят `mcache` и `mcentral`: для них `mheap` выделяет отдельный span. Точный порог и size classes — детали версии. `mcache` означает cache allocator на P, а не гарантию попадания объекта в hardware CPU cache.

Практический вывод для интервью: такая иерархия амортизирует глобальную синхронизацию. Но allocation по-прежнему увеличивает объём работы GC, может обнулить память, вызвать refill span или рост heap. Поэтому «маленький `new` всегда почти бесплатен» — неверный контракт.

### 2. Почему в Go `1.23.4` heap arena на Windows меньше?

![[90 Вложения/CurseHunter/6817/Кадры/28-windows-heap-arena-00-09-00.jpg]]

Это прямой вопрос со слайда. В исходнике Go `1.23.4` `heapArenaBytes` равен `64 MiB` на большинстве 64-битных non-Windows платформ и `4 MiB` на Windows, 32-bit, wasm и iOS/arm64. Комментарий runtime объясняет Windows-исключение моделью committed memory: процессу начисляется весь committed range, даже если страницы ещё не трогали. Крупная arena несоразмерно раздувала бы учёт малых heap; race detector усиливает этот эффект.

Это хороший пример версионно-платформенного вопроса. Размер arena нельзя выводить из `unsafe.Sizeof`, размера OS page или объёма RAM, а число `64 MiB` нельзя переносить на любую `GOOS/GOARCH`.

### 3. Чем garbage collector отличается от scavenger и почему `HeapSys` не равен RSS?

![[90 Вложения/CurseHunter/6817/Кадры/38-go-scavenger-madvise-00-19-00.jpg]]

GC определяет недостижимые объекты и освобождает slots; полностью свободный span можно вернуть в пул страниц `mheap`. Scavenger решает другую задачу: для достаточно старых свободных страниц он сообщает ОС, что physical backing можно переиспользовать. Конкретный `madvise` и его semantics зависят от Go version и OS; кадр с `MADV_DONTNEED` иллюстрирует один platform path, а не универсальный контракт runtime.

Из этого следуют разные метрики:

- `HeapAlloc` отражает байты выделенных heap objects;
- `HeapInuse` включает spans, где находится хотя бы один объект, вместе с внутренней фрагментацией;
- `HeapIdle` относится к spans без объектов;
- `HeapReleased` — часть idle memory, которую runtime уже сообщил ОС как возвращённую;
- `HeapSys` показывает virtual address space, полученный runtime для heap, поэтому не обязан совпадать с resident set size процесса.

`debug.FreeOSMemory` принудительно запускает GC и затем пытается вернуть ОС как можно больше памяти. Это диагностический рычаг, а не исправление удержания объектов, плохой preallocation или неверно настроенного memory limit.

### 4. Каков минимальный контракт корректного Go benchmark?

Одна итерация должна выполнять одну и ту же логическую операцию над сопоставимым input. Setup и cleanup исключаются из времени, результат остаётся наблюдаемым для compiler, а benchmark сообщает allocations, если они относятся к вопросу.

В Go до `1.24` обычный шаблон использует `for i := 0; i < b.N; i++`. Harness несколько раз вызывает benchmark function с растущим `b.N`, пока измерение не станет достаточно длинным. Начиная с Go `1.24` доступен и рекомендован `for b.Loop()`: setup до первого `Loop` и cleanup после завершения цикла автоматически не входят в timer, вызов benchmark function выполняется один раз на measurement, а compiler не должен полностью удалить calls внутри benchmark loop.

Полезные флаги отвечают на разные вопросы:

- `-bench=...` выбирает benchmark;
- `-benchmem` добавляет `B/op` и `allocs/op`;
- `-benchtime=2s` задаёт целевую длительность, а `-benchtime=100x` — ровно `100` итераций;
- `-count=N` даёт повторные samples для сравнения distributions;
- `-cpu=1,2,4` запускает тестовый binary с указанными значениями `GOMAXPROCS`, а не обещает физически выдать процессу столько cores.

Сравнение имеет смысл только при фиксированных Go version, compiler flags, `GOOS/GOARCH`, governor/power mode, background load и данных. Один `ns/op` без повторов годится как наблюдение курса, но не как универсальная константа.

### 5. Почему три benchmark со слайда некорректны?

![[90 Вложения/CurseHunter/6817/Кадры/30-benchmark-common-errors-00-43-00.jpg]]

На слайде показаны три характерные ошибки:

- `BenchmarkWrongSingleAction` вызывает функцию один раз и не использует `b.N`. Harness всё равно увеличивает `b.N`, поэтому одна фактическая операция делится на вымышленное число итераций.
- `BenchmarkWrongRecursive` передаёт `b.N` как размер задачи внутри каждой итерации. При калибровке одновременно растут число итераций и стоимость одной итерации; измеряемая операция постоянно меняется.
- `testing.Benchmark`, внутри которого вызывается `recursiveFunc(i)`, также смешивает разные input sizes. Порог `NsPerOp() <= 1000` в обычном unit test зависит от CPU, toolchain и шума и не проверяет функциональную корректность.

Правильная постановка фиксирует размер задачи или создаёт отдельные sub-benchmarks для размеров. Performance regression оценивают по повторным samples и заранее определённой среде; correctness проверяют отдельными tests.

### 6. Как устроен parallel benchmark и что не так с `SetParallelism(runtime.NumCPU())`?

`RunParallel` запускает несколько goroutines и распределяет между ними суммарно `b.N` итераций. По умолчанию число workers равно `GOMAXPROCS`; `ns/op` отражает wall time всего benchmark, а не сумму CPU time всех goroutines. Внутри callback нельзя управлять `StartTimer`, `StopTimer` или `ResetTimer`, потому что timer общий.

`SetParallelism(p)` устанавливает число goroutines равным `p * GOMAXPROCS`. Поэтому пример курса `SetParallelism(runtime.NumCPU())` на обычной машине с `GOMAXPROCS == NumCPU` создаёт примерно `N²` workers, а не «по одному worker на CPU». Для CPU-bound benchmark обычно хватает default. Множитель нужен, когда намеренно моделируется большая concurrency, например блокирующий I/O, и это должно быть частью условия.

### 7. Что хранит slice и почему нельзя конструировать `reflect.SliceHeader` вручную?

![[90 Вложения/CurseHunter/6817/Кадры/31-slice-header-lifetime-00-52-00.jpg]]

Ментальная модель slice — descriptor из pointer, length и capacity, который ссылается на backing array. Но `reflect.SliceHeader` не является безопасным конструктором этого descriptor: его `Data` имеет тип `uintptr`, а integer не удерживает объект живым для GC. В Go `1.23.4` тип помечен deprecated; документация предлагает `unsafe.Slice` и `unsafe.SliceData`.

`unsafe` не снимает требования lifetime и bounds. Pointer должен оставаться внутри исходного allocation, длина не может выходить за него, а правильно типизированная ссылка должна удерживать backing storage живым. `runtime.KeepAlive` помогает только с lifetime; он не исправляет неверную арифметику указателя.

В уроке также показан внутренний `iface`/`itab`. Это полезно для объяснения interface dispatch, но layout runtime нестабилен. Само присваивание в interface не гарантирует heap allocation: решение зависит от escape analysis, размера значения и call context. Проверяют его через compiler diagnostics и benchmark, а не по рисунку из одной версии runtime.

### 8. Что на самом деле делает zero-copy conversion через `unsafe.String`?

![[90 Вложения/CurseHunter/6817/Кадры/32-unsafe-zero-copy-01-00-00.jpg]]

`unsafe.String`, `unsafe.StringData` и `unsafe.SliceData` появились в Go `1.20`. Они создают новый header, который ссылается на те же bytes; копирования payload нет. Взамен вызывающий код принимает aliasing contract:

- bytes, переданные в `unsafe.String`, нельзя менять, пока живёт string;
- память, полученную через `unsafe.StringData`, нельзя менять вообще через этот pointer;
- для empty string `StringData` может вернуть `nil`;
- возвращать `[]byte`, который ссылается на storage строки, опасно: тип обещает mutability, которой фактически нет.

Особенно опасен такой string как map key. Если исходный byte slice после вставки изменить, логическое значение ключа изменится без нормальной повторной вставки и нарушит предположения hash table.

Benchmark курса сообщает `0.3955 ns/op`, `0 B/op`, `0 allocs/op` на `darwin/arm64`. Но функция делает круг `[]byte → string → []byte`, возвращает alias исходного slice, а caller игнорирует результат. Старый `b.N`-benchmark не ставит надёжного compiler barrier, поэтому столь малая цифра совместима с удалением большей части работы. Она не доказывает стоимость полезной конвертации в production.

### 9. Почему перестановка полей меняет размер struct?

![[90 Вложения/CurseHunter/6817/Кадры/33-struct-alignment-01-08-00.jpg]]

На показанной `darwin/arm64` конфигурации:

```go
type T1 struct {
    a int8
    b int64
    c int16
} // 24 bytes

type T2 struct {
    a int8
    c int16
    b int64
} // 16 bytes
```

Каждое поле начинается с допустимого alignment; compiler вставляет padding между полями и округляет общий размер struct так, чтобы элементы массива таких structs тоже были выровнены. Группировка полей от большего alignment к меньшему часто сокращает padding, но это не универсальное правило оптимизации.

Перед перестановкой проверяют `unsafe.Sizeof`, `Alignof`, `Offsetof` на целевых architectures и последствия для cache locality. Field order может быть внешним binary/FFI contract, участвовать в atomic alignment или намеренно разделять горячие поля. Экономия восьми байт не оправдывает молчаливое изменение layout на границе системы.

### 10. Когда preallocation слайса или map действительно помогает?

![[90 Вложения/CurseHunter/6817/Кадры/34-slice-preallocation-01-16-00.jpg]]

Если известна верхняя граница результата, `make([]T, 0, n)` снижает число reallocations и копирований backing array. Для map `make(map[K]V, n)` передаёт ожидаемое число элементов как hint, но не обещает точную capacity. Цена — заранее удержанная память; если filter обычно пропускает один элемент из тысячи, capacity `len(input)` может ухудшить peak RSS, а для pointer-containing `T` ещё и увеличить GC scan work.

В «быстром» варианте на слайде одновременно применены preallocation и `unsafe`-alias `string → []byte`. Такой benchmark не изолирует эффект одного изменения. Более того, возвращаемый `[][]byte` выглядит изменяемым, хотя элементы ссылаются на immutable string storage. Для честного сравнения нужны отдельные варианты: только capacity hint, только conversion strategy, затем их комбинация.

### 11. Почему специализированный `strconv` может выиграть у `fmt.Sprintf`?

![[90 Вложения/CurseHunter/6817/Кадры/35-string-conversion-benchmark-01-17-00.jpg]]

`fmt.Sprintf` разбирает format string и обслуживает общий variadic/interface-based path. `strconv.FormatInt` решает узкую задачу: перевод одного integer в заданную base. На слайде для константы `42` показано `30.28 ns/op`, `2 B/op`, `1 allocs/op` против `1.387 ns/op`, `0 B/op`, `0 allocs/op` на `darwin/arm64`.

Правило выбора не «`fmt` медленный», а «не оплачивать общий formatter для узкой операции в подтверждённом hot path». Сам benchmark требует усиления: результат нужно использовать, inputs — варьировать, а вывод подтвердить несколькими запусками. Constant input и discarded result дают compiler больше возможностей специализировать или удалить работу.

### 12. Что не так с benchmark `strings.Builder` на слайде?

![[90 Вложения/CurseHunter/6817/Кадры/36-string-builder-benchmark-01-20-00.jpg]]

`strings.Builder` предназначен для последовательной сборки одной строки и минимизирует копирование; если финальный размер известен, `Grow` убирает промежуточные расширения. Нельзя копировать non-zero `Builder` по значению.

Однако слайд создаёт один Builder перед `for i < b.N` и на каждой итерации дописывает byte. Вторая версия так же растит одну строку через `+=`. Значит, `b.N` одновременно играет роль iteration count и конечной длины результата. Для `+=` суммарное копирование растёт квадратично, тогда как harness ожидает примерно постоянную работу на итерацию при калибровке.

Показанные `1.154 ns/op` и `0 allocs/op` у Builder не означают, что вся строка построена без allocations: итоговые allocations делятся на огромное `b.N` и округляются на операцию. Корректный benchmark фиксирует длину одной строки, строит её заново на каждой итерации, вызывает `Grow` отдельным вариантом и сохраняет результат в sink.

### 13. Всегда ли нужно выносить allocation из цикла?

![[90 Вложения/CurseHunter/6817/Кадры/37-loop-allocation-benchmark-01-22-00.jpg]]

В примере курса перенос `d := Data{}` за цикл перед повторным `json.Unmarshal` меняет результат с `228.2 ns/op`, `160 B/op`, `3 allocs/op` на `205.5 ns/op`, `152 B/op`, `2 allocs/op`. Здесь повторно используется (reuse) один destination object на выбранном типе данных.

Но reuse корректен только после проверки semantics. `json.Unmarshal` может переиспользовать capacity slices/maps и оставлять большие backing arrays доступными дольше; поля, отсутствующие в следующем JSON, требуют понимания правил обновления destination. В concurrent path один объект нельзя переиспользовать без ownership или синхронизации. Сначала доказывают эквивалентность результата, затем измеряют CPU, allocations и retained memory.

Три финальных шаблона со слайда тоже требуют условий:

- регулярное выражение компилируют один раз, если pattern повторяется; dynamic patterns требуют bounded cache и политики eviction;
- `sync.Pool` подходит для временных объектов, разделяемых независимыми callers, но runtime может удалить любой элемент в любой момент без уведомления; это не storage и не cache с гарантией hit;
- перед `Put` объект сбрасывают, после `Put` им не пользуются, а очень большие buffers часто отбрасывают, чтобы pool не удерживал высокий memory watermark.

### Что проверяет хороший ответ по уроку 9

Кандидат связывает allocation path с contention и GC work, различает live heap, reserved virtual memory и resident pages. В benchmark он фиксирует одну операцию, управляет setup, объясняет `b.N`/`B.Loop`, не путает `-cpu` с physical cores и замечает опасные aliases `unsafe`. Оптимизацию выбирает после профиля и подтверждает воспроизводимым сравнением, а не одной красивой цифрой.

## Урок 10. Оптимизации в Go

На кадрах открыт Go SDK `1.24.0`, а документация `weak` показывает `go1.24.1`. Поэтому compiler-dependent наблюдения ниже относятся к Go `1.24.0` на `darwin/arm64`. `runtime.AddCleanup` и пакет `weak` появились в Go `1.24`, memory limit — в Go `1.19`, а учёт GC roots в формуле `GOGC` — в Go `1.18`. Актуальная стабильная версия на дату проверки — Go `1.26.5`; escape analysis и bounds-check elimination не входят в Go 1 compatibility contract.

### 1. Как `GOGC` задаёт момент запуска следующего GC cycle?

![[90 Вложения/CurseHunter/6817/Кадры/39-gc-controls-00-03-00.jpg]]

![[90 Вложения/CurseHunter/6817/Кадры/40-gogc-target-formula-00-12-00.jpg]]

Начиная с Go `1.18`, упрощённая модель target heap выглядит так:

```text
Target heap = Live heap + (Live heap + GC roots) × GOGC / 100
```

`Live heap` измерен в конце предыдущего цикла. В `GC roots` входят, в частности, goroutine stacks и pointers в global variables. Target — цель pacer, а не жёсткая граница: крупная allocation или неточность оценки могут её превысить до завершения mark phase.

Пример со страницы, показанной в уроке: `8 MiB` live heap, `1 MiB` stacks и `1 MiB` global roots. При `GOGC=100` runtime допускает примерно `10 MiB` new heap, то есть target равен `18 MiB`. При `GOGC=50` headroom составляет примерно `5 MiB`, при `GOGC=200` — `20 MiB`.

Отсюда trade-off:

- больше `GOGC` — реже GC и обычно меньше CPU на collection, но выше peak heap;
- меньше `GOGC` — ниже heap headroom, но больше циклов и CPU overhead;
- приближение «удвоили `GOGC` — примерно вдвое снизили GC CPU» работает для достаточно большого steady-state heap, где marginal scan cost доминирует. Это не SLA для произвольной нагрузки.

### 2. Почему `GOMEMLIMIT` не равен лимиту RSS или cgroup?

![[90 Вложения/CurseHunter/6817/Кадры/41-gomemlimit-tradeoff-00-19-00.jpg]]

Memory limit появился в Go `1.19` и задаётся через `GOMEMLIMIT` либо `debug.SetMemoryLimit`. Runtime считает управляемую им память по модели:

```text
runtime.MemStats.Sys - runtime.MemStats.HeapReleased
```

Эквивалент через `runtime/metrics`:

```text
/memory/classes/total:bytes - /memory/classes/heap/released:bytes
```

Это soft limit для памяти Go runtime. Он не охватывает executable mappings, память C через cgo, произвольный `mmap` и память, которую ОС держит от имени процесса. Поэтому ставить `GOMEMLIMIT` ровно в cgroup limit опасно. Официальный GC guide предлагает оставлять хотя бы `5–10%` headroom в полностью контролируемом контейнере; если есть заметная non-Go memory, запас считают отдельно по измерениям.

Когда usage приближается к limit, runtime снижает эффективный heap target и чаще запускает GC. Если limit ниже live set плюс runtime overhead, начинается thrashing: приложение тратит CPU на циклы, которые почти ничего не освобождают. Именно поэтому limit мягкий. Описанный в GC guide limiter допускает выход за limit, чтобы не отдавать сборщику больше примерно половины CPU в окне `2 × GOMAXPROCS` CPU-seconds. Это implementation detail, а не вечная гарантия Go.

`GOGC=off` отключает обычный процентный trigger, но memory limit продолжает действовать. Практическое правило для сервиса: сначала измерить live heap, allocation rate и внешнюю память; затем согласованно настроить `GOGC`, `GOMEMLIMIT` и container limit. Memory limit не лечит retention leak.

### 3. Когда допустимы `runtime.GC`, `SetGCPercent` и `SetMemoryLimit`?

- `runtime.GC()` синхронно доводит до конца полный GC cycle и блокирует caller; часть цикла может остановить весь program. Это инструмент для редких фазовых переходов, тестов и диагностики, а не вызов на каждом request.
- `debug.SetGCPercent(p)` возвращает предыдущее значение. Отрицательный `p` отключает процентный trigger, но не memory-limit trigger.
- `debug.SetMemoryLimit(n)` можно менять во время работы, например когда контролируемая cgo-фаза временно требует больше памяти. Значение должно следовать реальному resource budget, а не колебаться по каждому allocation.

Слайд упоминает ballast и отдельные GC tuners как workaround до Go `1.19`. С появлением memory limit они перестали быть основным способом ограничить runtime memory. Механическое добавление ballast увеличивает искусственный live heap и может ухудшить прогноз поведения.

### 4. Что на самом деле измеряют функции пакета `runtime` со слайда?

- `runtime.NumCPU()` возвращает число logical CPUs, доступных process при старте; это не число goroutines и не текущий `GOMAXPROCS`.
- `runtime.GOMAXPROCS(0)` читает текущий limit параллельного исполнения Go code, а положительный argument меняет его. В Go `1.24` default не учитывал container CPU quota автоматически; container-aware default появился в Go `1.25` на поддерживаемых Linux systems.
- `runtime.Gosched()` уступает processor другим goroutines, но не блокирует текущую goroutine на событии и не заменяет channel, mutex или condition variable.
- `runtime.NumGoroutine()` даёт моментальный счётчик, пригодный как symptom утечки, но сам по себе не показывает owners, states и stacks.
- `runtime.FuncForPC()` связывает program counter с function metadata; для анализа задержки и blocking нужны профили и trace, а не одиночное имя функции.

### 5. Чем `AddCleanup` отличается от finalizer и явного `Close`?

![[90 Вложения/CurseHunter/6817/Кадры/42-addcleanup-contract-00-30-00.jpg]]

`runtime.AddCleanup` появился в Go `1.24`. Cleanup становится eligible после того, как объект недостижим, и выполняется runtime worker. Порядок не задан. Если `cleanup` или его `arg` удерживает `ptr`, объект не станет недостижимым; простейший случай `arg == ptr` вызывает panic при регистрации.

Здесь в самом уроке есть версионная ловушка. Открытая в браузере документация ветки `master` говорит, что cleanups могут выполняться concurrently друг с другом. В Go `1.24.0` один worker вызывал их последовательно. Начиная с Go `1.25`, runtime запускает cleanups конкурентно, в том числе параллельно; это зафиксировано в release notes. В обеих версиях cleanup идёт concurrently с user goroutines, а долгую или блокирующую работу лучше передать отдельной goroutine, чтобы не задерживать очередь.

Сравнение ближайших механизмов:

| Механизм | Когда использовать | Главная граница |
|---|---|---|
| Явный `Close` | File descriptors, sockets, transactions, locks и любой correctness-critical resource | Caller обязан вызвать и обработать error; зато момент release детерминирован |
| `AddCleanup` | Safety net для объекта-обёртки, low-level cache/lifetime machinery | Не гарантирован до exit, порядок не задан; нельзя строить correctness только на cleanup |
| `SetFinalizer` | Legacy code и редкие случаи, где cleanup не подходит | Finalizers сложнее, задерживают reclamation и проблемнее с cycles; новый code предпочитает `AddCleanup` |

`AddCleanup` допускает несколько cleanups и interior pointers, обычно не создаёт cycle leak и не задерживает освобождение pointee graph так, как finalizer. Но он не превращает GC в resource manager. Для file/socket остаётся схема «явный `Close` плюс cleanup как страховка».

### 6. Как weak pointer позволяет построить cache и где возникает race?

![[90 Вложения/CurseHunter/6817/Кадры/43-weak-pointer-contract-00-40-00.jpg]]

![[90 Вложения/CurseHunter/6817/Кадры/44-weak-cache-example-00-42-00.jpg]]

`weak.Pointer[T]` не делает `T` reachable. Если на object остались только weak references, `Value()` может вернуть `nil`. Полученный non-nil `*T` уже служит strong reference на время использования. Если compiler может считать последнее использование завершённым раньше внешнего low-level operation, lifetime фиксируют `runtime.KeepAlive` после этой операции.

Показанный cache хранит `key → weak.Pointer[V]`. При miss он создаёт value, пытается поставить weak pointer через `LoadOrStore`, а к value привязывает cleanup. Cleanup удаляет entry через `CompareAndDelete(key, wp)`, а не обычный `Delete(key)`: пока cleanup ожидал запуска, другой caller мог установить под тем же key новый value. Без сравнения старый cleanup удалил бы новую запись — классический ABA-like race.

Две тонкости контракта:

- equality weak pointers сохраняет identity исходного pointer даже после reclamation, что и позволяет сравнить cache entry;
- `Value()` не обязана когда-либо стать `nil`: tiny pointer-free objects могут делить allocation slot с другим живым object. Поэтому weak pointer нельзя использовать как уведомление с гарантированным сроком.

### 7. Почему два почти одинаковых фрагмента дали разный escape analysis?

![[90 Вложения/CurseHunter/6817/Кадры/45-escape-direct-assignment-00-46-40.jpg]]

![[90 Вложения/CurseHunter/6817/Кадры/46-escape-indirect-assignment-00-48-20.jpg]]

В первом sub-benchmark Go `1.24.0` видит pointer сразу в composite literal:

```go
var number int
data := &Data{pointer: &number}
_ = data
```

Во втором pointer присваивается через уже созданный `*Data`:

```go
var number int
data := &Data{}
data.pointer = &number
_ = data
```

На `darwin/arm64`, Apple M3 Pro курс показывает:

```text
direction_assignment     1.276 ns/op   0 B/op   0 allocs/op
indirection_assignment   6.808 ns/op   8 B/op   1 allocs/op
```

Compiler diagnostics для второго path помечает `number` как `moved to heap`: escape graph консервативно проводит flow через indirect field assignment. Это не правило «инициализируй struct только literal». Escape analysis зависит от inlining, call graph, compiler version и формы использования. `&T{}` само по себе не означает heap allocation, а возврат pointer не всегда её требует после inlining.

У benchmark есть отдельное ограничение: результат присваивается в blank identifier, полезного observable effect нет. `B.Loop` защищает calls и их arguments/results от полного удаления, но в этом loop нет измеряемого function call. Поэтому `ns/op` нельзя трактовать как стоимость реальной business operation; здесь ценен прежде всего allocation report и compiler escape trace.

Проверка для production code: `go build -gcflags='all=-m=2' ./...`, затем benchmark с `-benchmem` и профиль allocations. В доступной среде Go toolchain не установлен, поэтому пример здесь не перезапускался; зафиксирован наблюдаемый output курса и точная его платформа.

### 8. Как одна проверка `_ = out[31]` устраняет несколько bounds checks?

![[90 Вложения/CurseHunter/6817/Кадры/47-bounds-check-elimination-00-53-00.jpg]]

После успешного чтения `out[31]` compiler знает на этом control-flow path, что `len(out) >= 32`. Следующие срезы `out[0:4]`, `out[4:8]`, …, `out[28:32]` уже доказуемо находятся в bounds, поэтому отдельные checks можно убрать.

Это не отключение safety. Если slice короче `32`, program предсказуемо panic на первой строке. Приём полезен в codec/crypto hot path, когда minimum length уже часть precondition. Для внешнего input обычно лучше явная проверка с error:

```go
if len(out) < 32 {
    return io.ErrShortBuffer
}
```

Наличие checks смотрят в assembly или через внутренний SSA diagnostic `-d=ssa/check_bce/debug=1`. Флаг внутренний и может меняться между Go versions; оптимизацию нельзя считать гарантией языка.

### 9. Зачем `slices.Delete` очищает невидимый tail backing array?

![[90 Вложения/CurseHunter/6817/Кадры/48-slices-delete-clears-tail-00-52-00.jpg]]

Операция `append(s[:i], s[j:]...)` уменьшает length, но в slots от нового `len(s)` до старого length остаются копии удалённых elements. Если `E` содержит pointers, GC сканирует backing allocation и эти stale references удерживают объекты живыми, хотя caller их через slice уже не видит.

Начиная с Go `1.22`, mutating functions вроде `slices.Delete` zero obsolete tail:

```go
oldlen := len(s)
s = append(s[:i], s[j:]...)
clear(s[len(s):oldlen])
```

Для собственного in-place algorithm действует то же правило: если tail больше не нужен и element type несёт pointers, очистить его. Цена `clear` реальна, но обычно ниже риска скрытого retention. Для pointer-free `E` GC retention нет, хотя semantic cleanup иногда всё равно полезен.

### 10. Почему generic JSON через reflection проигрывает generated code?

![[90 Вложения/CurseHunter/6817/Кадры/49-reflection-unmarshal-table-00-54-00.jpg]]

![[90 Вложения/CurseHunter/6817/Кадры/50-reflection-course-benchmark-00-55-00.jpg]]

`encoding/json` должен работать с произвольными types: разбирать struct tags, обходить field metadata, проверять `reflect.Kind`, создавать values нужного dynamic type, поддерживать interfaces, maps, custom marshalers и error semantics. Metadata для structs кешируется, но dynamic dispatch и проверки остаются.

Generator вроде easyjson переносит type traversal в build time и выпускает прямой code для конкретной struct. Он сокращает reflection и часть allocations, но добавляет generated files, шаг regeneration, риск рассинхронизации и отдельный compatibility surface.

На слайде курса показаны такие observations:

| Library | JSON | MB/s | allocs/op | B/op |
|---|---:|---:|---:|---:|
| standard | regular | 22 | 218 | 10229 |
| standard | small | 9.7 | 14 | 720 |
| easyjson | regular | 125 | 128 | 9794 |
| easyjson | small | 67 | 3 | 128 |

Отдельный запуск курса сообщает `531515 ns/op`, `265098 B/op`, `3937 allocs/op` для `convertThroughSerialization`; `274356 ns/op`, `282184 B/op`, `4389 allocs/op` для `convertThroughSerializationNew`; `57407 ns/op`, `53259 B/op`, `499 allocs/op` для `easyjsonSerialization`. Это результаты конкретных payload и environment, а не обещанная кратность ускорения. На интервью ожидается предложение benchmark на реальной schema с одинаковой correctness и error handling.

### 11. Чем `reflect.Type`, `Kind`, addressability и `CanSet` отличаются друг от друга?

![[90 Вложения/CurseHunter/6817/Кадры/51-reflect-settable-00-58-00.jpg]]

- `Type` хранит точный Go type, включая defined type и methods;
- `Kind` сводит его к базовой категории: `Struct`, `Pointer`, `Slice`, `Uint8` и так далее;
- addressable `Value` ссылается на реальное storage, а не на неперезаписываемую copy;
- `CanSet` требует addressability и право изменения через reflection.

`reflect.ValueOf(x)` получает copy interface value, поэтому её нельзя менять. `reflect.ValueOf(&x)` тоже нельзя менять: это `Value` самого pointer. Но `.Elem()` у valid non-nil pointer адресует `x`, и для exported/доступного значения `CanSet()` возвращает `true`.

Перед `Elem`, `Set`, `Interface`, numeric accessors и `Call` нужно проверить контракт конкретного `Kind`; reflection сообщает ошибки panic, а не всегда `error`.

### 12. Почему `reflect.Value.IsNil` легко вызывает panic?

![[90 Вложения/CurseHunter/6817/Кадры/52-reflect-isnil-kinds-01-02-00.jpg]]

Документированный контракт `IsNil` допускает `Chan`, `Func`, `Interface`, `Map`, `Pointer` и `Slice`. Для `Struct`, `Int` и других kinds он panic. Реализация Go `1.24.0` также обрабатывает `UnsafePointer`, хотя prose документации его отдельно не называет; переносимый helper лучше строить по документированному набору. Ещё одна ловушка: `reflect.ValueOf(nil)` возвращает zero `Value`; у него сначала проверяют `IsValid`, а не вызывают `IsNil`.

Typed nil внутри interface ведёт себя иначе:

```go
var p *int = nil
var x any = p

x == nil // false: interface содержит dynamic type *int
reflect.ValueOf(x).IsNil() // true
```

Безопасный generic helper сначала проверяет `x == nil`, затем `Value.IsValid`, после чего вызывает `IsNil` только для nil-able kinds. Если набор типов закрыт, type switch проще, быстрее и проверяется compiler.

### 13. Можно ли через `reflect` изменить unexported field другого package?

![[90 Вложения/CurseHunter/6817/Кадры/53-unsafe-private-field-01-06-00.jpg]]

Обычный `reflect` запрещает такое изменение: unexported field не проходит `CanSet`, а некоторые операции `Interface` panic. В демонстрации ограничение обходится через `field.UnsafeAddr()`, `unsafe.Pointer`, `reflect.NewAt(field.Type(), p)` и последующий `Set`.

Технически это запись по адресу storage. Она обходит package encapsulation, invariants и compatibility contract владельца type. Field layout или type может измениться без предупреждения; неверный `Value` приводит к panic или memory corruption, а concurrent access — к data race. Требуются addressable object, точное совпадение type и корректный lifetime.

Такой код оправдан лишь в тесно контролируемом low-level integration, где нет публичного API и есть tests на каждую поддержанную version. Для application code лучше добавить constructor/method, adapter либо generated code.

### 14. Задание: реализовать RPC поверх HTTP/JSON через reflection

![[90 Вложения/CurseHunter/6817/Кадры/54-rpc-homework-contract-01-15-00.jpg]]

Исходное домашнее задание задаёт два контракта:

```go
Invoke(
    ctx context.Context,
    url *url.URL,
    rmiMethod string,
    data []any,
    response any,
) error

CreateObjectStub(serverObject any) http.HandlerFunc
```

Client должен сериализовать список arguments в JSON, выполнить request с переданным context и декодировать result через pointer `response`. Server получает object с RPC methods, выбирает method по имени, восстанавливает аргументы в declared types, вызывает его через reflection и кодирует result.

Для интервью сначала уточняют protocol, потому что signature не отвечает на критические вопросы:

1. Где передаётся method name: path, header или JSON envelope? Как versionируется protocol?
2. Какие method signatures разрешены: нужен ли первым argument `context.Context`, допустимы ли variadic methods, сколько return values и где `error`?
3. Как JSON numbers превращаются в `int`, structs и pointers? Декодирование всего в `[]any` даёт `float64` и теряет declared types; надёжнее создавать `reflect.New(method.Type().In(i))` и декодировать непосредственно туда.
4. Что происходит при unknown method, неверной arity/type, malformed JSON, panic внутри method, application error и transport failure?
5. Какие limits стоят на body, deadlines и concurrency? Есть ли authentication, allowlist methods и защита от вызова нежелательного exported API?
6. Как закрывается response body, сохраняется error cause и различается retryable failure? Удалённый вызов не эквивалентен локальному: возможны timeout после фактического выполнения и повторный side effect.

Минимальный server pipeline:

```text
validate transport → resolve allowed method → validate signature
→ allocate typed arguments → decode JSON → call with panic boundary
→ map result/error to stable response envelope → encode
```

Trade-off: reflection даёт один generic stub, но переносит ошибки signatures в runtime и добавляет dynamic checks. Generated stubs ловят больше ошибок при build, быстрее работают и проще документируют schema, зато требуют generation pipeline. JSON удобно отлаживать; Protobuf/gRPC дают формальную schema, компактный wire format и развитую ecosystem ценой IDL и codegen.

### Что проверяет хороший ответ по уроку 10

Кандидат вычисляет heap target, но не выдаёт `GOGC` и `GOMEMLIMIT` за hard limits. Он отделяет deterministic `Close` от GC-driven cleanup, понимает reachability weak pointer и ABA-like race cache cleanup. Оптимизацию compiler подтверждает escape/BCE diagnostics и benchmark, а reflection применяет с проверкой `Kind`, addressability и failure semantics. В RPC-задаче сначала фиксирует wire contract и partial failures, затем пишет generic invocation.

Название урока в выдаче CourseHunter — «Управление внутренним состоянием рантайма Go», но фактически это первая часть лекции о profiling. Runtime contracts ниже поэтому привязаны к Go `1.24.1`, а source примера — к отдельному commit курса.

### 1. Чем instrumentation отличается от sampling и когда выбирать каждый подход?

![[90 Вложения/CurseHunter/6817/Кадры/55-instrumentation-types-00-12-00.jpg]]

Instrumentation добавляет наблюдаемые события в конкретные точки выполнения. Курс перечисляет пять уровней: ручные вызовы, переписывание source, вставки в intermediate representation, изменение binary и runtime instrumentation. Такой подход может точно посчитать каждый интересующий event и сохранить его arguments, но цена растёт вместе с частотой событий: дополнительные instructions, allocations, synchronization и объём данных способны изменить само поведение программы.

Sampling периодически снимает текущий stack и затем статистически восстанавливает, где process тратил ресурс.

![[90 Вложения/CurseHunter/6817/Кадры/56-sampling-timeline-00-14-00.jpg]]

Если функция занимала долю `p` от измеряемого CPU time, то при достаточно длинном и репрезентативном измерении около доли `p` samples попадёт в её stack. Отсюда следуют обе стороны trade-off:

- overhead обычно ниже, потому что recorder не срабатывает на каждый call;
- короткий профиль, редкий path или неудачный период sampling могут дать нерепрезентативный результат;
- sample показывает stack в момент наблюдения, а не точное начало, конец и duration каждой операции;
- удлинение окна уменьшает случайный шум, но не исправляет нерепрезентативную нагрузку.

Практическое правило: instrumentation подходит для точных business/runtime events и trace chronology, sampling — для поиска устойчивых hot spots с приемлемым production overhead. Результат любого sampling profile — гипотеза, которую подтверждают повторным профилем и измерением после изменения.

### 2. Как устроен CPU profiler Go и зачем ему lock-free `profBuf`?

![[90 Вложения/CurseHunter/6817/Кадры/57-cpu-profiler-architecture-00-18-00.jpg]]

В Unix-like варианте Go `1.24.1` цепочка выглядит так:

```text
profiling clock → SIGPROF → unwind текущего stack
→ запись sample в runtime.profBuf → reader goroutine
→ построение pprof protobuf → io.Writer или HTTP response
```

Signal handler работает в ограниченном контексте: он не может безопасно выделять память, брать произвольные locks или использовать большой stack. `cpuProfile.add` поэтому записывает готовый stack в заранее выделенный `profBuf`. Сам buffer рассчитан на одного reader и одного writer; конкурирующие signal callbacks сериализуются внутренним signal lock. Reader goroutine забирает chunks и преобразует их в profile вне signal handler.

Если reader не успевает, runtime не блокирует execution бесконечно: заполненный ring buffer отбрасывает новые records и позже добавляет synthetic overflow record с числом потерь. Поэтому absence samples не доказывает absence работы — сначала проверяют lost records и достаточность окна.

`runtime/pprof.StartCPUProfile` в Go `1.24.1` жёстко использует `100 Hz`. На каждые CPU-second получается примерно `100` sampling ticks, а не «100 samples на request». Обработка stack trace сама не бесплатна, поэтому стандартный API выбирает фиксированный компромисс и не предлагает caller менять частоту.

![[90 Вложения/CurseHunter/6817/Кадры/67-start-cpu-profile-100hz-01-15-00.jpg]]

Одновременно в process может идти только один CPU profile. Повторный `StartCPUProfile` возвращает error, а `StopCPUProfile` отключает profiler и ждёт, пока writer полностью запишет данные. `defer pprof.StopCPUProfile()` ставят только после успешного `StartCPUProfile`; file нужно закрывать уже после `StopCPUProfile`, иначе хвост профиля потеряется.

### 3. Какой Go profile выбирать по наблюдаемому симптому?

![[90 Вложения/CurseHunter/6817/Кадры/58-on-off-cpu-profiles-00-30-00.jpg]]

| Симптом или вопрос | Инструмент | Что он действительно показывает |
|---|---|---|
| Process потребляет CPU | CPU profile | Samples on-CPU call stacks; это не wall-clock latency и не полная история goroutine |
| Растёт live heap | `heap`, обычно `-inuse_space` | Sampling allocation sites объектов, остающихся live на учтённом GC snapshot |
| Высокий allocation churn или GC pressure | `allocs` / `-alloc_space` | Sampling всех allocations с начала process, включая уже собранные |
| Goroutines ждут channels, mutexes, `WaitGroup`, `Cond` | `block` | Cumulative blocked time и stack в точке, где goroutine заблокировалась; профиль нужно включить |
| Lock holder задерживает конкурентов | `mutex` | Approximate cumulative wait других goroutines, attributed к stack завершения critical section, обычно `Unlock`; профиль нужно включить |
| Подозрение на goroutine leak или зависание | `goroutine`, часто `debug=2` | Snapshot текущих goroutines и stacks без duration и причинной chronology |
| Нужно понять создание OS threads | `threadcreate` | Stacks, приведшие к созданию новых OS threads |
| Нужна временная последовательность scheduler, GC, syscalls и goroutine states | execution trace | Event chronology; данных и overhead больше, окно обычно короче |

CPU profile отвечает на вопрос «где process тратил CPU», а не «почему request ждал». При высокой latency и низком CPU следующий шаг — block/mutex/goroutine profile или trace. Block profile не покрывает всё off-CPU время: I/O syscalls, `time.Sleep`, scheduler backlog и ожидание внешнего сервиса требуют других signals и сопоставления с application telemetry.

Heap profile тоже не равен byte-perfect live snapshot. В Go `1.24.1` allocations sampled, а `runtime.MemProfile` предупреждает, что данные могут отставать до двух GC cycles. Query `gc=1` у HTTP heap handler запускает GC перед записью, но сам forced GC меняет исследуемый process и не отменяет sampling.

### 4. Чем программный `runtime/pprof` отличается от `net/http/pprof`?

![[90 Вложения/CurseHunter/6817/Кадры/59-runtime-pprof-api-00-36-00.jpg]]

Programmatic API подходит для CLI, tests и строго ограниченного участка:

```go
p := pprof.Lookup("heap")
if p == nil {
    return errors.New("heap profile is unavailable")
}
if err := p.WriteTo(heapOut, 0); err != nil {
    return err
}

if err := pprof.StartCPUProfile(cpuOut); err != nil {
    return err
}
defer pprof.StopCPUProfile()
```

Здесь `heapOut` и `cpuOut` — разные outputs: два binary profiles нельзя последовательно склеивать в один stream.

`WriteHeapProfile(w)` — backward-compatible shorthand для `Lookup("heap").WriteTo(w, 0)`. `debug=0` пишет binary protobuf, `debug=1` — legacy text; у goroutine profile `debug=2` печатает stacks в panic-like форме. Не следует обращаться к низкоуровневым `runtime.MemProfile`, `MutexProfile` или `GoroutineProfile`, если задачу уже решает `runtime/pprof`: низкоуровневый API заставляет самостоятельно обрабатывать races размера snapshot и формат.

HTTP API удобен для долгоживущего service и remote collection:

![[90 Вложения/CurseHunter/6817/Кадры/60-pprof-http-endpoints-00-38-00.jpg]]

- `/debug/pprof/profile?seconds=20` — новый CPU profile за заданное окно;
- `/debug/pprof/heap` — memory snapshot, `/debug/pprof/heap?gc=1` предварительно запускает GC;
- `/debug/pprof/allocs`, `/block`, `/mutex`, `/goroutine`, `/threadcreate` — named profiles;
- `/debug/pprof/trace?seconds=5` — execution trace;
- `seconds=N` для named profiles в Go `1.24.1` строит delta между двумя snapshots.

Начиная с Go `1.22`, endpoints принимают только `GET`. При custom `ServeMux` handlers нужно зарегистрировать явно. На слайде курса есть ошибка: `/debug/pprof/threadcreate` связан с `pprof.Handler("goroutine")`. Правильная registration:

```go
mux.Handle("GET /debug/pprof/threadcreate", pprof.Handler("threadcreate"))
```

Profiling endpoints раскрывают command line, stacks, function names и особенности нагрузки, а CPU profile и trace ещё и создают измеримый overhead. Их не публикуют в открытый Internet: отдельный listener/private network, authentication и time limits являются частью production design.

### 5. Что означают profile rates и почему значение `1` опасно оставлять постоянно?

![[90 Вложения/CurseHunter/6817/Кадры/61-profile-rates-00-39-00.jpg]]

| Настройка Go `1.24.1` | Единица и sampling | Специальные значения | Default |
|---|---|---|---|
| `SetBlockProfileRate(rate)` | В среднем один blocking event на `rate` nanoseconds blocked time | `1` — каждый event; `<=0` — off | `0`, off |
| `SetMutexProfileFraction(rate)` | В среднем `1/rate` mutex contention events | `1` — каждый event; `0` — off; `<0` — только прочитать rate | `0`, off |
| `MemProfileRate` | В среднем одна allocation на `rate` allocated bytes | `1` — каждый block; `0` — off | `512 * 1024` bytes |
| `StartCPUProfile` | Около `100` samples на CPU-second | Частота фиксирована API | Профиль off до старта |

![[90 Вложения/CurseHunter/6817/Кадры/68-mem-profile-rate-contract-01-21-00.jpg]]

`MemProfileRate` меняют не динамически по каждому endpoint call, а один раз как можно раньше: pprof tools предполагают постоянный rate на протяжении process. Значение `1` полезно для короткого контролируемого experiment, но recording каждого allocation или contention event способен резко увеличить overhead и memory footprint.

![[90 Вложения/CurseHunter/6817/Кадры/69-mutex-profile-rate-contract-01-22-00.jpg]]

Слайд ставит `runtime.SetCPUProfileRate(1)` рядом с остальными knobs, однако это не рекомендуемая настройка для `runtime/pprof`: `1 Hz` даёт слишком мало samples для большинства коротких экспериментов, а прямой API нельзя перенастроить, пока profiler включён. Обычный code использует `StartCPUProfile`, который в этой версии сам включает `100 Hz`.

### 6. Как читать `Duration`, `Total samples`, `flat` и `cum`?

![[90 Вложения/CurseHunter/6817/Кадры/62-pprof-commands-00-45-00.jpg]]

Лекция собирает profiles под нагрузкой и открывает их командами:

```bash
go tool pprof -http=:8081 -seconds=20 http://127.0.0.1:8080/debug/pprof/profile
go tool pprof -http=:8081 http://127.0.0.1:8080/debug/pprof/heap
curl 'http://127.0.0.1:8080/debug/pprof/trace?seconds=20' -o trace.out
```

В header `Duration: 20.13s, Total samples = 7.64s (37.96%)` первое число — wall interval сбора, второе — суммарный sampled CPU time. Это примерно `0.38` занятого CPU core в среднем, а не latency request. В multi-core process `Total samples / Duration` может быть больше `100%`, потому что несколько threads исполняются параллельно.

![[90 Вложения/CurseHunter/6817/Кадры/63-pprof-top-flat-cum-00-54-00.jpg]]

- `flat` — resource attributed непосредственно function: sample остановился в её собственном code;
- `cum` — `flat` этой function плюс resource descendants ниже неё в sampled call stacks;
- большой `flat` указывает на self hot spot;
- маленький `flat`, но большой `cum` указывает на orchestration/caller, под которым дорога другая работа;
- `sum%` — running sum rows текущей сортировки, а не отдельная метрика function;
- `Dropped ... (cum <= ...)` означает pruning малых nodes из отображения, а не потерю исходных samples.

Поэтому `top` и `top -cum` отвечают на разные вопросы. Оптимизировать wrapper только из-за высокого `cum` бессмысленно: нужно спуститься к descendants с высоким `flat` или понять, почему caller запускает дорогой path.

В показанном graph UI цвет node кодирует `cum`, размер — `flat`; цвет и толщина edge показывают ресурс на call path, а dotted edge означает пропущенные промежуточные nodes. Это визуальные defaults конкретного pprof view: при других options сначала читают legend, а не запоминают цвета.

### 7. Почему line view и disassembly не являются точным таймером строки Go?

![[90 Вложения/CurseHunter/6817/Кадры/66-pprof-source-disassembly-01-11-00.jpg]]

CPU sample привязан к program counter machine instruction. Pprof затем через symbol/debug information отображает PC на source line и учитывает inline frames. Compiler мог объединить expressions, переместить code, inline call, удалить dead branch или оставить несколько instructions для одной строки. Поэтому `160 ms` возле `a++` означает samples, сопоставленные с instructions этой строки, а не встроенный stopwatch вокруг оператора.

Рабочий порядок анализа:

1. `top` находит дорогие functions по `flat` и `cum`.
2. `tree`/graph показывает callers и callees.
3. `list <regexp>` или source view локализует строки.
4. Disassembly объясняет, какие instructions действительно выполнялись.
5. Hypothesis проверяется benchmark, новым profile под той же нагрузкой и application metrics.

Демонстрация курса помечает `f1`–`f5` как `//go:noinline`, чтобы call graph не исчез после inlining. В `f3` дорогая ветка срабатывает случайно примерно в одном случае из ста. Короткое окно может её не увидеть либо, наоборот, переоценить; это намеренный пример sampling variance.

### 8. Как собрать профиль, которому можно доверять, и использовать его для regression/PGO?

Минимальный protocol experiment:

1. Зафиксировать binary revision, Go version, `GOOS/GOARCH`, config и profile rates.
2. Прогреть service и подать репрезентативный mix requests, а не один удобный endpoint.
3. Собирать baseline и candidate одинаковое wall time при сопоставимой throughput/concurrency.
4. Сохранить raw profiles: UI screenshot не позволяет повторно фильтровать и сравнивать data.
5. Проверить CPU utilization, GC, errors и latency рядом с profile; profile без контекста легко объясняет не тот режим.
6. Повторить несколько раз и искать устойчивый signal, а не единственный красивый graph.

Profile-guided optimization (PGO) использует тот же CPU pprof как compiler input. Compiler поддерживает PGO начиная с Go `1.20`, а автоматический поиск `default.pgo` включён по умолчанию начиная с Go `1.21`. Representative production profile помогает compiler принимать решения об inlining и devirtualization; узкий microbenchmark описывает слишком малую часть application и способен оптимизировать не тот workload. PGO не заменяет diagnosis: сначала проверяют correctness и bottleneck, затем сравнивают обычный и `-pgo` build на одинаковой нагрузке.

### Что проверяет хороший ответ по уроку 11

Кандидат отделяет on-CPU sampling от wall-clock tracing, выбирает profile по симптому и знает blind spots каждого. Он объясняет signal-handler constraints и `profBuf`, различает `flat` и `cum`, правильно читает `Duration` и CPU-seconds, не включает rate `1` без оценки overhead. В production design он защищает HTTP endpoints, фиксирует workload и версии, сохраняет raw profile и подтверждает improvement повторным experiment.

### 1. Чем `allocs` отличается от `heap`, а `objects` — от `space`?

![[90 Вложения/CurseHunter/6817/Кадры/70-heap-profile-types-00-03-00.jpg]]

Один memory profile несёт несколько sample types. Выбор отвечает на разные вопросы:

| Sample type | Что агрегируется | Какой вопрос задаём |
|---|---|---|
| `alloc_objects` | Число всех sampled allocations с начала process, включая уже собранные | Где создаётся больше всего объектов и откуда берётся GC pressure? |
| `alloc_space` | Суммарные bytes всех sampled allocations с начала process | Где возникает allocation traffic по объёму? |
| `inuse_objects` | Число sampled объектов, остающихся live на учтённом GC snapshot | Где удерживается много объектов независимо от их размера? |
| `inuse_space` | Bytes sampled объектов, остающихся live на учтённом GC snapshot | Какие allocation sites удерживают heap по объёму? |

`alloc_*` — накопительный счётчик: снижение текущего heap не уменьшает уже накопленные allocations. `inuse_*` — snapshot live set, но не «память прямо сейчас»: Go `1.24.1` описывает heap profile по последнему завершённому GC и отбрасывает более новые allocations, чтобы не смещать профиль в сторону ещё не собранного garbage. Если GC ещё ни разу не завершался, действует отдельное исключение и показываются все известные allocations.

Из этого получается простой порядок диагностики leak:

1. `alloc_space` растёт, `inuse_space` после GC остаётся примерно постоянным — это allocation churn, а не доказанный leak.
2. `inuse_space` устойчиво растёт на одинаковых фазах нагрузки и одинаковых post-GC snapshots — есть гипотеза об удержании memory.
3. `inuse_objects` растёт быстрее `inuse_space` — вероятны многочисленные мелкие объекты.
4. `inuse_space` растёт при умеренном `inuse_objects` — вероятны крупные buffers или backing arrays.

Profiles sampled, поэтому их нельзя вычитать из RSS. RSS также включает stacks, executable mappings, runtime metadata, non-Go allocations и resident pages, которые allocator ещё не вернул ОС.

### 2. Почему mutex profile указывает на `Unlock`, а block profile — на ожидающую сторону?

![[90 Вложения/CurseHunter/6817/Кадры/71-block-mutex-profiling-00-06-00.jpg]]

Оба профиля измеряют ожидание, но сохраняют разные causal stacks:

- **block profile** записывает stack в точке, где goroutine заблокировалась: `Mutex.Lock`, channel send/receive/select, `WaitGroup`, `Cond` и другие synchronization primitives. Sample value — приблизительное накопленное blocked time этого stack;
- **mutex profile** записывает stack завершения critical section, которая задержала конкурентов, то есть обычно `Mutex.Unlock` или `RWMutex.Unlock`. Sample value — приблизительная сумма времени, которое другие goroutines ждали этот lock.

Если holder держал lock `1 s`, а пять contenders ждали всю секунду, mutex profile вправе приписать его `Unlock` примерно `5 s` contention. Это не wall duration critical section и не невозможные «пять секунд внутри одной секунды»: агрегируется время пяти ожидающих goroutines.

Rates тоже имеют разные единицы:

```go
runtime.SetMutexProfileFraction(n) // в среднем 1/n contention events
runtime.SetBlockProfileRate(ns)    // цель: один event на ns blocked time
```

Для mutex `0` выключает сбор, отрицательное значение только читает текущий rate. Для block `rate <= 0` выключает сбор, `1` включает каждый blocking event. В курсе оба rate выставлены в `1`, чтобы быстро получить наглядный профиль. Это диагностический режим, а не production default: частые короткие события делают overhead особенно заметным.

### 3. Когда `sync.RWMutex` действительно лучше `sync.Mutex` и что доказывает профиль курса?

В демонстрации один endpoint последовательно вызывает четыре writes по `10 ms` и один read `readF3` по `200 ms`. В варианте с обычным `Mutex` read берёт exclusive lock и сериализует все запросы. В варианте с `RWMutex` `readF3` использует `RLock`, поэтому несколько readers могут выполнять долгий участок одновременно, пока нет writer.

![[90 Вложения/CurseHunter/6817/Кадры/72-mutex-rwmutex-diff-00-18-00.jpg]]

Это не общее доказательство превосходства `RWMutex`. Он выигрывает при достаточно длинных, параллельных и преобладающих reads. Короткие critical sections, частые writes, низкая concurrency и cache-coherence cost bookkeeping могут сделать обычный `Mutex` дешевле. Сначала нужна корректность: `RLock` допустим только для реально read-only инварианта; скрытая mutation под `RLock` создаёт data race.

На diff graph зелёный `readF3` с отрицательным значением означает, что в выбранном направлении сравнения contention в candidate меньше baseline. Знак зависит от порядка source и `-diff_base`; по одному цвету нельзя объявлять улучшение, не записав команду, нагрузку, длительность и totals обоих профилей.

### 4. Зачем named profiles собирать как delta и почему в них виден сам HTTP collector?

`heap`, `allocs`, `block` и `mutex` содержат cumulative state. В Go `1.24.1` query `seconds=N` для поддерживаемого named profile делает два snapshots, ждёт `N` секунд и возвращает delta. Это удобнее, чем анализировать всю историю process, когда нужен конкретный workload window:

```bash
go tool pprof -http=:8081 \
  'http://127.0.0.1:8080/debug/pprof/block?seconds=20'
```

![[90 Вложения/CurseHunter/6817/Кадры/73-block-profile-delta-00-27-00.jpg]]

В показанном block delta заметна ветка `pprof.handler.serveDeltaProfile → runtime.selectgo`: сам request ждёт timer окончания окна и попадает в собираемый profile. Это observer effect, а не обязательно bottleneck приложения. При анализе нужно отделить collector infrastructure от business stacks, проверить абсолютный вклад и повторить опыт с внешним pull collector, если self-observation мешает.

### 5. Чем execution trace отличается от pprof и когда он нужен?

Pprof агрегирует samples по stacks и отвечает «где накопился ресурс». Execution trace сохраняет временную последовательность runtime events и отвечает «что происходило до и после»: running/blocked goroutines, scheduler latency, syscalls, network waits, GC phases, STW и causal unblock flows.

![[90 Вложения/CurseHunter/6817/Кадры/75-execution-trace-gc-timeline-00-33-00.jpg]]

Trace получают программно через `runtime/trace`, из `go test -trace` или через HTTP endpoint:

```bash
curl 'http://127.0.0.1:8080/debug/pprof/trace?seconds=20' -o trace.out
go tool trace trace.out
```

Из trace можно построить pprof-like views:

```bash
go tool trace -pprof=net trace.out > net.pprof
go tool trace -pprof=sync trace.out > sync.pprof
go tool trace -pprof=syscall trace.out > syscall.pprof
go tool trace -pprof=sched trace.out > sched.pprof
```

![[90 Вложения/CurseHunter/6817/Кадры/76-trace-goroutine-wait-table-00-39-00.jpg]]

Урок показывает разложение времени каждой goroutine на execution, network/select/sync/syscall block, scheduler wait и syscall execution. Это помогает отличить «медленно исполнялась» от «долго ждала». Цена — больший поток событий, объём файла и perturbation, поэтому trace обычно снимают коротким контролируемым окном, а pprof оставляют для более длинного statistical observation.

### 6. Что показывает Minimum Mutator Utilization и как читать эксперимент с `sync.Pool`?

Mutator — application code, выполняющий полезную работу между затратами GC. Для window длительности `W` mutator utilization — доля CPU capacity, доступная mutator внутри этого окна. Minimum Mutator Utilization (MMU) берёт худшее окно такой длины. Поэтому точка `MMU = 0.917` при `W ≈ 658 ms` означает: в худшем примерно 658-миллисекундном интервале около `91.7%` capacity оставалось mutator, а не «GC всегда съедал 8.3% CPU».

![[90 Вложения/CurseHunter/6817/Кадры/77-trace-mmu-00-42-00.jpg]]

UI позволяет отдельно включать STW, background GC workers, mark assists и sweep, а также переключаться между system-wide и per-goroutine view. Выбранные флаги входят в определение «GC cost», поэтому две кривые сравнивают только при одинаковой конфигурации.

В примере endpoint либо каждый раз создаёт `1 MiB` buffer, либо берёт его из `sync.Pool`. Pool может снизить allocation rate и GC pressure, но его контракт слабее cache:

- любой item может исчезнуть автоматически без уведомления;
- `Get` не обещает вернуть ранее положенный object;
- Pool подходит для временных объектов, разделяемых concurrent clients, а не для обязательного владения или resource pool;
- крупный buffer после редкого пика может временно увеличить retained memory; нужны limits и измерения representative workload.

Задача кандидата — сопоставить `alloc_space`, GC events, latency и MMU, а не выбрать Pool только потому, что одна кривая выше.

### 7. Какие форматы goroutine profile выбирать для incident и continuous profiling?

![[90 Вложения/CurseHunter/6817/Кадры/78-goroutine-profile-formats-00-47-00.jpg]]

В курсе сопоставлены четыре интерфейса:

- `pprof.Lookup("goroutine").WriteTo(w, 2)` внутри вызывает `runtime.Stack(..., true)` и выдаёт human-readable dump отдельных goroutines: goroutine id, state/wait reason и symbolized stacks. Это удобно для единичного incident и поиска групп одинаковых waits;
- `WriteTo(w, 1)` выдаёт legacy text count profile: одинаковые stacks агрегированы, доступны labels, но теряется индивидуальная metadata;
- `WriteTo(w, 0)` выдаёт protobuf pprof с агрегированными stacks и labels — удобнее для storage, diff и continuous profiling;
- `runtime.GoroutineProfile` возвращает `[]StackRecord` с program counters. Это низкоуровневый API; документация прямо советует большинству clients использовать `runtime/pprof`.

Goroutine snapshot не содержит историю: stack «ждёт channel receive» не говорит, сколько она ждёт, кто должен был отправить и был ли это нормальный worker. Для leak-гипотезы нужны snapshots во времени, группировка одинаковых stacks, request/domain metrics и trace причинной последовательности.

### 8. Почему `threadcreate` редко помогает найти причину роста OS threads?

![[90 Вложения/CurseHunter/6817/Кадры/74-threadcreate-limitations-00-28-00.jpg]]

Идея profile — показать stacks, приведшие к созданию OS threads, например когда cgo DNS calls блокируют threads и runtime создаёт новые. Но в Go `1.24.1` `runtime/pprof` всё ещё содержит прямую оговорку к issue `6104`: profile оборачивается отдельно, потому что полезные stack traces часто отсутствуют. Поэтому `threadcreate` может подтвердить рост числа threads, но отсутствие причинного stack не опровергает проблему.

Практический fallback: сопоставить thread count, goroutine dump, cgo metrics, execution trace, OS-level thread stacks и blocked syscalls. Не путать OS thread leak с goroutine leak: M:N scheduler может менять число M без пропорционального изменения G.

### 9. Как выглядит воспроизводимый PGO workflow и почему нужен representative profile?

![[90 Вложения/CurseHunter/6817/Кадры/79-pgo-overview-00-50-00.jpg]]

Go PGO принимает CPU pprof и использует hotness при compiler decisions, прежде всего более агрессивном inlining и devirtualization. Типовой цикл:

```text
обычная production build → representative CPU profiles
→ merge/отбор профиля → build с PGO
→ одинаковый benchmark/load test → rollout и новые profiles
```

Стандартный путь — положить `default.pgo` в каталог `main` package. При `-pgo=auto` `go build` найдёт его автоматически; `-pgo=off` создаёт честный non-PGO baseline, а `-pgo=/path/profile.pprof` выбирает файл явно.

![[90 Вложения/CurseHunter/6817/Кадры/80-default-pgo-00-57-00.jpg]]

PGO применяется ко всей программе, включая standard library и dependencies, которые пересобираются с учётом профиля. Один path, переданный `-pgo`, применяется ко всем `main` packages в данной build invocation, поэтому разные binaries обычно собирают отдельно со своими profiles.

Длинный CPU profile сам по себе ещё не становится representative. Нужны production traffic mix, разные instances и фазы дня либо аккуратно взвешенный merge. При merge `go tool pprof -proto a.pprof b.pprof` просто суммирует samples, поэтому более длинное окно получает больший вес. Microbenchmark обычно покрывает слишком узкий path для whole-program PGO.

### 10. Насколько PGO устойчив к изменению исходников?

Go PGO рассчитан на source skew между profiled и новой версиями. Samples сопоставляются с symbol и line offsets внутри function; несовпадение одной function приводит к graceful degradation её optimizations, а не делает весь profile недействительным.

![[90 Вложения/CurseHunter/6817/Кадры/83-pgo-source-stability-01-14-00.jpg]]

Обычно не ломают matching изменения вне hot function и перенос function в другой файл того же package. Могут сломать matching:

- изменение строк внутри hot function;
- rename function или receiver type;
- перенос function в другой package;
- крупный refactoring сразу многих hot symbols.

Новый code path отсутствует в старом profile и не получит PGO benefit до следующего цикла. Поэтому `default.pgo` — versioned build input, который регулярно обновляют; это не вечный artifact. Курс отдельно показывает compiler diagnostics `-gcflags='-m=4'`: они помогают увидеть inlining/escape decisions, но сами по себе не доказывают runtime improvement.

### 11. Чем `-diff_base`, `-base` и `-normalize` отличаются друг от друга?

![[90 Вложения/CurseHunter/6817/Кадры/81-pprof-diff-base-command-01-09-00.jpg]]

В pprof, vendored в Go `1.24.1` на commit `b92577c0c142`, semantics такие:

- `-diff_base=base.pprof source.pprof` — сравнить два совместимых профиля; percentages считаются относительно total base, entries могут быть отрицательными;
- `-base=earlier.pprof later.pprof` — вычесть более ранний cumulative profile из более позднего того же process; percentages считаются относительно разницы totals, в штатном cumulative случае values положительные;
- `-normalize` перед subtraction масштабирует source до total base и помогает сравнивать доли, когда totals различаются.

![[90 Вложения/CurseHunter/6817/Кадры/82-pprof-diff-negative-values-01-12-00.jpg]]

Отрицательное `-0.12 s` означает уменьшение относительно base. Это не latency ниже нуля. Для diff flame graph width учитывает сумму абсолютных increases/decreases subtree, а shaded area показывает net change; поэтому широкая frame может иметь небольшой net.

Сравнение валидно лишь при одинаковых binary symbols или корректной symbolization, равных окнах, сопоставимой throughput/concurrency и representative workload. Для CPU profile абсолютные CPU-seconds полезнее одного процента: percentage может вырасти лишь потому, что сильнее уменьшился другой path.

### 12. Что даёт continuous profiling и какую цену нужно заложить?

Continuous profiler периодически собирает profiles со множества instances, добавляет labels и сохраняет time series. Это позволяет:

- увидеть медленный drift, который не попал в ручное 30-секундное окно;
- сравнить releases, instances, tenants и time ranges;
- связать regression с deployment;
- получить более representative input для PGO.

В демонстрации используется `github.com/grafana/pyroscope-go v1.2.0` и включены CPU, четыре memory sample types, goroutine, mutex count/duration и block count/duration.

![[90 Вложения/CurseHunter/6817/Кадры/84-pyroscope-profile-config-01-16-00.jpg]]

У SDK CPU и четыре memory profiles входят в defaults, остальные включаются явно. Mutex/block требуют отдельно настроенных runtime rates. `Profiler.Stop` останавливает session и отправляет остаток данных; игнорирование возвращённого profiler может потерять последнее окно при graceful shutdown. Labels делают данные полезнее, но high-cardinality values вроде request ID или user ID раздувают storage и ухудшают агрегацию.

![[90 Вложения/CurseHunter/6817/Кадры/86-pyroscope-profile-types-01-27-00.jpg]]

Overhead складывается из sampling, stack unwinding, symbolization/compression, GC для heap snapshots, network upload и хранения. Все rates `1` одновременно, full goroutine snapshots и forced GC на большом heap нельзя включать без измерения. Production rollout начинают с одной instance, сравнивают CPU/RSS/latency и lost samples, затем расширяют охват.

### 13. Какие ошибки намеренно заложены в leak demo и как их найти по профилям?

Курс запускает демонстрационный service с отключённым automatic GC и практически снятым soft memory limit:

![[90 Вложения/CurseHunter/6817/Кадры/85-leak-demo-gc-disabled-01-19-00.jpg]]

В source endpoint делает `go s.leak()`, а `leak` выглядит так:

```go
func (s *server) leak() {
    h, _ := http.Get("https://www.google.com")
    body, _ := io.ReadAll(h.Body)

    s.mx.Lock()
    defer s.mx.Unlock()
    global = append(global, body)

    go s.leak()
}
```

Это специально небезопасный код. Хороший разбор перечисляет независимые failure modes:

1. Каждый request запускает цепочку, где каждая итерация создаёт следующую goroutine. Число network operations и goroutines растёт без bound и lifecycle cancellation.
2. `http.Get` использует default client без общего request timeout. Зависший upstream может удерживать goroutine и resources неопределённо долго.
3. Errors отброшены. При ошибке `h` может быть `nil`, и `h.Body` вызовет panic.
4. `Body.Close` не вызывается. Это мешает корректному освобождению/переиспользованию connection resources.
5. `io.ReadAll` не ограничивает response size.
6. `global` навсегда удерживает каждый `body`; это собственно reachable heap leak. Mutex защищает append от data race, но не вводит limit.
7. `debug.SetGCPercent(-1)` отключает automatic GC, а `debug.SetMemoryLimit(math.MaxInt64)` практически убирает soft memory bound. Это усиливает рост и искажает обычное production поведение; само по себе не является исправлением или причиной reachable leak.
8. В `docker-compose.yml` указан `grafana/pyroscope:latest` и memory limit `1000000M`: результат невоспроизводим и не моделирует реальный container limit.

Для диагностики нужны одновременно:

- `inuse_space` во времени — увидеть удерживаемые bytes;
- `alloc_space` — отделить lifetime allocations от live set;
- goroutine profile — увидеть множащиеся stacks `leak`/HTTP;
- network/trace/metrics — увидеть blocked requests и connection pressure;
- comparison одного и того же profile type на равных time ranges.

![[90 Вложения/CurseHunter/6817/Кадры/87-pyroscope-inuse-comparison-01-28-00.jpg]]

Исправление строится вокруг ownership: bounded worker/concurrency, `context` и timeout, обязательные error checks и `defer resp.Body.Close()` после успешного response, `io.LimitReader`, bounded storage/eviction и production-like GC/container limits. Просто «чаще запускать GC» не освобождает объекты, на которые всё ещё ссылается `global`.

### Практические задания по материалу урока

#### Задание 1. Выбрать memory sample type по четырём симптомам

Для каждого случая назвать первый profile, blind spot и подтверждающий signal:

1. `alloc_space` быстро растёт, post-GC `inuse_space` стабилен.
2. Post-GC `inuse_objects` и `inuse_space` растут вместе.
3. RSS растёт, Go `inuse_space` стабилен.
4. Latency скачет одновременно с GC, live heap не растёт.

Ожидается reasoning, а не список: churn → `alloc_*` и GC metrics; retention → series `inuse_*`; RSS gap → stacks/mappings/non-Go memory; latency → trace/MMU/mark assists и allocation sites.

### 1. Как выбирать `GOMAXPROCS` для Go-сервиса в Kubernetes?

![[90 Вложения/CurseHunter/6817/Кадры/88-gomaxprocs-kubernetes-00-10-45.jpg]]

В Q&A лектор открывает пример Kubernetes manifest с `limits.cpu: 1000m`, `requests.cpu: 750m` и `GOMAXPROCS: "1"`. Для версии Go, на которой записан курс, это осмысленная ручная настройка: до Go `1.25` runtime по умолчанию ориентировался на число logical CPUs машины, а не на cgroup CPU quota контейнера. На большой node процесс мог получить высокий `GOMAXPROCS`, быстро исчерпать quota и попасть под CPU throttling.

`request` и `limit` отвечают на разные вопросы:

- `requests.cpu` участвует в scheduling и резервировании capacity. Контейнер вправе использовать больше request, если node не перегружена;
- `limits.cpu` задаёт жёсткий throughput budget, который Linux kernel обеспечивает через cgroup CPU throttling;
- `GOMAXPROCS` ограничивает параллелизм Go-кода: сколько OS threads одновременно исполняют goroutines. Это не CPU quota и не гарантия выделенного core.

Поэтому в примере курса ручное значение выводится из `limits.cpu = 1000m`, а не из `requests.cpu = 750m`. Но равенство приблизительное по модели: quota ограничивает CPU time за период, тогда как `GOMAXPROCS` ограничивает одновременное исполнение. Для bursty workload более низкий `GOMAXPROCS` иногда убирает throttling и улучшает tail latency, но ценой запрета полезного краткого burst. Решение проверяют по `container_cpu_cfs_throttled_*`, CPU utilization, throughput и latency percentiles.

Начиная с language version Go `1.25` Linux runtime использует container-aware default. В Go `1.26.5` он обычно выбирает минимум из:

1. числа logical CPUs;
2. размера CPU affinity mask;
3. cgroup quota / period, округлённого вверх.

Одна лишь низкая quota не опускает default ниже `2`; значение `1` возможно, если меньше двух logical/affinity CPUs. Runtime проверяет изменения до одного раза в секунду. Явная переменная окружения `GOMAXPROCS` или вызов `runtime.GOMAXPROCS` выключает automatic updates; восстановить default можно через `runtime.SetDefaultGOMAXPROCS`. Для `go 1.24` и ниже container-aware режим по умолчанию выключен даже при новом toolchain, если его не переопределили через `GODEBUG`.

Практический ответ должен сначала назвать Go language version и platform. Для `go 1.25+` ручное дублирование limit обычно не требуется; для старой версии нужны явная настройка или совместимая библиотека. `requests.cpu` runtime намеренно не использует: иначе процесс не смог бы задействовать свободную CPU capacity сверх request.

### 2. Доступ к heap сам по себе медленнее доступа к stack?

![[90 Вложения/CurseHunter/6817/Кадры/89-heap-stack-access-00-25-00.jpg]]

Короткий ответ: у CPU нет отдельной «медленной heap-памяти» и «быстрой stack-памяти». И stack, и heap находятся в virtual address space процесса; конкретный load проходит через TLB, cache hierarchy и при промахе — через RAM или page fault. Cache-hot heap object может читаться быстрее, чем stack page, которую пришлось вернуть из swap.

Репутация stack как более быстрого пути возникает из типичной формы кода:

- local value часто остаётся в register, и обращения к памяти вообще нет;
- stack slot адресуется относительно stack pointer с известным offset;
- bump/debump stack pointer дешевле общей heap allocation;
- близкие stack values обычно имеют хорошую locality и короткий lifetime;
- heap object чаще требует pointer chasing, может лежать дальше от потребителя и добавляет работу allocator/GC.

Это различия generated code, lifetime и locality, а не latency физического типа памяти. В Go escape analysis переносит value в heap, когда compiler не может доказать, что его lifetime укладывается в stack frame, или когда объект слишком велик для выбранной реализации. Сам факт escape не доказывает дополнительный cache miss; он добавляет allocation/GC obligations и нередко indirection. Обратная ситуация тоже возможна: большой stack working set вытесняет cache, а многократно читаемый heap object остаётся hot.

На интервью полезно разложить ошибку так: неверное предположение «heap-load всегда медленнее» приводит к попыткам любой ценой удержать всё на stack; причина реального slowdown обычно в allocations, GC, pointer indirection или locality; исправление начинают с escape diagnostics, allocation profile и hardware/runtime measurements, а не с `unsafe`-трюков.

### 3. Чем `const string` отличается от `var string` и обязаны ли одинаковые literals занимать одно место?

![[90 Вложения/CurseHunter/6817/Кадры/90-string-literal-assembly-00-29-00.jpg]]

String constant — compile-time value. Untyped constant получает default type `string`, когда контекст требует обычного value. Переменная — storage location: `var s = "text"` создаёт изменяемый string header, который можно переназначить, но bytes строки через Go API по-прежнему нельзя изменять.

Из `var` не следует отдельная heap-копия literal. В Go `1.24.1` compiler помещает literal data в read-only symbol через `staticdata.StringSym`. Имя короткой строки выводится из её содержимого, длинной — из длины и hash; symbol помечается `RODATA`, `DUPOK` и content-addressable. Поэтому две функции из демонстрации с literal `"Hello, Go course"` ссылаются на один и тот же string-data symbol в показанном binary.

Это свойство конкретной toolchain, не гарантия Go language. Спецификация гарантирует значения и операции над strings, но не identity backing storage, адрес symbol или deduplication между packages/build modes. Нельзя строить correctness на сравнении `unsafe.StringData` либо считать, что любое повторение literal увеличивает binary ровно на его длину.

`const` тоже не означает «всегда быстрее». Его преимущество появляется только там, где compile-time knowledge разрешает folding, propagation, удаление dead code или специализацию. Если обе версии в итоге выполняют одинаковые loads и loop, разницы может не быть. Если compiler полностью вычисляет или выбрасывает один вариант, benchmark уже сравнивает разные programs.

### 4. Почему benchmark курса сначала показывает `const` быстрее в десятки раз, а затем разница исчезает?

В live coding сравниваются два sub-benchmarks примерно такой формы:

```go
for b.Loop() {
    for i := range myString {
        a := i
        _ = a
    }
}
```

Здесь нет observable result. Присваивание в локальную `a`, а затем blank identifier не заставляют compiler сохранить работу. `B.Loop` в Go `1.24.1` защищает от удаления только function calls, синтаксически находящиеся внутри `for b.Loop() { ... }`; произвольный dead loop такой гарантии не получает.

![[90 Вложения/CurseHunter/6817/Кадры/91-const-var-empty-loop-00-35-00.jpg]]

Показанные на Apple M3 Pro результаты меняются вместе с формой input:

| Вариант демонстрации | `var` | `const` | Что можно заключить |
| --- | ---: | ---: | --- |
| Строка из `100` ASCII bytes | `18.29 ns/op` | `18.43 ns/op` | В этой сборке остаточная работа почти одинакова |
| Строка из `10 000` ASCII bytes; `var` получена через `strings.Repeat`, `const` известна compiler | `2802 ns/op` | `31.53 ns/op` | Compiler удалил или резко упростил больше работы в constant branch; это не измерение latency чтения 10 000 bytes |
| После дальнейшего изменения кода | `2810 ns/op` | `3430 ns/op` | Результат чувствителен к code shape; один запуск не задаёт устойчивого ranking |

Go `1.24.1` lowering превращает `range` по string в index loop с ASCII fast path и вызовом `decoderune` для остальных bytes. Когда весь input известен и body не имеет эффекта, последующие SSA passes получают больше возможностей убрать loop. Runtime-built или mutable global string оставляет compiler меньше доказательств, хотя семантически результат цикла всё равно никому не нужен.

Затем лектор создаёт string через `[]byte`, меняет первый byte и возвращает `string(buffer)`. В отдельной goroutine читается исходная global variable. После этих изменений на коротких strings показаны близкие результаты.

![[90 Вложения/CurseHunter/6817/Кадры/92-runtime-string-setup-00-40-00.jpg]]

![[90 Вложения/CurseHunter/6817/Кадры/93-final-var-const-benchmark-00-40-45.jpg]]

Финальный запуск даёт `15.77 ns/op` для `var` и `15.79 ns/op` для `const`, по `0 B/op` и `0 allocs/op`. Это хорошо демонстрирует хрупкость исходного вывода, но не доказывает равенство: изменились input length и construction, goroutine не синхронизирована, `_ = internal` не создаёт полезного result, повторных samples и statistical comparison нет.

Корректная постановка начинается с вопроса, что именно сравнивается:

- если нужен cost обработки runtime string, обе строки должны приходить из одинакового runtime source, иметь одинаковые bytes/length и выполнять одинаковый observable checksum;
- если нужен выигрыш compile-time constant, specialization разрешают намеренно и затем объясняют generated code. Это уже другой experiment;
- если проверяется allocation, setup выносят до первого `b.Loop`, включают `ReportAllocs` и не смешивают construction со traversal;
- результат сохраняют в package-level sink или проверяют checksum, затем смотрят assembly, повторяют `-count`, фиксируют Go/flags/`GOOS`/`GOARCH` и сравнивают samples через `benchstat`.

### 1. Задание курса: реализовать watchdog и server с двусторонней связью через pipes

![[90 Вложения/CurseHunter/6817/Кадры/94-watchdog-pipes-00-01-45.jpg]]

Это буквальное домашнее задание курса. Нужно запустить два процесса и связать их двумя FIFO-каналами ОС: один pipe передаёт команды от watchdog в `stdin` server, второй возвращает результаты из `stdout` server в watchdog. Один anonymous pipe однонаправленный, поэтому для request/response нужны две пары read/write descriptors.

Watchdog предоставляет CLI, запускает server и поддерживает четыре команды:

```text
setmeml <digit>
setproc <digit>
doreq <http method> <url> <request body>
exit
```

![[90 Вложения/CurseHunter/6817/Кадры/95-watchdog-cli-commands-00-02-15.jpg]]

Путь к server приходит через `-sp`, аргументы — через `-sargs`; дочерний процесс должен получить environment родителя. В моделях задания команды превращаются во frames вида `opByte + JSON + '#'`:

| `opByte` | Операция | Запрос и ответ |
| ---: | --- | --- |
| `0x1` | изменить `GOMAXPROCS` | `GoMaxProcAction{Value}` → прежнее значение |
| `0x2` | изменить soft memory limit | `SetMemoryLimitAction{Value}` → прежнее значение |
| `0x3` | выполнить HTTP request | method, URL, body и `RequestID` → response/error с тем же ID |

`runtime.GOMAXPROCS(0)` и `debug.SetMemoryLimit` с отрицательным аргументом используются как query без изменения текущего значения. Это часть контракта, которую проверяют tests, а не особый случай парсера CLI.

Server читает frames до `#`, применяет операцию, пишет frame ответа и продолжает обслуживать поток. По условию collision с delimiter допускается; для production-протокола это слабое место. Если arbitrary body может содержать `#`, нужны length prefix, escaping либо формат, который однозначно задаёт границу сообщения.

![[90 Вложения/CurseHunter/6817/Кадры/96-server-protocol-platform-details-00-02-45.jpg]]

Решение должно работать на Linux, Darwin и Windows. На Unix-like системах server завершает работу по `SIGALRM`; Windows-ветка собирается отдельно и не может опираться на этот signal. Показанный CI запускает один test suite на `ubuntu-22.04`, `windows-2022` и `macos-14`.

![[90 Вложения/CurseHunter/6817/Кадры/97-cross-platform-ci-00-08-45.jpg]]

#### Ментальная модель решения

`os/exec.Cmd` уже скрывает platform-specific process creation. На Unix это путь через fork/exec primitives, на Windows — создание нового process средствами Windows; вызывать raw `fork` из многопоточного Go process не требуется. До `Start` watchdog связывает ends так:

```text
watchdog serverWriter ──> child stdin  (serverReader)
watchdogReader        <── child stdout (watchdogWriter)
child stderr           ──> parent stderr
```

После успешного `Start` parent закрывает собственные копии ends, предназначенных только child. Иначе лишний открытый writer удержит pipe живым: reader не увидит EOF и shutdown зависнет. На любой ветке ошибки уже созданные files тоже закрываются.

Жизненный цикл child задают явно:

1. запустить и передать environment/arguments;
2. обслуживать команды, сохраняя границы frames и полностью обрабатывая short writes;
3. по `exit` закрыть write side или отправить отдельную shutdown-команду;
4. дать server время завершиться, вызвать `Wait` ровно один раз;
5. после timeout принудительно остановить process и всё равно дождаться `Wait`.

Одного `exec.CommandContext` мало для graceful shutdown: стандартный `Cancel` у созданного им `Cmd` вызывает `Process.Kill`. Контекст задаёт аварийный механизм, а протокол завершения и reap child остаются ответственностью watchdog.

#### Что проверять при code review показанного reference solution

Commit `6b0e0926bd6a42f3781c506310a382aa943f9e16`, который открыт в видео, полезен именно как материал для review. Кандидат должен заметить как минимум такие failure modes:

- `strings.Split(sargs, " ")` теряет quoting, escaping и пустые arguments; аргументы лучше принимать структурированно, например повторяемым flag или JSON-массивом;
- parent не закрывает child-side descriptors сразу после `Start`, поэтому EOF и освобождение ресурсов зависят от случайного порядка shutdown;
- ветка `exit` возвращается до `cmd.Wait`; отмена context может убить child, но не заменяет обязательный reap и проверку `ProcessState`;
- goroutine записывает результат `Close` в общий `err`, пока main goroutine использует ту же переменную: это data race;
- при `cErr != nil` logger выводит старый `err`, а `processDoReq` после `doAction` возвращает `nil`, то есть реальная ошибка теряется;
- `unsafe.Slice(unsafe.StringData(body), len(body))` создаёт writable `[]byte` поверх immutable string storage; обычное `[]byte(body)` безопаснее, пока профиль не доказал цену копии;
- `fmt.Fscan` читает body как один token, поэтому JSON с whitespace не соответствует ожидаемому CLI без отдельного framing;
- protocol допускает delimiter collision и не задаёт version, maximum response size, backpressure или поведение при partial/corrupt frame.

Сильное решение не обязано копировать reference implementation. Его принимают по наблюдаемым свойствам: нет leaked descriptors/goroutines, server завершается при EOF, signal и cancellation; каждый response коррелирует с request; malformed frame не рассинхронизирует весь stream; tests проходят на трёх OS и под race detector там, где он поддерживается.

### 2. Что происходит при `panic(nil)` и почему одной версии toolchain недостаточно?

![[90 Вложения/CurseHunter/6817/Кадры/98-panic-nil-go121-contract-00-19-00.jpg]]

До Go `1.21` `panic(nil)` создавал двусмысленность: прямой вызов `recover` в deferred function останавливал panicking, но возвращал `nil`, как и вызов вне активной panic. Код вида `if r := recover(); r != nil` не мог отличить эти состояния.

Начиная с semantics Go `1.21`, runtime заменяет nil panic value на `*runtime.PanicNilError`. Поэтому во время активной panic прямой `recover` возвращает ненулевое значение:

```go
defer func() {
    r := recover()
    fmt.Printf("%T: %v\n", r, r)
}()

panic(nil) // *runtime.PanicNilError: panic called with nil argument
```

![[90 Вложения/CurseHunter/6817/Кадры/99-panic-nil-runtime-00-20-45.jpg]]

В Go `1.23.4` `runtime.gopanic` проверяет `debug.panicnil`: при обычном значении создаёт `PanicNilError`, при `panicnil=1` оставляет старое поведение. Здесь есть интервью-ловушка. Одного ответа «начиная с toolchain 1.21 всегда будет `PanicNilError`» недостаточно:

- default выбирается с учётом строки `go` главного module или workspace;
- новый toolchain сохраняет старое поведение для module, объявляющего Go `1.20` или ниже;
- `GODEBUG=panicnil=1`, а начиная с Go `1.23` ещё и директивы `godebug`, могут переопределить default.

Проверять нужно toolchain, language version главного module и effective GODEBUG. `panic/recover` при этом не превращают в обычный error handling: bounds violation, nil dereference и нарушенный invariant обычно нужно либо исправить, либо после локальной cleanup повторно вызвать `panic`.

### 3. Почему `a[10]` не читает чужую память и где возникает bounds check?

![[90 Вложения/CurseHunter/6817/Кадры/100-bounds-check-slice-00-21-45.jpg]]

В live coding создаётся `a := make([]int, 3)`, затем выполняется `a[10]`. Для slice допустим индекс `0 <= i < len(a)`, поэтому этот код вызывает runtime panic `index out of range`. Рядом набросана ручная проверка `len(a) >= 10`; она сама содержит off-by-one. Для доступа `a[10]` корректное предусловие — `10 < len(a)`, то есть `len(a) > 10`.

Language semantics требуют panic при out-of-range. Реализация не обязана исполнять отдельное сравнение перед каждым access: compiler удаляет bounds check, если докажет диапазон из loop bounds, предыдущей проверки или другого dominating condition. Это bounds-check elimination (BCE), а не отказ от memory safety. Не удалось доказать — check остаётся.

Граница compile time тоже важна:

- constant index у array с известной длиной должен быть in range, иначе программа не компилируется;
- длина slice — runtime state, поэтому constant `10` может пройти type checking и упасть при выполнении;
- subslice меняет `len`, хотя backing array может быть больше: возможность физически добраться до элемента не делает индекс допустимым.

Фраза «Go memory-safe» относится к safe language operations. `unsafe`, некорректный cgo и data race выходят за эту модель. Практическое правило: проверять business boundary на входе, а внутри простого цикла писать форму, из которой compiler видит invariant; использовать recover вместо проверки индекса нельзя.

### 4. Можно ли нескольким goroutines одновременно вызывать `PacketConn.ReadFrom`?

![[90 Вложения/CurseHunter/6817/Кадры/101-packetconn-concurrency-contract-00-34-40.jpg]]

Да. Контракт `net.PacketConn` в Go `1.23.4` прямо разрешает нескольким goroutines одновременно вызывать его методы. Но thread-safe API не обещает, что два `ReadFrom` выполнят два kernel syscalls параллельно, что packets будут справедливо распределены между readers или что application-level порядок сохранится.

В демонстрации две goroutines читают один UDP `PacketConn`, каждая в собственный `[]byte`.

![[90 Вложения/CurseHunter/6817/Кадры/102-udp-two-readers-01-04-40.jpg]]

Отдельный buffer обязателен: `ReadFrom` записывает payload в переданный slice. Общий slice без ownership protocol создаст data race, даже если сам `PacketConn` безопасен. Каждый успешный receive потребляет одну datagram; какая goroutine получит следующую, API не определяет. Нельзя использовать номер worker как часть protocol ordering.

В показанной Unix-реализации путь чтения примерно такой:

```text
UDPConn.ReadFrom
→ UDPConn.readFromUDP
→ UDPConn.readFrom
→ internal/poll.FD.ReadFromInet4
→ unix.RecvfromInet4
```

`FD.ReadFromInet4` сначала берёт `fd.readLock`. Внутренний `fdMutex` допускает один active reader и отдельно один active writer; конкурирующие reads ждут на semaphore. Lock state хранится в bit field и меняется через atomic compare-and-swap.

![[90 Вложения/CurseHunter/6817/Кадры/103-fdmutex-read-lock-00-48-20.jpg]]

Поэтому в Go `1.23.4` reads одного socket безопасны, но внутри этого process сериализуются. Это implementation detail, а не часть public contract: строить correctness или performance estimate на точной форме `fdMutex` нельзя.

![[90 Вложения/CurseHunter/6817/Кадры/104-udp-readlock-recvfrom-00-50-00.jpg]]

После `prepareRead` runtime вызывает `recvfrom`. `EINTR` приводит к повтору syscall, `EAGAIN` на pollable descriptor — к ожиданию readiness в network poller и новой попытке. Отсюда ещё четыре правила ответа:

- сначала обрабатывать `n > 0`, затем `err`, как требует контракт;
- `SetReadDeadline` общий для connection и влияет на все pending/future reads;
- `Close` разблокирует ожидающие `ReadFrom` с ошибкой и подходит для общего shutdown;
- несколько blocking readers не увеличивают число одновременно выполняемых reads в этой реализации. Их польза может быть только в том, что получившая packet goroutine сразу переходит к обработке, пока другая ждёт следующий.

Для управляемого server чаще удобнее один read loop и bounded worker queue. Reader получает buffer из pool, передаёт worker исключительное владение `buf[:n]`, а worker возвращает buffer после обработки. Если reader переиспользует slice сразу после send, workers увидят уже следующую datagram. Bounded queue задаёт backpressure внутри process, но при медленной обработке заполнится kernel receive buffer и UDP packets начнут теряться — это свойство protocol, а не ошибка `PacketConn`.

### 5. Почему thread-safe методы не делают составную операцию atomic?

![[90 Вложения/CurseHunter/6817/Кадры/105-thread-safe-compound-operation-01-07-20.jpg]]

В конце Q&A открыт классический пример с Java `Vector`: `isEmpty` и `add` синхронизированы по отдельности, но последовательность «если пусто, добавить» не atomic. Между вызовами другой thread успевает изменить container.

Тот же принцип переносится на Go API. Разрешение конкурентных calls гарантирует внутренний invariant объекта на границе каждого метода. Оно не защищает business invariant, который охватывает несколько calls, shared buffer или несколько объектов. Исправления три:

1. предоставить одну atomic operation на уровне abstraction, например `LoadOrStore` вместо `Load` + `Store`;
2. удерживать внешний lock на всей последовательности;
3. передать ownership одной goroutine и сериализовать commands через channel.

Выбор зависит от требуемого invariant. Lock на отдельном method слишком узок, global lock часто слишком широк, owner goroutine упрощает reasoning, но добавляет queueing и backpressure.

### 1. Какие системные вызовы совершает программа с `printf`?

![[90 Вложения/CurseHunter/6817/Кадры/106-syscalls-printf-question-00-02-40.jpg]]

У вопроса нет одного списка, верного для любого запуска. Сильный ответ сначала фиксирует окружение: статическая или динамическая линковка, libc и её версия, архитектура, kernel, состояние стандартных дескрипторов, терминал или файл на `stdout`. Затем разделяет три слоя:

1. исходная программа вызывает функцию `printf` из C library, а не kernel напрямую;
2. startup/termination code до и после `main` тоже обращается к ОС;
3. `strace` показывает наблюдаемые system calls всего process lifetime, включая начальный `execve`.

В демонстрации binary собран как `gcc hello.c -o hello -static`. На экране видны `execve`, `arch_prctl`, `brk`, `uname`, `readlink`, `mprotect`, `fstat`, `write` и `exit_group`.

![[90 Вложения/CurseHunter/6817/Кадры/107-static-hello-strace-00-06-40.jpg]]

`fstat(1, ...)` и один `write(1, "Hello, world!\n", 14)` относятся к реализации buffered stdio в этом запуске. Но из source нельзя вывести, что каждый `printf` немедленно породит один `write`: stream в terminal обычно line-buffered, stream в regular file обычно fully buffered, а несколько вызовов могут слиться в один flush. Динамически связанный binary обычно добавит работу dynamic loader, например открытия и mappings shared libraries. Поэтому правильный короткий ответ звучит так: «минимально ожидаю вывод через `write` и завершение process, точный trace зависит от runtime; проверю `strace` на заданном binary».

Дальше лектор убирает `printf`: сначала вызывает `write(STDOUT_FILENO, ...)`, затем задаёт собственную точку входа `_start` и использует `syscall(SYS_write, ...)` с `syscall(SYS_exit, 0)`.

![[90 Вложения/CurseHunter/6817/Кадры/108-start-syscall-write-exit-00-15-20.jpg]]

Тонкость для Senior+ ответа: `syscall()` на слайде всё ещё небольшая функция libc, которая выполняет architecture-specific переход в kernel. Это ближе к syscall ABI, но код по-прежнему не содержит handwritten `syscall` instruction. `SYS_exit` на Linux завершает только calling thread; wrapper `_exit` в glibc с версии 2.3 вызывает `exit_group`, чтобы завершить все threads process. Для однопоточного примера результат совпадает. Низкоуровневый exit также не flushes `FILE*`, поэтому печатать через buffered stdio перед ним нельзя без явного flush.

### 2. Что такое file descriptor и почему это не сам файл?

![[90 Вложения/CurseHunter/6817/Кадры/109-file-descriptor-read-write-00-19-20.jpg]]

File descriptor (FD) — маленькое неотрицательное число, индекс записи в таблице descriptors конкретного process. Запись ссылается на open file description в kernel, где лежат file offset и status flags. Один и тот же интерфейс `read`/`write` поэтому работает с regular file, terminal, pipe и socket, хотя поведение объекта различается.

При обычном запуске `0`, `1`, `2` назначены `stdin`, `stdout`, `stderr`, но это соглашение, а не зарезервированные навсегда числа. Если закрыть `0`, следующий успешный `open` вправе вернуть `0`. На слайде сказано «минимальное положительное число»; для Linux это неточно: `open` возвращает lowest-numbered descriptor, который сейчас свободен, включая ноль.

`read` и `write` возвращают число реально переданных bytes. Успех не означает `n == count`, поэтому generic I/O loop обязан:

- продвигать pointer/offset после partial read или write;
- повторять вызов после `EINTR`, если operation ещё нужна;
- трактовать `EAGAIN`/`EWOULDBLOCK` как отсутствие готовности у nonblocking descriptor;
- считать `read == 0` EOF только для подходящего объекта;
- отдельно решать вопрос durability: успешный `write` не равен записи на стабильный носитель, для этого нужен как минимум `fsync` с учётом guarantees filesystem/storage.

`FILE*` находится уровнем выше. C stdio хранит userspace buffer, flags и состояние stream; на POSIX stream обычно связан с FD, который можно получить через `fileno`. Смешивать `fwrite` и `write` на одном underlying open file description без согласования buffering/offset опасно: два слоя имеют разное состояние. Portable C code работает с `FILE*`; descriptor semantics начинаются на POSIX boundary.

### 3. Чем отличаются `open`, `pipe`, `dup` и `dup2`?

![[90 Вложения/CurseHunter/6817/Кадры/110-open-pipe-dup-00-29-40.jpg]]

`open` создаёт новую open file description и возвращает FD. `pipe` создаёт bounded byte stream и два FD: `pipefd[0]` для чтения, `pipefd[1]` для записи. `dup(oldfd)` создаёт новый FD на ту же open file description, а `dup2(oldfd, newfd)` атомарно заставляет конкретный `newfd` ссылаться туда же.

У duplicated descriptors общие file offset и file status flags, но descriptor flags разные. В частности, `FD_CLOEXEC` относится к конкретному FD. Отсюда два частых вопроса:

- два FD, созданных через `dup`, последовательно читают один file — второй продолжит с offset, изменённого первым;
- `close(oldfd)` после успешного `dup2` не уничтожит объект, пока на open file description остаётся хотя бы одна ссылка.

Атомарность `dup2` нужна не для красоты. Последовательность `close(newfd); dup(oldfd)` содержит race: signal handler или другой thread может занять освободившийся номер между calls. `dup2` закрывает прежний `newfd` и переназначает его одним atomic step.

В примере pipe записываются четыре bytes, затем они читаются из второго конца:

![[90 Вложения/CurseHunter/6817/Кадры/111-pipe-example-00-31-40.jpg]]

Pipe не хранит application messages. Это byte stream с ограниченной capacity. Если buffer заполнен, blocking writer ждёт, а nonblocking получает `EAGAIN`; запись больше `PIPE_BUF` при нескольких writers может interleave. EOF придёт читателю лишь после закрытия **всех** copies write end. Если все read ends закрыты, writer получает `SIGPIPE`, а при ignored/blocked signal — `EPIPE`. Одного лишнего write end в parent достаточно, чтобы consumer бесконечно ждал EOF.

Перенаправление работает как замена ссылки под стандартным номером:

![[90 Вложения/CurseHunter/6817/Кадры/112-dup2-stdout-redirection-00-39-40.jpg]]

```c
int fd = open("output", O_WRONLY | O_CREAT | O_TRUNC, 0666);
if (fd == -1) { /* handle error */ }
if (dup2(fd, STDOUT_FILENO) == -1) { /* handle error */ }
if (fd != STDOUT_FILENO) close(fd);
```

После `dup2` обычный `printf` пишет в `output`. Effective permissions нового файла равны `0666 & ~umask`.

На учебном слайде у `open(..., O_CREAT | ...)` нет третьего аргумента `mode`. Это correctness и security defect: при `O_CREAT` mode обязателен, иначе в variadic call за него принимаются произвольные bytes. Позже аналогично показан `creat(outfile)` без обязательного `mode`; с видимым prototype такой код должен быть отвергнут compiler. На собеседовании учебный pseudocode нельзя копировать буквально: нужно заметить обе ошибки, проверить каждый return value и закрыть FD во всех branches.

### 4. Что именно копирует `fork`?

![[90 Вложения/CurseHunter/6817/Кадры/113-fork-memory-descriptors-00-55-00.jpg]]

`fork` логически создаёт child с копией virtual address space. На Linux физические pages не копируются немедленно: page tables дублируются, а память работает через copy-on-write (COW) до первой записи. Поэтому фраза «полная копия process» описывает видимую semantics, но не стоимость реализации.

Return value разделяет branches: в child — `0`, в parent — PID child, при ошибке в parent — `-1`, и child не появляется. Порядок дальнейшего выполнения не определён scheduler. Если оба process печатают без синхронизации, порядок строк нельзя обещать.

Таблица descriptors копируется, но corresponding FD parent и child ссылаются на одни open file descriptions. Они разделяют offsets и file status flags. Это объясняет сразу два эффекта:

- parent и child могут организовать IPC через pipe, созданный до `fork`;
- случайно унаследованный descriptor способен удержать pipe открытым или сдвинуть общий file offset.

В multithreaded process child получает только thread, который вызвал `fork`, зато memory содержит снимки состояния mutexes остальных threads. Lock мог остаться «занятым» thread, которого в child уже нет. До `execve` разрешены лишь async-signal-safe functions. Поэтому pair `fork` + большой произвольный callback в child — плохая граница для threaded runtime; ближайшая альтернатива для простого запуска — `posix_spawn`, а platform runtime обычно предоставляет более безопасную high-level abstraction.

### 5. Создаёт ли `exec` новый process?

![[90 Вложения/CurseHunter/6817/Кадры/114-execve-process-image-00-57-00.jpg]]

Нет. `execve` заменяет program image текущего process: code, initialized/uninitialized data, heap и stack. PID остаётся тем же. При успехе `execve` не возвращается; возврат `-1` означает, что замена не состоялась.

Открытые descriptors по умолчанию переживают `execve`; закрываются только помеченные `FD_CLOEXEC`. Поэтому `O_CLOEXEC`, `pipe2(O_CLOEXEC)` и atomic spawn file actions защищают от утечки descriptors в случайно запущенный program. Для intentional redirection нужный FD, наоборот, оставляют открытым под номером `0`, `1` или `2`.

Семейство `exec*` добавляет convenience поверх `execve`:

- `l` принимает variadic list arguments, `v` — null-terminated vector;
- `p` ищет executable по `PATH`, если name не содержит slash;
- `e` позволяет передать environment явно.

На слайде `execl` завершается literal `0`. Для variadic `execl` нужен именно null pointer, то есть `(char *)NULL`: обычный `int 0` после default argument promotions не обязан иметь representation/width pointer. Ещё опаснее проигнорировать ошибку `exec`. Child продолжит выполнять код shell, если call вернулся. Минимальный error path после `fork` заканчивается `_exit(127)`, если command не найден, или `_exit(126)`, если найден, но не может быть выполнен; `exit` может повторно flush-нуть stdio buffers, скопированные от parent. Если parent должен узнать точный `errno`, надёжный вариант — отдельный close-on-exec error pipe: child пишет `errno` при failure, а successful `exec` автоматически закрывает write end.

### 6. Зачем нужны `wait` и `waitpid`?

![[90 Вложения/CurseHunter/6817/Кадры/115-wait-status-macros-01-07-40.jpg]]

Завершившийся child сохраняет минимальную запись в kernel, пока parent не заберёт status. Это zombie: он уже не выполняется, но занимает slot process table. `wait` ждёт любого подходящего child; `waitpid(pid, ...)` позволяет ждать конкретный PID и поэтому лучше выражает invariant shell.

Status декодируют только после соответствующей проверки:

- `WIFEXITED(status)` → разрешено читать `WEXITSTATUS(status)`;
- `WIFSIGNALED(status)` → разрешено читать `WTERMSIG(status)`;
- при options для job control также обрабатываются stopped/continued states.

`waitpid` может вернуть `-1` с `EINTR`, поэтому ожидание повторяют. Для pipeline shell сохраняет PID всех stages, закрывает свои copies pipe ends, затем reaps каждого child. Без `pipefail` status берётся у последней команды pipeline. POSIX.1-2024 Issue 8 уже стандартизует option `pipefail`, но Issue 7 его не требовал; переносимый ответ обязан назвать версионную границу и не предполагать эту option у старого shell.

### 7. Как из этих primitives собрать shell?

Первая версия на слайде делает `readline → parse_args → fork → execvp → wait`:

![[90 Вложения/CurseHunter/6817/Кадры/116-minimal-shell-fork-exec-wait-01-10-45.jpg]]

Для одной external command схема достаточна как mental model, но production path обязан обработать empty input, EOF, syntax error, `fork == -1`, `execvp == -1`, interrupted `waitpid`, освобождение parsed arguments и exit status. `execvp` не понимает shell syntax: quotes, `$`, `>`, `|` разбирает сам shell до запуска program.

Перенаправление `echo foo > myfile` добавляет `open/creat`, затем в child выполняет `dup2(fd, STDOUT_FILENO)` и закрывает исходный FD:

![[90 Вложения/CurseHunter/6817/Кадры/117-shell-output-redirection-01-24-00.jpg]]

Открывать output удобнее в child: тогда parent не должен чистить FD при failed `fork`, и side effect создания/truncation связывается с execution branch. Но shell semantics требуют соблюдать порядок redirections. Например, `cmd >out 2>&1` и `cmd 2>&1 >out` дают разный graph descriptors, потому что `dup2` копирует текущую ссылку в момент операции.

Pipeline `echo hello | less` требует создать pipe до forks и связать `stdout` producer с `pipefd[1]`, а `stdin` consumer — с `pipefd[0]`:

![[90 Вложения/CurseHunter/6817/Кадры/118-shell-pipeline-diagram-01-28-00.jpg]]

Для pipeline из `N` commands корректный порядок такой:

1. создать нужные pipes с close-on-exec;
2. fork всех `N` children, не ожидая первый сразу;
3. в каждом child сделать `dup2` только нужных input/output ends и закрыть **все** исходные pipe FDs;
4. в parent закрывать ends, которые больше не понадобятся, особенно все write ends;
5. после успешного запуска stages вызвать `waitpid` для всех сохранённых PID.

Если parent ждёт producer до запуска consumer, producer может заполнить bounded pipe и навсегда заблокироваться. Если запущены оба, но parent сохранил write end, consumer не увидит EOF. Эти два failure mode проверяют понимание lifecycle лучше, чем просьба перечислить signatures syscalls.

Есть редкий, но показательный edge case. Если standard descriptor был закрыт и `pipe2(O_CLOEXEC)` сразу вернул его номер, например write end уже равен `STDOUT_FILENO`, то `dup2(fd, fd)` ничего не делает и не снимает `FD_CLOEXEC`. После `execve` output исчезнет. Реализация должна отдельно обработать равенство source/target и очистить close-on-exec flag либо построить allocation так, чтобы номера не совпали.

Builtins меняют архитектуру. `cd` нельзя безусловно выполнить в child: его `chdir` не изменит current directory parent-shell. То же относится к `exit`, `export`, `umask` и управлению jobs. Interactive shell сверх показанного минимума нужны process groups, foreground terminal ownership, forwarding/handling `SIGINT`, `SIGTSTP`, `SIGCHLD`, background jobs и quoting/expansion grammar. Иными словами, `fork+exec+wait` запускает одну команду; полноценный shell — parser плюс manager процессов, descriptors и terminal state.

### Перенос модели на Go

Go `os/exec` скрывает platform-specific spawn, но не отменяет lifecycle:

- `exec.Command` принимает уже разделённые arguments и не запускает shell; `|`, `>`, glob и variable expansion сами по себе не работают;
- после `Start` нужен ровно один `Wait`, иначе child resources не будут освобождены;
- при pipeline downstream command запускают до producer, parent закрывает свои copies ends, а outputs читаются concurrently;
- `CommandContext` связывает process с cancellation: по умолчанию context вызывает `Process.Kill`, а `Cmd.Cancel` и `WaitDelay` позволяют настроить завершение и границу ожидания I/O; caller всё равно обрабатывает partial output и итоговую ошибку;
- переданный `*os.File` сохраняет FD semantics: неверно оставленный конец pipe так же удерживает EOF.

На интервью полезно сначала нарисовать descriptor graph и ownership, затем переносить его в `exec.Cmd`. High-level API убирает boilerplate; invariant «кто владеет каждым концом и кто reap-ит child» остаётся.

### Что проверяет хороший ответ по уроку 15

Кандидат не угадывает trace по одной строке C, а отделяет source, libc, loader и process lifecycle. Он различает FD и open file description, замечает возможность descriptor `0`, partial I/O и обязательный `mode` при `O_CREAT`. `dup2` объясняет через atomic replacement, `fork` — через logical copy/COW и shared open file descriptions, `exec` — как replacement того же process, `waitpid` — как reap конкретного child.
