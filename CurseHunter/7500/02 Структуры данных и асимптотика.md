# Структуры данных и асимптотика

## TL;DR

Уроки 8–16 не содержат ещё одной самостоятельной задачи с формальным входом и выходом. Это теоретическая часть, из которой на собеседовании проверяют другое: умеет ли кандидат назвать инвариант структуры, выразить стоимость операции через фактический параметр (`n`, height `h`, load factor `α`), отличить average/amortized от worst case и выбрать ближайшую альтернативу.

Короткая карта выбора:

| Нужная операция | Базовый выбор | Главная оговорка |
| --- | --- | --- |
| random access по индексу | array / slice | вставка в середину сдвигает suffix |
| частые вставки около известного node | linked list | поиск node сам стоит `O(n)` |
| LIFO | stack поверх slice | `append` amortized, не worst-case `O(1)` |
| FIFO | circular buffer или list с head/tail | удаление `slice[0]` может удерживать backing array |
| min/max с динамическими вставками | binary heap | произвольный поиск остаётся `O(n)` |
| ordered set/map | balanced search tree | обычный BST вырождается до `O(n)` |
| lookup по exact key | hash table | expected `O(1)`, worst `O(n)` |

## Область применимости

- Восстановленное содержание: уроки `8–16` CourseHunter `7500`.
- Go-специфичные пояснения: language version `go1.26`, standard library `go1.26.5`, проверено `2026-07-19`.
- Слайды и речь объясняют общие структуры, а не контракт конкретной реализации. Там, где курс приводит C-подобный layout, это модель, а не обещание Go runtime.
- Код и формулы проверены статически; локального Go toolchain нет.

## Ментальная модель: ADT сначала, layout потом

Abstract data type (ADT) задаёт допустимые операции и их семантику. Concrete data structure определяет, как эти операции реализованы и сколько стоят.

Например, `dictionary` — это контракт `put/get/delete`. Его можно реализовать hash table, balanced tree или даже линейным массивом пар. `priority queue` — контракт `push/peek/extract`; binary heap — распространённая, но не единственная реализация.

Урок делит структуры на «смежные» и «связанные». Это полезная интуиция про locality, но не строгая классификация ADT:

- элементы array действительно лежат последовательно;
- binary heap обычно хранится в array, но heap property не требует конкретного физического layout;
- hash table с separate chaining сочетает bucket array и связанные цепочки;
- nodes linked list или tree **могут**, но не обязаны физически лежать далеко друг от друга;
- arena или pool может сделать pointer-based структуру компактной, не меняя её логический инвариант.

На собеседовании лучше идти от workload: `какие операции доминируют → какая гарантия нужна → какой layout и overhead приемлемы`.

### Что именно означает нотация

Если `T(n) = O(g(n))`, то начиная с некоторого `n` функция `T` не превосходит constant multiple от `g`. Это upper bound, а не знак точного равенства. Если верхняя и нижняя границы совпадают с точностью до constant factor, tight bound записывают `Θ(g(n))`. `Ω(g(n))` задаёт lower bound.

Поэтому для `T(n) = 3n + 7` верны обе записи `O(n)` и `O(n²)`, но полезная tight-оценка — `Θ(n)`. В интервью разговорное «алгоритм работает за O(n)» обычно подразумевает tight order роста, однако в доказательстве полезно не смешивать `O` и `Θ`.

Big-O также не является синонимом worst case. Сначала фиксируется функция, которую оцениваем: worst-case `T_worst(n)`, average-case при заданном распределении или best-case. Затем к ней применяется `O`, `Θ` или `Ω`. Например, worst-case линейного поиска — `Θ(n)`, best-case — `Θ(1)`.

### Как считать

1. Определить размер входа. У двух независимых массивов это обычно `n` и `m`, а не один искусственный параметр.
2. Зафиксировать case и computation model: считаются comparisons, memory accesses, network round trips или другое узкое место.
3. Получить recurrence или сумму стоимости blocks.
4. Оставить доминирующий член и убрать constant factors.
5. Отдельно посчитать auxiliary space, call stack и output space.

### Минимальные примеры из урока

Два последовательных независимых цикла:

```go
for range a { /* O(1) */ }
for range b { /* O(1) */ }
```

дают `Θ(n + m)`, а не `Θ(nm)`. Вложенные циклы по `a` и `b` дают `Θ(nm)`.

Binary search требует sorted input и после каждого comparison отбрасывает половину диапазона: `T(n) = T(n/2) + Θ(1) = Θ(log n)`. Основание логарифма в асимптотике несущественно из-за constant factor.

Один рекурсивный вызов с `n-1`:

```go
func sum(n int) int {
	if n == 0 {
		return 0
	}
	return n + sum(n-1)
}
```

даёт `Θ(n)` time и `Θ(n)` stack. Два вызова `f(n-1)` в каждом node создают recurrence `T(n) = 2T(n-1) + Θ(1)`, то есть `Θ(2^n)` time, но максимальная глубина stack остаётся `Θ(n)`: одновременно в памяти находится один root-to-leaf path, а не всё recursion tree.

### Space complexity

Если функция строит result длины `n`, total extra memory может быть `Θ(n)`, даже если алгоритмическое состояние состоит из двух индексов. Полезно явно разделять:

- auxiliary space — рабочая память помимо результата;
- output space — память обязательного возвращаемого значения;
- call stack — скрытая, но реальная память recursion.

### Ошибки, замеченные в изложении

- Фраза «Big-O — это худший случай» смешивает выбор case и тип bound. Правильно: сначала выбрать worst/average/best function, затем оценить её.
- Фраза «у linear search нет Theta, потому что лучший и худший случаи различаются» неверна без контекста. Есть `Θ(n)` для worst case и `Θ(1)` для best case; не существует одной tight function для всех inputs размера `n`, если case не зафиксирован.
- `quicksort` не безусловно `Θ(n log n)`: average/expected time обычно `Θ(n log n)`, а при неудачном pivot worst case — `Θ(n²)`.

## Static array, dynamic array и Go slice

### Array

Contiguous layout позволяет вычислить address элемента как `base + index*elementSize`, поэтому indexing — `O(1)`. Последовательный обход обычно хорошо использует cache lines и prefetch.

Цена:

- размер static array фиксирован;
- вставка или удаление внутри сохраняющего порядок массива требует сдвинуть suffix: `Θ(n)` worst case;
- поиск по значению без дополнительной структуры — `Θ(n)`.

В Go `[N]T` — value type, и `N` входит в тип. Передача или assignment array копирует все элементы, если не используется pointer. `[]T` — descriptor underlying array с pointer, length и capacity; копирование slice value копирует descriptor, а не elements.

### Dynamic array и amortization

Когда `len < cap`, append пишет в свободный slot. Когда capacity исчерпана, нужна новая область и копирование старых элементов: отдельный append стоит `Θ(n)`.

![[90 Вложения/CurseHunter/7500/Кадры/08-dynamic-array-amortized.jpg|720]]

При geometric growth общая стоимость копирований за `n` append ограничена геометрической суммой:

```text
n/2 + n/4 + n/8 + ... < n
```

Поэтому последовательность из `n` append стоит `Θ(n)`, а один append — amortized `Θ(1)`. Это не worst-case guarantee каждого вызова: редкий resize по-прежнему `Θ(n)` и может дать latency spike.

Go specification обещает поведение `append`, но не конкретный growth factor. Нельзя строить portable logic на предположении «capacity всегда удваивается». После allocation новый slice ссылается на другой backing array; aliases старого массива больше не видят новые записи.

### Trade-offs

| Dynamic array | Linked list |
| --- | --- |
| `O(1)` random access | `O(n)` доступ по индексу |
| хорошая locality | pointer chasing и слабее locality |
| middle insert/delete `O(n)` | splice `O(1)`, если node уже известен |
| редкие reallocations | allocation и pointer overhead на node |
| простая compact representation | стабильные node addresses, если implementation это гарантирует |

## Linked list

Singly linked node хранит value и `next`; doubly linked — ещё `prev`. Head позволяет вставить или удалить первый node за `O(1)`. Tail даёт `O(1)` append, но удалить последний node singly linked list за `O(1)` нельзя: нужен predecessor, который обычно ищется за `O(n)`.

Фраза «вставка в list — O(1)» требует указать предусловие. Если дан pointer на нужный node или predecessor, rewiring — `O(1)`. Если дана позиция либо value, сначала выполняется `O(n)` traversal.

Удаление:

- leaf-like случай для list отсутствует: важно сохранить связь predecessor → successor;
- в doubly list удаление известного node `O(1)`, но каждый node хранит два pointers;
- после удаления внешняя ссылка на node может оставаться валидной или нет — это уже contract реализации.

Go package `container/list` реализует doubly linked list. Он удобен для `O(1)` moves/removes по `*list.Element`, но хранит `any` и обычно проигрывает slice по locality и allocation overhead. Для простых последовательностей slice чаще дешевле; list выигрывает, когда элементы уже представлены стабильными nodes и часто переставляются.

## Stack и queue

Stack — LIFO ADT с `push`, `pop`, `top`. На slice верхушкой удобно считать конец: index последнего элемента `len(s)-1`, pop — reslice до `len-1`. Push amortized `O(1)`, pop/top `O(1)` при непустом stack. На linked list top обычно совпадает с head.

Критические edge cases: empty `top/pop`, необходимость занулить удалённый pointer-bearing element перед reslice, если большой backing array живёт долго, и recursion depth, когда stack неявный.

Queue — FIFO ADT с `enqueue` в tail, `dequeue/peek` в head. Linked implementation с head и tail даёт `O(1)` обеих операций; при удалении последнего элемента надо обнулить оба pointers. Для array implementation production-вариант — circular buffer с head/tail indexes.

Наивное `q = q[1:]` в Go само по себе не сдвигает элементы и работает `O(1)`, но маленький tail slice может удерживать весь backing array. Периодическая compaction или ring buffer снимает retention. Удаление через `copy`/`append(q[:0], q[1:]...)` освобождает prefix логически, но сдвигает элементы и стоит `O(n)`.

## Binary tree и BST

Binary tree ограничивает число children двумя. Binary search tree (BST) добавляет order invariant. Для unique keys один распространённый контракт:

```text
все keys(left subtree) < node.key < все keys(right subtree)
```

Если duplicates разрешены, политика (`count`, всегда влево, всегда вправо, список values) должна быть явной и согласованной во всех операциях.

![[90 Вложения/CurseHunter/7500/Кадры/09-bst-invariant.jpg|720]]

Search, insert и delete работают за `Θ(h)`, где `h` — height. Для сбалансированного дерева `h = Θ(log n)`. Для sorted insertion в обычный BST дерево превращается в chain и `h = Θ(n)`. Поэтому название BST само по себе не обещает логарифм; гарантия требует AVL, red-black tree, B-tree, treap или другого механизма balancing.

In-order traversal `left → node → right` выдаёт keys в sorted order за `Θ(n)` time и `Θ(h)` stack при рекурсивной реализации.

Delete имеет три случая:

1. нет children — убрать ссылку parent;
2. один child — связать parent напрямую с child;
3. два children — заменить key/value inorder predecessor (максимум left subtree) или successor (минимум right subtree), затем удалить replacement node, у которого не может быть двух children в выбранном направлении.

Курс упоминает database indexes как применение BST. Практическая оговорка: storage engines обычно используют B/B+ tree из-за большого fan-out и page locality, а не pointer-heavy binary tree. Hash index подходит exact-match workload, но не сохраняет range order. Реализацию ordered map на обычном BST с теми же failure modes можно увидеть в [[CurseHunter/6860/01 Базовые типы, коллекции и строки#Задание 4. Ordered map на BST|задании CourseHunter 6860]].

## Binary heap

Binary heap соединяет два независимых инварианта:

1. **shape:** complete binary tree — все уровни, кроме последнего, заполнены, последний заполняется слева направо;
2. **order:** в max-heap каждый parent `>=` children, в min-heap `<=` children.

Heap не является BST: отношения между siblings и разными subtrees не заданы. Root — экстремум, но поиск произвольного value остаётся `O(n)`.

### Array representation

Для node с index `i` в zero-based array:

```text
left  = 2*i + 1
right = 2*i + 2
parent = (i-1)/2, если i > 0
```

Complete shape устраняет необходимость хранить pointers и пустые внутренние позиции.

### Операции

- `peek root` — `O(1)`;
- `push` — append в конец + sift-up, `O(log n)` worst case;
- `extract root` — swap root с последним, убрать последний, sift-down, `O(log n)`;
- найти arbitrary value без дополнительного index — `O(n)`;
- удалить элемент по **известному index** можно за `O(log n)`: заменить последним и восстановить invariant sift-up или sift-down. Поэтому фраза курса «удалять можно только root» описывает обычный priority-queue API, а не фундаментальное ограничение heap.

### Почему heapify — O(n)

Вставить `n` элементов по одному — `O(n log n)`. Bottom-up Floyd heapify начинает с последнего internal node и делает sift-down. Половина nodes — leaves и не двигается, четверть может пройти не более одного уровня, восьмая — двух:

![[90 Вложения/CurseHunter/7500/Кадры/10-bottom-up-heapify.jpg|720]]

```text
n/4 * 1 + n/8 * 2 + n/16 * 3 + ... = O(n)
```

Это не противоречит `O(log n)` одному sift-down: дорогую максимальную высоту проходит только малая доля nodes.

Heap нужен для priority queue, scheduler, top-k, multiway merge и heapsort. Go `container/heap` задаёт min-heap через `Less`; max-heap получается инверсией ordering. `heap.Init` выполняет `O(n)`, `Push`, `Pop`, `Remove`, `Fix` — `O(log n)` по контракту package.

## Hash table

### Ментальная модель

Hash table не хранит «key прямо по hash». Hash сжимает key space до bucket/probe position, а equality окончательно подтверждает key. Collision неизбежны, когда возможных keys больше slots.

Для обычной таблицы hash function должна быть:

- deterministic в рамках принятого seed/instance;
- достаточно быстрой;
- хорошо распределять ожидаемый набор keys;
- устойчивой к adversarial clustering, если input недоверенный.

Необратимость не является обязательным свойством hash-table hash. Слайд курса смешивает требования обычной и cryptographic hash function. Integrity checks и content deduplication из того же урока требуют collision-resistant cryptographic/content hash по другому threat model; bucket selection такой гарантии не требует.

### Separate chaining

Bucket array хранит chains элементов с одинаковым bucket index. Insert/find/delete сначала вычисляют bucket, затем сравнивают keys внутри chain.

![[90 Вложения/CurseHunter/7500/Кадры/11-hash-chaining-load-factor.jpg|720]]

Load factor `α = size / bucketCount`. При разумном hash и controlled load expected chain length ограничена, поэтому операции expected `O(1)`. При плохом распределении или adversarial collisions одна chain содержит `n` elements, и операция становится `O(n)`.

При grow нельзя просто скопировать buckets по тем же indexes: обычно `hash % newBucketCount` отличается, поэтому entries надо перераспределить. Stop-the-world rehash даёт latency spike `O(n)`; incremental migration переносит buckets постепенно, но усложняет lookup, потому что во время перехода надо учитывать old и new tables. Это design choice, не универсальное свойство всех hash tables.

### Open addressing

Все entries лежат в slot array. Collision запускает probe sequence: linear, quadratic, double hashing или другой алгоритм, который должен гарантировать достижение подходящих slots.

![[90 Вложения/CurseHunter/7500/Кадры/12-hash-open-addressing.jpg|720]]

Open addressing лучше использует locality и не создаёт отдельный node на entry, но чувствительнее к load factor и clustering. `α` обязан оставаться меньше `1`; практический threshold зависит от layout и probe strategy, поэтому показанное в курсе `0.7–0.75` — пример, а не закон.

Удалённый slot нельзя сразу помечать never-used. Lookup завершает probe на первом truly empty slot; если middle entry цепочки сделать empty, key дальше по chain станет недостижим. Нужен tombstone/deleted marker либо backward-shift deletion, корректность которого зависит от probing algorithm.

Хранение короткого hash fragment рядом с metadata позволяет отбрасывать большинство несовпадающих slots до дорогого сравнения полного key. Финальные рисунки курса подводят к grouped open-addressing designs. Современная Go `map` начиная с Go `1.24` использует Swiss Tables, но точный runtime layout не является частью языка; устройство H1/H2, groups и incremental growth подробно разобрано в [[CurseHunter/6860/05 Контексты, итераторы и Swiss Tables#Swiss Tables в Go 1.24|заметке CourseHunter 6860]].

Go specification гарантирует, что `map` — неупорядоченная коллекция unique comparable keys, но не обещает конкретную asymptotic complexity или физическую реализацию. Iteration order не определён.

## Сводка failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| `O` означает точное равенство | слишком грубая или ложная оценка | `O` — upper bound | для tight order использовать `Θ` |
| Big-O автоматически означает worst case | смешаны разные функции стоимости | case не зафиксирован | сначала назвать worst/average/best |
| каждый `append` — `O(1)` worst case | неожиданный latency spike | resize копирует `n` elements | говорить amortized `O(1)`, учитывать reserve |
| slice не alias underlying array | изменение видно в неожиданном месте | descriptors разделяют storage | отслеживать `len/cap`, при изоляции копировать |
| list insert всегда `O(1)` | фактическое время `O(n)` | сначала искали позицию | уточнить, известен ли node/iterator |
| queue через reslice освобождает память | маленькая queue держит большой buffer | backing array остаётся reachable | ring buffer или compaction |
| BST всегда `O(log n)` | sorted input даёт chain | нет balancing | balanced tree либо другая структура |
| heap хранит sorted order | произвольный lookup медленный | invariant связывает только parent/children | использовать root operations или другую структуру |
| heapify — `O(n log n)` | завышенная оценка | большинство nodes близко к leaves | суммировать работу по высотам: `O(n)` |
| hash lookup гарантирован `O(1)` | collision attack или длинная chain | это expected, не worst-case bound | хороший seeded hash, load control, fallback design |
| hash обязан быть необратимым | лишняя cryptographic стоимость | перепутаны use cases | для table нужны distribution и threat-aware collision resistance |
| delete в open addressing делает slot empty | существующий key перестаёт находиться | probe chain преждевременно завершается | tombstone/backward shift |
