# Битовые операции и задачи

## TL;DR

В уроках 1–7 курса `7500` разобраны порядок прохождения алгоритмической секции, базовые битовые операции и пять самостоятельных задач: `Power of Two`, `Single Number`, `Reverse Bits`, `Number of 1 Bits`, `Missing Number`. Бонусный урок 17 повторяет `Squares of a Sorted Array` из курса `7253`: условие, длительность ролика и показанный код совпадают, поэтому здесь зафиксировано повторное появление, а полный разбор оставлен в [[CurseHunter/7253/Бланк вопросов и заданий#5. Squares of a Sorted Array — LeetCode 977|канонической карточке]].

Главная ментальная модель битовых задач: у значения есть две проекции, математическое целое и битовый вектор фиксированной ширины. В Go это особенно важно для 32-битных контрактов: `uint32` фиксирует ширину и даёт логический правый сдвиг, тогда как `int` зависит от архитектуры, а его правый сдвиг для отрицательных значений арифметический.

## Область применимости

- Восстановленное содержание: уроки `1–7` и `17` CourseHunter `7500`.
- Язык решений на видео: Go.
- Версионная область пояснений: language version `go1.26`, проверено `2026-07-19`.
- Локального Go toolchain нет: решения прошли статическую проверку по спецификации, а показанные реализации — проверку по результатам `Accepted` в видео. Локальная компиляция не заявляется.

## Как проходить алгоритмическую секцию

![[90 Вложения/CurseHunter/7500/Кадры/01-interview-flow.jpg|720]]

Первый урок предлагает девять этапов. Полезно воспринимать их не как жёсткий ритуал, а как порядок снятия рисков:

1. **Осознать задачу.** Пересказать условие своими словами и назвать, что должно вернуться. Если модель задачи неверна, оптимизация неправильного решения ничего не даст.
2. **Собрать требования.** Уточнить размер и природу входа, допустимые значения, наличие duplicates, sortedness, возможность менять вход, формат ошибок и требуемую память. Ограничения подсказывают допустимый класс сложности.
3. **Разобрать показательный пример.** Он должен включать существенное ветвление, а не только массив из одного элемента. На нём удобно проверить понимание и найти инвариант.
4. **Перечислить границы.** Пустой вход, один элемент, ноль, отрицательные значения, максимум диапазона, все элементы одинаковы, решение отсутствует или неоднозначно.
5. **Предложить алгоритм и оценить его.** Сначала допустим baseline, затем улучшение. Надо отдельно назвать time, auxiliary space и output space.
6. **Согласовать API.** Имя и сигнатура функции, изменяемость аргументов, формат результата, поведение при ошибке.
7. **Проговорить структуру кода.** Какие переменные хранят инвариант, где обновляются указатели или state, почему цикл завершится.
8. **Написать код.** Стиль, имена и обработка границ оцениваются вместе с алгоритмом.
9. **Протрассировать тесты.** Вслух провести нормальный и граничный пример по значениям переменных. Это одновременно проверка и доказательство для интервьюера.

Короткая формула ответа: `контракт → пример → baseline → инвариант оптимального решения → сложность → код → трассировка`.

## Битовые операции: минимальная модель

Для двух битов `a` и `b`:

| Go | Операция | Когда бит результата равен `1` |
| --- | --- | --- |
| `a & b` | AND | оба бита равны `1` |
| `a \| b` | OR | хотя бы один бит равен `1` |
| `a ^ b` | XOR | биты различаются |
| `a &^ b` | AND NOT / bit clear | бит есть в `a`, но отсутствует в `b` |
| `^a` | complement | соответствующий бит `a` равен `0` |
| `a << k` | left shift | биты `a` сдвинуты влево на `k` |
| `a >> k` | right shift | биты `a` сдвинуты вправо на `k` |

Нужные для задач тождества:

```text
x ^ x = 0
x ^ 0 = x
x ^ y = y ^ x
(x ^ y) ^ z = x ^ (y ^ z)
x & (x - 1) удаляет младший установленный бит x, если x != 0
```

Объяснение последнего свойства: пусть младшая единица `x` стоит в позиции `k`. При вычитании единицы она превращается в `0`, а все младшие нули — в `1`. AND оставляет общий prefix, но обнуляет позицию `k` и всё справа от неё.

Фраза «сдвиг влево умножает на два, вправо делит на два» верна только с оговорками. Для unsigned значения левый сдвиг вычисляется по модулю ширины типа и может отбросить старшие биты. Для signed значения правый сдвиг арифметический, поэтому отрицательное число округляется к минус бесконечности, а не к нулю. В задачах на сырой 32-битный pattern безопаснее говорить о перемещении битов и использовать `uint32`.

## Задача 1. Power of Two — LeetCode 231

### Условие

Для целого `n` определить, является ли оно положительной степенью двойки. Ноль и отрицательные числа ответом не считаются; `1 = 2^0` считается.

Прямой baseline делит положительное число на `2`, пока оно чётное. Он выполняет `O(log n)` итераций. Битовый вариант использует тот факт, что положительная степень двойки содержит ровно одну единицу:

```go
func isPowerOfTwo(n int) bool {
	return n > 0 && n&(n-1) == 0
}
```

![[90 Вложения/CurseHunter/7500/Кадры/02-power-of-two.jpg|720]]

Для `n = 8`:

```text
8     = 1000₂
8 - 1 = 0111₂
AND   = 0000₂
```

Для `n = 10` результат `1010₂ & 1001₂ = 1000₂`, поэтому ответ `false`.

Time и space для машинного `int` — `O(1)`: число битов типа ограничено. Если тот же алгоритм перенести на arbitrary-precision integer, стоимость зависит от длины битового представления и уже не является константной.

### Что проверить на собеседовании

- `n == 0`: без проверки `n > 0` выражение ошибочно вернуло бы `true`.
- `n < 0`: signed representation может содержать единственную единицу только у минимального значения типа, но математической положительной степенью двойки оно не является.
- `n == 1`: должно вернуться `true`.
- Если спрашивают «ровно один установленный бит» для raw word, это другой контракт: там знак и математическая положительность могут быть не нужны.

## Задача 2. Single Number — LeetCode 136

### Условие

Дан непустой массив, в котором каждый элемент встречается дважды, кроме одного. Найти одиночный элемент за линейное время и с `O(1)` дополнительной памятью.

Hash set даёт понятный baseline: при первом появлении вставлять, при втором удалять. Time — expected `O(n)`, space — `O(n)`. Требование по памяти приводит к XOR:

```go
func singleNumber(nums []int) int {
	result := 0
	for _, num := range nums {
		result ^= num
	}
	return result
}
```

![[90 Вложения/CurseHunter/7500/Кадры/03-single-number.jpg|720]]

Инвариант после обработки prefix: `result` равен XOR всех значений, встретившихся в prefix нечётное число раз. Пары взаимно уничтожаются независимо от порядка благодаря associativity и commutativity XOR. После полного прохода остаётся единственный непарный элемент.

Time — `O(n)`, auxiliary space — `O(1)`.

### Границы решения

- Если остальные элементы встречаются трижды, пары больше не уничтожаются: нужна bit-count по разрядам modulo `3` или конечный автомат масок.
- Если одиночных элементов два, XOR даёт их `a ^ b`, но не разделяет. Один установленный бит результата позволяет разбить значения на две группы и восстановить оба.
- Решение работает и с отрицательными `int`, потому что равные bit patterns всё равно взаимно уничтожаются.
- Контракт должен гарантировать непустой input и ровно один непарный элемент; иначе `0` не отличим от корректного ответа `0`.

## Задача 3. Reverse Bits — LeetCode 190

### Ошибка в названии урока

В API урок 5 называется второй задачей `Single Number`, однако видео показывает `Reverse Bits`: пример `43261596 → 964176192`, цикл по `32` битам и результат `Accepted`. Это не вариант `Single Number`, а самостоятельное условие LeetCode 190.

### Условие и инвариант

Развернуть порядок битов 32-битного unsigned значения. После `i` итераций `result` содержит в обратном порядке `i` младших битов исходного значения, а `number` уже сдвинут вправо на `i`.

```go
func reverseBits(number uint32) uint32 {
	var result uint32
	for bit := 0; bit < 32; bit++ {
		result = (result << 1) | (number & 1)
		number >>= 1
	}
	return result
}
```

![[90 Вложения/CurseHunter/7500/Кадры/04-reverse-bits.jpg|720]]

Нельзя завершать цикл, когда `number == 0`: оставшиеся ведущие нули исходного 32-битного слова должны стать младшими нулями результата, а их позиции входят в контракт. Выполняются ровно `32` итерации, то есть `O(32) = O(1)` time и `O(1)` space.

Код курса использует `int`. На показанном положительном примере он даёт правильный результат, но не фиксирует 32-битную ширину и для отрицательного значения применяет arithmetic right shift. Для контракта «32-битная беззнаковая последовательность» нужен `uint32`. В production-коде готовая операция — `bits.Reverse32(number)` из `math/bits`.

### Follow-up

- Если функция вызывается много раз, интервьюер может спросить о lookup table по байтам: четыре разворота байта плюс перестановка позиций.
- `bits.Reverse32` предпочтительнее самописного цикла в прикладном коде; цикл нужен для демонстрации понимания.
- Разворот **битов** не равен `bits.ReverseBytes32`: вторая функция меняет порядок байтов, сохраняя порядок битов внутри каждого байта.

## Задача 4. Number of 1 Bits — LeetCode 191

### Условие

Посчитать число единичных битов 32-битного unsigned значения — population count / Hamming weight.

Проверка всех `32` позиций работает за `O(32)`. Алгоритм Brian Kernighan выполняет одну итерацию на установленный бит:

```go
func hammingWeight(number uint32) int {
	count := 0
	for number != 0 {
		number &= number - 1
		count++
	}
	return count
}
```

![[90 Вложения/CurseHunter/7500/Кадры/05-number-of-1-bits.jpg|720]]

Инвариант: после `count` итераций удалены ровно `count` младших единиц исходного слова. Если установлено `k` битов, time — `O(k)`, где `0 ≤ k ≤ 32`; space — `O(1)`.

Курс снова использует `int`. Для положительных значений при подходящей ширине результат совпадает, но fixed-width contract лучше выразить через `uint32`. Для прикладного Go есть `bits.OnesCount32(number)`.

### Follow-up

- Если надо посчитать Hamming distance двух слов, ответ равен `popcount(a ^ b)`.
- Для массива слов можно суммировать `bits.OnesCount32`; самостоятельный цикл редко выигрывает у оптимизированной стандартной библиотеки.
- Заявление `O(k)` полезнее безусловного `O(1)`, потому что показывает зависимость от числа единиц; для фиксированной ширины обе оценки совместимы.

## Задача 5. Missing Number — LeetCode 268

### Условие

Дан массив длины `n` из различных чисел диапазона `[0, n]`. Ровно одно число отсутствует. Найти его за `O(n)` time и `O(1)` auxiliary space.

Baseline через сумму вычисляет `n(n+1)/2 - sum(nums)`, но для более широких контрактов может переполнить тип. XOR не зависит от порядка и уничтожает каждую присутствующую пару `i ^ nums[j]`:

```go
func missingNumber(nums []int) int {
	result := 0
	for i := 0; i <= len(nums); i++ {
		result ^= i
	}
	for _, num := range nums {
		result ^= num
	}
	return result
}
```

![[90 Вложения/CurseHunter/7500/Кадры/06-missing-number.jpg|720]]

После обоих проходов каждое присутствующее число диапазона встречается в XOR дважды, а пропущенное — один раз. Time — `O(n)`, space — `O(1)`.

Тот же инвариант можно выразить одним проходом:

```go
func missingNumberOnePass(nums []int) int {
	missing := len(nums)
	for i, num := range nums {
		missing ^= i ^ num
	}
	return missing
}
```

### Что сломает решение

- duplicate вместо одного из значений: контракт «все различны» нарушен, XOR больше не доказывает правильность;
- произвольный диапазон `[lo, hi]`: надо изменить множество ожидаемых значений;
- два пропуска: одного XOR недостаточно для однозначного восстановления без дополнительных условий;
- желание вернуть одновременно duplicate и missing: нужна другая задача и отдельный инвариант.

## Задача 6. Squares of a Sorted Array — LeetCode 977

В бонусном уроке дан sorted array, возможно содержащий отрицательные значения. Требуется вернуть квадраты в неубывающем порядке. После возведения в квадрат максимум ещё не обработанного диапазона всегда находится на одном из его краёв; больший модуль записывается в output справа налево. Time — `O(n)`, output space — `O(n)`, auxiliary state — `O(1)`.

Видео полностью совпадает с уроком 6 курса `7253`: длительность `00:05:46`, функция `abs`, два указателя и тот же финальный код. Поэтому полный инвариант, overflow-границы и альтернативы сохранены в [[CurseHunter/7253/Бланк вопросов и заданий#5. Squares of a Sorted Array — LeetCode 977|исходной карточке]], а не продублированы здесь.

![[90 Вложения/CurseHunter/7253/Кадры/006-sorted-squares.jpg|720]]

К двум оговоркам курса стоит отнестись явно:

- `math.Abs` принимает `float64`, поэтому комментарий около вызова целочисленной `abs` не является прямой заменой без conversion;
- `-MinInt` и квадрат большого `int` могут overflow. Ограничения LeetCode удерживают значения в безопасном диапазоне, но production API должен сформулировать собственный числовой контракт.

## Сводка типичных ошибок

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| `n & (n-1) == 0` достаточно без `n > 0` | `0` принимается за степень двойки | у нуля тоже получается нулевой AND | отдельно проверить положительность |
| XOR решает задачу при любых multiplicities | неверный survivor | взаимно уничтожаются только чётные повторения | зафиксировать contract, для `3` использовать bit counts modulo `3` |
| `int` означает 32-битное слово | platform-dependent результат | ширина `int` зависит от архитектуры | использовать `uint32` |
| Reverse Bits можно остановить на `number == 0` | потеря позиций ведущих нулей | контракт требует ровно 32 позиции | выполнить 32 итерации |
| `>>` всегда логический | отрицательное signed значение сохраняет sign bits | Go использует arithmetic shift для signed | сдвигать unsigned type |
| Сумма для Missing Number всегда безопасна | overflow при расширенном диапазоне | промежуточное `n(n+1)` может не поместиться | XOR либо более широкий/проверяемый тип |
| Sorted input остаётся sorted после квадрата | отрицательная часть нарушает порядок | функция `x²` не монотонна на всей числовой оси | сравнивать модули на краях |
