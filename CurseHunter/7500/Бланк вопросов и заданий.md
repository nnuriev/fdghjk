## Пять новых задач

### 1. Power of Two — LeetCode 231

Определить, является ли положительный `n` степенью двойки. Решение курса: `n > 0 && n&(n-1) == 0`. На собеседовании надо объяснить, почему вычитание единицы меняет младшую единицу и все биты справа от неё.

### 2. Single Number — LeetCode 136

Все числа встречаются дважды, кроме одного. XOR accumulator за `O(n)` time и `O(1)` space уничтожает пары. Важный follow-up: другой multiplicity contract требует другого алгоритма.

### 3. Reverse Bits — LeetCode 190

Развернуть ровно `32` бита. В метаданных урок ошибочно назван вторым `Single Number`, но видео, пример `43261596 → 964176192` и код однозначно соответствуют `Reverse Bits`. Для Go-контракта fixed 32-bit нужна сигнатура на `uint32`, а не показанный `int`.

### 4. Number of 1 Bits — LeetCode 191

Brian Kernighan: повторять `n &= n-1`, пока слово не станет нулём. Каждая итерация удаляет одну единицу, поэтому time `O(k)` по числу установленных битов, не более `32` для `uint32`.

### 5. Missing Number — LeetCode 268

В массиве из distinct values диапазона `[0,n]` отсутствует одно число. XOR всех ожидаемых и фактических значений оставляет пропуск за `O(n)` time и `O(1)` space. Preconditions «distinct», полный диапазон и ровно один пропуск входят в доказательство.

## Главные корректировки курса

1. `Reverse Bits` и `Number of 1 Bits` работают с фиксированным 32-битным pattern. Показанный `int` не фиксирует ширину и signed shift semantics; в Go нужен `uint32`.
2. Big-O — asymptotic upper bound, а не другое название worst case. Для worst-case linear search tight bound `Θ(n)`, для best case `Θ(1)`.
3. `append` dynamic array — amortized `O(1)`, но resize одного вызова остаётся `O(n)`. Конкретный growth factor Go specification не обещает.
4. Обычный BST даёт `O(h)`, а не безусловный `O(log n)`. Sorted insertion без balancing вырождает дерево.
5. Heap ограничивает только parent/child order; произвольный поиск не становится логарифмическим. Удаление по известному index возможно за `O(log n)`, хотя обычный priority-queue API извлекает root.
6. Необратимость не нужна обычной hash-table hash function. Это свойство из cryptographic threat model.
7. Hash-table lookup — expected/average `O(1)` при принятых предпосылках, worst case — `O(n)`. Threshold load factor зависит от реализации.
8. Database index обычно строят на B/B+ tree для range/page workload; «binary tree» на слайде слишком общее описание.

## Быстрый прогон перед собеседованием

1. Как constraints меняют допустимую сложность?
2. Какой invariant поддерживает цикл и почему он доказывает результат?
3. Что означает фиксированная ширина bit pattern в Go?
4. Где оценка worst-case, expected или amortized?
5. Как отделить auxiliary memory от output и call stack?
6. Какая операция является сильной стороной выбранной структуры, а какая деградирует?
7. От какого параметра зависит стоимость: `n`, height `h`, число единиц `k` или load factor `α`?
8. Что произойдёт при adversarial или просто неудачном input?
9. Как ближайшая альтернатива меняет time, memory, order и implementation complexity?
10. Какие утверждения являются contract языка, а какие — деталями runtime/library implementation?
