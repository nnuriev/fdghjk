## 1. Safety: data race и логическая гонка

![[90 Вложения/CurseHunter/6609/Кадры/084.jpg|720]]

В первой задаче одна goroutine записывает `text`, вторая читает. Обе вызывают `Done`, а main ждёт `Wait`. Ошибка в рассуждении «раз main ждёт, всё синхронизировано»: happens-before существует от каждого `Done` к возврату из `Wait`, но не между двумя workers. Поэтому detector показывает data race.

На собеседовании важно разделить:

- **data race** — конфликтующие concurrent memory accesses без требуемой synchronization, хотя бы одно из них write;
- **race condition** — более широкая логическая зависимость результата от interleaving; она возможна и в формально data-race-free коде.

Исправление выбирают по требуемому порядку: mutex защищает shared state, channel передаёт value/ownership, а запуск reader после `Wait` явно делает чтение последующим.

## 2. Liveness: deadlock, spin-deadlock и lock order

Две goroutines берут `A→B` и `B→A`. Если каждая успела удержать первый lock, образуется cycle ожидания. Базовое правило — глобально одинаковый порядок захвата. `TryLock` сам по себе проблему не решает: если участник удерживает первый lock и бесконечно крутится на втором, состояние владения не меняется и полезного progress нет.

Курс называет такой пример livelock, но это неточная классификация. При livelock участники активно меняют состояние и уступают друг другу; здесь они удерживают locks, поэтому точнее говорить busy-wait/spin-deadlock.

Отдельная ловушка: наличие ticker goroutine может не дать runtime вывести `all goroutines are asleep`, хотя business operation навсегда заблокирована. Отсутствие fatal message не является доказательством liveness.

## 3. Mutex: identity, granularity и protocol

Mutex защищает не строку кода, а согласованный invariant конкретного shared state. Отсюда следуют три задачи курса:

1. mutex, созданный локально каждой goroutine, не создаёт mutual exclusion между ними;
2. общий mutex вокруг локального state синхронизирует лишнее и создаёт contention без пользы;
3. слишком широкая critical section блокирует peers на I/O, но механически сузить её нельзя, если между copy и действием должен сохраняться invariant.

`sync.Mutex` не reentrant. Повторный `Lock` без `Unlock` зависает. Для `RWMutex` нельзя делать upgrade из read lock в write lock. `Unlock` незаблокированного mutex и несогласованные `RWMutex.Unlock/RUnlock` приводят не к обычной recoverable panic, а к fatal runtime error.

## 4. Копирование `WaitGroup` и других primitives

Функция `done(wg sync.WaitGroup)` получает значение и уменьшает counter копии. Original после `Add(1)` остаётся равен единице, поэтому `Wait` не возвращается. После начала использования не копируют `Mutex`, `RWMutex`, `Once`, `Cond` и `WaitGroup`; объект передают по pointer или держат внутри non-copying owner.

`go vet` с analyzer `copylocks` ловит многие такие случаи, но design rule важнее конкретного предупреждения. Начиная с Go 1.25, `WaitGroup.Go(f)` упрощает пару `Add` + запуск goroutine, однако не отменяет запрет копирования и требования к liveness.

## 5. CAS — это не транзакция и не готовый publication protocol

![[90 Вложения/CurseHunter/6609/Кадры/091.jpg|720]]

В lazy initialization код курса сначала публикует `initialized=true`, затем создаёт `data`. Даже если Store заменить на успешный CAS, другая goroutine может увидеть флаг до готового объекта. Для one-time initialization подходит `sync.Once`, который связывает возврат из initializer с последующими возвратами из `Do`.

Вторая ловушка: `Add(1)` и затем `CompareAndSwap(100, 0)` атомарны по отдельности, но не образуют одну transaction. Между ними другой increment может изменить значение. Нужен CAS loop и заранее определённая semantic: reset каждой сотни, modulo counter, saturation или ticket sequence.

## 6. `sync.Cond`: ждать нужно predicate

Корректный skeleton:

```go
cond.L.Lock()
for !ready {
	cond.Wait()
}
// работа с состоянием под тем же lock
cond.L.Unlock()
```

`Wait` atomically освобождает `cond.L`, приостанавливает goroutine и снова захватывает lock перед возвратом. `Signal` не является сохранённым событием: если waiter ещё не ждёт, сигнал потеряется. Поэтому источник истины — predicate под lock, а не факт вызова `Signal`.

## 7. Concurrent structures: граница защиты должна совпадать с API

Урок объединяет четыре независимые ошибки:

- getter возвращает internal slice после unlock — caller получает доступ к backing array без защиты;
- метод под lock вызывает другой public method, который берёт тот же non-reentrant mutex — self-deadlock;
- embedded `sync.Mutex` продвигает `Unlock` в public method set — внешний код может нарушить invariant;
- value receiver копирует mutex и slice header — изменения выполняются над копией owner, а check и operation оказываются неатомарны.

Практический вариант: named unexported mutex, pointer receivers, snapshot/clone вместо утечки backing storage и одна операция `Pop() (T, bool)` вместо раздельных `Top` и `Pop`.

## 8. False sharing и корректный benchmark

![[90 Вложения/CurseHunter/6609/Кадры/094.jpg|720]]

False sharing связывает независимые frequently-written variables через одну cache line: coherence invalidations заставляют cores мешать друг другу без логически shared state. Возможные меры — sharding и padding/alignment, но cache-line size зависит от platform, а padding расходует memory.

Benchmark из курса не доказывает эффект: название не соответствует создаваемому counter, нет сравнительных variants, total work меняется с `NumCPU`, а shards расположены подряд без явного padding. Для вывода нужны fixed total operations, отдельные implementations, `RunParallel`, проверка результата и при серьёзном анализе hardware counters/profile.
