## Опорная модель ответа

Базовые правила уже собраны в [[60 Go/Массивы и слайсы|заметке о массивах и слайсах]]. Для задач этого курса достаточно держать в голове четыре инварианта:

1. Array `[N]T` — самостоятельное значение из `N` элементов. Длина входит в тип. Присваивание и передача array по значению копируют элементы.
2. Slice `[]T` — значение, описывающее окно в backing array. Языковой контракт задаёт элементы, `len` и `cap`; конкретный runtime Go реализует descriptor через pointer, length и capacity.
3. Копирование slice копирует descriptor, а не backing array. Aliasing сохраняется, пока данные не скопированы явно или `append` не выбрал новый массив.
4. `append` возвращает обновлённый slice. При достаточной capacity он пишет в прежний массив, при недостаточной выделяет новый. Точная стратегия роста capacity не входит в контракт языка.

Из этих правил выводятся почти все ответы ниже. Если приходится запоминать каждый результат отдельно, ментальная модель ещё не сложилась.

## Урок 1. Внутреннее устройство массивов

### Что такое array в Go и что хранится в его значении?

![[90 Вложения/CurseHunter/6754/Кадры/01-массив-непрерывное-хранение.jpg]]

**Утверждение курса:** элементы одного типа лежат последовательно, индекс меняется от `0` до `N-1`, а размер массива фиксирован.

В терминах спецификации тип записывается как `[N]T`: `N` — неотрицательная константа, представимая типом `int`, а `T` — тип элемента. `[3]int` и `[4]int` — разные типы. Само значение array состоит из элементов; отдельного runtime-header с pointer, `len` и `cap` у него нет.

![[90 Вложения/CurseHunter/6754/Кадры/02-массив-длина-часть-типа.jpg]]

Фраза на слайде «длина, как и размер элементов, является частью типа» требует уточнения. В синтаксис типа входят длина и **тип элемента**. Размер элемента выводится реализацией из `T`; отдельным параметром `[N]T` он не записан и рядом с данными не хранится.

### Что произойдёт при присваивании или передаче массива в функцию?

```go
func mutate(a [3]int) {
	a[0] = 99
}

func main() {
	a := [3]int{1, 2, 3}
	b := a
	b[1] = 42
	mutate(a)
	println(a[0], a[1]) // 1 2
}
```

Array подчиняется обычной value semantics: `b := a` и параметр `a [3]int` создают копию всего значения. Изменение копии не затрагивает исходный array. Если нужен доступ к тем же элементам, передают `*[3]int` или slice. Общая граница shallow copy разобрана в [[60 Go/Zero values, семантика значений и копирование|заметке о семантике значений]].

**Интервью-применение:** большая копия array семантически существует, но компилятор вправе устранить физическое копирование, если это не меняет наблюдаемого поведения. Поэтому нельзя доказывать стоимость только синтаксисом; нужны benchmark и анализ конкретной сборки.

## Урок 2. Особенности итерации по массивам

### Задача 1. Изменится ли array при изменении переменной `value` внутри `range`?

```go
values := [...]int{100, 200, 300}

for idx, value := range values {
	value += 50
	fmt.Println(&value, &values[idx])
}

fmt.Println(values)
```

![[90 Вложения/CurseHunter/6754/Кадры/03-range-копия-значения.jpg]]

**Ответ:** нет, результат остаётся `[100 200 300]`. На каждой итерации `value` получает значение элемента присваиванием. `value += 50` меняет iteration variable, а не `values[idx]`. Для мутации нужен индекс:

```go
for i := range values {
	values[i] += 50
}
```

В демонстрации адрес `&value` меняется между итерациями. Это соответствует семантике Go начиная с `1.22`: переменные, объявленные в range clause через `:=`, создаются заново для каждой итерации. В старом коде с языковой версией до Go `1.22` переменная могла переиспользоваться.

### Задача 2. Как отличаются три формы `range` по array?

![[90 Вложения/CurseHunter/6754/Кадры/04-range-массив-указатель-срез.jpg]]

В коде курса показаны:

```go
data := [...]int{1, 2, 3}

for value := range data { /* ... */ }
for value := range &data { /* ... */ }
for value := range data[:] { /* ... */ }
```

Во всех трёх циклах единственная переменная получает **индекс**, а не элемент: `0`, `1`, `2`.

**Коррекция к комментарию курса:** пометка `copy of array` у первой формы неверна именно для этого кода. По спецификации range expression не вычисляется, если переменных не больше одной и `len(x)` — константа. Для array `len(data)` константен, поэтому `for i := range data` не требует семантического копирования array.

Различие появляется в двухпеременной форме:

```go
for i, v := range data { /* array value evaluated once */ }
for i, v := range &data { /* pointer copied, elements read through it */ }
for i, v := range data[:] { /* slice descriptor copied */ }
```

В первом случае range expression — array value; оно вычисляется до цикла, то есть язык задаёт снимок array. Во втором копируется pointer, в третьем — slice descriptor. Физическое устранение копии компилятором допустимо, если программа не замечает разницы.

**Типичная ошибка benchmark:** сравнение большого array value и `*[N]T` часто одновременно меняет layout, allocation, escape и число indirections. Результат не доказывает правило «pointer всегда быстрее»; он доказывает только разницу конкретных функций в конкретной сборке.

## Урок 3. Внутреннее устройство срезов

### Что именно копируется при присваивании slice?

![[90 Вложения/CurseHunter/6754/Кадры/05-дескриптор-среза.jpg]]

Спецификация называет slice descriptor непрерывного сегмента backing array. В runtime Go `1.23.4` это реализовано значением из pointer, length и capacity. Это полезная модель, но не обещание ABI для произвольного `unsafe`-кода.

```go
a := []int{1, 2, 3}
b := a
b[0] = 9
fmt.Println(a) // [9 2 3]
```

`b := a` копирует descriptor. Оба значения указывают в один backing array, поэтому изменение элемента видно через оба slice. `len` ограничивает допустимое индексирование; `cap` показывает, до какой длины окно можно расширить вправо без обязательной смены массива.

### Как работает `append` при нехватке capacity?

![[90 Вложения/CurseHunter/6754/Кадры/06-append-перевыделение.jpg]]

Причинная цепочка:

1. вычисляется требуемая новая длина;
2. если capacity достаточно, slice расширяется поверх прежнего array;
3. иначе runtime выделяет backing array большей ёмкости и копирует старые элементы;
4. новый элемент записывается после прежнего `len`;
5. возвращается новый descriptor, который обязательно нужно сохранить.

Точная новая capacity не гарантирована. В Go `1.23.4` функция `runtime.nextslicecap` удваивает небольшую capacity, после порога `256` плавно уменьшает коэффициент в направлении `1.25`; затем allocator округляет объём под size class и размер элемента.

![[90 Вложения/CurseHunter/6754/Кадры/07-рост-capacity-go123.jpg]]

Это implementation detail конкретной версии. Код должен зависеть только от контракта: при нехватке capacity появится массив, достаточный для новой длины. Даже значение, рассчитанное `nextslicecap`, может измениться после округления allocation size.

### Уменьшает ли slicing capacity и освобождает ли память?

![[90 Вложения/CurseHunter/6754/Кадры/08-capacity-не-уменьшается.jpg]]

Обычное `s = s[:n]` меняет length, но оставляет прежнюю capacity и не отделяет backing array. Полное выражение `s[low:high:max]` может ограничить **видимую** capacity, однако тоже не освобождает массив. Чтобы небольшой результат перестал удерживать крупный backing array, элементы копируют в новый slice.

## Урок 4. Итерация по срезу и aliasing

### Задача 1. Станет ли этот `range` бесконечным?

```go
data := []int{0, 1, 2}
for range data {
	data = append(data, 10)
	fmt.Println("iteration")
}
```

![[90 Вложения/CurseHunter/6754/Кадры/09-range-append-три-итерации.jpg]]

**Ответ:** нет. Будет ровно три строки `iteration`. Range expression вычисляется один раз до цикла. Для slice фиксируются descriptor и исходный `len == 3`; последующее присваивание нового descriptor переменной `data` не меняет число iteration values.

![[90 Вложения/CurseHunter/6754/Кадры/10-range-копия-дескриптора.jpg]]

А этот вариант бесконечен, пока процесс не завершится из-за внешнего ограничения:

```go
for i := 0; i < len(data); i++ {
	data = append(data, 10)
}
```

Условие вычисляет текущий `len(data)` перед каждой итерацией. И `i`, и длина растут на единицу, поэтому `i < len(data)` остаётся истинным.

### Задача 2. Что напечатает код с мутациями и `append` внутри `range`?

![[90 Вложения/CurseHunter/6754/Кадры/11-range-алиасинг-условие.jpg]]

```go
x := []string{"A", "M", "C"}

for i, s := range x {
	println(i, s)
	x[i+1] = "M"
	x = append(x, "Z")
	x[i+1] = "Z"
}
```

**Вывод:**

```text
0 A
1 M
2 C
```

![[90 Вложения/CurseHunter/6754/Кадры/12-range-алиасинг-вывод.jpg]]

До входа в цикл range сохраняет исходный descriptor: pointer на старый массив и length `3`. В начале первой итерации `len(x) == cap(x) == 3`, поэтому первый `append` выделяет новый backing array. Дальнейшие записи идут в новый массив через текущую переменную `x`, а iteration values `s` читаются из старого массива `[A M C]`.

После трёх итераций текущий `x` равен:

```text
[A Z Z Z Z Z]
```

![[90 Вложения/CurseHunter/6754/Кадры/13-range-алиасинг-финальное-состояние.jpg]]

Граница рассуждения важна. Если бы исходной capacity хватало для первого `append`, range и `x` некоторое время смотрели бы в один array, и записи могли бы изменить ещё не прочитанные iteration values. Ответ зависит от `len`, `cap` и момента первой reallocation, а не от одной фразы «range копирует slice».

## Урок 5. Копирование срезов

### Задача. Почему `copy` ничего не скопировал?

```go
src := []int{1, 2, 3}
var dst []int
n := copy(dst, src)

fmt.Println(n, dst) // 0 []
```

![[90 Вложения/CurseHunter/6754/Кадры/14-copy-в-пустой-dst.jpg]]

`copy` переносит `min(len(dst), len(src))` элементов. У nil slice `dst` длина равна нулю, поэтому результат — `0`, а allocation автоматически не происходит.

```go
dst := make([]int, len(src))
n := copy(dst, src) // n == 3
```

![[90 Вложения/CurseHunter/6754/Кадры/15-copy-минимум-длин.jpg]]

Начиная с Go `1.21` для того же намерения есть `slices.Clone(src)`. Оба решения создают независимый backing array, однако элементы копируются присваиванием. Это shallow clone: для `[][]byte`, `[]*T` или struct со slice внутренние ссылки останутся общими.

```go
outer := make([][]byte, len(src2D))
for i := range src2D {
	outer[i] = slices.Clone(src2D[i])
}
```

**Типичная ошибка:** называть любой `copy` «глубокой копией». Независимость верхнего slice ещё не означает рекурсивного копирования графа элементов.

## Урок 6. Создание среза без инициализации

### Задача. Можно ли получить mutable `[]byte` без zeroing через `strings.Builder` и `unsafe`?

В курсе предложен такой обход:

```go
func makeDirty(size int) []byte {
	var sb strings.Builder
	sb.Grow(size)
	p := unsafe.StringData(sb.String())
	return unsafe.Slice(p, size)
}
```

![[90 Вложения/CurseHunter/6754/Кадры/16-unsafe-грязный-срез.jpg]]

**Коррекция:** это не поддерживаемое решение и его нельзя рекомендовать в production или на интервью как корректный Go-код.

1. После `Grow(size)` длина builder всё ещё равна нулю. `sb.String()` возвращает пустую строку.
2. Контракт `unsafe.StringData` для пустой строки не задаёт конкретный адрес: результат может быть `nil`.
3. `unsafe.Slice(nil, size)` при ненулевом `size` вызывает panic.
4. Если реализация всё же вернула pointer на внутренний buffer, байты принадлежат строковому представлению. Документация `unsafe` запрещает менять bytes, пока существует соответствующая string value; Go strings immutable.
5. Код зависит от внутренности Go `1.23.4`: `strings.Builder.grow` использует `internal/bytealg.MakeNoZero`. Это не публичный контракт и может измениться.

Benchmark курса тоже сравнивает разные результаты:

```go
make([]byte, 0, 10<<20) // len == 0
makeDirty(10 << 20)     // заявленный len == 10 MiB
```

У первой функции нет 10 MiB доступных элементов, только capacity. Сравнение времени не отвечает на вопрос, сколько стоит получить slice одинаковой длины и семантики.

Безопасного общего API для mutable uninitialized `[]byte` в Go нет: `make([]byte, n)` даёт zero values. Если zeroing измеримо мешает, используют reuse буфера или pool с явным ownership и полностью перезаписывают нужный диапазон до первого чтения. Иначе «грязные» bytes превращаются ещё и в риск утечки старых данных.

## Урок 7. Срез как аргумент функции

### Задача 1. Какие изменения увидит вызывающая функция?

```go
func process1(data []int) {
	data[2] = 5
}

func process2(data []int) {
	data = append(data, 6)
}

func main() {
	data := []int{1, 2, 3, 4} // len=4 cap=4
	process1(data)
	process2(data)
	fmt.Println(data)
}
```

![[90 Вложения/CurseHunter/6754/Кадры/17-срез-в-функции.jpg]]

**Ответ:** `[1 2 5 4]`. Параметр получает копию slice descriptor, но его pointer ведёт в тот же backing array. Поэтому `process1` меняет общий элемент. В `process2` capacity исчерпана: `append` выделяет другой array и меняет только локальную копию descriptor. Вызывающая функция не получает ни новую длину, ни новый pointer.

Правильный контракт обычно возвращает slice:

```go
func process(data []int) []int {
	return append(data, 6)
}

data = process(data)
```

Pointer `*[]T` нужен редко: он позволяет менять descriptor вызывающего непосредственно, но усложняет nil-состояния и ownership. Возврат `[]T` обычно яснее.

### Задача 2. Что изменится, если свободная capacity есть?

```go
func process(data []int) {
	data = append(data, 5)
}

func main() {
	data := make([]int, 3, 6)
	process(data)
	fmt.Println(data)     // [0 0 0]
	fmt.Println(data[:4]) // [0 0 0 5]
}
```

![[90 Вложения/CurseHunter/6754/Кадры/18-append-скрытый-элемент.jpg]]

`append` использует общий array и пишет `5` в четвёртый слот. Но length хранится в descriptor, а локальное изменение `len` вызывающая функция не видит. Reslicing до четырёх элементов раскрывает записанное значение.

**Практическое правило:** документируйте отдельно право менять существующие элементы, право делать `append` и право удерживать slice после возврата. Формулировка «slice передаётся по ссылке» скрывает три разных эффекта и потому вредна.

## Урок 8. Nil и пустые срезы

### Задача 1. Чем отличаются nil slice и empty non-nil slice?

```go
var a []string
b := []string(nil)
c := []string{}
d := make([]string, 0)
```

Для всех четырёх `len == 0`, `cap == 0`, `range` делает ноль итераций, а `append` допустим. Но `a == nil` и `b == nil` истинны; `c == nil` и `d == nil` ложны. Сами slices между собой сравнивать нельзя, кроме сравнения с `nil`; общая модель comparability разобрана в [[60 Go/Равенство и comparability типов|заметке о равенстве типов]].

![[90 Вложения/CurseHunter/6754/Кадры/19-nil-и-пустой-срез.jpg]]

Показанные `24` bytes — результат конкретного `darwin/arm64` запуска Go `1.23.4`, а не размер, гарантированный языком. Совпавшие адреса пустого slice и `struct{}{}` тоже не задают identity: zero-size objects и empty values могут использовать общий runtime address.

### Задача 2. Почему обработчик не замечает отсутствие операций?

```go
func handleOperations(id string) {
	operations := getOperations(id)
	if operations == nil {
		// handling...
	}
}

func getOperations(id string) []float32 {
	operations := []float32{}
	if id == "" {
		return operations
	}
	return operations
}
```

![[90 Вложения/CurseHunter/6754/Кадры/20-ошибка-проверки-на-nil.jpg]]

`getOperations` возвращает non-nil empty slice, поэтому проверка `operations == nil` ложна. Если бизнес-смысл — «операций нет», проверяют длину:

```go
if len(operations) == 0 {
	// handling...
}
```

В конце видео автор начинает заменять условие, но на кадре получается `len(operations) == nil`. Такой код не компилируется: `len` возвращает `int`, который нельзя сравнить с `nil`. Корректное сравнение — с нулём.

Альтернатива — намеренно возвращать `nil` как отдельный protocol state. Тогда контракт должен объяснять, чем `nil` отличается от пустого результата. Внутри программы чаще всего достаточно `len(s) == 0`.

**Интервью-применение:** `encoding/json` кодирует nil slice как `null`, а non-nil empty slice как `[]`. Если внешний API различает эти значения, выбор становится частью wire contract; подробнее это связано с [[60 Go/Пакет encoding-json|контрактом encoding/json]].

## Урок 9. Slice expressions, capacity и overlap

### Как вычислить `len` и `cap` после slicing?

Для простого выражения:

```go
t := s[low:high]
```

получаем:

```text
len(t) = high - low
cap(t) = cap(s) - low
```

У slice верхняя граница `high` может доходить до `cap(s)`, а не только до `len(s)`. Поэтому `make([]int, 2, 6)[1:6]` допустимо и создаёт slice длины `5`.

Полное выражение:

```go
t := s[low:high:max]
```

задаёт `cap(t) = max - low` при инварианте `0 <= low <= high <= max <= cap(s)`.

![[90 Вложения/CurseHunter/6754/Кадры/21-full-slice-expression.jpg]]

Full slice expression полезно на API-границе:

```go
visible := s[:n:n]
```

Следующий `append(visible, x)` вынужден перейти на новый array и не перезапишет скрытый хвост исходного slice. Но существующие `visible[:n]` элементы всё ещё общие, а сам большой backing array остаётся достижимым.

### Когда `append` одного slice меняет другой?

![[90 Вложения/CurseHunter/6754/Кадры/22-append-меняет-общий-массив.jpg]]

Если два slice указывают в один array и у изменяемого slice хватает capacity, `append` может записать в область, которую второй slice уже видит или сможет увидеть после reslicing. После reallocation связь для новых записей разрывается, однако старые элементы, скопированные в новый array, становятся независимыми только с этого момента.

### Почему capacity стала `8`, а не `12`?

```go
data1 := make([]int, 4, 6)
data2 := data1[2:6] // len=4 cap=4
data2 = append(data2, 1)
```

У `data2` capacity равна `4`, потому что окно начинается с индекса `2` backing array длины `6`. Для пятого элемента нужна reallocation. В Go `1.23.4` малая capacity удваивается с `4` до `8`, что и показано в курсе.

![[90 Вложения/CurseHunter/6754/Кадры/23-capacity-восемь-go123.jpg]]

Число `8` — наблюдение о Go `1.23.4` и этом element size. Контракт языка обещает лишь `cap >= 5`; код и тесты не должны требовать ровно `8`.

### Задача. Что напечатает код с перекрывающимися срезами?

```go
a := [...]int{0, 1, 2, 3}
x := a[:1]
y := a[2:]

x = append(x, y...)
x = append(x, y...)

fmt.Println("a:", a)
fmt.Println("x:", x)
```

**Ответ:**

```text
a: [0 2 3 3]
x: [0 2 3 3 3]
```

![[90 Вложения/CurseHunter/6754/Кадры/24-перекрывающиеся-срезы-вывод.jpg]]

Разбор по шагам:

1. Изначально `x` имеет `len=1`, `cap=4`; `y` смотрит на `a[2:]`, то есть `[2 3]`.
2. Первый `append` помещается в capacity `x` и копирует `[2 3]` в `a[1:3]`. Array становится `[0 2 3 3]`.
3. Поскольку `y` всё ещё ссылается на `a[2:]`, его текущее содержимое теперь `[3 3]`.
4. У `x` после первого append длина `3`, capacity `4`. Второй append требует длину `5`, поэтому создаётся новый array и в него добавляется текущее `[3 3]`.
5. Старый `a` больше не меняется, новый `x` равен `[0 2 3 3 3]`.

`copy` и `append(s, other...)` корректно обрабатывают overlap, но это не делает aliasing понятным. На границах владения лучше клонировать данные или явно запретить изменение входа.

## Урок 10. Retention и «утечки» памяти

В примерах этого урока память не потеряна для программы: объекты остаются достижимыми через маленький slice или pointer. Точнее называть это **непреднамеренным retention**, а не классической утечкой недостижимой памяти. Для диагностики полезна общая схема из [[70 Практические кейсы/Диагностика memory leaks|runbook по memory leaks]].

### Задача 1. Почему slice длины 20 bytes удерживает 1 GiB?

```go
func FindData() []byte {
	data := make([]byte, 1<<30)
	for i := 0; i < len(data)-1; i++ {
		if data[i] == 0 && data[i+1] == 0 {
			return data[i : i+20]
		}
	}
	return nil
}
```

![[90 Вложения/CurseHunter/6754/Кадры/25-малый-срез-держит-гигабайт.jpg]]

Возвращённый descriptor указывает внутрь allocation размером 1 GiB. GC не может освободить часть heap object: пока достижим subslice, достижим весь backing array. В записанном запуске `runtime.GC()` оставляет около `1024 MB` live allocation.

Корректное отделение результата:

```go
func FindData(data []byte) []byte {
	for i := 0; i+20 <= len(data); i++ {
		if data[i] == 0 && data[i+1] == 0 {
			return append([]byte(nil), data[i:i+20]...)
		}
	}
	return nil
}
```

Вместо `append` подойдёт `slices.Clone(data[i:i+20])`. Новые 20 bytes стоят `O(20)` времени и памяти, зато большой вход можно собрать после исчезновения остальных ссылок.

### Коррекция «правильного» примера курса

![[90 Вложения/CurseHunter/6754/Кадры/26-ошибка-в-исправлении-копии.jpg]]

В файле `big_slice_correct` есть две независимые ошибки:

```go
if data[i] == 0xFF && data[i+1] == 0x00 {
	partData := make([]byte, 20)
	copy(partData, data)
	return partData
}
```

- `data` создан через `make` и целиком заполнен нулями, поэтому условие с `0xFF` не выполняется. Функция возвращает `nil`, и вывод `0 MB` не доказывает, что копия освободила большой array.
- Если совпадение всё же встретится при `i > 0`, `copy(partData, data)` копирует первые 20 bytes, а не найденное окно. Нужно `copy(partData, data[i:i+20])`.

### Задача 2. Почему pointer на один элемент удерживает большой array?

```go
func FindElement(numbers []int, target int) *int {
	for i := range numbers {
		if numbers[i] == target {
			return &numbers[i]
		}
	}
	return nil
}
```

![[90 Вложения/CurseHunter/6754/Кадры/27-указатель-держит-весь-массив.jpg]]

Interior pointer `&numbers[i]` указывает внутрь backing allocation, поэтому GC сохраняет весь объект. В курсе slice из `1<<30` значений `int` занимает около `8192 MB` на 64-bit платформе и остаётся live после GC.

Выбор исправления зависит от контракта:

- если нужен только результат поиска, вернуть `(value int, ok bool)`;
- если нужен независимый pointer, скопировать значение в новую переменную и вернуть `&value`;
- если вызывающий должен менять элемент исходного array, retention — часть контракта и его нельзя «исправить» без смены API.

### Задача 3. Почему `data[:2]` удерживает вложенные буферы оставшихся 998 элементов?

```go
type Data struct {
	values []byte
}

func GetData() []Data {
	data := make([]Data, 1000)
	for i := range data {
		data[i].values = make([]byte, 1<<20)
	}
	return data[:2]
}
```

Backing array из 1000 structs остаётся достижим. GC сканирует pointer-поля всего heap object, поэтому slice descriptors в хвосте продолжают удерживать 998 вложенных буферов по 1 MiB.

```go
clear(data[2:])
return data[:2]
```

![[90 Вложения/CurseHunter/6754/Кадры/28-clear-хвоста-среза.jpg]]

`clear` записывает zero value в хвостовые structs. В показанном запуске live allocation после GC падает примерно с `1000 MB` до `2 MB`. Точные числа зависят от runtime и accounting, но механизм устойчив: исчезают pointers на вложенные buffers.

Полное выражение `data[:2:2]` само по себе не помогает: оно меняет capacity descriptor, но не очищает pointer slots backing array. Если исходный slice нельзя мутировать, безопаснее клонировать первые два элемента в новый outer slice; при необходимости полной независимости нужно отдельно клонировать и `values`.

`runtime.KeepAlive(x)` в демонстрациях лишь гарантирует, что `x` считается live до точки вызова. Оно не освобождает память и не лечит retention.

## Урок 11. Граничные операции: compile error или panic

Задача урока — быстро классифицировать выражение до запуска. Главный критерий: компилятор может отвергнуть нарушение, доказанное из constant expressions и типов; нарушение динамической границы приводит к runtime panic.

![[90 Вложения/CurseHunter/6754/Кадры/29-операции-с-массивами.jpg]]

### Массивы

| Выражение | Результат | Причина |
| --- | --- | --- |
| `[3]int{1,2,3}[4]` | compile error | constant index вне длины array |
| `i := 4; a[i]` для `[3]int` | panic | значение индекса известно только при выполнении |
| `[10]int` и `[11]int` | разные типы | длина входит в array type |
| `len([10]int{})`, `cap([10]int{})` | `10`, `10` | для array capacity равна длине |
| `[3]int{1,2,3} == [3]int{1,2,3}` | допустимо, `true` | element type `int` comparable |
| `[1][]int{} == [1][]int{}` | compile error | array несравним, если element type несравним |
| `a < b` для arrays | compile error | arrays поддерживают `==` и `!=`, но не ordering operators |
| `var a [0]int` | допустимо | нулевая длина разрешена; размер значения равен нулю |
| `var a [-1]int` | compile error | длина должна быть неотрицательной константой |
| `n := 10; var a [n]int` | compile error | переменная `n` не constant expression |
| `make([10]int, 10)` | compile error | `make` принимает slice, map или channel, не array |
| `append([10]int{}, 1)` | compile error | первый аргумент `append` должен быть slice |

### Срезы

| Выражение | Результат | Причина |
| --- | --- | --- |
| `make([]int, 3, 6)[4]` | panic | индексирование ограничено `len == 3`, а не capacity |
| `var s []int; s[0]` | panic | у nil slice длина нулевая |
| `var s []int; s = append(s, 1)` | допустимо | `append` умеет начать с nil slice |
| `for range ([]int)(nil) {}` | 0 итераций | length nil slice равна нулю |
| `make([]int, 0)` | non-nil empty slice | `len == 0`, `cap == 0` |
| `make([]int, -5)` | compile error | отрицательная constant length |
| `n := -5; make([]int, n)` | panic | отрицательная динамическая length |
| `make([]int, 10, 5)` | compile error | constant capacity меньше length |
| `n := 5; make([]int, n*2, n)` | panic | динамическая capacity меньше length |

![[90 Вложения/CurseHunter/6754/Кадры/30-make-и-границы-среза.jpg]]

### Slice expressions

| Выражение при `s := make([]int, 2, 6)` | Результат | Причина |
| --- | --- | --- |
| `s[1:6]` | допустимо, `len == 5` | для slice simple upper bound может доходить до `cap(s)` |
| `s[1:7]` | panic | upper bound превышает capacity |
| `s[2:1]` | compile error | constant bounds нарушают `low <= high` |
| `low, high := 2, 1; s[low:high]` | panic | нарушение обнаруживается при выполнении |
| `var n []int; n[:]` | допустимо, nil slice | границы `0:0` допустимы |
| `var n []int; n[0:0]` | допустимо, nil slice | тот же нулевой диапазон |
| `var n []int; n[0:1]` | panic | high превышает нулевую capacity |
| `make([]int, 0, 10)[:10:100]` | panic | `max` превышает исходную capacity |

![[90 Вложения/CurseHunter/6754/Кадры/31-nil-и-full-slice-границы.jpg]]

### Как отвечать на такую задачу вслух

1. Сначала назвать статический тип выражения: array, pointer-to-array или slice.
2. Для array выписать длину из типа; для slice отдельно выписать текущие `len` и `cap`.
3. Проверить, constant ли индекс или границы. Доказуемое constant-нарушение обычно становится compile error.
4. Для индексирования сравнивать с `len`; для верхней границы simple slicing slice — с `cap`; для full slicing проверить `low <= high <= max <= cap`.
5. Только после этого определять aliasing и возможность reallocation.
