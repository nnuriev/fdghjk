## Сквозное system-design задание

Спроектируйте оформление заказа, где API пишет заказ в PostgreSQL, публикует `OrderCreated` в Kafka, downstream резервирует товар, а чтение карточки ускоряет Redis.

В хорошем ответе должны появиться:

- end-to-end deadline и отдельные connection/request timeouts;
- retry только для классифицированных transient errors, exponential backoff с jitter и ограниченный budget;
- `Idempotency-Key`, привязанный к fingerprint запроса и сохранённому результату;
- bounded admission control до дорогой работы, bulkheads для разных зависимостей;
- outbox в транзакции с заказом и idempotent/inbox consumer;
- partition key по `order_id`, явная стратегия commit offset и DLQ/replay;
- cache invalidation/versioning, stampede protection, TTL jitter;
- SLI/SLO, burn-rate alerts, trace correlation и runbook;
- безопасный rollout, rollback boundary и план деградации.

Ответ считается неполным, если в нём только перечислены паттерны. Для каждого паттерна надо назвать защищаемый ресурс, инвариант, failure mode, метрику и цену.
