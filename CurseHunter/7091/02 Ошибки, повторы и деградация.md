# Ошибки, повторы и деградация

## TL;DR

Retry полезен только тогда, когда следующий шанс действительно отличается от предыдущего и клиент способен безопасно повторить эффект. Его контракт включает retryable error, maximum attempts, общий deadline, backoff+jitter, budget, idempotency и observability. Hedging уменьшает tail latency ценой дополнительной нагрузки. Circuit breaker прекращает заведомо бесполезные вызовы, а graceful degradation сохраняет ограниченную продуктовую ценность. Эти механизмы дополняют друг друга, но не взаимозаменяемы.

## 1. Retry начинается с классификации

Retry оправдан, если ошибка transient и повтор может попасть в изменившееся состояние: leader election, краткий network loss, connection reset, throttling с `Retry-After`. Permanent errors — invalid argument, authentication failure, schema mismatch, deterministic conflict — повтором не исправляются.

Одного HTTP/gRPC code недостаточно. Например, timeout оставляет результат неизвестным: read обычно можно повторить, write — только при идемпотентном protocol. `RESOURCE_EXHAUSTED` может означать временный rate limit или устойчивый capacity shortage; клиент должен уважать server pushback.

Retry policy должна отвечать на семь вопросов:

1. какие operations безопасно повторять;
2. какие errors retryable;
3. сколько attempts разрешено;
4. какой total deadline;
5. какой backoff и jitter;
6. кто единственный owner retry в call chain;
7. какой budget ограничивает дополнительную нагрузку.

## 2. Backoff, jitter и amplification

Exponential backoff для попытки `n` можно представить как `min(cap, base * 2^n)`. Он даёт dependency время восстановиться. Jitter разводит синхронных клиентов по времени; без него все clients, получившие ошибку одновременно, снова придут одной волной.

Практические варианты:

- full jitter: случайное значение от `0` до текущего exponential cap;
- equal jitter: половина cap плюс random до второй половины;
- decorrelated jitter: следующая пауза зависит от предыдущей и случайности.

Выбор менее важен, чем отсутствие синхронизации, cap и total deadline.

![[90 Вложения/CurseHunter/7091/Кадры/03-retry-amplification.jpg|720]]

Если каждый из `L` последовательных layers делает до `A` attempts, худший fan-out к нижнему layer равен `A^L`. Два слоя по три attempts дают девять вызовов, пять слоёв — 243. Поэтому retry обычно располагают в одном подходящем layer: там, где известны semantics операции и осталось достаточно deadline.

**Retry budget** ограничивает долю дополнительных attempts относительно исходного трафика, например не более 10%. Это защищает dependency независимо от конкретного `maxAttempts`. Breaker может быстро запретить заведомо неуспешные calls, но budget всё равно нужен при частичных ошибках.

### Ошибка формулировки

`maxRetries=3` нередко означает один initial attempt плюс три retries, то есть четыре фактических вызова. В API лучше использовать `maxAttempts` или явно документировать смысл.

## 3. Идемпотентность при потерянном ответе

![[90 Вложения/CurseHunter/7091/Кадры/04-idempotency-lost-response.jpg|720]]

Идемпотентность — повтор одной логической операции даёт тот же внешний эффект, а не просто тот же HTTP status. Для `POST /orders` клиент генерирует `Idempotency-Key`, сервер атомарно связывает его с:

- tenant/user scope;
- canonical request fingerprint;
- состоянием `in_progress/completed/failed`;
- business resource ID;
- сохранённым response/status;
- retention/expiry policy.

Алгоритм:

1. в транзакции попытаться создать запись с unique `(scope, key)`;
2. если ключ новый — стать owner выполнения;
3. если ключ есть и fingerprint другой — вернуть conflict, а не старый ответ;
4. если `completed` — вернуть сохранённый результат;
5. если `in_progress` — ждать bounded time, вернуть `202`, либо повторить чтение состояния;
6. business effect и idempotency result фиксировать атомарно или через protocol, исключающий окно между ними.

`SELECT`, затем `INSERT` без unique constraint оставляет race: два workers увидят отсутствие записи и оба создадут заказ. Distributed lock тоже не заменяет durable idempotency record: process может упасть после side effect до записи результата.

### Idempotency и optimistic concurrency — разные инварианты

- `Idempotency-Key` отвечает: «это повтор той же команды?»
- `If-Match: version` отвечает: «команда построена на всё ещё актуальном состоянии?»

Бронирование может требовать оба механизма. Повтор той же команды не должен создать вторую бронь, а новая команда со старой version не должна перезаписать чужое изменение.

## 4. Hedging против tail latency

Обычный retry ждёт failure/timeout. Hedging отправляет дополнительную копию ещё незавершённого запроса после задержки, например после `p95`; первый успешный response побеждает, остальные calls отменяются.

![[90 Вложения/CurseHunter/7091/Кадры/05-hedging-pathway.jpg|720]]

Польза: случайный slow backend или network tail перестаёт определять latency клиента. Цена: дополнительные requests, CPU, queueing и риск сделать перегрузку каскадной.

Безопасная схема:

- операция read-only либо имеет полноценную idempotency semantics;
- копия идёт в другой независимый backend/fault domain;
- есть hedge delay, max copies и общий deadline;
- loser отменяется, а сервер умеет прекращать работу по cancellation;
- hedge rate входит в отдельный budget и виден в telemetry;
- механизм выключается при overload или breaker open.

Отправлять копии сразу во все replicas — fan-out, а не аккуратный hedging. Это снижает latency ценой постоянного умножения нагрузки.

## 5. Timeout и deadline propagation

**Connection timeout** ограничивает establishment transport connection. **Request timeout** ограничивает ожидание одной операции. **Deadline** — абсолютный конец end-to-end budget. Deadline удобнее относительного timeout: downstream видит, сколько времени реально осталось после уже потраченной работы.

Если клиент дал 500 ms, gateway потратил 80 ms, service A — 120 ms, service B не должен получить новые 500 ms. Он получает оставшийся budget за вычетом network/cleanup reserve. Иначе call chain продолжит дорого работать после того, как ответ уже никому не нужен.

Практические правила:

- propagation через `context`/RPC metadata;
- child deadline не позже parent deadline;
- перед дорогой фазой проверять remaining time;
- cancellation передавать в DB/query/HTTP client;
- различать «клиент отменил» и «наш dependency просрочил» в metrics;
- timeout не отменяет side effect автоматически.

Если каждый layer независимо выбирает `5s`, суммарная latency может стать десятками секунд. Если timeout слишком мал и близок к обычному p99, он создаёт retry storm из здорового, но медленного трафика.

## 6. Circuit breaker

![[90 Вложения/CurseHunter/7091/Кадры/06-circuit-breaker-states.jpg|720]]

- **Closed**: requests проходят, outcome попадает в rolling window.
- **Open**: calls fail fast; dependency получает время на восстановление.
- **Half-open**: проходит небольшое число probes; успех закрывает breaker, ошибка снова открывает.

Threshold по пяти последовательным ошибкам прост, но нестабилен при редком трафике. Для production полезны minimum sample size, failure/slow-call rate, rolling window, open timeout и ограничение probes.

Generations нужны, чтобы late result старого периода не испортил counters нового. Например, медленный request из старого closed generation завершился после open→half-open→closed и не должен увеличить failure count уже восстановившейся generation.

Breaker должен быть scoped по реальному failure domain: endpoint/operation/tenant/host. Один global breaker может отключить здоровые endpoints из-за одной тяжёлой операции. Слишком мелкий scope, наоборот, не остановит общий overload.

Circuit breaker:

- не исправляет ошибку;
- не гарантирует recovery;
- не заменяет timeout, retry budget, rate limiting или capacity planning;
- обязан иметь metrics состояний, rejected calls, transition reason и manual override/runbook.

## 7. Graceful degradation и fallback

Fallback сохраняет ограниченную пользовательскую ценность: stale cache вместо рекомендаций, read-only mode, упрощённый ranking, отключение необязательного enrichment. Он опасен, если использует ту же сломанную dependency или оказывается дороже основной ветки.

Нужно заранее определить:

- какой инвариант нельзя нарушать даже в degraded mode;
- какую freshness/полноту увидит клиент;
- как пометить degraded response;
- сколько времени можно жить в таком режиме;
- как избежать скрытого постоянного fallback;
- какая метрика покажет вход, длительность и выход из деградации.

Fallback на пустой список без сигнала может превратить outage в тихую потерю выручки. Технически HTTP 200 не означает хороший SLI.

## Failure modes в формате собеседования

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| любой 5xx надо retry | dependency падает сильнее | постоянные ошибки и лишняя работа | error classification + budget |
| backoff без jitter достаточно | периодические пики | clients синхронизированы | randomized delay |
| idempotency key можно хранить в памяти | дубликаты после restart | нет durable ownership/result | unique durable record |
| timeout означает rollback | повторный side effect | сервер уже сделал commit | idempotency/status lookup |
| hedging всегда ускоряет | tail latency растёт | нагрузка создаёт queueing | delayed bounded hedges |
| breaker на весь client | здоровые операции недоступны | слишком широкий scope | breaker по failure domain |
| fallback всегда безопасен | скрытая деградация или overload | ветка не дешевле/не независима | explicit degraded contract |

## Практическое задание

Спроектируйте `POST /payments` со сторонним provider, который иногда отвечает за 4 секунды, а иногда списывает деньги и теряет response.

Сильный ответ:

1. принимает client idempotency key и связывает с request fingerprint;
2. атомарно резервирует local operation ID;
3. передаёт provider его idempotency key, если protocol поддерживает;
4. использует end-to-end deadline и bounded attempts только для retryable outcomes;
5. при неизвестном результате переходит в `pending/reconciliation`, а не немедленно повторяет charge;
6. отделяет breaker provider от local admission control;
7. даёт клиенту status endpoint и возвращает сохранённый результат;
8. измеряет unknown outcomes, duplicate prevention, retry ratio и provider latency.
