# Очереди и асинхронная коммуникация

## TL;DR

Kafka масштабирует ordered log через partitions. Порядок гарантируется внутри partition, а consumer group распределяет каждую partition одному active member группы. Offset — позиция чтения, не доказательство бизнес-эффекта. Commit до обработки рискует потерей, commit после — дубликатом; поэтому основной практический контракт — at-least-once плюс idempotent consumer/inbox. Transactional outbox устраняет окно между commit бизнес-данных и публикацией события, но не делает весь мир exactly-once.

## 1. Partition — единица порядка и параллелизма

Topic разбит на append-only partitions. Record получает offset только внутри своей partition. Consumer group назначает partition одному active consumer; один consumer может читать несколько partitions.

![[90 Вложения/CurseHunter/7091/Кадры/11-kafka-ordering.jpg|720]]

Следствия:

- общий порядок topic с несколькими partitions отсутствует;
- одинаковый business key, например `order_id`, должен стабильно попадать в одну partition, если операции по заказу требуют порядка;
- maximum parallelism одной группы ограничен числом partitions;
- лишние consumers простаивают;
- горячий key создаёт hot partition, которую нельзя распараллелить внутри группы без изменения business ordering.

При увеличении числа partitions стандартный hash mapping может изменить destination для key. Старые и новые records одного key окажутся в разных partitions, поэтому migration требует отдельной стратегии, если порядок через границу изменения критичен.

## 2. Producer durability

Replication factor определяет число копий partition. Leader принимает writes, followers реплицируют log; ISR — replicas, достаточно синхронизированные с leader.

При `acks=all` broker подтверждает write по правилам ISR, а `min.insync.replicas` задаёт минимальное число ISR, необходимое для успешной записи. Типичная комбинация RF=3, min ISR=2, `acks=all` выдерживает потерю одного участника без тихого подтверждения записи только одной копией. Цена — при недостатке ISR writes отклоняются: durability повышается за счёт write availability.

`enable.idempotence=true` использует producer ID и sequence numbers, чтобы broker не добавил повторно record, повторно отправленный этим producer из-за неопределённого ACK. Это не end-to-end idempotency:

- новый producer/session и новая бизнес-команда могут создать логический duplicate;
- downstream side effect в PostgreSQL/HTTP не становится exactly-once;
- одинаковое payload без стабильного event ID не распознаётся как тот же бизнес-event.

В Apache Kafka 4.3 idempotence включена по умолчанию, если конфигурация ей не противоречит; version-dependent defaults нельзя переносить на произвольные clients и старые clusters без проверки.

## 3. Consumer groups и offsets

Committed offset — следующая позиция, которую группа считает обработанной. Он хранится в compacted internal topic `__consumer_offsets`. Log compaction сохраняет актуальное значение по key, но не означает, что business transaction завершилась.

Два базовых порядка:

### Commit до business effect

1. прочитать record;
2. commit offset;
3. выполнить effect;
4. crash между 2 и 3.

После restart consumer начнёт дальше: record потерян для этой группы. Это at-most-once processing.

### Commit после business effect

1. прочитать record;
2. выполнить effect;
3. crash до commit;
4. record приходит повторно.

Effect может выполниться дважды. Это at-least-once delivery; безопасность требует idempotency.

Auto-commit по таймеру не знает, завершился ли business code. Он может случайно фиксировать ещё не обработанные records. Manual commit даёт контроль, но не устраняет crash window между side effect и offset commit.

## 4. Delivery semantics без магии

![[90 Вложения/CurseHunter/7091/Кадры/12-kafka-delivery-semantics.jpg|720]]

На слайде exactly-once сформулировано слишком широко. Корректная область зависит от protocol:

- at-most-once: duplicate исключён ценой возможной потери;
- at-least-once: потеря предотвращается повтором, duplicate допустим;
- Kafka transaction может атомарно публиковать records в Kafka topics и commit consumer offsets в той же transaction;
- Kafka Streams EOS обеспечивает exactly-once processing для поддерживаемого Kafka read-process-write topology;
- произвольный write во внешнюю DB, email или HTTP call в эту atomic boundary не входит.

Поэтому «exactly-once delivery по сети» редко является полезным системным контрактом. Практическая цель — **effectively-once business effect**: at-least-once transport + stable event ID + atomic idempotent consumer.

## 5. Transactional outbox

Dual-write problem:

1. сервис commit заказ в DB;
2. process падает до Kafka publish;
3. заказ существует, событие потеряно.

Если поменять порядок, Kafka event может появиться, а DB transaction откатиться. Две независимые системы нельзя сделать атомарными простым порядком вызовов.

![[90 Вложения/CurseHunter/7091/Кадры/13-transactional-outbox.jpg|720]]

Outbox решает это так:

1. business row и outbox row с unique event ID пишутся в одной DB transaction;
2. relay/CDC читает committed outbox;
3. публикует событие в Kafka с retry;
4. помечает/удаляет запись либо полагается на CDC log.

Если relay падает после publish до отметки, он опубликует event повторно. Поэтому outbox гарантирует отсутствие окна «business state есть, event навсегда потерян», но delivery остаётся at-least-once. Consumer должен дедуплицировать event ID.

Aggregate ID удобно использовать как Kafka key: события одного aggregate сохраняют partition order. Outbox table требует retention/cleanup, monitoring backlog/age и безопасного replay.

## 6. Inbox и idempotent consumer

Inbox table содержит `(consumer_name, message_id)` с unique constraint и результат/статус. В **одной транзакции** consumer:

1. пытается claim message ID;
2. если запись уже есть — считает повтор обработанным;
3. применяет business effect;
4. фиксирует inbox и effect вместе;
5. только потом позволяет commit Kafka offset.

Если inbox insert и business write в разных transactions, crash между ними снова оставляет окно duplicate/loss. Если effect находится во внешней системе, нужен её idempotency protocol, saga/reconciliation или другой durable state machine.

Inbox растёт, поэтому retention выбирают дольше максимального replay horizon. Удалять dedup records раньше, чем Kafka/DLQ может повторно доставить event, опасно.

## 7. Retry, retry topics и DLQ

Errors надо разделить:

- transient: timeout, temporary unavailability, throttling;
- permanent for message: validation/schema/business rejection;
- poison: deterministic crash/bug на конкретном payload;
- systemic: DB down, credentials expired, consumer code broken.

Долгий retry внутри handler блокирует partition: последующие records не обрабатываются, lag растёт. Короткий bounded retry полезен для действительно кратких faults. Для минут/часов message публикуют в retry topic с attempt metadata и delayed schedule.

Цена retry topic:

- исходная partition может продолжить работу;
- relative ordering с отложенным message нарушается;
- требуется idempotency и correlation с original topic/partition/offset;
- ошибочная массовая retry может умножить traffic;
- нужен конечный max attempts/age.

Если строгий per-key order важнее availability, partition может быть остановлена до успешной обработки или key направляется в отдельную ordered state machine. Это явный product trade-off.

**DLQ — не мусорная корзина.** Нужны:

- reason/error class и original metadata;
- payload redaction/encryption и ACL;
- retention;
- alert на arrival rate/age;
- owner и runbook;
- инструмент inspect/fix/replay;
- защита от повторной петли DLQ → основной topic → DLQ.

## 8. Lag и эксплуатация

Consumer lag — разница между log end offset и consumer position/committed offset. Само число без rate и возраста недостаточно.

Наблюдать стоит:

- lag и oldest record age по каждой partition;
- ingress/processing rate и estimated catch-up time;
- rebalance count/duration;
- processing latency/error/retry/DLQ rate;
- commit latency/failures;
- hot partitions и skew;
- outbox backlog/oldest unpublished event;
- inbox dedup hit rate.

Autoscaling consumers не поможет, если partitions уже полностью распределены или одна hot partition определяет lag.

## Failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| Kafka даёт global order | события сущности переставлены | разные partitions | stable key и contract порядка |
| auto-commit означает success | тихая потеря | offset впереди effect | manual boundary + idempotency |
| manual commit исключает дубликаты | повторный charge | crash после effect | inbox/idempotent endpoint |
| idempotent producer даёт EOS в DB | duplicate side effects | узкая область producer dedup | end-to-end event ID protocol |
| outbox публикует ровно раз | duplicate event | crash relay после publish | idempotent consumer |
| DLQ решила проблему | сообщения забыты | нет owner/replay | operational workflow |
| больше consumers всегда быстрее | часть простаивает | partitions меньше consumers | partitions/cost/hot-key анализ |

## Практическое задание

Заказ должен последовательно пройти `Created → Paid → Shipped`. События публикуются в Kafka, payment consumer пишет PostgreSQL, shipping временно недоступен.

Сильный ответ:

- key = `order_id`, чтобы один заказ сохранял partition order;
- RF/min ISR/acks выбираются по durability requirement;
- producer имеет stable event ID и outbox;
- payment consumer применяет inbox+business effect в одной DB transaction;
- offset commit после transaction;
- short transient retry bounded, долгий retry — через retry topic;
- если shipping order строгий, нельзя пропустить `Paid` и обработать следующий state без state-machine check;
- poison messages уходят в DLQ с owner/replay;
- SLI: event age, processing success до deadline, outbox lag, DLQ rate.
