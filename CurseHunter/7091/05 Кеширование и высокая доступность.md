# Кеширование и высокая доступность

## TL;DR

Cache ускоряет чтение ценой нового состояния, которое может устареть, исчезнуть или стать точкой перегрузки. Корректный дизайн начинает не с Redis command, а с контракта: источник истины, допустимая staleness, invalidation path, поведение при miss/failure, stampede protection и memory policy. `WATCH`, lock, singleflight, TTL и Cluster решают разные задачи. Ни один из них сам по себе не даёт атомарность между DB и cache.

## 1. Stale data — часть контракта

В cache-aside read path приложение сначала читает cache, на miss — source of truth, затем кладёт value. Write path меняет DB и удаляет/обновляет cache. Между независимыми operations есть race.

Пример:

1. reader получил старое значение `v1` из DB после cache miss, но ещё не записал cache;
2. writer commit `v2` в DB и удалил cache;
3. reader после удаления записал `v1` обратно;
4. cache хранит stale `v1` до TTL или следующей invalidation.

Поэтому «сначала DB, потом delete cache» уменьшает окно, но не является строгой guarantee. Подход выбирают по staleness budget:

- TTL only: просто, но stale до TTL и burst на expiry;
- invalidate after commit: быстро, но dual-write/race;
- event-driven invalidation через outbox/CDC: durable propagation, но eventual lag и duplicates;
- versioned value: client/cache отвергает version старше известной;
- read-through/write-through: логика централизована, но atomicity с внешней DB всё равно требует protocol.

Если нужна strict read-your-writes, можно bypass cache после write, передавать minimum version или читать authoritative store. Нельзя обещать blanket strong consistency от DB+Redis без distributed transaction/consensus protocol и явной цены.

## 2. `WATCH/MULTI/EXEC` и concurrency

![[90 Вложения/CurseHunter/7091/Кадры/14-redis-watch-multi-exec.jpg|720]]

`WATCH key` реализует optimistic check-and-set: если watched key изменился до `EXEC`, transaction aborts, а client повторяет read-modify-write. `MULTI` ставит commands в очередь, `EXEC` выполняет их последовательно без interleaving других client commands в середину этого batch.

Это не SQL transaction:

- rollback уже выполненных commands при runtime error нет;
- arbitrary reads/logic выполняются client-side до `EXEC`;
- consistency с PostgreSQL не появляется;
- optimistic conflict требует retry;
- в Cluster multi-key transaction требует keys в одном hash slot.

Если операция сводится к атомарному Redis command (`INCR`, conditional `SET NX`) или Lua/function на одном shard, это проще и обычно надёжнее client-side read-modify-write.

## 3. Distributed lock: lease, token, fencing

Lock полезен для mutual exclusion, но не доказывает, что старый owner прекратил работу. Process может зависнуть дольше lease, lock истечёт, новый owner начнёт работу, старый проснётся — оба воздействуют на resource.

Минимальные свойства:

- unique random owner token;
- bounded lease;
- release только compare-and-delete своего token, не просто `DEL`;
- renewal с ограничениями и cancellation;
- fencing token, который authoritative resource сравнивает монотонно;
- поведение при потере lock явно определено.

Если resource не проверяет fencing token, lease lock даёт best-effort mutual exclusion, но не защищает от delayed old owner. Для финансового инварианта лучше DB constraint/transaction или consensus-backed coordinator, который интегрирован с resource.

## 4. Thundering herd и singleflight

![[90 Вложения/CurseHunter/7091/Кадры/15-thundering-herd.jpg|720]]

Когда hot key истекает, тысячи concurrent misses одновременно идут в DB. Источник замедляется, requests удерживают ресурсы, timeout и retry усиливают волну.

Защиты дополняют друг друга:

- request coalescing/singleflight;
- stale-while-revalidate через soft TTL;
- TTL jitter;
- prewarm hot keys;
- bounded refresh concurrency;
- per-key lock;
- admission control/fallback;
- negative cache для повторяющегося известного отсутствия.

Go `singleflight.Group` объединяет concurrent calls с одним key **внутри процесса**: один выполняет function, остальные получают результат. После завершения следующий miss снова запускает call. Несколько pods имеют независимые groups, поэтому нагрузка уменьшается примерно до числа pods, а не до одного global request. Он не является cache и не гарантирует durable result.

Нужно решить, разделять ли errors: если первый call завершился transient error, все waiters получат ту же ошибку. Иногда нужен короткий error TTL/backoff, иногда лучше не кешировать error вообще.

## 5. Soft TTL, hard TTL и negative caching

![[90 Вложения/CurseHunter/7091/Кадры/16-soft-hard-ttl.jpg|720]]

- до soft TTL value fresh;
- между soft и hard TTL можно вернуть stale value и запустить один async refresh;
- после hard TTL value больше не выдаётся, request ждёт refresh/источник или получает controlled error.

Это уменьшает tail latency и herd, но stale interval становится продуктовым контрактом. Для баланса счёта hard TTL может быть нулевым; для рекомендаций минуты могут быть допустимы.

Negative cache хранит «объект точно отсутствует» на короткое время. Он полезен против repeated misses и probing random IDs. Риски:

- объект создаётся, а negative entry продолжает скрывать его;
- authorization failure ошибочно кешируется как `not found` между users;
- transient DB error превращается в ложное отсутствие;
- attacker заполняет cache уникальными misses.

Поэтому key включает security scope, TTL короткий, кешируются только классифицированные authoritative not-found results, а capacity ограничена.

## 6. Cache avalanche и TTL jitter

Cache avalanche — много keys истекают одновременно, например после массовой загрузки с одинаковым TTL, deploy с новым namespace или cold start. Random jitter распределяет expiration:

```text
effectiveTTL = baseTTL + random(-delta, +delta)
```

Jitter не лечит один hot key — там нужен request coalescing/soft TTL. Он разводит независимые expirations во времени. Deterministic jitter от hash(key) полезен для воспроизводимости и стабильного распределения.

## 7. L1/L2 cache

L1 in-process даёт минимальную latency и снимает network RPS с Redis; L2 shared Redis повышает hit rate между pods. Read path обычно L1 → L2 → source.

Цена L1:

- разные pods держат разные версии;
- invalidation fan-out сложнее;
- memory умножается на число replicas;
- deploy вызывает cold cache;
- local hot set может быть skewed.

Контракт включает local short TTL/version, event invalidation с защитой от пропусков, max size/eviction и fallback на L2. «Запихнуть как можно больше в L1» без memory budget может перенести outage из Redis в application OOM.

## 8. Redis Cluster, hash slots и Sentinel

![[90 Вложения/CurseHunter/7091/Кадры/17-redis-sharding.jpg|720]]

Redis Cluster использует 16384 hash slots. Key отображается в slot через CRC16; slots распределены по primary nodes, replicas обеспечивают failover. Client должен обрабатывать topology и redirects (`MOVED`/`ASK`).

Multi-key operation обычно требует keys в одном slot. Hash tag в `{...}` заставляет связанные keys использовать общую hash-часть, например `order:{42}:state` и `order:{42}:items`. Цена — риск hot slot; нельзя поместить все keys в один tag ради удобства.

При network partition Cluster предпочитает consistency подтверждённых slots в рамках своей модели и может стать частично/полностью недоступным; asynchronous replication всё равно оставляет окно потери недавно подтверждённых writes при failover. Redis cache должен иметь fallback и не считаться единственным durable source без отдельного анализа.

**Sentinel** мониторит non-sharded primary/replicas, выполняет failover и сообщает clients новый primary. Он даёт HA без horizontal sharding. **Cluster** даёт sharding плюс failover, но сложнее для clients, multi-key operations и resharding. Sentinel не превращает один dataset в несколько shards.

## 9. Eviction policies

![[90 Вложения/CurseHunter/7091/Кадры/18-redis-eviction.jpg|720]]

При достижении `maxmemory` policy определяет, какие keys удалять:

- `noeviction`: writes, требующие memory, получают error; подходит, когда silent eviction недопустим, но application обязано пережить OOM response;
- `allkeys-lru`: удалять давно не использованные keys из всего keyspace;
- `allkeys-lfu`: защищать часто используемые hot keys;
- `allkeys-random`: дешёвый вариант при близкой ценности keys;
- `volatile-*`: выбирать только keys с expiration; опасно, если таких keys мало — non-TTL data съест memory;
- `volatile-ttl`: сначала keys с ближайшим expiration.

Redis использует приближённые LRU/LFU, а не идеальную глобальную структуру. Выбор подтверждают hit ratio и miss penalty, а не универсальной таблицей.

В current Redis docs на дату проверки присутствуют также `allkeys-lrm`/`volatile-lrm` для least recently modified; это version-dependent дополнение Redis 8.6, которого нет на слайдах курса. Для старого deployment надо проверять его фактическую версию.

## 10. Observability cache

Минимальный набор:

- hit/miss ratio по workload, а не только global;
- latency/error/timeout по cache operations;
- source load caused by misses;
- memory used, fragmentation, evictions, expirations;
- hot keys/slots, command distribution, network throughput;
- connection pool wait/in-use;
- replication lag/failover/cluster state;
- refresh duration/error, singleflight shared count;
- stale served age и hard-expiry failures;
- invalidation lag/drop;
- negative-cache hit ratio.

Высокий hit ratio может быть бесполезен, если редкие misses очень дорогие и совпадают по времени. Полезна метрика **origin protection**: сколько concurrent source loads допущено на key/workload.

## Failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| delete-after-write строго согласован | старое value вернулось | concurrent stale fill | version/outbox/contract staleness |
| Redis transaction = SQL ACID | DB и cache расходятся | нет общей atomic boundary | source of truth + durable invalidation |
| lock исключает старого owner | двойной side effect | lease expired | fencing token/resource validation |
| singleflight глобальный | DB получает по call на pod | local process scope | L2/per-key distributed protection |
| один TTL для всех keys | avalanche | синхронное expiry | jitter/prewarm |
| negative cache любой error | ложный not-found | transient/tenant mix | classify + scoped key + short TTL |
| volatile policy всегда evicts | writes получают OOM | нет TTL candidates | memory/key policy audit |

## Практическое задание

Карточка объявления читается 50k RPS, меняется редко, но статус «продано» должен появляться не позже 5 секунд. Redis иногда недоступен, один hot ad получает 10k concurrent requests.

Сильный дизайн:

- PostgreSQL — source of truth;
- cache-aside с versioned value и hard staleness ≤5 s для status;
- outbox/CDC invalidation после commit, TTL как safety net;
- L1 short TTL + L2 Redis, key содержит schema/version;
- local singleflight и, при необходимости, bounded distributed refresh;
- soft TTL для некритичных полей, но strict/bypass path для status;
- jitter для массовых keys, prewarm hot ad;
- cache failure не делает liveness false; controlled DB fallback ограничен admission control;
- Redis Cluster нужен только при data/throughput beyond single primary; иначе Sentinel может быть проще;
- SLI: freshness violation, origin concurrency, hit ratio, stale age, invalidation lag.
