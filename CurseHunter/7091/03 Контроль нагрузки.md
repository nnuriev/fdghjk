# Контроль нагрузки

## TL;DR

Overload — не просто «много RPS». Он возникает, когда arrival rate и стоимость запросов дольше допустимого превышают service rate ограничивающего ресурса. Bulkhead локализует исчерпание ресурса, rate limiter формирует вход по квоте, backpressure сообщает producer снизить темп, load shedding отбрасывает работу, которую система уже не способна выполнить вовремя. Bounded queue допустима как короткий buffer; unbounded queue превращает overload в latency, memory pressure и timeout amplification.

## 1. Сначала найдите ограничивающий ресурс

Capacity описывается не одним числом RPS. Запросы различаются по CPU, memory, DB connections, fan-out, payload и lock contention. Поэтому полезны weighted cost, concurrency и saturation signals.

По Little's Law для устойчивой системы `L = λW`: среднее число одновременно находящихся в системе requests равно arrival rate, умноженному на среднее время. Если сервис обслуживает 1000 RPS со средней latency 100 ms, ожидаемый in-flight около 100. Рост latency до 1 s при прежнем RPS поднимает in-flight до 1000 — даже без роста входного трафика.

Это объясняет cascading overload: медленная dependency удерживает goroutines, connections и memory; очереди растут; timeout провоцирует retry; дополнительная работа ещё больше увеличивает latency.

## 2. Bulkhead

Bulkhead разделяет shared resource на failure domains. Если одна тяжёлая операция занимает весь общий connection pool, простые requests тоже перестают работать. Разделение может быть:

- concurrency semaphores по endpoint/tenant/work class;
- отдельные HTTP/DB connection pools;
- worker pools и queues;
- process/pod/node pools;
- отдельные clusters или accounts для наиболее критичных workloads.

Инвариант: исчерпание quota одного compartment не забирает весь ресурс у другого. Цена — fragmentation: один pool простаивает, пока другой отклоняет запросы. Поэтому quotas требуют telemetry и иногда controlled borrowing, которое не уничтожает reserved capacity.

Плохой bulkhead по числу goroutines не поможет, если все compartments используют один маленький DB pool. Делить надо реальную точку saturation.

## 3. Rate limiting algorithms

![[90 Вложения/CurseHunter/7091/Кадры/07-token-bucket.jpg|720]]

### Token bucket

Tokens добавляются со скоростью `r` до capacity `b`; request тратит token. Средний rate ограничен `r`, а накопленные `b` tokens допускают burst. Это хороший выбор, когда короткий burst приемлем.

### Leaky bucket

Requests помещаются в bounded queue и выходят с фиксированным темпом. Поток сглаживается сильнее, но появляется queueing latency. При переполнении requests должны отбрасываться; бесконечная очередь нарушает смысл алгоритма.

### Fixed window

Counter сбрасывается на границе окна. Дёшево, но клиент может отправить почти полный лимит в конце одного окна и ещё раз в начале следующего, создав двойной boundary burst.

### Sliding log/window

Sliding log хранит timestamps и точен, но дорог по памяти/операциям. Sliding-window counter аппроксимирует предыдущий и текущий windows, дешевле, но не абсолютно точен.

| Алгоритм | Burst | Стоимость | Типичный выбор |
| --- | --- | --- | --- |
| token bucket | разрешён до capacity | O(1) state | API quotas, bursty clients |
| leaky bucket | сглаживается | queue + latency | контролируемая подача в dependency |
| fixed window | boundary burst | очень низкая | грубые quotas |
| sliding window | точнее | выше | fairness/contract с меньшей погрешностью |

## 4. Distributed limiter: точность против доступности

Local limiter быстрый и переживает недоступность coordinator, но общий лимит приблизителен: при `N` instances каждая может пропустить свою долю или весь quota. Central Redis/Lua counter даёт более согласованный global view, но добавляет network hop и новый failure domain.

Вопросы, которые надо задать:

- quota global, per-region, per-tenant или per-instance?
- допустимы overshoot и temporary underutilization?
- что делать при недоступности limiter: fail-open или fail-closed?
- как распределить quota при autoscaling и skewed traffic?
- насколько важна fairness относительно availability?

Для security/financial cap чаще нужен fail-closed и authoritative storage. Для защиты best-effort feed от собственного overload fail-open с local emergency limiter может быть лучше.

## 5. Backpressure

![[90 Вложения/CurseHunter/7091/Кадры/08-backpressure.jpg|720]]

Backpressure — downstream сообщает upstream, какую скорость или объём он способен принять. Простое наличие queue ещё не backpressure: если producer продолжает бесконечно писать, давление не дошло до источника.

Возможные сигналы:

- blocking send/limited channel;
- credits/window;
- pause/resume consumer;
- reduced fetch/batch size;
- `429`/`RESOURCE_EXHAUSTED` с retry delay;
- broker lag, на основе которого producer снижает rate;
- adaptive concurrency.

TCP receive window защищает buffer конкретного connection, но не знает capacity application, DB или tenant fairness. HTTP/2/gRPC flow control ограничивает bytes, а не число дорогих requests. Поэтому application-level admission control всё равно нужен.

Unbounded queue создаёт ложную availability: requests приняты, но их deadline истечёт до выполнения. Полезная очередь ограничена:

- max length/bytes;
- max queue wait или deadline;
- per-priority/tenant quota;
- shedding policy;
- metrics age, depth, enqueue/dequeue/reject.

## 6. Load shedding

![[90 Вложения/CurseHunter/7091/Кадры/09-rate-limit-vs-load-shedding.jpg|720]]

Rate limiter обычно про договорённый rate/fairness/abuse prevention. Load shedding реагирует на фактическое состояние capacity: in-flight, queue delay, CPU, memory, dependency saturation, latency. Первый может отклонить здоровый excess quota; второй — сохранить критичную функцию при overload.

Admission control размещают до parsing большого payload, allocation, authentication fan-out и DB acquisition, насколько позволяет security model. Иначе отклонённый request уже потратил защищаемый ресурс.

Приоритет shedding:

1. уже просроченные/cancelled requests;
2. speculative/prefetch/background work;
3. low-priority и дорогие optional enrichments;
4. requests сверх tenant quota;
5. случайная доля, если все requests эквивалентны.

`429 Too Many Requests`/`RESOURCE_EXHAUSTED` сообщает client quota/admission limit и часто допускает later retry. `503 Service Unavailable` уместен при общей временной неспособности. В обоих случаях retry должен иметь backoff, jitter и budget; мгновенный retry отменяет пользу shedding.

## 7. Adaptive concurrency и материал QA

В QA на экране разбиралась библиотека `go-concurrency-limits`: limiter контролирует in-flight, blocking wrapper держит bounded backlog, проверяет deadline, извлекает/evict request из queue после получения capacity и сообщает `OnDropped` при timeout/rejection.

![[90 Вложения/CurseHunter/7091/Кадры/10-concurrency-limit-queue-code.jpg|720]]

Ключевой инвариант не в названии библиотеки: request либо получает permit и затем обязательно освобождает его с outcome, либо покидает bounded queue и не выполняет expensive work. Permit leak постепенно обнулит capacity; double release исказит лимит.

Adaptive limiter использует latency/queueing/gradient и меняет concurrency. Он реагирует быстрее ручного RPS limit на изменение cost, но может oscillate. Нужны min/max bounds, smoothing, warm-up и независимый emergency cap.

Остальная часть QA визуально не содержит технических слайдов или читаемого кода; устные вопросы без аудиорасшифровки не приписываются курсу.

## Failure modes

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| RPS один для всех requests | дорогой endpoint валит сервис | разная стоимость | weighted/per-class limit |
| очередь сгладит любой пик | latency/OOM | queue не bounded | size+time bounds, shed early |
| global Redis limiter всегда лучше | limiter стал SPOF | coordination hop | local/global hybrid и fail policy |
| TCP backpressure достаточно | DB pool исчерпан | transport не видит app capacity | application admission control |
| CPU — единственный overload signal | latency растёт при низком CPU | lock/IO/pool saturation | golden signals + resource metrics |
| reject и retry безопасны | retry storm | client не получил pushback/budget | explicit retry policy |

## Практическое задание

API имеет два endpoints: дешёвый `GET /status` и тяжёлый `POST /report`, который занимает DB connection на 2 секунды. На пике reports занимают весь pool из 100 connections.

Сильный дизайн:

- отдельный pool или semaphore для reports и reserved capacity для status;
- per-tenant token bucket для quota;
- bounded queue reports по deadline, без queue для уже просроченных requests;
- concurrency limit на основании DB saturation и latency;
- early load shedding до генерации report;
- `429` для tenant quota, `503` для общей деградации;
- retry только при безопасной idempotency и с server pushback;
- metrics: in-flight, queue age/depth, reject reason, DB acquire latency, goodput.

Goodput важнее throughput: число успешно завершённых до deadline полезных requests. Система, которая «обработала» тысячу уже отменённых reports, только усугубила outage.
