# Практические реализации и учебная БД

## TL;DR

В практической части два уровня задач:

1. сквозной проект — in-memory key-value БД с parser, TCP protocol, WAL, recovery и asynchronous primary/replica replication;
2. четыре интервью-задачи — bounded worker pool, keyed scheduler, size/time batcher и Redis read-through cache с фоновой синхронизацией.

Исходники полезны именно для code review: демонстрационные решения выглядят законченными, но их lifecycle contract неполон, а в `Close` и background coordination есть гонки.

## Сквозная in-memory БД

Официальная программа разбивает проект на последовательные задания:

1. архитектура основных слоёв и примитивный query parser;
2. продолжение parser и storage;
3. TCP server, CLI client и application protocol;
4. развитие network layer;
5. WAL и recovery после restart;
6. продолжение WAL;
7. интеграция WAL с context/lifecycle;
8. asynchronous replication одного primary на replica через WAL segments;
9–10. развитие replication и итоговая сборка.

Четыре видео с review показывают checkpoints проекта; финальный reference находится в каталоге `database` репозитория автора.

![[90 Вложения/CurseHunter/6593/Кадры/003-database-layers.jpg|720]]

### Архитектурные границы

- **Compute/parser:** строка запроса превращается в typed command; проверяются arity и syntax до storage.
- **Storage:** `Set/Get/Del` скрывают engine и принимают context.
- **In-memory engine:** partitioning по hash снижает contention, каждый partition защищает состояние.
- **WAL:** durable ordered log предшествует подтверждённому изменению state.
- **Network:** framing, max message size, idle timeout и bounded connections.
- **Replication:** primary отдаёт ordered WAL records/segments, replica применяет их в том же порядке.

### Вопросы к parser и protocol

1. Как отделить syntax error от storage error?
2. Можно ли разбить запрос обычным `strings.Fields`, если values содержат пробелы/escaping?
3. Как frame-ить TCP stream: delimiter, fixed header, length prefix?
4. Как ограничить declared length до allocation?
5. Что происходит с partial read/write и несколькими messages в одном packet?
6. Как поддерживать protocol version и backward compatibility?
7. Где backpressure: connection limit, request queue, per-client in-flight limit?
8. Как завершить server, не оборвав уже принятую WAL write между log и state apply?

### WAL: инварианты

Перед ответом нужно выбрать durability contract. Для подтверждённой записи обычно требуется:

1. сериализовать record с length/version/checksum;
2. добавить его в WAL в определённом order;
3. выполнить требуемый flush/fsync;
4. применить state либо обеспечить однозначный recovery order;
5. только после точки durability ответить клиенту.

Group commit/batching уменьшает fsync cost, но увеличивает latency и усложняет ошибку одного batch. Segment rotation требует атомарного metadata/update protocol. Recovery должен отличать корректный prefix от torn/partial tail и быть идемпотентным относительно уже применённого состояния.

### Asynchronous replication

Нужно определить:

- global position: LSN/segment+offset;
- как replica сообщает последний применённый position;
- что primary хранит до ACK и как чистит segments;
- catch-up через WAL или snapshot, если lag вышел за retention;
- как обрабатываются duplicate/reordered records;
- что обещает ACK клиента: только primary durability или replica тоже;
- что происходит при split brain и promotion;
- как fence-ить старый primary после failover.

Слово `master-slave` сохранено только при описании формулировки курса; в проектировании удобнее `primary/replica`.

## Live coding 1: bounded worker pool

### Условие курса

- конструктор принимает положительное число workers;
- `AddTask(func())` ставит работу в bounded queue либо возвращает ошибку `pool is full`;
- после `Close` новые задачи запрещены;
- `Close` закрывает queue и ждёт уже принятые задачи.

![[90 Вложения/CurseHunter/6593/Кадры/040-worker-pool.jpg|720]]

### Что должен спросить кандидат

- `AddTask` блокируется, отбрасывает, возвращает ошибку или принимает context?
- сохраняется ли FIFO start order; обещается ли completion order?
- что делать с panic task?
- может ли task вызывать `Close` того же pool?
- `Close` идемпотентен и допускает concurrent callers?
- нужно ли отменять queued tasks или drain-ить их?

### Ошибка решения

`Close` читает `wp.closed` вне mutex. Два concurrent caller могут оба увидеть `false`, последовательно установить flag и дважды закрыть `tasksCh`; второй получит panic. Проверка и state transition `open → closing` должны быть одной critical section либо `sync.Once`.

Panic task также не изолирован: `defer wg.Done` сработает при раскрутке stack, но необработанный panic завершит процесс. Production pool либо документирует это, либо ставит recovery boundary и возвращает/логирует failure.

## Live coding 2: keyed scheduler

### Условие курса

`SetTimeout(key, delay, action)` планирует функцию; новый timeout того же key заменяет старый. `CancelTimeout` отменяет key, `Close` отменяет все и запрещает новые.

![[90 Вложения/CurseHunter/6593/Кадры/041-scheduler.jpg|720]]

### Failure modes

- `Timer.Stop()` не откатывает callback, который уже начал выполняться.
- После fire entry остаётся в `actions`, поэтому map растёт до cancel/replace/close.
- Если callback удаляет entry, старый callback не должен удалить новый timer того же key: нужен generation/token identity check.
- `action` panic завершит процесс без recovery boundary.
- `Close` отменяет будущий start, но не ждёт callbacks, которые уже выполняются; этот contract нужно назвать.
- Огромное число `time.AfterFunc` создаёт иной cost profile, чем один heap/loop scheduler.

Linearization point `SetTimeout` — публикация нового timer под lock; для `Cancel` нужно определить, считается ли успехом предотвращение ещё не начавшегося callback или ожидание полного завершения.

## Live coding 3: batcher

### Условие курса

`Append` добавляет message. Flush происходит при достижении `maxSize` либо по `flushInterval`. `Close` прекращает приём, flush-ит остаток и ждёт worker.

![[90 Вложения/CurseHunter/6593/Кадры/042-batcher.jpg|720]]

### Ошибки решения

- `Run` проверяет/создаёт `ticker` без общей синхронизации: два caller могут поднять несколько workers.
- `Close` сначала проверяет `closeCh` вне lock, поэтому concurrent close способен дважды закрыть channel.
- `Close` до `Run` ждёт `closeDoneCh` вечно, а затем попытался бы вызвать `Stop` у `nil` ticker.
- `flushAction` запускается отдельной goroutine. Поэтому `Close` ждёт loop, но не ждёт завершения callbacks, а несколько batches могут примениться не по порядку.
- Ошибка callback не возвращается caller и не участвует в retry/dead-letter policy.

### Уточнённый contract

Хорошее решение имеет state machine `new → running → closing → closed`, один owner goroutine и команды через channel либо единый mutex. `Close(ctx)` идемпотентно инициирует closing, flush-ит final batch, определённо ждёт callback и возвращает error/timeout. Если batches могут исполняться параллельно, ordering explicitly не гарантируется.

## Live coding 4: Redis cache с синхронизацией

### Условие курса

Wrapper реализует `Get`. При hit читает local map, при miss обращается в Redis и заполняет cache. Background loop периодически получает `Keys + MGet` и атомарно заменяет snapshot. Внешние вызовы повторяются до трёх раз под общим timeout.

![[90 Вложения/CurseHunter/6593/Кадры/043-cache.jpg|720]]

### Ошибки и недостающие решения

1. `withRetries` делает `time.Sleep`, не выбирая `ctx.Done`; после отмены продолжает ждать. После последней неудачной попытки sleep вообще не нужен.
2. Все attempts используют один timeout, но action обязана реально принимать тот же context.
3. Если `MGet` вернул меньше элементов, чем `Keys`, indexing panic; response contract нужно проверить.
4. Concurrent miss может записать свежее значение, а поздняя full snapshot replacement потеряет его.
5. При старте cache пуст до первого tick; нужно решить, делать ли initial warm-up синхронно/асинхронно.
6. Concurrent misses одного key создают stampede; нужен singleflight или per-key coordination.
7. Нет freshness contract: TTL, maximum staleness, negative caching и invalidation.
8. Background goroutine отменяется context, но constructor не даёт `Close/Wait`, поэтому caller не может подтвердить завершение.

Senior+ ответ отделяет три механизма: cache-aside на miss, periodic snapshot refresh и retry policy. Их consistency/failure semantics нельзя склеивать одним mutex.

## Как усилить четыре решения

| Задача | Минимальный production contract |
| --- | --- |
| Worker pool | bounded queue, backpressure, panic policy, idempotent concurrent `Close`, drain/cancel semantics |
| Scheduler | keyed generation, cleanup fired timers, race с callback, `Close`/wait semantics |
| Batcher | single owner loop, size/time trigger, ordered flush, callback error, final drain, bounded close |
| Cache | freshness, atomic snapshot, miss deduplication, context-aware retry, lifecycle join, metrics |

## Финальный чек-лист интервьюера

Для любой background component спросите:

1. Кто владеет goroutine и как её остановить?
2. Есть ли единственный state transition в closing?
3. Что происходит при двух concurrent `Close`?
4. Что происходит при panic callback/task?
5. Где backpressure и ограничена ли память?
6. Что обещает возврат `Close`: сигнал отправлен, loop завершён или все side effects durable?
7. Как наблюдаются ошибки background работы?
8. Какие события являются linearization points?
9. Какие операции могут продолжаться после timeout/cancel?
10. Какие метрики и traces подтвердят failure mode в production?
