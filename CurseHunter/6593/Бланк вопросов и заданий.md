## Интервью-ядро

Если времени мало, сначала отработайте эти вопросы и задачи:

1. Провести happens-before для кода и отличить data race от race condition, deadlock, livelock и starvation.
2. Объяснить GMP без легенды «одна горутина равна одному потоку»: кто владеет локальной очередью, зачем GRQ, work stealing, netpoller и sysmon.
3. Найти бесполезный локальный mutex, слишком крупную критическую секцию, копирование `sync.Mutex` и `WaitGroup`, `defer` внутри цикла и публикацию частично инициализированного состояния.
4. Объяснить normal/starvation mode mutex и почему fairness, throughput и latency конфликтуют.
5. Реализовать ограниченный worker pool с контрактом `AddTask`/`Close`, backpressure, обработкой panic и идемпотентным завершением.
6. Заполнить матрицу операций для `nil`, открытого и закрытого channel; назвать владельца `close`.
7. Реализовать fan-in, fan-out, pipeline и отмену так, чтобы ранний выход consumer не оставлял заблокированных producer goroutine.
8. Объяснить, что `context` только сообщает об отмене: вызываемый код обязан наблюдать `Done` либо передать контекст дальше.
9. Найти ошибку публикации `data` через обычный `bool`; исправить с mutex, channel, `sync.Once` или atomic protocol и доказать happens-before.
10. Сравнить coarse-grained, fine-grained, optimistic и lock-free синхронизацию по contention, сложности доказательства и reclamation памяти.
11. Объяснить ABA на стеке Трайбера и почему один успешный CAS ещё не доказывает корректность структуры.
12. Сравнить 2PL и MVCC не по лозунгу, а по конфликтам чтения/записи, дедлокам, версиям, очистке и требуемому уровню изоляции.
13. Посчитать предел ускорения по закону Амдала и найти последовательную долю, которая делает добавление goroutine бессмысленным.
14. Спроектировать scheduler с заменой и отменой задач по ключу; разобрать гонку `Timer.Stop` с уже запущенной функцией.
15. Спроектировать batcher с flush по размеру и времени; определить семантику `Close`, владение ticker и ожидание callback.
16. Провести code review read-through cache: атомарная замена снапшота, retries с context, stampede, stale data, удаление и гонка фоновой синхронизации с cache miss.

## Существенные коррекции курса

- `make(chan T)` возвращает значение типа channel, а не значение Go-типа «указатель». В текущем runtime это дескриптор со ссылочной семантикой на `hchan`, но layout `hchan` — implementation detail.
- Слайд «MVCC выдаёт каждому пользователю копию данных» полезен только как образ. Обычно это логический snapshot и выбор видимой версии, а не физическое копирование всей базы.
- Правило «2PL для пересекающихся данных, MVCC для непересекающихся» слишком грубое. Выбор зависит от типа конфликтов, уровня изоляции, read/write ratio, стоимости версий, очистки и реализации конкретной СУБД.
- В Go atomic-операции имеют sequentially consistent semantics. Перенос терминов LoadLoad/LoadStore/StoreStore из CPU-модели не заменяет доказательство через Go memory model.
- Начиная с Go `1.25`, `sync.WaitGroup` получил метод `Go`; видео на Go `1.22.5` закономерно использует `Add` + goroutine + `Done`.
- В live-coding коде есть реальные failure modes: конкурентный двойной `Close` в worker pool и batcher, `Close` batcher до `Run`, неостанавливаемая уже запущенная timer-функция, context-unaware sleep в retries и потеря cache update при замене снапшота.
