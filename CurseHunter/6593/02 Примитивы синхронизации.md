# Примитивы синхронизации

## TL;DR

Примитив выбирают не по привычке, а по инварианту:

- нужно дождаться конечного набора работ — `WaitGroup`;
- нужно сериализовать переход состояния — `Mutex`;
- нужно ждать предикат, который меняют другие goroutine — `Cond` либо channel-based protocol;
- нужна однократная публикация полностью готового состояния — `Once`;
- нужен атомарный переход одного машинного состояния — `sync/atomic`;
- нужен временный reuse объектов — `Pool`;
- нужен concurrent map для конкретного профиля доступа — `sync.Map` либо обычная map под lock.

Главный навык на интервью — провести happens-before и отдельно проверить liveness. Отсутствие data race ещё не исключает deadlock, starvation, leak или нарушение бизнес-инварианта.

## `WaitGroup`

### Ментальная модель

Это счётчик незавершённых работ, а не контейнер goroutine и не mutex. `Done` уменьшает counter, `Wait` блокируется до нуля. В Go `1.22.5` безопасный шаблон — вызвать `Add` до старта goroutine, внутри сделать `defer Done`. Начиная с Go `1.25`, стандартная библиотека также предлагает `wg.Go(f)`.

### Что спрашивают

- Почему `Add(1)` внутри новой goroutine может привести к преждевременному возврату `Wait`?
- Можно ли копировать `WaitGroup` после первого использования? Нет: копия отделит внутреннее состояние и нарушит протокол.
- Упорядочивает ли `Wait` две sibling goroutine между собой? Нет. Он связывает завершение каждой работы с возвратом `Wait`, но не сериализует конкурентные чтение и запись до завершения.
- Можно ли переиспользовать `WaitGroup`? Да, новый цикл `Add` начинается только после возврата всех предыдущих `Wait`; racing reuse запрещён.
- Что произойдёт при отрицательном counter? Panic.

Практические задачи `wait_group_order`, `wait_group_copying` и `data_race` совпадают с [[CurseHunter/6609/12 Примитивы синхронизации|курсом 6609]].

## `Mutex`: инвариант важнее lock/unlock

Правильный ответ начинается с формулировки защищаемого состояния: «пока lock удерживается, поля `balance` и `version` меняются атомарно относительно других операций этого объекта». Фраза «закрою map mutex-ом» недостаточна, если часть методов использует другой lock или возвращает ссылку на внутренние данные.

### Частые задачи на code review

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| mutex, созданный внутри каждого вызова, общий | потерянные increments | goroutine блокируют разные mutex | хранить один lock рядом с общим состоянием |
| `WaitGroup` защищает `value++` | нестабильный результат, race detector | read-modify-write не сериализован | mutex либо atomic RMW |
| lock можно держать во время логирования/сети | latency и convoy | критическая секция включает медленную работу | скопировать защищённые данные и отпустить lock |
| `defer Unlock` всегда оптимален | lock живёт до конца большой функции | область `defer` шире нужной секции | вынести секцию в маленькую функцию или явный unlock |
| panic не влияет на lock | последующий deadlock | unlock не выполнен | `defer` сразу после успешного `Lock` на правильной границе |
| mutex рекурсивный | self-deadlock | `sync.Mutex` не отслеживает владельца и не допускает reentrancy | убрать повторный захват, разделить locked/unlocked helpers |
| копия struct с mutex независима и безопасна | две блокировки защищают общие ссылки | mutex скопирован, data всё ещё alias | pointer receiver, не копировать после использования |

### Normal и starvation mode

В runtime Go `1.22.5` mutex имеет fast path через CAS и slow path с semaphore queue. В normal mode новый contender может конкурировать с уже разбуженным waiter, что повышает throughput, но создаёт риск повторного проигрыша. После длительного ожидания mutex способен перейти в starvation mode и передавать ownership ожидающему напрямую. Порог и bit layout — implementation detail; их нельзя использовать как контракт программы.

### Priority inversion

Классический сценарий: низкоприоритетный исполнитель держит lock, высокоприоритетный ждёт, а среднеприоритетная работа вытесняет владельца lock. Go не даёт приложению API приоритетов goroutine, но аналогичный latency effect возникает, когда некритичная работа задерживает владельца ресурса, нужного latency-sensitive path. Практические меры: короткие critical sections, разделение locks, исключение blocking I/O под lock и измерение contention.

## `sync.Once`

`Once.Do(f)` гарантирует, что завершение единственного `f` happens-before возврата любого `Do`. Получается безопасная публикация с ожиданием полного завершения `f`; один CAS-флаг такого контракта не даёт.

Интервью-ловушки:

- если `f` паникует, конкретный `Once` считает вызов выполненным; обычный `Do` больше не повторит `f`;
- вызывать `Do` того же `Once` из `f` нельзя — получится deadlock;
- `Once` защищает инициализацию, но не делает дальнейшие операции с map конкурентно безопасными;
- double-checked locking с обычным `bool` не гарантирует видимость полей объекта.

## `sync.Cond`

`Cond` нужен, когда goroutine ждёт не событие как единичное сообщение, а истинность предиката над общим состоянием.

Канонический протокол:

```go
c.L.Lock()
for !ready() {
    c.Wait()
}
consume()
c.L.Unlock()
```

`Wait` атомарно отпускает `L`, park-ит goroutine, затем перед возвратом снова захватывает `L`. Цикл обязателен: сигнал не резервируется для будущего waiter, а после пробуждения состояние мог изменить другой contender. `Signal` будит одного, `Broadcast` — всех; изменение предиката выполняют под тем же lock.

Задачи курса:

- вызвать `Wait` без lock — runtime panic из-за unlock незахваченного mutex;
- сделать `Signal`, затем начать ждать — сигнал потерян, возможен deadlock;
- реализовать counting semaphore как `count/max + Cond`;
- объяснить, когда закрытие channel проще и безопаснее `Broadcast`.

## `sync.Pool`

Pool — кэш временных объектов для снижения allocation pressure. Runtime может удалить элементы в любой момент; Pool нельзя использовать как bounded cache, очередь, storage или механизм обязательного возврата. `Put(x)` synchronizes-before `Get`, который вернул тот же `x`, но программа не может требовать, чтобы конкретный объект вообще был возвращён.

Задача: реализовать упрощённый pool и объяснить, какие свойства production `sync.Pool` потеряны — per-P storage, victim cache, взаимодействие с GC, memory ordering и конкуренция.

## Atomic и публикация состояния

### Когда atomic уместен

- независимый counter;
- state machine с явно перечисленными переходами;
- immutable snapshot через `atomic.Pointer[T]`;
- редко меняемая конфигурация через `atomic.Value`.

Atomic плохо подходит для инварианта нескольких независимо изменяемых полей: отдельная атомарность каждого поля не делает снимок целиком согласованным.

### CAS-loop

Правильный loop каждый раз:

1. загружает наблюдаемое состояние;
2. вычисляет новое без побочных эффектов;
3. делает CAS;
4. при неудаче перечитывает состояние и повторяет.

Нельзя вызывать необратимую операцию до успешного CAS: loop может повториться.

### `atomic.Value` и `atomic.Pointer`

`atomic.Value` удобен для immutable snapshot, но все непустые `Store` должны иметь один concrete type; `Store(nil)` panic. `atomic.Pointer[T]` типобезопаснее для `*T`, но lifetime объектов и отсутствие мутации после публикации всё равно доказывает автор кода.

### False sharing

Два независимых atomic counter на одной cache line могут непрерывно инвалидировать кэш друг друга. Семантически код корректен, но масштабируется плохо. Padding или sharding помогают только после измерения и с учётом реального cache-line layout платформы.

## Самодельные locks

Курс показывает spinlock, spin + `Gosched`, hybrid spin, ticket lock, Peterson lock, timed и recursive mutex. Это интервью-упражнения, а не рекомендация заменить `sync.Mutex`.

### Что нужно доказать

- mutual exclusion;
- progress: deadlock-free, starvation-free или хотя бы lock-free — в зависимости от обещания;
- memory ordering публикации данных;
- корректность при panic/отмене;
- overflow ticket counter;
- поведение при oversubscription и долгой critical section.

Простой CAS spinlock жжёт CPU и мешает владельцу lock получить время на oversubscribed машине. `Gosched` снижает busy spin, но не даёт fairness. Ticket lock упорядочивает contenders, но все они читают одну cache line; отмена ticket без дополнительного протокола трудна.

## `RWMutex`, map и `sync.Map`

### Когда `RWMutex` может проиграть `Mutex`

Если critical section мала, readers мало, writes часты или cache contention велик, bookkeeping `RLock/RUnlock` съедает выигрыш. Решение принимают benchmark под похожей нагрузкой, а не по слову «чтений больше».

Нельзя рекурсивно брать read lock при ожидающем writer, upgrade `RLock → Lock` и downgrade `Lock → RLock` без release gap. Копировать `RWMutex` после использования также нельзя.

### Обычная map или `sync.Map`

- Обычная map + lock даёт типобезопасность, составные операции и явный инвариант.
- `sync.Map` оптимизирован для write-once/read-many либо disjoint-key concurrent access и имеет специальные атомарные операции (`LoadOrStore`, `CompareAndSwap` и другие).
- Последовательность `Load` затем `Store` не атомарна как единая бизнес-операция.
