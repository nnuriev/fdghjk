# Каналы, время и паттерны

## TL;DR

Channel одновременно передаёт значение, синхронизирует goroutine и задаёт ownership protocol. Он не устраняет race вокруг других переменных автоматически. Для каждого channel на интервью нужно назвать:

1. кто создаёт;
2. кто отправляет и сколько sender;
3. кто получает и может ли уйти раньше;
4. кто единственный закрывает;
5. как отменяются заблокированные send/receive;
6. кто ждёт завершения созданных goroutine.

## Матрица состояний

| Операция | `nil` channel | открытый channel | закрытый channel |
| --- | --- | --- | --- |
| send | блокируется навсегда | синхронно передаёт либо ждёт места в buffer | panic |
| receive | блокируется навсегда | получает buffered/новое значение либо ждёт | сначала drain buffer, затем zero value и `ok=false` |
| `close` | panic | закрывает один раз | panic |
| `len` / `cap` | `0` / `0` | моментальный snapshot | buffer может ещё содержать значения |

![[90 Вложения/CurseHunter/6593/Кадры/023-channel-state-matrix.jpg|720]]

`nil` channel полезен внутри `select`: case с ним выключен. Вне управляемого `select` send/receive на `nil` обычно означает утечку goroutine или deadlock.

## Что гарантирует коммуникация

- send happens-before завершения соответствующего receive;
- close happens-before receive, который вернул zero value из-за закрытия;
- для unbuffered channel receive happens-before завершения соответствующего send;
- для capacity `C` k-й receive happens-before завершения k+`C`-го send.

Эти связи относятся к конкретной коммуникации. Если две goroutine одновременно меняют внешнюю map, наличие рядом channel не защищает map без подходящего протокола.

## Внутреннее устройство и граница контракта

В Go `1.22.5` runtime представляет channel структурой `hchan`: кольцевой buffer, counters/indices, queues ожидающих send/receive и lock. `sudog` связывает parked goroutine с операцией ожидания. При прямой передаче значение может копироваться между stack sender и receiver через runtime protocol.

![[90 Вложения/CurseHunter/6593/Кадры/024-hchan.jpg|720]]

Это помогает объяснить blocking и contention, но поля `hchan`, queue discipline и оптимизации не входят в спецификацию. `make(chan T)` возвращает значение типа `chan T`; фраза курса «make возвращает указатель на hchan» описывает текущую реализацию слишком буквально.

## Базовые задачи с channel

### Кто закрывает channel?

Обычно sender-side owner, который знает, что новых значений больше не будет. Receiver не должен закрывать чужой channel только потому, что больше не хочет читать: concurrent sender получит panic. При нескольких sender нужен coordinator, `WaitGroup` и отдельная goroutine, закрывающая output после их завершения.

### Как проверить, закрыт ли channel?

Универсального безопасного `IsClosed` нет. Receive меняет состояние: может забрать реальное значение. Даже проверка мгновенно устаревает перед следующим send/close. Нужно проектировать ownership или отдельный done signal.

### Что делает копирование channel?

Копируется descriptor/reference на тот же channel, не новый buffer. Закрытие через одну переменную влияет на все aliases.

### Почему `select { default: ... }` опасен?

Он делает операцию неблокирующей, но tight loop с `default` превращается в busy spin. Кроме того, «не готово сейчас» не означает «не случится» и может молча терять данные, если это не часть контракта.

### Гарантирует ли `select` приоритет case?

Если готовы несколько case, выбирается pseudo-random ready case. Дублирование case создаёт вес, но не строгий приоритет. Для строгого предпочтения нужен двухэтапный protocol с повторной проверкой high-priority input.

### Что делают `break` и `continue` внутри `select`?

Без label они относятся к ближайшей конструкции по правилам языка: `break` завершает `select`, а не окружающий `for`; `continue` продолжает loop. Это частый источник бесконечного цикла после закрытия input. Закрытый input обычно присваивают `nil`, чтобы отключить case.

### Как возникает goroutine leak?

- producer навсегда блокируется на send, потому что consumer вернулся;
- consumer ждёт channel, который никто не закроет;
- send результата в unbuffered channel переживает timeout caller;
- ticker, worker или bridge не имеет cancellation path.

Buffer размером `1` может спасти одиночный поздний результат, но не заменяет cancellation для произвольного потока.

## Время: Timer и Ticker

Курс записан на Go `1.22.5`, а timer semantics заметно изменились в Go `1.23`.

### Практические вопросы

- `time.After` удобно для разового timeout, но в hot loop лучше контролировать и переиспользовать `Timer`.
- `Ticker` нужно останавливать, когда owner закончил работу; закрытия `Ticker.C` после `Stop` ждать нельзя.
- До Go `1.23` после `Stop`/`Reset` приходилось учитывать buffered stale value и аккуратно drain-ить channel.
- Начиная с Go `1.23`, timer channels синхронные, а после успешного `Stop` или `Reset` не должно быть stale receive от прежней конфигурации; unreferenced timers могут быть собраны GC.
- Callback `time.AfterFunc` имеет другой contract: `Reset` может запланировать новый callback, а `Stop=false` означает, что функция уже стартовала или timer уже истёк. Ждать её завершения нужно отдельно.

## Composable channel patterns

### Generator, transformer, filter

Каждый stage владеет своим outbound channel и закрывает его после завершения всех send. Если downstream может уйти раньше, каждое потенциально блокирующее действие выбирает между data path и cancellation.

### Fan-out

Несколько workers читают один input. Это распределяет элементы, а не копирует каждый элемент всем. Нужно выбрать число workers, порядок результата и backpressure.

### Fan-in

Отдельная goroutine копирует каждый input в общий output; coordinator ждёт все copy goroutine и затем закрывает output. Закрыть output до завершения send — panic.

![[90 Вложения/CurseHunter/6593/Кадры/027-fan-in.jpg|640]]

### Pipeline

Stage получает input, преобразует и отправляет output. Инварианты:

- stage закрывает только свой output;
- cancellation наблюдается и на receive, и на send;
- error path останавливает upstream либо продолжает drain;
- порядок и degree of parallelism являются частью API.

![[90 Вложения/CurseHunter/6593/Кадры/027-pipeline.jpg|640]]

### Tee и bridge

- Tee копирует каждое значение во все outputs. Чтобы не зависеть от того, какой consumer готов первым, outputs временно выключают присваиванием `nil` после успешного send.
- Bridge превращает channel of channels в один поток. Sequential bridge сохраняет порядок внутренних потоков, concurrent merge — нет; это разные contracts.

### Or-channel и or-done

- Or-channel закрывается/сигналит при завершении любого input и объединяет несколько cancellation sources.
- Or-done оборачивает receive, чтобы caller мог прекратить ожидание. Он не отменяет producer сам по себе, если producer не наблюдает тот же signal.

### Barrier и semaphore

- Barrier ждёт заданное число участников перед продолжением фазы; `WaitGroup` подходит для однонаправленного barrier, reusable barrier требует поколений.
- Buffered channel моделирует counting semaphore. Нужно определить, send — acquire или release, и во всех путях, включая error/panic, симметрично вернуть permit.

### Future и Promise

Одно значение либо ошибка публикуются один раз, readers ждут result channel. Для нескольких readers channel с одним value недостаточен: один reader заберёт значение. Нужен immutable result + close broadcast либо иной shared state protocol.

### `errgroup`

`errgroup.WithContext` объединяет ожидание и отмену sibling tasks после первой ошибки. Он не ограничивает concurrency без `SetLimit`, не восстанавливает panic и не гарантирует сбор нескольких ошибок. Функции должны реально наблюдать context.

### Singleflight

Одновременные запросы одного ключа делят один in-flight call. Нужно решить:

- получают ли waiters одну и ту же ошибку;
- может ли caller отменить только своё ожидание;
- когда entry удаляется;
- кешируется ли результат после завершения — обычный singleflight этого не обещает.

### Rate limiter

На интервью сравнивают fixed window, sliding window, leaky bucket и token bucket. Нужно назвать burst semantics, granularity времени, распределённую координацию и поведение при отмене. Ticker-per-request плохо масштабируется.

### Graceful shutdown

Owner ловит process signal, прекращает приём новой работы, отменяет context, ждёт in-flight tasks с deadline и только затем освобождает ресурсы. Закрытие общего channel — broadcast, но закрывать его должен один coordinator.
