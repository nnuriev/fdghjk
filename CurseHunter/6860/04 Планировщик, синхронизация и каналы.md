# Планировщик, синхронизация и каналы

## TL;DR

Scheduler отвечает за порядок runnable work, synchronization — за допустимые interleavings shared state, channel — за communication и ownership. Домашние задания моделируют все три слоя: mutable priority queue, writer-preferring `RWMutex` и bounded worker pool. Сильный ответ всегда называет state, owner, linearization point, happens-before edge и shutdown protocol.

## Задание 12. Priority scheduler с изменением priority

![[90 Вложения/CurseHunter/6860/Кадры/12-homework-priority-scheduler.jpg|720]]

### Контракт курса

```go
type Task struct {
    Identifier int
    Priority   int
}

type Scheduler struct { /* ... */ }

func NewScheduler() Scheduler
func (s *Scheduler) AddTask(task Task)
func (s *Scheduler) ChangeTaskPriority(taskID, newPriority int)
func (s *Scheduler) GetTask() Task
```

Из tasks с priorities `10,20,30,40,50` извлекаются сначала `50`, затем `40`. После изменения task 1 с `10` на `100` он извлекается следующим, затем идёт `30`. Источник: [`homework/goroutines_and_scheduler`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/goroutines_and_scheduler/homework_test.go).

### Структура данных

Max-heap даёт `AddTask` и `GetTask` за `O(log n)`, но одного heap недостаточно для быстрого `ChangeTaskPriority`: по `taskID` пришлось бы искать `O(n)`. Нужен дополнительный index:

```text
heap[index] = Task
position[taskID] = index
```

Каждый swap heap elements обязан синхронно обновлять `position`. После изменения priority вызывается `heap.Fix`, который перемещает element вверх или вниз за `O(log n)`. После extract удалённый ID убирается из map.

### Недоопределённые части

- duplicate `Identifier`: reject, replace или две tasks?
- equal priority: FIFO stability, ID tie-breaker или arbitrary order?
- `GetTask` на empty scheduler: zero Task, `(Task,bool)` или error?
- change missing task: no-op или error?
- concurrent access: task не требует, но production scheduler нуждается в lock или single owner goroutine.

Если требуется FIFO для equal priority, task получает monotonic sequence; comparator использует `(priority desc, sequence asc)`.

- Что делать с priority inversion и starvation низкоприоритетных tasks?
- Как aging меняет comparator?

## Задание 13. Writer-preferring RWMutex

![[90 Вложения/CurseHunter/6860/Кадры/13-homework-rwmutex.jpg|720]]

### Контракт курса

Реализовать `Lock`, `Unlock`, `RLock`, `RUnlock` поверх готового `sync.Mutex`. Несколько readers могут владеть lock одновременно; writer exclusive. Если writer уже ждёт, новые readers обязаны ждать за ним — тест явно проверяет writer priority. Задание со звёздочкой — non-blocking `TryLock/TryRLock`. Источник: [`homework/sync_primitives`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/sync_primitives/homework_test.go).

### State и predicates

Одна из моделей:

```text
activeReaders  >= 0
writerActive   ∈ {false,true}
waitingWriters >= 0
```

Все поля защищает internal `sync.Mutex`. Удобно иметь две `sync.Cond` с одним locker: отдельные очереди ожидания для readers и writers. С одной общей condition `Signal` не умеет выбрать writer среди смешанных waiters; тогда пришлось бы использовать `Broadcast` и повторную проверку predicates.

- Reader может войти, только если `!writerActive && waitingWriters==0`.
- Writer может войти, только если `!writerActive && activeReaders==0`.
- `RLock` ждёт predicate на reader condition в loop, затем увеличивает `activeReaders`.
- `RUnlock` уменьшает count; последний reader сигналит writer condition.
- `Lock` заранее увеличивает `waitingWriters`, ждёт на writer condition, затем переводит state в active writer.
- `Unlock` снимает writer; если writers ждут, сигналит одного writer, иначе делает broadcast readers.

`Cond.Wait` должен вызываться с locked `Cond.L`; он сам unlocks lock while sleeping и relocks перед возвратом. Predicate повторяется после wakeup, потому что сигнал не резервирует condition за конкретной goroutine.

### Linearization points

- успешный read lock — увеличение `activeReaders` под mutex;
- успешный write lock — установка `writerActive=true`;
- unlock — изменение state перед signal/broadcast.

Try methods проверяют тот же predicate один раз под mutex и возвращают `false`, не ожидая.

### Failure modes

- разрешить reader при `waitingWriters>0` — writer starvation;
- ждать на `Cond` через `if`, а не `for` — вход после устаревшего signal;
- забыть decrement waitingWriters — readers навсегда blocked;
- `Broadcast` на каждом transition — thundering herd;
- copy mutex после use — разделённое state и undefined synchronization behavior;
- `RUnlock` без matching `RLock` — invalid state/panic policy.

Тесты используют `time.Sleep` как доказательство blocking и не выполняют полный cleanup blocked goroutines. Это достаточно для учебного шаблона, но production tests должны использовать handshakes/barriers, time-bounded completion и проверку race detector.

### Trade-off со стандартным `sync.RWMutex`

Учебная реализация объясняет protocol, но production code должен предпочитать стандартный primitive. Его memory-model edges документированы, implementation оптимизирована и протестирована на редких interleavings. Собственный lock оправдан лишь при реально ином contract и строгой верификации.

## Задание 14. Bounded worker pool

![[90 Вложения/CurseHunter/6860/Кадры/14-homework-worker-pool.jpg|720]]

### Контракт курса

```go
type WorkerPool struct { /* ... */ }
func NewWorkerPool(workersNumber int) *WorkerPool
func (wp *WorkerPool) AddTask(task func()) error
func (wp *WorkerPool) Shutdown()
```

Требования слайда:

- фиксированное число workers;
- если task queue full, `AddTask` сразу возвращает error, а не blocks;
- `Shutdown` прекращает приём, drains принятые tasks и ждёт workers.

Задание со звёздочкой — task, выполняемая заданное число раз. Источник: [`homework/channels`](https://github.com/Balun-courses/deep_go/blob/aacfeb63b810479f98cebcaa16a444e3c8710ee0/homework/channels/homework_test.go).

### Ownership model

Pool владеет:

- buffered `tasks chan func()` с явной capacity;
- lifecycle state `open/closing/closed`;
- worker `WaitGroup`;
- единственным close канала.

Worker выполняет `for task := range tasks { task() }`. После close он drains уже buffered tasks, затем выходит; `WaitGroup.Wait` доказывает завершение.

### Безопасный admission против shutdown

Опасный вариант проверяет boolean `closed`, отпускает lock и отправляет в channel: между проверкой и send другой caller закроет channel, возникнет `send on closed channel`.

Безопасный вариант выглядит так:

1. `AddTask` держит read lock на lifecycle, проверяет `open` и делает non-blocking `select` send под этим lock.
2. `Shutdown` берёт write lock, переводит state в closing и закрывает channel ровно один раз.
3. После unlock ждёт worker group.

Поскольку close требует write lock, он не пересекается с send под read lock. Send остаётся non-blocking благодаря `default` case.

### Backpressure contract

Queue capacity нельзя неявно приравнивать к worker count. Это два разных knobs:

- workers ограничивают concurrency и защищают CPU/dependency;
- queue ограничивает допустимое ожидание и память.

`ErrFull` означает load shedding. Caller должен знать, retry, drop, fallback или propagate rejection. Unbounded queue лишь превращает overload в latency и memory pressure.

### Слабость теста курса

Тест отправляет три tasks в pool из двух workers, игнорирует errors и ожидает, что все три выполнятся. Он не фиксирует queue capacity и не доказывает full-path. Результат зависит от того, успели ли workers принять первые sends. Поэтому сильная реализация добавляет deterministic tests:

- workers заблокированы barrier;
- queue заполнена до capacity;
- следующий `AddTask` обязан вернуть `ErrFull` немедленно;
- concurrent `AddTask/Shutdown` многократно проверяется с race detector;
- repeated `Shutdown` имеет idempotent contract.

### Panic и cancellation

Если task panic и worker не recover, process обычно завершится. Если pool вводит recovery boundary, он обязан записать panic/stack и определить, продолжает ли worker работу. Cancel running task невозможно «силой»; task должен принимать context или собственный cancellation signal.

## Сквозная задача

Спроектируйте in-process image processing service:

- jobs имеют tenant, priority и deadline;
- одновременно выполняется не больше `N` CPU-heavy jobs;
- queue bounded;
- priority можно повысить до начала выполнения;
- одинаковый tenant не должен занять всю capacity;
- shutdown имеет 10 seconds на drain.

Сильный ответ связывает:

1. max-heap + `position` map для pending jobs;
2. admission limits per tenant и global;
3. mutex/cond либо single scheduler goroutine для ownership state;
4. worker pool с context-aware tasks;
5. state transitions `queued/running/completed/canceled`;
6. idempotent shutdown и timeout policy;
7. metrics: queue depth, rejection, wait time, run time, cancellations, panic count.
