## Как распознать паттерн

![[90 Вложения/CurseHunter/7253/Кадры/001-two-pointer-patterns.jpg|720]]

- **Навстречу:** ответ зависит от пары на краях отсортированного/симметричного диапазона.
- **В одном направлении:** один указатель читает, другой обозначает место записи или границу валидного prefix.
- **По двум входам:** merge/intersection двух отсортированных последовательностей.
- **С конца:** output пишется в свободный tail либо крупнейший элемент определяется краями.

Ускорение появляется не от двух переменных, а от монотонности: после сравнения можно доказать, что целый класс кандидатов больше не нужен.

## 1. Reverse String — LeetCode 344

### Вопрос

Развернуть `[]byte` in-place с `O(1)` auxiliary space.

### Инвариант

До каждой итерации элементы вне `[left, right]` уже стоят на конечных местах. Swap краёв и сдвиг обоих указателей сохраняет инвариант. Time `O(n)`, space `O(1)`.

### Что уточнить

Официальная задача даёт массив символов, а Go-решение работает с bytes. Для произвольной UTF-8 строки reversal bytes портит многобайтовые code points; нужен `[]rune` либо более точный контракт о grapheme clusters.

## 2. Valid Palindrome — LeetCode 125

### Вопрос

Игнорируя регистр и неалфавитно-цифровые символы, определить palindrome.

Левый и правый указатели независимо пропускают noise, затем сравнивают нормализованные символы. При совпадении сдвигаются оба; при различии ответ окончательно `false`.

![[90 Вложения/CurseHunter/7253/Кадры/003-valid-palindrome.jpg|720]]

### Важная граница реализации

Код курса реализует ASCII-проверку над `byte` и ручной `toLower`. Комментарии упоминают `unicode.IsLetter/IsNumber/ToLower`, но для них надо декодировать runes; заменить одну функцию, сохранив byte indexes, недостаточно. Официальные constraints ограничивают input printable ASCII, поэтому показанное решение принято.

Time `O(n)`, space `O(1)` для ASCII. Edge cases: пустая строка, только punctuation, один значимый символ.

## 3. Merge Sorted Array — LeetCode 88

### Почему merge идёт с конца?

В начале `nums1` лежат ещё не обработанные элементы. Запись слева могла бы затереть их. Tail `nums1[m:m+n]` свободен, поэтому сравниваем `nums1[m-1]` и `nums2[n-1]`, а больший кладём в `nums1[m+n-1]`.

![[90 Вложения/CurseHunter/7253/Кадры/004-merge-sorted.jpg|720]]

### Инвариант

Suffix после write pointer уже содержит крупнейшие элементы объединения в правильном порядке. Если `nums2` исчерпан, оставшийся prefix `nums1` уже на месте. Если `nums1` исчерпан, остаток `nums2` копируется в начало.

Time `O(m+n)`, space `O(1)`. Follow-up: если буфера в `nums1` нет, нужен отдельный output либо более сложный in-place merge.

## 4. Intersection of Two Arrays — LeetCode 349

### Вариант курса

Отсортировать оба массива, идти `i/j`. Меньший элемент продвигает свой указатель; равенство добавляет значение один раз, после чего оба указателя пропускают все его duplicates.

Time `O(n log n + m log m)`, traversal `O(n+m)`. Сортировка `slices.Sort` меняет оба input. Это надо назвать: если mutation запрещена, копировать или использовать hash set.

### Ближайшая альтернатива

Hash set даёт expected `O(n+m)` time и `O(min(n,m))` space без требования sorted input. Sort + two pointers выигрывает при уже отсортированных данных, ограниченной памяти или необходимости deterministic traversal.

### Типичная ошибка

В ветке равенства после пропуска duplicates важно проверять свой массив своим указателем: `nums1[i]` с `i`, `nums2[j]` с `j`. OCR промежуточного кадра искажает вторую петлю, но финальный запуск на видео имеет `Accepted`.

## 5. Squares of a Sorted Array — LeetCode 977

### Почему смотреть на края?

После возведения отрицательные числа меняют порядок, но максимальный модуль всегда находится на одном из краёв ещё не обработанного диапазона. Больший квадрат пишем в output справа налево.

![[90 Вложения/CurseHunter/7253/Кадры/006-sorted-squares.jpg|720]]

Инвариант: suffix output уже заполнен крупнейшими квадратами. Time `O(n)`, output space `O(n)`, auxiliary state `O(1)`.

В общем API `abs(minInt)` и `x*x` могут overflow. Constraints LeetCode ограничивают значения; production-контракт должен задать range или использовать более широкий тип/проверку.

## 6. Remove Duplicates from Sorted Array — LeetCode 26

`slow` указывает на последний unique element, `fast` сканирует input. Когда `nums[fast] != nums[slow]`, увеличиваем `slow` и копируем новый unique value. После цикла длина результата `slow+1`; для пустого input — `0`.

Инвариант: `nums[:slow+1]` содержит все уникальные значения просмотренного prefix в исходном порядке. Time `O(n)`, space `O(1)`.

Follow-up: разрешить не более `k` повторов. Тогда write pointer сравнивает текущий элемент с `nums[write-k]`.

## 7. Move Zeroes — LeetCode 283

`fast` ищет non-zero, `slow` — позицию следующего non-zero в compact prefix. Swap сохраняет относительный порядок non-zero элементов и постепенно отправляет zeros вправо.

![[90 Вложения/CurseHunter/7253/Кадры/008-move-zeroes.jpg|720]]

Time `O(n)`, space `O(1)`, результат stable. Когда `slow == fast`, self-swap безвреден. Альтернатива «сначала записать non-zero, затем заполнить tail нулями» делает меньше swap, но два прохода по части массива.

## 8. Two Sum II — LeetCode 167

Для sorted array:

- если `a[left] + a[right] < target`, любой pair с текущим `left` и меньшим right тоже слишком мал — двигаем `left`;
- если сумма больше target, любой pair с текущим `right` и большим left тоже слишком велик — двигаем `right`;
- иначе пара найдена.

Это и есть доказательство безопасного отбрасывания, которого часто не хватает в ответе. Time `O(n)`, space `O(1)`. Официальный output использует 1-based indexes.

## 9. 3Sum — LeetCode 15

### Декомпозиция

Сортируем. Для каждого `idx` фиксируем `nums[idx]`, а на suffix решаем Two Sum для target `-nums[idx]`. Duplicate anchors пропускаем; после найденной тройки двигаем обе границы и пропускаем равные значения.

![[90 Вложения/CurseHunter/7253/Кадры/010-three-sum.jpg|720]]

Time `O(n²)` после `O(n log n)` sort, итого `O(n²)`. Auxiliary space зависит от sort implementation и не включает output.

### Pruning и ошибки

- если `nums[idx] > 0`, дальше zero sum невозможна;
- если минимально возможная сумма уже > 0 или максимально возможная < 0, итерацию можно завершить/пропустить;
- duplicate skipping должен происходить после pointer move и сравнивать с предыдущим/следующим уже использованным значением;
- возвращаются уникальные value triples, не index triples.

## 10. String Compression — LeetCode 443

### Read/write pointers

`right` стоит в начале группы и измеряет `groupSize`. `left` пишет символ, затем decimal digits размера, если он больше одного. После этого `right += groupSize`.

![[90 Вложения/CurseHunter/7253/Кадры/011-string-compression.jpg|720]]

Инвариант: `chars[:left]` — полное сжатие `chars[:right]`, а `left <= right`, поэтому запись не уничтожает непрочитанные данные. Time `O(n)`.

Курс использует `strconv.Itoa(groupSize)`: это создаёт временную string длины `O(log groupSize)`. Для требований задачи обычно говорят constant extra space относительно фиксированных integer bounds; при строгом асимптотическом учёте произвольного `n` временный buffer не `O(1)`. Можно записывать digits вручную в небольшой локальный buffer.

Edge cases: singleton не получает `'1'`; count `10+` записывается несколькими символами; возвращается новая logical length, tail не важен.

## 11. Compare Version Numbers — LeetCode 165

Два cursor независимо читают очередную numeric revision до точки. Отсутствующий tail трактуется как нули, leading zeros не влияют. После сравнения равных revisions указатели пропускают разделитель и переходят дальше.

![[90 Вложения/CurseHunter/7253/Кадры/012-compare-versions.jpg|720]]

Time `O(n+m)`, space `O(1)` в рамках официальных ограничений. Парсинг revision в `int` полагается на ограниченный размер. Для version API с произвольно длинными сегментами безопаснее убрать leading zeros, сравнить длины, затем lexicographically — без integer overflow.

Пограничные пары: `1.0 == 1`, `1.01 == 1.001`, `1.0.0 == 1`, `1.10 > 1.2`.

## 12. Container With Most Water — LeetCode 11

Площадь краёв: `(right-left) * min(height[left], height[right])`. После вычисления двигаем сторону меньшей высоты.

![[90 Вложения/CurseHunter/7253/Кадры/013-container-water.jpg|720]]

### Почему это корректно?

Если левая высота меньше или равна правой, любая пара с тем же `left` и более узким диапазоном имеет ширину меньше, а высоту не выше `height[left]`. Она не улучшит текущую площадь. Поэтому текущий `left` можно отбросить. Симметрично для правой стороны.

Time `O(n)`, space `O(1)`. Если высоты равны, можно сдвинуть любую сторону или обе: текущая меньшая высота одинакова, пары с каждым из этих краёв и меньшей шириной не дадут больше.
