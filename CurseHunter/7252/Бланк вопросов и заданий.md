## Ментальная модель

Хеш-таблица полезна, когда во время линейного прохода нужно быстро отвечать на запрос вида «что уже известно про этот ключ?». Значением может быть индекс, частота, группа, соответствующий символ или промежуточное состояние.

![[90 Вложения/CurseHunter/7252/Кадры/001-hash-use-cases.jpg|720]]

В задачах курса встречаются четыре формы:

1. `ключ → индекс` — найти ранее просмотренное дополнение в Two Sum;
2. `символ → символ` — проверить взаимно-однозначное соответствие;
3. `символ → частота` — сравнить мультимножества;
4. `канонический ключ → группа` — собрать эквивалентные элементы.

На собеседовании `O(1)` для lookup обычно означает expected/average complexity принятой модели hash table. Полный алгоритм всё равно может быть `O(n)`, потому что каждый элемент надо прочитать. Спецификация Go задаёт семантику `map`, но не обещает конкретную асимптотику реализации; это важно не выдавать за языковую гарантию.

## 1. Two Sum — LeetCode 1

### Вопрос

Дан неотсортированный массив и `target`. Вернуть индексы двух разных элементов, сумма которых равна `target`.

### Решение курса

Идём слева направо. Перед вставкой текущего `num` проверяем, есть ли `target-num` в `map[value]index`. Если есть, возвращаем сохранённый и текущий индексы; иначе запоминаем `num → idx`.

![[90 Вложения/CurseHunter/7252/Кадры/002-two-sum.jpg|720]]

Инвариант: перед обработкой `nums[idx]` таблица содержит элементы только из `nums[:idx]`. Поэтому найденный индекс всегда отличается от текущего. Именно порядок «сначала lookup, потом insert» не позволяет использовать один элемент дважды при `target == 2*num`.

Time — expected `O(n)`, space — `O(n)`. Если решения нет, курс возвращает `[-1, -1]`; официальный контракт гарантирует ровно одно решение, поэтому этот fallback не участвует в корректности на LeetCode.

### Follow-up

- Нужны все пары, а не одна: одного последнего индекса на значение уже недостаточно; надо хранить список индексов или аккуратно считать комбинации.
- Массив отсортирован: два указателя дают `O(n)` time и `O(1)` space.
- Нельзя менять вход и важен deterministic output: заранее определить, какую из нескольких допустимых пар возвращать.

## 2. Isomorphic Strings — LeetCode 205

### Вопрос

Можно ли заменить каждый символ `s` всегда одним и тем же символом так, чтобы получить `t`, причём два разных символа `s` не должны отображаться в один символ `t`?

Одной таблицы `s → t` мало: она проверяет функцию, но не injectivity. Для примера `s = "ab"`, `t = "cc"` прямое отображение непротиворечиво по каждому ключу, однако строки не изоморфны. Код курса ведёт две таблицы: `s → t` и `t → s`.

![[90 Вложения/CurseHunter/7252/Кадры/003-isomorphic-strings.jpg|720]]

Инвариант: после каждого шага обе таблицы описывают одну и ту же bijection для уже просмотренного prefix. Любое расхождение в одном из направлений немедленно даёт `false`.

Для фиксированного ASCII-алфавита time `O(n)`, space `O(1)`. В показанном Go-коде используются `[128]byte`, byte indexes и `0` как sentinel «соответствия ещё нет».

### Границы решения

- Для произвольного UTF-8 нужны `rune` и таблицы по code point, иначе сравниваются байты кодировки.
- Sentinel `0` конфликтует с реальным NUL byte. Более общий вариант хранит индексы последнего появления `idx+1` в массивах `[256]int` либо использует `map[rune]rune` вместе с отдельным признаком наличия.
- Равенство `len(s) == len(t)` в коде означает одинаковое число bytes, не Unicode code points.

Альтернативный interview-приём: для каждой позиции требовать равенство последних позиций появления символов в `s` и `t`. Тогда две таблицы хранят `symbol → lastIndex+1`, а bijection следует из одинакового pattern повторов.

## 3. Roman to Integer — LeetCode 13

### Вопрос

Преобразовать корректную римскую запись в целое число.

Курс сканирует справа налево. Последний символ сразу добавляется. Каждый следующий символ вычитается, если его значение меньше значения символа справа; иначе прибавляется.

![[90 Вложения/CurseHunter/7252/Кадры/004-roman-to-int.jpg|720]]

Для `MCMXCIV` проход справа налево даёт `V-I + C-X + M-C + M = 1994`. Причинный механизм такой: меньший символ непосредственно перед большим кодирует subtractive pair; остальные значения складываются.

Time `O(n)`, дополнительное space `O(1)`: таблица `[128]int` имеет фиксированный размер.

### Что спросить у интервьюера

- Вход гарантирован непустым? Показанный код обращается к `s[len(s)-1]`.
- Вход гарантирован канонической валидной римской записью? Решение вычисляет значение, но не валидирует grammar: например, не отвергает произвольные repetition или недопустимую subtractive pair.
- Если алфавит фиксирован семью ASCII-символами, массив проще и дешевле `map`; если набор правил динамический, таблица становится удобнее.

## 4. Valid Anagram — LeetCode 242

### Вопрос

Проверить, являются ли две строки анаграммами: совпадают ли у них частоты каждого символа.

После проверки длины один проход одновременно увеличивает `frequency[s[i]]` и уменьшает `frequency[t[i]]`. Затем все счётчики должны быть равны нулю.

![[90 Вложения/CurseHunter/7252/Кадры/005-valid-anagram.jpg|720]]

Инвариант после позиции `i`: каждый счётчик равен разности частот символа в просмотренных prefixes `s` и `t`. Нулевой итог по всему алфавиту эквивалентен равенству мультимножеств символов.

Для официального контракта с lowercase English letters: time `O(n)`, space `O(1)` из-за `[26]int`. Код с `s[i]-'a'` нельзя без изменений применять к uppercase, punctuation или UTF-8: возможен выход за границы массива либо подсчёт байтов вместо символов.

### Альтернативы

- `map[rune]int` поддерживает Unicode code points: expected `O(n)` time, `O(k)` space для `k` различных runes.
- Сортировка даёт `O(n log n)` time и обычно требует копии, зато не зависит от hash table; для Unicode сначала надо определить, сравниваются code points или grapheme clusters и нужна ли normalization.

## 5. Group Anagrams — LeetCode 49

### Вопрос

Сгруппировать строки, являющиеся анаграммами друг друга.

Нужна каноническая функция `key(s)`, одинаковая ровно для анаграмм. Курс считает 26 частот и сериализует их как `#count#count...`; затем выполняет `groups[key] = append(groups[key], s)`.

![[90 Вложения/CurseHunter/7252/Кадры/006-group-anagrams.jpg|720]]

Разделитель в ключе обязателен. Простая конкатенация чисел создаёт collision представлений: частоты `[1, 11]` и `[11, 1]` обе дали бы `"111"`. С `#` получаются разные строки `"#1#11"` и `"#11#1"`.

Если `N` — число строк, `S` — суммарное число символов, а алфавит фиксирован 26 букв, подсчёт занимает `O(S)`, а построение ключей — ещё `O(26N)` операций плюс запись decimal digits. В памяти находятся группы и один созданный string-key на обрабатываемую строку до объединения одинаковых ключей; точная оценка зависит от того, считаем ли возвращаемые группы output space.

### Trade-offs и edge cases

- Ключ из отсортированных символов проще обобщить на произвольный алфавит, но стоит `O(k log k)` на строку длины `k`.
- Частотный `[26]int` даёт `O(k+26)`, но жёстко привязан к `a-z`.
- Порядок обхода Go `map` не определён, поэтому порядок групп в результате курса нестабилен. Официальная задача принимает группы в любом порядке; production API должен либо это задокументировать, либо отсортировать output/хранить порядок первого появления.
- Пустая строка получает нулевой frequency key; несколько пустых строк попадают в одну группу.

## Сводка типичных ошибок

| Неверное предположение | Симптом | Причина | Исправление |
| --- | --- | --- | --- |
| В Two Sum можно вставить элемент до lookup | один индекс используется дважды | текущий элемент уже виден самому себе | сначала искать complement |
| Для изоморфизма достаточно `s → t` | `ab` и `cc` ошибочно проходят | не проверена injectivity | хранить оба направления или last-seen pattern |
| `[26]int` работает с любой строкой | panic или неверные частоты | индексирование рассчитано на ASCII `a-z` | уточнить alphabet, использовать `map[rune]int` |
| Любая конкатенация частот уникальна | разные vectors дают одинаковый ключ | нет границ между числами | добавлять delimiter или использовать comparable fixed array там, где язык это позволяет |
| Roman parser заодно валидирует запись | некорректная запись получает число | алгоритм только вычисляет сумму | отдельно проверять grammar и canonical form |

## Вопросы для самопроверки

1. Что именно лежит в `map`: факт наличия, индекс, count, группа или state?
2. Какой инвариант связывает таблицу с уже просмотренным prefix?
3. Почему lookup делается до insert или после него?
4. Требуется mapping или bijection?
5. Как строится канонический ключ и почему у него нет ложных collisions представления?
6. Какой alphabet обещает контракт: bytes, ASCII, runes или пользовательские grapheme clusters?
7. Входит ли output и память ключей в заявленную space complexity?
8. Нужен ли стабильный порядок результата при обходе `map`?
