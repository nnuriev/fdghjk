# Горутины и планировщик: теория и 6 задач курса

## Урок 75. Goroutine как управляемая runtime задача

![[90 Вложения/CurseHunter/6609/Кадры/075.jpg]]

Goroutine начинается с небольшого растущего stack, имеет runtime descriptor `G` и планируется Go runtime поверх OS threads. Когда stack перестаёт вмещать frames, runtime может выделить больший stack, скопировать содержимое и поправить известные ему pointers. Поэтому address, превращённый в `uintptr`, опасен: integer runtime не отслеживает.

Goroutine заканчивается при return её top-level function, `runtime.Goexit` или завершении process. Garbage collector не отменяет заблокированную goroutine только потому, что application больше не ждёт результат. Lifecycle и ownership подробно разобраны в [[60 Go/Goroutines и lifecycle|заметке о lifecycle goroutine]].

## Урок 76. GMP scheduler

![[90 Вложения/CurseHunter/6609/Кадры/076.jpg]]

Модель курса:

- `G` — runnable/blocked computation;
- `M` — OS thread, на котором исполняется Go code;
- `P` — scheduler resource с local run queue, необходимый `M` для выполнения Go code;
- `GOMAXPROCS` ограничивает число одновременно исполняющих Go code `P`, а не число goroutines или созданных OS threads.

Scheduler ищет работу в local run queue, периодически учитывает global queue, выполняет work stealing и забирает готовые network operations из netpoller. Blocking syscall может отделить `M` от `P`, чтобы другой thread продолжил Go work. Это implementation model, а не детерминированный порядок запуска. Полный механизм — в [[60 Go/Планировщик GMP|заметке о GMP scheduler]].

## Урок 77. Сколько goroutines в пустой программе

![[90 Вложения/CurseHunter/6609/Кадры/077.jpg]]

`runtime.NumGoroutine()` возвращает текущее число существующих goroutines. В коротком `main` часто видна только одна, но точное число зависит от imports, runtime activity, profiling и момента измерения. Константного ответа «в Go всегда N служебных goroutines» нет.

## Урок 78. Loop variable в goroutine

![[90 Вложения/CurseHunter/6609/Кадры/078.jpg]]

Для module с language version Go `1.22+` variable, объявленная в `for` через `:=`, имеет отдельный instance на каждой iteration. Пять closures печатают `0…4` в недетерминированном порядке. В language version до `1.22` один reused variable часто приводил к повтору финального значения; idiom был передавать `i` argument-ом или создавать `i := i`.

`time.Sleep` не является synchronization: main может закончить раньше либо тест стать flaky. Используют `WaitGroup`, channel или structured cancellation.

## Урок 79. `GOMAXPROCS`

![[90 Вложения/CurseHunter/6609/Кадры/079.jpg]]

В Go `1.23.4` default обычно следовал `runtime.NumCPU()`, если его не переопределили environment/call. `runtime.GOMAXPROCS(16)` возвращает старое значение и задаёт новый limit, даже если logical CPU меньше; лишние `P` не создают вычислительную мощность.

**Эволюция:** Go `1.25` на Linux учитывает cgroup CPU bandwidth limit при выборе default и периодически обновляет его при изменении доступных CPU/limit. Ручная установка через environment или `runtime.GOMAXPROCS` отключает соответствующую автоматику. Поэтому ответ курса требует версии и deployment context.

## Урок 80. Async preemption

![[90 Вложения/CurseHunter/6609/Кадры/080.jpg]]

Начиная с Go `1.14` goroutines asynchronously preemptible: tight loop без function calls больше не обязан монополизировать один `P` и задерживать GC. `runtime.Gosched` добровольно уступает execution, но не нужен как универсальная «точка справедливости».

**Коррекции к примеру курса:** бесконечный `Result += str` постоянно аллоцирует/копирует string и обращается к shared variable, поэтому это слабая демонстрация чистого tight loop и одновременно data race, если `Result` читается/пишется конкурентно. `GOMAXPROCS(1)` не устраняет race по memory model.

## Урок 81. Panic в любой goroutine

![[90 Вложения/CurseHunter/6609/Кадры/081.jpg]]

Unrecovered panic в handler goroutine завершает весь process после выполнения её defers. Server обязан поставить recovery boundary внутри каждой запускаемой goroutine, залогировать stack/context и закрыть connection. Но recovery не заменяет error handling и не гарантирует, что shared state после нарушенного invariant пригодно к продолжению.

В обоих server examples после ошибки `Accept` отсутствует `continue`: код может запустить handler с nil/invalid `conn`. После transient error нужно решить retry/backoff, после terminal — выйти; нельзя безусловно продолжать с результатом failed call.

## Урок 82. Можно ли recover panic дочерней goroutine

![[90 Вложения/CurseHunter/6609/Кадры/082.jpg]]

Нет. Deferred `recover` в parent goroutine не видит panic, возникший в child. Panic unwinding и defer stack локальны goroutine. Recovery ставят непосредственно вокруг body child goroutine; затем outcome передают через channel/error group.
