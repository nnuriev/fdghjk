# Сборщик мусора: теория и 2 задачи курса

## Урок 72. Concurrent tracing GC

![[90 Вложения/CurseHunter/6609/Кадры/072.jpg]]

Курс показывает tracing, tri-color abstraction и цикл `stack scan → concurrent mark → mark termination → sweep`. Roots включают globals и stacks; marker следует по pointer graph, а write barrier сохраняет invariant при одновременной работе mutator. Stop-the-world остаётся для коротких фаз перехода, хотя основная marking work concurrent.

GC освобождает **unreachable**, а не «ненужное по мнению программы». Поэтому retention через slice, map, goroutine, timer или global остаётся logical leak даже при исправном collector. Подробная причинная модель — в [[60 Go/Аллокации, GC и GC pressure|заметке об allocation и GC pressure]].

## Урок 73. Большие allocation

![[90 Вложения/CurseHunter/6609/Кадры/073.jpg]]

Цикл присваивает global `data` новый slice на `1 GiB`. При обычном GC предыдущий backing array после следующего присваивания становится unreachable и может быть собран. При `GOGC=off` он не собирается, поэтому процесс растёт до OOM.

Если bytes не touch-ить, OS может лишь зарезервировать virtual address space, и RSS будет заметно меньше requested size. Версия с записью по одной byte на page materializes memory и лучше демонстрирует resident growth. OOM остаётся fatal, а не recoverable panic.

## Урок 74. Memory ballast

![[90 Вложения/CurseHunter/6609/Кадры/074.jpg]]

Ballast — большой live object, исторически увеличивавший GC target при percentage-based `GOGC`, чтобы collector запускался реже. Цена — завышенный live heap, reserved address space и конфликт с реальным memory budget.

Начиная с Go `1.19` runtime поддерживает soft memory limit через `GOMEMLIMIT`/`debug.SetMemoryLimit`. Для container budget это прямой механизм; ballast не является универсальной современной рекомендацией. `GOGC` по-прежнему задаёт CPU/memory trade-off, а limit не гарантирует жёсткое удержание процесса ниже cgroup limit при external memory и экстремальных условиях.
