# Типы данных: 6 задач курса

## Опорная модель

Числовая задача на Go начинается с трёх вопросов: операнд — constant или variable, тип signed или unsigned, какая у него bit width. Для `unsafe` добавляется четвёртый: является ли значение всё ещё pointer с точки зрения GC.

## Урок 2. Переполнение целых чисел

![[90 Вложения/CurseHunter/6609/Кадры/002.jpg]]

### Задача курса

Что напечатает increment максимальных `int8` и `uint8`, а что произойдёт с `math.MaxInt8 + 1` при инициализации typed variable?

**Ответ:** runtime-значения переходят `127 → -128` и `255 → 0`. Integer overflow в Go не вызывает panic: unsigned arithmetic работает modulo `2^n`, а signed result детерминирован представлением типа. Но constant вычисляется точно и должен быть representable в целевом типе, поэтому `var x int8 = math.MaxInt8 + 1` не компилируется.

### Как проверять overflow

Курс показывает корректные проверки `Inc` и `Add` через границы `math.MaxInt`/`math.MinInt`, но его `Mul` ошибочен:

```go
if lhs > math.MaxInt/rhs || lhs < math.MinInt/rhs {
	return 0, ErrIntOverflow
}
```

При `lhs=2`, `rhs=-2` знак неравенства после деления на отрицательное число учтён неверно, и функция ложно сообщает overflow. Рабочая проверка обязана разбирать квадранты знаков и отдельно `MinInt * -1`:

```go
func Mul(a, b int) (int, error) {
	if a == 0 || b == 0 {
		return 0, nil
	}
	if (a == math.MinInt && b == -1) || (b == math.MinInt && a == -1) {
		return 0, ErrIntOverflow
	}
	switch {
	case a > 0 && b > 0 && a > math.MaxInt/b,
		a > 0 && b < 0 && b < math.MinInt/a,
		a < 0 && b > 0 && a < math.MinInt/b,
		a < 0 && b < 0 && a < math.MaxInt/b:
		return 0, ErrIntOverflow
	}
	return a * b, nil
}
```

**Интервью-применение:** нельзя переносить на Go правило C/C++ «signed overflow — undefined behavior». Компилятор Go не вправе считать `x < x+1` всегда истинным.

## Урок 3. Bitwise complement signed и unsigned

![[90 Вложения/CurseHunter/6609/Кадры/003.jpg]]

### Задача курса

```go
var x uint8 = 3
var y int8 = 3
fmt.Println(^x, ^y)
```

**Ответ:** `252 -4`. Для unsigned mask состоит из всех единиц соответствующей ширины: `11111111 XOR 00000011 = 11111100`. Для signed mask равен `-1`, а в two's complement `^3 == -4`.

**Типичная ошибка:** объяснить результат бесконечной строкой битов, забыв bit width `uint8`. Для `uint16(3)` результат уже `65532`.

## Урок 4. Системы счисления

![[90 Вложения/CurseHunter/6609/Кадры/004.jpg]]

### Задача курса

```go
sum := 100 + 010
fmt.Println(sum)
```

**Ответ:** `108`: literal с ведущим нулём интерпретируется как octal, поэтому `010 == 8`. В современном коде явная запись `0o10` читается лучше и не маскируется под decimal.

## Урок 5. Почему `uintptr` не является pointer

![[90 Вложения/CurseHunter/6609/Кадры/005.jpg]]

### Задача курса

Сохранить адрес как `uintptr`, вызвать `runtime.GC`, затем преобразовать обратно и записать значение. Надёжно ли это?

**Ответ:** нет. `uintptr` — integer, а не reference. Он не удерживает объект живым и не обновляется, если runtime перемещает stack при росте. Обратное преобразование допустимо лишь в перечисленных документацией `unsafe` patterns; арифметика pointer → `uintptr` → pointer обычно должна оставаться в одном expression и внутри исходного allocated object.

`runtime.KeepAlive` может продлить liveness исходного pointer, но не превращает произвольно сохранённый address в безопасную ссылку. Запуск с `-gcflags=-d=checkptr` помогает находить часть нарушений, но отсутствие сообщения ничего не доказывает.

## Урок 6. Endianness

![[90 Вложения/CurseHunter/6609/Кадры/006.jpg]]

### Задача курса

Как будут лежать bytes числа `int32(0x12345678)` на little-endian машине?

**Ответ:** по возрастающим адресам `78 56 34 12`. Проверка курса читает первый byte числа `0x0001`: `1` означает little endian, `0` — big endian.

**Практическое правило:** application code обычно не должен определять endianness через `unsafe`. Для protocol/file formats порядок bytes задаётся явно через `encoding/binary`; host order не должен протекать в wire format.

## Урок 7. Почему присваивание параметру-pointer не меняет pointer вызывающего кода

![[90 Вложения/CurseHunter/6609/Кадры/007.jpg]]

### Задача курса

```go
func process(p *int32) {
	v := int32(200)
	p = &v
}
```

Почему caller по-прежнему указывает на старое значение? Потому что Go передаёт значение pointer по значению: функция меняет локальную копию address. Чтобы заменить pointer caller, нужно вернуть новый `*int32` или передать `**int32` и выполнить `*p = &v`.

Локальная `v` не становится висячей ссылкой: если её address переживает frame, escape analysis размещает значение там, где liveness безопасен.
