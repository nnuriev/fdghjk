# Каналы: теория и 10 задач курса

## Урок 95. `hchan`, buffer и wait queues

![[90 Вложения/CurseHunter/6609/Кадры/095.jpg]]

Channel value ссылается на runtime object. В Go `1.23.4` `hchan` хранит circular buffer, send/receive indices, element metadata, closed state, queues blocked senders/receivers и lock. Unbuffered send rendezvous с receiver; buffered send может закончиться после записи в свободный slot. Layout — implementation detail, а communication/happens-before — контракт.

Основной state matrix разобран в [[60 Go/Буферизация, ownership и закрытие каналов|заметке о buffering и channel ownership]]. Закрывает channel producer/owner, когда новых values больше не будет; receivers обычно не закрывают input.

## Урок 96. `select`, `break`, `continue`

![[90 Вложения/CurseHunter/6609/Кадры/096.jpg]]

В shown loop value `2` выполняет `continue` внешнего `for`, value `3` — обычный `break`, который выходит только из `select`; затем `3` печатается. После закрытия channel loop получает zero/`ok=false` и возвращается. Вывод: `1 3 4`.

Чтобы выйти из surrounding loop, используют labeled break/return. Empty `select {}` блокируется навсегда; если другие goroutines продолжают быть runnable, это может быть намеренный keep-alive, но lifecycle обычно лучше выражать `Wait`/context.

## Урок 97. Non-blocking operations

![[90 Вложения/CurseHunter/6609/Кадры/097.jpg]]

`len(ch)>0` перед receive и `len(ch)<cap(ch)` перед send — TOCTOU: состояние меняется до operation, а unbuffered channel всегда имеет `len==cap==0`, хотя partner может быть готов. Правильная форма — `select` с `default`.

При non-blocking receive важно не потерять closed state:

```go
select {
case v, ok := <-ch:
	return v, ok, true // operation состоялась
default:
	return zero, false, false
}
```

Иначе zero value из closed channel ошибочно выглядит как обычные данные.

## Урок 98. Priority в `select`

![[90 Вложения/CurseHunter/6609/Кадры/098.jpg]]

Если ready несколько communications, выбирается один pseudo-random ready case; source order приоритета не задаёт. Дублирование одинакового receive case дважды примерно удваивает его шанс относительно одного другого case, но это не точный fairness/SLA contract и ухудшает код. Настоящий priority выражают staged selects/queues и отдельно решают starvation.

## Уроки 99 и 104. API перепутал названия видео

CourseHunter API называет урок 99 «data race с каналами», но его MP4 показывает задачу про последовательное вычисление. Урок 104 в API назван «последовательное выполнение», а фактически показывает assignment race. Ниже задачи разложены по содержимому видео, а не по ошибочным labels.

### Последовательное вычисление arguments

![[90 Вложения/CurseHunter/6609/Кадры/099.jpg]]

Function arguments вычисляются left-to-right. Поэтому `Process(<-FetchData1(), <-FetchData2())` сначала полностью ждёт первый two-second result и только затем вызывает вторую function: около 4 секунд. Для concurrency сначала запускают обе operations:

```go
ch1, ch2 := FetchData1(), FetchData2()
Process(<-ch1, <-ch2) // около 2 секунд
```

### Data race на channel variable

![[90 Вложения/CurseHunter/6609/Кадры/104.jpg]]

Concurrent sends/receives на одном channel runtime synchronizes. Но `buffer = make(chan int)` — обычная запись в shared variable; сто goroutines выполняют её с data race. «Channels thread-safe» не распространяется на variable, в которой лежит channel pointer.

## Урок 101. Goroutine leaks

![[90 Вложения/CurseHunter/6609/Кадры/101.jpg]]

Первый example range-ит input, который caller никогда не закрывает, поэтому worker остаётся blocked. Нужен owner, закрывающий channel, либо context/done path.

Во втором first-response-wins пять goroutines send в unbuffered channel, а caller принимает одно value; четыре losers блокируются навсегда. Buffer size 1 спасает только одного sender. Используют buffer на число producers, cancellation, non-blocking loser send или errgroup. Практические patterns — в [[60 Go/Goroutine и channel leaks|заметке о goroutine leaks]].

## Урок 102. Можно ли проверить, закрыт ли channel

![[90 Вложения/CurseHunter/6609/Кадры/102.jpg]]

Универсального безопасного `IsClosed` peek нет. `select { case <-ch: ... }` потребляет buffered value и объявляет обычный ready channel «закрытым». Receiver узнаёт closure через `v, ok := <-ch`, принимая, что operation потребляет data. Sender не может без race проверить «можно ли отправить»; owner protocol должен исключать send-after-close.

## Урок 103. Nil channel как выключенный case

![[90 Вложения/CurseHunter/6609/Кадры/103.jpg]]

После closure closed channel всегда ready. Если оставить его в `select`, case может spin и голодать другие. Receiver присваивает local variable `nil`, чтобы отключить case. Одного boolean flag недостаточно, если сам channel остаётся в select.

В отдельной задаче buffered channel сначала допускает send case (`1`), затем receive (`2`), код делает `ch=nil`, после чего остаётся только `default` (`3`): вывод детерминированно `1 2 3`.

## Урок 105. Close при blocked sender

![[90 Вложения/CurseHunter/6609/Кадры/105.jpg]]

Sender блокируется на unbuffered `ch <- 1`. Другой поток закрывает channel, затем receive получает zero/closed; blocked sender пробуждается panic `send on closed channel`. Close не «отменяет» senders безопасно. Правило владения: sender side координирует остановку producers, дожидается их и только затем закрывает data channel.
