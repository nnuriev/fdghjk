# Строки: теория и 10 задач курса

Базовая модель — в [[60 Go/Строки, байты, руны и UTF-8|заметке о strings, bytes, runes и UTF-8]]. Главное для задач: string — immutable sequence of bytes; Unicode code point (`rune`) может занимать несколько UTF-8 bytes, а slicing работает по byte offsets.

## Урок 19. Внутреннее устройство string

![[90 Вложения/CurseHunter/6609/Кадры/019.jpg]]

Курс показывает runtime-модель из pointer на bytes и длины. Она объясняет дешёвое копирование header и retention подстрокой, но не является обещанием ABI для самодельных `unsafe` headers. У string нет `cap`.

## Урок 20. Что возвращает `len`

![[90 Вложения/CurseHunter/6609/Кадры/020.jpg]]

### Задача курса

Для `"hello"` ответ `5`, для `"汉hello"` — `8`: китайский rune кодируется тремя UTF-8 bytes. Число runes даёт `utf8.RuneCountInString`, но и оно считает invalid byte sequence как `RuneError` ширины один byte.

## Урок 21. Индекс при `range string`

![[90 Вложения/CurseHunter/6609/Кадры/021.jpg]]

`for i, r := range s` декодирует UTF-8, а `i` — byte offset начала rune, не его порядковый номер. Цикл `for i := 0; i < len(s); i++` читает raw bytes и может печатать части multi-byte rune.

**Коррекция:** комментарий курса `range []rune(text)` не является точным эквивалентом. После `[]rune` индекс уже rune index, а conversion обычно создаёт новый slice; `range string` allocation для slice не требует.

## Урок 22. String против `[]byte` при обходе

![[90 Вложения/CurseHunter/6609/Кадры/022.jpg]]

Benchmark курса показывает более дешёвый обход ASCII как bytes: `range string` всё равно выполняет UTF-8 decoding. Но выбор обязан идти от semantics: если нужны code points, byte loop неверен; если protocol гарантирует ASCII/opaque bytes, декодирование лишнее. Нельзя экстраполировать цифры одного Go `1.23.4` benchmark на другой CPU и input distribution.

## Урок 23. Подстрока на byte boundary

![[90 Вложения/CurseHunter/6609/Кадры/023.jpg]]

```go
s := "字Hello"
fmt.Println(s[:6])
```

Первый rune занимает 3 bytes, затем берутся `Hel`, поэтому результат `字Hel`. Граница внутри rune дала бы syntactically valid string со invalid UTF-8. Go разрешает произвольные bytes; валидность проверяется отдельно через `utf8.ValidString`.

## Урок 24. Способы конкатенации

![[90 Вложения/CurseHunter/6609/Кадры/024.jpg]]

Курс сравнивает `fmt.Sprintf`, `+` и `strings.Join`. Для фиксированного малого числа strings оператор `+` часто проще и быстрее; `Sprintf` нужен форматированию, `Join` — готовой коллекции с delimiter, `Builder` — пошаговой сборке. Benchmark должен удерживать результат и фиксировать объём работы; иначе compiler elimination или рост output вместе с `b.N` искажает вывод.

## Урок 25. `strings.Builder`

![[90 Вложения/CurseHunter/6609/Кадры/025.jpg]]

`Builder` накапливает bytes и возвращает string без обязательной финальной copy. `Grow(n)` резервирует место ещё как минимум для `n` bytes, не задаёт абсолютную capacity и не сокращает buffer. Negative `n` должен panic.

**Коррекция к учебной реализации:** показанный `Grow` при меньшем аргументе обрезает `buffer`, а при отрицательном молча возвращается. Это не контракт standard `strings.Builder`.

## Урок 26. Можно ли копировать `strings.Builder`

![[90 Вложения/CurseHunter/6609/Кадры/026.jpg]]

Zero-value Builder можно копировать до первого use. После непустого use копирование запрещено: copies могут alias один buffer, а методы standard library обнаруживают это и panic. Чтобы получить независимую сборку, создают новый Builder и пишут в него `old.String()`; при необходимости заранее вызывают `Grow`.

## Урок 27. String ↔ `[]byte`

![[90 Вложения/CurseHunter/6609/Кадры/027.jpg]]

Обычные `string(bytes)` и `[]byte(s)` семантически создают независимое mutable/immutable представление. Compiler может убрать copy в ограниченных non-escaping patterns, но на это нельзя строить ownership.

`unsafe.String(unsafe.SliceData(b), len(b))` и `unsafe.Slice(unsafe.StringData(s), len(s))` создают alias без copy. Тогда caller обязан гарантировать lifetime, а bytes backing string нельзя менять. Выигрыш допустим только после profiling и с узким ownership contract.

## Урок 28. «Изменение immutable string»

![[90 Вложения/CurseHunter/6609/Кадры/028.jpg]]

Если string создан zero-copy поверх mutable `[]byte`, изменение slice наблюдается через string — уже нарушение обычного immutable contract. Попытка записать в storage string literal через `unsafe` может упасть, потому что данные могут лежать в read-only memory. Это нарушение правил `unsafe`, а не техника мутации.

## Урок 29. Retention большой строки

![[90 Вложения/CurseHunter/6609/Кадры/029.jpg]]

Маленькая substring может удерживать весь исходный byte storage. Если исходная строка большая и результат живёт долго, используют `strings.Clone(sub)`.

В `FindData` курса есть ещё один дефект: после совпадения `data[i+2:i+22]` не проверяет, что `i+22 <= len(data)`, и может panic у конца строки. Сначала проверяют boundary, затем при нужном lifetime клонируют результат.
