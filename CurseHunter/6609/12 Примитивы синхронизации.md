# Примитивы синхронизации: 11 задач курса

## Урок 84. Data race и race condition

![[90 Вложения/CurseHunter/6609/Кадры/084.jpg]]

### Задача курса

Одна goroutine присваивает `text`, другая печатает его, обе вызывают `Done`, а main делает `Wait`. Есть ли race?

**Ответ:** да. `Wait` упорядочивает завершение workers с возвратом из `Wait`, но между read и write двух workers нет happens-before. Возможны empty/новое значение, а `-race` сообщает конфликт. Нужны channel, mutex или запуск reader после `Wait`.

Data race — конфликтующие concurrent accesses без synchronization, хотя бы один write. Race condition шире: output зависит от scheduling и может быть логически неверным даже при отсутствии data race. Модель — в [[60 Go/Модель памяти Go и happens-before|заметке о memory model]].

## Урок 85. Deadlock

![[90 Вложения/CurseHunter/6609/Кадры/085.jpg]]

Две группы goroutines захватывают `mutex1→mutex2` и `mutex2→mutex1`. Circular wait устраняют единым global lock order, одним mutex, `TryLock` с корректным rollback/backoff либо redesign без одновременного владения двумя locks.

Вторая версия запускает бесконечный ticker goroutine. Runtime может не вывести `all goroutines are asleep`, потому что timer продолжает становиться runnable, но полезная работа всё равно навсегда зависла. Отсутствие runtime fatal message не доказывает liveness.

## Урок 86. «Livelock»

![[90 Вложения/CurseHunter/6609/Кадры/086.jpg]]

Каждая goroutine удерживает один mutex и бесконечно крутит `TryLock` второго. Состояние владения не меняется, поэтому это busy-wait deadlock/spin-deadlock, а не классический livelock. При livelock участники активно меняют состояние и уступают друг другу, но общий progress не происходит. Различие и диагностика — в [[60 Go/Data races, deadlocks и livelocks|заметке о races и liveness failures]].

## Урок 87. Local mutex

![[90 Вложения/CurseHunter/6609/Кадры/087.jpg]]

Создание `mutex := sync.Mutex{}` внутри каждого `inc` не защищает global `value`: goroutines блокируют разные objects. Lock должен быть shared вместе с защищаемым state.

Во втором example mutex общий, но защищаемое `value` local каждой goroutine — synchronization избыточна. Кроме того, код делает `wg.Add(1000)`, запускает только 10 goroutines и поэтому deadlock на `Wait`: отдельная ошибка bookkeeping.

## Урок 88. Granularity critical section

![[90 Вложения/CurseHunter/6609/Кадры/088.jpg]]

Печать под lock удерживает mutex во время потенциально медленного I/O. Обычно под lock только читают/copy `item`, затем unlock и выполняют `fmt.Println`. Но слишком узкая critical section опасна, если между copy и действием требуется invariant. Granularity выбирают по atomicity requirement, затем уменьшают contention.

## Урок 89. Опасные операции с Mutex/RWMutex

![[90 Вложения/CurseHunter/6609/Кадры/089.jpg]]

- Повторный `Lock` одного `Mutex` без `Unlock` deadlock: mutex не reentrant.
- `Unlock` unlocked mutex и mismatched `RWMutex.Unlock/RUnlock` — unrecoverable runtime fatal errors.
- Go mutex не имеет goroutine ownership: одна goroutine технически может unlock lock другой. Это допустимо contract-ом, но требует явного handoff protocol.
- Upgrade `RLock→Lock` и попытка `RLock` при удерживаемом write lock той же goroutine deadlock.

Полные правила — в [[60 Go/Mutex, RWMutex и примитивы координации sync|заметке о sync primitives]].

## Урок 90. Copying synchronization primitives

![[90 Вложения/CurseHunter/6609/Кадры/090.jpg]]

`done(wg sync.WaitGroup)` копирует internal state после `Add`. `Done` уменьшает copy, а original остаётся равен 1 и `Wait` висит. `Mutex`, `RWMutex`, `Once`, `Cond`, `WaitGroup` после начала использования не копируют; передают pointer или держат в non-copying owner. `go vet` analyzer `copylocks` ловит многие случаи.

Начиная с Go `1.25`, `WaitGroup.Go(f)` объединяет Add и запуск goroutine для common pattern, но не отменяет правила liveness и prohibition на copying.

## Урок 91. CAS и публикация

![[90 Вложения/CurseHunter/6609/Кадры/091.jpg]]

`if !initialized.Load() { initialized.Store(true); data=make(...) }` допускает несколько initializers и data race. Замена на CAS флага тоже недостаточна, если winner выставляет `true` **до** заполнения `data`: readers увидят published flag и неготовое состояние. Для once-only initialization нужен `sync.Once`, чей return из `f` synchronizes-before return последующих `Do`.

`count.Add(1); count.CompareAndSwap(100,0)` — две atomic operations, но не одна transaction. Между ними другой increment может изменить count, и reset пропустится. Нужны CAS loop и точная semantic: сбрасывать каждую сотню событий, saturate, modulo или отдавать ticket. Atomics разобраны в [[60 Go/Пакет sync-atomic|заметке о sync/atomic]].

## Урок 92. `sync.Cond`

![[90 Вложения/CurseHunter/6609/Кадры/092.jpg]]

`Wait` вызывают при удерживаемом `cond.L`; он atomically unlocks, suspends и перед возвратом снова locks. Predicate проверяют в loop:

```go
cond.L.Lock()
for !ready {
	cond.Wait()
}
cond.L.Unlock()
```

`Signal` до появления waiter не накапливается, поэтому `Signal(); Wait()` может зависнуть. Cond не переносит состояние; состояние хранится в predicate под тем же lock.

## Урок 93. Синхронизация структур данных

![[90 Вложения/CurseHunter/6609/Кадры/093.jpg]]

Курс показывает четыре независимые ошибки:

1. `Buffer.Data` возвращает internal slice после unlock. Caller мутирует/читает backing array без lock; возвращают clone или выполняют callback/snapshot под lock.
2. `Cache.Get` держит mutex и вызывает `Size`, повторно locking тот же mutex: self-deadlock. Делают unexported helper, предполагающий уже удерживаемый lock.
3. Embedded `sync.Mutex` продвигает public `Unlock`, и caller ломает invariant. Mutex лучше хранить named unexported field.
4. `Stack` использует value receivers, поэтому копирует mutex и slice header; `Push` не меняет original header. Проверка `len < 0` невозможна, checks находятся вне lock, `Top`+`Pop` не atomic. Нужны pointer receivers, `len == 0` под lock и единая operation `Pop() (T, bool)`.

## Урок 94. False sharing

![[90 Вложения/CurseHunter/6609/Кадры/094.jpg]]

False sharing возникает, когда независимые frequently-written variables попадают в одну cache line: coherence invalidations связывают cores, хотя data logically independent. Помогают sharding per worker/P и padding/alignment, но размер line platform-specific, а padding расходует memory.

**Коррекция:** benchmark курса не доказывает false sharing. Функция называется `BenchmarkAtomicCounter`, но создаёт `MutexCounter`; нет сравнительных benchmarks; total work растёт вместе с `NumCPU`; shards расположены рядом без padding. Нужны fixed total operations, отдельные variants, `RunParallel`, проверка результата и hardware counters/profile.
