# Интерфейсы: теория и 7 задач курса

## Урок 44. Dynamic type и dynamic value

![[90 Вложения/CurseHunter/6609/Кадры/044.jpg]]

Interface value описывается парой **dynamic type + dynamic value**. Nil interface — только пара без обоих компонентов. Runtime Go `1.23.4` различает empty interface и interface с methods и использует type metadata/dispatch table, но конкретные `eface`/`iface` fields не входят в language ABI. Для ответа важна модель из [[60 Go/Интерфейсы и неявная реализация|заметки об интерфейсах]], а не запоминание internal struct.

## Урок 45. Сравнение interface values

![[90 Вложения/CurseHunter/6609/Кадры/045.jpg]]

Два interface values равны, когда равны их dynamic types и dynamic values. Если одинаковый dynamic type не comparable, comparison panic. Поэтому `any(3) == any(3)` даёт `true`, а сравнение двух `any`, содержащих `[]int`, — panic. Нельзя считать, что boxing делает slice comparable.

## Урок 46. Присваивание интерфейсов

![[90 Вложения/CurseHunter/6609/Кадры/046.jpg]]

`SyncStorage` с `Close` и `Sync` assignable в `BaseStorage`, требующий только `Close`: любой dynamic value `SyncStorage` удовлетворяет меньшему method set. Обратное присваивание запрещено статически, потому что произвольный `BaseStorage` может не иметь `Sync`.

Type assertion из `SyncStorage` в `interface{ Close() }` здесь избыточна: assignability уже известна compiler.

## Урок 47. Копирование interface

![[90 Вложения/CurseHunter/6609/Кадры/047.jpg]]

```go
value := 100
var x any = value
value = 200
fmt.Println(x) // 100
```

В interface скопировано значение `100`, а не reference на variable `value`. Однако boxing slice, map или pointer копирует их value semantics: header/pointer может вести к тем же данным. «Interface всегда копирует объект целиком» — неверное обобщение.

## Урок 48. Почему `[]int` не является `[]any`

![[90 Вложения/CurseHunter/6609/Кадры/048.jpg]]

Slice types имеют разный element layout: `int` и interface value представлены по-разному. Ковариантность сделала бы mutation небезопасной. Нужна element-wise boxing:

```go
dst := make([]any, len(src))
for i, v := range src {
	dst[i] = v
}
```

Это `O(n)` work и может влиять на allocation/escape.

## Урок 49. Typed nil

![[90 Вложения/CurseHunter/6609/Кадры/049.jpg]]

```go
var p *MyError
var err error = p
fmt.Println(err == nil) // false
```

Interface содержит dynamic type `*MyError`, хотя dynamic value nil. Поэтому `ReadFile`, который возвращает typed nil `*os.PathError` как `error`, сообщает вызывающему non-nil error. Если ошибки нет, возвращают literal `nil`, а typed pointer создают только для настоящей ошибки. Больше edge cases — в [[60 Go/Nil, type assertions и type switches в интерфейсах|заметке о nil и assertions]].

## Урок 50. Type assertion

![[90 Вложения/CurseHunter/6609/Кадры/050.jpg]]

- `v, ok := x.(T)` возвращает zero value и `false` при mismatch.
- `v := x.(T)` panic при mismatch.
- Assertion проверяет dynamic type, поэтому interface с меньшим static method set можно проверить на дополнительные methods.
- Assertion из `foobarer` в `fooer` успешна, если dynamic type реализует `foo`.

## Урок 51. Type switch

![[90 Вложения/CurseHunter/6609/Кадры/051.jpg]]

Cases проверяются сверху вниз, срабатывает первый matching type. `*thing`, реализующий `foo` и `bar`, уже совпадает с первым `case fooer`; последующий `case foobarer` недостижим для этого value. Более специфические cases ставят раньше, если порядок выражает намерение.
