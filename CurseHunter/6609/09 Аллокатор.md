# Аллокатор: теория и 3 задачи курса

## Урок 68. Size classes и уровни allocator

![[90 Вложения/CurseHunter/6609/Кадры/068.jpg]]

Курс объясняет hierarchy `mcache → mcentral → mheap`: per-P cache обслуживает frequent small allocations без global contention, central lists пополняют spans нужного size class, heap управляет spans/pages и large objects. Это runtime model Go `1.23.4`, а не language contract.

В benchmark две 33-byte conversions округляются до 48-byte classes, а 66-byte result — до 80, поэтому автор получает `176 B/op` и `3 allocs/op`. Конкретные classes, compiler copies и `AllocsPerOp` зависят от версии, architecture, escape и optimization. Главный вывод: internal fragmentation — цена быстрого allocation по классам, но цифры перепроверяют benchmark-ом.

## Урок 69. Stack или heap

![[90 Вложения/CurseHunter/6609/Кадры/069.jpg]]

### Задача курса

Определить размещение local value, returned pointer, address argument и `new`.

**Ответ:** синтаксис `new` и взятие address сами по себе не решают. Compiler помещает value в heap, если оно escapes lifetime/analysis возможностей stack frame; inlining и dead-code elimination меняют результат. Returned pointer часто вызывает escape, но compiler может scalar-replace или inline так, что физической heap allocation нет.

В курсе `createPointer()` с отключённым inlining демонстрирует heap, а локальный `new(int)`, не escaping, — stack. Проверять нужно `go build -gcflags=-m=2` и production-like benchmark. Механизм разобран в [[60 Go/Стеки и escape analysis|заметке об escape analysis]].

## Урок 70. Interface allocations

![[90 Вложения/CurseHunter/6609/Кадры/070.jpg]]

Boxing value в interface не гарантирует heap allocation. Если interface не escapes и representation доступно compiler, данные могут остаться на stack или быть встроены/устранены. Передача в `fmt.Println` меняет escape picture. Поэтому ответ «interface всегда аллоцирует» неверен; измеряют конкретный call path.

## Урок 71. Array и slice thresholds

![[90 Вложения/CurseHunter/6609/Кадры/071.jpg]]

Курс показывает пороги Go `1.23.4`: большой local array около `10 MiB` и `make([]byte, 64 KiB+1)` уходят в heap в данной сборке. Это compiler heuristics, не гарантии Go. Размер stack каждой goroutine динамически растёт, но compiler может заранее вынести слишком крупный object, а constant-size slice рассматривается иначе, чем dynamic-size.

**Интервью-применение:** нельзя использовать эти числа как design limit. Большой object в hot path оценивают по escape report, benchmark, GC pressure и peak concurrency.
