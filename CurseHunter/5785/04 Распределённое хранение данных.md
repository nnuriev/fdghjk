# Распределённое хранение данных

## Backup и replication

Replica повышает availability и/или read capacity, но быстро размножает accidental delete, corruption и bad migration. Backup — независимая recovery point с retention; его ценность доказывает restore drill, а не факт создания файла.

На интервью определить:

- RPO: сколько подтверждённых данных допустимо потерять;
- RTO: сколько длится восстановление;
- full/incremental/PITR и WAL retention;
- encryption, access separation и geographic failure domain;
- регулярную проверку целостности и восстановления.

## Репликация

### Single-leader и multi-leader

Single-leader упрощает order writes, но требует failover, fencing старого leader и политики чтения. Multi-leader снижает latency локальных writes и переживает disconnection, но переносит сложность в conflict detection/resolution. Last-write-wins теряет обновления при clock skew и редко выражает бизнес-инвариант.

### Sync, async и semi-sync

Синхронное подтверждение до replica уменьшает RPO, но связывает write availability/latency с quorum. Асинхронное быстрее и доступнее при lagging replica, но acknowledged writes могут исчезнуть при failover. Semi-sync — конкретный контракт ack, а не магическое среднее.

### Quorum

![[90 Вложения/CurseHunter/5785/Кадры/022-quorum-overview.jpg|720]]

Пересечение `W + R > N` означает, что read и write quorum имеют общий replica. Для возврата последнего значения нужны version/timestamp semantics, корректная обработка concurrent writes, read repair или coordinator logic. Linearizability этим не доказывается.

### Client-centric consistency

- read-your-writes: после собственного write route к leader или replica, догнавшей version token;
- monotonic reads: session не должна переходить на более старую replica;
- monotonic writes: writes пользователя применяются в causal order;
- consistent prefix: reader не видит более позднее событие без его причин.

![[90 Вложения/CurseHunter/5785/Кадры/022-read-own-writes.jpg|720]]

Таймер «минуту читать с master» — эвристика; version/session token точнее и не зависит от фактического lag.

## CAP без лозунга

При network partition узлы не могут одновременно гарантировать linearizable consistency и ответ каждому запросу. Практическое решение принимается per operation: платеж может отклонить write без quorum, а лента показать stale copy. Latency/failure detection и PACELC-подобные trade-offs остаются и без полного partition.

Интервью-задача: разорвать связь между двумя DC и пройти write/read/failover. Кто принимает write? Как предотвращается split brain? Что происходит с acknowledged data после heal?

## Partitioning и sharding

Partitioning делит logical table на physical pieces внутри выбранной системы; sharding распределяет данные и ответственность между независимыми nodes/databases. Термины в продуктах пересекаются, поэтому важнее routing и failure domain.

### Как выбрать shard key?

Хороший key:

- присутствует в главных запросах;
- даёт высокую cardinality и равномерную нагрузку;
- локализует transaction/aggregation;
- стабилен и не меняет владельца часто;
- допускает controlled rebalance.

Range sharding поддерживает range scan, но создаёт moving hotspot. Hash равномернее, но уничтожает locality. Consistent hashing уменьшает число перемещаемых key при смене membership; virtual nodes помогают балансировать неодинаковые machines.

![[90 Вложения/CurseHunter/5785/Кадры/025-consistent-hashing.jpg|720]]

### Rebalance как state machine

1. зафиксировать новую topology version;
2. копировать snapshot;
3. догнать delta/log;
4. временно читать/писать по migration protocol;
5. атомарно переключить routing;
6. проверить checksum и удалить старую копию после grace period.

Наивный dual write во время migration создаёт divergence. Нужны idempotent replay, ownership epoch и защита от stale router.

### Hot key

Разбить key искусственным suffix, кэшировать read, локально агрегировать writes, выделить отдельный shard или изменить продуктовый access pattern. Случайное распределение writes усложняет read fan-out и ordering; это осознанный trade-off.

## CDC и альтернативные read models

Change Data Capture читает database change log и публикует ordered changes в downstream systems. Это снимает application dual write, но добавляет snapshot bootstrap, schema evolution, duplicates, lag и offset management.

![[90 Вложения/CurseHunter/5785/Кадры/026-cdc.jpg|720]]

Search/analytics index через CDC — derived view, который можно перестроить. Source of truth остаётся transactional store; query path должен понимать lag и иметь reconciliation.

## CDN

CDN переносит bytes ближе к пользователю и снижает origin traffic. Вопросы интервьюера: cache key, TTL, signed URL, purge, regional miss, origin shielding, range requests, immutable object versioning и защита private content. Metadata consistency и object delivery — разные paths.

## Практика Coin Keeper

Предложение курса: PostgreSQL, asynchronous replication factor 2 и sharding по `user_id`. Защитить решение можно только после уточнений:

- RF=2 переживает один crash, но maintenance во время failure оставляет нулевой запас;
- async replication допускает потерю acknowledged writes при failover;
- balance/report требуют понятного source of truth и isolation;
- cross-user transfer пересекает shards;
- backup/PITR и restore остаются обязательны;
- shard count выбирают по load/capacity и migration cost, не только по числу дисков.
