# Архитектура, устойчивость и консенсус

## Monolith, SOA и microservices

Monolith сохраняет in-process calls и локальные транзакции; небольшой команде его проще развёртывать и отлаживать. Microservices дают независимые границы владения, развёртывания и масштабирования, но превращают function call в ненадёжное сетевое взаимодействие и требуют distributed observability, compatibility, data ownership и automation.

Хорошая граница следует business capability и ownership данных. «Один сервис на таблицу» создаёт chatty calls и distributed monolith. Shared database облегчает joins, но связывает release и schema; database-per-service усиливает autonomy ценой saga/CDC/read models.

## Event-driven взаимодействия

- event notification сообщает факт и оставляет consumer дочитать детали;
- event-carried state transfer несёт достаточно данных, снижая coupling на read path ценой duplicated state;
- event sourcing хранит ordered domain events как source of truth и требует versioning, replay и snapshot strategy.

![[90 Вложения/CurseHunter/5785/Кадры/029-event-notification.jpg|720]]

Не каждое сообщение — domain event. Command просит выполнить действие и может быть отклонён; event утверждает уже произошедший факт.

## Resilience patterns

### Timeout и retry

Timeout обязан учитывать end-to-end deadline и включать connection/DNS/TLS semantics. Retry полезен для transient failure только если request idempotent и остаётся budget. Exponential backoff ограничивает частоту, jitter рассинхронизирует клиентов, token bucket/limit ограничивает общий retry amplification.

Ошибка: retry на каждом из пяти уровней по три раза даёт до `3^5 = 243` downstream attempts. Обычно retry концентрируют в одном слое.

### Rate limiting, load shedding и backpressure

- rate limiting защищает policy/fairness до исчерпания ресурса;
- load shedding быстро отклоняет низкоприоритетную работу при saturation;
- backpressure сообщает upstream, что consumer не успевает, и ограничивает очередь;
- bulkhead разделяет pools/quotas, чтобы один tenant/dependency не съел всё.

### Circuit breaker

Closed пропускает запросы и измеряет failures; open fail-fast; half-open допускает ограниченные probes. Breaker без timeout, bounded concurrency и per-destination scope легко становится глобальным выключателем.

![[90 Вложения/CurseHunter/5785/Кадры/030-circuit-breaker.jpg|720]]

### Graceful degradation и fallback

Отключить recommendations, показать stale cache, перейти на read-only или снизить качество media. Fallback не должен нарушать security и money invariants; stale price или разрешение доступа хуже ошибки.

### DLQ

Dead-letter queue не кладбище. Нужны причина, payload/schema version, attempt history, alert, replay tool и idempotent consumer. Poison message нельзя бесконечно возвращать в основную очередь.

## Distributed workflow

### Saga

Saga разбивает workflow на локальные транзакции и compensating actions. Compensation — новое бизнес-действие, а не rollback времени: возврат может сам отказать, иметь fee и требовать audit.

- choreography снижает central coupling, но causal flow трудно наблюдать;
- orchestration делает state machine явной, но orchestrator становится важным компонентом;
- 2PC даёт атомарное решение участвующих resource managers, но блокировка/availability и operational support ограничивают применимость.

### Transactional outbox/inbox

Entity и outbox event записываются одной локальной транзакцией; publisher повторно доставляет событие. Consumer inbox/unique key дедуплицирует side effect.

![[90 Вложения/CurseHunter/5785/Кадры/031-outbox.jpg|720]]

Outbox даёт at-least-once publication, а не глобальный exactly-once. Порядок надо определить per aggregate/partition.

## Consensus, leader election и lock

Consensus — согласование replicated state при failures. Raft разделяет leader election, log replication и safety; majority из `2f+1` nodes позволяет продолжать после crash `f` nodes. Consensus не делает application transaction автоматически serializable.

### Lease против lock

Lease истекает по времени, но старый holder может продолжить работу после pause/network delay. Поэтому resource принимает monotonically increasing fencing token и отвергает операции со старым epoch. Без fencing distributed lock защищает только намерение участников, не сам resource.

### Что спросит интервьюер?

1. Что произойдёт, если leader принял write, но умер до replication/response?
2. Как новый leader отличает committed entry от uncommitted?
3. Почему два nodes недостаточны для автоматического safe failover без внешнего witness?
4. Как меняется membership и где quorum intersection?
5. Что происходит с availability при потере majority?

## Развёртывание

- rolling: постепенно меняет instances, требует backward-compatible protocol/schema;
- blue-green: быстрое переключение и rollback ценой двойной capacity;
- canary: ограничивает blast radius, но требует comparable traffic, metrics и automated abort;
- feature toggle разделяет deploy/release, но stale flags создают permanent complexity.

Database migration должна быть expand → migrate/backfill → switch reads/writes → contract. Одновременное destructive schema change с rolling deploy ломает старые instances.
