# Горутины в цикле

## TL;DR

В исходнике три независимые проблемы: lifecycle goroutines не связан с `main`, общий `max` читается и пишется с data race, а ответ про capture `i` зависит от module language version. Для Go 1.21 loop variable, объявленная в `for`, была общей; для Go 1.22+ каждая iteration получает новую variable. Это исправление не добавляет ожидание и не защищает `max`.

Рабочий вариант ниже явно создаёт iteration-local copy, ждёт все задачи через `WaitGroup` и держит сравнение вместе с записью под одним `Mutex`.

## Условие и контракт

Исходный код запускает goroutine для каждого `i` от `1000` до `1` и пытается найти максимальное чётное число. Нужно провести code review и сделать результат детерминированным.

Контракт решения:

- для положительного `limit` вернуть максимальное положительное чётное число `<= limit`;
- если такого числа нет, вернуть `(0, false)`;
- завершить все запущенные goroutines до возврата;
- не иметь data races при `go test -race`;
- одинаково корректно захватывать `i` при language versions до Go 1.22 и начиная с Go 1.22.

### Неоднозначности исходника

- Без synchronization исходная программа не имеет стабильного числового результата: `main` может напечатать `0` до выполнения workers, а race на `max` делает остальные исходы недоказуемыми.
- Замечание исходника «closure обязательно увидит финальное `i`» описывает старую семантику и всё равно слишком категорично: при race конкретное interleaving не гарантировалось.
- Параллелить тысячу тривиальных сравнений невыгодно. Concurrency сохранена как предмет задачи, а не как performance-рекомендация.

## Ментальная модель

Корректность распадается на три оси:

1. **Capture:** какое значение `i` доступно closure.
2. **Lifetime:** кто доказывает, что все goroutines завершились до чтения результата.
3. **Atomicity invariant:** сравнение `i > max` и последующая запись должны быть одной критической секцией.

Исправление только одной оси не чинит две другие. `WaitGroup` задаёт completion barrier, но не mutual exclusion во время работы. `Mutex` защищает `max`, но не удерживает `main` от раннего завершения. Новая loop semantics Go 1.22 убирает типичный capture bug, но не создаёт happens-before для `max`.

## Concurrency invariants

- Каждая goroutine читает собственную iteration variable `i`.
- `max` и `found` образуют один инвариант и всегда читаются/меняются под одним `mu`.
- `Wait` возвращается только после каждой учитываемой функции; `Done` внутри `WaitGroup.Go` synchronizes-before разблокированный `Wait`.
- `WaitGroup` не копируется после первого использования.
- Function, переданная `WaitGroup.Go`, не должна panic; это часть публичного контракта Go 1.26.5.

## Код Go 1.26.5

`loop.go`:

```go
package loop

import "sync"

// MaxEven returns the largest positive even number not greater than limit.
func MaxEven(limit int) (int, bool) {
	var (
		mu    sync.Mutex
		wg    sync.WaitGroup
		max   int
		found bool
	)

	for i := limit; i > 0; i-- {
		i := i // Safe for modules with language versions both before and after Go 1.22.
		wg.Go(func() {
			if i%2 != 0 {
				return
			}

			mu.Lock()
			if !found || i > max {
				max = i
				found = true
			}
			mu.Unlock()
		})
	}

	wg.Wait()
	return max, found
}
```

Явное `i := i` избыточно для variables, объявленных циклом в module language version Go 1.22+, но сохраняет правильную семантику и в старом модуле. Это локальная compatibility-мера, а не универсальная защита любого captured state.

## Tests

`loop_test.go`:

```go
package loop

import "testing"

func TestMaxEven(t *testing.T) {
	for run := 0; run < 20; run++ {
		got, ok := MaxEven(1000)
		if !ok || got != 1000 {
			t.Fatalf("run %d: MaxEven(1000) = (%d, %v), want (1000, true)", run, got, ok)
		}
	}
}

func TestMaxEvenWithoutCandidate(t *testing.T) {
	if got, ok := MaxEven(1); ok || got != 0 {
		t.Fatalf("MaxEven(1) = (%d, %v), want (0, false)", got, ok)
	}
}
```

## Проверка результата

Tests прошли `go test`, `go vet` и `go test -race` на Go 1.26.5 `darwin/arm64`, проверено 2026-07-18.

Для отдельной программы без `i := i` toolchain Go 1.26.5 дал:

```text
go.mod: go 1.21 -> 3 3 3
go.mod: go 1.26 -> 0 1 2
```

Оба временных модуля запускались одним toolchain Go 1.26.5. Различие результата задаёт директива `go` в `go.mod`, то есть language version модуля; одной версии установленного toolchain для вывода недостаточно.

Пример специально сохранял closures и вызывал их после цикла, поэтому показывает language semantics без scheduler race.

| --- | --- | --- | --- | --- |
| Module language version Go 1.21 и ниже | Variables, объявленные `for`/`range`, переиспользовались iterations | — | Closure мог читать общую variable; нужна iteration-local copy или параметр | [Go 1.22 Release Notes](https://go.dev/doc/go1.22) |
| Module language version Go 1.22+ | — | Каждая iteration создаёт новые declared variables | Исходный capture `i` больше не вызывает эту ошибку; race на `max` и ранний exit остаются | [For statements](https://go.dev/ref/spec#For_statements) |

## Сложность и ресурсы

Время полезной работы — `O(limit)`, число goroutines и дополнительная память — `O(limit)`. Critical section короткая, но все чётные candidates сериализуются на одном mutex. Для `limit=1000` это демонстрация примитивов; для большой CPU-задачи тысячи goroutines и общий lock добавляют overhead без ускорения.

## Trade-offs и альтернативы

- Последовательный цикл остаётся лучшим production-решением этой конкретной задачи: `O(n)` без scheduler и lock overhead.
- Atomic `Store` только на строке присваивания неверен: проверка `i > max` остаётся отдельной read-modify-write гонкой. Корректная atomic-версия использует CAS loop, как объяснено в [[60 Go/Пакет sync-atomic|заметке о `sync/atomic`]].
- Для дорогой обработки inputs workers могут вычислять local maxima, а один reducer — общий максимум. Это уменьшает contention и позволяет bounded concurrency через [[60 Go/Worker pool, fan-in, fan-out и bounded concurrency|worker pool]].
- `defer wg.Done()` в ручном `Add`/`Done` варианте защищает от ранних `return`; выбирать его по мифу о «запредельной цене defer» не стоит. Но `WaitGroup.Go` в Go 1.26.5 уже инкапсулирует нормальное завершение.

## Типичные ошибки

- **Исправить только capture → иногда печатается `0` → `main` не ждёт workers → добавить completion barrier.**
- **Добавить `WaitGroup`, но не lock → race detector сообщает конфликт на `max` → completion не сериализует параллельные accesses → защитить весь compare-and-update.**
- **Сделать atomic только присваивание → теряется более крупный candidate → сравнение и запись не образуют одну atomic operation → CAS loop или mutex.**
- **Всегда повторять ответ про финальный `i` → неверное объяснение для Go 1.22+ → перепутана историческая и текущая language semantics → назвать module language version и форму объявления variable.**
- **Вызвать `Add(1)` внутри goroutine → `Wait` может увидеть zero раньше Add → нарушен порядок публичного контракта → `Add` выполнять до запуска либо использовать `WaitGroup.Go`.**

## Когда применять выводы

Разделение capture, lifecycle и shared-state invariant полезно при review любого concurrent loop. Сначала определите ownership данных и момент завершения, затем выбирайте `Mutex`, atomic, channel или reducer. Историческую loop-variable проблему проверяйте только после определения language version.
