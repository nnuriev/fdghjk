# Сборка сниппета

## TL;DR

Вычисление — DAG из двух независимых цепочек: `itemDescription → prettify` и `itemPrice → priceToRub`. Цепочки запускаются параллельно, но порядок внутри каждой сохраняется. Результат публикуется только после обеих веток; первая ошибка отменяет общий дочерний context, а caller получает обе наблюдённые причины через `errors.Join`.

При исходных задержках последовательное выполнение заняло бы около четырёх секунд, параллельное — около двух: latency определяется более длинной веткой, а не суммой обеих. Это верно лишь пока dependencies действительно независимы и располагают capacity для параллельных запросов.

## Условие и контракт

Для товара нужно получить описание, затем отформатировать его; независимо получить цену в долларах, затем перевести её в рубли. После этого вернуть один `Snippet`.

Нормализованный контракт:

- все четыре dependencies обязательны и принимают `context.Context`;
- две цепочки выполняются параллельно, шаги одной цепочки — последовательно;
- если любой шаг завершился ошибкой, частичный `Snippet` не возвращается;
- первая ошибка вызывает cancellation sibling-ветки; каждая dependency обязана наблюдать `ctx`;
- deadline задаёт caller из общего request budget;
- функции не должны panic: `WaitGroup.Go` в Go 1.26.5 требует нормального возврата function.

### Неоднозначности исходника

- Исходник не задаёт политику деградации: цену в долларах, пустое описание, HTTP 500 или retry. Здесь выбран строгий all-or-nothing контракт; fallback должен быть отдельным продуктовым решением.
- `float64` оставлен ради совместимости с условием. Для реальных денег предпочтительны minor units в integer или decimal, иначе binary floating point не выражает многие десятичные суммы точно.
- Универсального «правильного timeout» нет. Он выводится из входного deadline, собственного server budget, latency distributions dependencies и резерва на сериализацию/ответ, как объяснено в [[20 Бэкенд/Дедлайны запросов и распространение отмены|заметке о дедлайнах]].
- Автоматический retry не встроен: неизвестный outcome, idempotency и общий retry budget должны быть определены отдельно.

## Ментальная модель

Сначала рисуем зависимости, а не goroutines:

```text
itemDescription(itemID) -> prettify(raw) -------+
                                                  +-> Snippet
itemPrice(itemID)       -> priceToRub(price) ----+
```

Ребро означает обязательный порядок, отсутствие ребра — возможность параллельного запуска. `WaitGroup` отвечает только на вопрос «все функции вернулись?». Он не переносит values, errors или cancellation. Поэтому результаты каждой ветки передаются через отдельный buffered channel, а cancellation кодируется дочерним context.

## Concurrency invariants

- У каждой ветки один producer и один result channel capacity `1`; producer может опубликовать итог даже до начала receive.
- Каждая ветка отправляет ровно один result на любом нормальном пути возврата.
- Main goroutine читает results только после `Wait`, поэтому lifecycle всех producers завершён.
- После ошибки вызывается idempotent `cancel`; sibling должен вернуть по `ctx.Done()`, иначе `BuildSnippet` продолжит ждать его. Context не умеет насильно остановить чужой код.
- Ни одна mutable variable не записывается двумя goroutines. Ownership передаётся через channels, поэтому отдельный mutex для результатов не нужен; выбор каналов вместо lock соответствует критерию из [[60 Go/Каналы или mutex|заметки о channels и mutex]].

## Код Go 1.26.5

`snippet.go`:

```go
package snippet

import (
	"context"
	"errors"
	"fmt"
	"sync"
)

type Snippet struct {
	Price       float64
	Description string
}

type Dependencies struct {
	ItemDescription func(context.Context, int) (string, error)
	Prettify        func(context.Context, string) (string, error)
	ItemPrice       func(context.Context, int) (float64, error)
	PriceToRub      func(context.Context, float64) (float64, error)
}

type descriptionResult struct {
	value string
	err   error
}

type priceResult struct {
	value float64
	err   error
}

func (d Dependencies) validate() error {
	if d.ItemDescription == nil || d.Prettify == nil || d.ItemPrice == nil || d.PriceToRub == nil {
		return errors.New("all snippet dependencies are required")
	}
	return nil
}

// BuildSnippet executes two independent dependency chains concurrently.
func BuildSnippet(ctx context.Context, itemID int, deps Dependencies) (Snippet, error) {
	if err := deps.validate(); err != nil {
		return Snippet{}, err
	}

	ctx, cancel := context.WithCancel(ctx)
	defer cancel()

	descriptionCh := make(chan descriptionResult, 1)
	priceCh := make(chan priceResult, 1)

	var wg sync.WaitGroup
	wg.Go(func() {
		raw, err := deps.ItemDescription(ctx, itemID)
		if err == nil {
			raw, err = deps.Prettify(ctx, raw)
		}
		if err != nil {
			err = fmt.Errorf("description chain: %w", err)
			cancel()
		}
		descriptionCh <- descriptionResult{value: raw, err: err}
	})
	wg.Go(func() {
		price, err := deps.ItemPrice(ctx, itemID)
		if err == nil {
			price, err = deps.PriceToRub(ctx, price)
		}
		if err != nil {
			err = fmt.Errorf("price chain: %w", err)
			cancel()
		}
		priceCh <- priceResult{value: price, err: err}
	})

	wg.Wait()
	description := <-descriptionCh
	price := <-priceCh
	if err := errors.Join(description.err, price.err); err != nil {
		return Snippet{}, err
	}

	return Snippet{Price: price.value, Description: description.value}, nil
}
```

## Tests

Проверка параллельного старта не опирается на `Sleep`: обе dependency-функции сообщают о старте и ждут один release channel.

`snippet_test.go`:

```go
package snippet

import (
	"context"
	"errors"
	"strings"
	"testing"
	"time"
)

func TestBuildSnippetRunsChainsConcurrently(t *testing.T) {
	started := make(chan string, 2)
	release := make(chan struct{})
	deps := Dependencies{
		ItemDescription: func(ctx context.Context, _ int) (string, error) {
			started <- "description"
			select {
			case <-release:
				return "hello world", nil
			case <-ctx.Done():
				return "", ctx.Err()
			}
		},
		Prettify: func(_ context.Context, value string) (string, error) {
			return strings.ToUpper(value), nil
		},
		ItemPrice: func(ctx context.Context, _ int) (float64, error) {
			started <- "price"
			select {
			case <-release:
				return 10, nil
			case <-ctx.Done():
				return 0, ctx.Err()
			}
		},
		PriceToRub: func(_ context.Context, price float64) (float64, error) {
			return price * 70, nil
		},
	}

	type outcome struct {
		snippet Snippet
		err     error
	}
	done := make(chan outcome, 1)
	go func() {
		got, err := BuildSnippet(context.Background(), 42, deps)
		done <- outcome{snippet: got, err: err}
	}()

	seen := map[string]bool{<-started: true, <-started: true}
	if !seen["description"] || !seen["price"] {
		t.Fatalf("chains did not both start: %v", seen)
	}
	close(release)

	result := <-done
	if result.err != nil {
		t.Fatalf("BuildSnippet() error = %v", result.err)
	}
	if result.snippet.Description != "HELLO WORLD" || result.snippet.Price != 700 {
		t.Fatalf("BuildSnippet() = %+v", result.snippet)
	}
}

func TestBuildSnippetCancelsSiblingOnError(t *testing.T) {
	descriptionErr := errors.New("description unavailable")
	deps := Dependencies{
		ItemDescription: func(context.Context, int) (string, error) {
			return "", descriptionErr
		},
		Prettify: func(context.Context, string) (string, error) {
			t.Fatal("Prettify must not run after source error")
			return "", nil
		},
		ItemPrice: func(ctx context.Context, _ int) (float64, error) {
			<-ctx.Done()
			return 0, ctx.Err()
		},
		PriceToRub: func(context.Context, float64) (float64, error) {
			t.Fatal("PriceToRub must not run after cancellation")
			return 0, nil
		},
	}

	_, err := BuildSnippet(context.Background(), 42, deps)
	if !errors.Is(err, descriptionErr) || !errors.Is(err, context.Canceled) {
		t.Fatalf("BuildSnippet() error = %v, want both source error and cancellation", err)
	}
}

func TestBuildSnippetHonorsDeadline(t *testing.T) {
	block := func(ctx context.Context) error {
		<-ctx.Done()
		return ctx.Err()
	}
	deps := Dependencies{
		ItemDescription: func(ctx context.Context, _ int) (string, error) {
			return "", block(ctx)
		},
		Prettify: func(context.Context, string) (string, error) { return "", nil },
		ItemPrice: func(ctx context.Context, _ int) (float64, error) {
			return 0, block(ctx)
		},
		PriceToRub: func(context.Context, float64) (float64, error) { return 0, nil },
	}

	ctx, cancel := context.WithTimeout(context.Background(), 20*time.Millisecond)
	defer cancel()
	_, err := BuildSnippet(ctx, 42, deps)
	if !errors.Is(err, context.DeadlineExceeded) {
		t.Fatalf("BuildSnippet() error = %v, want deadline exceeded", err)
	}
}
```

## Проверка результата

Tests доказывают параллельный старт обеих цепочек, сборку `Snippet{Price: 700, Description: "HELLO WORLD"}`, отмену sibling при ошибке и соблюдение caller deadline. Код прошёл `go test`, `go vet` и `go test -race` на Go 1.26.5 `darwin/arm64`, проверено 2026-07-18.

## Сложность и ресурсы

Число шагов и дополнительная память — `O(1)`, запускаются две goroutines и два channels capacity `1`. При длительностях веток `Tdesc` и `Tprice` идеальная latency равна `max(Tdesc, Tprice)` плюс небольшой coordination overhead; последовательная — `Tdesc + Tprice`. Реальная latency ограничена dependency capacity, очередями и deadline.

## Trade-offs и альтернативы

- `sync.WaitGroup` из standard library достаточен для фиксированных двух веток, но error/cancellation protocol приходится писать явно. `errgroup` из официального модуля `golang.org/x/sync` компактнее для динамической группы и умеет отменять context по первой ошибке; лимит concurrency и semantics результата всё равно задаёт приложение.
- Возврат fallback-цены или описания повышает availability, но меняет correctness и UX. Такой ответ должен явно показывать degraded state и не маскировать системную ошибку как свежие данные.
- Retry полезен только для transient error, безопасной операции и оставшегося budget. Без jitter/limit он создаёт retry storm; критерии описаны в [[40 Распределённые системы/Retry, exponential backoff и jitter|заметке о retry]].

## Типичные ошибки

- **Запустить четыре шага параллельно → `prettify` получает пустое описание → нарушены зависимости DAG → параллелить только независимые цепочки.**
- **Вызвать `wg.Add` внутри goroutine → `Wait` может вернуться раньше Add → registration гоняется с ожиданием → использовать `WaitGroup.Go` или делать `Add` до `go`.**
- **Вернуть по первой ошибке без остановки sibling → лишняя работа и goroutine leak при блокирующей публикации результата → отменять общий context и делать result channels buffered.**
- **Считать timeout константой dependency → верхний request уже исчерпал budget → вложенный вызов переживает caller → распространять deadline сверху и выделять подбюджеты.**
- **Retry любой ошибки → latency и нагрузка умножаются → не определены transient class, idempotency и budget → классифицировать ошибки до retry.**

## Когда применять выводы

Тот же подход работает для агрегаторов, Backend-for-Frontend и fan-out чтений: сначала DAG и consistency контракта, затем concurrency. Если количество задач приходит с входом, переходите от «goroutine на ветку» к bounded concurrency, чтобы fan-out не превратился в resource exhaustion.
