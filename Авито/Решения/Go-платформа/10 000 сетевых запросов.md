# 10 000 сетевых запросов

## TL;DR

Исходный цикл выполняет `10_000` вызовов по `1 ms` последовательно, поэтому печатает `10000` примерно через десять секунд. Простое добавление `go` создаёт две новые проблемы: `main` не ждёт workers, а `count++` содержит data race.

Production-вариант ниже использует bounded worker pool. Он ограничивает одновременно выполняемые операции, ждёт их завершения, считает successes через `atomic.Int64` и отменяет оставшуюся работу после первой ошибки или caller cancellation.

## Условие и контракт

Нужно ускорить `10_000` независимых I/O-like вызовов, каждый из которых в исходнике делает `time.Sleep(time.Millisecond)` и увеличивает счётчик.

Нормализованный контракт `Run`:

- выполнить не более `count` операций `request(ctx)`;
- держать не больше `workers` активных вызовов;
- вернуть число успешно завершившихся операций;
- после первой ошибки отменить выдачу новой работы и вернуть эту ошибку;
- соблюдать cancellation/deadline caller;
- не иметь data races и goroutine leaks.

### Неоднозначности исходника

- `Sleep(1 ms)` задаёт минимальную паузу, а не точный wall-clock. Поэтому «ровно 10 секунд» не гарантируется scheduler и ОС.
- У реального HTTP-вызова есть DNS, dial, TLS, connection pool, response body и errors; учебный `Sleep` этого не моделирует.
- Требуемая concurrency не задана. Она должна следовать из downstream limit, connection pool, memory/FD budget и latency target, а не из числа inputs.
- После частичной ошибки бизнес-контракт может требовать retry или список per-item results. Здесь выбран fail-fast и счётчик successes.

## Ментальная модель

Для I/O-bound работы goroutine большую часть времени ждёт. Другие goroutines могут использовать thread, поэтому несколько запросов перекрывают ожидание. Но «дешевле thread» не значит «бесплатно»: каждая операция удерживает goroutine stack, request state, socket/FD, buffers и downstream capacity.

Worker pool разделяет число inputs и число одновременно активных resources. Канал `jobs` — bounded handoff без накопления очереди, `workers` — concurrency budget, context — общий stop signal. Это частный случай [[60 Go/Worker pool, fan-in, fan-out и bounded concurrency|bounded concurrency]].

## Concurrency invariants

- Только coordinator закрывает `jobs`; workers канал не закрывают.
- В каждый момент не более `min(workers, count)` workers вызывают `request`.
- `completed` увеличивается только после успешного возврата `request`.
- `sync.Once` публикует ровно первую ошибку и вызывает `cancel` один раз; повторный `cancel` всё равно безопасен.
- `WaitGroup.Wait` происходит после `close(jobs)` и дожидается всех workers.
- `request` обязана принимать `ctx`: cancellation — кооперативный protocol, а не принудительное прерывание function.

## Код Go 1.26.5

`requests.go`:

```go
package requests

import (
	"context"
	"errors"
	"sync"
	"sync/atomic"
)

type RequestFunc func(context.Context) error

// Run executes count I/O-like operations with a bounded worker pool.
func Run(ctx context.Context, count, workers int, request RequestFunc) (int, error) {
	if count < 0 {
		return 0, errors.New("count must not be negative")
	}
	if workers <= 0 {
		return 0, errors.New("workers must be positive")
	}
	if request == nil {
		return 0, errors.New("request function is required")
	}

	ctx, cancel := context.WithCancel(ctx)
	defer cancel()
	jobs := make(chan struct{})
	var (
		completed atomic.Int64
		wg        sync.WaitGroup
		once      sync.Once
		firstErr  error
	)

	for range min(workers, count) {
		wg.Go(func() {
			for {
				select {
				case <-ctx.Done():
					return
				case _, ok := <-jobs:
					if !ok {
						return
					}
					if err := request(ctx); err != nil {
						once.Do(func() {
							firstErr = err
							cancel()
						})
						return
					}
					completed.Add(1)
				}
			}
		})
	}

sendLoop:
	for range count {
		select {
		case jobs <- struct{}{}:
		case <-ctx.Done():
			break sendLoop
		}
	}
	close(jobs)
	wg.Wait()

	if firstErr != nil {
		return int(completed.Load()), firstErr
	}
	if err := ctx.Err(); err != nil {
		return int(completed.Load()), err
	}
	return int(completed.Load()), nil
}
```

## Tests

`requests_test.go`:

```go
package requests

import (
	"context"
	"errors"
	"sync/atomic"
	"testing"
	"time"
)

func TestRunTenThousandRequestsWithBoundedConcurrency(t *testing.T) {
	const (
		count   = 10_000
		workers = 64
	)
	var inFlight, peak atomic.Int64

	completed, err := Run(context.Background(), count, workers, func(context.Context) error {
		current := inFlight.Add(1)
		for {
			old := peak.Load()
			if current <= old || peak.CompareAndSwap(old, current) {
				break
			}
		}
		time.Sleep(50 * time.Microsecond)
		inFlight.Add(-1)
		return nil
	})
	if err != nil {
		t.Fatalf("Run() error = %v", err)
	}
	if completed != count {
		t.Fatalf("completed = %d, want %d", completed, count)
	}
	if got := peak.Load(); got > workers {
		t.Fatalf("peak concurrency = %d, limit = %d", got, workers)
	}
}

func TestRunCancelsAfterError(t *testing.T) {
	sentinel := errors.New("network failed")
	var calls atomic.Int64
	completed, err := Run(context.Background(), 1000, 8, func(context.Context) error {
		if calls.Add(1) == 5 {
			return sentinel
		}
		return nil
	})
	if !errors.Is(err, sentinel) {
		t.Fatalf("Run() error = %v, want sentinel", err)
	}
	if completed >= 1000 {
		t.Fatalf("completed = %d, want partial execution", completed)
	}
}
```

## Проверка результата

Тест выполняет все `10_000` operations, измеряет peak и доказывает, что он не превышает `64`; отдельный test проверяет fail-fast cancellation. Код прошёл `go test`, `go vet` и `go test -race` на Go 1.26.5 `darwin/arm64`, проверено 2026-07-18.

Для исходного `Sleep(1 ms)` нижняя оценка идеального времени с `64` workers — примерно `ceil(10_000/64) × 1 ms ≈ 157 ms` плюс scheduler overhead. Утверждение «после goroutines будет 1–2 ms» возможно лишь при почти неограниченной concurrency и не подходит как безопасная capacity-модель реального сервиса.

## Сложность и ресурсы

Время работы — `O(count/workers)` волн при одинаковой latency операций, общее CPU-действие — `O(count)`. Память и число goroutines — `O(min(workers, count))`; unbuffered `jobs` не накапливает `O(count)` tasks. Реальные sockets/FD также ограничиваются числом workers и настройками transport.

## Trade-offs и альтернативы

- Одна goroutine на input плюс semaphore проще записывается, но всё равно создаёт `O(count)` goroutines, ожидающих token. Worker pool ограничивает и активную работу, и число goroutines.
- `atomic.Int64` подходит для независимого счётчика. Если вместе со счётчиком нужно согласованно менять несколько полей, mutex проще и сохраняет общий invariant; выбор разобран в [[60 Go/Пакет sync-atomic|заметке об atomics]].
- Сбор всех per-request errors требует result array/channel и policy partial success. Один `firstErr` намеренно реализует другой контракт.
- Для CPU-bound функции увеличение workers сверх доступного CPU не уменьшает объём вычислений и может добавить context switching. Здесь ускорение появляется из перекрытия ожидания.

## Типичные ошибки

- **Добавить `go request()` без ожидания → программа печатает прежний счётчик или `0` → `main` завершает process → ждать lifecycle workers.**
- **Сделать обычный `count++` → race и потерянные increments → read-modify-write не атомарен → atomic counter либо mutex.**
- **Запустить 10 000 реальных запросов одновременно → `EMFILE`, exhausted connection pool или overload downstream → concurrency не имеет resource budget → bounded workers и transport limits.**
- **Отменить coordinator, но request игнорирует context → `Run` ждёт зависший worker → cancellation кооперативна → передавать `ctx` до socket/API boundary и задавать сетевые timeouts.**
- **Использовать worker count как магическую константу → latency или errors меняются при иной capacity dependency → лимит должен вытекать из нагрузки, SLO и измерений.**

## Когда применять выводы

Worker pool подходит для большого числа независимых I/O tasks: HTTP calls, files, object storage, batch enrichment. Перед запуском фиксируют максимальную concurrency, timeout, error policy и реакцию на overload. Проверку FD, connections и memory связывают с [[10 Основы CS/Исчерпание ресурсов процесса|общим resource budget процесса]].
