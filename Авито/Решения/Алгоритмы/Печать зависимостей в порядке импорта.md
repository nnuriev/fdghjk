# Печать зависимостей в порядке импорта

## TL;DR

Рассматриваем `library → dependencies` как directed graph и запускаем DFS postorder: библиотека добавляется в ответ только после рекурсивного обхода всех зависимостей. Три состояния `unvisited/visiting/visited` одновременно исключают повторную работу и обнаруживают цикл.

## Нормализованное условие и контракт

Вход — map, где ключ — библиотека, значение — список её прямых зависимостей. Нужно вернуть порядок всех встреченных библиотек, в котором каждая зависимость расположена раньше зависящей от неё библиотеки. Библиотека, указанная только в списке зависимостей и отсутствующая как ключ, считается leaf без собственных зависимостей — именно так заданы `core`, `gpu` и `requests` в примере.

Пустые имена и повтор одной зависимости в списке — ошибки. Цикл делает корректный порядок невозможным и возвращает ошибку. Несвязные компоненты допустимы. Чтобы один и тот же вход-map всегда давал один ответ, вершины и соседей обходим лексикографически.

### Неоднозначности исходника

- Исходник требует «печать в stdout». Для тестируемости основная функция возвращает `[]string`; внешний слой печатает `strings.Join(order," ")`. Алгоритмический контракт от этого не меняется.
- Утверждается, что циклов нет, но безопасная реализация проверяет это предусловие. Иначе DFS либо зациклится, либо выдаст порядок, нарушающий зависимости.
- Отсутствующий ключ зависимости — не «разрыв»: пример сам использует такие leaf-библиотеки. Ошибкой было бы молча не включить их в результат.
- Корректных topological orders может быть много. Лексикографический DFS задаёт один воспроизводимый вариант, не заявляя его единственным.

## Ментальная модель

Postorder означает «закрой все prerequisites, затем текущую работу». Цвет `visiting` отмечает активный стек. Ребро в такую вершину требует импортировать библиотеку до самой себя через цепочку, то есть доказывает цикл.

## Алгоритм и корректность

1. Собираем множество всех ключей и всех упомянутых зависимостей, валидируем и сортируем копии adjacency lists.
2. Сортируем полный список вершин.
3. `visit(v)`:
   - при `visiting` возвращает ошибку цикла;
   - при `visited` ничего не делает;
   - иначе помечает `visiting`, посещает зависимости, помечает `visited` и добавляет `v` в ответ.
4. Вызываем `visit` для каждой вершины.

По индукции после успешного `visit(v)` каждая вершина в подграфе `v` добавлена один раз, а все зависимости `v` расположены раньше `v`, потому что append выполняется после рекурсивных вызовов. Внешний цикл покрывает также несвязные компоненты. Если найден back-edge в `visiting`, никакого topological order не существует; функция не возвращает частичный порядок как успешный.

## Код Go 1.26.5

`solution.go`:

```go
package imports

import (
	"errors"
	"slices"
)

func ImportOrder(deps map[string][]string) ([]string, error) {
	all := make(map[string]struct{})
	normalized := make(map[string][]string, len(deps))

	for library, requirements := range deps {
		if library == "" {
			return nil, errors.New("library name must not be empty")
		}
		all[library] = struct{}{}
		copyOfRequirements := slices.Clone(requirements)
		seen := make(map[string]struct{}, len(copyOfRequirements))
		for _, dependency := range copyOfRequirements {
			if dependency == "" {
				return nil, errors.New("dependency name must not be empty")
			}
			if _, duplicate := seen[dependency]; duplicate {
				return nil, errors.New("duplicate dependency")
			}
			seen[dependency] = struct{}{}
			all[dependency] = struct{}{}
		}
		slices.Sort(copyOfRequirements)
		normalized[library] = copyOfRequirements
	}

	vertices := make([]string, 0, len(all))
	for vertex := range all {
		vertices = append(vertices, vertex)
	}
	slices.Sort(vertices)

	state := make(map[string]uint8, len(vertices))
	order := make([]string, 0, len(vertices))
	var visit func(string) error
	visit = func(library string) error {
		switch state[library] {
		case 1:
			return errors.New("dependency graph contains a cycle")
		case 2:
			return nil
		}
		state[library] = 1
		for _, dependency := range normalized[library] {
			if err := visit(dependency); err != nil {
				return err
			}
		}
		state[library] = 2
		order = append(order, library)
		return nil
	}

	for _, vertex := range vertices {
		if err := visit(vertex); err != nil {
			return nil, err
		}
	}
	return order, nil
}
```

Для печати достаточно тонкого I/O-слоя:

```go
order, err := ImportOrder(deps)
if err != nil {
	return err
}
fmt.Println(strings.Join(order, " "))
```

```go
package imports

import (
	"reflect"
	"testing"

func TestImportOrder(t *testing.T) {
	tests := []struct {
		name    string
		deps    map[string][]string
		want    []string
		wantErr bool
	}{
		{
			name: "source graph with disconnected component",
			deps: map[string][]string{
				"tensorflow": {"nvcc", "gpu", "linux"},
				"nvcc":       {"linux"},
				"linux":      {"core"},
				"mylib":      {"tensorflow"},
				"mylib2":     {"requests"},
			},
			want: []string{"core", "gpu", "linux", "nvcc", "tensorflow", "mylib", "requests", "mylib2"},
		},
		{
			name: "dependency absent as key is a leaf",
			deps: map[string][]string{"app": {"runtime"}},
			want: []string{"runtime", "app"},
		},
		{name: "empty graph", deps: nil, want: []string{}},
		{
			name: "cycle", deps: map[string][]string{
				"a": {"b"}, "b": {"a"},
			}, wantErr: true,
		},
		{
			name: "duplicate edge", deps: map[string][]string{
				"a": {"b", "b"},
			}, wantErr: true,
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := ImportOrder(tc.deps)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && !reflect.DeepEqual(got, tc.want) {
				t.Fatalf("got %v, want %v", got, tc.want)
			}
		})
	}
}
```

Topological order — специальный случай directed acyclic graph (DAG); базовые понятия направленности и обхода есть в [[10 Основы CS/Графы и union-find|заметке о графах]].

## Сложность

Сам DFS занимает `O(V+E)` времени и `O(V)` памяти. Детерминированная сортировка добавляет `O(V log V + Σ d(v) log d(v))`; без требования фиксированного порядка её можно убрать и получить ожидаемое `O(V+E)` при hash map. Результат содержит `V` имён.

## Trade-offs

Алгоритм Kahn через indegree и очередь тоже даёт `O(V+E)` без детерминированной сортировки и естественно показывает, сколько вершин осталось в цикле. DFS короче для representation `library→dependencies` и сразу выдаёт dependencies-first postorder. Рекурсивный DFS рискует переполнить call stack на патологически глубокой цепочке; iterative frames или Kahn лучше для недоверенного огромного graph.

## Типичные ошибки

- **Добавлять библиотеку до рекурсивного обхода → dependent печатается раньше dependency → использован preorder → нужен postorder append.**
- **Хранить только boolean `visited` → цикл неотличим от уже завершённой вершины → потерян active stack → нужны три состояния.**
- **Обходить только ключи map и не посещать отсутствующие зависимости → `core`, `gpu`, `requests` исчезают → key set не равен vertex set → собирать keys ∪ values.**
- **Ожидать один конкретный порядок без tie-break → корректный результат кажется нестабильным → topological order не единственен → либо проверять property, либо задавать сортировку.**
