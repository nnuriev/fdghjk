# Поиск маршрута

## TL;DR

Строим map `from → ticket` и множество всех `to`. Начало — единственный город отправления, которого нет среди пунктов назначения. Затем идём по `from → to`, пока продолжение не закончится, и проверяем, что использовали ровно все билеты.

Именно последняя проверка отличает полный маршрут от правдоподобного префикса: разрыв или отдельная компонента иначе останутся незамеченными. Нулевое число стартов у непустого входа указывает на цикл; несколько стартов — на разрыв/несколько цепочек.

## Нормализованное условие и контракт

Каждый билет — ориентированное ребро `From → To` между непустыми строками. Корректный непустой вход должен образовывать одну простую направленную цепь, использующую каждый билет ровно один раз:

- у города не больше одного исходящего билета;
- у города не больше одного входящего билета;
- существует ровно один старт с исходящим и без входящего билета;
- циклов и отдельных компонент нет.

Функция возвращает билеты в порядке маршрута, не меняя вход. Пустой вход даёт пустой маршрут. Любое нарушение, включая цикл или разрыв, возвращает ошибку.

### Неоднозначности и повреждения исходника

Кавычки в примере повреждены; он нормализован как билеты `London→Moscow`, `NY→London`, `Moscow→SPb` и ответ `NY→London→Moscow→SPb`.

Исходник обещает единственный неразрывный маршрут без петель и повторов, но production-safe функция всё равно проверяет обещание. Иначе map молча перезапишет развилку, чистый цикл не имеет старта, а две независимые цепочки дадут неполный «успешный» ответ.

## Ментальная модель

В правильной цепи степени вершин создают fingerprint:

```text
start: indegree=0, outdegree=1
middle: indegree=1, outdegree=1
finish: indegree=1, outdegree=0
```

Map восстанавливает следующий шаг, а fingerprint находит старт. Счётчик использованных билетов доказывает связность всего входа.

## Алгоритм и корректность

1. За один проход строим `next[from]` и множество `destinations`, отвергая повторные `from` и `to`.
2. Среди ключей `next` ищем города, которых нет в `destinations`; требуется ровно один.
3. От старта последовательно берём `next[current]`, добавляем билет и переходим в `ticket.To`.
4. Отвергаем повторный departure и проверяем `len(route)==len(tickets)`.

В корректной цепи единственный старт обязательно найден. Из каждой внутренней вершины существует ровно один следующий билет, поэтому walk не имеет выбора и воспроизводит единственный порядок. Если использованы все `N` билетов, маршрут покрывает весь вход. Если меньше, оставшиеся билеты лежат в другой компоненте или цикле; если вершина повторена, найден цикл. Поэтому успешный результат удовлетворяет полному контракту.

## Код Go 1.26.5

`solution.go`:

```go
package route

import "errors"

type Ticket struct {
	From string
	To   string
}

func BuildRoute(tickets []Ticket) ([]Ticket, error) {
	if len(tickets) == 0 {
		return []Ticket{}, nil
	}

	next := make(map[string]Ticket, len(tickets))
	destinations := make(map[string]struct{}, len(tickets))
	for _, ticket := range tickets {
		if ticket.From == "" || ticket.To == "" {
			return nil, errors.New("ticket endpoints must not be empty")
		}
		if _, exists := next[ticket.From]; exists {
			return nil, errors.New("more than one ticket leaves a city")
		}
		if _, exists := destinations[ticket.To]; exists {
			return nil, errors.New("more than one ticket arrives at a city")
		}
		next[ticket.From] = ticket
		destinations[ticket.To] = struct{}{}
	}

	start := ""
	startCount := 0
	for from := range next {
		if _, hasIncoming := destinations[from]; !hasIncoming {
			start = from
			startCount++
		}
	}
	if startCount != 1 {
		return nil, errors.New("tickets contain a cycle or disconnected chains")
	}

	route := make([]Ticket, 0, len(tickets))
	visited := make(map[string]struct{}, len(tickets))
	current := start
	for {
		ticket, exists := next[current]
		if !exists {
			break
		}
		if _, repeated := visited[current]; repeated {
			return nil, errors.New("route contains a cycle")
		}
		visited[current] = struct{}{}
		route = append(route, ticket)
		current = ticket.To
	}
	if len(route) != len(tickets) {
		return nil, errors.New("tickets do not form one continuous route")
	}
	return route, nil
}
```

## Table-driven tests

`solution_test.go`:

```go
package route

import (
	"reflect"
	"testing"
)

func TestBuildRoute(t *testing.T) {
	tests := []struct {
		name    string
		tickets []Ticket
		want    []Ticket
		wantErr bool
	}{
		{
			name: "shuffled source example",
			tickets: []Ticket{
				{From: "London", To: "Moscow"},
				{From: "NY", To: "London"},
				{From: "Moscow", To: "SPb"},
			},
			want: []Ticket{
				{From: "NY", To: "London"},
				{From: "London", To: "Moscow"},
				{From: "Moscow", To: "SPb"},
			},
		},
		{name: "one ticket", tickets: []Ticket{{From: "A", To: "B"}}, want: []Ticket{{From: "A", To: "B"}}},
		{name: "empty", tickets: nil, want: []Ticket{}},
		{
			name: "cycle", tickets: []Ticket{
				{From: "A", To: "B"}, {From: "B", To: "A"},
			}, wantErr: true,
		},
		{
			name: "gap creates two chains", tickets: []Ticket{
				{From: "A", To: "B"}, {From: "C", To: "D"},
			}, wantErr: true,
		},
		{
			name: "branch", tickets: []Ticket{
				{From: "A", To: "B"}, {From: "A", To: "C"},
			}, wantErr: true,
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := BuildRoute(tc.tickets)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && !reflect.DeepEqual(got, tc.want) {
				t.Fatalf("got %v, want %v", got, tc.want)
			}
		})
	}
}
```

Эта задача использует directed graph, но не требует общего обхода: ограничения степеней сводят его к chain. Общая модель и failure modes графов есть в [[10 Основы CS/Графы и union-find|заметке о графах]].

## Сложность

Построение индексов, поиск старта и walk занимают `O(N)` ожидаемого времени и `O(N)` памяти. Возвращаемый маршрут сам требует `O(N)`; maps валидации также линейны.

## Trade-offs

Можно отсортировать билеты или перебирать оставшиеся на каждом шаге, но это даёт соответственно нерелевантный `O(N log N)` или `O(N²)`. Общая topological sort избыточна: здесь каждая допустимая вершина имеет не больше одного successor. Если допускаются повторные города и нужно использовать все билеты как Euler trail, это уже другой контракт и нужен алгоритм Hierholzer.

## Типичные ошибки

- **Строить только `from→to` и не проверять дубликат `from` → одна ветка перезаписывает другую → map скрывает invalid graph → проверять существование до записи.**
- **Вернуть walk при первом городе без продолжения → разорванный вход выглядит успешным → использована только одна компонента → сравнить число использованных билетов с `N`.**
- **Искать старт как первый билет во входе → shuffled пример остаётся несортированным → порядок массива не имеет семантики → вычислять `from \ to`.**
- **Не обработать чистый цикл → старт пуст, lookup ведёт к странному результату → у цикла нет indegree-zero вершины → требовать ровно один старт и вернуть ошибку.**
