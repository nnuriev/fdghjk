# Суммарная неудовлетворённость покупателей

## TL;DR

Сортируем товары, а для каждой потребности бинарным поиском находим первый товар не меньше неё. Ближайшим может быть только найденный элемент или его левый сосед; меньшую из двух дистанций прибавляем к ответу.

## Нормализованное условие и контракт

Каждый товар и каждая потребность заданы неотрицательным `int64`. Для потребности `x` неудовлетворённость равна `min |x-g|` по всем товарам `g`. Запас каждого товара не ограничен, поэтому выборы покупателей независимы. Нужно вернуть сумму неудовлетворённостей.

Функция не меняет входной слайс товаров. Пустой список покупателей даёт `0`. Если покупатели есть, а товаров нет, вход некорректен. Отрицательные значения и переполнение итоговой суммы также возвращают ошибку.

### Неоднозначности исходника

- Диапазон чисел не задан. Выбран `int64`; неотрицательность позволяет безопасно вычислять разность как `max-min`.
- Не сказано, отсортирован ли вход и можно ли его менять. Здесь функция принимает произвольный порядок и сортирует копию.
- При равноудалённых товарах можно выбрать любой: в сумму входит одна и та же дистанция.

## Ментальная модель

В отсортированном массиве позиция вставки делит товары на две части: слева все меньше потребности, справа все не меньше. Любой более далёкий элемент внутри каждой части не может быть лучше её граничного элемента. Поэтому вместо полного сканирования достаточно проверить не более двух кандидатов.

## Алгоритм и корректность

1. Валидируем вход и сортируем копию `goods`.
2. Для каждой потребности `need` находим lower bound — первый индекс `i`, где `goods[i] >= need`.
3. Если `i < len(goods)`, рассматриваем правого кандидата; если `i > 0`, левого `goods[i-1]`.
4. Прибавляем меньшую дистанцию, предварительно проверяя переполнение суммы.

Все элементы левее `i-1` не превосходят `goods[i-1]`, значит их расстояние до `need` не меньше. Аналогично все элементы правее `i` не меньше `goods[i]` и не ближе. Следовательно, минимум среди двух границ равен глобальному минимуму для покупателя. Независимость запасов позволяет сложить эти минимумы.

## Код Go 1.26.5

`solution.go`:

```go
package dissatisfaction

import (
	"errors"
	"math"
	"slices"
	"sort"
)

func TotalDissatisfaction(goods, needs []int64) (int64, error) {
	sortedGoods := slices.Clone(goods)
	for _, good := range sortedGoods {
		if good < 0 {
			return 0, errors.New("goods must be non-negative")
		}
	}
	if len(needs) == 0 {
		return 0, nil
	}
	if len(sortedGoods) == 0 {
		return 0, errors.New("at least one good is required")
	}
	slices.Sort(sortedGoods)

	var total int64
	for _, need := range needs {
		if need < 0 {
			return 0, errors.New("needs must be non-negative")
		}
		i := sort.Search(len(sortedGoods), func(i int) bool {
			return sortedGoods[i] >= need
		})

		best := int64(math.MaxInt64)
		if i < len(sortedGoods) {
			best = sortedGoods[i] - need
		}
		if i > 0 && need-sortedGoods[i-1] < best {
			best = need - sortedGoods[i-1]
		}
		if math.MaxInt64-total < best {
			return 0, errors.New("total dissatisfaction overflows int64")
		}
		total += best
	}
	return total, nil
}
```

package dissatisfaction

import "testing"

func TestTotalDissatisfaction(t *testing.T) {
	tests := []struct {
		name    string
		goods   []int64
		needs   []int64
		want    int64
		wantErr bool
	}{
		{name: "source example", goods: []int64{8, 3, 5}, needs: []int64{5, 6}, want: 1},
		{name: "both sides and equal distance", goods: []int64{2, 8}, needs: []int64{1, 5, 10}, want: 6},
		{name: "goods may be reused", goods: []int64{7}, needs: []int64{6, 7, 9}, want: 3},
		{name: "no buyers", goods: nil, needs: nil, want: 0},
		{name: "negative good without buyers", goods: []int64{-1}, needs: nil, wantErr: true},
		{name: "buyers without goods", goods: nil, needs: []int64{1}, wantErr: true},
		{name: "negative need", goods: []int64{1}, needs: []int64{-1}, wantErr: true},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := TotalDissatisfaction(tc.goods, tc.needs)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && got != tc.want {
				t.Fatalf("got %d, want %d", got, tc.want)
			}
		})
	}
}

## Сложность

Для `G` товаров и `B` покупателей время равно `O(G log G + B log G)`, дополнительная память — `O(G)` из-за копии. Если товары уже отсортированы и контракт разрешает доверять этому предусловию, подготовка становится `O(G)` на валидацию без копии либо `O(1)` без неё.

## Trade-offs

Сортировка потребностей и проход двумя указателями дают `O(G log G + B log B + G + B)` и могут быть выгоднее при очень большом `B`, но требуют сохранить соответствие покупателей только если нужны индивидуальные ответы. Полный перебор `O(G·B)` проще, однако не масштабируется.

## Типичные ошибки

- **Проверять только lower bound → потребность `6` выбирает `8` вместо `5` → проигнорирован левый сосед → сравнивать обе границы.**
- **Удалять купленный товар → результат зависит от порядка покупателей → ошибочно введено ограничение запаса → товары разрешено переиспользовать.**
- **Сортировать исходный слайс → вызывающая сторона наблюдает mutation → контракт владения не определён → сортировать clone либо явно документировать mutation.**
- **Использовать `abs(a-b)` для произвольных signed limits → вычитание переполняется до `abs` → арифметика уже повреждена → ограничить домен или проверять операции до вычисления.**
