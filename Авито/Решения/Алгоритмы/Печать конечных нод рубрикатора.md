# Печать конечных нод рубрикатора

## TL;DR

Запускаем depth-first search (DFS) от каждого корня и несём текущий путь в рекурсивном стеке. Если у ноды нет детей, соединяем путь через ` => ` и добавляем в результат. Отдельная маркировка нод обнаруживает цикл и повторное использование одного указателя в нескольких ветках — оба случая нарушают контракт дерева.

## Нормализованное условие и контракт

Рубрикатор — упорядоченный лес: каждый `Node` имеет непустой `Text` и ноль или больше детей. У каждой ноды не больше одного родителя, циклов нет, `nil`-нод нет. Нужно вернуть пути от корней до всех листьев в порядке корней и детей. Лист — нода с `len(Children)==0`; путь соединяется строкой ` => `.

Функция возвращает ошибку для `nil`, пустого текста, цикла либо shared node. Пустой лес даёт пустой список.

### Повреждение исходного примера

В текстовой схеме есть ветка `Вещи → Одежда → Мужская`, но ожидаемый вывод дополнительно содержит `Вещи => Одежда => Женская`. Нода `Женская` во входной схеме потеряна. В нормализованном примере она восстановлена как второй ребёнок `Одежда`, потому что это единственное положение, прямо заданное ожидаемым полным путём. Это восстановление относится только к примеру, а не к алгоритму.

Нормализованный лес:

```text
Вещи
└── Одежда
    ├── Мужская
    └── Женская          ← восстановлено из expected output
Хобби
├── Велосипеды
│   └── Горные
└── Мангалы
Транспорт
```

## Ментальная модель

Рекурсивный стек уже хранит ancestry текущей ноды. Нужно лишь синхронно добавлять label при входе и убирать при возврате. Лист превращает текущее временное состояние в самостоятельную строку результата.

## Алгоритм и корректность

DFS помечает ноду как `visiting`, добавляет её текст к пути и рекурсивно обходит детей. Лист материализует путь. После детей нода становится `visited`.

Инвариант при входе в `walk(node,path)`: `path` содержит labels ровно от корня до родителя `node`. После добавления `node.Text` это полный путь до `node`. Для листа он и есть требуемый ответ; для внутренней ноды рекурсивное предположение даёт все пути её поддерева. Обход каждого ребёнка по порядку сохраняет порядок результата. Состояние `visiting` означает back-edge и цикл, `visited` при повторном входе — более одного родителя или повторный корень.

## Код Go 1.26.5

`solution.go`:

```go
package rubricator

import (
	"errors"
	"strings"
)

type Node struct {
	Text     string
	Children []*Node
}

func LeafPaths(roots []*Node) ([]string, error) {
	paths := make([]string, 0)
	state := make(map[*Node]uint8)

	var walk func(node *Node, path []string) error
	walk = func(node *Node, path []string) error {
		if node == nil {
			return errors.New("tree contains a nil node")
		}
		if node.Text == "" {
			return errors.New("node text must not be empty")
		}
		switch state[node] {
		case 1:
			return errors.New("rubricator contains a cycle")
		case 2:
			return errors.New("node is shared by multiple paths")
		}

		state[node] = 1
		path = append(path, node.Text)
		if len(node.Children) == 0 {
			paths = append(paths, strings.Join(path, " => "))
		} else {
			for _, child := range node.Children {
				if err := walk(child, path); err != nil {
					return err
				}
			}
		}
		state[node] = 2
		return nil
	}

	for _, root := range roots {
		if err := walk(root, nil); err != nil {
			return nil, err
		}
	}
	return paths, nil
}
```

package rubricator

import (
	"reflect"
	"testing"
)

func TestLeafPaths(t *testing.T) {
	clothes := &Node{Text: "Одежда", Children: []*Node{
		{Text: "Мужская"},
		{Text: "Женская"},
	}}
	valid := []*Node{
		{Text: "Вещи", Children: []*Node{clothes}},
		{Text: "Хобби", Children: []*Node{
			{Text: "Велосипеды", Children: []*Node{{Text: "Горные"}}},
			{Text: "Мангалы"},
		}},
		{Text: "Транспорт"},
	}
	cycle := &Node{Text: "Цикл"}
	cycle.Children = []*Node{cycle}
	shared := &Node{Text: "Общий лист"}

	tests := []struct {
		name    string
		roots   []*Node
		want    []string
		wantErr bool
	}{
		{
			name: "normalized source tree", roots: valid,
			want: []string{
				"Вещи => Одежда => Мужская",
				"Вещи => Одежда => Женская",
				"Хобби => Велосипеды => Горные",
				"Хобби => Мангалы",
				"Транспорт",
			},
		},
		{name: "empty forest", roots: nil, want: []string{}},
		{name: "cycle", roots: []*Node{cycle}, wantErr: true},
		{
			name: "shared node",
			roots: []*Node{{Text: "Корень", Children: []*Node{
				{Text: "A", Children: []*Node{shared}},
				{Text: "B", Children: []*Node{shared}},
			}}},
			wantErr: true,
		},
		{name: "nil root", roots: []*Node{nil}, wantErr: true},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := LeafPaths(tc.roots)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && !reflect.DeepEqual(got, tc.want) {
				t.Fatalf("got %v, want %v", got, tc.want)
			}
		})
	}
}

## Сложность

DFS посещает `N` нод и `N-R` рёбер леса. Без учёта строк результата время `O(N)`, состояние маркировки `O(N)`, рекурсивный стек `O(H)`. Материализация путей стоит `Θ(L)`, где `L` — суммарное число символов во всех возвращаемых строках; в вырожденном дереве с большим числом длинных путей эта цена существенна.

## Trade-offs

Если вход гарантированно корректное дерево, map состояний можно убрать и оставить `O(H)` рабочей памяти. Но pointer-based структура из внешнего источника легко содержит цикл или shared node, и без защиты DFS уйдёт в бесконечную рекурсию либо напечатает неоднозначные пути. Итеративный DFS устраняет риск глубины call stack, но требует явно хранить frame и текущий путь.

## Типичные ошибки

- **Печатать только `node.Text` у листа → теряется контекст категории → путь не переносится через DFS → хранить ancestry.**
- **Считать листом `Children == nil` → пустой, но non-nil slice не распознаётся → nil и empty имеют одну длину → проверять `len==0`.**
- **Игнорировать повреждение `Женская` → expected output невозможно получить из показанного входа → источник внутренне противоречив → явно восстановить ноду и обозначить вывод.**
- **Считать любой visited безопасным → cycle/shared node молча пропускаются → вход уже не дерево → вернуть диагностируемую ошибку.**
