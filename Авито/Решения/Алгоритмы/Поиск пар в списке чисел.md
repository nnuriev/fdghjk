# Поиск пар в списке чисел

## TL;DR

Идём слева направо и храним уже увиденные числа в hash map. Для текущего `x` ищем `target-x`; если дополнение уже встречалось, добавляем пару. Затем записываем `x`. Уникальность входа гарантирует, что каждая неупорядоченная пара появится ровно один раз.

## Нормализованное условие и контракт

Вход — слайс попарно различных `int64` и целевая сумма `target`. Нужно вернуть все пары разных входных элементов, чья математическая сумма равна `target`. Пары `(a,b)` и `(b,a)` считаются одной. Результат упорядочен моментом обнаружения второго элемента; внутри пары первым стоит ранее встреченный элемент.

Повтор во входе считается нарушением контракта. Если `target-x` не представим в `int64`, соответствующей пары с `int64`-элементом существовать не может, поэтому этот `x` безопасно пропустить как кандидата на завершение пары.

### Неоднозначности исходника

- Повреждённая строка вывода восстановлена как пары `(2,5)` и `(4,3)` для входа `[2,4,5,3]`, `target=7`.
- Исходник обещает неповторяющиеся числа. Вариант с повторами потребовал бы уточнить, возвращать пары значений или все пары индексов.
- Порядок ответа не задан. Здесь он фиксирован порядком сканирования, чтобы тесты проверяли точный контракт.

## Ментальная модель

Map — это память о левой части массива. Когда рассматриваем `x`, все пары с ним и более ранним элементом определяются единственным дополнением `target-x`; будущие элементы будут обработаны позже и сами найдут `x`.

## Алгоритм и корректность

Инвариант перед обработкой позиции `i`: `seen` содержит ровно `nums[:i]`, а результат содержит все подходящие пары, у которых оба индекса меньше `i`, ровно по одному разу.

Для `x=nums[i]` вычисляем дополнение без signed overflow. Если оно в `seen`, существует и единственен ранний элемент пары — добавляем её. Другого раннего значения, способного дать нужную сумму с `x`, нет. После добавления `x` map и список пар удовлетворяют инварианту для следующего префикса. В конце префикс равен всему входу, значит найдены все пары.

## Код Go 1.26.5

`solution.go`:

```go
package pairsum

import (
	"errors"
	"math"
)

func PairsWithSum(nums []int64, target int64) ([][2]int64, error) {
	seen := make(map[int64]struct{}, len(nums))
	pairs := make([][2]int64, 0)

	for _, value := range nums {
		if _, duplicate := seen[value]; duplicate {
			return nil, errors.New("input values must be unique")
		}
		if complement, ok := safeDifference(target, value); ok {
			if _, exists := seen[complement]; exists {
				pairs = append(pairs, [2]int64{complement, value})
			}
		}
		seen[value] = struct{}{}
	}
	return pairs, nil
}

func safeDifference(a, b int64) (int64, bool) {
	if b > 0 && a < math.MinInt64+b {
		return 0, false
	}
	if b < 0 && a > math.MaxInt64+b {
		return 0, false
	}
	return a - b, true
}
```

## Table-driven tests

`solution_test.go`:

```go
package pairsum

import (
	"math"
	"reflect"
	"testing"
)

func TestPairsWithSum(t *testing.T) {
	tests := []struct {
		name    string
		nums    []int64
		target  int64
		want    [][2]int64
		wantErr bool
	}{
		{
			name: "source example", nums: []int64{2, 4, 5, 3}, target: 7,
			want: [][2]int64{{2, 5}, {4, 3}},
		},
		{
			name: "negative values", nums: []int64{-3, 8, 1, 4}, target: 5,
			want: [][2]int64{{-3, 8}, {1, 4}},
		},
		{name: "no pairs", nums: []int64{1, 2, 4}, target: 10, want: [][2]int64{}},
		{name: "overflowing complement is skipped", nums: []int64{math.MaxInt64, 0}, target: math.MinInt64, want: [][2]int64{}},
		{name: "duplicate input", nums: []int64{1, 1}, target: 2, wantErr: true},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := PairsWithSum(tc.nums, tc.target)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && !reflect.DeepEqual(got, tc.want) {
				t.Fatalf("got %v, want %v", got, tc.want)
			}
		})
	}
}
```

## Сложность

Ожидаемое время `O(N)` при обычных гарантиях hash map, память `O(N)` на `seen` и до `O(N)` на ответ для уникального входа. Асимптотика ответа не скрывается: все найденные пары нужно вернуть.

## Trade-offs

Сортировка копии и два указателя дают `O(N log N)` времени и `O(N)` памяти из-за сохранения входа; если mutation разрешена, дополнительную память можно уменьшить. Этот вариант детерминированнее по worst case и естественно поддерживает упорядоченный вывод. Двойной цикл использует `O(1)` памяти, но `O(N²)` времени.

## Типичные ошибки

- **Записать `x` до поиска дополнения → при `target=2x` элемент образует пару сам с собой → нарушено требование двух элементов → искать, затем вставлять.**
- **Добавлять и `(a,b)`, и `(b,a)` → каждая пара дублируется → просмотрены оба направления → искать только среди уже увиденных.**
- **Вычислять `target-x` без учёта границ → signed overflow создаёт ложное дополнение → машинная арифметика разошлась с математической → проверять разность до вычисления.**
