# Собери команду

## TL;DR

Держим по одному указателю в четырёх отсортированных списках. Текущая четвёрка задаёт диапазон `[min,max]`; запоминаем лучший и двигаем указатель списка, который дал минимум. Двигать любой другой список бессмысленно: текущий минимум останется, а максимум может только вырасти.

## Нормализованное условие и контракт

Даны четыре непустых неубывающих списка неотрицательных уровней: backend, frontend, QA и design. Нужно выбрать ровно один уровень из каждого списка и минимизировать `max(levels)-min(levels)`. Если оптимумов несколько, разрешён любой; реализация детерминированно оставляет первый найденный.

Функция возвращает выбранную команду и величину разброса. Пустой, несортированный список или отрицательный уровень — ошибка.

### Неоднозначности исходника

- «Ровность» определена только разницей экстремумов; среднее, медиана и попарные различия в objective не входят.
- Требуется вернуть уровни, а не индексы сотрудников. При одинаковом уровне разные сотрудники неразличимы.
- Исходник ожидает `O(N1+N2+N3+N4)` времени и `O(1)` памяти для фиксированных четырёх функций. Поэтому решение не объединяет все записи в общий массив.

## Ментальная модель

Текущие указатели образуют «окно», которое касается каждого списка. Чтобы сделать окно уже, нужно поднять его левую границу — текущий минимум. Продвижение элемента выше минимума не меняет левую границу и не может улучшить разброс.

## Алгоритм и корректность

1. Ставим четыре указателя в начало списков.
2. Вычисляем минимум и максимум текущих значений, обновляем лучший ответ.
3. Продвигаем один указатель, на котором находится минимум.
4. Заканчиваем, когда этот список исчерпан: больше нельзя собрать полную команду.

Рассмотрим текущий минимум `m` из списка `L`. Пока указатель `L` не сдвинут, любая комбинация с более поздними элементами остальных списков всё ещё содержит `m`, а её максимум не меньше текущего. Значит, она не может иметь меньший разброс и её безопасно пропустить. Алгоритм проверяет текущую комбинацию перед сдвигом и не пропускает ни одного потенциально лучшего диапазона. Когда какой-либо список исчерпан, комбинаций с элементом не меньше уже пройденного минимума больше нет.

## Код Go 1.26.5

`solution.go`:

```go
package team

import (
	"errors"
	"math"
)

type Groups struct {
	Backend  []int64
	Frontend []int64
	QA       []int64
	Design   []int64
}

type Team struct {
	Backend  int64
	Frontend int64
	QA       int64
	Design   int64
}

func AssembleTeam(groups Groups) (Team, int64, error) {
	lists := [4][]int64{groups.Backend, groups.Frontend, groups.QA, groups.Design}
	for _, list := range lists {
		if len(list) == 0 {
			return Team{}, 0, errors.New("every function must have at least one engineer")
		}
		if !validLevels(list) {
			return Team{}, 0, errors.New("levels must be non-negative and sorted")
		}
	}

	var indexes [4]int
	bestSpread := int64(math.MaxInt64)
	var best Team

	for {
		values := [4]int64{
			lists[0][indexes[0]], lists[1][indexes[1]],
			lists[2][indexes[2]], lists[3][indexes[3]],
		}
		minPos, maxPos := 0, 0
		for i := 1; i < len(values); i++ {
			if values[i] < values[minPos] {
				minPos = i
			}
			if values[i] > values[maxPos] {
				maxPos = i
			}
		}

		spread := values[maxPos] - values[minPos]
		if spread < bestSpread {
			bestSpread = spread
			best = Team{values[0], values[1], values[2], values[3]}
		}
		if bestSpread == 0 {
			return best, bestSpread, nil
		}

		indexes[minPos]++
		if indexes[minPos] == len(lists[minPos]) {
			return best, bestSpread, nil
		}
	}
}

func validLevels(levels []int64) bool {
	for i, level := range levels {
		if level < 0 || (i > 0 && level < levels[i-1]) {
			return false
		}
	}
	return true
}
```

## Table-driven tests

`solution_test.go`:

```go
package team

import "testing"

func TestAssembleTeam(t *testing.T) {
	tests := []struct {
		name       string
		groups     Groups
		want       Team
		wantSpread int64
		wantErr    bool
	}{
		{
			name: "perfect team from source",
			groups: Groups{
				Backend: []int64{1, 2, 2, 3}, Frontend: []int64{1, 3},
				QA: []int64{3, 4, 4}, Design: []int64{2, 3},
			},
			want: Team{Backend: 3, Frontend: 3, QA: 3, Design: 3},
		},
		{
			name: "forced wide range",
			groups: Groups{
				Backend: []int64{5}, Frontend: []int64{3, 6, 7, 10},
				QA: []int64{3, 9, 11, 18}, Design: []int64{20},
			},
			want: Team{Backend: 5, Frontend: 6, QA: 9, Design: 20}, wantSpread: 15,
		},
		{
			name: "one element in each list",
			groups: Groups{
				Backend: []int64{1}, Frontend: []int64{2},
				QA: []int64{3}, Design: []int64{4},
			},
			want: Team{Backend: 1, Frontend: 2, QA: 3, Design: 4}, wantSpread: 3,
		},
		{name: "empty group", groups: Groups{}, wantErr: true},
		{
			name: "unsorted group",
			groups: Groups{
				Backend: []int64{2, 1}, Frontend: []int64{1},
				QA: []int64{1}, Design: []int64{1},
			},
			wantErr: true,
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, spread, err := AssembleTeam(tc.groups)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && (got != tc.want || spread != tc.wantSpread) {
				t.Fatalf("got %+v/%d, want %+v/%d", got, spread, tc.want, tc.wantSpread)
			}
		})
	}
}
```

## Сложность

Каждая итерация продвигает ровно один указатель, поэтому их не больше `N1+N2+N3+N4`. Для четырёх списков время `O(N1+N2+N3+N4)`, дополнительная память `O(1)`.

## Trade-offs

Для произвольного числа `K` функций минимум удобно поддерживать min-heap, а максимум — отдельной переменной. Тогда время становится `O(N log K)`, память — `O(K)`. Фиксированная четвёрка выигрывает простотой и константами; heap выигрывает расширяемостью.

## Типичные ошибки

- **Продвигать текущий максимум → минимум остаётся прежним, диапазон не сужается → изменяется не та граница → двигать список минимума.**
- **Остановиться до оценки комбинации, чей указатель последний → пропускается оптимум на конце → сначала оценить, затем двигать.**
- **Сравнивать соседние уровни, а не экстремумы → objective подменён → «ровность» определяется `max-min` → вычислять обе границы всей четвёрки.**
