# Чемпионат по шагам

## TL;DR

За один проход агрегируем число посещённых дней и сумму шагов каждого участника. После прохода рассматриваем только тех, у кого число дней равно длине соревнования, находим максимальную сумму и возвращаем всех победителей с этой суммой.

Главный инвариант: после обработки первых `d` дней запись участника содержит сумму только его записей из этих дней и точное число дней, в которых он встретился. Дубликат `userId` внутри одного дня отвергается, иначе «не пропустил день» и сумма становятся неоднозначными.

## Нормализованное условие и контракт

Вход — `statistics`, где внешний слайс задаёт дни по порядку, а каждый внутренний слайс содержит пары `userId, steps` за один день. Нужно вернуть участников, которые:

1. имеют ровно одну запись в каждом дне соревнования;
2. среди таких участников набрали максимальную суммарную величину `steps`.

Если максимум разделяют несколько участников, функция возвращает все их `userId` по возрастанию и общую для них сумму. Пустое соревнование и соревнование без подходящих участников дают пустой список и `0`. `steps` должны быть неотрицательными, сумма должна помещаться в `int64`.

### Неоднозначности исходника

- В примерах не задан порядок победителей. Здесь он нормализован как возрастающий, чтобы результат и тесты были детерминированными.
- Не сказано, возможны ли две записи одного `userId` за день. Складывать их молча опасно: это меняет смысл дневной статистики, поэтому такой вход считается ошибкой.
- Не описан пустой календарь. Здесь он не создаёт «вакуумных чемпионов», а возвращает пустой результат.

## Ментальная модель

У каждого участника есть аккумулятор `(days, total)`. Условие участия — это фильтр `days == len(statistics)`, а победа — обычный максимум по `total` после фильтра. Хранить пересечение множеств участников отдельно не нужно: счётчик дней кодирует то же условие.

## Алгоритм и корректность

Для каждого дня создаём временное множество `seen`. Для каждой записи:

1. отвергаем отрицательные шаги и повторный `userId` в `seen`;
2. увеличиваем `days` и прибавляем `steps` в map по `userId`;
3. после всех дней сканируем map, пропуская записи с `days != число дней`;
4. поддерживаем текущий максимум и список участников, которые его разделяют;
5. сортируем только итоговый список идентификаторов.

После дня `d` инвариант следует по индукции: перед днём агрегаты точны для предыдущих дней; `seen` гарантирует ровно одно прибавление на текущем дне. Поэтому после последнего дня `days == D` эквивалентно присутствию во всех `D` днях. Второй проход рассматривает ровно допустимых участников и стандартным обновлением максимума сохраняет всех и только тех, чья сумма максимальна.

## Код Go 1.26.5

`solution.go`:

```go
package championship

import (
	"errors"
	"math"
	"slices"
)

type DayStat struct {
	UserID int
	Steps  int64
}

type Champions struct {
	UserIDs []int
	Steps   int64
}

func ChampionsFor(statistics [][]DayStat) (Champions, error) {
	empty := Champions{UserIDs: []int{}}
	if len(statistics) == 0 {
		return empty, nil
	}

	type aggregate struct {
		days  int
		total int64
	}
	byUser := make(map[int]aggregate)

	for _, day := range statistics {
		seen := make(map[int]struct{}, len(day))
		for _, stat := range day {
			if stat.Steps < 0 {
				return Champions{}, errors.New("steps must be non-negative")
			}
			if _, exists := seen[stat.UserID]; exists {
				return Champions{}, errors.New("duplicate user in one day")
			}
			seen[stat.UserID] = struct{}{}

			a := byUser[stat.UserID]
			if math.MaxInt64-a.total < stat.Steps {
				return Champions{}, errors.New("steps sum overflows int64")
			}
			a.days++
			a.total += stat.Steps
			byUser[stat.UserID] = a
		}
	}

	result := empty
	found := false
	for userID, a := range byUser {
		if a.days != len(statistics) {
			continue
		}
		if !found || a.total > result.Steps {
			found = true
			result.Steps = a.total
			result.UserIDs = []int{userID}
		} else if a.total == result.Steps {
			result.UserIDs = append(result.UserIDs, userID)
		}
	}
	if !found {
		return empty, nil
	}
	slices.Sort(result.UserIDs)
	return result, nil
}
```

## Table-driven tests

`solution_test.go`:

```go
package championship

import (
	"reflect"
	"testing"
)

func TestChampionsFor(t *testing.T) {
	tests := []struct {
		name    string
		stats   [][]DayStat
		want    Champions
		wantErr bool
	}{
		{
			name: "participant who attended every day wins",
			stats: [][]DayStat{
				{{UserID: 1, Steps: 1000}, {UserID: 2, Steps: 1500}},
				{{UserID: 2, Steps: 1000}},
			},
			want: Champions{UserIDs: []int{2}, Steps: 2500},
		},
		{
			name: "shared maximum",
			stats: [][]DayStat{
				{{UserID: 1, Steps: 2000}, {UserID: 2, Steps: 1500}},
				{{UserID: 2, Steps: 4000}, {UserID: 1, Steps: 3500}},
			},
			want: Champions{UserIDs: []int{1, 2}, Steps: 5500},
		},
		{
			name:  "empty day removes every candidate",
			stats: [][]DayStat{{{UserID: 1, Steps: 10}}, {}},
			want:  Champions{UserIDs: []int{}},
		},
		{
			name: "duplicate in one day is invalid",
			stats: [][]DayStat{{
				{UserID: 1, Steps: 10},
				{UserID: 1, Steps: 20},
			}},
			wantErr: true,
		},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			got, err := ChampionsFor(tc.stats)
			if (err != nil) != tc.wantErr {
				t.Fatalf("error = %v, wantErr %v", err, tc.wantErr)
			}
			if !tc.wantErr && !reflect.DeepEqual(got, tc.want) {
				t.Fatalf("got %+v, want %+v", got, tc.want)
			}
		})
	}
}
```

Проверки оформлены по модели из [[60 Go/Table-driven tests в Go|заметки о table-driven tests]]: отдельно покрыты пропущенный день, равенство максимумов и некорректный дубликат.

## Сложность

Пусть `R` — общее число дневных записей, `U` — число разных участников, `C` — число победителей. Время — `O(R + U + C log C)`, память — `O(U + M)`, где `M` — максимальное число записей за один день для `seen`. Если порядок победителей не фиксировать, сортировку можно убрать.

## Trade-offs

Можно начать с множества участников первого дня и удалять тех, кто отсутствует в очередном дне. Это раньше освобождает заведомо проигравших, но всё равно требует дневных множеств и отдельного хранения сумм. Счётчик дней проще и допускает один общий проход.

## Типичные ошибки

- **Суммировать всех, затем выбрать максимум → побеждает участник с пропуском → условие присутствия потеряно → фильтровать по `days == D`.**
- **Перезаписывать map дневным значением → учитывается только последний день → accumulator не сохранён → читать, изменять и записывать агрегат.**
- **Считать дубликаты двумя днями → участник ошибочно проходит фильтр → счётчик означает записи, а не дни → проверять дневной `seen`.**
- **Сравнивать с максимумом, начатым с нуля, без флага → невозможно отличить отсутствие кандидата от кандидата с нулём → хранить `found`.**
