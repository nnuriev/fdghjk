# Ограничения размера входных данных и исчерпание ресурсов

## TL;DR

Ограничить только HTTP body недостаточно. Один внешний input потребляет вектор ресурсов: connection time, header bytes, compressed и decompressed bytes, parser depth/nodes, memory, CPU, temporary disk, database work, queue slots и response bandwidth. Маленький payload тоже может быть дорогим, если запускает decompression, pathological parsing, большой query, fan-out или создание множества объектов.

Надёжная система задаёт budget перед каждой границей расширения и считает фактически потреблённый ресурс, а не доверяет заявленному `Content-Length`. Per-request limits дополняются deadlines, bounded concurrency, queue bounds, per-principal/tenant quotas, backpressure и load shedding. Эти механизмы отвечают на разные вопросы и не заменяют друг друга.

Policy должна быть endpoint-specific и измеримой: ресурс, единица, scope, enforcement point, реакция и telemetry. Limit, применённый после `ReadAll`, decompression или allocation, уже не защищает ресурс, который успели потратить.

## Область применимости

Заметка рассматривает CWE-400/CWE-770 в CWE 4.20 от 2026-04-30, HTTP semantics RFC 9110/RFC 6585 и серверные API Go `net/http` 1.26.5. Сюда входят JSON/forms/uploads, compressed content, archive/import jobs, GraphQL/RPC batches, search/export endpoints и asynchronous consumers.

Вне scope остаются точные численные значения для конкретного продукта: их получают из domain constraints, capacity tests и SLO. Значения в сквозном примере — иллюстративная policy, а не универсальная рекомендация.

## Ментальная модель

Input — программа расходов, даже если формат считается «данными»:

```text
wire bytes
  -> framing and headers
  -> transfer/content decoding
  -> parser nodes and allocations
  -> domain objects and lookups
  -> downstream work and response
```

На каждом переходе возможна amplification:

```text
small compressed input -> large plaintext
small nested document  -> deep recursion / many nodes
one filter              -> full table scan / huge result
one request             -> fan-out to many services
one valid body          -> long-lived upload slot
```

Per-request bound задаёт максимум одной операции. Aggregate bound задаёт число одновременно активных операций:

```text
worst-case pressure ≈ per-request cost × active concurrency
```

Это не формула точного capacity planning, а полезный invariant: безопасный body limit при unlimited concurrency всё равно допускает исчерпание [[10 Основы CS/Исчерпание ресурсов процесса|memory, file descriptors, threads/goroutines и CPU]].

## Как устроено

### Сначала составить resource budget

Для каждого endpoint фиксируют не одно число, а таблицу:

| Ресурс | Что измерять | Где останавливать | Scope |
| --- | --- | --- | --- |
| Request line/URI | bytes | edge и HTTP server parser | connection/request |
| Headers | total bytes, count, per-field bytes | до handler | connection/request |
| Wire body | фактически прочитанные bytes | bounded reader перед parser | request |
| Decoded body | bytes после decompression/decoding | bounded output reader | request |
| Syntax | depth, tokens/nodes, string/number length | parser | document |
| Domain cardinality | items, ranges, batch size | validation до work | command |
| Downstream | rows scanned/returned, execution time, fan-out | database/client/service | request/tenant |
| Aggregate work | active jobs, queued jobs, memory/disk | semaphore/queue/quota | instance/cluster/tenant |
| Response | rows/bytes/duration | pagination/stream budget | request/principal |

Единица и scope обязательны. «Limit 1000» без ответа «1000 чего, на request или tenant, до или после decompression» не является проверяемой policy.

### Ограничивать фактическое чтение

`Content-Length` полезен для раннего reject, но не является enforcement. Length может отсутствовать у streaming/framed request, а intermediary и application могут по-разному обрабатывать framing. Handler оборачивает body в reader, который сам считает прочитанные bytes и возвращает отдельную size error при попытке выйти за cap.

В Go 1.26.5 `http.MaxBytesReader` предназначен именно для incoming request body. В отличие от `io.LimitReader`, он возвращает `ReadCloser`, выдаёт `*http.MaxBytesError`, когда consumer пытается читать сверх limit, и закрывает underlying reader через `Close`. Limit ставят **до** JSON decoder, `ParseForm`, multipart parser или `io.ReadAll`.

Контракт требует различать bytes, доступные parser, и bytes, которые ограничивающий wrapper получил от underlying `Request.Body`. Чтобы отличить body ровно в limit от превышения, реализация может запросить у underlying reader до `remaining+1` bytes, но caller суммарно получает не больше limit. Ошибка появится только если consumer продолжит чтение и за границей действительно есть byte. Один `json.Decoder.Decode` этого не доказывает: он читает следующее JSON value, использует собственный buffer и может успешно остановиться перед trailing data. Для endpoint, который принимает ровно один JSON document, после первого `Decode` явно проверяют, что следующий `Decode` возвращает `io.EOF`; второе value, мусор и size error отвергают до business effect. `*http.MaxBytesError` означает `413`, а trailing data внутри cap — malformed request. Это не лимит bytes, которые HTTP server или network stack уже успели прочитать в собственные buffers; их memory/time budget задаётся на других границах.

`Request.ParseForm` имеет fallback cap 10 MB для form body, если body ранее не ограничен `MaxBytesReader`. Это implementation default, а не endpoint policy: он не покрывает произвольный JSON и может быть слишком большим или маленьким. Для multipart `ParseMultipartForm(maxMemory)` ограничивает memory для file parts, но остаток пишет во временные files; следовательно, `maxMemory` не ограничивает весь body или disk. `MultipartReader` позволяет stream processing, однако каждый part, суммарный body, disk и part count всё равно требуют bounds.

### Ограничивать до и после decompression

Wire cap защищает network ingress, но compressed payload способен развернуться во много раз. CWE-409 выделяет data amplification как отдельный случай uncontrolled resource consumption. Поэтому ставят два независимых счётчика:

```text
bounded compressed reader
  -> decompressor
  -> bounded decompressed reader
  -> parser
```

Ratio limit может служить дополнительным signal, но абсолютный decompressed cap проще доказывает memory/disk bound. При archive import также ограничивают entry count, per-entry size, total expanded size, nesting и temporary disk. Эти проверки идут вместе с [[20 Бэкенд/Path traversal|containment имён archive entries]].

### Структурные лимиты после byte limit

Одинаковое число bytes может создать разную стоимость. Parser и schema ограничивают:

- nesting depth и recursion;
- число object fields, array items, form keys и multipart parts;
- длину string, key, numeric literal и identifier;
- duplicate/unknown fields по выбранной policy;
- число references, aliases или expansion directives у форматов, где они есть;
- domain batch size, date range, polygon points и requested page size.

Byte limit не предотвращает expensive regular expression, pathological algorithm или huge integer conversion. Валидация до domain work связывает syntactic bounds с [[20 Бэкенд/Валидация и модель ошибок API|business constraints]]. Parser выбирают с известными complexity guarantees; fuzz/property tests проверяют boundary values и malformed inputs.

### Ограничивать downstream amplification

После успешного parse request способен создать дорогой database query, N downstream calls или большой response. Поэтому budget продолжается:

- database statement timeout, row/scan limits и index-aware query shape;
- pagination/cursor и server maximum page size;
- fan-out cap и bounded parallelism;
- output byte/row cap для exports;
- job cost class и отдельная queue для тяжёлых операций;
- cancellation всех дочерних операций при abort/deadline.

[[20 Бэкенд/Дедлайны запросов и распространение отмены|Deadline]] ограничивает wall-clock time, но не гарантирует memory cap: process может быстро выделить много памяти до timeout. Resource counter ограничивает amount, но не зависший I/O. Нужны оба.

### Aggregate bounds, backpressure и load shedding

Per-request cap не защищает от множества допустимых requests. Semaphore ограничивает active expensive operations; bounded queue задаёт максимум ожидающих; per-tenant quotas не дают одному tenant занять весь shared pool. Когда capacity исчерпана, system перестаёт принимать новую работу или снижает её качество вместо бесконтрольного queue buildup.

[[20 Бэкенд/Rate limiting и quotas|Rate limiting]] измеряет arrivals за окно и полезен против sustained abuse. Concurrency limit измеряет одновременно удерживаемый ресурс и защищает от slow requests. [[40 Распределённые системы/Backpressure и queue buildup|Backpressure]] сообщает producer о недостатке capacity, а [[40 Распределённые системы/Load shedding|load shedding]] сохраняет работу системы, отклоняя часть нагрузки. Один token bucket не заменяет эти механизмы.

Global limit без tenant reservation допускает noisy-neighbor effect. Только per-tenant limit может оставить instance без общей защиты при большом числе tenants. Обычно есть и local/global ceiling, и fair per-principal/tenant allocation; identity для quota берут из authenticated context, а не из client header.

### Time и connection limits

Slow sender удерживает socket и goroutine при маленьком body. HTTP server задаёт header/read/write/idle timeouts и maximum header bytes; в Go это связывается с [[60 Go/Тайм-ауты HTTP-сервера и клиента|тайм-аутами HTTP-сервера и клиента]]. `Server.MaxHeaderBytes` ограничивает bytes request line и header keys/values, но не body.

Timeout должен отменять downstream work, иначе client уже получил error, а database или worker продолжает расходовать ресурс. При streaming endpoint различают inactivity timeout и total duration: слишком короткий общий deadline обрывает легитимный большой transfer, а отсутствие inactivity bound допускает slow-client exhaustion.

### Реакция и HTTP status

RFC 9110 и RFC 6585 дают разные сигналы:

- `413 Content Too Large` — request content больше, чем server готов обработать;
- `414 URI Too Long` — target URI превышает допустимую длину;
- `431 Request Header Fields Too Large` — слишком велики headers в сумме или одно field;
- `429 Too Many Requests` — user/client превысил rate policy; допустим `Retry-After`;
- `503 Service Unavailable` — server временно не справляется из-за overload/maintenance; server не обязан отвечать 503 и под атакой может закрыть connection.

Статус выбирают по причине, а не по расположению middleware. Очередь заполнена из-за текущей capacity — обычно `503`; индивидуальная quota/rate — `429`; oversized body — `413`. Error body и logging тоже bounded: иначе reject path сам становится amplification channel.

### Наблюдаемость без новой уязвимости

Метрики разделяют причины: `body_limit`, `decoded_limit`, `depth_limit`, `queue_full`, `deadline`, `tenant_quota`. Записывают endpoint, policy ID, authenticated tenant и bucketed observed size, но не целиком hostile body или PII. Cardinality label и log volume имеют собственные caps/sampling.

Alert строят не только на числе rejects. Важны active concurrency, queue age, temporary disk, allocator/GC pressure, downstream cancellation rate и процент limit hits около legitimate p99. Это позволяет отличить abuse от слишком тесной policy.

## Пример или трассировка

Import endpoint использует иллюстративную policy:

```text
compressed body <= 1 MiB
decompressed JSON <= 4 MiB
nesting depth <= 16
items <= 100
active imports per instance <= 32
queued imports <= 16
```

### Сценарий 1: неизвестная длина и body больше cap

Client отправляет framed/streaming body без пригодного для доверия `Content-Length`; фактический поток — 2 MiB. Возможны два варианта: либо первое JSON value само пересекает границу 1 MiB, либо оно заканчивается раньше, но после него остаются trailing bytes за границей.

1. Handler ставит counting body limit 1 MiB до decompressor и JSON decoder.
2. Если границу пересекает первое value, уже первый `Decode` получает `*http.MaxBytesError`.
3. Если первое value помещается в cap, endpoint запускает второй `Decode`, чтобы подтвердить отсутствие второго value и получить `io.EOF`; trailing bytes за cap дают тот же size error на этой проверке.
4. В обоих вариантах `MaxBytesReader` может получить от underlying `Request.Body` один дополнительный byte, но возвращает caller суммарно не больше 1 MiB.
5. Domain list, job и database transaction не создаются. Service классифицирует size error как `413`, возвращает короткий error code и закрывает body.

Наблюдаемый результат: ранний header check не был единственной защитой; parser увидел максимум policy cap, а underlying `Request.Body` вернул wrapper не более cap плюс один byte для обнаружения превышения. Нижележащий HTTP/TCP stack мог заранее наполнить свои buffers. Если parser отвергает malformed input раньше, ответ может быть `400` без определения полной длины — resource bound на consuming boundary от этого не нарушается.

### Сценарий 2: compressed payload проходит wire cap

Payload занимает 700 KiB на wire, но разворачивается более чем в 4 MiB.

1. Compressed reader пропускает поток: 700 KiB меньше 1 MiB.
2. Decompressed counter останавливает output до передачи полного document parser.
3. Service возвращает `413`/domain-specific `content_too_large`; temporary disk и parsed graph не растут до expanded payload.

Наблюдаемый результат: wire limit и decoded limit защищают разные ресурсы.

### Сценарий 3: каждый request допустим, instance уже занят

32 imports находятся в active pool, ещё 16 ждут в bounded queue. Следующий valid request не начинает parse/import work и получает `503` с retry guidance. Если тот же authenticated tenant превысил свою quota раньше общего ceiling, он получает `429`.

Наблюдаемый результат: per-request size не превращается в unlimited aggregate pressure, а причина overload не смешивается с индивидуальным rate violation.

Трассировка сверена с контрактами и исходным кодом `MaxBytesReader`, `json.Decoder`, `ParseForm`, `ParseMultipartForm`, `MultipartReader` и `Server.MaxHeaderBytes` в Go 1.26.5, а status semantics — с RFC 9110 и RFC 6585. Числа policy намеренно примерные; исполняемый Go-код локально не запускался из-за отсутствия toolchain.

## Trade-offs

### Reject целиком или stream

Buffering упрощает atomic validation и повторное чтение, но memory растёт с cap. Streaming ограничивает retained memory и начинает work раньше, зато требует incremental validation, cleanup partial state и невозможность «откатить» уже отправленный response. Large upload/import обычно stream/spool с checksum, quota и commit after validation; маленькая JSON command — bounded buffer.

### Static или adaptive limits

Static cap легко документировать, тестировать и одинаково применять на edge/app. Он плохо учитывает разные cost classes. Adaptive admission использует текущую load/cost estimate, лучше сохраняет SLO, но сложнее объясняется client и может быть нестабилен. Hard safety ceilings оставляют статическими; admission threshold может быть динамическим.

### Edge или application enforcement

Edge рано экономит bandwidth/connections и защищает все handlers. Он не знает decoded size, schema depth и domain cost. Application понимает endpoint semantics, но request уже прошёл часть stack. Limits дублируют согласованно: внешний слой не должен разрешать меньше legitimate application contract без явной причины, а внутренний не полагается на единственный proxy.

### Отказ или очередь

Очередь сглаживает короткий burst и повышает utilization. Неограниченная очередь превращает overload в latency, memory pressure и retry storm. Bounded queue имеет явный maximum age; после него дешевле отказать и дать контролируемый retry, чем выполнять устаревшую работу.

### Жёсткий cap или удобство клиента

Малый cap снижает blast radius, но ломает legitimate bulk operation. Альтернатива — не увеличивать synchronous endpoint без меры, а предоставить pagination, chunked/resumable upload, asynchronous export или pre-signed object upload с отдельной storage quota.

## Типичные ошибки

### Доверяют `Content-Length`

- **Неверное предположение:** заявленная длина равна фактически прочитанной.
- **Симптом:** streaming request без длины обходит early check или разные HTTP layers спорят о framing.
- **Причина:** header использован как enforcement вместо bounded reader.
- **Исправление:** header — только fast reject; фактические bytes считает reader на consuming boundary.

### Проверяют размер после `ReadAll` или decompression

- **Неверное предположение:** сравнение длины вернёт контроль над уже потраченной memory.
- **Симптом:** OOM/temporary-disk spike возникает до `413`.
- **Причина:** limit стоит после amplification/allocation.
- **Исправление:** bounded input и bounded decoded output до parser/storage.

### Ограничивают bytes, но не структуру

- **Неверное предположение:** одинаковый размер означает одинаковую стоимость.
- **Симптом:** маленький deeply nested document вызывает recursion, CPU spike или много allocations.
- **Причина:** parser/domain complexity не входит в budget.
- **Исправление:** depth/node/item/string limits, complexity-safe parser и domain cardinality bounds.

### Считают timeout полным resource limit

- **Неверное предположение:** всё дорогое обязательно работает долго.
- **Симптом:** process быстро расходует memory или disk до deadline; downstream продолжает работать после response.
- **Причина:** ограничено wall time, но не amount и cancellation propagation.
- **Исправление:** resource counters + deadline + отмена дочерней работы.

### Есть per-request cap, но нет aggregate cap

- **Неверное предположение:** безопасный максимум одного request делает безопасной любую concurrency.
- **Симптом:** множество валидных requests исчерпывает pool/heap или строит длинную очередь.
- **Причина:** не ограничен множитель active/queued work.
- **Исправление:** concurrency semaphore, bounded queue, global и per-tenant admission.

### Ставят только rate limit

- **Неверное предположение:** requests per second описывает удерживаемый ресурс.
- **Симптом:** небольшое число slow uploads занимает все sockets/workers.
- **Причина:** rate и concurrency измеряют разные dimensions.
- **Исправление:** rate + concurrent in-flight + read/inactivity deadlines.

### Reject path сам дорогой

- **Неверное предположение:** после отказа ресурс больше не тратится.
- **Симптом:** logs, metrics cardinality или подробный error отражают/хранят весь hostile input.
- **Причина:** error handling не имеет собственного budget.
- **Исправление:** bounded generic response, sampled/bucketed telemetry и запрет raw body в logs.

## Когда применять

Resource review выполняют для каждого external boundary и каждой стадии expansion, а не только для upload endpoint. Нужны документированные limits на ingress, decode, parse, domain cardinality, downstream work, response, concurrency и queue. Тесты проходят значения `limit-1`, `limit`, `limit+1`, missing/misleading length, compressed expansion, maximum depth и concurrent saturation.

Практическое правило: у любой операции должен быть ответ на два вопроса — «сколько максимум стоит одна?» и «сколько таких операций могут одновременно удерживать ресурс?». Если известен только body size или только RPS, модель исчерпания ресурсов неполна.
