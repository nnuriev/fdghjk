# Фильтрация, сортировка и частичные ответы

## TL;DR

Filtering и sorting превращают list endpoint в публичный query language. Контракт должен зафиксировать grammar, types, null/missing semantics, collation, timezone, разрешённые поля, стоимость запроса и полный стабильный порядок. Строку клиента нельзя подставлять в SQL: её разбирают в AST, проверяют по schema/allowlist и переводят в parameterized query.

Partial response управляет projection, а не authorization. Field mask, sparse fieldset или GraphQL selection set сокращают response только тогда, когда server проталкивает projection в storage и expensive resolvers. Omitted field означает «не запрошено», а не обязательно `null`; default projection и views входят в backward-compatibility contract.

## Область применимости

- Filter grammar и list ordering опираются на AIP-160 и AIP-132 с changelog до 2025-10-03 и 2025-02-25 соответственно.
- Partial response соответствует AIP-157, field masks — AIP-161; обе страницы проверены 2026-07-18.
- JSON:API 1.1 и GraphQL September 2025 используются как официальные альтернативные wire models.
- Вне scope: full-text relevance ranking, database-specific optimizer hints и authorization model целиком.

## Ментальная модель

List request строит последовательный query pipeline:

```text
principal и authorization scope
    -> parse query language
    -> validate schema, types и cost
    -> filter authorized set
    -> total stable sort
    -> paginate
    -> project разрешённые fields
    -> encode response
```

Порядок операций — часть результата. Если сначала взять `LIMIT 50`, а затем отфильтровать, client получает не первые 50 matching resources, а случайную часть. Если сначала project и потерять sort key, continuation становится неоднозначным. Если authorization применяется после filter/count, query может раскрыть существование скрытых resources.

## Как устроено

### Filtering как язык

Filter string удобен для evolution, но требует настоящей спецификации:

- lexical grammar: identifiers, string escaping, literals, parentheses;
- operators и precedence;
- type conversion для numbers, booleans, enums, timestamps и durations;
- смысл `null`, missing field, empty list и wildcard;
- traversal по nested fields и collections;
- case sensitivity, Unicode normalization, collation и timezone;
- limits на длину, AST nodes, nesting, disjunctions и expensive functions.

AIP-160, например, задаёт `AND`, `OR`, `NOT`, comparisons, traversal и `:` как has operator. У него `OR` имеет более высокий precedence, чем `AND`, в отличие от большинства языков программирования. Поэтому documentation должна показывать скобки: пользовательское ожидание нельзя восстанавливать догадкой.

Server разбирает filter в typed AST:

```text
AND
├── EQ(field=customer_id, string="c-42")
└── GE(field=create_time, timestamp=2026-07-01T00:00:00Z)
```

Затем validator проверяет существование field, разрешённый operator, type literal, access policy и cost. SQL builder выбирает только заранее сопоставленные columns/functions и передаёт values parameters. Даже корректный parser не должен разрешать arbitrary traversal или function call, если storage не способен выполнить их с bounded cost.

Если filter ссылается на неизвестное поле или значение несовместимого типа, сервер возвращает стабильную [[20 Бэкенд/Валидация и модель ошибок API|validation error]], а не пустой список. Иначе typo выглядит как «данных нет».

Когда filter слишком велик или чувствителен для URI, RFC 10008 позволяет передать его в content safe/idempotent метода QUERY. На 2026-07-18 это новый protocol feature: перед внедрением нужно проверить clients, CORS preflight, gateway и cache, причём cache key обязан учитывать content. Детали границы разобраны в [[20 Бэкенд/HTTP-методы, статус-коды, заголовки и семантика кэширования|заметке о HTTP-семантике]].

### Sorting и total order

`order_by` обычно содержит упорядоченный список fields и direction, например `create_time desc, id desc`. Контракт дополнительно фиксирует:

- natural comparator каждого type;
- порядок `NULL`/missing values;
- case и locale collation для strings;
- timezone/precision timestamp;
- разрешённые combinations и maximum fields;
- implicit unique tie-breaker.

Client-visible sort должен соответствовать storage query. Если API принимает `name`, а database сортирует по locale-specific collation, эту collation нельзя менять без semantic review. Сортировка по mutable или computed field влияет на stability [[20 Бэкенд/Пагинация offset, cursor и continuation token|cursor pagination]].

Даже если client запрашивает `create_time desc`, server добавляет `id desc` как документированный tie-breaker. Continuation token связывается с normalized complete order, включая implicit part. Иначе две rows с одинаковым timestamp могут меняться местами между pages.

### Partial responses

Есть три распространённые модели.

**Field mask/sparse fieldset.** Client перечисляет paths, например `id,total,status` или `author.name`. Fine-grained control уменьшает bytes, но усложняет validation, nested collections и cache key.

**Named view.** `BASIC` и `FULL` задают server-owned projections. Их проще оптимизировать, документировать и авторизовать, но client может получить лишние fields или не получить нужную комбинацию.

**GraphQL selection set.** Projection встроена в query и совпадает с response shape. Server всё равно обязан ограничить cost, batch nested loads и проверить field-level access.

AIP-157 рекомендует для своей field-mask модели optional side-channel parameter. Если API выбирает эту модель, parameter должен быть optional, `*` означает все fields, а изменение default mask считается breaking change. Отдельный `read_mask` в request message для Google APIs deprecated в пользу system parameter. AIP-157 также допускает named view, если вариантов немного.

Projection уменьшает работу только при pushdown:

```text
fields=id,total,status
  -> SELECT id, total, status
  -> не запускать resolver margin_forecast
  -> сериализовать только выбранные fields
```

Если application сначала загружает полный object graph и вычисляет все derived fields, partial response экономит лишь network bytes. Нужны tracing и DB query inspection, чтобы подтвердить выигрыш.

Omitted и `null` различаются. `null` обычно означает известное отсутствующее/недоступное значение в выбранном field; omission означает, что field не вошёл в representation или не существует в этой version. Generated client и cache merge должны сохранять это различие.

### Authorization и утечки через query shape

Projection нельзя применять вместо authorization. Server сначала определяет доступный resource set и разрешённые query fields. Иначе client может не запрашивать `salary`, но фильтровать `salary > 100000` и по count узнать скрытую информацию. Sort по secret field раскрывает относительный порядок, а error «field существует, но запрещено» может стать schema oracle.

Policy должна отдельно решать:

- разрешено ли читать field;
- разрешено ли фильтровать и сортировать по нему;
- можно ли использовать его в aggregate/count;
- какой error не раскрывает лишний schema/resource context.

Cache и ETag относятся к итоговому representation. Filter, order, projection и authorization context должны участвовать в cache key или запрещать shared caching. Byte-разным projections нужны разные strong ETag; общий weak ETag допустим только при явно определённой semantic equivalence и всё равно не заменяет корректный variant key.

## Пример или трассировка

Request:

```http
GET /orders?filter=customer_id%20%3D%20%22c-42%22%20AND%20create_time%20%3E%3D%20%222026-07-01T00%3A00%3A00Z%22&order_by=create_time%20desc%2C%20id%20desc&fields=id%2Ctotal%2Cstatus&page_size=50&page_token=TOKEN HTTP/1.1
```

После parse и validation server получает typed plan:

```sql
SELECT id, total, status, create_time
FROM orders
WHERE tenant_id = $1
  AND customer_id = $2
  AND create_time >= $3
  AND (create_time, id) < ($4, $5)
ORDER BY create_time DESC, id DESC
LIMIT $6;
```

`create_time` нужен внутри для cursor, но не попадает в encoded response. Наблюдаемый результат содержит только `id`, `total`, `status`; DB trace не показывает загрузку остальных columns или вызов expensive resolver.

Contract checks:

- `customer_id = 42` отклоняется как type error;
- unknown field не превращается в empty result;
- `margin` нельзя ни выбрать, ни использовать в filter/sort без permission;
- token от `customer_id="c-42"` отклоняется после смены filter или order;
- две rows с одинаковым `create_time` стабильно упорядочены по `id`;
- head insert не создаёт duplicate на следующей cursor page.

## Trade-offs

- Структурированный filter string легче расширять, чем отдельное поле на каждый predicate. Цена: parser, документация, cost model и более сложный compatibility surface. Небольшому API может хватить нескольких typed query parameters.
- Свободный multi-field sort даёт client flexibility, но требует множества indexes и cursor combinations. Named sort modes ограничивают UX, зато дают предсказуемый query plan.
- Fine-grained field mask минимизирует payload. Named views проще кешировать, авторизовать и оптимизировать. GraphQL даёт максимальную projection flexibility, но переносит cost control на runtime.
- Reject unknown input быстро ловит typo. Forward-compatible output readers обычно игнорируют новые fields. Одинаковая strictness в обе стороны редко полезна.
- Exact filtered count удобен UI, но способен доминировать по стоимости и раскрывать hidden cardinality. Estimate или отсутствие count безопаснее, если exact value не входит в product requirement.

## Типичные ошибки

- Неверное предположение: filter — это SQL fragment. Симптом: injection или доступ к незапланированным columns/functions. Причина: public query language напрямую смешан с storage language. Исправление: parse в AST, allowlist и parameterized translation.
- Неверное предположение: любой валидный query допустим по стоимости. Симптом: один request сканирует всю таблицу или строит взрывной join. Причина: grammar ограничивает syntax, но не work. Исправление: complexity budget, bounded traversal, supported combinations, indexes и timeout.
- Неверное предположение: filtering можно выполнить после page limit. Симптом: короткие pages, пропущенные matching rows и неверный total. Причина: операции переставлены. Исправление: filter authorized set до sort/pagination.
- Неверное предположение: visible fields определяют authorization. Симптом: secret field не виден в body, но утечка идёт через filter, sort или count. Причина: projection применена после query. Исправление: отдельная policy для read/filter/sort/aggregate и authorized candidate set.
- Неверное предположение: requested order уже total. Симптом: cursor pages дают duplicates/gaps при ties. Причина: нет unique comparator. Исправление: добавить и документировать stable tie-breaker, включить его в token.
- Неверное предположение: partial response автоматически ускоряет server. Симптом: bytes меньше, DB/CPU те же. Причина: полный graph вычисляется до projection. Исправление: pushdown selection и проверка trace/query plan.
- Неверное предположение: omission эквивалентен `null`. Симптом: client стирает cached value или считает field пустым. Причина: representation presence потеряна. Исправление: presence-aware model и documented merge semantics.

## Когда применять

Добавляйте filtering и sorting только под подтверждённые use cases и доступные indexes. Сначала выберите небольшой grammar и supported fields, затем versioned extensions. Каждый новый operator должен пройти security и cost review.

Partial responses оправданы для больших или дорогих resources и UI с разными projections. Если shape стабильна и дешева, полный resource проще для caching, declarative clients и compatibility. В любом варианте default projection, order и token binding входят в контракт.
