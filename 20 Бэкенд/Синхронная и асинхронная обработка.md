# Синхронная и асинхронная обработка

## TL;DR

Синхронный API удерживает request-response до терминального бизнес-результата. Асинхронный принимает команду долговечно, возвращает operation handle и завершает работу позже. Различие задаёт клиентский контракт, а не наличие goroutine, event loop или broker внутри реализации.

`202 Accepted` не означает успех операции: RFC 9110 прямо допускает, что принятая работа ещё не началась и позже может быть отклонена. Поэтому async API требует durable acceptance, state machine, terminal result/error, idempotent submission, правила cancellation, retention и способ наблюдать прогресс. Без них он лишь переносит timeout в непрозрачное «когда-нибудь».

## Область применимости

- HTTP-семантика `202 Accepted` соответствует RFC 9110 от июня 2022 года.
- Preference `respond-async` рассматривается по RFC 7240 от июня 2014 года.
- Long-running operation model сверена с Google AIP-151, обновлённым 2025-02-04.
- Message-driven contracts показаны по AsyncAPI 3.1.0 от 2026-01-31.
- Вне scope: внутренняя модель async I/O runtime, конкретный workflow engine и устройство scheduler для [[20 Бэкенд/Фоновые задачи и scheduled jobs|фоновых и scheduled jobs]].

## Ментальная модель

Синхронный вызов обещает: «когда ответ вернулся, известен итог этой операции». Асинхронный обещает другое: «команда надёжно принята под идентификатором, а итог появится в отдельном состоянии».

Это две разные транзакционные границы:

```text
synchronous: request -> execute -> terminal response

asynchronous: request -> durable acceptance -> 202/operation
                                      |
                                      v
                              execute -> terminal state
```

Если сервер возвращает `202` до durable acceptance, между ответом и записью задания есть окно потери. Клиент уверен, что работа принята, а восстановить её некому. Потому acceptance point должен переживать crash.

## Как устроено

### Синхронный контракт

Синхронная обработка подходит, когда работа укладывается в request deadline и клиенту нужен немедленный итог для следующего решения. Ошибки возвращаются в исходном response, cancellation естественно распространяется по call tree, а trace остаётся одним causal path.

Внутри server может использовать queues и concurrency, но это не делает API асинхронным, пока клиент ждёт терминальный outcome. И наоборот, `202` не становится надёжным только потому, что код запустил goroutine.

Длинный синхронный request удерживает connection и admission slot, а иногда ещё и транзакцию. При client disconnect outcome может стать неизвестным. Если ожидаемая latency близка к proxy/client timeouts либо работа должна пережить restart, operation resource обычно честнее.

### Durable acceptance и operation resource

Асинхронная команда сначала фиксируется в durable storage. В одной локальной транзакции удобно создать domain command или job, operation state и запись [[40 Распределённые системы/Transactional outbox и Change Data Capture|transactional outbox]], после чего вернуть `202`.

Operation имеет стабильный ID, owner и state machine. Минимальный набор состояний можно выразить так:

```text
PENDING -> RUNNING -> SUCCEEDED
                   -> FAILED
PENDING/RUNNING -> CANCEL_REQUESTED -> CANCELLED | SUCCEEDED | FAILED
```

`CANCEL_REQUESTED` отделяет желание клиента от фактической отмены. Worker мог уже зафиксировать необратимый эффект, поэтому cancel не обещает rollback без отдельной compensating semantics.

Terminal state хранит result reference либо структурированную error model. Progress и metadata не должны менять meaning результата. Google AIP-151 также отделяет ошибку запуска самой операции от ошибки, записанной в уже созданной long-running operation.

### Идемпотентное принятие

Клиент способен не получить `202` после успешной записи. Повтор того же submit должен вернуть прежнюю operation, а не запустить вторую работу. Для этого request связывают с [[20 Бэкенд/Ключи идемпотентности и дедупликация запросов|idempotency key]] или естественным business key, проверяют fingerprint и сохраняют operation ID вместе с результатом acceptance.

Worker тоже переживает redelivery. Operation ID не заменяет event ID и inbox на границе consumer; гарантии очередей описаны в [[40 Распределённые системы/Очереди, streams, группы потребителей и DLQ|заметке о messaging]].

### Как клиент узнаёт итог

Основные варианты:

- polling operation resource: просто восстанавливается после потери связи, но создаёт read load и требует backoff;
- webhook/callback: снижает polling, зато нужен authenticated delivery, retry, дедупликация и endpoint lifecycle;
- event/stream: хорошо для нескольких подписчиков, но клиент обязан хранить position и переживать duplicate delivery;
- long polling или streaming: даёт быстрые обновления, однако само ожидание снова становится долгоживущим connection contract.

RFC 7240 позволяет клиенту выразить `Prefer: respond-async`, но preference — пожелание, а не приказ. Сервер может обработать запрос синхронно. Если он выбирает async, response должен объяснить состояние и дать способ следить за ним. RFC 9110 рекомендует representation статуса и ссылку на status monitor.

AsyncAPI 3.1.0 описывает channels, messages и send/receive operations независимо от конкретного broker topology. Такой контракт нужен рядом с HTTP API: async transport не отменяет versioning schema, authentication и error semantics.

### Авторизация и retention

Operation resource проверяет право на чтение результата, отмену и повтор. Нельзя полагаться на трудноугадываемый ID. Для worker заранее выбирают, действует ли authority на момент acceptance или authorization повторно проверяется перед эффектом; эти варианты дают разное поведение после отзыва доступа.

Operation и result не хранят бесконечно. Retention должен быть длиннее гарантированного окна получения результата, а удалённая operation возвращает документированное terminal поведение. Иначе медленный клиент не отличит «никогда не было» от «успешно завершилось и уже удалено».

## Пример или трассировка

Клиент создаёт отчёт, который занимает минуты.

1. `POST /reports` приходит с `Idempotency-Key: k-42`. В одной DB transaction сервер создаёт `operation op-7` со state `PENDING`, связывает `k-42 -> op-7` и пишет outbox event.
2. После commit сервер отвечает `202 Accepted`, `Location: /operations/op-7`. Crash после ответа не теряет задание: outbox publisher дочитает запись.
3. Worker получает event, атомарно переводит operation в `RUNNING` и строит отчёт. Duplicate delivery видит тот же `event_id` в inbox и не запускает второй расчёт.
4. Клиент опрашивает `/operations/op-7` с backoff. Сначала получает `RUNNING` и progress metadata, затем `SUCCEEDED` с ссылкой на immutable result.
5. Повтор исходного `POST` с `k-42` возвращает `op-7`. Тот же key с другим fingerprint получает conflict.

Если расчёт падает, исходный `POST` всё равно остаётся успешно принятым; бизнес-ошибка записывается в terminal `FAILED`. Это и есть отличие acceptance от completion.

## Trade-offs

Синхронный API проще для клиента: один вызов, один error path, read-your-result сразу после ответа. Цена — связь availability и latency всех участников, удержание ресурсов и хрупкость на длинных операциях.

Асинхронный API сглаживает burst и позволяет работе пережить disconnect/restart. Цена заметно выше: durable queue, operation storage, дедупликация, progress, cancellation, cleanup и observability lag.

Polling устойчив к потере callback и легко отлаживается, но создаёт herd. Callback быстрее доставляет итог, однако переносит retry и authentication на сервер. Event stream удобен для многих результатов, но заставляет клиента стать полноценным consumer.

Operation resource подходит для одной адресуемой команды и её результата. Durable Job resource по Google AIP-152 лучше, когда пользователь должен многократно запускать, останавливать и наблюдать самостоятельную сущность, а не временный execution handle.

## Типичные ошибки

### `202` возвращается до durable записи

- **Неверное предположение:** goroutine обязательно переживёт ответ и restart.
- **Симптом:** operation ID есть у клиента, но status навсегда отсутствует.
- **Причина:** acceptance и enqueue разделены crash window.
- **Исправление:** сначала durable transaction/queue acknowledgement, затем `202`.

### Нет terminal error model

- **Неверное предположение:** любой принятый job когда-нибудь станет successful.
- **Симптом:** operation бесконечно остаётся `RUNNING` или исчезает.
- **Причина:** failure и timeout worker не записываются как outcome.
- **Исправление:** terminal `FAILED`, machine-readable cause, retryability и reconciliation зависших leases.

### Cancellation обещает rollback

- **Неверное предположение:** cancel request мгновенно отменяет уже выполненный side effect.
- **Симптом:** API показывает `CANCELLED`, хотя платёж или экспорт уже создан.
- **Причина:** смешаны прекращение будущей работы и компенсация commit.
- **Исправление:** `CANCEL_REQUESTED`, explicit point of no return и отдельная compensation.

### Polling без backoff

- **Неверное предположение:** частый GET ускоряет worker.
- **Симптом:** status endpoint получает больше трафика, чем рабочая система.
- **Причина:** каждый клиент опрашивает с фиксированным коротким интервалом.
- **Исправление:** backoff, jitter, cache validators либо push channel.

## Когда применять

Синхронную обработку выбирают для короткой работы, где клиенту нужен конечный результат в том же interaction и вся цепочка укладывается в SLO.

Асинхронная нужна, когда latency непредсказуема или велика, работу требуется пережить restart, burst следует сгладить, а клиент готов работать с operation state. До реализации фиксируют acceptance guarantee, state machine, deduplication, cancellation, notification, authorization и retention. Без этих пунктов async API сложнее синхронного, но не надёжнее.
