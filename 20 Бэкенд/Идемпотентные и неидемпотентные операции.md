# Идемпотентные и неидемпотентные операции

## TL;DR

Операция идемпотентна, если несколько одинаковых запросов имеют тот же intended effect, что один. Речь идёт об эффекте, а не об одинаковых status, body, latency или записях в логах. GET, PUT и DELETE имеют идемпотентную HTTP-семантику; POST и PATCH требуют знания конкретной операции.

Idempotency key делает повторяемой логическую команду: сервер атомарно связывает уникальный ключ с tenant, endpoint и fingerprint запроса, а повтор возвращает уже известный outcome. Это bounded deduplication, не гарантия exactly-once delivery и не замена concurrency control.

## Область применимости

- Базовая HTTP-семантика соответствует RFC 9110 от июня 2022 года; safe/idempotent QUERY учитывается по RFC 10008 от июня 2026 года. Источники проверены 2026-07-18.
- Request identification опирается на AIP-155, Stripe API v1 и Amazon EC2 API как на подтверждённые design examples. Их retention и replay policy нельзя переносить в другой API молча.
- Рассматриваются клиентские повторы из-за timeout, разрыва соединения и transient transport failure.
- Вне scope: broker delivery guarantees, transactional outbox и межсервисная saga.

## Ментальная модель

Клиент отправил команду, но потерял ответ. У него три возможных мира: сервер не видел запрос; сервер начал и откатил его; сервер завершил side effect, а response пропал в сети. По одному timeout эти миры не различить.

Idempotency превращает повтор из новой команды в запрос о той же логической операции. Operation identity переживает отдельное TCP-соединение и HTTP attempt. Сервер узнаёт identity, сравнивает request fingerprint и либо продолжает первый запуск без нового intended effect, либо возвращает сохранённый результат, либо сообщает, что тот же ключ уже связан с другим запросом.

## Как устроено

### Семантика метода и семантика операции

RFC 9110 определяет PUT, DELETE и safe methods как idempotent. RFC 10008 добавляет к safe/idempotent methods QUERY. Это контракт intended effect:

- два `PUT /users/42` с одним representation оставляют один и тот же целевой state;
- два `DELETE /users/42` не удаляют два разных ресурса, даже если ответы отличаются;
- повтор GET не должен сам по себе создавать запрошенное изменение состояния.
- повтор QUERY заново вычисляет safe query и не запрашивает изменение target resource, хотя response может измениться вместе с исходными данными.

POST по умолчанию неидемпотентен: два `POST /payments` естественно могут создать два платежа. PATCH тоже не имеет общего свойства. Patch `{"email":"new@example.com"}` с replace semantics идемпотентен, а `{"increment":1}` — нет. Клиенту нельзя угадывать это по синтаксису URI или body; гарантия должна быть частью [[20 Бэкенд/Контракты API и обратная совместимость|API-контракта]].

Даже идемпотентный метод не всегда можно автоматически повторить. Streaming body может быть non-replayable, общий deadline может закончиться, а credentials или preconditions — устареть. RFC 9110 запрещает автоматически повторять non-idempotent request, если client не знает, что семантика операции идемпотентна, или не установил, что исходный request не был применён.

### Контракт idempotency key

На 2026-07-18 общего действующего RFC для `Idempotency-Key` нет: Internet-Draft версии 07 истёк 2026-04-18. Поэтому имя field, scope, retention и replay semantics остаются provider-specific контрактом; совпадение имени header у двух API ещё не означает одинакового поведения.

Практичный контракт отвечает на семь вопросов.

1. **Кто создаёт identity.** Обычно клиент генерирует UUID v4 или другой высокоэнтропийный opaque key до первого attempt.
2. **Каков scope.** Минимум tenant/account и logical operation; часто также method и endpoint. Один key в разных tenants не должен пересекаться.
3. **Что считается тем же запросом.** Сервер связывает key с canonical fingerprint существенных параметров. Тот же key с другой суммой или получателем отклоняется.
4. **Когда key фиксируется.** Claim key и переход операции в состояние `processing` должны быть атомарны относительно конкурирующих requests.
5. **Что сохраняется.** Operation ID, terminal state и достаточно данных для договорённого replay. Stripe API v1 сохраняет status и body первого начавшегося выполнения, включая `500`; другой API может возвращать текущее representation операции. Политику нужно назвать явно.
6. **Что получает конкурентный duplicate.** Возможны ожидание первого выполнения, документированный `409 Conflict` либо `202 Accepted` со ссылкой на operation resource. Бесконечное параллельное выполнение недопустимо.
7. **Каков retention window.** После TTL тот же key может считаться новым. Клиент должен знать нижнюю границу, иначе поздний retry создаст duplicate.

Упрощённая state machine:

```text
absent --atomic claim--> processing --commit--> succeeded
                             |                    |
                             +----failure------> failed

same key + same fingerprint:
  processing -> documented in-progress response
  succeeded/failed -> documented replay

same key + different fingerprint -> mismatch error
```

Claim, бизнес-эффект и result record должны переживать crash в согласованном порядке. Если локальная база владеет эффектом, их можно связать одной транзакцией. Если эффект выполняет внешний payment provider, нужен его idempotency mechanism или отдельная reconciliation state machine. Локальная запись ключа после внешнего charge оставляет crash window.

### Граница гарантии

Idempotency key не доказывает exactly-once execution во всей системе. Он ограничивает повторы там, где один authority хранит deduplication state. TTL, region и endpoint образуют границу гарантии. Amazon EC2, например, документирует regional и zonal scope для разных операций; одинаковый client token в другом scope может обозначать другой запрос.

Idempotency также не сериализует разные операции. Два PATCH с разными keys могут одновременно прочитать version 7 и записать несовместимые значения. Для lost update нужны preconditions (`If-Match`), version column или transaction isolation; HTTP-вариант разобран в [[20 Бэкенд/HTTP-методы, статус-коды, заголовки и семантика кэширования|заметке о семантике HTTP]].

## Пример или трассировка

Клиент создаёт платёж:

```http
POST /payments HTTP/1.1
Idempotency-Key: "8b6f2d8a-8f95-4ac4-b51d-c5aa228f8111"
Content-Type: application/json

{"account":"A-7","amount":5000,"currency":"RUB"}
```

Два одинаковых requests приходят конкурентно.

```text
t0 request A: INSERT operation(key, tenant, fingerprint, state=processing) -> success
t1 request B: INSERT same unique key -> waits for transaction A
t2 request A: INSERT ledger(payment_id=P-91, amount=5000)
t3 request A: UPDATE operation SET state=succeeded, result=P-91; COMMIT
t4 response A: connection breaks before client receives bytes
t5 request B: unique-key wait ends -> conflict; rollback claim transaction; read completed operation
t6 response B: 200 {"id":"P-91","status":"succeeded"}
```

Наблюдаемый результат: в ledger одна запись `P-91`, а retry возвращает тот же logical payment. Внешнего payment provider в этой трассировке нет; если charge выполняется вне локальной транзакции, ему нужен собственный idempotency key и reconciliation из state machine. Запрос с тем же key и `amount=7000` получает стабильную mismatch error, например `409` с machine code `IDEMPOTENCY_KEY_REUSED`. Новый key с исходным payload создаёт новый платёж, потому что это новая operation identity.

Fault injection должен оборвать процесс после `COMMIT`, но до отправки response. Без этого тест проверяет happy path, а не ambiguous outcome.

## Trade-offs

- Идемпотентный PUT по известному resource URI проще дедуплицировать: target identity уже находится в URI. POST с server-generated ID удобнее клиенту, но для безопасного retry требует отдельного operation key.
- Хранить полный response удобно для byte-stable replay, но это увеличивает storage, удерживает потенциально чувствительные данные и фиксирует старое representation. Хранить operation ID дешевле, зато поздний duplicate может получить текущее состояние, а не исходный response. Контракт должен выбрать один смысл.
- Длинный TTL снижает риск поздних duplicates, но увеличивает storage и период уникальности ключей. Короткий TTL дешевле, но должен превышать максимальный клиентский retry и reconciliation horizon.
- Ожидание in-flight результата даёт единый ответ, но занимает connection и server resources. Быстрый `409` или operation resource разгружает сервер, однако усложняет client state machine.
- Fingerprint всего raw body чувствителен к незначимому JSON formatting и порядку полей. Canonical fingerprint существенных typed parameters устойчивее, но требует versioned canonicalization policy.

## Типичные ошибки

- Неверное предположение: key можно записать после side effect. Симптом: после crash появляются два платежа. Причина: retry не видит завершённую первую попытку. Исправление: атомарно claim-ить operation до эффекта и связывать commit эффекта с durable outcome либо использовать downstream idempotency.
- Неверное предположение: совпадения key достаточно. Симптом: response старой операции возвращается на новый payload или другому tenant. Причина: нет scope и fingerprint check. Исправление: key lookup включает tenant/operation, а параметры сравниваются с canonical fingerprint.
- Неверное предположение: нужно кешировать только `2xx`. Симптом: ambiguous failure выполняется повторно, хотя первая попытка уже изменила состояние. Причина: failure semantics не определены. Исправление: сохранять terminal outcome после начала execution согласно контракту и различать validation failure, in-flight conflict и execution failure.
- Неверное предположение: idempotency key решает гонку любых writes. Симптом: две команды с разными keys затирают друг друга. Причина: deduplication отвечает только за повтор одной identity. Исправление: добавить resource version/precondition или транзакционную сериализацию.
- Неверное предположение: локальной дедупликации достаточно для внешнего эффекта. Симптом: одна локальная payment row, но два charge у провайдера. Причина: crash window находится между двумя authorities. Исправление: передавать стабильный downstream key, применять outbox/reconciliation и моделировать intermediate states.
- Неверное предположение: key действует всегда и везде. Симптом: поздний или cross-region retry создаёт новую операцию. Причина: retention и scope ограничены. Исправление: документировать границу и не повторять за ней без read/reconciliation.

## Когда применять

Idempotency key нужен для команд, которые клиент или transport вправе повторить после ambiguous failure: платежей, создания заказов, резервирования, запуска jobs. Он особенно полезен, когда resource ID выдаёт сервер и PUT по client-chosen URI не подходит.

Не добавляйте key к каждой read operation по привычке: GET уже имеет идемпотентную семантику. Для конкурентных updates используйте version preconditions. Для операций, чей внешний эффект невозможно дедуплицировать, сначала проектируйте reconciliation и честно показывайте клиенту состояние `unknown` или `processing`, а не обещайте exactly-once.
