# Защитная обработка невалидного ввода

## TL;DR

Невалидный input — штатная ветка управления на любой trust boundary, а не исключительная ситуация и не только признак атаки. Надёжный обработчик сначала ограничивает объём данных, затем разбирает их в transport DTO, проверяет позитивный контракт и cross-field invariants, создаёт domain type и только после этого запускает side effects. Каждый пройденный этап усиливает гарантии типа, но не превращает данные в «абсолютно доверенные» для всех следующих контекстов.

Validation не заменяет parameterized queries, output encoding, authorization и constraints хранилища. Эти механизмы защищают разные границы. Ошибка клиента должна завершаться контролируемым machine-readable response без panic, частичной записи и утечки внутренних деталей; timeout или отказ dependency нельзя маскировать под «невалидный запрос».

## Ментальная модель

Полезно рассматривать вход как последовательность ворот:

```text
untrusted bytes
-> bounded reader
-> syntax decoder
-> transport DTO
-> contract and cross-field validation
-> domain constructor
-> authorization and current-state checks
-> side effects under storage constraints
```

После `bounded reader` известно лишь, что input помещается в установленный budget. После decoder известно, что bytes образуют допустимое transport-представление. После domain constructor можно получить тип `Transfer`, который невозможно создать с отрицательной суммой или одинаковыми счетами. Но даже такой объект ещё не доказывает, что principal вправе списать деньги и что balance не изменился конкурентно.

Trust boundary есть не только у публичного HTTP endpoint. Сообщение из очереди, файл, environment variable, запись старой версии в БД, cache value и callback поставщика тоже пришли из компонента с независимым lifecycle. Фраза «это внутренние данные» не задаёт инвариант формата.

## Как устроено

### Проверки идут от дешёвых и общих к контекстным

| Этап | Что становится известно | Что ещё не доказано |
| --- | --- | --- |
| Ограничение framing и byte size | input укладывается в byte budget | syntax, nesting/count и смысл |
| Decode с parser limits | представление синтаксически корректно, parser budgets не превышены | required fields, ranges, domain rules |
| Structural validation | shape, types, presence, enum и локальные ranges допустимы | согласованность полей и current state |
| Domain constructor | command удовлетворяет invariants, не зависящим от внешнего state | authorization, uniqueness, balance, version |
| State transition под constraint/lock | переход допустим относительно authoritative state | успешность внешних зависимостей после commit |

Физические ограничения request body, nesting и decompression раскрыты в [[20 Бэкенд/Ограничения размера входных данных и исчерпание ресурсов|заметке об ограничении входных данных]]. Здесь важен порядок: parser не должен получить неограниченный stream, а проверки `Content-Length` недостаточно — значение сообщает отправитель и оно может отсутствовать или не совпасть с фактически прочитанным объёмом.

### Позитивный контракт сильнее denylist

Для перечислимого поля задают допустимые значения, для числа — representation и range, для collection — тип элементов и cardinality, для object — required/unknown-field policy. Затем проверяют отношения: `start < end`, currency счёта совпадает с currency операции, `from != to`, сумма частей равна total.

Denylist вроде «запретить `<script>` и несколько SQL-слов» не описывает допустимую domain value и легко пропускает другое представление. При этом корректная строка всё равно может стать опасной в другом sink. Validation отвечает «соответствует ли значение контракту», parameter binding — «может ли оно изменить структуру SQL», а context-specific encoding — «будет ли оно интерпретировано как HTML/URL/shell syntax». CWE-20 прямо предупреждает, что input validation не всегда достаточна и не заменяет другие protection mechanisms.

### Нормализация должна иметь одного владельца

Unicode normalization, case folding, trimming, URL decoding и преобразование чисел могут менять equivalence classes. Если gateway сравнил исходную строку, application нормализовало её иначе, а storage применило collation, два слоя могут принять разные решения об одном identity.

Контракт должен определить, что именно принимается и где выполняется canonicalization. Без такого правила безопаснее отвергнуть неоднозначное представление. Подписанные данные нельзя самовольно нормализовать до verification: signer и verifier должны построить одну и ту же signature base по правилам конкретного protocol, а не по локальным defaults одного слоя. Raw value также может понадобиться для forensic record, но в обычные логи её нельзя писать без ограничения и redaction.

### Invalid state лучше не представлять обычным типом

Transport DTO допускает частично заполненное состояние, потому что decoder должен куда-то поместить request. Domain object создаётся только через constructor, возвращающий либо значение с доказанными invariants, либо typed validation error. Тогда business function принимает `Transfer`, а не набор `string` и не повторяет проверки в каждой ветке.

Это не отменяет constraints БД. Предварительные `SELECT` и validation выполняются до конкурентной записи, поэтому два запроса могут одновременно увидеть свободный key. Unique/check/foreign-key constraint или conditional write остаётся authority; application переводит violation в стабильную domain error. Слои и wire-модель ошибок подробнее разобраны в [[20 Бэкенд/Валидация и модель ошибок API|валидации и модели ошибок API]].

### Ошибка ввода — ограниченный и наблюдаемый outcome

Handler обязан:

- завершить обработку до business side effect, если нарушен precondition этого side effect;
- вернуть документированную категорию и стабильный machine code, не заставляя client разбирать текст;
- не включать stack trace, SQL, filesystem path, credentials или полное исходное значение;
- ограничить cardinality метрик и объём логов, иначе поток bad requests сам создаст resource incident;
- различать client error, authorization denial, state conflict и transient dependency failure.

`panic` допустим как обнаружение нарушенного внутреннего invariant, но не как обычная реакция на внешний malformed input. Recovery middleware — последний containment layer, а не замена явной error branch.

### Совместимость — часть политики входа

Запрет unknown fields быстро обнаруживает typo и schema drift, но мешает additive rollout, если старый consumer должен переживать новое поле. Игнорирование unknown fields облегчает forward compatibility, но может скрыть ошибку имени. Выбор фиксируют для конкретной boundary и проверяют [[20 Бэкенд/Контракты API и обратная совместимость|контрактными правилами совместимости]], а не оставляют случайным default decoder.

Особенно опасна polymorphic deserialization по имени типа из payload: schema-valid discriminator не делает произвольный class безопасным. Allowlist типов и отсутствие выполнения constructors/gadgets на untrusted payload относятся к [[20 Бэкенд/Unsafe deserialization|защите от unsafe deserialization]].

## Пример или трассировка

Пусть endpoint принимает перевод:

```json
{
  "from": "account-17",
  "to": "account-17",
  "amount": "10.00",
  "currency": "USD"
}
```

Иллюстративная трассировка, без запуска кода:

1. Body помещается в byte limit, JSON содержит ровно один object, поля и string representations допустимы.
2. Parser суммы создаёт точное значение в minor units; float в domain model не попадает.
3. `NewTransfer` проверяет `amount > 0`, поддерживаемую currency и invariant `from != to`.
4. Последний invariant нарушен, поэтому transaction, ledger write и outbox insert даже не начинаются.
5. Handler возвращает `422` с problem type и code `SAME_ACCOUNT`; человекочитаемый `detail` можно изменить без изменения client branch.
6. Метрика увеличивается по bounded label `reason="same_account"`; исходные account IDs не становятся labels и не выводятся целиком в общий log.

Наблюдаемый invariant: после ответа нет transfer/ledger/outbox records. Запись о факте отклонённого запроса в защищённый audit trail возможна как отдельный, явно спроектированный security side effect; она не должна выглядеть как частично выполненный перевод.

## Trade-offs

| Выбор | Когда выигрывает | Цена и риск |
| --- | --- | --- |
| Strict rejection | typo опаснее forward compatibility; internal command должен быть точным | additive producer rollout требует согласования |
| Tolerant reader | version skew ожидаем и unknown fields не меняют semantics | ошибки имён и незамеченный drift |
| Reject non-canonical form | identity/security comparison должен быть однозначным | client обязан исправить представление |
| Canonicalize once | protocol явно задаёт equivalence и удобство важнее исходной формы | нужна единая реализация во всех слоях |
| Fail fast | дорогие проверки и side effects надо остановить рано | client исправляет ошибки по одной |
| Aggregate violations | form/batch client может исправить несколько полей за round trip | больше CPU, сложнее concealment и порядок errors |

Validation на edge уменьшает плохой traffic внутри системы, но не освобождает service от проверки своей boundary. Повторение всех правил в каждом слое тоже вредно: representation rules принадлежат transport, domain invariants — domain constructor, конкурентные invariants — authoritative storage.

## Типичные ошибки

- **«Внутренний producer уже валидировал» → malformed message вызывает panic после deploy → producer и consumer обновляются независимо, а очередь хранит старые сообщения → consumer проверяет собственный versioned contract и отправляет непринимаемое сообщение в bounded quarantine/DLQ.**
- **«JSON успешно распарсился» → zero value превращается в реальную команду → decoder не различил absent, `null`, empty и zero → DTO хранит presence отдельно, а constructor проверяет обязательность.**
- **«Проверим denylist» → payload проходит фильтр и меняет запрос или markup → validation перепутана с sink-specific neutralization → задают позитивный domain contract и отдельно используют parameterization/encoding.**
- **«Сделаем `SELECT`, затем `INSERT`» → под concurrency появляются duplicates → между check и write есть race → invariant закрепляют unique/conditional constraint и переводят conflict в domain error.**
- **«Любая ошибка во время handler означает bad request» → client не retry временный outage → dependency timeout классифицирован как validation error → сохраняют причинную категорию и применяют документированную retry policy.**
- **«Залогируем весь плохой request» → секреты попадают в logs, а attacker создаёт log/metric flood → наблюдаемость сама не ограничена → используют redaction, truncation, sampling и bounded reason codes.**

## Когда применять

Подход нужен на каждой границе, где данные могут иметь другую версию, владельца или lifecycle: HTTP/RPC, queue, file import, CLI/config, database migration, cache, webhook и third-party SDK. Интенсивность зависит от риска, но явные limits, typed parsing и domain invariants нужны даже для trusted operator input: человеческая ошибка не становится безопасной из-за источника.

Для исследования неизвестных malformed representations defensive checks дополняет [[60 Go/Fuzzing|fuzz testing]]. Для систематического перебора invariants на множестве сгенерированных valid inputs полезен [[20 Бэкенд/Property-based testing|property-based testing]]. Они находят контрпримеры, но не определяют контракт вместо команды.
