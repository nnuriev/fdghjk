# Управление секретами и rotation

## TL;DR

Secret management — управление полным lifecycle значения, владение которым даёт полномочие: создание, выдача, хранение, доставка, использование, renewal, rotation, revocation, audit и уничтожение. Secret manager закрывает только часть задачи. Если workload получает к нему общий вечный token, читает все paths, пишет secret в environment dump и не умеет обновиться без outage, защищённого lifecycle нет.

Rotation — распределённый протокол смены credential, а не команда «сгенерировать новое значение». Безопасная смена обычно проходит через versioning и overlap: создать новую версию, разрешить её у producer/verifier, обновить consumers, переключить новое использование, доказать сходимость, затем отозвать старую. Для encryption keys старый key material часто надо сохранить для decrypt или rewrap; бездумное удаление уничтожит данные.

Предпочтительный путь — не раздавать долгоживущий shared secret вообще: workload identity и short-lived dynamic credentials уменьшают окно компрометации и дают отдельный audit trail. Это переносит требования на bootstrap identity, control-plane availability, clock и автоматическое renewal.

## Ментальная модель

Секрет — time-bounded capability. Его полезно описывать не строкой, а записью:

```text
(secret_id, version, owner, consumers, allowed_use,
 issued_at, expires_at, rotation_state, revocation_state)
```

У lifecycle есть две плоскости:

- **control plane** выдаёт, версионирует, меняет, отзывает и аудитирует;
- **data plane** получает credential, применяет его к конкретной dependency и переживает обновление.

Цепочка доверия начинается раньше secret manager. Workload должен доказать, кто он, чтобы получить секрет. Этот bootstrap credential называют secret zero problem. Перенос вечного root token из файла в переменную не решает проблему; platform attestation, workload identity или ограниченный одноразовый bootstrap уменьшают её.

## Как устроено

### Что считать секретом

К секретам относятся passwords, API keys, OAuth client secrets, private signing/encryption keys, database credentials, webhook shared secrets и recovery codes. Public certificate, key ID и обычная configuration не секретны, хотя их целостность может быть критична. Разделение важно: secrecy не заменяет integrity, а попытка «спрятать» public identifier усложняет систему без защиты.

Cryptographic key management шире хранения secrets. У key есть algorithm, purpose, cryptoperiod, generation source, protection level, backup/recovery и правила уничтожения. NIST SP 800-57 Part 1 Rev. 5 требует связывать защиту key material с его типом и lifecycle; один и тот же key не стоит использовать для разных cryptographic purposes.

### Inventory и ownership

Нельзя менять то, существование и consumers чего неизвестны. Для каждой записи хранят:

- owner и on-call, который способен отозвать credential;
- issuer/source of truth и target system;
- consumers, environments и разрешённые actions/resources;
- способ delivery и места возможных copies;
- дату создания, последний rotation, expiry и максимальный incident revoke time;
- dependency order: что должно начать принимать новую version раньше, чем consumer её отправит.

Secret без owner, expiry и проверенного runbook уже operational debt. Discovery scanner помогает найти material в Git, images и configs, но совпадение по entropy не доказывает, что значение активно. Найденный возможный secret сначала классифицируют, затем revoke/rotate; удаление из последнего commit не удаляет историю и clones.

### Generation и хранение

Machine secret генерируют cryptographically secure random source с entropy, достаточной против offline/online guessing. Human password подчиняется другому threat model и не должен становиться signing key.

Secret store защищает ciphertext at rest, access policy, version history и audit. Но encryption at rest не мешает процессу с разрешением `get` получить plaintext. Главный boundary — authorization к конкретному path/version и изоляция key-encryption material от хранилища ciphertext.

Envelope encryption разделяет data encryption keys (DEK) и key encryption key (KEK): data шифруются DEK, а небольшой DEK wrapping выполняется KEK в KMS/HSM. Rotation KEK тогда может rewrap DEK без чтения и повторного шифрования всех data. Полная re-encryption нужна, если меняется DEK, algorithm или threat model.

### Bootstrap и доставка

Workload сначала аутентифицируется к control plane. Варианты:

- platform-issued identity по attestation процесса, pod, VM или hardware;
- short-lived certificate/token, автоматически доставляемый локальным agent;
- ограниченный bootstrap credential, который однократно меняется на рабочую identity;
- pre-shared secret как крайний вариант с малым scope и отдельной rotation.

SPIFFE Workload API v1.15.1 описывает локальную выдачу X.509-SVID/JWT-SVID и streaming updates при rotation, CRL или bundle changes. Workload не хранит заранее application secret, но доверяет локальному endpoint и его attestation caller. Stream client должен применять полный новый snapshot и удалять отсутствующий material; иначе revoked bundle способен остаться в памяти.

Delivery выбирают по приложению:

- direct pull через SDK даёт явный refresh/error handling, но связывает код с provider;
- local agent/sidecar пишет atomic file или предоставляет socket, упрощая integration, но приложение должно reload без partial read;
- mounted file лучше обновляется, чем immutable environment variable; env остаётся process-wide copy и часто читается только при startup;
- secret, записанный в обычный config object, image layer или command line, расширяет число copies и журналов.

Независимо от канала приложение не должно печатать raw value в logs, metrics, traces, crash dump или error. Redaction по имени поля недостаточна: secret способен оказаться в URL, nested JSON или library error.

### Static и dynamic credentials

Static secret существует независимо от consumer request и меняется по schedule или incident. Он совместим почти со всеми dependencies, зато распространяется по fleet и требует overlap/cutover.

Dynamic credential создаётся для конкретного workload/request с TTL и lease. Например, database secrets engine выдаёт отдельного DB user с ограниченной ролью, а lease expiry/revoke удаляет или инвалидирует его. Компрометация одной instance не раскрывает общий fleet password, audit точнее. Цена — issuer должен быть доступен для issuance/renewal, target integration обязана действительно отзывать право, а consumer должен переживать expiry.

Короткий TTL не равен мгновенному revoke. Уже открытая database connection может остаться активной после смены password, а certificate может приниматься до expiry, если verifier не проверяет revocation. Для каждого credential надо знать enforcement semantics target system.

### Rotation как state machine

Для shared authentication secret безопасный normal rotation выглядит так:

```text
OLD_ONLY
  -> NEW_CREATED
  -> ACCEPT_OLD_AND_NEW
  -> CONSUMERS_ON_NEW
  -> NEW_ONLY
  -> OLD_DESTROYED
```

1. Создать independent version `v2` с тем же минимальным scope; `v1` продолжает работать.
2. Сначала настроить verifier/target принимать `v2`. Для signature key опубликовать новый public key/JWK вместе со старым; для database создать второй login.
3. Доставить `v2` consumers. Приложение открывает новые connections/requests с `v2`, а старые постепенно drain.
4. Наблюдать usage по key/version и подтвердить, что все ожидаемые consumers перешли. Отсутствие error не доказывает это: редко запускаемый job мог не обратиться.
5. Переключить issuer/new writes на `v2`; выждать максимальные caches, queues и offline consumers согласно contract.
6. Отозвать `v1`, проверить negative use, затем удалить material по retention/recovery policy.

Emergency rotation при известной утечке сокращает или убирает overlap. Это увеличивает outage risk, поэтому runbook заранее определяет, кого отключить, как принудительно закрыть active sessions и как восстановить service.

Rotation encryption key отличается. Reader должен уметь определить key version из metadata и расшифровать старый ciphertext до re-encryption/rewrap. Сначала переключают новые writes, мигрируют old data и доказывают отсутствие нужных ссылок и backup-зависимостей от old key. Затем уничтожают все target copies старого key material и только после этого валидируют cryptographic erase; одна созданная новая key version ничего не стирает.

### Revocation, audit и failure modes

Revocation должна доходить до реального enforcement point. Secret manager может пометить lease revoked, но target DB/cloud provider должен удалить credential. Метрики измеряют issuance, renewal failures, age, usage старой version, revoke latency и unauthorized reads; raw secret не входит в audit.

Control plane outage требует выбранного поведения. Уже запущенный workload способен продолжить с cached credential до TTL, но не должен использовать его после expiry без явного degraded contract. Слишком короткий TTL превращает issuer outage или clock skew в массовый отказ; слишком длинный увеличивает blast radius. Renewal выполняют заранее с jitter, чтобы fleet не обновлялся одновременно.

## Пример или трассировка

`payments-api` использует общий PostgreSQL credential `payments_app/v17`. Требуется rotation без остановки fleet из 20 instances.

1. DBA создаёт login `payments_app_v18` с той же узкой DB role: `SELECT/INSERT/UPDATE` только нужных tables, без DDL и role management.
2. Secret manager сохраняет version `v18`; `v17` остаётся active. Application instances получают update и сначала проверяют новый credential отдельным connection.
3. Каждая instance создаёт новый pool с `v18`, атомарно делает его current для новых queries, а pool `v17` переводит в drain. Ошибка подключения `v18` не уничтожает ещё рабочий `v17` pool.
4. Telemetry по DB username показывает, что все живые instances и scheduled jobs использовали `v18`; deployment inventory подтверждает отсутствие старой revision.
5. DBA запрещает login `payments_app_v17` и принудительно завершает его sessions, если rotation вызван compromise. Negative probe с `v17` получает authentication failure.
6. После retention окна secret manager удаляет доступ к plaintext `v17`, а audit сохраняет только ID, timestamps, actors и outcome.

Наблюдаемый результат: новые запросы проходят через `v18`, украденный `v17` больше не создаёт и не сохраняет connection, а cutover не обрывает healthy fleet до проверки новой version. Остаточный риск во время overlap — оба credentials действуют; его ограничивают короткое окно, одинаковый least privilege и мониторинг usage `v17`.

## Trade-offs

### Static или dynamic secret

Static credential проще для legacy dependency и работает при outage issuer, пока не истёк/не изменён. Он дольше живёт и чаще разделяется. Dynamic lease уменьшает lifetime и улучшает attribution, но добавляет issuance/renewal path и требует capacity для churn. Выбор задаёт target capability и допустимый revoke window.

### Push или pull

Push быстро доставляет update, но control plane должен адресовать consumers и безопасно повторять события. Pull проще масштабирует ownership: consumer сам обновляется по TTL/version, зато rollout задерживается до poll. Streaming API объединяет свойства, но требует reconnect, full-snapshot semantics и backoff.

### Короткий TTL или availability

Чем короче TTL, тем меньше окно украденного credential. Одновременно растут load issuer, чувствительность к clock/network outage и вероятность synchronized renewal. TTL выбирают вместе с renewal margin, jitter, offline tolerance и emergency revoke, а не как отдельное «безопасное» число.

### Центральный secret manager или platform identity

Manager даёт inventory, policy, audit и integrations, но становится критичным control plane и привлекательной целью. Platform identity убирает application secrets и автоматизирует rotation, зато переносит trust на attestation, node isolation и federation. На практике identity выдаёт доступ к manager или напрямую к dependency; оба слоя ограничивают независимо.

### Environment variable или file/socket

Environment поддерживается почти везде, но обычно фиксируется при startup, наследуется child processes и попадает в diagnostics. Atomic file/socket позволяет live reload и более узкие permissions, но приложение должно корректно обрабатывать replacement и не читать partial state. Ни один канал сам по себе не защищает plaintext после чтения процессом.

## Типичные ошибки

### Secret зашифрован в Git

- **Неверное предположение:** encryption файла делает repository подходящим runtime secret store.
- **Симптом:** decryption key широко раздан, old versions остаются в history, revoke неясен.
- **Причина:** distribution и lifecycle ключа шифрования не отделены от ciphertext.
- **Исправление:** хранить reference/config в Git, material — в управляемом store; найденный active secret revoke/rotate, а не только удалить строку.

### Один credential у всех consumers

- **Неверное предположение:** общий ключ снижает operational cost.
- **Симптом:** невозможно определить источник утечки, revoke останавливает весь fleet.
- **Причина:** identities и blast radius объединены.
- **Исправление:** per-service/per-environment, а где возможно per-instance dynamic credentials с минимальным scope.

### Сначала отозвать старую version

- **Неверное предположение:** consumers мгновенно прочитают новое значение.
- **Симптом:** fleet одновременно теряет DB/provider access.
- **Причина:** verifier readiness, caches, connection pools и offline jobs не вошли в protocol.
- **Исправление:** versioned overlap, readiness probe, usage telemetry, затем revoke; emergency path тестировать отдельно.

### Удалить старый encryption key

- **Неверное предположение:** rotation ключа касается только новых writes.
- **Симптом:** старые records и backups больше не decrypt.
- **Причина:** нет key version metadata и migration inventory.
- **Исправление:** сохранить decrypt capability до verified rewrap/re-encryption всех copies, затем уничтожить по policy.

### Secret manager с broad read

- **Неверное предположение:** центральное защищённое хранилище делает любой доступ безопасным.
- **Симптом:** compromised service выгружает secrets других environments/services.
- **Причина:** storage protection подменила authorization.
- **Исправление:** workload identity, path-level least privilege, запрет list/watch без необходимости и anomaly audit.

### Rotation существует только на бумаге

- **Неверное предположение:** schedule достаточно, даже если никто не проверял cutover.
- **Симптом:** первый incident rotation заканчивается outage или оставляет старый credential active.
- **Причина:** нет observable state machine и negative verification.
- **Исправление:** регулярно выполнять rotation, измерять age/old-version usage и проверять revoke у target.

## Когда применять

Управляемый lifecycle нужен каждому credential, который открывает production data, control plane, signing, encryption или внешнюю интеграцию. Сначала удаляют ненужный secret: заменяют shared cloud key на workload identity, password — на short-lived database lease, ручной certificate — на автоматизированную PKI. Оставшийся material получает owner, scope, expiry, version и tested revoke.

Перед production проверяют normal rotation, emergency compromise, control-plane outage, missed update, stale consumer и rollback. Rotation считается завершённой, когда старое значение отклоняет target system и telemetry не показывает его использование, а не когда в secret manager появилась новая version. Принцип ограничения каждого consumer подробнее раскрыт в [[20 Бэкенд/Least privilege|заметке о least privilege]].
