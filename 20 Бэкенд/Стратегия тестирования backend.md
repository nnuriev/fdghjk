# Стратегия тестирования backend

## TL;DR

Стратегия тестирования backend — это не фиксированная пропорция unit, integration и end-to-end tests, а портфель доказательств под конкретные риски. Для каждого значимого свойства выбирают самый узкий и быстрый тест, который способен наблюдать его нарушение на правильной границе. Более широкие тесты оставляют для стыков, конфигурации и пользовательских путей, которые нельзя доказать внутри одного компонента.

Хорошая стратегия отвечает на пять вопросов: что обязано оставаться истинным, где это свойство может сломаться, какой тест действительно увидит поломку, когда он запускается и кто разбирает его отказ. Процент покрытия кода и число тестов могут помочь найти пробелы, но сами по себе не отвечают ни на один из этих вопросов.

## Область применимости

Заметка относится к backend-сервисам с HTTP/RPC API, базами данных, очередями и внешними зависимостями. Она охватывает функциональную корректность, совместимость, reliability, security и performance как разные классы риска. Конкретная CI-система и выбор test framework остаются вне scope.

Названия уровней неоднозначны между командами. Поэтому у каждого suite фиксируют:

1. **system under test (SUT)** — какой код и какие процессы проверяются;
2. **реальные и заменённые зависимости** — например, реальный PostgreSQL, но fake платёжной системы;
3. **точку входа** — вызов функции, adapter API или публичный endpoint;
4. **test oracle** — какой наблюдаемый результат означает успех или нарушение.

## Ментальная модель

Тест уменьшает неопределённость только относительно сформулированной гипотезы. Если unit test доказал правило расчёта комиссии, из этого не следует, что миграция создала нужное ограничение в базе. Если end-to-end test провёл один платёж, из этого не следует, что все граничные значения рассчитаны верно.

Удобная цепочка выбора выглядит так:

```text
риск -> место возможной поломки -> минимальная наблюдаемая граница
     -> управляемый сценарий -> oracle -> release gate
```

Отсюда следует практическое правило: уровень теста выбирают по месту риска, а не по важности функции. Критический предметный инвариант часто лучше всего доказывается быстрыми [[20 Бэкенд/Unit tests|unit tests]], а несовместимость SQL, wire format или конфигурации требует более широкой границы независимо от объёма кода.

## Как устроено

### 1. Начать с рисков и контрактов

Перед выбором тестов выписывают свойства, потеря которых изменит решение пользователя или оператора:

- предметные инварианты и переходы состояния;
- внешние API и message contracts;
- транзакционные границы, migrations и constraints;
- поведение при invalid input, timeout, cancellation, retry и частичном отказе;
- concurrency и ownership ресурсов;
- SLO, capacity и контролируемое поведение при overload;
- security controls и границы tenant/authorization.

Риск описывают наблюдаемым нарушением: не «проверить базу», а «две конкурентные команды не должны создать две операции с одним idempotency key». Такое утверждение уже подсказывает нужный scope, нагрузку и oracle.

### 2. Разместить доказательство на минимальной достаточной границе

| Риск | Основное доказательство | Что остаётся проверить шире |
| --- | --- | --- |
| Ошибка в decision logic и граничных значениях | Unit test без внешнего I/O | Wiring и реальные adapters |
| Несовпадение SQL, driver, migration или broker semantics | [[20 Бэкенд/Integration tests|Integration test]] с рискованной реальной зависимостью | Полный пользовательский поток |
| Независимо выпускаемые consumer и provider стали несовместимы | [[20 Бэкенд/Contract tests|Contract test]] обеих сторон | Emergent behavior собранной системы |
| Критический путь не проходит через собранные процессы и конфигурацию | Небольшое число [[20 Бэкенд/End-to-end tests|end-to-end tests]] | Production-only topology и реальные масштабы |
| Система нарушает SLO или рушится после saturation | [[70 Практические кейсы/Load и stress testing|Load, stress и recovery test]] | Неизвестный production workload |
| Дефект зависит от scheduling, input space или отказа | Race, deterministic concurrency, fuzz/property или fault-injection test | Комбинации, не вошедшие в модель |

Эти категории пересекаются. Contract test может запускать настоящий provider process, а integration test — идти через HTTP. Решающее различие не transport, а проверяемое свойство и объявленная граница.

### 3. Проверять не только happy path

Для каждого критического пути строят минимум три класса сценариев:

- допустимый вход и ожидаемый side effect;
- граничный или недопустимый вход без запрещённого side effect;
- отказ зависимости до, во время и после возможного commit.

Особенно важна последняя граница. Timeout не сообщает, был ли удалённый side effect выполнен. Поэтому тест должен различать «операция точно не началась», «откатилась» и «результат неизвестен»; иначе suite закрепляет опасную модель ошибки.

### 4. Разделить feedback loops

- На каждом изменении запускают быстрые deterministic suites, contract verification затронутых границ и критические integration tests.
- Более дорогие end-to-end, compatibility matrix и security tests можно шардировать или запускать как merge/release gates.
- Load, stress, soak, fault injection и disaster scenarios запускают по расписанию и перед capacity-sensitive изменениями.
- После deploy используют canary signals и безопасные synthetic probes. Они проверяют реальную конфигурацию, но не заменяют pre-production tests: production state не герметичен, а диагностика и rollback дороже.

Gate должен быть связан с риском. Flaky test нельзя молча повторять до зелёного: первый отказ остаётся сигналом о недетерминизме, а retry лишь даёт диагностические данные.

### 5. Управлять качеством самого suite

Тесты — production-код с собственными failure modes. Для каждого suite нужны владелец, ограниченный runtime, изолированные данные, фиксированные версии окружения и артефакты диагностики. Метрики suite включают не только pass rate, но и время feedback, долю flaky runs, частоту quarantine, время локализации и дефекты, дошедшие до production.

Coverage полезен как карта выполненных путей: нулевая coverage почти наверняка показывает пробел. Но высокая statement coverage не доказывает assertions, граничные значения, concurrency histories или правильность внешнего контракта.

## Пример или трассировка

Команда добавляет `POST /transfers`. Контракт: сумма положительна, счета различны, один `Idempotency-Key` создаёт не более одного перевода, debit и outbox event фиксируются атомарно, а старый mobile consumer продолжает понимать ответ.

1. Unit table проверяет `amount <= 0`, одинаковые счета, нехватку средств и допустимый переход состояния. Он доказывает decision logic, но не SQL transaction.
2. Integration test поднимает поддерживаемую версию PostgreSQL, применяет production migrations и дважды конкурентно вызывает repository с одним ключом. Oracle: одна запись transfer, один debit и один outbox row.
3. Consumer-driven contract фиксирует минимально нужные mobile client поля и error shape. Provider verification блокирует удаление или переименование используемого поля.
4. Один end-to-end scenario отправляет запрос через публичный API, ждёт обработку outbox и читает итоговый статус тем же клиентским API. Внешний банк заменён sandbox, явно находящимся вне SUT.
5. Отдельный fault scenario обрывает ответ после commit и проверяет, что повтор с тем же ключом возвращает исходную операцию, а не создаёт новую.

Если все unit tests проходят, а integration test на второй записи получает duplicate debit, причина локализована на transaction/constraint boundary. Добавление ещё одного unit mock не закроет этот риск. Если integration проходит, но старый consumer не декодирует новый ответ, отсутствует или неверен contract oracle.

## Trade-offs

Чем уже тест, тем обычно быстрее feedback, проще управление входами и точнее локализация. Цена — меньше реальной сборки и protocol semantics. Чем шире тест, тем выше fidelity конкретной конфигурации, но больше причин отказа, runtime и стоимость поддержки. Портфель нужен именно потому, что один уровень не оптимизирует обе стороны одновременно.

Большое число unit tests выгодно для combinatorial decision logic. Большое число integration tests оправдано в тонком сервисе, чья основная сложность — SQL, broker или cloud API. Универсальное соотношение слоёв без учёта архитектуры превращается в метрику активности.

Mocks удешевляют failure injection, но могут доказать только ожидания автора. Реальная dependency повышает fidelity, но делает suite дороже и иногда недетерминированнее. Правило выбора раскрыто в [[20 Бэкенд/Mocks vs fakes|сравнении mocks и fakes]]: заменяют только ту границу, чья реальная semantics не является предметом текущего теста.

## Типичные ошибки

- **Неверное предположение:** высокий code coverage означает достаточную проверку. **Симптом:** production ломается на хорошо покрытом path. **Причина:** строки выполнялись без сильного oracle или без нужной history. **Исправление:** вести risk-to-test matrix и проверять observable invariants, boundary и failure paths.
- **Неверное предположение:** критический сценарий нужно целиком проверять только end-to-end. **Симптом:** медленный flaky suite и долгий поиск причины. **Причина:** все комбинации подняты на самую дорогую границу. **Исправление:** вынести decision space в unit/property tests, стыки — в integration/contract, оставить E2E для нескольких journeys.
- **Неверное предположение:** mock внешнего сервиса доказывает совместимость. **Симптом:** тест зелёный, реальный provider отвергает запрос. **Причина:** mock повторяет представление consumer. **Исправление:** provider verification или integration с поддерживаемой реальной версией.
- **Неверное предположение:** retry превращает flaky test в надёжный gate. **Симптом:** изменения проходят после нескольких попыток. **Причина:** недетерминизм скрыт агрегацией результата. **Исправление:** сохранять первый failure, локализовать shared state/time/readiness и quarantine только с владельцем и сроком.
- **Неверное предположение:** happy path представляет failure semantics. **Симптом:** retry после timeout дублирует side effect. **Причина:** не проверена commit boundary. **Исправление:** тестировать отказ до commit, после commit и ambiguous outcome отдельно.
- **Неверное предположение:** тестовый binary достаточно, а migration/configuration не являются кодом. **Симптом:** release падает при старте или работает с неверным default. **Причина:** собранный artifact и production-like configuration не вошли в SUT. **Исправление:** startup/release test на том же artifact, migrations и versioned config.

## Когда применять

Стратегию формируют вместе с design: у каждого инварианта и внешней границы должен появиться владелец доказательства до реализации. Её пересматривают после изменения архитектуры, dependency version, SLO и после каждого escaped defect: новый тест размещают на самой узкой границе, которая воспроизводит корневую причину, а не только production-симптом.

Практический критерий готовности — команда может для каждого критического риска указать тест, oracle, среду, cadence и остаточную неопределённость. Если ответ сводится к общему числу тестов или coverage, стратегии ещё нет.
