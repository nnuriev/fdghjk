# Ключи идемпотентности и дедупликация запросов

## TL;DR

Idempotency key связывает несколько попыток запроса с одной логической операцией. Сервер атомарно резервирует ключ в области пользователя и endpoint, сверяет fingerprint значимых параметров, а после выполнения сохраняет терминальный исход. Повтор с тем же ключом и тем же запросом возвращает сохранённый исход; тот же ключ с другим запросом отклоняется.

Ключ делает безопасным retry только в пределах объявленного retention и только для эффекта, охваченного протоколом. Он не закрывает окно между локальной транзакцией и внешним побочным эффектом, не заменяет доменный уникальный ключ и не превращает произвольный workflow в `exactly once`.

## Область применимости

Заметка относится прежде всего к HTTP-операциям создания и запуска действия, обычно `POST`, которые клиент может повторить после неопределённого исхода. Идемпотентность методов определена RFC 9110. Заголовок `Idempotency-Key` на дату проверки описан только истёкшим Internet-Draft `draft-ietf-httpapi-idempotency-key-header-07`, а не стандартом RFC; конкретный контракт должен документировать сам API.

## Ментальная модель

Idempotency key — не идентификатор сетевого запроса, а имя логического намерения клиента:

```text
(principal, operation, key) -> fingerprint + state + outcome + expires_at
```

Например, повторные попытки «создать платёж для заказа 42» получают один ключ. Два разных платежа, даже с одинаковыми параметрами, получают разные ключи. Сервер хранит не факт «такой пакет уже видел», а состояние конечного автомата операции.

```text
absent -> in_progress -> succeeded
                     \-> failed_terminal
```

`in_progress` нужен, чтобы два параллельных запроса не начали один эффект одновременно. Для зависшей записи требуется lease или процедура reconciliation: автоматически считать её успехом либо неуспехом нельзя.

## Как это устроено

### 1. Контракт ключа

API определяет:

- на каких операциях ключ обязателен или поддерживается;
- формат, максимальную длину и источник уникальности;
- область уникальности: tenant/principal плюс конкретная операция;
- сколько хранится результат и что происходит после expiry;
- что возвращается при параллельном запросе и при несовпадении payload;
- воспроизводятся ли terminal `4xx` и `5xx` либо только успешные результаты.

Клиент генерирует непрозрачный высокоэнтропийный ключ и повторяет его только для того же намерения. Сервер не должен глобально объединять одинаковые ключи разных tenants. После истечения retention прежний ключ может означать новую операцию, поэтому retention должен покрывать максимальный retry horizon клиентов и посредников.

RFC 9110 называет `PUT`, `DELETE` и safe methods идемпотентными по их **предполагаемому эффекту**: одинаковые запросы могут вернуть разные статусы и ответы. `POST` не получает эту семантику автоматически, но API может сделать конкретную POST-операцию повторяемой через свой контракт ключа.

### 2. Fingerprint запроса

С ключом сохраняют hash канонического набора параметров, влияющих на эффект. Повтор с другим fingerprint должен завершаться конфликтом контракта, а не возвращать чужой старый результат. Нельзя бездумно хешировать сырые bytes JSON: различия порядка полей и пробелов не меняют смысл. Лучше строить fingerprint после валидации из нормализованной команды и версии операции.

Authorization context входит в область ключа, но bearer token и чувствительный payload не следует сохранять в открытом виде. Смена прав между попытками требует отдельного решения: replay ранее созданного ресурса не должен раскрывать его principal, который больше не имеет доступа.

### 3. Атомарная резервация и эффект

Уникальное ограничение на `(principal_id, operation, key)` — точка сериализации конкурирующих попыток. Дальше возможны две схемы:

- если бизнес-эффект локален одной БД, запись результата дедупликации и бизнес-изменение фиксируют одной транзакцией;
- если операция длительная или вызывает внешнюю систему, коротко фиксируют `in_progress`, выполняют workflow по собственному бизнес-ключу, а затем сохраняют исход. Падение оставляет запись для reconciliation, а не разрешение слепо повторить внешний эффект.

Нельзя удерживать открытую DB-транзакцию и row lock во время долгого сетевого вызова только ради дедупликации: это увеличивает lock contention и не делает внешний ресурс участником транзакции. Внешней стороне передают тот же бизнес-ключ, если она поддерживает идемпотентность, либо отделяют публикацию через [[40 Распределённые системы/Transactional outbox и Change Data Capture|outbox]].

### 4. Replay исхода

После терминального завершения сервер хранит status code, стабильное представление результата или идентификатор ресурса. Первый ответ мог потеряться после коммита; повтор должен позволить клиенту узнать уже состоявшийся исход.

Возвращать byte-for-byte тот же ответ не всегда разумно: представление ресурса могло измениться, а временные headers устареть. Контракт может обещать сохранённый status/body либо ссылку на тот же outcome. Важно не смешивать эти варианты неявно. Stripe API v1, например, документирует сохранение первого status и body, включая `500`, но это политика конкретного API, а не общее требование HTTP.

## Сквозной пример: создание платежа

Клиент отправляет:

```http
POST /v1/payments
Idempotency-Key: "8b7a53d1-4124-4d3d-b7f2-3f2d13f7272c"

{"order_id":"42","amount":1000,"currency":"RUB"}
```

1. Сервер аутентифицирует tenant `t1`, валидирует команду и вычисляет fingerprint `(42, 1000, RUB, v1)`.
2. Уникальная вставка `(t1, create-payment, 8b7a53d1-4124-4d3d-b7f2-3f2d13f7272c)` создаёт `in_progress`. Параллельная попытка с тем же fingerprint получает документированный `409 Conflict` и может повторить позже.
3. Платёж создаётся с доменным уникальным ключом `(t1, order_id=42, kind=charge)`. Результат `payment_id=p7` и `201 Created` фиксируются вместе с терминальным состоянием дедупликации.
4. Ответ теряется в сети. Клиент повторяет запрос с тем же ключом; сервер возвращает сохранённый исход для `p7`, не создавая `p8`.
5. Повтор с тем же ключом, но `amount=2000` имеет другой fingerprint и получает ошибку переиспользования ключа.

Если платёжный provider успел списать деньги, а процесс упал до локальной фиксации, одной таблицы ключей недостаточно. Provider должен принять согласованный idempotency key либо workflow должен сверять его состояние. Именно это окно отличает дедупликацию попыток от end-to-end `exactly once`.

## Trade-offs и альтернативы

### Idempotency key и доменный уникальный ключ

Idempotency key принадлежит протоколу клиента и имеет retention. Доменный ключ выражает постоянный инвариант, например «не больше одного capture для payment». Первый удобен для восстановления ответа, второй защищает данные даже от другого producer. Для денежных и иных критичных операций обычно нужны оба.

### Сохранённый ответ или повторное чтение ресурса

Снимок ответа точно воспроизводит первоначальный outcome, но занимает место и может хранить чувствительные данные или устаревшие headers. Ссылка на созданный ресурс компактнее и отдаёт актуальное представление, но не воспроизводит исходный body. Выбор фиксируют в API contract.

### Клиентский или серверный ключ

Только клиент знает, какие transport attempts относятся к одному намерению, поэтому ключ обычно создаёт клиент. Серверный request ID полезен для tracing, но появляется слишком поздно, чтобы связать потерянный ответ с новой попыткой клиента.

## Типичные ошибки

### Проверка «сначала SELECT, потом INSERT»

- **Неверное предположение:** отсутствие строки останется истинным до записи.
- **Симптом:** два параллельных запроса оба выполняют эффект.
- **Причина:** check и reserve не атомарны.
- **Исправление:** уникальное ограничение и одна атомарная вставка/транзакция.

### Ключ без fingerprint

- **Неверное предположение:** клиент никогда не переиспользует ключ.
- **Симптом:** запрос на новую сумму получает результат старого платежа.
- **Причина:** сервер сравнил только строку ключа.
- **Исправление:** сохранять fingerprint нормализованной команды и отклонять несовпадение.

### Слишком короткий retention

- **Неверное предположение:** все retry происходят сразу.
- **Симптом:** отложенный повтор создаёт второй ресурс.
- **Причина:** запись дедупликации удалена раньше retry horizon.
- **Исправление:** согласовать TTL с клиентским SLA и хранить постоянный доменный инвариант там, где дубль недопустим никогда.

### `in_progress` без восстановления

- **Неверное предположение:** owner записи обязательно завершит её.
- **Симптом:** ключ навсегда возвращает conflict после crash.
- **Причина:** нет lease, статуса внешней операции или reconciliation.
- **Исправление:** описать recovery state machine и сделать повтор эффекта безопасным.

## Когда применять

Ключ нужен для операций, которые клиент вправе повторить после timeout или разрыва соединения, но повторный эффект недопустим: платежей, заказов, выдачи ресурсов, запуска долгой задачи. Для естественно идемпотентного `PUT` он всё равно может быть полезен, если надо воспроизвести outcome сложного workflow, но не должен дублировать простой ресурсный инвариант без причины.

Перед добавлением заголовка следует определить scope, fingerprint, поведение concurrent duplicate, terminal outcomes, retention и recovery зависшего `in_progress`. Без этих частей «поддерживаем idempotency key» остаётся названием, а не контрактом.
